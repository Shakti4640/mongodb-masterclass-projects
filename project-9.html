🔴 REDSHIFT MINDSET — How You Would Solve This

The Sort Key Approach You Know

-- REDSHIFT: You define performance optimization AT TABLE CREATION TIME
-- This is your instinct — plan performance BEFORE data arrives

CREATE TABLE store.customers (
    customer_id     BIGINT IDENTITY(1,1),
    email           VARCHAR(255) NOT NULL,
    name            VARCHAR(200),
    city            VARCHAR(100),
    loyalty_tier    VARCHAR(20),
    total_spent     DECIMAL(12,2),
    signup_date     TIMESTAMP,
    last_login      TIMESTAMP
)
DISTSTYLE KEY
DISTKEY(customer_id)           -- distribute rows across slices by customer_id
SORTKEY(email);                -- physically sort data by email on each slice

-- WHY email as sort key?
-- Because login (lookup by email) is the most frequent query
-- Sort key → zone maps → Redshift can skip blocks where email isn't in range

How Redshift Sort Keys Actually Work

REDSHIFT SORT KEY INTERNALS (what you already know):

1. DATA IS PHYSICALLY SORTED on disk by sort key column(s)
   → All rows on each slice are ordered by email
   → email: "alice@..." comes before "bob@..." on disk

2. ZONE MAPS are auto-created per 1MB block
   → Block 001: min="aaa@...", max="azz@..."
   → Block 002: min="baa@...", max="bzz@..."
   → Block 003: min="caa@...", max="czz@..."

3. QUERY: WHERE email = 'john@example.com'
   → Leader node checks zone maps
   → Block 001: max="azz@..." < "john@..." → SKIP
   → Block 002: max="bzz@..." < "john@..." → SKIP
   → ...
   → Block 010: min="jaa@..." ≤ "john@..." ≤ max="jzz@..." → READ
   → Only reads ~1 block out of thousands → FAST

4. LIMITATIONS:
   → Only ONE sort key order (compound sort key = ordered priority)
   → Cannot add sort key after table creation
   → Must recreate table to change sort key
   → Sort key only helps queries that filter on sort key columns
   → WHERE city = 'New York' gets ZERO benefit from email sort key
   → Must VACUUM after bulk loads to re-sort new data

REDSHIFT INTERLEAVED SORT KEY (alternative):
   → INTERLEAVED SORTKEY(email, city, signup_date)
   → Equal weight to all columns
   → Helps queries on ANY of the columns
   → But: slower VACUUM, less effective per column than compound

Redshift's Hidden Truth About Indexes

CRITICAL INSIGHT MOST REDSHIFT USERS DON'T REALIZE:

    REDSHIFT HAS NO INDEXES
    
    → No B-tree index
    → No hash index
    → No bitmap index
    → No index at all
    
    → Sort keys are NOT indexes
    → Sort keys are PHYSICAL SORT ORDER
    → Zone maps are NOT indexes
    → Zone maps are MIN/MAX metadata per block
    
    → Redshift relies on:
      1. Columnar storage (read only needed columns)
      2. Zone maps (skip blocks based on min/max)
      3. Compression (read less data from disk)
      4. MPP (parallel scan across nodes)
    
    → For point lookups (find ONE row by value):
      Redshift is TERRIBLE compared to MongoDB
      Even with perfect sort key, it scans entire blocks
      Minimum read unit = 1MB block
      
    → For analytical scans (aggregate millions of rows):
      Redshift is EXCELLENT compared to MongoDB
      Columnar + compression + MPP = blazing fast scans

THIS IS WHY YOU NEED MONGODB FOR OPERATIONAL WORKLOADS
    → MongoDB has REAL indexes (B-tree)
    → Point lookup: index → leaf node → document → done
    → No block scanning, no zone map checking
    → O(log n) lookup time regardless of collection size

🟢 MONGODB MINDSET — How This Same Problem Is Approached

The Fundamental Difference: Indexes Are Separate Structures

MONGODB INDEX PHILOSOPHY:

    REDSHIFT APPROACH:
        → Performance optimization = part of TABLE DEFINITION
        → Sort key chosen at CREATE TABLE time
        → Physical data order = the optimization
        → One sort key per table (compound or interleaved)
        → Change requires table recreation

    MONGODB APPROACH:
        → Performance optimization = SEPARATE from collection
        → Indexes created ANYTIME (before or after data)
        → Separate B-tree data structure = the optimization
        → UNLIMITED indexes per collection (practical limit ~50)
        → Add/remove indexes without touching data
        → Each index serves DIFFERENT query patterns

    ANALOGY:
        Redshift sort key = organizing a bookshelf alphabetically
            → ONE sort order
            → Finding by author is fast
            → Finding by title requires scanning everything
        
        MongoDB indexes = book's INDEX pages (back of book)
            → MULTIPLE indexes (by author, by topic, by date)
            → Each index points directly to the right page
            → Adding a new index = adding new pages at the back
            → Book content (documents) stays unchanged

What Happens Without an Index

CURRENT STATE (before we add any index):
    
    Collection: customers (10 million documents)
    Indexes: only _id (auto-created in Project 1)

    Query: db.customers.find({ email: "john@example.com" })
    
    WHAT MONGODB DOES:
        → No index on email field
        → Must perform COLLECTION SCAN (COLLSCAN)
        → Read EVERY document (all 10 million)
        → Check each document's email field
        → Return matching document(s)
        → Time: 8-15 seconds on 10M documents
    
    THIS IS LIKE:
        → Redshift table with NO sort key on the filtered column
        → Full table scan — every block on every slice
        → Except WORSE because MongoDB reads entire documents
          (Redshift at least only reads the email column)

🧩 HOW TO APPROACH IT IN MONGODB

THINKING FRAMEWORK FOR INDEXING:

REDSHIFT THINKING (what you're unlearning):
    1. Know the most important query → BEFORE creating table
    2. Choose sort key to match that query → AT table creation
    3. Accept that other queries won't benefit → TRADE-OFF
    4. Maybe use interleaved sort key for multiple queries → COMPROMISE
    5. Never change it without recreating table → LOCKED IN

MONGODB THINKING (what you're learning):
    1. Load data first → GET STARTED
    2. Run queries, observe what's slow → MEASURE
    3. Use explain() to see COLLSCAN → DIAGNOSE
    4. Create index on the slow query's filter field → FIX
    5. Use explain() again to verify IXSCAN → VERIFY
    6. Repeat for each slow query → ITERATE
    7. Drop unused indexes later → MAINTAIN

KEY INSIGHT:
    Redshift: optimize once, at design time, based on predicted queries
    MongoDB:  optimize iteratively, at runtime, based on actual queries
    
    This means MongoDB lets you DEFER optimization decisions
    But it also means you MUST come back and optimize
    "It works without indexes" ≠ "It works WELL without indexes"

Step 5: Understanding the explain() Output Deeply

// ═══════════════════════════════════════════════════════
// explain() DEEP DIVE — Your New EXPLAIN
// ═══════════════════════════════════════════════════════

// ─── READING EXPLAIN STAGES ──────────────────────────
// MongoDB explain shows a TREE of stages (like Redshift EXPLAIN)
// Data flows from INNER stages to OUTER stages

// COMMON STAGE TYPES:

// COLLSCAN
//   → Full collection scan (worst case)
//   → Reads EVERY document
//   → Equivalent to: Redshift sequential scan with no sort key benefit
//   → YOU NEVER WANT THIS ON LARGE COLLECTIONS

// IXSCAN
//   → Index scan (B-tree traversal)
//   → Reads index entries matching the query
//   → Equivalent to: Redshift zone map skip (but much more precise)
//   → THIS IS WHAT YOU WANT

// FETCH
//   → Retrieves full documents using pointers from IXSCAN
//   → Goes from index → actual document on disk
//   → Equivalent to: Redshift reading remaining columns after sort key filter

// PROJECTION_COVERED (or no FETCH stage)
//   → All needed fields are IN the index
//   → No need to fetch full document
//   → Equivalent to: Redshift reading only the sort key column
//   → FASTEST possible query

// SORT
//   → In-memory sort (index couldn't provide sort order)
//   → Expensive for large result sets

# Helper function to analyze explain output
def analyze_explain(collection, filter_doc=None, sort_doc=None, limit_val=None):
    """Run explain and extract key metrics"""
    cursor = collection.find(filter_doc or {})
    if sort_doc:
        cursor = cursor.sort(list(sort_doc.items()))
    if limit_val:
        cursor = cursor.limit(limit_val)
    
    explain = cursor.explain()
    
    plan = explain["queryPlanner"]["winningPlan"]
    stats = explain["executionStats"]
    
    # Find the deepest stage (IXSCAN or COLLSCAN)
    stage = plan
    index_name = None
    while "inputStage" in stage:
        stage = stage["inputStage"]
    scan_type = stage.get("stage", "unknown")
    index_name = stage.get("indexName", "none")
    
    return {
        "scan_type": scan_type,
        "index_used": index_name,
        "keys_examined": stats["totalKeysExamined"],
        "docs_examined": stats["totalDocsExamined"],
        "docs_returned": stats["nReturned"],
        "time_ms": stats["executionTimeMillis"],
        "efficiency": f"{stats['totalDocsExamined']}:{stats['nReturned']}"
    }

//   Direction DOESN'T matter for equality queries
//   { email: 1 } and { email: -1 } perform identically for:
//     db.customers.find({ email: "user@example.com" })

// DIRECTION MATTERS FOR:
//   1. SORT queries on this field
//   2. COMPOUND indexes (Project 10)

db.customers.find().sort({ signup_date: -1 }).limit(10).explain("executionStats")

// Stage: IXSCAN, direction: "backward" → ALSO EFFICIENT (slightly slower)

// FOR COMPOUND INDEXES: direction becomes CRITICAL (Project 10)

⚡ WHAT COULD BE DONE (MongoDB-Specific Index Optimizations)

OPTIMIZATIONS AVAILABLE NOW (Project 9 level):

1. SPARSE INDEX
   → Only index documents that HAVE the indexed field
   → Skip documents where field is missing (Project 8: null vs missing)
   
   db.customers.createIndex(
       { loyalty_tier: 1 },
       { sparse: true }
   )
   → Documents without loyalty_tier are NOT in the index
   → Smaller index size
   → BUT: find({ loyalty_tier: null }) won't use sparse index
   
   Redshift equivalent: NONE (every row has every column)

2. PARTIAL INDEX
   → Only index documents matching a condition
   
   db.customers.createIndex(
       { email: 1 },
       { partialFilterExpression: { is_active: true } }
   )
   → Only active customers are indexed
   → Much smaller index if most customers are inactive
   → Query MUST include the partial filter to use the index
   
   Redshift equivalent: NONE (sort key applies to all rows)

3. BACKGROUND INDEX BUILDING
   → Since MongoDB 4.2: ALL index builds are background by default
   → Does NOT block reads or writes during build
   → Redshift: changing sort key requires table recreation → downtime
   
4. TTL INDEX (revisiting from Project 5)
   → Index that auto-deletes documents after a time period
   → db.sessions.createIndex({ created_at: 1 }, { expireAfterSeconds: 3600 })
   → Documents auto-deleted 1 hour after created_at
   → Redshift: must schedule DELETE jobs manually

5. CASE-INSENSITIVE INDEX (using collation)
   → db.customers.createIndex(
         { email: 1 },
         { collation: { locale: "en", strength: 2 } }
     )
   → Queries with matching collation use this index case-insensitively
   → Redshift: ILIKE operator, no index benefit

SHOULD DO:

       → Verify COLLSCAN → IXSCAN transition
       → Check totalDocsExamined:nReturned ratio
    
    ✅ Create indexes based on ACTUAL query patterns
       → Don't guess — look at real application queries
       → MongoDB logs slow queries (>100ms) automatically
       → Atlas Performance Advisor suggests indexes automatically
    
    ✅ Use unique indexes for data integrity
       → Unlike Redshift, MongoDB ENFORCES uniqueness
       → Use it! email, username, order_id, etc.
    
    ✅ Name your indexes descriptively
       → { name: "idx_email" } not auto-generated "email_1"
       → Easier to reference in drop/hint operations
    
    ✅ Check index size relative to RAM
       → db.customers.stats().totalIndexSize
       → Indexes should fit in WiredTiger cache for best performance
       → If indexes > RAM: performance degrades significantly
    
    ✅ Start with single-field indexes
       → Compound indexes (Project 10) are more powerful but complex
       → Get single-field indexing right first

❌ WHAT SHOULD NOT DO

DO NOT:
    ❌ Create indexes on EVERY field "just in case"
       → Each index consumes RAM and slows down writes
       → Every insert/update/delete must update ALL indexes
       → 20 indexes on one collection = 20x write overhead
       → Redshift has no equivalent penalty (no indexes to maintain)
    
    ❌ Ignore COLLSCAN on large collections
       → "It works" is not "it works well"
       → 100K docs: COLLSCAN takes ~500ms (maybe acceptable)
       → 10M docs: COLLSCAN takes ~15 seconds (definitely not acceptable)
    
    ❌ Create index without checking if query actually uses it
       → ALWAYS verify with explain() after creating
       → MongoDB query planner might choose COLLSCAN anyway
         (e.g., when query returns >30% of collection)
    
    ❌ Think "sort key = index" (Redshift mental trap)
       → Sort key: physical data order, one per table
       → Index: separate B-tree, many per collection
       → Sort key helps range scans by skipping blocks
       → Index enables direct pointer-based lookups
       → Fundamentally different mechanisms
    
    ❌ Forget that indexes consume storage
       → db.customers.stats().totalIndexSize → check this
       → A collection with 1GB data might have 2GB+ of indexes
       → Redshift zone maps: negligible storage overhead
    
    ❌ Create descending index when ascending works fine
       → Single-field: direction rarely matters
       → Save the direction thinking for compound indexes (Project 10)
    
    ❌ Assume unique constraints work like Redshift
       → Redshift UNIQUE = not enforced (informational)
       → MongoDB unique = ENFORCED (inserts rejected)
       → If you create unique index on field with duplicates
         → createIndex will FAIL
         → Must clean data first

⚙️ HOW IT ONLY WORKS INTERNALLY — B-Tree vs Zone Maps

MongoDB B-Tree Index Internals

WHEN YOU CREATE: db.customers.createIndex({ email: 1 })

WHAT MONGODB BUILDS (B-tree structure):

                        ┌──────────────────────────┐
                        │     ROOT NODE              │
                        │  keys: [g..., m..., s...]  │
                        │  ptrs: [→, →, →, →]       │
                        └──────┬───┬───┬───┬────────┘
                               │   │   │   │
                    ┌──────────┘   │   │   └──────────┐
                    ▼              ▼   ▼              ▼
            ┌──────────────┐  ┌─────┐ ┌─────┐  ┌──────────────┐
            │ INTERNAL NODE │  │     │ │     │  │ INTERNAL NODE │
            │ a... - f...   │  │ ... │ │ ... │  │ s... - z...   │
            └──────┬────────┘  └─────┘ └─────┘  └──────┬────────┘
                   │                                     │
         ┌─────┬──┴──┬─────┐               ┌─────┬─────┴──┬─────┐
         ▼     ▼     ▼     ▼               ▼     ▼        ▼     ▼
      ┌──────┐ ┌──────┐ ┌──────┐        ┌──────┐ ┌──────┐ ┌──────┐
      │ LEAF │ │ LEAF │ │ LEAF │        │ LEAF │ │ LEAF │ │ LEAF │
      │a@... │ │c@... │ │e@... │        │s@... │ │u@... │ │w@... │
      │b@... │ │d@... │ │f@... │        │t@... │ │v@... │ │x@... │
      │→doc  │ │→doc  │ │→doc  │        │→doc  │ │→doc  │ │→doc  │
      │→doc  │ │→doc  │ │→doc  │        │→doc  │ │→doc  │ │→doc  │
      └──────┘ └──────┘ └──────┘        └──────┘ └──────┘ └──────┘

FINDING email = "user50000@example.com":
    Step 1: Start at ROOT
            → "user50000@..." starts with 'u'
            → Follow pointer for range [s..., z...]
    
    Step 2: INTERNAL NODE
            → Follow pointer for range [u..., v...]
    
    Step 3: LEAF NODE
            → Binary search within leaf for exact match
            → Found: "user50000@example.com" → pointer to document location
    
    Step 4: FETCH document from disk using pointer
    
    TOTAL I/O:
        → 3 index page reads (root → internal → leaf)
        → 1 document read
        → 4 reads total (regardless of collection size!)
        → For 10M documents: still just 4 reads (B-tree depth = ~4)
        → For 100M documents: maybe 5 reads (depth grows logarithmically)
    
    THIS IS O(log n) — logarithmic time
    → 10M docs: ~23 comparisons (log₂ 10M)
    → 100M docs: ~27 comparisons
    → 1B docs: ~30 comparisons
    → BARELY grows with data size

STORAGE:
    → Index stored in separate WiredTiger file: index-xxx.wt
    → Index size: ~40-60 bytes per entry (key value + document pointer)
    → 10M customers with email index: ~500MB-700MB of index data
    → Must fit in WiredTiger cache for optimal performance

    │ Contains: ~130,000 email column values            │

    └─────────────────────────────────────────────────┘
    ┌─────────────────────────────────────────────────┐
    │ Block 002 (1MB)                                  │
    │ Zone Map: min="baa@corp.com" max="bzz@web.com"   │

    └─────────────────────────────────────────────────┘
    ...
    ┌─────────────────────────────────────────────────┐
    │ Block 077 (1MB)                                  │
    │ Zone Map: min="uaa@..." max="uzz@..."            │

    │ ← "user50000@..." IS IN THIS BLOCK               │
    └─────────────────────────────────────────────────┘
    ...

FINDING email = "user50000@example.com":
    Step 1: Check zone map of Block 001
            → max "azz@..." < "user50000@..." → SKIP
    Step 2: Check zone map of Block 002
            → max "bzz@..." < "user50000@..." → SKIP
    ...
    Step 77: Check zone map of Block 077
            → min "uaa@..." ≤ "user50000@..." ≤ max "uzz@..." → READ
    Step 78: Scan entire 1MB block for exact match
            → Linear scan through ~130,000 values
    
    TOTAL I/O:
        → 77 zone map checks (fast — metadata in memory)
        → 1MB block read (minimum read unit)
        → Scan ~130,000 values within block
        → MUCH more I/O than MongoDB's 4 reads
    
    THIS IS O(n/B) — linear with block skipping
    → Effective for analytical scans (skip many blocks)
    → Wasteful for point lookups (still reads 1MB for 1 value)

│                             │ improvement          │ (recreate table)         │
└────────────────────────────┴──────────────────────┴──────────────────────────┘

THE FUNDAMENTAL INSIGHT:
    MongoDB B-tree index: POINTER-BASED → go directly to document
    Redshift zone maps:   ELIMINATION-BASED → skip irrelevant blocks, scan the rest

    MongoDB: "I know EXACTLY where document lives" (like GPS navigation)
    Redshift: "I know where document DOESN'T live" (like eliminating wrong neighborhoods)

    For finding 1 customer among 10 million:
        GPS (MongoDB):          Turn left, turn right, arrive → 1ms
        Elimination (Redshift): Not here, not here, not here... maybe here,
                                scan this whole block → 10-50ms

    For aggregating all 10 million customers:
        MongoDB:  Must visit every document (no shortcut) → 30-60s
        Redshift: Columnar scan + compression + MPP → 2-5s

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 9 SPECIFIC:

PITFALL 1: "I'll design my indexes before loading data"
    ❌ Planning all indexes at collection creation time
       (like choosing sort key at CREATE TABLE time)
    ✅ Load data first → run queries → see what's slow → add indexes
    WHY: Redshift trained you to decide sort key upfront
         MongoDB lets you observe actual query patterns first
         Real-world queries often differ from initial assumptions
    EXCEPTION: _id index (auto) and known unique constraints (email)

PITFALL 2: "One index should be enough" (sort key thinking)
    ❌ Creating one index and expecting all queries to benefit
    ✅ Create dedicated index per query pattern
    WHY: Redshift has one sort key → you accept the trade-off
         MongoDB has unlimited indexes → each query gets optimal performance
         But each index costs RAM and write performance

PITFALL 3: "Indexes are free" (opposite extreme)
    ❌ Creating 30 indexes on a collection "to cover everything"
    ✅ Monitor index usage → drop unused indexes
    WHY: Redshift has no index maintenance overhead
         MongoDB must update EVERY index on EVERY write
         10 indexes = 10x the write work
         db.customers.aggregate([{$indexStats: {}}]) → find unused indexes

PITFALL 4: "I'll just use COLLSCAN — the data isn't that big"
    ❌ Accepting COLLSCAN because "it only takes 500ms"
    ✅ Add index now — data will grow, 500ms becomes 15 seconds
    WHY: Redshift scales scans via MPP (add nodes → faster scans)
         MongoDB COLLSCAN on single node doesn't scale
         Collection grows 10x → COLLSCAN takes 10x longer
         Index stays O(log n) → barely grows with data size

PITFALL 5: "explain() is like EXPLAIN — I'll check it once"
    ❌ Running explain() once and never again
    ✅ Check explain() after data distribution changes
    WHY: MongoDB query planner can CHANGE its plan
         With more data, it might stop using an index
         (when query returns >30% of collection, COLLSCAN can be faster)
         Regular explain() checks catch regressions

PITFALL 6: "Unique index = Redshift UNIQUE constraint"
    ❌ Expecting unique index to be informational only
    ✅ Understanding unique index REJECTS duplicate inserts
    WHY: Redshift UNIQUE is a hint → duplicates silently accepted
         MongoDB unique is enforced → DuplicateKeyError on violation
         Your application code must handle this error
         This is actually BETTER — real data integrity

PITFALL 7: "I need to VACUUM after bulk inserts"
    ❌ Looking for MongoDB's VACUUM equivalent
    ✅ MongoDB B-tree indexes are self-maintaining
    WHY: Redshift appends unsorted data → needs VACUUM to re-sort
         MongoDB inserts directly into B-tree → always sorted
         No VACUUM needed (compact command exists but rarely needed)
         This is a Redshift habit you can unlearn with relief

📈 SCALABILITY CONSIDERATIONS

INDEX SCALABILITY — What Changes at Scale:

SCALE: 100K documents (current test)
    → Index size: ~5MB per single-field index
    → All indexes fit easily in RAM
    → Index builds: <1 second
    → COLLSCAN: ~500ms (annoying but survivable)

SCALE: 10M documents (real production)
    → Index size: ~500MB per single-field index
    → 5 indexes = ~2.5GB of index data
    → Need sufficient RAM (WiredTiger cache)
    → Index builds: 30-120 seconds (background, non-blocking)
    → COLLSCAN: 8-15 seconds (unacceptable for user-facing)

SCALE: 100M documents (large production)
    → Index size: ~5GB per single-field index
    → 5 indexes = ~25GB of index data
    → MUST have enough RAM for working set of indexes
    → If indexes don't fit in RAM → disk reads → 10-100x slower
    → Consider sharding (Project 21) at this scale

SCALE: 1B+ documents (massive scale)
    → Single server cannot hold all indexes in RAM
    → MUST shard (split data across multiple servers)
    → Each shard holds subset of data + subset of indexes
    → Shard key selection becomes critical (Project 22)
    → MongoDB sharding = Redshift adding compute nodes

COMPARISON TO REDSHIFT SCALING:
    Redshift at 100M rows:
        → Zone maps: negligible overhead (~KB)
        → Add more nodes → parallel scan → stays fast
        → No RAM pressure from indexes (no indexes!)
        → Sort key benefit maintained automatically across nodes
    
    MongoDB at 100M documents:
        → Index overhead: significant (~5GB per index)
        → RAM pressure: indexes MUST fit in cache
        → Must shard to distribute index load
        → Each shard handles subset → RAM per shard manageable

KEY INSIGHT:
    Redshift scales ANALYTICS by adding compute nodes (MPP)
    MongoDB scales OPERATIONS by sharding (distribute data + indexes)
    Both scale horizontally but for DIFFERENT workload types

📐 DATA MODELING COMPARISON — Index Impact on Design

REDSHIFT DESIGN THINKING:
    → Choose sort key based on most common filter
    → Design star schema with fact + dimension tables
    → JOINs are expected and optimized by MPP
    → One table can serve many different queries
      (columnar storage reads only needed columns)
    → Sort key helps ONE query pattern most

MONGODB DESIGN THINKING:
    → Design documents around ACCESS PATTERNS (Project 1 lesson)
    → Create indexes to match those access patterns
    → Each index is a commitment: RAM + write overhead
    → Too many indexes → slow writes
    → Too few indexes → slow reads
    → The balance is the art of MongoDB design

DESIGN DECISIONS INFLUENCED BY INDEXES:

    DECISION: "Should I embed addresses in customer document?"
    
    Redshift thinking:
        → Separate addresses table
        → JOIN when needed

    MongoDB thinking:
        → If you index customer.addresses.city for geo queries:
          → Multikey index on array of embedded addresses
          → ONE query returns customer + matching address
          → BUT: array growth = larger documents = index maintenance
        → If addresses are a separate collection:
          → Simple index on address.city
          → Requires $lookup (JOIN) to get customer info
          → More indexes but simpler documents
    
    THE ANSWER DEPENDS ON QUERY PATTERNS — which we'll explore in Project 14

def analyze_query(collection, filter_doc=None, sort_doc=None,
                  projection=None, limit_val=None, label="Query"):
    """
    Analyze a query's execution plan and performance.
    Equivalent to running EXPLAIN VERBOSE in Redshift.
    """
    cursor = collection.find(filter_doc or {}, projection or {})
    if sort_doc:
        cursor = cursor.sort(list(sort_doc.items()))
    if limit_val:
        cursor = cursor.limit(limit_val)

    explain = cursor.explain()

    # Extract plan info
    plan = explain["queryPlanner"]["winningPlan"]
    stats = explain["executionStats"]

    # Walk the plan tree to find scan type
    def walk_plan(node, depth=0):
        """Recursively walk the plan tree"""
        stages = []
        stages.append({
            "depth": depth,
            "stage": node.get("stage", "unknown"),
            "index": node.get("indexName"),
            "key_pattern": node.get("keyPattern"),
            "direction": node.get("direction"),
            "is_multi_key": node.get("isMultiKey"),
            "index_bounds": node.get("indexBounds"),
            "filter": node.get("filter"),
        })
        if "inputStage" in node:
            stages.extend(walk_plan(node["inputStage"], depth + 1))
        if "inputStages" in node:
            for sub in node["inputStages"]:
                stages.extend(walk_plan(sub, depth + 1))
        return stages

    stages = walk_plan(plan)

    # Find the scan stage (deepest level)
    scan_stage = stages[-1]

    # Performance metrics
    docs_examined = stats["totalDocsExamined"]
    keys_examined = stats["totalKeysExamined"]
    docs_returned = stats["nReturned"]
    exec_time = stats["executionTimeMillis"]

    # Efficiency calculation
    if docs_returned > 0:
        exam_ratio = docs_examined / docs_returned
    else:
        exam_ratio = docs_examined if docs_examined > 0 else 0

    # Rating
    if scan_stage["stage"] == "COLLSCAN":
        rating = "🔴 COLLSCAN — needs index"
    elif exam_ratio <= 1.5:
        rating = "🟢 EXCELLENT — near perfect efficiency"
    elif exam_ratio <= 10:
        rating = "🟡 GOOD — acceptable but could improve"
    else:
        rating = "🟠 POOR — index not selective enough"

    # Print report
    print(f"\n{'='*65}")
    print(f"  QUERY ANALYSIS: {label}")
    print(f"{'='*65}")
    print(f"  Filter:          {filter_doc}")
    print(f"  Sort:            {sort_doc}")
    print(f"  Limit:           {limit_val}")
    print(f"{'─'*65}")
    print(f"  Plan stages:")
    for s in stages:
        indent = "    " + "  " * s["depth"]
        stage_info = s["stage"]
        if s["index"]:
            stage_info += f" (index: {s['index']})"
        if s["direction"]:
            stage_info += f" [{s['direction']}]"
        print(f"{indent}→ {stage_info}")
    print(f"{'─'*65}")
    print(f"  Keys examined:   {keys_examined:,}")
    print(f"  Docs examined:   {docs_examined:,}")
    print(f"  Docs returned:   {docs_returned:,}")
    print(f"  Exam/Return:     {exam_ratio:.1f}:1")
    print(f"  Execution time:  {exec_time}ms")
    print(f"  Rating:          {rating}")
    print(f"{'='*65}")

    return {
        "scan_type": scan_stage["stage"],
        "index_used": scan_stage.get("index"),
        "keys_examined": keys_examined,
        "docs_examined": docs_examined,
        "docs_returned": docs_returned,
        "exec_time_ms": exec_time,
        "efficiency_ratio": exam_ratio,
        "rating": rating,
    }

def index_health_check(collection):
    """
    Analyze all indexes on a collection.
    Shows size, usage stats, and recommendations.
    """
    coll_name = collection.name
    stats = collection.index_information()
    coll_stats = db.command("collstats", coll_name)

    total_index_size = coll_stats.get("totalIndexSize", 0)
    data_size = coll_stats.get("size", 0)
    doc_count = coll_stats.get("count", 0)

    print(f"\n{'='*65}")
    print(f"  INDEX HEALTH CHECK: {collection.database.name}.{coll_name}")
    print(f"{'='*65}")
    print(f"  Documents:        {doc_count:,}")
    print(f"  Data size:        {data_size / 1024 / 1024:.1f} MB")
    print(f"  Total index size: {total_index_size / 1024 / 1024:.1f} MB")
    print(f"  Index/Data ratio: {total_index_size / max(data_size, 1) * 100:.0f}%")
    print(f"{'─'*65}")

    # Index usage stats
    try:
        usage_stats = list(collection.aggregate([{"$indexStats": {}}]))
        usage_map = {s["name"]: s["accesses"]["ops"] for s in usage_stats}
    except Exception:
        usage_map = {}

    print(f"  {'INDEX NAME':<25} {'KEYS':<25} {'UNIQUE':<8} {'USAGE':<10}")
    print(f"  {'─'*25} {'─'*25} {'─'*8} {'─'*10}")

    for name, info in stats.items():
        keys = str(dict(info["key"]))
        unique = "✓" if info.get("unique") else ""
        usage = usage_map.get(name, "N/A")
        print(f"  {name:<25} {keys:<25} {unique:<8} {usage}")

    # Recommendations
    print(f"\n  Recommendations:")
    if total_index_size > data_size * 2:
        print(f"  ⚠️  Index size > 2x data size — consider dropping unused indexes")
    if len(stats) > 10:
        print(f"  ⚠️  {len(stats)} indexes — write performance may suffer")
    for name, ops in usage_map.items():
        if ops == 0 and name != "_id_":
            print(f"  ⚠️  Index '{name}' has ZERO usage — candidate for dropping")
    print(f"{'='*65}")

def compare_with_without_index(collection, field, filter_doc, label="Test"):
    """
    Create index, compare performance before and after.
    Drops the test index at the end.
    """
    index_name = f"test_idx_{field}"

    # Check if index already exists
    existing = collection.index_information()
    had_index = index_name in existing

    # Phase 1: Without index (drop if exists)
    if had_index:
        collection.drop_index(index_name)

    print(f"\n{'='*65}")
    print(f"  BEFORE vs AFTER INDEX: {label}")
    print(f"  Field: {field}")
    print(f"  Filter: {filter_doc}")
    print(f"{'='*65}")

    # Measure without index
    start = time.time()
    result_before = list(collection.find(filter_doc))
    time_before = (time.time() - start) * 1000

    explain_before = collection.find(filter_doc).explain()
    stats_before = explain_before["executionStats"]

    print(f"\n  WITHOUT INDEX:")
    print(f"    Time: {time_before:.1f}ms")
    print(f"    Docs examined: {stats_before['totalDocsExamined']:,}")
    print(f"    Docs returned: {stats_before['nReturned']:,}")

    # Phase 2: Create index and measure
    collection.create_index([(field, ASCENDING)], name=index_name)

    start = time.time()
    result_after = list(collection.find(filter_doc))
    time_after = (time.time() - start) * 1000

    explain_after = collection.find(filter_doc).explain()
    stats_after = explain_after["executionStats"]

    print(f"\n  WITH INDEX:")
    print(f"    Time: {time_after:.1f}ms")
    print(f"    Docs examined: {stats_after['totalDocsExamined']:,}")
    print(f"    Docs returned: {stats_after['nReturned']:,}")

    # Improvement
    if time_before > 0:
        speedup = time_before / max(time_after, 0.001)
        print(f"\n  IMPROVEMENT: {speedup:.0f}x faster")
        print(f"  Docs examined reduced: "
              f"{stats_before['totalDocsExamined']:,} → "
              f"{stats_after['totalDocsExamined']:,}")

    # Cleanup (drop test index unless it existed before)
    if not had_index:
        collection.drop_index(index_name)
        print(f"  (test index dropped)")

    print(f"{'='*65}")
