PROJECT 5: Deleting Documents vs DELETE Statements

Problem: Remove Inactive Users and Purge Old Temporary Cart Data

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform has been running for months
    Several cleanup tasks are needed:
    
    1. Remove a single customer who requested account deletion (GDPR)
    2. Remove all customers who haven't logged in since 2022
    3. Purge temporary cart data older than 30 days
    4. Delete a customer AND return their data (for audit log)
    5. Auto-expire session tokens after 24 hours (without manual cleanup)
    6. Drop an entire collection of test data
    7. Remove all documents but keep the collection (TRUNCATE equivalent)
    
    In Redshift: DELETE FROM table WHERE condition, DROP TABLE, TRUNCATE
    In MongoDB: deleteOne, deleteMany, findOneAndDelete, TTL indexes, drop()
    
    PLUS: MongoDB has something Redshift CANNOT do —
    automatic time-based expiry (TTL indexes)
    Documents DELETE THEMSELVES after a specified duration

🔴 REDSHIFT MINDSET — How You Would Solve This

All 7 Deletion Scenarios in Redshift SQL

-- ─── SCENARIO 1: Delete single customer (GDPR request) ──
DELETE FROM store.customers
WHERE email = 'alice@example.com';
-- Also need to delete from ALL related tables:
DELETE FROM store.orders WHERE customer_id = 1234;
DELETE FROM store.customer_preferences WHERE customer_id = 1234;
-- Must know ALL tables that reference this customer
-- No CASCADE DELETE in Redshift (unlike PostgreSQL)

-- ─── SCENARIO 2: Delete inactive customers ──────────────
DELETE FROM store.customers
WHERE last_login < '2022-01-01';
-- Redshift: this creates ghost rows (soft-deleted internally)
-- Must VACUUM to reclaim disk space

-- ─── SCENARIO 3: Purge old cart data ────────────────────
DELETE FROM store.temp_carts
WHERE created_at < DATEADD(day, -30, GETDATE());
-- Must schedule this as a cron job / Step Function
-- Runs periodically (daily/hourly)
-- Manual process — Redshift has NO auto-expiry

-- ─── SCENARIO 4: Delete and return data ─────────────────
-- Redshift has NO "DELETE ... RETURNING" clause
-- Must SELECT first, then DELETE (two operations, not atomic)
SELECT * FROM store.customers WHERE email = 'alice@example.com';
-- Save result in application
DELETE FROM store.customers WHERE email = 'alice@example.com';
-- Race condition: data could change between SELECT and DELETE

-- ─── SCENARIO 5: Auto-expire session tokens ─────────────
-- IMPOSSIBLE in Redshift natively
-- Must create a scheduled job:
--   1. CloudWatch Event (cron schedule)
--   2. Lambda function
--   3. Connects to Redshift
--   4. Runs: DELETE FROM sessions WHERE created_at < NOW() - INTERVAL '24 hours'
-- Heavy infrastructure for a simple auto-expiry

-- ─── SCENARIO 6: Drop entire collection ─────────────────
DROP TABLE store.test_data;
-- Table definition AND data gone permanently
-- Cannot be undone (unless snapshot exists)

-- ─── SCENARIO 7: Remove all rows but keep table ─────────
TRUNCATE store.temp_carts;
-- Faster than DELETE (no row-by-row processing)
-- Resets table to empty state
-- Cannot be rolled back (auto-commits)
-- Keeps table definition, indexes, permissions

What Redshift Does Behind the Scenes for DELETE

REDSHIFT DELETE INTERNALS:

    DELETE FROM customers WHERE last_login < '2022-01-01'
    
    1. Leader node creates execution plan
    2. Each compute slice scans its data:
       → Uses sort key / zone maps to skip blocks if possible
       → Identifies rows matching the WHERE condition
    3. Matching rows are MARKED AS DELETED (ghost rows)
       → Rows are NOT physically removed from disk
       → A delete marker is written
       → The space is NOT reclaimed
    4. Subsequent queries must skip ghost rows
       → Queries get SLOWER as ghost rows accumulate
       → This is invisible performance degradation
    5. Must run VACUUM DELETE to reclaim space:
       → VACUUM DELETE store.customers;
       → Physically removes ghost rows
       → Rewrites affected blocks
       → Can take HOURS for large tables
    6. Must run ANALYZE to update statistics:
       → ANALYZE store.customers;
       → Query planner uses stale stats otherwise

    COST OF DELETE IN REDSHIFT:
    → DELETE itself: relatively fast (just marks rows)
    → But VACUUM afterward: can be very slow
    → Ghost rows degrade query performance until VACUUM
    → This is why frequent deletes in Redshift are discouraged
    
    TRUNCATE is different:
    → Drops all data blocks at once (not row-by-row)
    → Instant regardless of table size
    → No VACUUM needed
    → But removes ALL rows — no WHERE clause

🟢 MONGODB MINDSET — How This Same Problem Is Approached

The Fundamental Difference

REDSHIFT DELETE MODEL:
    → DELETE marks rows as ghosts → VACUUM reclaims space later
    → TRUNCATE drops all blocks → instant but removes everything
    → DROP TABLE removes table definition + data
    → No auto-expiry capability
    → Cascade deletes must be manual (delete from all related tables)
    → Cannot return deleted data atomically

MONGODB DELETE MODEL:
    → deleteOne/deleteMany removes documents immediately
    → WiredTiger reclaims space automatically (no VACUUM needed)
    → drop() removes collection + data + indexes
    → TTL indexes: documents auto-expire after a time period
    → Embedded data deleted WITH the parent document (no cascade needed)
    → findOneAndDelete: atomic delete + return deleted document

KEY DIFFERENCES:
    1. No ghost rows — MongoDB truly removes documents
    2. No VACUUM — WiredTiger handles space reclamation
    3. TTL indexes — automatic time-based expiry (unique to MongoDB)
    4. Embedded data — deleting a document deletes ALL its nested data
    5. findOneAndDelete — atomic delete + return

│                       │ (two ops, race condition)    │ (one atomic operation)            │

├──────────────────────┼──────────────────────────────┼──────────────────────────────────┤
│ Space reclamation     │ VACUUM DELETE (manual)        │ Automatic (WiredTiger)            │
│ after delete          │ Can take hours               │ No action needed                  │
├──────────────────────┼──────────────────────────────┼──────────────────────────────────┤
│ Ghost rows / bloat    │ YES — deleted rows remain    │ NO — documents truly removed      │
│                       │ until VACUUM                 │ Space reused automatically        │
├──────────────────────┼──────────────────────────────┼──────────────────────────────────┤
│ Statistics after      │ ANALYZE needed               │ Automatic (no action needed)      │
│ delete                │                              │                                   │
├──────────────────────┼──────────────────────────────┼──────────────────────────────────┤
│ Transaction support   │ DELETE can be in transaction │ deleteOne/Many can be in session  │
│                       │ ROLLBACK supported           │ Transaction ROLLBACK supported    │
│                       │ TRUNCATE cannot rollback     │ drop() cannot rollback            │
└──────────────────────┴──────────────────────────────┴──────────────────────────────────┘

🧩 HOW TO APPROACH IT IN MONGODB

DELETE DECISION TREE:

    "I need to remove data from MongoDB"
            │
            ├── Remove ONE specific document?
            │       → deleteOne({ filter })
            │       → Need the deleted data? → findOneAndDelete({ filter })
            │
            ├── Remove MANY matching documents?
            │       → deleteMany({ filter })
            │       → How many will match? → countDocuments(filter) first
            │
            ├── Remove ALL documents (keep collection)?
            │       → deleteMany({})
            │       → Keeps indexes, keeps collection, removes all docs
            │       → Like TRUNCATE but slower (document-by-document)
            │
            ├── Remove entire collection (definition + data)?
            │       → db.collection.drop()
            │       → Instant regardless of size
            │       → Like DROP TABLE in Redshift
            │       → Indexes also removed
            │
            ├── Data should expire automatically over time?
            │       → TTL index on a date field
            │       → Documents auto-delete after specified seconds
            │       → NO Redshift equivalent
            │
            └── Remove database entirely?
                    → db.dropDatabase()
                    → All collections in database removed
                    → Like DROP DATABASE in Redshift

REDSHIFT DEVELOPER MENTAL SHIFT:
    Redshift: "DELETE creates ghost rows → VACUUM later"
    MongoDB:  "deleteOne/Many truly removes → no cleanup needed"
    
    Redshift: "TRUNCATE for fast removal of all rows"
    MongoDB:  "drop() for fast removal, deleteMany({}) for keeping collection"
    
    Redshift: "Schedule jobs to purge old data"
    MongoDB:  "TTL index handles purging automatically"

)

// ┌──────────────────────────────────────────────────────────────┐
// │ TTL INDEX BEHAVIOR:                                          │
// │                                                               │
// │ MongoDB runs a BACKGROUND THREAD every 60 seconds that:      │
// │   1. Checks the TTL index                                    │
// │   2. Finds documents where:                                  │
// │      created_at + expireAfterSeconds < current_time           │
// │   3. DELETES those documents automatically                   │
// │                                                               │
// │ sess_001: created 40 days ago → 40 > 30 → WILL BE DELETED   │
// │ sess_002: created 15 days ago → 15 < 30 → KEPT (for now)    │
// │ sess_003: created just now    → 0 < 30  → KEPT              │
// │                                                               │
// │ TIMING: The background thread runs every ~60 seconds          │
// │ → Deletion is NOT instant — there's up to 60s delay           │
// │ → On busy systems, TTL deletion may lag further               │
// │                                                               │
// │ REDSHIFT EQUIVALENT: NONE                                    │
// │ → Must create: CloudWatch Event → Lambda → DELETE SQL         │
// │ → Infrastructure overhead for something MongoDB does natively│

// │ → No other process can modify it between find and delete     │
// │ → If document doesn't exist → returns null, deletes nothing  │
// │ → Atomic at the storage engine level                        │
// │                                                               │
// │ USE CASES:                                                   │
// │ → GDPR deletion with audit trail                             │
// │ → Queue processing (pop item from queue)                     │
// │ → Claim-and-delete patterns (take ownership + remove)        │
// │                                                               │
// │ RELATED METHODS:                                             │
// │ → findOneAndDelete: delete + return deleted doc               │
// │ → findOneAndUpdate: update + return (Project 4)               │

// │ → All three are ATOMIC find-and-modify operations            │

// → Two workers calling simultaneously get DIFFERENT tasks
// → No duplicate processing — each task dequeued exactly once
// → Redshift: NOT designed for this pattern (OLAP, not OLTP)

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Delete Path — What Happens During deleteOne()

WHEN YOU RUN: db.customers.deleteOne({ email: "alice@example.com" })

STEP 1: FIND THE DOCUMENT
    → Same as query execution (Project 3 internals)
    → Uses index if available on email field → IXSCAN
    → Without index → COLLSCAN (scans all documents)

STEP 2: ACQUIRE DOCUMENT-LEVEL LOCK
    → WiredTiger locks ONLY this document (same as updates — Project 4)
    → Other documents can be read/written concurrently

STEP 3: REMOVE FROM WIREDTIGER B-TREE
    → Document removed from the B-tree leaf page
    → Page marked as having free space
    → Space available for reuse by future inserts
    
    CRITICAL DIFFERENCE FROM REDSHIFT:
    ┌──────────────────────────────────────────────────────────────┐
    │ REDSHIFT DELETE:                                              │
    │   → Row marked with delete flag (ghost row)                  │
    │   → Physical space NOT freed                                 │
    │   → Ghost row still scanned during queries (skipped)         │
    │   → VACUUM needed to reclaim space                           │
    │   → Table can become "bloated" with ghost rows               │
    │                                                               │
    │ MONGODB DELETE:                                               │
    │   → Document removed from B-tree page                        │
    │   → Space marked as reusable on that page                    │
    │   → Future inserts can use this space                        │
    │   → No ghost rows, no bloat                                  │
    │   → WiredTiger compaction happens automatically              │
    │   → No VACUUM ever needed                                    │
    │                                                               │
    │ WHY THE DIFFERENCE:                                          │
    │   Redshift: columnar storage → each column block affected    │
    │             → complex to physically remove one row            │
    │   MongoDB: document storage → one B-tree entry removed       │
    │             → simple to physically remove one document        │
    └──────────────────────────────────────────────────────────────┘

STEP 4: UPDATE INDEXES
    → Remove document's entry from ALL indexes
    → _id index: remove the ObjectId entry
    → Any other indexes: remove entries for this document's field values
    → Index B-trees rebalance as needed

STEP 5: JOURNAL ENTRY
    → Delete operation written to journal (WAL)
    → Journal ensures crash recovery
    → If server crashes after journal write but before B-tree update
      → Recovery replays journal → delete is applied

STEP 6: OPLOG ENTRY
    → Delete recorded in oplog for replication
    → Secondary nodes read oplog and apply same delete
    → Change streams (Project 25) can detect this delete event
    
    OPLOG ENTRY:
    {
        "op": "d",                          // d = delete
        "ns": "ecommerce.customers",
        "o": { "_id": ObjectId("...") }    // identifies deleted document
    }

TTL Index Internals — How Auto-Expiry Works

TTL BACKGROUND THREAD:

    MongoDB runs a dedicated background thread called TTLMonitor
    
    EVERY 60 SECONDS:
    ┌────────────────────────────────────────────────────────────────┐
    │ 1. TTLMonitor wakes up                                        │
    │ 2. Scans all collections with TTL indexes                     │
    │ 3. For each TTL index:                                        │
    │    a. Query: find documents where                              │
    │       date_field + expireAfterSeconds < current_time           │
    │    b. Delete matching documents (using internal deleteMany)    │
    │ 4. TTLMonitor goes back to sleep for ~60 seconds              │
    └────────────────────────────────────────────────────────────────┘
    
    TIMING IMPLICATIONS:
    → Document expired at 10:00:00
    → TTLMonitor last ran at 9:59:45
    → Next run at 10:00:45
    → Document actually deleted at 10:00:45 (45 seconds late)
    → Maximum delay: ~60 seconds under normal load
    → Under heavy load: TTL deletion can lag further
    
    PERFORMANCE IMPACT:
    → TTLMonitor uses the TTL index for efficient scanning
    → Only expired documents are found (no full collection scan)
    → Deletes happen in background (don't block application queries)
    → Heavy TTL deletion can cause brief I/O spikes
    
    REDSHIFT COMPARISON:
    → No equivalent internal mechanism
    → Must build: CloudWatch Event → Lambda → SQL DELETE
    → Minimum scheduling: 1 minute (CloudWatch)
    → Additional latency: Lambda cold start + Redshift query time
    → Infrastructure to maintain and monitor
    → MongoDB: zero infrastructure, built into the database engine

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 5:

PITFALL 1: "I need to VACUUM after deletes"
    REDSHIFT TRAINED YOU: DELETE creates ghost rows → VACUUM required
    MONGODB REALITY: WiredTiger reclaims space automatically
    FIX: Just delete — no post-delete maintenance needed
    SEVERITY: LOW (you just waste time looking for a VACUUM command)
    
    NOTE: MongoDB DOES have a compact command:
    db.runCommand({ compact: "customers" })
    → Defragments storage and reclaims disk space
    → Rarely needed (WiredTiger handles this)
    → Blocks reads/writes during compaction
    → Only use if disk utilization is critically high after massive deletes

PITFALL 2: "deleteOne deletes ALL matching documents"
    REDSHIFT TRAINED YOU: DELETE FROM ... WHERE ... deletes ALL matches
    MONGODB REALITY: deleteOne deletes EXACTLY ONE (first match)
    
    # You expect this to delete all FL customers:
    db.customers.deleteOne({ "address.state": "FL" })
    # → Only ONE Florida customer deleted!
    
    # What you actually want:
    db.customers.deleteMany({ "address.state": "FL" })
    
    FIX: Use deleteMany when you intend to delete all matches
    SEVERITY: HIGH — silently deletes fewer documents than expected

PITFALL 3: "TTL index on a string date field"
    # WRONG — stored date as string (Project 2 CSV trap)
    { "created_at": "2024-01-15T00:00:00Z" }    # STRING, not Date
    db.collection.createIndex(
        { "created_at": 1 },
        { expireAfterSeconds: 86400 }
    )
    # → TTL thread ignores non-Date values → NOTHING expires
    # → No error, no warning — completely silent failure
    
    # RIGHT — must be a BSON Date type
    { "created_at": ISODate("2024-01-15T00:00:00Z") }    # Date type
    
    FIX: Verify field type before creating TTL index:
    db.collection.findOne()  // inspect created_at type
    SEVERITY: CRITICAL — data never expires, storage grows forever

PITFALL 4: "TTL index on compound fields"
    # WRONG — TTL index must be on a SINGLE field
    db.collection.createIndex(
        { "created_at": 1, "status": 1 },    # compound index
        { expireAfterSeconds: 86400 }          # TTL on compound → IGNORED
    )
    # → MongoDB creates the index but TTL behavior is IGNORED
    
    # RIGHT — single field only
    db.collection.createIndex(
        { "created_at": 1 },
        { expireAfterSeconds: 86400 }
    )
    
    FIX: TTL only works on single-field indexes with Date values
    SEVERITY: HIGH — silent failure, data never expires

PITFALL 5: "drop() when you meant deleteMany({})"
    # drop() removes the COLLECTION ITSELF (definition + indexes)
    db.customers.drop()
    # → Collection gone, indexes gone, must recreate everything
    
    # deleteMany({}) removes all DOCUMENTS but keeps collection + indexes
    db.customers.deleteMany({})
    # → Collection still exists, indexes preserved, ready for new data
    
    REDSHIFT EQUIVALENT:
    → DROP TABLE = db.collection.drop()
    → TRUNCATE = db.collection.deleteMany({}) (roughly)
    
    FIX: Double-check which operation you need
    SEVERITY: CRITICAL — losing indexes requires recreation
              (especially expensive on large collections)

PITFALL 6: "Deleting referenced data without cleaning up references"
    # Customer document deleted:
    db.customers.deleteOne({ "_id": customer_id })
    
    # But orders still reference this customer:
    # { "customer_id": ObjectId("..."), "amount": 99.99 }
    # → Orphaned reference — order points to non-existent customer
    
    # MongoDB has NO foreign key constraints
    # → No error when deleting referenced document
    # → Must clean up manually
    
    FIX: Either:
    a) Embed orders IN customer document (delete together)
    b) Delete from both collections in application code
    c) Use a transaction (Project 28) for multi-collection atomic delete
    d) Use a change stream (Project 25) to trigger cascading deletes
    
    SEVERITY: MEDIUM — orphaned references cause data inconsistency

PITFALL 7: "Expecting deleteMany to be as fast as TRUNCATE"
    # Large collection with 10 million documents:
    db.large_collection.deleteMany({})
    # → Processes document-by-document → can take MINUTES
    # → Each document: find → lock → delete → update indexes → journal
    
    # TRUNCATE equivalent (instant):
    db.large_collection.drop()
    # → Drops files at filesystem level → instant regardless of size
    
    # If you need to keep the collection structure:
    # → drop() then recreate indexes
    db.large_collection.drop()
    db.large_collection.createIndex({ email: 1 }, { unique: true })
    db.large_collection.createIndex({ "address.state": 1 })
    # → Still much faster than deleteMany({}) for millions of documents
    
    SEVERITY: MEDIUM — performance trap for large collections

📐 DATA MODELING COMPARISON

HOW DOCUMENT DESIGN AFFECTS DELETIONS:

SCENARIO: Customer deletes their account (GDPR)
Must remove: customer profile + orders + preferences + reviews + sessions

REDSHIFT NORMALIZED MODEL (5 tables):
    DELETE FROM store.sessions WHERE customer_id = 1234;
    DELETE FROM store.reviews WHERE customer_id = 1234;
    DELETE FROM store.order_items WHERE order_id IN 
        (SELECT order_id FROM store.orders WHERE customer_id = 1234);
    DELETE FROM store.orders WHERE customer_id = 1234;
    DELETE FROM store.customer_preferences WHERE customer_id = 1234;
    DELETE FROM store.customers WHERE customer_id = 1234;
    
    → 6 DELETE statements across 6 tables
    → Must know ALL related tables
    → Order matters (foreign key dependencies)
    → Should be in a transaction for atomicity
    → VACUUM needed on all 6 tables afterward
    → Easy to miss a table → orphaned data → compliance violation

MONGODB EMBEDDED MODEL (1 collection):
    // If orders, preferences, reviews are EMBEDDED in customer document:
    {
        _id: 1234,
        name: "Alice",
        email: "alice@ex.com",
        address: { ... },
        preferences: { ... },
        recent_orders: [ ... ],
        recent_reviews: [ ... ]
    }
    
    db.customers.deleteOne({ _id: 1234 })
    
    → ONE delete removes EVERYTHING
    → No tables to miss
    → No foreign key ordering
    → No transaction needed (single-document atomic)
    → No VACUUM needed
    → GDPR compliance is simpler

MONGODB MIXED MODEL (embedded + referenced):
    // Customer has embedded preferences but referenced orders:
    // customers collection: { _id: 1234, preferences: {...} }
    // orders collection: { customer_id: 1234, ... }
    // sessions collection: { user_id: 1234, ... } (TTL handles this)
    
    db.customers.deleteOne({ _id: 1234 })           // customer + preferences
    db.orders.deleteMany({ customer_id: 1234 })      // orders
    // sessions auto-expire via TTL — no action needed
    
    → 2 operations (vs 6 in Redshift)
    → Sessions handled automatically by TTL
    → Still simpler than fully normalized model

LESSON FOR DOCUMENT DESIGN:
    ┌──────────────────────────────────────────────────────────────┐
    │ When you design documents, think about the DELETION story:   │
    │                                                               │
    │ → Data that should be deleted TOGETHER → EMBED it            │
    │ → Data that has independent lifecycle → REFERENCE it         │
    │ → Data that expires on a timer → TTL index it                │
    │                                                               │
    │ The easier your deletion story, the better your design       │
    │ This is a MongoDB-specific design consideration              │
    │ Redshift: deletion story is always "find all related tables" │
    └──────────────────────────────────────────────────────────────┘
