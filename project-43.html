PROJECT 43: Event Sourcing with Change Streams

Problem: Build Complete Audit Trail + Event Replay Capability for Financial System

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your global fintech platform (Projects 41-42) faces regulatory audit
    
    REGULATORY REQUIREMENTS:
        → Complete audit trail: every state change to every account EVER
        → Immutable history: no record can be modified or deleted
        → Point-in-time reconstruction: "What was this account's state
          at 2024-03-15 14:30:00 UTC?"
        → Event replay: ability to rebuild current state from event history
        → Compliance proof: demonstrate that current balances match
          the sum of all historical events
        → Debugging: trace exactly how an account reached its current state
        → Cross-system sync: events must flow to downstream systems
          (Redshift, compliance engine, ML pipeline)
    
    CURRENT PROBLEM:
        → accounts collection stores CURRENT state only
          { account_id: "ACC-001", balance: 8920.30, status: "active" }
        → When balance changes: UPDATE replaces old value
        → Previous balance LOST forever
        → "What was the balance yesterday?" → CANNOT ANSWER
        → "Show me every change to this account" → CANNOT ANSWER
        → "Prove the balance is correct" → CANNOT PROVE
    
    THE SOLUTION: EVENT SOURCING
        → Don't store current state — store EVENTS that caused state changes
        → Current state = replay all events from beginning
        → Every event is immutable — append-only
        → Full audit trail built into the architecture
        → Point-in-time reconstruction: replay events up to timestamp T
        → Debugging: examine the exact sequence of events
    
    IN REDSHIFT:
        → You'd create a fact table of events (append-only INSERT)
        → Current state: SELECT ... WHERE timestamp = MAX(timestamp)
        → Historical queries: WHERE timestamp <= '2024-03-15 14:30:00'
        → Redshift is GOOD at scanning event history (columnar + sort key)
        → Redshift is BAD at real-time event capture (batch ETL lag)
        → Redshift CANNOT push events to downstream systems (no change streams)
    
    IN MONGODB:
        → Events collection: append-only, immutable documents
        → Change streams: push events to downstream systems in real-time
        → Snapshots: periodic materialized state for fast reconstruction
        → CQRS: separate read model (optimized) from write model (events)
        → MongoDB handles BOTH event capture AND event distribution

CREATE TABLE account_events (

    event_type      VARCHAR(50) NOT NULL,
    -- e.g.: 'account_created', 'deposit', 'withdrawal',
    --       'transfer_sent', 'transfer_received',

    user_agent       VARCHAR(255)        -- who triggered it
)
DISTSTYLE KEY
DISTKEY(account_id)                     -- co-locate all events for one account
SORTKEY(account_id, event_timestamp);   -- fast range scans per account

    account_id,
    resulting_balance AS balance_at_time,
    event_timestamp,
    event_type
FROM account_events
WHERE account_id = 'ACC-EU-001'
  AND event_timestamp <= '2024-03-15 14:30:00'

-- FULL EVENT HISTORY FOR AUDIT
SELECT
    event_id,
    event_type,
    event_timestamp,
    JSON_EXTRACT_PATH_TEXT(event_data, 'amount') AS amount,
    JSON_EXTRACT_PATH_TEXT(event_data, 'currency') AS currency,
    resulting_balance,
    correlation_id
FROM account_events
WHERE account_id = 'ACC-EU-001'
ORDER BY event_timestamp ASC;
-- Returns: complete chronological history
-- Execution: 1-5 seconds (sort key optimized)

-- BALANCE VERIFICATION (prove current = sum of events)
SELECT
    account_id,
    SUM(CASE
        WHEN event_type IN ('deposit', 'transfer_received', 'interest_applied')
            THEN CAST(JSON_EXTRACT_PATH_TEXT(event_data, 'amount') AS DECIMAL)
        WHEN event_type IN ('withdrawal', 'transfer_sent', 'fee_charged')
            THEN -CAST(JSON_EXTRACT_PATH_TEXT(event_data, 'amount') AS DECIMAL)
        ELSE 0
    END) AS calculated_balance,
    MAX(resulting_balance) AS stored_balance
FROM account_events
WHERE account_id = 'ACC-EU-001'
GROUP BY account_id;
-- calculated_balance MUST equal stored_balance

🟢 MONGODB MINDSET — How This Same Problem Is Approached

Event Sourcing Architecture in MongoDB

MONGODB EVENT SOURCING — CORE CONCEPTS:

1. EVENT STORE (events collection)
   → Append-only: insertOne() only, never updateOne() or deleteOne()
   → Each document = one event (one thing that happened)
   → Immutable: once written, never changed
   → Ordered: timestamp + sequence number guarantee ordering
   → The SOURCE OF TRUTH — everything else is derived

2. CURRENT STATE (projections/snapshots)
   → DERIVED from events — not the source of truth
   → Materialized for performance (avoid replaying all events)
   → Can be rebuilt from scratch by replaying events
   → Updated by change stream processors or in-application logic

3. CHANGE STREAMS (event distribution)
   → Watch the events collection for new inserts
   → Push to downstream systems in real-time
   → Redshift (via Kafka), compliance engine, ML pipeline
   → Resume tokens guarantee exactly-once delivery

4. SNAPSHOTS (optimization)
   → Periodic materialized state at a point in time
   → "Account ACC-001 had balance $8920.30 at 2024-06-01T00:00:00Z"
   → To reconstruct at any time T:
     → Find latest snapshot BEFORE T
     → Replay events AFTER snapshot up to T
   → Without snapshots: must replay ALL events from beginning (slow)

🧩 HOW TO APPROACH IT IN MONGODB

EVENT SOURCING DESIGN FRAMEWORK:

STEP 1 — DEFINE YOUR EVENTS
    → What things HAPPEN to an account?
    → Each event = a verb in past tense
    → "AccountCreated", "DepositMade", "WithdrawalMade"
    → "TransferSent", "TransferReceived", "InterestApplied"
    → "FeeCharged", "AccountFrozen", "AccountClosed"
    → Each event carries the data that describes WHAT happened

STEP 2 — DESIGN EVENT DOCUMENTS
    → Every event has: who, what, when, where, why
    → Metadata: event_id, event_type, timestamp, sequence
    → Aggregate: account_id (the entity this event belongs to)
    → Payload: event-type-specific data
    → Context: correlation_id, causation_id, user, source

STEP 3 — DESIGN PROJECTION (Read Model)
    → Current state derived from events
    → Optimized for how the application READS data
    → Updated in real-time via change stream
    → Can be completely rebuilt by replaying events

STEP 4 — DESIGN SNAPSHOT STRATEGY
    → How often to snapshot? Every N events? Every N hours?
    → Trade-off: more snapshots = faster reconstruction
                  fewer snapshots = less storage overhead
    → Financial system: snapshot every 100 events or every day

STEP 5 — DESIGN EVENT DISTRIBUTION
    → Change stream watches events collection
    → Routes events to: projection updater, Kafka, compliance, ML
    → Resume tokens ensure no events lost on restart

Phase B: Command Handler — Appending Events

// ═══════════════════════════════════════════════════════
// COMMAND HANDLER
// Processes commands → validates → appends events
// This is the WRITE side of CQRS

// ─── CORE EVENT APPENDING FUNCTION ───────────────────

function appendEvent(streamId, eventType, eventData, metadata, country) {
    // Step 1: Get current sequence number for this stream
    let lastEvent = db.events.findOne(
        { stream_id: streamId },
        { sequence: 1, resulting_state: 1 },
        { sort: { sequence: -1 } }
    )
    
    let nextSequence = lastEvent ? lastEvent.sequence + 1 : 1
    let previousState = lastEvent ? lastEvent.resulting_state : null
    
    // Step 2: Calculate resulting state (apply event to previous state)
    let resultingState = applyEvent(previousState, eventType, eventData)
    
    // Step 3: Build event document
    let event = {
        event_id: `EVT-${Date.now()}-${Math.random().toString(36).substr(2, 6)}`,
        stream_id: streamId,
        stream_type: "account",
        sequence: NumberInt(nextSequence),
        event_type: eventType,
        event_data: eventData,
        resulting_state: resultingState,
        timestamp: new Date(),
        metadata: {
            correlation_id: metadata.correlation_id || null,
            causation_id: metadata.causation_id || null,
            source: metadata.source || "unknown",
            user: metadata.user || null,
            ip_address: metadata.ip_address || null
        },
        country: country
    }
    
    // Step 4: Insert (unique index on stream_id + sequence prevents duplicates)
    try {
        db.events.insertOne(event)
        return { success: true, event_id: event.event_id, sequence: nextSequence }
    } catch (error) {
        if (error.code === 11000) {
            // Duplicate key — concurrent write detected
            // OPTIMISTIC CONCURRENCY: retry with new sequence
            return { success: false, error: "concurrent_write", retry: true }
        }
        throw error
    }
}

// ─── STATE TRANSITION FUNCTION ───────────────────────
// Pure function: (previousState, event) → newState
// This is the HEART of event sourcing

function applyEvent(previousState, eventType, eventData) {
    // Default initial state
    let state = previousState || {
        balance: NumberDecimal("0.00"),
        status: "pending",
        transaction_count: 0,
        created_at: null,
        last_activity: null
    }
    
    // Copy state (don't mutate)
    state = Object.assign({}, state)
    
    switch (eventType) {
        case "AccountCreated":
            state.balance = eventData.initial_balance || NumberDecimal("0.00")
            state.status = "active"
            state.transaction_count = 0
            state.created_at = new Date()
            break
        
        case "DepositMade":
            state.balance = NumberDecimal(
                (parseFloat(state.balance.toString()) +
                 parseFloat(eventData.amount.toString())).toFixed(2)
            )
            state.transaction_count++
            break
        
        case "WithdrawalMade":
            let newBalance = parseFloat(state.balance.toString()) -
                             parseFloat(eventData.amount.toString())
            if (newBalance < 0) {
                throw new Error("Insufficient funds")
            }
            state.balance = NumberDecimal(newBalance.toFixed(2))
            state.transaction_count++
            break
        
        case "TransferSent":
            let afterTransfer = parseFloat(state.balance.toString()) -
                                parseFloat(eventData.amount.toString())
            if (afterTransfer < 0) {
                throw new Error("Insufficient funds for transfer")
            }
            state.balance = NumberDecimal(afterTransfer.toFixed(2))
            state.transaction_count++
            break
        
        case "TransferReceived":
            state.balance = NumberDecimal(
                (parseFloat(state.balance.toString()) +
                 parseFloat(eventData.amount.toString())).toFixed(2)
            )
            state.transaction_count++
            break
        
        case "InterestApplied":
            state.balance = NumberDecimal(
                (parseFloat(state.balance.toString()) +
                 parseFloat(eventData.amount.toString())).toFixed(2)
            )
            break
        
        case "FeeCharged":
            state.balance = NumberDecimal(
                (parseFloat(state.balance.toString()) -
                 parseFloat(eventData.amount.toString())).toFixed(2)
            )
            break
        
        case "AccountFrozen":
            state.status = "frozen"
            break
        
        case "AccountUnfrozen":
            state.status = "active"
            break
        
        case "AccountClosed":
            state.status = "closed"
            break
        
        default:
            throw new Error(`Unknown event type: ${eventType}`)
    }
    
    state.last_activity = new Date()
    return state
}

function getStateAtTime(streamId, targetTime) {
    // Step 1: Find latest snapshot BEFORE targetTime
    let snapshot = db.snapshots.findOne(
        {
            stream_id: streamId,
            snapshot_timestamp: { $lte: targetTime }
        },
        {},
        { sort: { snapshot_timestamp: -1 } }
    )
    
    // Step 2: Determine starting point
    let startSequence = snapshot ? snapshot.sequence : 0
    let state = snapshot ? snapshot.state : null
    
    // Step 3: Replay events from snapshot to targetTime
    let events = db.events.find({
        stream_id: streamId,
        sequence: { $gt: startSequence },
        timestamp: { $lte: targetTime }
    }).sort({ sequence: 1 }).toArray()
    
    // Step 4: Apply each event to reconstruct state
    for (let event of events) {
        state = applyEvent(state, event.event_type, event.event_data)
    }
    
    return {
        stream_id: streamId,
        state_at: targetTime,
        state: state,
        events_replayed: events.length,
        from_snapshot: snapshot ? snapshot.sequence : "beginning",
        reconstruction_method: snapshot ? "snapshot + replay" : "full replay"
    }
}

def apply_event(previous_state, event_type, event_data):
    """
    Pure function: (state, event) → new_state
    This is the CORE of event sourcing
    Deterministic: same inputs ALWAYS produce same output
    
    Redshift equivalent:
        -- No equivalent — Redshift stores current state, not events
        -- Closest: SUM(CASE...) over event history
    """
    
    if previous_state is None:
        state = {
            "balance": Decimal128("0.00"),
            "status": "pending",
            "transaction_count": 0,
            "created_at": None,
            "last_activity": None
        }
    else:
        state = dict(previous_state)
    
    balance = Decimal(str(state["balance"]))
    
    if event_type == "AccountCreated":
        initial = Decimal(str(event_data.get("initial_balance", "0.00")))
        state["balance"] = Decimal128(str(initial))
        state["status"] = "active"
        state["transaction_count"] = 0
        state["created_at"] = datetime.now()
    
    elif event_type == "DepositMade":
        amount = Decimal(str(event_data["amount"]))
        new_balance = balance + amount
        state["balance"] = Decimal128(str(new_balance))
        state["transaction_count"] = state.get("transaction_count", 0) + 1
    
    elif event_type == "WithdrawalMade":
        amount = Decimal(str(event_data["amount"]))
        new_balance = balance - amount
        if new_balance < 0:
            raise ValueError(f"Insufficient funds: {balance} - {amount}")
        state["balance"] = Decimal128(str(new_balance))
        state["transaction_count"] = state.get("transaction_count", 0) + 1
    
    elif event_type == "TransferSent":
        amount = Decimal(str(event_data["amount"]))
        new_balance = balance - amount
        if new_balance < 0:
            raise ValueError(f"Insufficient funds for transfer")
        state["balance"] = Decimal128(str(new_balance))
        state["transaction_count"] = state.get("transaction_count", 0) + 1
    
    elif event_type == "TransferReceived":
        amount = Decimal(str(event_data["amount"]))
        new_balance = balance + amount
        state["balance"] = Decimal128(str(new_balance))
        state["transaction_count"] = state.get("transaction_count", 0) + 1
    
    elif event_type == "InterestApplied":
        amount = Decimal(str(event_data["amount"]))
        state["balance"] = Decimal128(str(balance + amount))
    
    elif event_type == "FeeCharged":
        amount = Decimal(str(event_data["amount"]))
        state["balance"] = Decimal128(str(balance - amount))
    
    elif event_type == "AccountFrozen":
        state["status"] = "frozen"
    
    elif event_type == "AccountUnfrozen":
        state["status"] = "active"
    
    elif event_type == "AccountClosed":
        state["status"] = "closed"
    
    else:
        raise ValueError(f"Unknown event type: {event_type}")
    
    state["last_activity"] = datetime.now()
    return state

def append_event(stream_id, event_type, event_data, metadata, country):
    """
    Append a new event to the event store
    
    Redshift equivalent:
        INSERT INTO account_events
            (event_type, account_id, country, event_data, resulting_balance, ...)
        VALUES (...);
        -- But: arrives via ETL, not real-time
        -- And: no change stream to notify downstream
    """
    
    wc = WriteConcern(w="majority", j=True)
    events_coll = db.get_collection("events", write_concern=wc)
    
    # Get current state
    last_event = db.events.find_one(
        {"stream_id": stream_id},
        {"sequence": 1, "resulting_state": 1},
        sort=[("sequence", DESCENDING)]
    )
    
    next_sequence = (last_event["sequence"] + 1) if last_event else 1
    previous_state = last_event["resulting_state"] if last_event else None
    
    # Calculate new state
    resulting_state = apply_event(previous_state, event_type, event_data)
    
    # Build event document
    event_doc = {
        "event_id": f"EVT-{int(time.time() * 1000)}-{uuid.uuid4().hex[:8]}",
        "stream_id": stream_id,
        "stream_type": "account",
        "sequence": next_sequence,
        "event_type": event_type,
        "event_data": event_data,
        "resulting_state": resulting_state,
        "timestamp": datetime.now(),
        "metadata": {
            "correlation_id": metadata.get("correlation_id"),
            "causation_id": metadata.get("causation_id"),
            "source": metadata.get("source", "unknown"),
            "user": metadata.get("user"),
            "ip_address": metadata.get("ip_address")
        },
        "country": country
    }
    
    # Insert with optimistic concurrency (unique index protects)
    max_retries = 3
    for attempt in range(max_retries):
        try:
            events_coll.insert_one(event_doc)
            
            # Update current_state projection inline (simple approach)
            # In production: change stream processor handles this
            db.current_state.update_one(
                {"_id": stream_id},
                {"$set": {
                    "stream_id": stream_id,
                    "country": country,
                    "state": resulting_state,
                    "last_event_type": event_type,
                    "last_event_time": event_doc["timestamp"],
                    "last_sequence": next_sequence
                }},
                upsert=True
            )
            
            return {
                "success": True,
                "event_id": event_doc["event_id"],
                "sequence": next_sequence,
                "resulting_state": resulting_state
            }
        
        except Exception as e:
            if "duplicate key" in str(e).lower():
                # Concurrent write — recalculate sequence
                last_event = db.events.find_one(
                    {"stream_id": stream_id},
                    {"sequence": 1, "resulting_state": 1},
                    sort=[("sequence", DESCENDING)]
                )
                next_sequence = last_event["sequence"] + 1
                previous_state = last_event["resulting_state"]
                resulting_state = apply_event(
                    previous_state, event_type, event_data
                )
                event_doc["sequence"] = next_sequence
                event_doc["resulting_state"] = resulting_state
                event_doc["event_id"] = (
                    f"EVT-{int(time.time()*1000)}-{uuid.uuid4().hex[:8]}"
                )
                continue
            raise
    
    return {"success": False, "error": "max_retries_exceeded"}

def send_transfer(from_account, to_account, amount, currency,

            "source": "transfer-service",
            "user": user_id,

    )

    credit_result = append_event(
        stream_id=to_account,
        event_type="TransferReceived",
        event_data={
            "amount": Decimal128(str(amount)),

        "debit": debit_result,
        "credit": credit_result,
        "correlation_id": correlation_id
    }


def apply_interest(account_id, rate, country):
    """Apply interest to account balance"""
    
    current = db.current_state.find_one({"_id": account_id})
    if not current or current["state"]["status"] != "active":
        return None
    
    balance = Decimal(str(current["state"]["balance"]))
    interest = (balance * Decimal(str(rate)) / Decimal("12")).quantize(Decimal("0.01"))
    
    if interest <= 0:

    return append_event(
        stream_id=account_id,
        event_type="InterestApplied",
        event_data={
            "amount": Decimal128(str(interest)),
            "rate": rate,
            "period": "monthly",
            "base_balance": Decimal128(str(balance))
        },
        metadata={
            "source": "interest-service",
            "correlation_id": f"CORR-INT-{int(time.time())}"
        },
        country=country
    )

def create_snapshot(stream_id):
    """
    Create a point-in-time snapshot of account state
    Used to speed up state reconstruction (avoid replaying all events)
    
    Redshift equivalent:
        REFRESH MATERIALIZED VIEW mv_account_current_state;
        -- But: refreshes ALL accounts, not one
        -- And: scheduled, not triggered by event count
    """
    
    last_event = db.events.find_one(
        {"stream_id": stream_id},
        {"sequence": 1, "resulting_state": 1, "country": 1, "timestamp": 1},
        sort=[("sequence", DESCENDING)]
    )
    
    if not last_event:
        return None
    
    snapshot = {
        "stream_id": stream_id,
        "stream_type": "account",
        "sequence": last_event["sequence"],
        "state": last_event["resulting_state"],
        "snapshot_timestamp": datetime.now(),
        "event_count_at_snapshot": last_event["sequence"],
        "country": last_event.get("country", "??")
    }
    
    db.snapshots.insert_one(snapshot)
    return snapshot


def create_snapshots_bulk(snapshot_every_n=100):
    """
    Create snapshots for all accounts that need them
    Run as scheduled job (e.g., hourly)
    """
    
    print(f"\n{'=' * 70}")
    print(f"BULK SNAPSHOT CREATION (every {snapshot_every_n} events)")
    print(f"{'=' * 70}")
    
    created = 0
    skipped = 0
    
    # Find all unique streams
    streams = db.events.distinct("stream_id")
    
    for stream_id in streams:
        # Get latest snapshot
        last_snapshot = db.snapshots.find_one(
            {"stream_id": stream_id},
            {"sequence": 1},
            sort=[("sequence", DESCENDING)]
        )
        
        # Get latest event
        last_event = db.events.find_one(
            {"stream_id": stream_id},
            {"sequence": 1},
            sort=[("sequence", DESCENDING)]
        )
        
        if not last_event:
            skipped += 1
            continue
        
        last_snap_seq = last_snapshot["sequence"] if last_snapshot else 0
        events_since = last_event["sequence"] - last_snap_seq
        
        if events_since >= snapshot_every_n:
            create_snapshot(stream_id)
            created += 1
        else:
            skipped += 1
    
    print(f"  Snapshots created: {created}")
    print(f"  Accounts skipped (not enough events): {skipped}")
    return created

def get_state_at_time(stream_id, target_time):
    """
    Reconstruct account state at any point in history
    THE killer feature of event sourcing
    
    Redshift equivalent:
        SELECT resulting_balance FROM account_events
        WHERE account_id = 'ACC-EU-001'
          AND event_timestamp <= '2024-03-15 14:30:00'
        ORDER BY event_timestamp DESC LIMIT 1;
        -- Simple query, but requires resulting_balance stored per event
        -- If not stored: must SUM(CASE...) all events up to that point
    
    MongoDB:
        → Find latest snapshot before target_time
        → Replay events from snapshot to target_time
        → Return reconstructed state
    """
    
    # Step 1: Find latest snapshot before target_time
    snapshot = db.snapshots.find_one(
        {
            "stream_id": stream_id,
            "snapshot_timestamp": {"$lte": target_time}
        },
        sort=[("snapshot_timestamp", DESCENDING)]
    )
    
    # Step 2: Determine replay starting point
    if snapshot:
        start_sequence = snapshot["sequence"]
        state = snapshot["state"]
        method = "snapshot + replay"
    else:
        start_sequence = 0
        state = None
        method = "full replay"
    
    # Step 3: Replay events from start to target_time
    events = list(db.events.find(
        {
            "stream_id": stream_id,
            "sequence": {"$gt": start_sequence},
            "timestamp": {"$lte": target_time}
        }
    ).sort("sequence", ASCENDING))
    
    # Step 4: Apply each event
    for event in events:
        state = apply_event(state, event["event_type"], event["event_data"])
    
    return {
        "stream_id": stream_id,
        "state_at": target_time,
        "state": state,
        "events_replayed": len(events),
        "from_snapshot": snapshot["sequence"] if snapshot else "beginning",
        "reconstruction_method": method
    }

def verify_balance(stream_id):
    """
    Prove current balance matches complete event replay
    CRITICAL for regulatory compliance
    """
    
    events = list(db.events.find(
        {"stream_id": stream_id},
        {"event_type": 1, "event_data": 1, "resulting_state": 1, "sequence": 1}
    ).sort("sequence", ASCENDING))
    
    if not events:
        return {"stream_id": stream_id, "error": "no events found"}
    
    # Replay ALL events from scratch
    replayed_state = None
    for event in events:
        replayed_state = apply_event(
            replayed_state, event["event_type"], event["event_data"]
        )
    
    # Get stored values
    last_event = events[-1]
    stored_balance = Decimal(str(last_event["resulting_state"]["balance"]))
    replayed_balance = Decimal(str(replayed_state["balance"]))
    
    # Get projection value
    projection = db.current_state.find_one({"_id": stream_id})
    projection_balance = (
        Decimal(str(projection["state"]["balance"]))
        if projection else None
    )
    
    return {
        "stream_id": stream_id,
        "total_events": len(events),
        "stored_balance": float(stored_balance),
        "replayed_balance": float(replayed_balance),
        "projection_balance": float(projection_balance) if projection_balance else None,
        "balances_match": stored_balance == replayed_balance,
        "projection_matches": stored_balance == projection_balance,
        "verified": (stored_balance == replayed_balance and
                     stored_balance == projection_balance)
    }

def start_event_processor(duration_seconds=60):
    """
    Watches events collection → updates projections + downstream
    This is the BRIDGE between write model and read model
    
    Redshift equivalent:
        -- No equivalent
        -- Would need: Kinesis → Lambda → Redshift COPY
        -- Plus: separate Lambda for each downstream system
        -- MongoDB: single change stream feeds everything
    """
    
    print(f"\n{'=' * 70}")
    print(f"🔄 EVENT PROCESSOR ({duration_seconds}s)")
    print(f"{'=' * 70}")
    
    # Load resume token
    token_doc = db.processor_state.find_one({"_id": "event_processor"})
    
    watch_options = {"full_document": "updateLookup"}
    if token_doc and token_doc.get("resume_token"):
        watch_options["resume_after"] = token_doc["resume_token"]
        print(f"  Resuming from saved position")
    
    pipeline = [{"$match": {"operationType": "insert"}}]
    
    processed = 0
    snapshots_created = 0
    alerts_raised = 0
    start_time = time.time()
    
    try:
        with db.events.watch(pipeline, **watch_options) as stream:
            while time.time() - start_time < duration_seconds:
                event_change = stream.try_next()
                
                if event_change is None:
                    time.sleep(0.1)
                    continue
                
                event = event_change.get("fullDocument", {})
                if not event:
                    continue
                
                processed += 1
                stream_id = event.get("stream_id")
                event_type = event.get("event_type")
                sequence = event.get("sequence", 0)
                
                # ─── 1. Update current state projection ──
                db.current_state.update_one(
                    {"_id": stream_id},
                    {"$set": {
                        "stream_id": stream_id,
                        "country": event.get("country"),
                        "state": event.get("resulting_state"),
                        "last_event_type": event_type,
                        "last_event_time": event.get("timestamp"),
                        "last_sequence": sequence
                    }},
                    upsert=True
                )
                
                # ─── 2. Auto-snapshot every 100 events ───
                if sequence > 0 and sequence % 100 == 0:
                    db.snapshots.insert_one({
                        "stream_id": stream_id,
                        "stream_type": "account",
                        "sequence": sequence,
                        "state": event.get("resulting_state"),
                        "snapshot_timestamp": datetime.now(),
                        "event_count_at_snapshot": sequence,
                        "country": event.get("country")
                    })
                    snapshots_created += 1
                
                # ─── 3. Route to alert handlers ──────────
                if event_type == "AccountFrozen":
                    db.compliance_alerts.insert_one({
                        "type": "account_frozen",
                        "stream_id": stream_id,
                        "country": event.get("country"),
                        "timestamp": event.get("timestamp"),
                        "metadata": event.get("metadata"),
                        "status": "pending_review"
                    })
                    alerts_raised += 1
                
                elif event_type == "TransferSent":
                    amount = Decimal(str(event["event_data"].get("amount", "0")))
                    if amount > 10000:
                        db.aml_alerts.insert_one({
                            "type": "high_value_transfer",
                            "stream_id": stream_id,
                            "amount": event["event_data"]["amount"],
                            "country": event.get("country"),
                            "timestamp": event.get("timestamp"),
                            "status": "pending_review"
                        })
                        alerts_raised += 1
                
                # ─── 4. Save resume token periodically ───
                if processed % 50 == 0:
                    db.processor_state.update_one(
                        {"_id": "event_processor"},
                        {"$set": {
                            "resume_token": event_change["_id"],
                            "last_processed": datetime.now(),
                            "events_processed": processed
                        }},
                        upsert=True
                    )
                
                # Progress logging
                if processed % 100 == 0:
                    print(f"  Processed: {processed} | "
                          f"Snapshots: {snapshots_created} | "
                          f"Alerts: {alerts_raised}")
    
    except KeyboardInterrupt:
        print("\n  Processor stopped by user")
    
    elapsed = round(time.time() - start_time, 1)
    print(f"\n  {'=' * 50}")
    print(f"  Processor ran: {elapsed}s")
    print(f"  Events processed: {processed}")
    print(f"  Snapshots created: {snapshots_created}")
    print(f"  Alerts raised: {alerts_raised}")
    print(f"  {'=' * 50}")
    
    return processed

✅ WHAT SHOULD DO

SHOULD DO:

    ✅ STORE resulting_state IN EVERY EVENT
       → Enables instant point-in-time queries without replay
       → Trade-off: larger events but dramatically faster reads
       → Redshift approach too: store resulting_balance per event row

    ✅ USE UNIQUE INDEX ON (stream_id, sequence) FOR ORDERING
       → Prevents duplicate sequence numbers (concurrent write protection)
       → Guarantees strict ordering within each stream
       → Optimistic concurrency: retry on duplicate key error

    ✅ IMPLEMENT SNAPSHOTS FOR LONG-LIVED STREAMS
       → Account with 10,000 events: replay takes seconds without snapshot
       → With snapshot every 100 events: replay max 100 events
       → Snapshot strategy: every N events OR every N hours

    ✅ USE CHANGE STREAMS FOR PROJECTION UPDATES
       → Don't update projection inside command handler (tight coupling)
       → Change stream processor: loosely coupled, independently scalable
       → Resume tokens: exactly-once processing guarantee

    ✅ ENFORCE IMMUTABILITY AT DATABASE LEVEL
       → RBAC role: insert + find only (no update, no remove)
       → Schema validation: strict on required fields
       → Application convention is NOT enough for compliance

    ✅ USE CORRELATION IDs TO LINK RELATED EVENTS
       → Transfer: 2 events (debit + credit) linked by correlation_id
       → Audit trail: trace complete business transaction across accounts
       → Index on metadata.correlation_id for fast lookups

    ✅ DESIGN EVENTS AS PAST-TENSE VERBS
       → "AccountCreated" not "CreateAccount"
       → "DepositMade" not "MakeDeposit"
       → Events record what HAPPENED, not what should happen
       → Commands are imperative, events are declarative

❌ WHAT SHOULD NOT DO

DO NOT:

    ❌ UPDATE OR DELETE EVENTS EVER
       → The ENTIRE point is immutability
       → "But the amount was wrong" → append a CorrectionApplied event
       → "But the account shouldn't exist" → append AccountClosed event
       → Never rewrite history — always append compensating events

    ❌ QUERY THE EVENTS COLLECTION FOR CURRENT STATE
       → db.events.find({ stream_id: "ACC-001" }).sort({ sequence: -1 }).limit(1)
       → This works but is SLOW at scale (index scan per query)
       → Use the current_state PROJECTION collection instead
       → Projection = read-optimized, pre-computed by change stream

    ❌ PUT BUSINESS LOGIC IN THE CHANGE STREAM PROCESSOR
       → Processor should: update projections, route events, create snapshots
       → Processor should NOT: validate commands, enforce business rules
       → Business logic belongs in COMMAND HANDLERS (before event creation)
       → Separation of concerns: commands validate, events record, processors project

    ❌ SKIP THE SNAPSHOT STRATEGY
       → Account open for 5 years = 50,000+ events
       → Replaying 50,000 events for point-in-time: 10+ seconds
       → With snapshots every 100: replay max 100 events: <50ms
       → Snapshots are NOT optional for production event sourcing

    ❌ USE MONGODB TRANSACTIONS FOR EVENT APPEND + PROJECTION UPDATE
       → Tempting: atomic append event + update projection
       → Anti-pattern: tight coupling between write and read model
       → Correct: append event → change stream → update projection
       → Eventual consistency between event and projection is OK (milliseconds)

    ❌ STORE EVENTS AND PROJECTIONS IN SAME COLLECTION
       → Events: append-only, immutable, chronological
       → Projections: mutable, updated frequently, query-optimized
       → Different access patterns = different collections
       → Different retention policies (events: forever, projections: rebuildable)

    ❌ ASSUME EVENT SOURCING IS ALWAYS BETTER
       → Simple CRUD with no audit requirement: traditional model is simpler
       → Event sourcing adds: complexity, storage, processing overhead
       → USE event sourcing when: audit trail, time-travel, event-driven architecture
       → DON'T use when: simple read/write with no history requirement

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 43:

PITFALL 1: "I'll just add a history table alongside my main table"
    ❌ accounts table (mutable) + accounts_history (copy-on-write)
    ✅ events collection (source of truth) + current_state (derived)
    WHY: History table is a COPY — can diverge from main table
         Events are THE truth — current state is always derivable
         If current_state gets corrupted: rebuild from events
         If history table diverges: which one is correct? Unknown

PITFALL 2: "I'll store events as one big JSON document per account"
    ❌ { account_id: "ACC-001", events: [ {}, {}, {} ... ] }
    ✅ One document per event, one collection for all events
    WHY: Array grows unbounded → 16MB document limit
         Can't efficiently query individual events
         Can't use change streams on array elements
         Redshift equivalent trap: storing all events in one SUPER column

PITFALL 3: "Event replay is just SELECT * ORDER BY timestamp"
    ❌ Assuming timestamp alone guarantees correct ordering
    ✅ Use explicit sequence number per stream
    WHY: Two events at same millisecond: which comes first?
         Clock skew across servers: timestamps may not be monotonic
         Sequence number: guaranteed monotonic per stream
         unique index on (stream_id, sequence) enforces this

PITFALL 4: "I don't need snapshots — replaying is fast"
    ❌ No snapshots, replaying from event #1 every time
    ✅ Snapshot every N events or every N hours
    WHY: 10 events: <1ms replay (fine)
         1,000 events: ~50ms replay (acceptable)
         100,000 events: ~5 seconds replay (unacceptable for API response)
         Snapshots reduce replay to constant time regardless of event count

PITFALL 5: "I'll use the same collection for events and current state"
    ❌ Mixing mutable projections with immutable events
    ✅ Separate collections: events (immutable) + current_state (mutable)
    WHY: Different access patterns, different retention, different security
         Events: append-only, never change, keep forever
         Projections: updated on every event, rebuildable, deletable
