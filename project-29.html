Problem: Build an E-Commerce Product Search with Fuzzy Matching, Autocomplete, Facets, and Relevance Scoring

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform (built across Projects 1-28) now needs
    SEARCH FUNCTIONALITY for the product catalog

    Users expect:
    → Type "wireles mous" (misspelled) → find "Wireless Mouse"
    → Type "lap" in search bar → autocomplete suggests "Laptop", "Laptop Stand"
    → Search "comfortable running shoes" → ranked by RELEVANCE, not insertion order
    → Filter results by facets: category, price range, brand — with counts
    → Highlight matching terms in results
    → Search across name, description, tags, brand simultaneously
    → Sub-100ms response time on 5 million products

    Your Redshift brain says: "I'll use LIKE or maybe a regex"
    Your MongoDB brain should say: "This is a SEARCH ENGINE problem, not a query problem"

    You have 5 million product documents that look like this:
    {
        name: "Ultra Comfort Running Shoes Pro",
        description: "Lightweight breathable mesh upper with responsive cushioning...",
        brand: "SpeedStride",
        category: "Footwear",
        subcategory: "Running",
        price: 129.99,
        rating: 4.7,
        reviews_count: 2340,
        tags: ["running", "lightweight", "cushioned", "breathable", "marathon"],

        created_at: ISODate("2024-03-15T10:00:00Z")
    }

🔴 REDSHIFT MINDSET — How You Would Solve This

Attempt 1: LIKE Operator (The Naive Approach)

-- REDSHIFT: Search for products matching "wireless mouse"
SELECT *
FROM products
WHERE LOWER(name) LIKE '%wireless%'
  AND LOWER(name) LIKE '%mouse%'
ORDER BY reviews_count DESC
LIMIT 20;

-- PROBLEMS:
-- → Full table scan EVERY TIME (LIKE '%...%' cannot use sort keys)
-- → No relevance scoring (which result is BEST match?)
-- → No fuzzy matching ("wireles" won't match "wireless")
-- → No stemming ("running" won't match "run" or "runner")
-- → No synonym support ("sneakers" won't match "shoes")
-- → No autocomplete capability
-- → No faceted counts (how many in each category?)
-- → Performance: 5M rows × full scan = 5-30 seconds
-- → This is UNACCEPTABLE for a search bar

Attempt 2: SIMILAR TO / REGEXP (Slightly Better, Still Terrible)

-- REDSHIFT: Regex-based search
SELECT *
FROM products
WHERE name ~* '(wireless|wifi).*(mouse|mice)'
   OR description ~* '(wireless|wifi).*(mouse|mice)'
ORDER BY reviews_count DESC
LIMIT 20;

-- PROBLEMS:
-- → Still full table scan (regex cannot use indexes in Redshift)
-- → Even SLOWER than LIKE (regex engine overhead)
-- → Still no relevance scoring
-- → Still no fuzzy matching
-- → Still no autocomplete
-- → You're writing increasingly complex regex — unmaintainable

CRITICAL DISTINCTION — TWO TYPES OF INDEXES:
    
    DATABASE INDEXES (Project 9-11):
    → B-tree indexes on fields
    → Used by find() and aggregation pipeline
    → Created with: db.collection.createIndex()
    → Purpose: speed up EXACT matches, ranges, sorts
    → Example: { email: 1 } for exact email lookup
    
    SEARCH INDEXES (This Project):
    → Lucene-based inverted indexes
    → Used ONLY by $search stage in aggregation pipeline
    → Created in Atlas UI, Atlas CLI, or via createSearchIndex()
    → Purpose: full-text search, fuzzy matching, scoring
    → Example: index on "name" and "description" fields
    
    THEY ARE COMPLETELY DIFFERENT SYSTEMS
    One does NOT replace the other
    → find({ email: "x@y.com" }) → uses database index (Project 9)
    → aggregate([ { $search: { text: "wireless mouse" } } ]) → uses search index

🧩 HOW TO APPROACH IT IN MONGODB

THINKING FRAMEWORK FOR SEARCH:

STEP 1: DEFINE WHAT USERS WILL SEARCH
    → Which fields? name, description, tags, brand
    → What behaviors? fuzzy, autocomplete, exact category filter
    → What results? scored, highlighted, faceted

STEP 2: CREATE A SEARCH INDEX (not a database index)
    → Define field mappings (which fields are searchable)
    → Choose analyzers (how text is tokenized and processed)
    → Configure autocomplete fields (edge n-gram)
    → This is done in Atlas UI or via API — NOT createIndex()

STEP 3: BUILD $search STAGE IN AGGREGATION PIPELINE
    → $search MUST be the FIRST stage in the pipeline
    → Follow with $project, $limit, $match for post-processing
    → This is where your Project 12-13 skills apply directly

STEP 4: TUNE RELEVANCE
    → Boost certain fields (name match > description match)
    → Apply function scoring (boost by rating, recency)
    → Test with real queries, iterate

ANALOGY FOR REDSHIFT USERS:
    Database Index (createIndex)  = Sort Key (query acceleration)
    Search Index (Atlas Search)   = External Elasticsearch (full-text search)
    
    EXCEPT: Atlas Search is NOT external — it's built in

Step 2: Create a Search Index (THE Critical Step)

IMPORTANT:
    Search indexes are NOT created with createIndex()
    They are created through:
    1. Atlas UI (Search tab on your cluster)
    2. Atlas CLI (atlas clusters search indexes create)
    3. mongosh: db.collection.createSearchIndex() (MongoDB 7.0+)
    4. API: Atlas Admin API

    This is fundamentally different from database indexes (Project 9-11)
    
    DATABASE INDEX:  db.products.createIndex({ email: 1 })
    SEARCH INDEX:    db.products.createSearchIndex({ ... lucene config ... })
    
    DIFFERENT command, DIFFERENT engine, DIFFERENT purpose
// ═══════════════════════════════════════════════════════
// mongosh (MongoDB 7.0+) — Create Search Index
// ═══════════════════════════════════════════════════════

// METHOD 1: Dynamic Mapping (Index ALL fields automatically)
// Good for getting started — indexes everything as searchable
db.products.createSearchIndex(
    "default",           // index name
    {
        mappings: {
            dynamic: true    // automatically index all fields
        }
    }
)

// METHOD 2: Static Mapping (RECOMMENDED for production)
// You control EXACTLY which fields are searchable and HOW
db.products.createSearchIndex(
    "products_search",   // custom index name
    {
        mappings: {
            dynamic: false,  // only index fields we specify
            fields: {
                // ─── FULL-TEXT SEARCHABLE FIELDS ─────────
                name: {
                    type: "string",
                    analyzer: "lucene.standard",
                    // Also index for autocomplete
                    multi: {
                        autocomplete: {
                            type: "autocomplete",
                            tokenization: "edgeGram",
                            minGrams: 2,
                            maxGrams: 15,
                            foldDiacritics: true
                        }

                },

                    type: "string",
                    analyzer: "lucene.english"  // English stemming

Step 3: Basic Full-Text Search — $search Stage

// ═══════════════════════════════════════════════════════
// SEARCH QUERY 1: Basic text search
// "Find products matching 'wireless mouse'"
// ═══════════════════════════════════════════════════════

// ── REDSHIFT EQUIVALENT (for reference) ──────────────
// SELECT *, 0 as score
// FROM products
// WHERE LOWER(name) LIKE '%wireless%'
//   AND LOWER(name) LIKE '%mouse%'
// ORDER BY reviews_count DESC
// LIMIT 10;
// -- No relevance scoring, full table scan, no fuzzy match

// ── MONGODB ATLAS SEARCH ─────────────────────────────
db.products.aggregate([
    {
        $search: {
            index: "products_search",    // which search index to use
            text: {
                query: "wireless mouse",
                path: ["name", "description", "tags"],  // search across multiple fields
                fuzzy: {
                    maxEdits: 1,         // allow 1 character difference
                    prefixLength: 2      // first 2 chars must match exactly
                }
            }
        }
    },
    {
        $project: {
            name: 1,
            brand: 1,
            price: 1,
            category: 1,
            rating: 1,
            score: { $meta: "searchScore" }  // THE MAGIC — relevance score
        }
    },
    { $limit: 10 }
])

// OUTPUT (ordered by relevance score — highest first):
// {
//     name: "Wireless Bluetooth Mouse Ergonomic",
//     brand: "TechFlow",
//     price: 34.99,
//     category: "Electronics",
//     rating: 4.5,
//     score: 8.72            ← highest relevance (both terms in name)
// },
// {
//     name: "Wireless Gaming Mouse RGB",
//     brand: "GameForce",
//     price: 79.99,
//     category: "Electronics",
//     rating: 4.8,
//     score: 7.91            ← second highest
// },
// ...

// KEY OBSERVATIONS:
// → Results are RANKED by relevance (not insertion order)
// → Score reflects how well document matches query
// → "wireless" AND "mouse" both in name = highest score
// → Fuzzy: "wireles" (misspelled) would STILL match with maxEdits: 1
// → Searched across name + description + tags simultaneously
// → Sub-10ms execution time (inverted index lookup)
$search MUST BE THE FIRST STAGE:

    ✅ aggregate([ $search, $project, $limit ])
    ❌ aggregate([ $match, $search, $project ])    ← ERROR
    
    WHY: $search runs on the Lucene engine (mongot process)
         It MUST execute first to produce the result set
         Then subsequent stages ($project, $match, $sort, etc.)
         process those results in the MongoDB query engine
    
    This is like: Elasticsearch query FIRST, then post-processing
    NOT like: SQL WHERE clause that can appear anywhere

// ═══════════════════════════════════════════════════════
// SEARCH QUERY 2: Compound search with must/should/filter
// "Find running shoes, prefer highly rated, only in-stock"
// ═══════════════════════════════════════════════════════

// ── REDSHIFT EQUIVALENT ──────────────────────────────
// SELECT *
// FROM products
// WHERE (LOWER(name) LIKE '%running%' OR LOWER(description) LIKE '%running%')
//   AND (LOWER(name) LIKE '%shoes%' OR LOWER(description) LIKE '%shoes%')
//   AND in_stock = true
//   AND category = 'Footwear'
// ORDER BY rating DESC, reviews_count DESC
// LIMIT 10;
// -- No relevance scoring, no "prefer" semantics (just hard filters)

// ── MONGODB ATLAS SEARCH ─────────────────────────────
db.products.aggregate([
    {
        $search: {
            index: "products_search",
            compound: {
                // ── MUST: required conditions (like AND in WHERE) ──
                must: [
                    {
                        text: {
                            query: "running shoes",
                            path: ["name", "description", "tags"],
                            fuzzy: { maxEdits: 1 }
                        }
                    }
                ],

                // ── SHOULD: boost score if matched (no Redshift equivalent) ──
                should: [
                    {
                        range: {
                            path: "rating",
                            gte: 4.5,
                            score: { boost: { value: 3 } }  // 3x score boost
                        }
                    },
                    {
                        range: {
                            path: "reviews_count",
                            gte: 1000,
                            score: { boost: { value: 2 } }  // 2x score boost
                        }
                    }
                ],

                // ── FILTER: hard filter, does NOT affect score ──
                filter: [
                    {
                        equals: {
                            path: "in_stock",
                            value: true
                        }
                    }
                ],

                // ── MUST NOT: exclude (like AND NOT in WHERE) ──
                mustNot: [
                    {
                        text: {
                            query: "trail hiking",
                            path: "tags"
                        }
                    }
                ]
            }
        }
    },
    {
        $project: {
            name: 1,
            brand: 1,
            price: 1,
            rating: 1,
            reviews_count: 1,
            score: { $meta: "searchScore" }
        }
    },
    { $limit: 10 }
])

// OUTPUT:
// "Ultra Comfort Running Shoes Pro" → score: 15.3
//    (matches "running shoes", rating 4.7 → boosted, 2340 reviews → boosted)
//
// "Comfortable Memory Foam Running Sneakers" → score: 9.1
//    (matches "running" + synonym "sneakers"→"shoes", rating 4.3 → no boost)
//
// "Trail Running Shoes Waterproof" → EXCLUDED (mustNot: "trail")
COMPOUND QUERY STRUCTURE — THE MENTAL MODEL:

┌──────────────────────────────────────────────────────┐
│  compound: {                                          │
│      must:     → AND (required, affects score)        │
│      should:   → BOOST (optional, increases score)    │
│      filter:   → AND (required, does NOT affect score)│
│      mustNot:  → AND NOT (excludes matches)           │
│  }                                                    │
└──────────────────────────────────────────────────────┘

REDSHIFT TRANSLATION:
    must     → WHERE clause conditions
    should   → No equivalent (can't boost relevance in SQL)
    filter   → WHERE clause conditions (same as must functionally)
    mustNot  → WHERE NOT / AND NOT conditions

WHY "filter" vs "must" MATTERS:
    must:   "running shoes" → AFFECTS the relevance score
    filter: in_stock = true → does NOT affect score (just excludes)
    
    Use filter for binary conditions (in_stock, category)
    Use must for text relevance conditions
    This distinction DOES NOT EXIST in SQL

Step 8: Score Boosting — Custom Relevance Tuning

// ═══════════════════════════════════════════════════════
// SEARCH QUERY 6: Custom scoring
// Boost by: rating, reviews count, recency
// "Newer, highly-rated, popular products rank higher"
// ═══════════════════════════════════════════════════════

// ── REDSHIFT EQUIVALENT ──────────────────────────────
// SELECT *,
//     (CASE WHEN name LIKE '%mouse%' THEN 5 ELSE 0 END
//    + rating * 2
//    + LOG(reviews_count)
//    + 1.0 / (DATEDIFF(day, created_at, GETDATE()) + 1)) as manual_score
// FROM products
// WHERE name LIKE '%mouse%' OR description LIKE '%mouse%'
// ORDER BY manual_score DESC;
//
// -- Fragile, manual scoring formula
// -- No real relevance from text matching
// -- Full table scan

// ── MONGODB ATLAS SEARCH ─────────────────────────────
db.products.aggregate([
    {
        $search: {
            index: "products_search",
            compound: {
                must: [
                    {
                        text: {
                            query: "mouse",
                            path: ["name", "description", "tags"],
                            score: {
                                boost: {
                                    path: "rating",    // multiply score by rating value
                                    undefined: 1       // default if field missing
                                }
                            }
                        }
                    }
                ],
                should: [
                    // Boost newer products
                    {
                        near: {
                            path: "created_at",
                            origin: new Date(),       // boost closer to today
                            pivot: 7776000000         // 90 days in ms (decay rate)
                        }
                    },
                    // Boost popular products (many reviews)
                    {
                        range: {
                            path: "reviews_count",
                            gte: 5000,
                            score: { boost: { value: 2 } }
                        }
                    }
                ],
                filter: [
                    { equals: { path: "in_stock", value: true } }
                ]
            }
        }
    },
    {
        $project: {
            name: 1,
            brand: 1,
            price: 1,
            rating: 1,
            reviews_count: 1,
            score: { $meta: "searchScore" }
        }
    },
    { $limit: 10 }
])

// OUTPUT:
// "Wireless Gaming Mouse RGB"       → score: 42.3
//     (rating: 4.8 boost + 15600 reviews boost + recent)
// "Wireless Bluetooth Mouse Ergon..." → score: 31.7
//     (rating: 4.5 boost + 8920 reviews boost)

// SCORING BREAKDOWN:
//   Base text relevance (BM25) × rating boost × recency decay × popularity boost
//   This is SEARCH ENGINE scoring — fundamentally impossible in SQL

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

OPTIMIZATION 1: STORED SOURCE
    → By default, $search returns document _id, then fetches full doc
    → With "storedSource": true in index definition, frequently accessed
      fields are stored IN the search index
    → Avoids the lookup back to MongoDB → faster results
    → Like Elasticsearch's _source field
    → Trade-off: larger search index size

OPTIMIZATION 2: COUNT ONLY (for pagination)
    → $searchMeta with "count" to get total results without fetching docs
    → Useful for "showing 1-20 of 3,456 results"
    db.products.aggregate([
        { $searchMeta: { 
            index: "products_search",
            count: { type: "total" },  // or "lowerBound" for faster estimate
            text: { query: "shoes", path: "name" }
        }}
    ])

OPTIMIZATION 3: CUSTOM ANALYZERS
    → Define your own tokenization rules
    → Example: keep "Wi-Fi" as one token, not ["Wi", "Fi"]
    → Example: treat "USB-C" as searchable unit
    → Configure in search index definition under "analyzers" array

OPTIMIZATION 4: SEARCH AFTER (cursor-based pagination)
    → Instead of $skip for page 2, 3, 4... (slow for deep pages)
    → Use searchAfter token for efficient deep pagination
    → Same concept as Redshift cursor-based pagination vs OFFSET

OPTIMIZATION 5: EXPLAIN SEARCH QUERIES
    → Add "scoreDetails": true to see WHY a score was assigned
    db.products.aggregate([
        { $search: { 
            text: { query: "mouse", path: "name" },
            scoreDetails: true 
        }},
        { $project: { 
            name: 1, 
            scoreDetails: { $meta: "searchScoreDetails" } 
        }}
    ])

✅ WHAT SHOULD DO (Best Practices)

✅ USE STATIC MAPPINGS IN PRODUCTION
    → dynamic: true indexes EVERYTHING (wasteful)
    → static mappings: index only fields users search on
    → Smaller index = faster search = lower cost

✅ USE APPROPRIATE ANALYZERS PER FIELD
    → name: "lucene.standard" (general tokenization)
    → description: "lucene.english" (stemming: "running" → "run")
    → brand: "lucene.keyword" (exact match, no tokenization)
    → tags: "lucene.standard" (tokenize tag phrases)
    → sku: "lucene.keyword" (never tokenize identifiers)

✅ USE filter INSTEAD OF must FOR NON-TEXT CONDITIONS
    → in_stock, category, price range → filter (no score impact)
    → search terms → must (score impact)
    → filter is FASTER (can use cached bit sets)

✅ SEPARATE SEARCH INDEXES FOR DIFFERENT USE CASES
    → "products_search" for full-text product search
    → "products_autocomplete" for search-as-you-type
    → Different analyzers, different field mappings

✅ MONITOR SEARCH INDEX SIZE AND PERFORMANCE
    → Atlas UI → Search tab → Index stats
    → Large indexes impact node memory
    → Reindex if mapping changes significantly

✅ USE $search + $match TOGETHER STRATEGICALLY
    → $search: text relevance (runs on Lucene)
    → $match AFTER $search: additional MongoDB-native filtering
    → Example: $search for text → $match for complex array queries

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT USE $search FOR EXACT LOOKUPS
    → find({ email: "user@example.com" }) ← use database index (Project 9)
    → $search for exact email ← OVERKILL, slower, wrong tool
    → $search is for TEXT RELEVANCE, not exact match
    → Redshift trap: thinking "search index replaces all other indexes"

❌ DO NOT PUT $search AFTER OTHER STAGES
    → $search MUST be first stage in pipeline
    → aggregate([ $match, $search ]) ← ERROR
    → aggregate([ $search, $match ]) ← CORRECT

❌ DO NOT USE dynamic: true IN PRODUCTION
    → Indexes every field including nested objects and arrays
    → Massive index size, slow indexing, wasted resources
    → Only for prototyping and exploration

❌ DO NOT RELY ON SEARCH FOR REAL-TIME CONSISTENCY
    → Search index updates are NEAR real-time (seconds lag)
    → Document inserted → searchable after 1-5 seconds
    → If you need instant consistency: use find() with database index
    → This is same limitation as Elasticsearch (eventual consistency)

❌ DO NOT BUILD PAGINATION WITH $skip FOR DEEP PAGES
    → $skip: 10000 → Lucene must score and skip 10000 results
    → Extremely slow for deep pagination
    → Use searchAfter or limit user to reasonable page depths
    → Same problem exists in Elasticsearch and Redshift (OFFSET)

❌ DO NOT FORGET: ATLAS SEARCH REQUIRES ATLAS
    → Atlas Search is an ATLAS FEATURE (cloud-managed MongoDB)
    → NOT available on self-hosted MongoDB Community/Enterprise
    → Self-hosted alternative: use text indexes (Project 11) or external Elasticsearch
    → Redshift users: this is like Redshift features only available on AWS

❌ DO NOT INDEX FIELDS WITH HIGH CARDINALITY AS stringFacet
    → product_id as stringFacet → millions of buckets → OOM
    → Facets are for LOW cardinality: category, brand, size
    → Like Redshift: don't GROUP BY on a unique ID column

MongoDB Atlas Search — Lucene Engine Internals

THE mongot PROCESS:

When you create a search index, MongoDB Atlas deploys
a SEPARATE process called "mongot" alongside "mongod":

┌─────────────────────────────────────────────────┐
│  Atlas Node                                      │
│                                                   │
│  ┌─────────────┐       ┌─────────────────────┐  │
│  │  mongod      │       │  mongot              │  │
│  │  (database   │ ←───→ │  (search engine)     │  │
│  │   engine)    │ oplog  │  (Apache Lucene)     │  │
│  │              │ feed   │                      │  │
│  │  WiredTiger  │       │  Inverted Indexes    │  │
│  │  B-tree      │       │  Term Dictionaries   │  │
│  │  Documents   │       │  Positional Indexes  │  │
│  │              │       │  Doc Values (facets)  │  │
│  └─────────────┘       └─────────────────────┘  │
│        ↑                         ↑                │
│        │     Query Router        │                │
│        └────────┬────────────────┘                │
│                 │                                  │
│            Application                             │
│         db.products.aggregate([{$search:...}])     │
└─────────────────────────────────────────────────┘

QUERY FLOW:
    1. Application sends aggregate([ $search, ... ])
    2. mongod sees $search → routes to mongot
    3. mongot executes Lucene query on inverted index
    4. mongot returns scored document IDs to mongod
    5. mongod fetches full documents (unless storedSource)
    6. mongod executes remaining pipeline stages ($project, $limit)
    7. Results returned to application

INDEX UPDATE FLOW:
    1. Application inserts/updates/deletes a document (mongod)
    2. Change appears in oplog (replication log — Project 23)
    3. mongot tails the oplog (like change streams — Project 25)
    4. mongot updates Lucene index segments
    5. New/updated document becomes searchable (1-5 second delay)
    
    THIS IS WHY:
    → Atlas Search is "near real-time" not "real-time"
    → Same as Elasticsearch refresh_interval (default 1s)
    → Much faster than Redshift ETL → Elasticsearch sync (minutes/hours)

Lucene Inverted Index — How Text Search Actually Works

WHAT HAPPENS WHEN YOU INDEX "Ultra Comfort Running Shoes Pro":

STEP 1: ANALYZER PIPELINE
    ┌──────────────────────────────────────────────┐
    │ Input: "Ultra Comfort Running Shoes Pro"      │
    │                                               │
    │ 1. Character Filter: (none in standard)       │
    │    → "Ultra Comfort Running Shoes Pro"        │
    │                                               │
    │ 2. Tokenizer (standard): split on whitespace  │
    │    → ["Ultra", "Comfort", "Running",          │
    │       "Shoes", "Pro"]                         │
    │                                               │
    │ 3. Token Filters:                             │
    │    a. Lowercase:                              │
    │       → ["ultra", "comfort", "running",       │
    │          "shoes", "pro"]                       │
    │    b. Stemmer (if lucene.english):            │
    │       → ["ultra", "comfort", "run",           │
    │          "shoe", "pro"]                        │
    │       "running" → "run"                       │
    │       "shoes" → "shoe"                        │
    │                                               │
    │ Output tokens: ["ultra","comfort","run",      │
    │                  "shoe","pro"]                 │
    └──────────────────────────────────────────────┘

STEP 2: INVERTED INDEX BUILD
    term → [document_ids with positions]
    
    "ultra"    → { doc1: [pos:0] }
    "comfort"  → { doc1: [pos:1], doc5: [pos:0] }
    "run"      → { doc1: [pos:2], doc5: [pos:3], doc7: [pos:1] }
    "shoe"     → { doc1: [pos:3], doc5: [pos:4], doc7: [pos:2] }
    "pro"      → { doc1: [pos:4] }
    "wireless" → { doc2: [pos:0], doc3: [pos:0], doc6: [pos:2] }
    "mouse"    → { doc2: [pos:3], doc3: [pos:2] }
    "game"     → { doc3: [pos:1] }
    ...

STEP 3: AT QUERY TIME
    Query: "running shoes"
    → Analyzer: "running" → "run", "shoes" → "shoe"
    → Lookup: "run" → [doc1, doc5, doc7]
    → Lookup: "shoe" → [doc1, doc5, doc7]
    → Intersection + scoring: doc1 has both → highest score
    → BM25 scoring considers:
        → Term frequency (how often term appears in doc)
        → Inverse document frequency (rarer terms score higher)
        → Field length (shorter fields → higher relevance per match)
    → Return: doc1 (score: 8.7), doc5 (score: 6.2), doc7 (score: 5.1)
    
    TIME: O(log n) for index lookup + O(k) for scoring k matches
    vs REDSHIFT LIKE: O(n) full table scan EVERY TIME

STEP 4: FUZZY MATCHING
    Query: "runing" (misspelled, missing 'n')
    → maxEdits: 1 → generate terms within edit distance 1
    → "runing" → ["running", "ruling", "runing", "tuning", ...]
    → Look up each variant in inverted index
    → "running" → "run" (after stemming) → found!
    → Result returned despite misspelling

Redshift Internals — Why Search is Impossible

REDSHIFT STORAGE — WHY LIKE '%x%' IS SLOW:

For: SELECT * FROM products WHERE name LIKE '%wireless%'

    1. Redshift stores "name" column in 1MB columnar blocks
    2. Zone maps store min/max per block:
       Block 1: min="Adjustable Laptop Stand", max="Comfort Running Shoes"
       Block 2: min="Gaming Mouse RGB", max="Wireless Mouse Ergonomic"
       
    3. LIKE '%wireless%' → zone maps CANNOT help
       WHY? Zone maps only know min/max alphabetical values
       "wireless" could be INSIDE any block's min-max range
       → Every single block must be scanned
       → Every single value must be string-compared
       
    4. LIKE 'wireless%' (prefix only — no leading %) 
       → Zone maps CAN help slightly
       → Skip blocks where max < "wireless" or min > "wireless~"
       → But still sequential string comparison within matching blocks
       
    5. LIKE '%wireless%mouse%' (two patterns)
       → Even worse: two substring scans per value
       → O(n × m × k) where n=rows, m=pattern1 length, k=pattern2 length
       
    RESULT:
    ┌──────────────────────────────────────────────┐
    │  5 million products                           │
    │  × ~100 bytes avg name length                 │
    │  × substring search per value                 │
    │  = 500MB of sequential string scanning         │
    │  = 5-30 seconds per query                     │
    │  = UNACCEPTABLE for search bar                │
    └──────────────────────────────────────────────┘
    
    vs MongoDB Atlas Search:
    ┌──────────────────────────────────────────────┐
    │  Inverted index lookup: "wireless"            │
    │  → Hash/B-tree lookup in term dictionary      │
    │  → Retrieve posting list: [doc2, doc3, doc6]  │
    │  → Score 3 documents                          │
    │  = <10ms                                      │
    └──────────────────────────────────────────────┘

FUNDAMENTAL ARCHITECTURAL MISMATCH:

    ┌─────────────────────────────────────────────────────────┐
    │                                                          │
    │  Redshift columnar storage:                              │
    │  OPTIMIZED FOR: "Give me SUM(price) across ALL rows"     │
    │  → Reads ONE column, skips others → FAST aggregation     │
    │                                                          │
    │  NOT DESIGNED FOR: "Find rows WHERE text CONTAINS word"  │
    │  → Must read entire column, compare each value → SLOW    │
    │  → No inverted index, no tokenization, no scoring        │
    │                                                          │
    │  ─────────────────────────────────────────────────────── │
    │                                                          │
    │  Lucene inverted index:                                  │
    │  OPTIMIZED FOR: "Find documents containing this word"    │
    │  → Direct term lookup → posting list → FAST search       │
    │                                                          │
    │  NOT DESIGNED FOR: "Compute SUM across all documents"    │
    │  → Must traverse all postings → SLOW aggregation         │
    │                                                          │
    └─────────────────────────────────────────────────────────┘

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE WITH ATLAS SEARCH:

PITFALL 1: "I'll just use LIKE in MongoDB too"
    ❌ db.products.find({ name: /wireless/i })    ← regex = full collection scan
    ❌ db.products.find({ name: { $regex: "wireless", $options: "i" } })
    ✅ db.products.aggregate([{ $search: { text: { query: "wireless", path: "name" } } }])
    WHY: regex in find() uses database engine → COLLSCAN (like Redshift LIKE)
         $search uses Lucene engine → inverted index lookup
         COMPLETELY different execution paths

PITFALL 2: "I'll create a text index instead of Atlas Search"
    ❌ db.products.createIndex({ name: "text", description: "text" })
       then: db.products.find({ $text: { $search: "wireless mouse" } })
    → This is Project 11's text index — basic, limited
    → No fuzzy matching, no autocomplete, no facets, no compound queries
    → No custom analyzers, no highlighting, no score boosting
    ✅ Use Atlas Search for any real search functionality
    WHEN text index is OK: simple keyword search, self-hosted MongoDB

PITFALL 3: "Search index = database index, just fancier"
    ❌ Thinking createSearchIndex replaces createIndex
    → They serve COMPLETELY different purposes
    → Database index (B-tree): exact match, range queries, sort
      → Used by find(), $match in aggregation
    → Search index (Lucene): text relevance, fuzzy, autocomplete
      → Used ONLY by $search in aggregation
    → You need BOTH on the same collection
    → Example:
      → createIndex({ category: 1, price: -1 })  ← for find() queries
      → createSearchIndex(...)                     ← for $search queries

PITFALL 4: "I'll put $match before $search to pre-filter"
    ❌ aggregate([ { $match: { in_stock: true } }, { $search: ... } ])
       → ERROR: $search must be first stage
    ✅ aggregate([ { $search: { compound: { filter: [{ equals: { path: "in_stock", value: true }}], must: [...] } } } ])
       → Use filter INSIDE compound $search
    ✅ aggregate([ { $search: ... }, { $match: { some_complex_condition } } ])
       → Or $match AFTER $search for conditions search can't handle

PITFALL 5: "Search results are inconsistent with find() results"
    → You insert a document, immediately $search for it → NOT FOUND
    → But find() returns it immediately
    WHY: Search index updates are ASYNC (1-5 second delay)
         Database writes are SYNC (immediately consistent)
    → Same issue as Redshift → Elasticsearch sync lag
       but measured in SECONDS not MINUTES
    SOLUTION: for critical "just inserted" lookups, use find() not $search

PITFALL 6: "I'll index everything for search"
    ❌ mappings: { dynamic: true }  on a collection with 50 fields
    → Massive search index (can exceed data size)
    → Slow index builds, high memory usage on mongot
    → Most fields are never searched by users
    ✅ Index only fields users actually search on (5-10 fields typically)

PITFALL 7: "Facets on high-cardinality fields"
    ❌ Facet on product_id (millions of unique values)
    ❌ Facet on description (unique per document)
    ✅ Facet on category (10-50 values)
    ✅ Facet on brand (100-500 values)
    ✅ Facet on price ranges (5-10 buckets)
    WHY: Facets compute counts per unique value
         High cardinality = millions of buckets = OOM or timeout

📈 SCALABILITY CONSIDERATIONS

ATLAS SEARCH SCALING:

    VERTICAL SCALING:
    → mongot process runs on SAME nodes as mongod
    → Larger Atlas tier → more RAM/CPU for search
    → Search index lives in memory for best performance
    → M10 tier: fine for <1M docs with search
    → M30 tier: good for 1-10M docs with search
    → M50+ tier: needed for 10M+ docs with heavy search

    → Atlas Search works WITH sharding (Project 21-22)
    → Each shard has its own mongot process
    → Each shard has its own local search index
    → Query: mongos → scatter to all shards → each shard searches locally
             → gather results → merge by score → return top N
    → Like: Elasticsearch shard-level search → coordinator merge
    → Like: Redshift MPP scatter-gather across compute slices
    
    SCALING COMPARISON:

    │ SCALE             │ REDSHIFT + ES          │ MONGODB ATLAS SEARCH │
    ├──────────────────┼──────────────────────┼─────────────────────┤

    │                   │ + dedicated Redshift   │ with dedicated search│

    └──────────────────┴──────────────────────┴─────────────────────┘
    
    DEDICATED SEARCH NODES (Atlas feature):
    → Separate compute for search workload
    → Search doesn't impact database read/write performance
    → Like having separate Elasticsearch cluster
       BUT still integrated (no sync pipeline)

📐 DATA MODELING COMPARISON

HOW DATA MODEL AFFECTS SEARCH:

REDSHIFT MODEL (Normalized):
    products: id, name, price, category_id, brand_id
    categories: category_id, category_name
    brands: brand_id, brand_name
    product_tags: product_id, tag
    product_descriptions: product_id, description
    
    TO SEARCH: must JOIN 5 tables first, THEN apply LIKE
    → Even worse performance
    → Elasticsearch sync must denormalize during ETL

MONGODB MODEL (Denormalized — from Project 6, 14):
    products: {
        name, description, brand, category,
        tags: [...], price, rating, ...
    }
    
    TO SEARCH: all searchable data in ONE document
    → Atlas Search indexes the document directly
    → No JOINs, no denormalization step
    → THIS is why MongoDB + Atlas Search is powerful together
    → The document model IS the search document

SEARCH-OPTIMIZED MODELING TIPS:
    
    ✅ EMBED searchable data in the document
       → Don't reference brand as brand_id
       → Embed brand name: brand: "SpeedStride"
       → Atlas Search can't follow $lookup references
    
    ✅ USE ARRAYS for multi-value search fields
       → tags: ["running", "lightweight", "cushioned"]
       → Atlas Search indexes each array element separately
       → Search for "lightweight" matches this document
    
    ✅ SEPARATE display fields from search fields
       → name: "Ultra Comfort Running Shoes Pro"          ← displayed to user
       → search_text: "ultra comfort running shoes pro     ← search-optimized
       →               sneakers trainers athletic footwear" ← includes synonyms
       → (alternative to synonym mapping — simpler but manual)
    
    ✅ PRE-COMPUTE search-relevant scores
       → popularity_score: reviews_count * rating          ← for boost functions
       → (from Project 20: Computed Pattern)
    
    ❌ DON'T normalize for search
       → Redshift habit: normalize everything
       → MongoDB search: denormalize everything searchable into one doc
