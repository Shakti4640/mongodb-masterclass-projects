Chapter 1 — Entering the MongoDB World

Project 1: Your First Database & Collection

Problem:    Store a list of products for an e-commerce platform
Redshift:   CREATE TABLE → define schema → INSERT INTO
MongoDB:    Use database → insert document → schema auto-created
Teaches:    Database/Collection creation vs CREATE TABLE
            Why there's NO "CREATE TABLE" in MongoDB
            Connection: mongosh vs psql/Query Editor
Internals:  WiredTiger first look vs Columnar block storage

Project 2: Inserting Your First Documents

Problem:    Bulk load 1000 customer records into the system
Redshift:   COPY from S3 / INSERT INTO with values
MongoDB:    insertOne() vs insertMany() vs mongoimport
Teaches:    Document structure (BSON) vs Row structure (columnar)
            _id field (auto-generated) vs IDENTITY column
            Data types: BSON types vs Redshift SQL types
Internals:  How BSON stores data vs how columnar blocks store data
Builds on:  Project 1 (database/collection now has data)

Project 3: Reading Data — Your First Queries

Problem:    Find all customers from "New York" who spent > $500
Redshift:   SELECT * FROM customers WHERE city='New York' AND total_spent > 500
MongoDB:    db.customers.find({ city: "New York", total_spent: { $gt: 500 } })
Teaches:    find() = SELECT, query filter = WHERE clause
            Projection (returning specific fields) vs SELECT col1, col2
            Comparison operators: $gt, $lt, $eq, $ne, $in vs SQL operators
Builds on:  Project 2 (querying the data we inserted)

Project 4: Updating Documents vs UPDATE Statements

Problem:    A customer changes their shipping address and loyalty tier upgrades
Redshift:   UPDATE customers SET address='...' , tier='Gold' WHERE id = 1234
MongoDB:    updateOne() with $set, $inc, $push, $unset
Teaches:    Atomic field-level updates vs full-row UPDATE
            Update operators ($set, $inc, $push, $pull, $unset)
            Why MongoDB updates are fundamentally different from SQL UPDATE
Builds on:  Project 3 (updating the data we queried)

Project 5: Deleting Documents vs DELETE Statements

Problem:    Remove inactive users and purge old temporary cart data
Redshift:   DELETE FROM users WHERE last_login < '2023-01-01'
MongoDB:    deleteMany({ last_login: { $lt: ISODate("2023-01-01") } })
Teaches:    deleteOne() vs deleteMany()
            TTL indexes (auto-expiry) — nothing equivalent in Redshift
            drop() vs DROP TABLE / TRUNCATE
Builds on:  Project 4 (complete CRUD cycle now finished)

Chapter 2 — Thinking in Documents, Not Rows

Project 6: Nested Documents — The First Mind Shift

Problem:    Store an order with line items, shipping address, payment info
Redshift:   3-4 normalized tables: orders, order_items, addresses, payments
            JOIN them for every query
MongoDB:    ONE document with nested objects and arrays
Teaches:    Embedding vs Normalizing — THE fundamental difference
            Nested objects (address: { street, city, zip })
            Arrays of objects (items: [{ sku, qty, price }])
            Why "one query, one document" is MongoDB's philosophy
Builds on:  Projects 1-5 (now you see WHY documents ≠ rows)

Project 7: Flexible Schema — No ALTER TABLE Ever Again

Problem:    Product catalog where electronics have specs, clothing has sizes,
            books have ISBN — all in one collection
Redshift:   One wide table with NULLable columns OR multiple tables
            ALTER TABLE ADD COLUMN when new product type arrives
MongoDB:    Each document has whatever fields it needs — no schema lock
Teaches:    Polymorphic data pattern
            Schema flexibility vs schema rigidity
            When flexibility is POWER vs when it's CHAOS
            Schema validation (voluntary guardrails)
Builds on:  Project 6 (nested documents + now varying shapes)

Project 8: Data Types Deep Dive — BSON vs SQL Types

Problem:    Store IoT sensor data with timestamps, geo-coordinates,
            binary blobs, and decimal precision financial amounts
Redshift:   TIMESTAMP, GEOMETRY, VARBYTE, DECIMAL(38,10)
MongoDB:    ISODate, GeoJSON, BinData, Decimal128, ObjectId
Teaches:    Complete BSON type system vs SQL type system
            ObjectId anatomy (timestamp + machine + process + counter)
            Date handling differences (critical for Redshift users)
            Null vs missing field (MongoDB distinction Redshift doesn't have)
Builds on:  Project 7 (now you understand WHAT goes inside documents)

Chapter 3 — Indexing — Sort Keys Reimagined

Project 9: Single Field Indexes — Your First Sort Key Equivalent

Problem:    Customer lookup by email is slow on 10M documents
Redshift:   Define email as sort key → zone maps skip blocks
MongoDB:    createIndex({ email: 1 }) → B-tree enables direct lookup
Teaches:    Index creation, types, direction (1 ascending, -1 descending)
            B-tree index vs Zone Maps — fundamentally different mechanisms
            explain() to verify index usage vs Redshift EXPLAIN
Builds on:  Project 3 (now we optimize the queries we learned)

Project 10: Compound Indexes — Multi-Column Sort Keys

Problem:    Query orders by customer_id + order_date range efficiently
Redshift:   COMPOUND SORTKEY(customer_id, order_date)
MongoDB:    createIndex({ customer_id: 1, order_date: -1 })
Teaches:    Index prefix rule (ESR: Equality → Sort → Range)
            Why index field ORDER matters (same as sort key column order)
            Covered queries (index-only scans) vs Redshift zone map skipping
Builds on:  Project 9 (single → compound index progression)

Project 11: Index Types Zoo — What Redshift Doesn't Have

Problem:    Full-text search on product descriptions + geo-queries on store locations
Redshift:   LIKE '%keyword%' (full scan) + custom UDF for geo
MongoDB:    Text indexes + 2dsphere indexes (built-in)
Teaches:    Unique, Text, Geospatial, Hashed, Wildcard, Partial indexes
            Each index type = a capability Redshift requires external tools for
            Index overhead: storage + write penalty (same concept in both)
Builds on:  Project 10 (expanding beyond simple B-tree indexes)

Chapter 4 — Aggregation Pipeline — Your New SQL Engine

Project 12: Aggregation Pipeline Basics — GROUP BY Reimagined

Problem:    Revenue per product category per month from order data
Redshift:   SELECT category, DATE_TRUNC('month', order_date), SUM(amount)
            GROUP BY 1, 2 ORDER BY 1, 2
MongoDB:    aggregate([ $match → $group → $sort → $project ])
Teaches:    Pipeline concept: data flows through stages (UNIX pipe analogy)
            $match = WHERE, $group = GROUP BY, $sort = ORDER BY
            $project = SELECT, $limit = LIMIT
            Accumulators: $sum, $avg, $min, $max, $push
Builds on:  Project 3 (basic find → now complex aggregation)

Project 13: Advanced Aggregation — Window Functions & Joins

Problem:    Rank customers by spending within each region + enrich with region data
Redshift:   RANK() OVER (PARTITION BY region ORDER BY total_spent DESC)
            + JOIN with regions table
MongoDB:    $setWindowFields (window functions equivalent)
            $lookup (LEFT JOIN equivalent)
Teaches:    $setWindowFields = window functions (added MongoDB 5.0)
            $lookup = LEFT OUTER JOIN
            $unwind = flattening arrays (like Redshift UNNEST or lateral flatten)
            Pipeline within $lookup = subquery in JOIN
Builds on:  Project 12 (simple aggregation → advanced pipeline stages)

Project 14: Embedding vs Referencing — The JOIN Decision

Problem:    Blog platform: posts have authors and comments
            — embed comments? reference authors? both?
Redshift:   Always normalize: posts, authors, comments = 3 tables + JOINs
MongoDB:    Decision tree: embed vs reference based on access patterns
Teaches:    THE most important MongoDB design decision
            Embed when: read together, bounded growth, owned relationship
            Reference when: unbounded growth, many-to-many, independent access
            Hybrid patterns: partial embedding + references
Builds on:  Project 6 (first saw embedding) + Project 13 ($lookup = JOIN)

Chapter 5 — Read/Write Pattern Awareness

Project 15: OLTP vs OLAP — Why MongoDB and Redshift Exist Differently

Problem:    E-commerce: real-time cart operations + end-of-day analytics
Redshift:   Optimized for: scan millions of rows, aggregate, report
            Struggles with: single-row lookup, high-concurrency writes
MongoDB:    Optimized for: single-document read/write, flexible schema, real-time
            Struggles with: massive analytical scans across all data
Teaches:    Read/write pattern analysis as THE architecture decision driver
            Latency profiles: MongoDB (ms) vs Redshift (seconds for complex queries)
            Concurrency: MongoDB (thousands of ops/sec) vs Redshift (limited WLM slots)
            THIS is why both exist — neither replaces the other
Builds on:  All Phase 1 + Phase 2 projects → now you see the WHY behind everything

Chapter 6 — Schema Design Patterns (MongoDB's Secret Weapon)

Project 16: Bucket Pattern — Time Series Grouping

Problem:    Store millions of IoT sensor readings per hour efficiently
Redshift:   One row per reading, sort key on timestamp, partition by date
MongoDB:    Bucket pattern: one document per sensor per hour containing array of readings
Teaches:    Bucket pattern reduces document count 60x
            Why this matters for WiredTiger (fewer documents = less overhead)
Builds on:  Project 8 (IoT data types) + Project 14 (embedding decisions)

Project 17: Outlier Pattern — Handling Unbounded Arrays

Problem:    Social media: most users have <100 followers, celebrities have millions
Redshift:   Separate followers table, JOIN always
MongoDB:    Embed up to threshold, overflow into linked documents
Teaches:    16MB document limit and how to design around it
            hasExtras flag pattern
Builds on:  Project 14 (embedding limits) + Project 16 (pattern thinking)

Project 18: Polymorphic Pattern — One Collection, Many Shapes

Problem:    Insurance system: auto, home, life policies with different fields
Redshift:   Multiple tables OR one wide sparse table
MongoDB:    Single collection, each document has policy-type-specific fields
Teaches:    Discriminator field pattern
            Partial indexes per document type
            Schema validation per type using JSON Schema
Builds on:  Project 7 (flexible schema) + Project 11 (partial indexes)

Project 19: Subset Pattern — Reducing Working Set

Problem:    Product page shows only 10 most recent reviews but product has 50,000
Redshift:   Query with LIMIT (columnar scan still touches all rows in sort order)
MongoDB:    Embed top-10 reviews in product document, full reviews in separate collection
Teaches:    Working set concept in WiredTiger cache
            Why document size directly affects RAM pressure
            Subset = keep hot data embedded, cold data referenced
Builds on:  Project 14 (embed vs reference gets nuanced)

Project 20: Computed Pattern — Pre-Aggregating Results

Problem:    Dashboard shows real-time product stats: view count, avg rating, purchase count
Redshift:   Materialized view refreshed periodically
MongoDB:    Store computed fields, update with $inc on each event
Teaches:    Pre-computation vs query-time computation trade-off
            $inc atomic updates for counters
            MongoDB approach: write-time compute vs Redshift approach: read-time compute
Builds on:  Project 4 ($inc updates) + Project 12 (aggregation)

Chapter 7 — Sharding vs Distribution — Scaling Horizontally

Project 21: Sharding Fundamentals — Distribution Keys Reimagined

Problem:    Order collection hits 2TB, single server can't handle load
Redshift:   Choose distribution key (DISTKEY) → data spread across slices
MongoDB:    Choose shard key → data spread across shards
Teaches:    Shard key = distribution key (same concept, different mechanics)
            Chunks, balancer, config servers, mongos router
            Range vs Hashed sharding vs Redshift KEY/EVEN/ALL distribution
Builds on:  Project 15 (scaling awareness) → now implementing it

Project 22: Choosing the Right Shard Key — The Hardest Decision

Problem:    Multi-tenant SaaS: shard by tenant_id? by date? compound key?
Redshift:   DISTKEY on tenant_id → co-locates tenant data
MongoDB:    Shard key on { tenant_id: 1, _id: 1 } → balanced + targeted queries
Teaches:    Cardinality, frequency, monotonicity — the three shard key factors
            Jumbo chunks problem (low cardinality trap)
            Scatter-gather queries (like Redshift broadcast joins)
            WHY shard key is immutable (biggest difference from Redshift DISTKEY)
Builds on:  Project 21 (fundamentals → now strategic decisions)

Chapter 8 — Replica Sets vs Redshift High Availability

Project 23: Replica Sets — Multi-AZ Reimagined

Problem:    Production database needs zero-downtime failover
Redshift:   Multi-AZ deployment, automatic failover, RA3 managed storage
MongoDB:    Replica set: Primary + Secondary + Secondary (or Arbiter)
Teaches:    Election process, oplog replication, automatic failover
            Read preferences: primary, secondary, nearest
            Write concern: w:1, w:"majority" — acknowledged writes
            Redshift has NONE of this granularity
Builds on:  Project 21 (sharding) → now replication within shards

Project 24: Read/Write Concern — Consistency Guarantees

Problem:    Financial transaction must be durably written; analytics read can be stale
Redshift:   Always serializable isolation (no choice)
MongoDB:    Tunable: { w: "majority", j: true } for critical writes
            { readPreference: "secondaryPreferred" } for analytics
Teaches:    Tunable consistency (MongoDB) vs fixed consistency (Redshift)
            CAP theorem in practice
            Read concern: local, majority, linearizable, snapshot
Builds on:  Project 23 (replica set → now configuring its behavior)

Chapter 9 — Change Streams vs Streaming Ingestion

Project 25: Change Streams — Real-Time Event Streaming

Problem:    Trigger notification when order status changes to "shipped"
Redshift:   No native equivalent — need Kinesis + Lambda + polling
MongoDB:    Change stream watches collection for insert/update/delete in real-time

            This is MongoDB's killer feature Redshift cannot match

Project 26: MongoDB Triggers (Atlas) — Serverless Reactions

Problem:    When new user signs up, auto-create welcome notification + audit log

            Database triggers vs scheduled triggers vs auth triggers
            Serverless functions tied to data events
Builds on:  Project 25 (change streams → triggers are built ON change streams)

Chapter 10 — Transactions — ACID in Document World

Project 27: Single-Document Transactions (Implicit ACID)

Problem:    Transfer items between cart and wishlist atomically
Redshift:   BEGIN; UPDATE cart; UPDATE wishlist; COMMIT;
MongoDB:    If modeled in ONE document: single update is automatically atomic
Teaches:    Single-document atomicity = free ACID (no transaction needed)
            THIS is why document design matters so much
            Redshift always needs explicit transactions for multi-table ops
Builds on:  Project 14 (embedding decisions have ACID consequences)

Project 28: Multi-Document Transactions

Problem:    Bank transfer: debit Account A, credit Account B — must be atomic
Redshift:   Standard SQL transaction: BEGIN → UPDATE → UPDATE → COMMIT
MongoDB:    session.startTransaction() → operations → session.commitTransaction()
Teaches:    Multi-document ACID (added MongoDB 4.0)
            Performance cost: transactions lock documents
            Why MongoDB says "design to avoid needing transactions"
            Redshift users over-use MongoDB transactions (anti-pattern)
Builds on:  Project 27 (single → multi-document transaction progression)
