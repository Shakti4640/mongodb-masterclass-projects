PROJECT 36: Atlas Data Federation vs Redshift Spectrum

Querying Data Across Boundaries Without Moving It

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    ShopStream's data now lives in THREE places (built in Projects 34-35):
    
    PLACE 1 — MONGODB ATLAS (operational)
        → Current orders, active carts, user profiles, inventory
        → 200GB of hot operational data
        → Sub-millisecond access for the application
    
    PLACE 2 — AMAZON REDSHIFT (analytical)
        → Historical orders, aggregated metrics, ML scores
        → 2TB of analytical data
        → Complex JOINs and aggregations for BI dashboards
    
    PLACE 3 — AMAZON S3 (data lake)
        → Raw event logs (clickstream, API logs, error logs)
        → Archived historical data (orders older than 2 years)
        → Third-party data feeds (marketing, weather, census)
        → Parquet, JSON, CSV files — 10TB and growing
        → Nobody wants to load ALL of this into Redshift or MongoDB
    
    NEW BUSINESS REQUIREMENTS:
    
    REQUIREMENT 1: "Query archived orders alongside current orders"
        → Analyst wants: "Revenue trend from 2020 to today"
        → 2020-2022 data: archived in S3 (Parquet)
        → 2023-2024 data: live in MongoDB Atlas
        → Currently: must query TWO systems and merge in Excel
    
    REQUIREMENT 2: "Enrich MongoDB data with S3 data without loading it"
        → Product catalog in MongoDB needs weather-adjusted demand forecasting
        → Weather data: 50GB CSV files in S3 (updated daily by vendor)
        → Don't want to import 50GB into MongoDB (waste of storage + working set)
        → Just query it when needed
    
    REQUIREMENT 3: "Run ad-hoc analytics across all data sources"
        → Data scientist needs: clickstream (S3) + orders (MongoDB) + ML scores (Redshift)
        → Building a pipeline for every ad-hoc query is too slow
        → Need: "just query it, wherever it lives"
    
    THE QUESTION:
        Can you query data WITHOUT moving it first?
    
    THE ANSWER:
        YES — this is what federated query engines do:
        → Redshift Spectrum: queries S3 from Redshift
        → Atlas Data Federation: queries S3 + Atlas + HTTP from MongoDB
        → Both: "virtual" data integration (no physical data movement)
    
    CONTRAST WITH PROJECTS 34-35:
        Projects 34-35: PHYSICAL integration (move data between systems)
        Project 36: VIRTUAL integration (query data where it lives)
        Both are valid — different trade-offs — often used TOGETHER

-- Step 1: Create external schema pointing to Glue Data Catalog
CREATE EXTERNAL SCHEMA spectrum
FROM DATA CATALOG
DATABASE 'shopstream_datalake'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftSpectrumRole'
REGION 'us-east-1';

-- Step 2: External table for archived orders in S3 (Parquet)
CREATE EXTERNAL TABLE spectrum.archived_orders (
    order_id        BIGINT,
    order_number    VARCHAR(50),
    customer_id     BIGINT,
    status          VARCHAR(20),
    total           DECIMAL(12,2),
    payment_method  VARCHAR(50),
    source_channel  VARCHAR(20),
    campaign_id     VARCHAR(100),
    created_at      TIMESTAMP
)
PARTITIONED BY (order_year INT, order_month INT)
STORED AS PARQUET
LOCATION 's3://shopstream-datalake/archived_orders/';

-- Step 3: Add partitions (or use ALTER TABLE ... ADD PARTITION)
ALTER TABLE spectrum.archived_orders ADD
PARTITION (order_year=2020, order_month=1)
LOCATION 's3://shopstream-datalake/archived_orders/year=2020/month=01/';
-- (repeat for each partition, or use MSCK REPAIR TABLE in Glue)

-- Step 4: External table for clickstream logs
CREATE EXTERNAL TABLE spectrum.clickstream (
    event_id        VARCHAR(36),
    user_id         BIGINT,
    session_id      VARCHAR(36),
    event_type      VARCHAR(50),
    page_url        VARCHAR(500),
    referrer        VARCHAR(500),
    device_type     VARCHAR(20),
    event_timestamp TIMESTAMP
)
PARTITIONED BY (event_date DATE)
STORED AS PARQUET
LOCATION 's3://shopstream-datalake/clickstream/';

-- Step 5: External table for weather data
CREATE EXTERNAL TABLE spectrum.weather_daily (
    date            DATE,
    region          VARCHAR(50),
    state           VARCHAR(2),
    temp_high_f     DECIMAL(5,1),
    temp_low_f      DECIMAL(5,1),
    precipitation   DECIMAL(5,2),
    weather_type    VARCHAR(20)
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ','
STORED AS TEXTFILE
LOCATION 's3://shopstream-datalake/weather/';

-- Step 6: QUERY — join internal + external seamlessly
-- "Revenue by month: current (internal) + archived (Spectrum)"
SELECT
    DATE_TRUNC('month', created_at) AS month,
    SUM(total) AS revenue,
    COUNT(*) AS orders,
    'current' AS source
FROM analytics.fact_orders
WHERE created_at >= '2023-01-01'
GROUP BY 1

UNION ALL

SELECT
    DATE_TRUNC('month', created_at) AS month,
    SUM(total) AS revenue,
    COUNT(*) AS orders,
    'archived' AS source
FROM spectrum.archived_orders
WHERE order_year BETWEEN 2020 AND 2022
GROUP BY 1

ORDER BY month;

-- Step 7: JOIN internal with external
-- "Weather-adjusted sales analysis"
SELECT
    w.state,
    w.weather_type,
    COUNT(DISTINCT fo.order_id) AS orders,
    SUM(fo.total) AS revenue,
    AVG(w.temp_high_f) AS avg_temp
FROM analytics.fact_orders fo
JOIN analytics.dim_customers dc ON fo.customer_id = dc.customer_id
JOIN spectrum.weather_daily w
    ON DATE(fo.created_at) = w.date
    AND dc.region = w.region
WHERE fo.created_at >= '2024-01-01'
GROUP BY 1, 2
ORDER BY revenue DESC;

-- WHAT HAPPENED INTERNALLY:
-- → Redshift leader node planned the query
-- → Internal tables: scanned by compute slices (local storage)
-- → External tables: Spectrum layer launched thousands of Lambda-like
--   workers to read S3 files in parallel
-- → Results merged in Redshift for final aggregation
-- → User sees ONE result set from TWO storage systems
-- → S3 data was NEVER loaded into Redshift

Spectrum Strengths and Limitations (What You Know)

REDSHIFT SPECTRUM STRENGTHS:
    ✅ Seamless SQL: same SQL for internal + external tables
    ✅ Predicate pushdown: filters applied at S3 read level
    ✅ Partition pruning: only reads relevant S3 partitions
    ✅ Format support: Parquet, ORC, JSON, CSV, Avro, Ion
    ✅ Joins: external tables can JOIN with internal tables
    ✅ Glue integration: shared catalog with Athena, EMR, Spark
    ✅ Scalable: reads from S3 scale independently of cluster size
    ✅ No data movement: query 10TB on S3 without loading

REDSHIFT SPECTRUM LIMITATIONS:
    ❌ SQL only: must define schema as external table DDL
    ❌ Read-only: cannot INSERT/UPDATE/DELETE external tables
    ❌ No MongoDB: cannot query MongoDB as external source
    ❌ Latency: slower than internal tables (S3 read overhead)
    ❌ Cost: $5 per TB scanned on S3 (per query)
    ❌ Schema rigidity: external tables have fixed columns
    ❌ No real-time: S3 data must be written before querying
    ❌ AWS-only: tied to S3 and Glue Data Catalog

THE CRITICAL LIMITATION FOR OUR SCENARIO:
    → Spectrum CANNOT query MongoDB Atlas
    → Current orders (MongoDB) + Archived orders (S3) = two separate queries
    → Must build a UNION in application code or use Redshift + pipeline
    → Atlas Data Federation solves this gap

🧩 HOW TO APPROACH IT IN MONGODB

THINKING FRAMEWORK — VIRTUAL vs PHYSICAL INTEGRATION:

PHYSICAL INTEGRATION (Projects 34-35):
    → Move data from System A to System B
    → Data exists in BOTH places (copy)
    → Queries are LOCAL (fast)
    → Pipeline maintains sync (operational cost)
    → Best for: frequent, performance-critical queries
    → Trade-off: data duplication, pipeline complexity, staleness

VIRTUAL INTEGRATION (Project 36):
    → Query data WHERE IT LIVES
    → Data exists in ONE place (no copy)
    → Queries are REMOTE (slower)
    → No pipeline needed (simpler architecture)
    → Best for: ad-hoc, infrequent, exploratory queries
    → Trade-off: query latency, network dependency, cost per query

WHEN TO USE WHICH:
    ┌──────────────────────────────────┬────────────┬────────────┐
    │ SCENARIO                          │ PHYSICAL    │ VIRTUAL     │
    ├──────────────────────────────────┼────────────┼────────────┤
    │ Dashboard refreshed every 5min    │ ✅          │ ❌ too slow │
    │ Ad-hoc analyst exploration        │ ❌ overkill │ ✅          │
    │ Application real-time query       │ ✅          │ ❌ too slow │
    │ Monthly compliance report         │ ❌ overkill │ ✅          │
    │ ML training data preparation      │ ❌ overkill │ ✅          │
    │ Data scientist notebook query     │ ❌ overkill │ ✅          │
    │ Production API serving            │ ✅          │ ❌ too slow │
    │ Cross-source one-time analysis    │ ❌ overkill │ ✅          │
    └──────────────────────────────────┴────────────┴────────────┘
    
    RULE OF THUMB:
    → Queried more than 10x/day on same data? → PHYSICAL (Project 34-35)
    → Queried less than 10x/day or ad-hoc?    → VIRTUAL (Project 36)
    → Need sub-second latency?                → PHYSICAL
    → Can tolerate seconds-to-minutes?         → VIRTUAL

⚡ WHAT COULD BE DONE (Federation-Specific Optimizations)

OPTIMIZATION 1: PARTITION PRUNING ON S3 DATA
    → Structure S3 paths with partition keys in the path:
      s3://bucket/orders/year=2024/month=05/day=31/file.parquet
    → Federation config parses path components as fields
    → Queries filtering on year/month/day SKIP entire folders
    → Like Redshift Spectrum partition pruning
    → 100x fewer files read = 100x faster + cheaper

OPTIMIZATION 2: USE PARQUET FOR S3 DATA (NOT JSON/CSV)
    → Parquet: columnar, compressed, typed, supports predicate pushdown
    → JSON: row-based, larger, requires full parsing
    → CSV: no types, no compression, no predicate pushdown
    → Parquet on S3 + Federation = closest to Redshift Spectrum performance
    → Use $out with format "parquet" when writing from Federation to S3

OPTIMIZATION 3: FILTER EARLY IN PIPELINE ($match FIRST)
    → Federation pushes $match predicates to source
    → $match on partitioned fields → partition pruning
    → $match on indexed fields (Atlas source) → index scan
    → If $match is NOT first stage → full scan of source → SLOW + EXPENSIVE
    → Same principle as Redshift: WHERE clause enables zone map/partition pruning

OPTIMIZATION 4: PROJECT ONLY NEEDED FIELDS
    → $project early → Federation requests fewer columns from Parquet
    → Parquet column pruning: reads only requested columns
    → Reduces data transfer from S3 → faster + cheaper
    → Same concept as Redshift: SELECT col1, col2 vs SELECT *

OPTIMIZATION 5: USE $merge FOR INCREMENTAL MATERIALIZATION
    → Instead of re-querying S3 every time for same data
    → Run federation query once → $merge results into Atlas collection
    → Subsequent queries read from Atlas (fast, indexed)
    → Re-run $merge daily to refresh
    → Like Redshift Materialized View but manual

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT USE FEDERATION FOR PRODUCTION APPLICATION QUERIES
    WRONG:
    → Application API calls federation for every page load
    → "Get user's order history" → federation query across Atlas + S3
    
    WHY:
    → Federation latency: 100ms-30s (depending on S3 data volume)
    → Application needs: <10ms
    → Federation is for analytics/exploration, not serving APIs
    
    RIGHT:
    → Application queries Atlas directly for current data
    → Federation for analyst ad-hoc queries only

❌ DO NOT SKIP PARTITION PRUNING
    WRONG:
    → db.archived_orders.aggregate([
        { $match: { "customer.email": "sarah@email.com" } }
      ])
    → This scans ALL S3 files (no partition filter)
    
    WHY:
    → S3 data is partitioned by year/month
    → Without year/month filter, reads ALL partitions
    → 10TB scan for one customer = slow + expensive
    
    RIGHT:
    → db.archived_orders.aggregate([
        { $match: { year: 2021, month: 6, "customer.email": "sarah@email.com" } }
      ])
    → Reads only June 2021 partition

❌ DO NOT TREAT FEDERATION AS A REPLACEMENT FOR REDSHIFT
    WRONG:
    → "We have Atlas Data Federation — we don't need Redshift anymore"
    → Running complex multi-JOIN analytics on federation
    
    WHY:
    → Federation is not an MPP engine
    → No distribution keys, no zone maps, no parallel hash joins
    → Complex analytical queries: 10-100x slower than Redshift
    → Federation reads from S3 serially (not MPP parallel)
    
    RIGHT:
    → Federation for ad-hoc, simple queries
    → Redshift for complex, repeated analytical workloads

❌ DO NOT STORE HOT OPERATIONAL DATA ONLY IN S3
    WRONG:
    → "S3 is cheap — let's put everything in S3 and query via federation"
    → Application reads from S3 via federation for real-time data
    
    WHY:
    → S3 read latency: 50-200ms per file
    → No indexes on S3 (every query = scan)
    → Not designed for point lookups
    
    RIGHT:
    → Hot data in Atlas (indexed, fast, real-time)
    → Warm/cold data in S3 (archived, federated access)
    → Federation unifies access across temperature tiers

❌ DO NOT WRITE SMALL FILES TO S3 VIA $out
    WRONG:
    → Running $out every minute, creating thousands of tiny files
    
    WHY:
    → S3 list operations slow with millions of small files
    → Each file has overhead (HTTP request per file)
    → Parquet files < 10MB don't compress efficiently
    
    RIGHT:
    → Batch $out operations (daily or hourly)
    → Target file size: 100-500MB
    → Use maxFileSize option in $out

WHEN YOU RUN: db.all_orders.aggregate([...])
ON A VIRTUAL COLLECTION SPANNING ATLAS + S3:

STEP 1: QUERY PARSING
    → Federation engine receives MQL query
    → Identifies target: "all_orders" virtual collection
    → Looks up storage config: 2 data sources (Atlas + S3)

STEP 2: QUERY PLANNING
    → Analyzes $match predicates
    → For Atlas source:
      → Can push $match to Atlas query engine
      → Atlas uses indexes (B-tree) for efficient filtering
    → For S3 source:
      → Can push partition filters (year/month) → skip folders
      → Can push column filters (Parquet) → skip columns
      → CANNOT push complex predicates to S3 (no query engine)

STEP 3: PARALLEL EXECUTION
    → Sub-query A sent to Atlas cluster
      → Atlas mongod processes query using indexes
      → Returns matching documents over network
    → Sub-query B sent to S3 reader
      → Lists S3 objects matching partition prefix
      → Downloads relevant Parquet files
      → Parses Parquet, applies remaining filters
      → Returns matching documents

STEP 4: MERGE
    → Results from both sources merged
    → Remaining pipeline stages ($group, $sort, $project) applied
    → Final result returned to client

STEP 5: COST CALCULATION
    → Data processed from Atlas: based on Atlas cluster usage
    → Data processed from S3: metered per bytes read
    → Federation compute: metered per processing time

WHEN YOU RUN: SELECT ... FROM internal JOIN spectrum.external ...

STEP 1: LEADER NODE parses SQL
STEP 2: LEADER NODE creates execution plan
    → Internal table: assigned to compute slices
    → External table: assigned to Spectrum layer

    → Compute slices read internal tables (local SSD/managed storage)
    → Spectrum layer launches thousands of workers to read S3

STEP 4: JOIN EXECUTION
    → Spectrum results streamed to compute slices
    → Hash join between internal and external results
    → MPP parallelism for aggregation
STEP 5: RESULT ASSEMBLY
    → Leader node collects results from all slices
    → Final sort/limit applied
    → Result returned to client

NEW TERMS IN THIS PROJECT:

    Virtual Collection:
        → A collection that doesn't store data itself
        → Routes queries to underlying data sources (Atlas, S3, HTTP)
        → Like a Redshift external table but more flexible

    Federated Query:
        → A query that spans multiple data sources
        → Engine handles routing, execution, merging transparently
        → User writes ONE query, engine queries MANY sources

    Partition Pruning:
        → Skipping entire S3 folders based on query filters
        → S3 path: year=2024/month=05/ → filter year=2024 → skip all other years
        → Same concept in Spectrum: partition elimination

    Column Pruning (Parquet):
        → Reading only requested columns from Parquet files
        → $project: { total: 1, status: 1 } → reads 2 columns, skips 20 others
        → Same concept in Spectrum: columnar read optimization

    Data Temperature:
        → Hot: frequently accessed, low latency needed → MongoDB Atlas
        → Warm: occasionally accessed, moderate latency OK → S3 Parquet
        → Cold: rarely accessed, high latency OK → S3 Glacier
        → Federation unifies access across all temperatures

    $out to S3:
        → Pipeline stage that writes results as files to S3
        → Supports Parquet, JSON, BSON, CSV formats
        → Redshift equivalent: UNLOAD command
        → Federation unique advantage: part of the query pipeline

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "FEDERATION IS SLOW — I'LL JUST USE REDSHIFT"
    CONTEXT: Analyst runs complex aggregation via federation → takes 2 minutes
             Same query on Redshift Spectrum → takes 10 seconds
    
    WHY IT'S NOT A FAIR COMPARISON:
    → Spectrum has MPP (massively parallel processing) behind it
    → Federation is serverless (no dedicated compute)
    → Spectrum costs: $0.25/hr × cluster nodes + $5/TB scanned
    → Federation costs: only data processed (no cluster)
    
    ✅ RIGHT APPROACH:
    → Use federation for ad-hoc/infrequent queries (cost-efficient)
    → Use Spectrum for repeated/production queries (performance-efficient)
    → If a federation query becomes frequent → materialize into Atlas or Redshift

PITFALL 2: "QUERYING ALL S3 DATA WITHOUT PARTITION FILTERS"
    ❌ db.archived_orders.find({ status: "shipped" })
    → Reads ALL archived orders across ALL years
    → Potentially scanning 8TB of Parquet files
    → Slow AND expensive
    
    ✅ db.archived_orders.find({ year: 2023, month: 6, status: "shipped" })
    → Reads only June 2023 partition
    → Maybe 50GB instead of 8TB
    → 160x less data = 160x faster + cheaper

PITFALL 3: "EXPECTING REAL-TIME S3 DATA"
    ❌ Writing file to S3 → immediately querying via federation
    → S3 eventual consistency can cause file not found
    → Parquet files take seconds to write and become queryable
    
    ✅ FIX:
    → Wait for S3 write confirmation before querying
    → For real-time data: query Atlas source (always current)
    → S3 source is for batch/historical data (seconds to minutes lag)

PITFALL 4: "USING FEDERATION FOR ETL INSTEAD OF AD-HOC"
    ❌ Scheduled job runs federation query every 5 minutes to "sync" data
    → Expensive: paying per-query charges 288 times/day
    → Unreliable: federation is serverless, cold starts happen
    → Wrong tool: this is a pipeline job (Project 34)
    
    ✅ FIX:
    → ETL/CDC: Use Change Streams → Kafka → COPY (Project 34)
    → Federation: Ad-hoc queries, one-time exports, exploration

PITFALL 5: "COMPLEX $lookup ACROSS LARGE S3 DATASETS"
    ❌ db.current_orders.aggregate([
        { $lookup: {
            from: "archived_orders",  // 8TB on S3
            localField: "customer.customer_id",
            foreignField: "customer.customer_id",
            as: "historical_orders"
        }}
    ])
    → For EACH current order, scans ALL archived orders on S3
    → Nested loop join × 8TB = never finishes
    
    ✅ FIX:
    → Add $match before $lookup to limit left side
    → Add pipeline inside $lookup with $match to limit right side
    → Use partition filters inside $lookup pipeline
    → Or: materialize the join result with $merge
    → For large JOINs: use Redshift (hash join with MPP)

PITFALL 6: "S3 FILE FORMAT MISMATCH"
    ❌ Config says defaultFormat: "parquet" but S3 contains JSON files
    → Federation tries to parse JSON as Parquet → cryptic errors
    
    ✅ FIX:
    → Verify file format matches configuration
    → Use consistent format per S3 prefix
    → Parquet for analytical data (best performance)
    → JSON for semi-structured logs (flexible schema)
    → Don't mix formats in same S3 prefix

PITFALL 7: "ASSUMING MONGODB INDEXES WORK ON S3 DATA"
    ❌ Creating index on virtual collection that includes S3 source
    → Indexes only apply to Atlas-sourced data
    → S3 data has no indexes (file-based storage)
    → $match on S3 data = file scan (partition pruning only optimization)
    
    ✅ RIGHT UNDERSTANDING:
    → Atlas source: indexes work (fast point lookups)
    → S3 source: partition + column pruning only (scan-based)
    → Design queries accordingly

    → Best for analytical queries via federation
    → Column pruning: read only needed columns
    → Predicate pushdown: filter at read time

    → Typed: no parsing overhead

    → Best for semi-structured data via federation
    → Flexible schema: each line can differ
    → No column pruning: reads entire document
    → Larger: no compression by default
    → USE FOR: event logs, API responses, raw data
    
    CSV (tabular — like Redshift rows on S3):
    → Simplest format, widely compatible
    → No types: everything is string
    → No compression by default
    → No column pruning: reads entire row
    → USE FOR: simple reference data, vendor feeds

    count = atlas_db.orders.count_documents({
        "created_at": {"$lt": cutoff},
        "status": {"$in": ["delivered", "cancelled", "refunded"]},
        "_archived": {"$ne": True}
    })
    
    logger.info(f"Found {count:,} orders eligible for archival")
    logger.info(f"Cutoff date: {cutoff.date()}")
    
    return cutoff, count

# ─── STEP 2: ARCHIVE TO S3 AS PARQUET ────────────────

def archive_to_s3(atlas_db, cutoff):
    """
    Export old orders from MongoDB to S3 as Parquet
    
    WHY PARQUET:
    → Columnar format: efficient for analytical queries
    → Compressed: 5-10x smaller than JSON
    → Typed: preserves data types
    → Queryable by: Federation, Spectrum, Athena, Spark
    → (Same reasons you'd use Parquet in any data lake)
    
    PROCESS:
    → Read documents from MongoDB in monthly batches
    → Flatten documents to tabular format (for Parquet)
    → Write Parquet files to S3 with partition structure
    """
    
    s3_client = boto3.client("s3")
    
    # Process month by month
    current = cutoff - timedelta(days=365)  # go back 1 year from cutoff
    total_archived = 0
    
    while current < cutoff:
        month_end = min(
            datetime(current.year, current.month + 1, 1) if current.month < 12
            else datetime(current.year + 1, 1, 1),
            cutoff
        )
        
        logger.info(f"\nArchiving: {current.date()} → {month_end.date()}")
        
        # Query MongoDB for this month's archivable orders
        cursor = atlas_db.orders.find(
            {
                "created_at": {"$gte": current, "$lt": month_end},
                "status": {"$in": ["delivered", "cancelled", "refunded"]},
                "_archived": {"$ne": True}
            }
        ).batch_size(5000)
        
        # Flatten documents to tabular rows for Parquet
        rows = []
        doc_ids = []
        
        for doc in cursor:
            doc_ids.append(doc["_id"])
            
            customer = doc.get("customer", {})
            payment = doc.get("payment", {})
            shipping = doc.get("shipping", {})
            address = shipping.get("address", {})
            
            # Flatten: nested document → flat row
            # This is the same flattening as Project 34 transformer
            row = {
                "order_id": str(doc["_id"]),
                "order_number": doc.get("order_number", ""),
                "status": doc.get("status", ""),
                "customer_id": str(customer.get("customer_id",
                                   customer.get("_legacy_customer_id", ""))),
                "customer_name": f"{customer.get('first_name', '')} {customer.get('last_name', customer.get('name', ''))}".strip(),
                "customer_segment": customer.get("segment", ""),
                "customer_email": customer.get("email", ""),
                "item_count": doc.get("item_count", 0),
                "payment_subtotal": float(str(payment.get("subtotal", 0))),
                "payment_shipping": float(str(payment.get("shipping", 0))),
                "payment_tax": float(str(payment.get("tax", 0))),
                "payment_total": float(str(payment.get("total", 0))),
                "payment_method": payment.get("method", ""),
                "source_channel": doc.get("source", ""),
                "campaign_id": doc.get("marketing_campaign", ""),
                "shipping_state": address.get("state", ""),
                "shipping_country": address.get("country", ""),
                "created_at": doc.get("created_at"),
                "updated_at": doc.get("updated_at")
            }
            rows.append(row)
            
            # Also flatten items into separate rows (for items archive)
            # Skipped for brevity — same pattern as Project 34
        
        if not rows:
            logger.info(f"  No documents for this period")
            current = month_end
            continue
        
        # Convert to PyArrow Table
        schema = pa.schema([
            ("order_id", pa.string()),
            ("order_number", pa.string()),
            ("status", pa.string()),
            ("customer_id", pa.string()),
            ("customer_name", pa.string()),
            ("customer_segment", pa.string()),
            ("customer_email", pa.string()),
            ("item_count", pa.int32()),
            ("payment_subtotal", pa.float64()),
            ("payment_shipping", pa.float64()),
            ("payment_tax", pa.float64()),
            ("payment_total", pa.float64()),
            ("payment_method", pa.string()),
            ("source_channel", pa.string()),
            ("campaign_id", pa.string()),
            ("shipping_state", pa.string()),
            ("shipping_country", pa.string()),
            ("created_at", pa.timestamp("ms")),
            ("updated_at", pa.timestamp("ms"))
        ])
        
        # Build columnar arrays
        arrays = [pa.array([r[field.name] for r in rows], type=field.type)
                  for field in schema]
        table = pa.table(arrays, schema=schema)
        
        # Write to S3 as Parquet with partition path
        year = current.year
        month = current.month
        s3_key = (
            f"{ARCHIVE_PREFIX}/"
            f"year={year}/month={month:02d}/"
            f"orders_{year}_{month:02d}.parquet"
        )
        
        # Write Parquet to buffer then upload to S3
        buf = io.BytesIO()
        pq.write_table(table, buf, compression="snappy")
        buf.seek(0)
        
        s3_client.put_object(
            Bucket=S3_BUCKET,
            Key=s3_key,
            Body=buf.getvalue()
        )
        
        logger.info(f"  Archived {len(rows):,} orders → s3://{S3_BUCKET}/{s3_key}")
        logger.info(f"  Parquet size: {buf.tell():,} bytes")
        
        # Mark documents as archived in MongoDB
        # (so we don't archive them again on re-run)
        atlas_db.orders.update_many(
            {"_id": {"$in": doc_ids}},
            {"$set": {"_archived": True, "_archived_at": datetime.utcnow()}}
        )
        
        total_archived += len(rows)
        current = month_end
    
    logger.info(f"\nTotal archived: {total_archived:,} orders")
    return total_archived

# ─── STEP 3: DELETE ARCHIVED DOCUMENTS ────────────────

def cleanup_archived_documents(atlas_db, grace_period_days=7):
    """
    Delete archived documents from MongoDB to free working set
    
    WHY DELETE:
    → Working set pressure (Project 33)
    → Archived data in MongoDB wastes WiredTiger cache
    → Data is safely in S3 Parquet now
    → Federation provides access when needed
    
    SAFETY:
    → Only delete documents marked _archived: true
    → Wait grace_period_days after archival (safety buffer)
    → Validate S3 files exist before deleting
    → This is IRREVERSIBLE — archived data must be in S3
    """
    
    grace_cutoff = datetime.utcnow() - timedelta(days=grace_period_days)
    
    # Count what we'll delete
    to_delete = atlas_db.orders.count_documents({
        "_archived": True,
        "_archived_at": {"$lt": grace_cutoff}
    })
    
    logger.info(f"\nDocuments eligible for cleanup: {to_delete:,}")
    logger.info(f"Grace period: {grace_period_days} days")
    logger.info(f"(Archived before: {grace_cutoff.date()})")
    
    if to_delete == 0:
        logger.info("Nothing to clean up")
        return 0
    
    # Delete in batches
    batch_size = 10000
    total_deleted = 0
    
    while True:
        result = atlas_db.orders.delete_many(
            {
                "_archived": True,
                "_archived_at": {"$lt": grace_cutoff}
            }
        )
        
        if result.deleted_count == 0:
            break
        
        total_deleted += result.deleted_count
        logger.info(f"  Deleted batch: {result.deleted_count:,} (total: {total_deleted:,})")
    
    logger.info(f"\nTotal cleaned up: {total_deleted:,} documents")
    logger.info(f"Working set freed: ~{total_deleted * 500 / 1024 / 1024:.1f} MB estimated")
    
    return total_deleted

def run_data_lifecycle():
    """
    Complete data lifecycle:
    1. Identify old data in MongoDB
    2. Archive to S3 as Parquet
    3. Delete from MongoDB (free working set)
    4. Verify federation access (unified queries work)
    
    SCHEDULE: Run nightly via cron/Airflow/Lambda
    """
    
    logger.info(f"{'='*60}")
    logger.info(f"DATA LIFECYCLE MANAGER")
    logger.info(f"Timestamp: {datetime.utcnow()}")
    logger.info(f"{'='*60}")
    
    atlas_client = MongoClient(ATLAS_URI)
    atlas_db = atlas_client["shopstream"]
    
    # Step 1: Identify
    cutoff, count = identify_archivable_orders(atlas_db)
    
    if count == 0:
        logger.info("No orders to archive. Done.")
        return
    
    # Step 2: Archive to S3
    archived = archive_to_s3(atlas_db, cutoff)
    
    # Step 3: Cleanup (with grace period)
    deleted = cleanup_archived_documents(atlas_db, grace_period_days=7)
    
    # Step 4: Verify federation
    verify_federation_access()
    
    logger.info(f"\n{'='*60}")
    logger.info(f"LIFECYCLE COMPLETE")
    logger.info(f"  Archived: {archived:,} orders to S3")
    logger.info(f"  Cleaned:  {deleted:,} documents from MongoDB")
    logger.info(f"  Federation: verified ✅")
    logger.info(f"{'='*60}")
    
    atlas_client.close()


if __name__ == "__main__":
    run_data_lifecycle()
