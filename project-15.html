PROJECT 15: OLTP vs OLAP — Why MongoDB and Redshift Exist Differently

Problem: E-Commerce Platform Needing Real-Time Operations AND End-of-Day Analytics

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    You're the lead data engineer at a growing e-commerce company
    
    The CEO asks TWO things in the same meeting:
    
    REQUEST 1 (Operations Team):
    "Our checkout page takes 3 seconds to load cart data.
     Users are abandoning carts. We need sub-50ms cart operations.
     Black Friday is coming — we'll have 50,000 concurrent users.
     Every millisecond of latency costs us $100K in lost revenue."
    
    REQUEST 2 (Analytics Team):
    "We need a dashboard showing:
     → Revenue by category by region for the last 2 years
     → Customer cohort analysis: retention by signup month

    YOUR REDSHIFT BRAIN SAYS:
    "Put everything in Redshift — it handles SQL, it handles scale"
    
    THE REALITY:
    → Request 1 needs OLTP (Online Transaction Processing)
    → Request 2 needs OLAP (Online Analytical Processing)
    → NO SINGLE DATABASE EXCELS AT BOTH
    → This is WHY MongoDB and Redshift both exist
    → This project explains the FUNDAMENTAL REASON

-- View cart (user clicks cart icon)

    ci.product_id,

WHERE c.customer_id = 98765
  AND c.status = 'active';

-- → Leader node parses query, creates plan
-- → Sends to compute nodes
-- → 2 JOINs across distributed data
-- → Even with DISTSTYLE ALL on products: 50-500ms
-- → MPP overhead for a SINGLE USER's cart = massive overkill
-- → Like using a freight train to deliver a letter

-- Add item to cart

UPDATE ecom.carts SET updated_at = GETDATE() WHERE cart_id = 1234;

-- REQUEST 2: Analytics Queries (OLAP needs)

-- Revenue by category by region (last 2 years)
SELECT 
    p.category,
    c.region,
    DATE_TRUNC('month', o.created_at) AS month,
    COUNT(DISTINCT o.order_id) AS order_count,
    SUM(oi.subtotal) AS revenue,
    AVG(o.total_amount) AS avg_order_value
FROM ecom.orders o
JOIN ecom.order_items oi ON o.order_id = oi.order_id
JOIN ecom.products p ON oi.product_id = p.product_id
JOIN ecom.customers c ON o.customer_id = c.customer_id
WHERE o.created_at >= DATEADD(year, -2, GETDATE())
  AND o.status = 'completed'
GROUP BY p.category, c.region, DATE_TRUNC('month', o.created_at)
ORDER BY month DESC, revenue DESC;

-- REDSHIFT PERFORMANCE FOR THIS QUERY:
-- → Scans 500M order rows (columnar: reads only needed columns)
-- → Zone maps skip blocks outside date range
-- → Parallel execution across all compute nodes
-- → Hash JOINs on co-located data (DISTKEY optimized)
-- → Result: 3-15 seconds for 500M rows
-- → THIS IS WHAT REDSHIFT IS BUILT FOR


-- Customer cohort retention
WITH cohorts AS (
    SELECT 
        customer_id,
        DATE_TRUNC('month', MIN(created_at)) AS cohort_month
    FROM ecom.orders
    WHERE status = 'completed'
    GROUP BY customer_id
),
activity AS (
    SELECT 
        c.customer_id,
        c.cohort_month,
        DATE_TRUNC('month', o.created_at) AS activity_month,
        DATEDIFF(month, c.cohort_month, DATE_TRUNC('month', o.created_at)) AS months_since
    FROM cohorts c
    JOIN ecom.orders o ON c.customer_id = o.customer_id
    WHERE o.status = 'completed'
)
SELECT 
    cohort_month,
    months_since,
    COUNT(DISTINCT customer_id) AS active_customers,
    COUNT(DISTINCT customer_id)::FLOAT / 
        FIRST_VALUE(COUNT(DISTINCT customer_id)) OVER (
            PARTITION BY cohort_month ORDER BY months_since
        ) AS retention_rate
FROM activity
GROUP BY cohort_month, months_since
ORDER BY cohort_month, months_since;

-- REDSHIFT PERFORMANCE:
-- → Window functions + CTEs + massive GROUP BY
-- → 5-30 seconds across 500M rows
-- → PERFECT workload for MPP columnar engine
-- → MongoDB would take MINUTES for this query

Why Redshift FAILS at Request 1 (OLTP)

REDSHIFT'S FUNDAMENTAL ARCHITECTURE PREVENTS FAST OLTP:

PROBLEM 1: MPP OVERHEAD
    → Every query goes through Leader Node → Compute Nodes → Leader Node
    → Even "SELECT * FROM carts WHERE customer_id = X" follows this path
    → Minimum latency: 20-50ms (network + parsing + distribution)
    → MongoDB: 1-3ms (direct B-tree lookup, no distribution overhead)
    
    ┌──────────┐     ┌──────────┐     ┌──────────┐
    │  Client   │────→│  Leader  │────→│ Compute  │
    │           │     │  Node    │     │ Node 1   │
    │           │     │          │────→│ Node 2   │
    │           │     │          │     │ Node 3   │
    │           │←────│          │←────│ Node 4   │
    └──────────┘     └──────────┘     └──────────┘
    
    ROUND TRIP: Client → Leader → Compute → Leader → Client
    For ONE user's cart = absurd overhead

PROBLEM 2: CONCURRENCY (WLM)
    → Redshift Workload Management (WLM) has LIMITED concurrent query slots
    → Default: 5 concurrent queries per queue
    → Maximum practical: ~50 concurrent queries
    → Black Friday: 50,000 users × 1 cart query each = 50,000 queries
    → Redshift queues them: most users wait 10+ seconds
    → MongoDB: handles 10,000+ concurrent operations per second per node
    
    CONCURRENCY COMPARISON:
    ┌─────────────────────┬────────────┬────────────────┐
    │ CONCURRENT USERS     │ REDSHIFT    │ MONGODB         │
    ├─────────────────────┼────────────┼────────────────┤
    │ 10                   │ ✅ Fine     │ ✅ Fine          │
    │ 50                   │ ⚠️ Queuing │ ✅ Fine          │
    │ 500                  │ ❌ Slow     │ ✅ Fine          │
    │ 5,000                │ ❌ Failing  │ ✅ Fine          │
    │ 50,000               │ ❌ Dead     │ ⚠️ Add shards   │
    └─────────────────────┴────────────┴────────────────┘

PROBLEM 3: SINGLE-ROW WRITE PENALTY
    → Redshift columnar storage: splits each INSERT into column blocks
    → INSERT one cart item = write to 5+ column files
    → Creates unsorted region → degrades sort key performance
    → Needs VACUUM to re-sort → maintenance overhead
    → MongoDB: insert one document = one write to one location
    
PROBLEM 4: NO REAL-TIME
    → Redshift optimized for batch loads (COPY from S3)
    → Individual INSERTs are anti-pattern
    → "Add item to cart" → individual INSERT → terrible pattern for Redshift
    → MongoDB: individual writes are the PRIMARY use case

BOTTOM LINE:
    Using Redshift for cart operations is like
    using a combine harvester to pick one apple

// ═══════════════════════════════════════════════════════

// ── Revenue by category by region (last 2 years) ──
db.orders.aggregate([
    // Step 1: Filter to completed orders in date range
    { $match: {
        status: "completed",
        created_at: { $gte: ISODate("2022-07-01") }
    }},
    
    // Step 2: Unwind order items (like UNNEST)
    { $unwind: "$items" },
    
    // Step 3: Need customer region → $lookup (JOIN)
    { $lookup: {
        from: "customers",
        localField: "customer_id",
        foreignField: "_id",
        as: "customer"
    }},
    { $unwind: "$customer" },
    
    // Step 4: Group by category + region + month
    { $group: {
        _id: {
            category: "$items.category",
            region: "$customer.region",
            month: { $dateToString: { format: "%Y-%m", date: "$created_at" } }
        },
        order_count: { $addToSet: "$_id" },
        revenue: { $sum: "$items.subtotal" },
        total_amounts: { $push: "$total_amount" }
    }},
    
    // Step 5: Compute derived fields
    { $project: {
        category: "$_id.category",
        region: "$_id.region",
        month: "$_id.month",
        order_count: { $size: "$order_count" },
        revenue: 1,
        avg_order_value: { $avg: "$total_amounts" }
    }},
    
    { $sort: { month: -1, revenue: -1 } }
])

// MONGODB PERFORMANCE FOR THIS QUERY:
// → Must read ALL 500M order documents (document storage = read entire document)
// → $unwind multiplies documents (3 items per order → 1.5B intermediate docs)
// → $lookup on customers = nested loop across 10M customer docs
// → $group in memory → may exceed 100MB RAM limit → spill to disk
// → TOTAL: 60-300 seconds (vs Redshift: 3-15 seconds)
//
// WHY SO SLOW?
// → Document storage: must read ALL fields of every order
//   (even though we only need items.category and created_at)
//   Redshift: reads ONLY the columns needed (columnar)
//
// → No distribution optimization
//   MongoDB aggregation runs on ONE node (primary) unless sharded
//   Even sharded: each shard runs independently, mongos merges
//   Redshift: parallel execution across ALL slices from the start
//
// → $lookup = nested loop join
//   For each of 500M orders → look up customer → 500M lookups
//   Redshift: hash join on co-located data → O(n + m)
//
// → No zone maps equivalent for analytical scans
//   MongoDB B-tree index helps for point lookups, not full scans
//   Redshift zone maps skip entire 1MB blocks → massive I/O savings

Why MongoDB STRUGGLES at Request 2 (OLAP)

MONGODB'S FUNDAMENTAL ARCHITECTURE PREVENTS FAST OLAP:

PROBLEM 1: DOCUMENT STORAGE (not columnar)
    → To compute SUM(items.subtotal), MongoDB must:
      Read ENTIRE document (customer_name, shipping, payment, etc.)
      Then extract just the subtotal field
    → Redshift reads ONLY the subtotal column
    
    DATA READ COMPARISON (500M orders):
    ┌────────────────────────────────┬──────────────┬──────────────┐
    │ QUERY: SUM of subtotals        │ MONGODB       │ REDSHIFT      │
    ├────────────────────────────────┼──────────────┼──────────────┤
    │ Avg document/row size          │ 2KB           │ 2KB (full row)│
    │ Data actually needed           │ 8 bytes       │ 8 bytes       │
    │ Data actually READ from disk   │ 2KB × 500M   │ 8B × 500M    │
    │                                │ = 1 TB        │ = 4 GB        │
    │ I/O RATIO                      │ 250x MORE     │ OPTIMAL       │
    └────────────────────────────────┴──────────────┴──────────────┘
    
    MongoDB reads 250x more data for the same analytical query
    THIS is why columnar storage exists

PROBLEM 2: NO PARALLEL EXECUTION (without sharding)
    → MongoDB aggregation runs on PRIMARY node
    → Single-threaded per pipeline (mostly)
    → Redshift: distributes across 4-128 slices, all work simultaneously
    
    PARALLELISM:
    ┌──────────────────────────┬──────────────┬──────────────┐
    │                           │ MONGODB       │ REDSHIFT      │
    ├──────────────────────────┼──────────────┼──────────────┤
    │ Execution threads         │ 1 (primary)   │ 32-512 slices │
    │ Data scanned in parallel  │ None          │ All slices    │
    │ JOIN parallelism          │ Nested loop   │ Hash join     │
    │                           │ (sequential)  │ (parallel)    │
    └──────────────────────────┴──────────────┴──────────────┘

PROBLEM 3: $lookup vs JOIN (revisited from Project 14)
    → Analytical queries JOIN 4-6 tables
    → Each $lookup in MongoDB = nested loop
    → 500M orders × customer lookup = 500M separate lookups
    → Redshift: hash join with pre-distributed data = one pass

PROBLEM 4: MEMORY LIMITS
    → MongoDB aggregation: 100MB RAM limit per stage (default)
    → allowDiskUse: true → spills to disk → 10x slower
    → Redshift: entire cluster memory available (64GB-16TB+)
    → Intermediate results distributed across all nodes

PROBLEM 5: NO COMPRESSION ADVANTAGE
    → Redshift columnar compression: 3-10x reduction
    → 1TB logical → 100-300GB physical (compressed columns)
    → MongoDB BSON compression (snappy/zstd): 2-4x reduction
    → But still reads entire documents, not just needed fields

BOTTOM LINE:
    Using MongoDB for complex analytics is like
    reading an entire encyclopedia to find one person's birthday

function addToCart(cartId, product) {
    const start = new Date()
    
    // Check if product already in cart
    const cart = db.carts.findOne({
        _id: cartId,
        "items.product_id": product._id
    })
    
    if (cart) {
        // Product exists → increment quantity
        db.carts.updateOne(
            { _id: cartId, "items.product_id": product._id },
            {
                $inc: {
                    "items.$.quantity": 1,
                    "items.$.subtotal": product.price,
                    total: product.price
                },
                $set: { updated_at: new Date() }
            }
        )
    } else {
        // New product → push to items array
        db.carts.updateOne(
            { _id: cartId },
            {
                $push: {
                    items: {
                        product_id: product._id,
                        name: product.name,           // embedded subset
                        price: product.price,
                        image: product.image,
                        category: product.category,
                        quantity: 1,
                        subtotal: product.price,
                        added_at: new Date()
                    }
                },
                $inc: { item_count: 1, total: product.price },
                $set: { updated_at: new Date() }
            }
        )
    }
    
    const elapsed = new Date() - start
    print("  Added " + product.name + " in " + elapsed + "ms")
    return elapsed
}

class OLTPBenchmark:
    """Benchmark OLTP operations with timing"""
    
    def __init__(self, db):
        self.db = db
        self.results = []
    
    def timed(self, operation_name, func, iterations=1):
        """Run and time an operation"""
        start = time.time()
        result = None
        for _ in range(iterations):
            result = func()
        elapsed = (time.time() - start) * 1000  # ms
        avg = elapsed / iterations
        self.results.append({
            "operation": operation_name,
            "iterations": iterations,
            "total_ms": round(elapsed, 2),
            "avg_ms": round(avg, 3)
        })
        return result
    
    def report(self):
        """Print benchmark results"""
        print(f"\n{'='*70}")
        print(f"{'OPERATION':<35} {'ITERS':>6} {'TOTAL':>10} {'AVG':>10}")
        print(f"{'='*70}")
        for r in self.results:
            print(f"{r['operation']:<35} {r['iterations']:>6} "
                  f"{r['total_ms']:>8.1f}ms {r['avg_ms']:>8.3f}ms")
        print(f"{'='*70}")

def add_to_cart(cart_id, product):
    """
    Add product to cart — single atomic operation
    
    Redshift equivalent would be:
        INSERT INTO ecom.cart_items (cart_id, product_id, quantity) VALUES (...);
        UPDATE ecom.carts SET updated_at = GETDATE() WHERE cart_id = ...;
        -- Two separate operations + product name needs JOIN at read time
    """
    existing = db.carts.find_one({
        "_id": cart_id,
        "items.product_id": product["_id"]
    })
    
    if existing:
        # Increment quantity of existing item
        db.carts.update_one(
            {"_id": cart_id, "items.product_id": product["_id"]},
            {
                "$inc": {
                    "items.$.quantity": 1,
                    "items.$.subtotal": product["price"],
                    "total": product["price"]
                },
                "$set": {"updated_at": datetime.now()}
            }
        )
    else:
        # Add new item
        db.carts.update_one(
            {"_id": cart_id},
            {
                "$push": {
                    "items": {
                        "product_id": product["_id"],
                        "name": product["name"],         # embedded — no JOIN needed
                        "price": product["price"],
                        "image": product["image"],
                        "category": product["category"],
                        "quantity": 1,
                        "subtotal": product["price"],
                        "added_at": datetime.now()
                    }
                },
                "$inc": {"item_count": 1, "total": product["price"]},
                "$set": {"updated_at": datetime.now()}
            }
        )

def remove_from_cart():
    """
    Atomic removal of item from embedded array.
    
    Redshift equivalent:
        DELETE FROM ecom.cart_items 
        WHERE cart_id = 1234 AND product_id = 5678;
        -- Creates tombstone row, needs VACUUM

        {"_id": cart_id, "items.product_id": target["_id"]},
        {"items.$": 1}
    )
    
    if current_cart and current_cart.get("items"):
        item = current_cart["items"][0]
        db.carts.update_one(
            {"_id": cart_id},
            {
                "$pull": {"items": {"product_id": target["_id"]}},
                "$inc": {
                    "item_count": -1,
                    "total": -item["subtotal"]
                },
                "$set": {"updated_at": datetime.now()}
            }
        )

def checkout():
    """
    Convert cart to order: read cart, create order, clear cart, update stock.
    
    Redshift equivalent:
        BEGIN;
        INSERT INTO ecom.orders (...) SELECT ... FROM ecom.carts WHERE ...;
        INSERT INTO ecom.order_items (...) SELECT ... FROM ecom.cart_items WHERE ...;
        UPDATE ecom.carts SET status = 'converted' WHERE cart_id = ...;
        UPDATE ecom.products SET stock_qty = stock_qty - ... WHERE product_id = ...;
        COMMIT;
        -- Multiple tables, transaction overhead: 200-500ms
    """
    active_cart = db.carts.find_one(
        {"customer_id": customer_id, "status": "active"}
    )
    
    if not active_cart or not active_cart.get("items"):
        return None
    
    customer = db.customers.find_one({"_id": customer_id})
    shipping_addr = next(
        (a for a in customer.get("addresses", []) if a["type"] == "shipping"),
        {}
    )
    
    # Create order
    order_doc = {
        "customer_id": customer["_id"],
        "customer_name": customer["name"],
        "customer_region": customer.get("region", "unknown"),
        "customer_segment": customer.get("segment", "standard"),
        "status": "pending",
        "items": [{
            "product_id": item["product_id"],
            "name": item["name"],
            "category": item.get("category", ""),
            "price": item["price"],
            "quantity": item["quantity"],
            "subtotal": item["subtotal"]
        } for item in active_cart["items"]],
        "item_count": active_cart["item_count"],
        "total_amount": active_cart["total"],
        "shipping": {
            "address": shipping_addr,
            "method": "standard",
            "cost": 5.99
        },
        "payment": {
            "method": "credit_card",
            "last_four": "[REDACTED:CREDIT_DEBIT_NUMBER]",
            "charged_at": datetime.now()
        },
        "created_at": datetime.now()
    }
    
    result = db.orders.insert_one(order_doc)
    
    # Mark cart as converted
    db.carts.update_one(
        {"_id": active_cart["_id"]},
        {"$set": {"status": "converted", "updated_at": datetime.now()}}
    )
    
    # Decrement stock
    for item in active_cart["items"]:
        db.products.update_one(
            {"_id": item["product_id"]},
            {"$inc": {"stock_qty": -item["quantity"]}}
        )
    
    return result.inserted_id

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Redshift Can Do Everything"
    ❌ Using Redshift for cart operations, user sessions, real-time features
    SYMPTOMS:
        → Cart page takes 2+ seconds
        → WLM queues fill up during peak traffic
        → Individual INSERTs creating unsorted regions
        → VACUUM running constantly
    FIX: Move operational workloads to MongoDB

PITFALL 2: "MongoDB Can Do Everything"
    ❌ Running complex multi-collection analytics in MongoDB
    SYMPTOMS:
        → Analytics queries taking minutes
        → Aggregation pipeline hitting 100MB memory limit
        → allowDiskUse: true → queries take 10x longer
        → Data team complaining they can't use SQL or BI tools
    FIX: ETL operational data to Redshift for analytics

PITFALL 3: "I'll Migrate Everything from Redshift to MongoDB"
    ❌ Assuming MongoDB replaces Redshift entirely
    REALITY:
        → MongoDB replaces Redshift for OPERATIONAL workloads
        → Redshift remains essential for ANALYTICAL workloads
        → Both serve different purposes — complement, not replace

PITFALL 4: "Same Schema in Both Systems"
    ❌ Using identical normalized schema in MongoDB as Redshift
    ❌ Using identical denormalized schema in Redshift as MongoDB
    FIX:
        → MongoDB: denormalized documents for application reads
        → Redshift: normalized tables for analytical queries
        → TRANSFORM data during pipeline between systems

PITFALL 5: "No Data Pipeline — Just Query Both Directly"
    ❌ Application queries MongoDB for ops AND Redshift for analytics
       with no data sync between them
    SYMPTOMS:
        → Data drift: MongoDB has orders Redshift doesn't
        → Analytics team works with stale data
        → No single source of truth
    FIX: Establish CDC pipeline: MongoDB → Transform → Redshift
         MongoDB = source of truth for operational data
         Redshift = derived analytical copy, refreshed regularly

PITFALL 6: "Real-Time Analytics in Redshift"
    ❌ Expecting Redshift to show real-time metrics
    REALITY:
        → Redshift streaming ingestion adds minutes of latency
        → Materialized views refresh on schedule
        → For TRUE real-time: use MongoDB aggregation + Atlas Charts
        → For HISTORICAL analytics: Redshift is superior
    DESIGN:
        → Real-time dashboard (last hour) → MongoDB aggregation
        → Historical dashboard (last 2 years) → Redshift queries

    → One read returns EVERYTHING for order detail page
