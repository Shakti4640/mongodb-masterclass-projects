PROJECT 30: Time Series Collections — Purpose-Built Storage

Problem: Ingest and Analyze Billions of Stock Price Ticks in Real-Time with Historical Trend Queries

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your fintech platform tracks stock prices for 5,000 tickers
    Each ticker generates a price tick every second during market hours
    
    DATA VOLUME:
    → 5,000 tickers × 1 tick/second × 23,400 seconds/trading day
    → = 117,000,000 ticks per day
    → = ~585 million ticks per week
    → = ~2.5 billion ticks per month
    → Must retain 2 years of history = ~60 billion documents
    
    REQUIREMENTS:
    → Ingest: 5,000 writes/second sustained (during market hours)
    → Real-time query: "Current price + last 5 minutes for AAPL" → <50ms
    → Recent analysis: "VWAP for TSLA in last hour" → <200ms
    → Historical analysis: "Monthly average close for AAPL over 2 years" → <5 seconds
    → Storage efficiency: tick data is highly compressible
    → Auto-expiry: delete data older than 2 years automatically
    
    Each tick looks like:
    {
        ticker: "AAPL",
        timestamp: ISODate("2024-06-15T14:30:22.500Z"),
        price: 195.42,
        volume: 1250,
        bid: 195.40,
        ask: 195.44,
        exchange: "NASDAQ"
    }
    
    YOUR REDSHIFT BRAIN:
    "This is EXACTLY what Redshift is built for — columnar storage,
     sort key on timestamp, massive analytical scans"
    
    YOUR MONGODB BRAIN SHOULD ASK:
    "Can MongoDB handle this? Isn't this Redshift territory?"
    
    ANSWER: MongoDB 5.0+ introduced TIME SERIES COLLECTIONS
    — a purpose-built storage engine that borrows columnar ideas
    — and beats Redshift for RECENT data + real-time ingestion
    — while Redshift still wins for deep historical analytics

🔴 REDSHIFT MINDSET — How You Would Solve This

Table Design — Redshift's Natural Strength

-- REDSHIFT: Stock price ticks — THIS IS REDSHIFT'S SWEET SPOT

CREATE TABLE market.stock_ticks (
    ticker          VARCHAR(10)     NOT NULL,
    tick_timestamp  TIMESTAMP       NOT NULL,    -- microsecond precision
    price           DECIMAL(12,4)   NOT NULL,
    volume          INTEGER         NOT NULL,
    bid             DECIMAL(12,4),
    ask             DECIMAL(12,4),
    exchange        VARCHAR(20)
)
DISTSTYLE KEY
DISTKEY(ticker)                     -- co-locate same ticker on same slice
COMPOUND SORTKEY(ticker, tick_timestamp)  -- sorted by ticker then time
;

-- WHY THIS IS EXCELLENT IN REDSHIFT:
-- → DISTKEY(ticker): all AAPL data on same compute slice
--   → "AAPL last 5 minutes" reads from ONE slice, not all
-- → SORTKEY(ticker, tick_timestamp): data physically sorted
--   → Zone maps skip blocks outside time range instantly
--   → "AAPL between 2:00-2:05 PM" skips 99.99% of blocks
-- → Columnar storage: AVG(price) reads ONLY price column
--   → 4 bytes per tick × 117M ticks/day = 468MB/day for price column
--   → Highly compressible (Delta encoding on sorted prices)
-- → Compression: AAPL prices vary by ~$1/day
--   → Delta encoding: store differences = ~0.5 bytes/tick effective
--   → 60 billion ticks × 0.5 bytes = ~30GB for price column (2 years!)

-- LOADING DATA:
-- High-throughput batch loading via COPY
COPY market.stock_ticks
FROM 's3://market-data/ticks/2024/06/15/'
IAM_ROLE 'arn:aws:iam::123456789:role/redshift-s3'
FORMAT AS PARQUET;
-- COPY loads millions of rows per second from S3

-- QUERY 1: Recent price for AAPL
SELECT price, tick_timestamp
FROM market.stock_ticks
WHERE ticker = 'AAPL'
  AND tick_timestamp >= DATEADD(minute, -5, GETDATE())
ORDER BY tick_timestamp DESC
LIMIT 300;
-- FAST: sort key skips to AAPL + recent time range
-- ~100-500ms (includes query planning overhead)

-- QUERY 2: VWAP (Volume Weighted Average Price) for TSLA last hour
SELECT
    DATE_TRUNC('minute', tick_timestamp) AS minute,
    SUM(price * volume) / SUM(volume) AS vwap,
    SUM(volume) AS total_volume
FROM market.stock_ticks
WHERE ticker = 'TSLA'
  AND tick_timestamp >= DATEADD(hour, -1, GETDATE())
GROUP BY 1
ORDER BY 1;
-- FAST: ~500ms-2s

-- QUERY 3: Monthly average close over 2 years
SELECT
    DATE_TRUNC('month', tick_timestamp) AS month,
    AVG(price) AS avg_price,
    MAX(price) AS high,
    MIN(price) AS low,
    SUM(volume) AS total_volume
FROM market.stock_ticks
WHERE ticker = 'AAPL'
  AND tick_timestamp >= DATEADD(year, -2, GETDATE())
GROUP BY 1
ORDER BY 1;
-- MODERATE: ~2-10s (scanning 2 years of one ticker)

-- QUERY 4: Cross-ticker correlation (Redshift's REAL strength)
SELECT
    a.ticker AS ticker_a,
    b.ticker AS ticker_b,
    CORR(a.price, b.price) AS price_correlation
FROM (
    SELECT ticker, DATE_TRUNC('hour', tick_timestamp) AS hour, AVG(price) AS price
    FROM market.stock_ticks
    WHERE tick_timestamp >= DATEADD(year, -1, GETDATE())
    GROUP BY 1, 2
) a
JOIN (
    SELECT ticker, DATE_TRUNC('hour', tick_timestamp) AS hour, AVG(price) AS price
    FROM market.stock_ticks
    WHERE tick_timestamp >= DATEADD(year, -1, GETDATE())
    GROUP BY 1, 2
) b ON a.hour = b.hour AND a.ticker < b.ticker
GROUP BY 1, 2
ORDER BY price_correlation DESC
LIMIT 50;
-- COMPLEX ANALYTICAL: ~30-120s
-- Scans billions of rows, JOINs, correlates
-- THIS is where Redshift SHINES — MongoDB cannot compete here

Where Redshift Struggles for This Problem

REDSHIFT WEAKNESSES FOR TIME SERIES:

1. INGESTION LATENCY
   → COPY from S3 = batch (not real-time)
   → Streaming ingestion (Kinesis) added recently but limited
   → INSERT INTO for single rows = very slow (~50-200ms per row)
   → 5,000 inserts/second = Redshift will choke
   → Must BATCH: collect in Kinesis → micro-batch COPY every 30-60s
   → RESULT: data is 30-60 seconds STALE

2. CONCURRENCY
   → Redshift WLM: ~15-50 concurrent queries (depending on config)
   → 100 traders each querying their watchlist = queue contention
   → Auto WLM helps but fundamentally limited
   → MongoDB: thousands of concurrent reads/writes

3. SINGLE-DOCUMENT LATENCY
   → "Get latest AAPL price" → Redshift: ~100-500ms (query overhead)
   → Same query in MongoDB: ~1-5ms (index lookup)
   → For real-time dashboards, 100ms is too slow

4. POINT-IN-TIME LOOKUPS
   → "What was AAPL price at exactly 2:30:15.500 PM?" 
   → Redshift: still scans blocks even with sort key
   → MongoDB with index: direct B-tree lookup

5. DATA RETENTION
   → Deleting old data: DELETE WHERE timestamp < X
   → In Redshift: DELETE is expensive (marks rows as deleted, needs VACUUM)
   → VACUUM on 60 billion rows = hours
   → MongoDB TTL index: automatic background deletion, zero overhead

Key Concepts — Time Series Collection Anatomy

WHEN YOU CREATE A TIME SERIES COLLECTION, YOU SPECIFY:

1. timeField (REQUIRED):
   → The field containing the timestamp
   → "When was this measurement taken?"
   → Like: Redshift SORTKEY on timestamp column

2. metaField (RECOMMENDED):
   → The field identifying the SOURCE of measurement
   → "Which sensor/ticker/device produced this?"
   → Like: Redshift DISTKEY — groups related measurements
   → Must be the SAME for measurements from same source
   → Can be a string OR an object (for compound metadata)

3. granularity (OPTIONAL but IMPORTANT):
   → How frequently measurements arrive: "seconds" | "minutes" | "hours"
   → Tells MongoDB how to size internal buckets
   → "seconds" → smaller buckets, more of them
   → "hours" → larger buckets, fewer of them
   → Like: Redshift block size optimization for scan patterns
   
   MongoDB 6.3+: bucketMaxSpanSeconds + bucketRoundingSeconds
   → Even finer control over bucket boundaries

4. expireAfterSeconds (OPTIONAL):
   → Auto-delete data after N seconds
   → Like TTL index (Project 5) but built into time series
   → Redshift: must manually DELETE + VACUUM (expensive)

VISUALIZATION:

    WHAT YOU INSERT (individual measurements):
    { ticker: "AAPL", timestamp: T1, price: 195.40, volume: 100 }
    { ticker: "AAPL", timestamp: T2, price: 195.42, volume: 150 }
    { ticker: "AAPL", timestamp: T3, price: 195.41, volume: 200 }
    { ticker: "TSLA", timestamp: T1, price: 178.30, volume: 300 }
    { ticker: "TSLA", timestamp: T2, price: 178.35, volume: 250 }
    
    WHAT MONGODB STORES INTERNALLY (bucketed):
    {
        _id: ObjectId(...),
        control: {
            min: { timestamp: T1, price: 195.40, volume: 100 },
            max: { timestamp: T3, price: 195.42, volume: 200 }
        },
        meta: { ticker: "AAPL" },
        data: {
            timestamp: [T1, T2, T3],          ← COLUMNAR arrays!
            price: [195.40, 195.42, 195.41],  ← COLUMNAR arrays!
            volume: [100, 150, 200]           ← COLUMNAR arrays!
        }
    }
    
    → Multiple measurements BUCKETED into ONE internal document
    → Measurement values stored as ARRAYS (columnar within bucket)
    → Compressed with column-specific algorithms
    → control.min/max = ZONE MAPS (same concept as Redshift!)
    
    THIS IS REDSHIFT'S COLUMNAR IDEA INSIDE MONGODB

🧩 HOW TO APPROACH IT IN MONGODB

THINKING FRAMEWORK FOR TIME SERIES:

STEP 1: IDENTIFY THE THREE COMPONENTS
    → timeField: which field is the timestamp?
       → "tick_timestamp" or "timestamp"
    → metaField: which field identifies the source?
       → "ticker" (or object: { ticker, exchange })
    → measurements: what values are recorded?
       → price, volume, bid, ask

STEP 2: CHOOSE GRANULARITY
    → How often do measurements arrive from ONE source?
    → Stock ticks: every 1 second → granularity: "seconds"
    → IoT sensors: every 1 minute → granularity: "minutes"
    → Weather stations: every 1 hour → granularity: "hours"
    → THIS affects internal bucket sizing

STEP 3: SET RETENTION
    → How long to keep data?
    → expireAfterSeconds: 63072000 (2 years in seconds)
    → Data auto-deletes — no VACUUM, no DELETE statement

STEP 4: CREATE SECONDARY INDEXES
    → Time series collections auto-create internal index on metaField + timeField
    → Add compound indexes for your specific query patterns
    → Like choosing Redshift SORTKEY columns

STEP 5: INSERT NORMALLY
    → insertOne() / insertMany() — same as regular collections
    → MongoDB handles bucketing INTERNALLY
    → You NEVER see the internal bucket structure
    → You query as if each measurement is a separate document

KEY DIFFERENCE FROM PROJECT 16 (Manual Bucket Pattern):
    Project 16: YOU designed the bucket document structure
               YOU managed the bucketing logic
               YOU wrote upsert queries to add to buckets
    
    Project 30: MongoDB does ALL of this automatically
               You insert individual measurements
               MongoDB buckets them internally
               You query individual measurements
               MongoDB un-buckets them transparently

Step 1: Create the Time Series Collection

// ═══════════════════════════════════════════════════════
// mongosh — Create Time Series Collection
// ═══════════════════════════════════════════════════════

use fintech

// ── REDSHIFT EQUIVALENT (for reference) ──────────────
// CREATE TABLE market.stock_ticks (
//     ticker VARCHAR(10),
//     tick_timestamp TIMESTAMP,
//     price DECIMAL(12,4),
//     volume INTEGER,
//     bid DECIMAL(12,4),
//     ask DECIMAL(12,4),
//     exchange VARCHAR(20)
// )
// DISTSTYLE KEY DISTKEY(ticker)
// COMPOUND SORTKEY(ticker, tick_timestamp);

// ── MONGODB TIME SERIES COLLECTION ───────────────────
db.createCollection("stock_ticks", {
    timeseries: {
        timeField: "timestamp",           // REQUIRED: the time column
                                          // Like SORTKEY(tick_timestamp)
        
        metaField: "metadata",            // RECOMMENDED: source identifier
                                          // Like DISTKEY(ticker)
                                          // Can be string OR object
        
        granularity: "seconds"            // How often data arrives per source
                                          // Options: "seconds" | "minutes" | "hours"
                                          // "seconds" → small buckets (many)
                                          // "hours" → large buckets (few)
    },
    expireAfterSeconds: 63072000          // Auto-delete after 2 years
                                          // = 2 × 365 × 24 × 3600
                                          // Redshift: no equivalent (manual DELETE + VACUUM)
})

// WHAT JUST HAPPENED:
// 1. Collection "stock_ticks" created with time series engine
// 2. Internal bucket collection "system.buckets.stock_ticks" created
//    (you never interact with this directly)
// 3. Internal clustered index on metadata + timestamp created
// 4. TTL background thread configured to delete expired data

// VERIFY:
db.getCollectionInfos({ name: "stock_ticks" })
// Shows: type: "timeseries" with timeseries options
CRITICAL — metaField DESIGN DECISION:

OPTION A: Simple string metaField
    metaField: "ticker"
    → Insert: { ticker: "AAPL", timestamp: ..., price: ... }
    → Bucketed by ticker value
    → Good for: queries always filter by ONE ticker

OPTION B: Object metaField (RECOMMENDED for this problem)
    metaField: "metadata"
    → Insert: { metadata: { ticker: "AAPL", exchange: "NASDAQ" }, timestamp: ..., price: ... }
    → Bucketed by metadata object
    → Good for: queries filter by ticker AND/OR exchange
    → Allows compound metadata for richer grouping

WE USE OPTION B — more flexible, like compound DISTKEY in Redshift

WHY metaField MATTERS:
    → Data with SAME metaField value goes into SAME buckets
    → Like Redshift DISTKEY: same key → same slice → co-located
    → Query filtering by metaField → skip entire buckets
    → Like Redshift zone maps skipping blocks by DISTKEY value

    for (let t of tickers) {
        // Simulate price movement (random walk)
        let priceChange = (Math.random() - 0.5) * 0.10

        let price = Math.round(t.basePrice * 100) / 100

            metadata: { 
                ticker: t.ticker, 
                exchange: t.exchange

            price: price,
            volume: Math.floor(Math.random() * 5000) + 100,
            bid: Math.round((price - 0.02) * 100) / 100,
            ask: Math.round((price + 0.02) * 100) / 100

    if (batch.length >= 5000) {
        db.stock_ticks.insertMany(batch, { ordered: false })
        batch = []
    }

// Insert remaining

// QUERY 3: VWAP (Volume Weighted Average Price) — last hour
// Using aggregation pipeline (Project 12-13 skills)
// ═══════════════════════════════════════════════════════

// ── REDSHIFT ─────────────────────────────────────────
// SELECT
//     DATE_TRUNC('minute', tick_timestamp) AS minute,
//     SUM(price * volume) / NULLIF(SUM(volume), 0) AS vwap,
//     SUM(volume) AS total_volume,
//     COUNT(*) AS tick_count
// FROM market.stock_ticks
// WHERE ticker = 'TSLA'
//   AND tick_timestamp >= DATEADD(hour, -1, GETDATE())
// GROUP BY 1
// ORDER BY 1;

// ── MONGODB ──────────────────────────────────────────

    // Stage 1: Filter (uses internal bucket index)
    {
        $match: {
            "metadata.ticker": "TSLA",
            timestamp: { $gte: oneHourAgo }
        }
    },
    // Stage 2: Group by minute (like DATE_TRUNC + GROUP BY)
    {
        $group: {
            _id: {
                $dateTrunc: {
                    date: "$timestamp",
                    unit: "minute"
                }
            },
            vwap_numerator: {
                $sum: { $multiply: ["$price", "$volume"] }
            },
            total_volume: { $sum: "$volume" },
            tick_count: { $sum: 1 }
        }
    },
    // Stage 3: Calculate VWAP
    {
        $project: {
            minute: "$_id",
            vwap: {
                $cond: {
                    if: { $eq: ["$total_volume", 0] },
                    then: 0,
                    else: {
                        $round: [
                            { $divide: ["$vwap_numerator", "$total_volume"] },
                            4
                        ]
                    }
                }
            },
            total_volume: 1,
            tick_count: 1
        }
    },
    // Stage 4: Sort by time
    { $sort: { minute: 1 } }
])

// ~50-200ms for last hour of one ticker
// Redshift: ~500ms-2s
// MongoDB wins here because:
//   → metaField filtering skips all non-TSLA buckets
//   → Time range skips old buckets
//   → Only reads relevant buckets (maybe 5-10)
//   → Columnar compression means less I/O

Step 4: Window Functions on Time Series ($setWindowFields)

// ═══════════════════════════════════════════════════════
// QUERY 4: Moving Average — 5-minute SMA for AAPL
// Using $setWindowFields (Project 13 skills)
// ═══════════════════════════════════════════════════════

// ── REDSHIFT ─────────────────────────────────────────
// SELECT
//     tick_timestamp,
//     price,
//     AVG(price) OVER (
//         PARTITION BY ticker
//         ORDER BY tick_timestamp
//         ROWS BETWEEN 299 PRECEDING AND CURRENT ROW
//     ) AS sma_5min,
//     price - LAG(price, 1) OVER (
//         PARTITION BY ticker
//         ORDER BY tick_timestamp
//     ) AS price_change
// FROM market.stock_ticks
// WHERE ticker = 'AAPL'
//   AND tick_timestamp >= DATEADD(hour, -1, GETDATE())
// ORDER BY tick_timestamp;

// ── MONGODB ──────────────────────────────────────────
db.stock_ticks.aggregate([
    // Stage 1: Filter to AAPL last hour
    {
        $match: {
            "metadata.ticker": "AAPL",
            timestamp: { $gte: oneHourAgo }
        }
    },
    // Stage 2: Sort by time (required for window functions)
    { $sort: { timestamp: 1 } },
    // Stage 3: Window functions
    {
        $setWindowFields: {
            partitionBy: "$metadata.ticker",
            sortBy: { timestamp: 1 },
            output: {
                // 5-minute Simple Moving Average (300 seconds = 300 ticks at 1/sec)
                sma_5min: {
                    $avg: "$price",
                    window: {
                        documents: [-299, 0]    // ROWS BETWEEN 299 PRECEDING AND CURRENT
                    }
                },
                // Price change from previous tick
                price_change: {
                    $subtract: [                // price - LAG(price, 1)
                        "$price",
                        { $shift: { output: "$price", by: -1, default: null } }
                    ]
                    // Note: $shift not available — use alternative below
                },
                // Previous price (LAG equivalent)
                prev_price: {
                    $shift: {
                        output: "$price",
                        by: -1,
                        default: null
                    }
                },
                // Cumulative volume
                cumulative_volume: {
                    $sum: "$volume",
                    window: {
                        documents: ["unbounded", "current"]
                    }
                },
                // Rank by volume within this hour
                volume_rank: {
                    $rank: {}
                }
            }
        }
    },
    // Stage 4: Add calculated price change
    {
        $addFields: {
            price_change: {
                $cond: {
                    if: { $eq: ["$prev_price", null] },
                    then: 0,
                    else: {
                        $round: [
                            { $subtract: ["$price", "$prev_price"] },
                            4
                        ]
                    }
                }
            }
        }
    },
    // Stage 5: Project final output
    {
        $project: {
            _id: 0,
            timestamp: 1,
            price: 1,
            sma_5min: { $round: ["$sma_5min", 4] },
            price_change: 1,
            volume: 1,
            cumulative_volume: 1
        }
    },
    { $limit: 20 }  // last 20 ticks for display
])

// OUTPUT:
// {
//     timestamp: ISODate("2024-06-15T14:59:40Z"),
//     price: 195.23,
//     sma_5min: 195.18,        ← 5-minute moving average
//     price_change: 0.03,       ← change from previous tick
//     volume: 2340,
//     cumulative_volume: 4521000 ← running total
// }

            trading_day: "$_id",
            open: { $round: ["$open", 2] },

            low: { $round: ["$low", 2] },

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

OPTIMIZATION 1: CUSTOM BUCKET PARAMETERS (MongoDB 6.3+)
    db.createCollection("high_freq_ticks", {
        timeseries: {
            timeField: "timestamp",
            metaField: "metadata",
            bucketMaxSpanSeconds: 3600,        // max 1 hour per bucket
            bucketRoundingSeconds: 3600         // align buckets to hour boundaries
        }
    })
    → Finer control than granularity: "seconds" | "minutes" | "hours"
    → Align buckets to natural boundaries (top of hour, midnight)
    → Like choosing Redshift block boundaries (except Redshift doesn't allow this)

OPTIMIZATION 2: COMPOUND metaField FOR RICHER BUCKETING
    metadata: {
        ticker: "AAPL",
        exchange: "NASDAQ",
        asset_class: "equity",
        sector: "technology"
    }
    → All metadata travels with each bucket
    → Enables filtering on any metadata dimension
    → Like multi-column DISTKEY (which Redshift doesn't support)

OPTIMIZATION 3: ORDERED INSERTS FOR BETTER COMPRESSION
    → Insert ticks IN CHRONOLOGICAL ORDER when possible
    → Same-time ticks grouped into same bucket → better compression
    → Out-of-order inserts work but may create more buckets
    → Like Redshift COPY loading pre-sorted data for better zone maps

OPTIMIZATION 4: SEPARATE COLLECTIONS BY RETENTION

    → "stock_ticks_daily" → pre-aggregated OHLCV, no expiry

    → Historical queries hit aggregated collection (compact)

✅ WHAT SHOULD DO (Best Practices)

✅ ALWAYS SET metaField
    → Without metaField, all data goes into same buckets
    → With metaField, data from same source is co-located
    → Queries filtering by source skip irrelevant buckets entirely
    → Like setting DISTKEY in Redshift — ALWAYS do it

✅ CHOOSE GRANULARITY MATCHING YOUR DATA RATE
    → 1 measurement/second → granularity: "seconds"
    → 1 measurement/minute → granularity: "minutes"
    → 1 measurement/hour → granularity: "hours"
    → WRONG granularity → suboptimal bucket sizes
    → Too fine: many tiny buckets → overhead
    → Too coarse: huge buckets → slow to scan within bucket

✅ USE expireAfterSeconds FOR AUTOMATIC RETENTION
    → Set at collection creation time
    → Data auto-deleted by background thread
    → Zero application code for retention management
    → MUCH better than Redshift DELETE + VACUUM

✅ INSERT IN TIME ORDER WHEN POSSIBLE
    → Chronological inserts → better bucket packing
    → Better packing → better compression → less storage
    → Out-of-order inserts work but are slightly less efficient

✅ CREATE SECONDARY INDEXES FOR YOUR QUERY PATTERNS
    → Internal index covers metaField + timeField
    → Add indexes for other access patterns
    → Remember Project 10: compound index field order matters

✅ USE AGGREGATION PIPELINE FOR ANALYTICS
    → $match → $group → $sort → $setWindowFields
    → All your Project 12-13 skills apply directly
    → $dateTrunc is your DATE_TRUNC equivalent
    → $setWindowFields is your window function equivalent

✅ PRE-AGGREGATE FOR HISTORICAL QUERIES
    → Raw ticks: keep for recent window (hours/days)
    → Minute bars: aggregate and store separately
    → Daily OHLCV: aggregate and store permanently
    → Reduces data volume 60-3600x for historical analysis
    → Like Redshift materialized views for pre-aggregation

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT UPDATE TIME SERIES DOCUMENTS
    → Time series documents are INSERT-ONLY by design
    → Updates to measurement fields are restricted (MongoDB 5.0-6.x)
    → MongoDB 7.0+: some update operations allowed on metaField
    → If you need to "correct" a tick: insert a correction document
    → Redshift habit: UPDATE old rows → does not apply here

❌ DO NOT USE TIME SERIES COLLECTION FOR NON-TEMPORAL DATA
    → Products catalog → regular collection
    → User profiles → regular collection
    → Stock ticks → time series collection ✅
    → IoT sensor data → time series collection ✅
    → Server metrics → time series collection ✅
    → The collection TYPE must match the data PATTERN

❌ DO NOT FORGET: metaField IS IMMUTABLE AFTER CREATION
    → Once you set metaField: "metadata" at collection creation
    → You CANNOT change it later
    → Choose wisely — like Redshift DISTKEY (also hard to change)
    → If wrong: must recreate collection and migrate data

❌ DO NOT PUT HIGH-CARDINALITY DATA IN metaField
    → metaField: { ticker: "AAPL", timestamp: "..." } ← WRONG
    → timestamp in metaField = every bucket has ONE document = no benefit
    → metaField should have LOW-TO-MODERATE cardinality
    → 5,000 tickers = fine (5,000 unique meta values)
    → 1,000,000 unique user IDs = probably too many for metaField

❌ DO NOT QUERY WITHOUT metaField FILTER WHEN POSSIBLE
    → db.stock_ticks.find({ price: { $gt: 200 } })  ← scans ALL buckets
    → db.stock_ticks.find({ "metadata.ticker": "AAPL", price: { $gt: 200 } })
      ← skips non-AAPL buckets
    → Same as Redshift: queries without DISTKEY filter = broadcast to all slices

❌ DO NOT CREATE REGULAR COLLECTION THEN TRY TO CONVERT
    → You cannot convert a regular collection to time series
    → Must create as time series FROM THE START
    → Or: create new time series collection + migrate data
    → Plan your collection type BEFORE inserting data

❌ DO NOT MIX MEASUREMENT SCHEMAS RANDOMLY
    → Doc 1: { price: 195.42, volume: 1250 }
    → Doc 2: { temperature: 72.5, humidity: 45 }
    → Same collection with wildly different fields = poor compression
    → Each bucket optimizes compression per field
    → Consistent fields across documents → better compression
    → Redshift equivalent: don't mix completely different row types in one table

📍 WHEN TO USE MONGODB TIME SERIES OVER REDSHIFT

USE MONGODB TIME SERIES WHEN:
    → Real-time ingestion required (sub-second latency)
    → High write concurrency (thousands of inserts/second)
    → Recent data queries dominate (last minutes/hours/days)
    → Point lookups needed ("latest value for sensor X")
    → Flexible metadata (sensors have different attributes)
    → Automatic data expiry needed (TTL)
    → Operational + time series in SAME database
      (e.g., user profiles + user activity timestamps)
    → High query concurrency (thousands of concurrent readers)
    → Data arrives continuously (not batch)

📍 WHEN TO STICK WITH REDSHIFT INSTEAD

USE REDSHIFT FOR TIME SERIES WHEN:
    → Deep historical analysis dominates (months/years of data)
    → Cross-source analytics needed (correlate 5000 tickers)
    → Complex JOINs with dimension tables (ticker → company → sector)
    → Window functions over MASSIVE datasets
    → Data arrives in BATCHES (hourly/daily ETL)
    → SQL compatibility required for BI tools
    → Team has SQL expertise, not MongoDB experience
    → Existing data warehouse infrastructure
    → Ad-hoc analytical queries from business analysts
    → The data volume exceeds 100TB+ (Redshift RA3 handles this well)

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Time Series Internals — Deep Dive

WHEN YOU INSERT INTO A TIME SERIES COLLECTION:

db.stock_ticks.insertOne({
    metadata: { ticker: "AAPL", exchange: "NASDAQ" },
    timestamp: ISODate("2024-06-15T14:30:22.500Z"),
    price: 195.42,
    volume: 1250,
    bid: 195.40,
    ask: 195.44
})

HERE'S WHAT HAPPENS INTERNALLY:

STEP 1: BUCKET SELECTION
    → MongoDB checks: "Is there an OPEN bucket for this metaField + time range?"
    → Bucket matching criteria:
      a. Same metaField value: { ticker: "AAPL", exchange: "NASDAQ" }
      b. Timestamp falls within bucket's time span
      c. Bucket is not CLOSED (not full, not expired)
      d. Bucket hasn't exceeded max document count (~1000)
    
    → IF matching bucket found: add measurement to existing bucket
    → IF no matching bucket: create NEW bucket
    
    ANALOGY TO REDSHIFT:
    → Redshift appends row to the correct 1MB block based on sort key
    → MongoDB appends measurement to the correct bucket based on meta + time
    → Same concept: physically co-locate related data

STEP 2: COLUMNAR INSERTION INTO BUCKET
    → The measurement's fields are DECOMPOSED into columnar arrays
    → NOT stored as a complete BSON document
    
    BEFORE insertion, bucket.data looks like:
    {
        timestamp: { "0": T1, "1": T2 },
        price:     { "0": 195.40, "1": 195.38 },
        volume:    { "0": 800, "1": 1100 }
    }
    
    AFTER insertion:
    {
        timestamp: { "0": T1, "1": T2, "2": T3 },     ← appended
        price:     { "0": 195.40, "1": 195.38, "2": 195.42 },  ← appended
        volume:    { "0": 800, "1": 1100, "2": 1250 }  ← appended
    }
    
    EACH FIELD IS ITS OWN ARRAY — THIS IS COLUMNAR STORAGE
    
    COMPARED TO REGULAR COLLECTION (Project 1-28):
    → Regular: each document stored as complete BSON blob
    → Time series: each measurement decomposed into column arrays
    → Time series reads: can scan ONLY price array (skip volume, bid, ask)
    → SAME optimization as Redshift columnar blocks

STEP 3: ZONE MAP UPDATE (control.min/max)
    → bucket.control.min.price = MIN(existing_min, 195.42) 
    → bucket.control.max.price = MAX(existing_max, 195.42)
    → bucket.control.min.timestamp = MIN(existing_min, T3)
    → bucket.control.max.timestamp = MAX(existing_max, T3)
    
    THESE ARE ZONE MAPS — identical concept to Redshift
    → Query: WHERE price > 200
    → Check bucket: control.max.price = 195.42 < 200
    → SKIP entire bucket (no need to read any data)
    → Redshift does EXACTLY this with 1MB block zone maps

STEP 4: COMPRESSION
    → WiredTiger compresses the bucket document
    → Column arrays compress EXCELLENTLY because:
      → Price values are similar (195.40, 195.38, 195.42)
      → Delta encoding: store differences [0, -0.02, +0.04]
      → Much smaller than absolute values
      → Timestamps are sequential: delta = 1 second each
      → Volume values: integer compression
    
    → Same compression advantage as Redshift columnar encoding:
      → Delta encoding for sorted/sequential values
      → Dictionary encoding for low-cardinality strings
      → Run-length encoding for repeated values

STEP 5: INTERNAL COLLECTION STRUCTURE
    → You see: db.stock_ticks (the time series collection)
    → Internally exists: db.system.buckets.stock_ticks (the actual data)
    → MongoDB maintains a VIEW-like abstraction:
      → When you query db.stock_ticks → MongoDB reads buckets
      → Decompresses and "unpacks" measurements from columnar arrays
      → Returns them as individual documents (as if stored separately)
      → This is TRANSPARENT to your application
    
    ABSTRACTION LAYERS:
    ┌──────────────────────────────────────────────┐
    │  YOUR VIEW:  db.stock_ticks                   │
    │  → Individual measurement documents            │
    │  → { metadata, timestamp, price, volume, ... } │
    │  → Query with find(), aggregate() normally     │
    ├──────────────────────────────────────────────┤
    │  MONGODB ENGINE:  Unpacking layer              │
    │  → Reads buckets from system.buckets           │
    │  → Decompresses columnar arrays                │
    │  → Reconstructs individual measurements        │
    │  → Applies your query filter                   │
    ├──────────────────────────────────────────────┤
    │  PHYSICAL STORAGE:  system.buckets.stock_ticks │
    │  → Bucketed documents with columnar arrays     │
    │  → Zone maps (control.min/max)                 │
    │  → Compressed with column-aware algorithms     │
    │  → Clustered by metaField + timeField          │
    └──────────────────────────────────────────────┘

Query Execution — How Time Range Queries Work

QUERY: "AAPL prices in last 5 minutes"

db.stock_ticks.find({
    "metadata.ticker": "AAPL",
    timestamp: { $gte: fiveMinutesAgo }
})

EXECUTION PATH:

STEP 1: REWRITE TO BUCKET QUERY
    → MongoDB rewrites this internally to query system.buckets:
    → Filter on: meta.ticker = "AAPL"
    → Filter on: control.max.timestamp >= fiveMinutesAgo
    → (If max.timestamp < fiveMinutesAgo, entire bucket is too old → SKIP)

STEP 2: BUCKET SCANNING
    → Assume AAPL has 1000 buckets over 7 days
    → 5 minutes of data at "seconds" granularity ≈ 1-2 buckets
    → 998 buckets SKIPPED by zone map check
    → Only 1-2 buckets READ from disk

STEP 3: WITHIN-BUCKET FILTERING
    → Read the 1-2 matching buckets
    → Decompress timestamp array
    → Filter individual measurements: timestamp >= fiveMinutesAgo
    → For matching timestamps: read corresponding price, volume values
    → Reconstruct individual documents

STEP 4: RETURN RESULTS
    → Return ~300 measurements (5 min × 60 sec/min = 300 ticks)
    → Total time: ~5-20ms

COMPARED TO REDSHIFT:
    → Redshift: Leader node → distribute to slice with AAPL data
    → Slice: check zone maps on tick_timestamp column blocks
    → Skip blocks outside time range
    → Read matching blocks, decompress price column
    → Return results via leader node
    → Total time: ~200-500ms (query planning overhead + network hops)
    
    SAME CONCEPT (zone map skip-scan), but:
    → MongoDB: single-process, in-memory index → faster for small results
    → Redshift: MPP distributed → faster for large analytical scans

Redshift Internals — For Direct Comparison

REDSHIFT TIME SERIES STORAGE:

    INSERT INTO market.stock_ticks VALUES ('AAPL', T1, 195.40, 800, ...);
    INSERT INTO market.stock_ticks VALUES ('AAPL', T2, 195.38, 1100, ...);
    INSERT INTO market.stock_ticks VALUES ('AAPL', T3, 195.42, 1250, ...);

    PHYSICAL STORAGE (per compute slice):
    
    Block: ticker_col_001.block
    ┌─────────────────────────┐
    │ "AAPL", "AAPL", "AAPL" │ ← same value repeated
    │ Zone map: min="AAPL"    │   → run-length encoded: AAPL × 3
    │           max="AAPL"    │   → extremely compact
    └─────────────────────────┘
    
    Block: timestamp_col_001.block
    ┌─────────────────────────┐
    │ T1, T2, T3              │ ← sequential values
    │ Zone map: min=T1        │   → delta encoded: [T1, +1s, +1s]
    │           max=T3        │   → very compact
    └─────────────────────────┘
    
    Block: price_col_001.block
    ┌─────────────────────────┐
    │ 195.40, 195.38, 195.42  │ ← similar values
    │ Zone map: min=195.38    │   → delta encoded: [195.40, -0.02, +0.04]
    │           max=195.42    │   → very compact
    └─────────────────────────┘
    
    Block: volume_col_001.block
    ┌─────────────────────────┐
    │ 800, 1100, 1250         │ ← integer values
    │ Zone map: min=800       │   → delta or LZO encoded
    │           max=1250      │
    └─────────────────────────┘

    KEY DIFFERENCE:
    → Redshift: SEPARATE FILE per column (true columnar)
    → MongoDB TS: columnar ARRAYS within bucket DOCUMENT
    
    → Redshift: block = 1MB fixed size
    → MongoDB TS: bucket = variable size (based on granularity)
    
    → Redshift: zone maps = per block per column
    → MongoDB TS: control.min/max = per bucket per field (same idea!)
    
    → Redshift: compression = per column file
    → MongoDB TS: compression = per bucket (WiredTiger zstd on entire bucket)
    
    FOR SCAN-HEAVY QUERIES (AVG price over 2 years):
    → Redshift reads ONLY price_col files → skips ticker, timestamp, volume
    → MongoDB TS reads ENTIRE bucket → decompresses → extracts price array
    → Redshift wins: true column pruning at file level
    → MongoDB TS: column pruning only WITHIN bucket (less efficient)
    
    FOR POINT QUERIES (latest AAPL price):
    → Redshift: query planning → slice routing → block scan → return
    → MongoDB TS: index lookup → latest bucket → last measurement
    → MongoDB wins: no query planning overhead, direct access

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE WITH TIME SERIES COLLECTIONS:

PITFALL 1: "I'll use a regular collection for time series data"
    ❌ db.createCollection("stock_ticks")  ← regular collection
       db.stock_ticks.insertOne({ ticker: "AAPL", timestamp: ..., price: ... })
    ✅ db.createCollection("stock_ticks", { timeseries: { ... } })
    WHY: regular collection stores each tick as separate BSON document
         → no columnar compression, no automatic bucketing
         → 10-50x more storage, 5-20x slower range queries
         → time series collection gives you Redshift-like efficiency

PITFALL 2: "I forgot metaField — I'll add it later"
    ❌ db.createCollection("ticks", { timeseries: { timeField: "ts" } })
       // No metaField → all data in same buckets → no source filtering
    ✅ db.createCollection("ticks", { timeseries: {
           timeField: "ts", metaField: "meta" } })
    WHY: metaField is IMMUTABLE after creation
         without it: query for ONE ticker scans ALL buckets
         like Redshift without DISTKEY: every query broadcasts to all slices
         CANNOT be added later — must recreate collection

PITFALL 3: "I'll UPDATE historical measurements"
    ❌ db.stock_ticks.updateMany(
           { "metadata.ticker": "AAPL", timestamp: { $lt: yesterday } },
           { $set: { price: corrected_price } }
       )
    → Time series collections have LIMITED update support
    → Measurement fields (price, volume) cannot be updated (MongoDB 5.0-6.x)
    → MongoDB 7.0+: some updates allowed but expensive (bucket rewrite)
    ✅ Insert correction document with flag:
       { metadata: { ticker: "AAPL", type: "correction" },
         timestamp: original_ts, price: corrected_price }
    → Or: use regular collection for data needing frequent updates
    WHY: buckets are optimized for append-only workloads
         updates require decompressing, modifying, recompressing entire bucket

PITFALL 4: "I'll treat metaField like a regular query field"
    ❌ db.stock_ticks.find({ "metadata.ticker": { $regex: /^AA/ } })
    → Regex on metaField = scan all bucket metadata = slow
    ✅ db.stock_ticks.find({ "metadata.ticker": "AAPL" })
    → Exact match on metaField = direct bucket filtering = fast
    WHY: metaField is indexed for EXACT matches in bucket selection
         complex operators ($regex, $in with large arrays) reduce effectiveness
         like Redshift: DISTKEY works best for equality filters

PITFALL 5: "Granularity doesn't matter — I'll just pick any"
    ❌ Data arrives every second → granularity: "hours"
       → Buckets span 1 hour → thousands of measurements per bucket
       → Single bucket too large → slow to decompress for point queries
    ❌ Data arrives every hour → granularity: "seconds"
       → Buckets span ~1 minute → only 1 measurement per bucket
       → Thousands of tiny buckets → overhead, no compression benefit
    ✅ Match granularity to actual data arrival rate
    WHY: granularity determines bucket time span
         wrong sizing = either too many small buckets or too few huge buckets
         like Redshift block sizing — there's an optimal range

PITFALL 6: "I'll use $lookup (JOIN) on time series collections"
    ❌ db.stock_ticks.aggregate([
           { $lookup: { from: "companies", ... } }
       ])
    → $lookup on time series = decompresses ALL matching buckets
    → then JOINs with external collection
    → extremely slow for large time ranges
    ✅ Enrich data BEFORE inserting into time series
       → metadata: { ticker: "AAPL", company: "Apple Inc.", sector: "Tech" }
       → All metadata travels with measurements
       → No $lookup needed at query time
    ✅ Or: aggregate first, THEN $lookup on small result set
       → aggregate → $group → $lookup (join 50 daily summaries, not 1M ticks)
    WHY: same as Redshift — JOIN on fact table is expensive
         denormalize into fact table for best scan performance

PITFALL 7: "I'll skip pre-aggregation — MongoDB can handle raw queries"
    ❌ Query 2 years of raw tick data for monthly averages every time
       → 23M documents per ticker × 5000 tickers = scan nightmare
    ✅ Pre-aggregate into daily_ohlcv collection
       → 2 years × 252 trading days × 5000 tickers = 2.5M documents
       → 10,000x less data to scan for monthly queries
    WHY: same as Redshift materialized views
         pre-aggregation trades write-time compute for read-time speed
         Project 20 (Computed Pattern) applies directly here

📈 SCALABILITY CONSIDERATIONS

MONGODB TIME SERIES SCALING:

SINGLE NODE CAPACITY:
    → 100K inserts/second sustained (with batching)
    → 1-10TB storage per node (depending on Atlas tier)
    → 5M documents: M30 Atlas tier sufficient
    → 500M documents: M50 Atlas tier
    → 5B+ documents: sharding required

    → Shard key MUST include metaField (or its subfields)
    → Optionally include timeField for range-based distribution

    sh.shardCollection("fintech.stock_ticks", { "metadata.ticker": "hashed" })
    
    // Shard by ticker + time (range) — time-based queries target fewer shards
    sh.shardCollection("fintech.stock_ticks", 
        { "metadata.ticker": 1, timestamp: 1 })

STORAGE EFFICIENCY AT SCALE:
    Raw tick: ~150 bytes (as regular BSON document)
    Time series bucket: ~15-30 bytes per measurement (after compression)
    → 5-10x compression ratio
    
    60 billion ticks:
    → Regular collection: ~9 TB (150 bytes × 60B)
    → Time series collection: ~0.9-1.8 TB (15-30 bytes × 60B)
    → Redshift (columnar): ~0.5-1.5 TB (highly optimized columnar encoding)
    
    Redshift still wins on pure compression for time series
    But MongoDB is within 1.5-3x — acceptable trade-off for real-time access

DATA LIFECYCLE SCALING:
    → expireAfterSeconds handles retention automatically
    → No manual DELETE + VACUUM cycles (Redshift's Achilles heel)
    → At 60 billion documents, Redshift VACUUM can take HOURS
    → MongoDB TTL deletes buckets continuously — zero impact
    → THIS is a massive operational advantage at scale

📐 DATA MODELING COMPARISON

TIME SERIES DATA MODELING:

REDSHIFT MODEL (Star Schema):
    ┌─────────────┐     ┌──────────────────┐
    │ dim_tickers  │     │ fact_stock_ticks   │
    │ ticker_id PK │←────│ ticker_id FK       │
    │ ticker_sym   │     │ tick_timestamp     │
    │ company_name │     │ price              │
    │ sector       │     │ volume             │
    │ exchange     │     │ bid                │
    │ market_cap   │     │ ask                │
    └─────────────┘     └──────────────────┘
    
    → Normalized: ticker info in dimension table
    → Fact table has FK to dimension
    → JOIN for enriched queries
    → DISTKEY on ticker_id for co-located JOINs
    → This is TEXTBOOK data warehouse design

MONGODB TIME SERIES MODEL (Embedded Metadata):
    stock_ticks collection:
    {
        metadata: {
            ticker: "AAPL",
            company: "Apple Inc.",      ← denormalized
            sector: "Technology",       ← denormalized
            exchange: "NASDAQ"          ← denormalized
        },
        timestamp: ISODate("2024-06-15T14:30:22Z"),
        price: 195.42,
        volume: 1250,
        bid: 195.40,
        ask: 195.44
    }
    
    → Denormalized: ALL metadata embedded in metaField
    → No JOINs needed at query time
    → metaField travels with every bucket
    → Trade-off: metadata duplicated across all measurements
    → But: metadata is per-BUCKET, not per-measurement
