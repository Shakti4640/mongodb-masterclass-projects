🔴 REDSHIFT MINDSET — Why This Problem Doesn't Exist in Redshift

Redshift's "Free" Optimization

WHY REDSHIFT DEVELOPERS NEVER MANAGE INDEXES:

1. ZONE MAPS — Automatic, Free, Zero Overhead
   → Created automatically for EVERY column
   → Min/max per 1MB block
   → NO storage overhead (tiny metadata per block)
   → NO write overhead (updated as part of normal write)
   → Cannot create "too many" zone maps — they're always there

2. SORT KEYS — One Per Table, No Ongoing Management
   → Defined at CREATE TABLE time
   → Physically sorts data on disk
   → ONE sort key per table (compound OK)
   → No "drop this sort key, add that one" operations
   → VACUUM SORT re-sorts if data becomes unsorted
   
3. NO B-TREE INDEXES
   → Redshift has NO traditional indexes
   → No index storage overhead
   → No write amplification from index maintenance
   → No "too many indexes" problem
   → Trade-off: less flexible access patterns

REDSHIFT'S LIMITATION:
    → ONE sort key = ONE optimal access pattern
    → Need different access pattern? Create materialized view
    → Or accept slower performance for non-sort-key queries
    → Cannot have 23 different "fast paths" to the same data

MONGODB'S ADVANTAGE AND BURDEN:
    → CAN have 23 different fast paths (indexes)
    → But each one costs: storage + write overhead + memory
    → Must MANAGE them actively
    → This is the PRICE of flexibility

🟢 MONGODB MINDSET — Understanding the Index Tax

The Write Amplification Problem

WHAT HAPPENS WHEN YOU INSERT ONE DOCUMENT WITH 23 INDEXES:

    db.orders.insertOne({ customer_id: "cust_1", order_date: new Date(), ... })

    STEP 1: Write document to collection (WiredTiger B-tree)
        → 1 write operation
    
    STEP 2: Update EACH index with new key entry
        → Index 1:  insert into _id index
        → Index 2:  insert into { customer_id: 1 } index
        → Index 3:  insert into { customer_id: 1, order_date: -1 } index
        → Index 4:  insert into { customer_id: 1, status: 1 } index
        → ...
        → Index 21: insert into { items.sku: 1 } index (MULTIKEY — one entry PER array element!)
        → Index 22: insert into { tags: 1 } index (MULTIKEY — one entry PER tag!)
        → Index 23: insert into { promo_code: 1 } index
    
    TOTAL WRITE OPERATIONS: 1 document + 23 index entries
    → If items has 5 SKUs: index 21 = 5 entries
    → If tags has 8 tags: index 22 = 8 entries
    → ACTUAL total: 1 + 21 + 5 + 8 = 35 write operations for ONE insert!
    
    WRITE AMPLIFICATION FACTOR: 35x
    
    WITH 5 WELL-CHOSEN INDEXES:
    → 1 document + 5 index entries = 6 write operations
    → Write amplification: 6x
    → 5.8x LESS write overhead

    IMPACT ON BATCH IMPORTS:
    → 100,000 documents × 35 writes = 3,500,000 operations
    → vs 100,000 × 6 = 600,000 operations
    → Batch import time: ~6x slower with 23 indexes vs 5 indexes

    REDSHIFT COMPARISON:
    → COPY 100,000 rows: writes to columnar blocks + zone map updates
    → Zone map updates: trivial (just track min/max)
    → No B-tree maintenance, no write amplification
    → THIS is why Redshift batch loading is inherently faster

The Memory Problem

INDEXES MUST FIT IN RAM FOR BEST PERFORMANCE:

    WiredTiger cache (default: 50% of RAM - 1GB)
    → 16GB server → ~7GB cache
    
    MUST FIT IN CACHE:
    → Hot data (frequently accessed documents)
    → Active indexes (frequently used for queries)
    → Internal overhead (~20% of cache)
    
    YOUR SITUATION:
    → Data size: 4.2 GB
    → Index size: 6.8 GB
    → Total: 11.0 GB
    → Available cache: 7 GB
    → PROBLEM: data + indexes > cache = EVICTION THRASHING
    
    WHAT HAPPENS WHEN INDEXES DON'T FIT:
    → Index pages evicted from cache to make room
    → Next query needs that index page → disk read
    → Disk read: ~1-10ms (vs ~0.01ms from cache)
    → Query latency becomes UNPREDICTABLE
    → Some queries fast (index in cache), some slow (index on disk)
    
    WITH 5 OPTIMIZED INDEXES:
    → Index size: ~1.5 GB (estimated)
    → Data + indexes: 5.7 GB
    → Fits in 7 GB cache with room to spare
    → ALL queries consistently fast

    REDSHIFT COMPARISON:
    → Zone maps: tiny (a few bytes per block per column)
    → Always fit in memory — never evicted
    → Sort key: not a separate structure — data IS sorted
    → No "index doesn't fit in memory" problem
    → RA3 managed storage handles data/compute separation

🧩 HOW TO APPROACH IT IN MONGODB

THE INDEX OPTIMIZATION METHODOLOGY — 6 STEPS:

STEP 1: AUDIT — What indexes exist?
    → db.collection.getIndexes()
    → Document every index: name, key pattern, size, type

STEP 2: MEASURE — Which indexes are actually used?
    → $indexStats → shows access count per index
    → Unused indexes (0 ops) → immediate DROP candidates

STEP 3: ANALYZE — Which indexes are REDUNDANT?
    → Prefix rule: { a: 1, b: 1 } makes { a: 1 } redundant
    → Direction duplicates: { date: 1 } and { date: -1 }
    → Overlapping compounds: find the minimal covering set

STEP 4: CONSOLIDATE — Merge overlapping indexes
    → Replace multiple single-field with fewer compound indexes
    → Apply ESR rule (Project 31) for optimal compound design
    → Consider partial indexes for conditional queries

STEP 5: DROP — Remove unused and redundant indexes
    → Drop in order: unused first, then redundant
    → Monitor after each drop — verify no query regression

STEP 6: VERIFY — Confirm improvement
    → Re-check: write latency, storage size, cache pressure
    → Re-run explain() on critical queries (Project 31 skills)
    → Compare before/after metrics

REDSHIFT EQUIVALENT METHODOLOGY:
    → Step 1: Check SVV_TABLE_INFO for table health
    → Step 2: Check STL_SCAN for sort key effectiveness
    → Step 3: VACUUM SORT if unsorted percentage high
    → Step 4: ANALYZE to update statistics
    → Much simpler — fewer knobs to turn

// THE PREFIX RULE — The Most Important Redundancy Rule:
//
// A compound index { a: 1, b: 1, c: 1 } can serve queries on:
//   → { a: 1 }            ← prefix match
//   → { a: 1, b: 1 }     ← prefix match
//   → { a: 1, b: 1, c: 1 } ← full match
//
// Therefore:
//   { a: 1, b: 1, c: 1 } makes { a: 1 } REDUNDANT
//   { a: 1, b: 1, c: 1 } makes { a: 1, b: 1 } REDUNDANT
//
// BUT:
//   { a: 1, b: 1, c: 1 } does NOT serve { b: 1 } (not a prefix)
//   { a: 1, b: 1, c: 1 } does NOT serve { a: 1, c: 1 } (b skipped)

Step 4: CONSOLIDATE — Advanced Index Techniques

// ═══════════════════════════════════════════════════════
// STEP 4A: PARTIAL INDEXES — Index only what matters
// ═══════════════════════════════════════════════════════

// PROBLEM: { promo_code: 1 } indexes ALL documents
// BUT: only 5% of orders have promo codes
// SOLUTION: Partial index — only index documents WHERE promo_code EXISTS

// ── BEFORE (full index): ─────────────────────────────
// db.orders.createIndex({ promo_code: 1 })
// → Indexes 10M documents → 10M index entries
// → 95% of entries are for { promo_code: null } — WASTED

// ── AFTER (partial index): ───────────────────────────
db.orders.createIndex(
    { promo_code: 1 },
    {
        name: "promo_code_partial",
        partialFilterExpression: {
            promo_code: { $exists: true, $ne: null }
        }
    }
)
// → Indexes only 500K documents (5% of 10M)
// → 95% smaller index!
// → 95% less write overhead for non-promo orders!

// QUERIES THAT USE THIS INDEX:
db.orders.find({ promo_code: "SAVE20" })  // ✅ uses partial index
db.orders.find({ promo_code: { $exists: true } })  // ✅ matches filter

// QUERIES THAT CANNOT USE THIS INDEX:
db.orders.find({ promo_code: null })  // ❌ excluded by partial filter
db.orders.find({})  // ❌ doesn't include partial filter criteria
// For these queries: COLLSCAN (but that's OK — different use case)

// ── REDSHIFT EQUIVALENT ──────────────────────────────
// No equivalent — Redshift zone maps index EVERYTHING
// Closest concept: Redshift Spectrum partition pruning
// (only scan partitions matching WHERE clause)
// But partitions are at TABLE level, not INDEX level

// ── MORE PARTIAL INDEX EXAMPLES ──────────────────────

// Index only active orders (skip historical/cancelled)
db.orders.createIndex(

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

OPTIMIZATION 1: INDEX INTERSECTION — Let MongoDB combine indexes
    
    Instead of compound { a: 1, b: 1 }:
    → Keep { a: 1 } and { b: 1 } separately
    → MongoDB CAN intersect two indexes at query time
    
    db.orders.find({ customer_id: "cust_5000", status: "delivered" })
    → Could use: IXSCAN(customer_id) ∩ IXSCAN(status) → intersection
    
    BUT: Index intersection is SLOWER than compound index
    → Compound { customer_id: 1, status: 1 } = ONE B-tree walk
    → Intersection = TWO B-tree walks + merge step
    → Use intersection as FALLBACK, not PRIMARY strategy
    
    WHEN INTERSECTION IS OK:
    → Too many compound combinations needed
    → Low-frequency queries that don't justify dedicated compound index
    → Temporary measure before creating proper compound index

OPTIMIZATION 2: TTL INDEXES — Auto-delete old data
    
    // (Project 5 concept, applied to index optimization)
    // Instead of indexing old data → auto-delete it
    db.orders.createIndex(
        { created_at: 1 },
        { expireAfterSeconds: 63072000, name: "ttl_2_years" }
    )
    // → Old orders automatically deleted
    // → Collection stays smaller → all indexes stay smaller
    // → Like time series expireAfterSeconds (Project 30)
    
    REDSHIFT EQUIVALENT:
    // DELETE WHERE created_at < DATEADD(year, -2, GETDATE());
    // VACUUM;  ← expensive manual process
    // MongoDB TTL: automatic, zero overhead

OPTIMIZATION 3: HIDDEN INDEXES — Test dropping without committing
    
    // MongoDB 4.4+: hide an index (planner ignores it)
    // If no queries break → safe to drop
    db.orders.hideIndex("total_-1")
    
    // Monitor for 24-48 hours
    // If no performance regression → drop it
    db.orders.dropIndex("total_-1")
    
    // If queries break → unhide it
    db.orders.unhideIndex("total_-1")
    
    // SAFER than dropping directly!
    // Like Redshift: no equivalent (can't "test" removing sort key)

OPTIMIZATION 4: COLLATION-SPECIFIC INDEXES
    
    // If queries use case-insensitive comparison:
    db.orders.createIndex(
        { customer_email: 1 },
        { collation: { locale: "en", strength: 2 }, name: "email_ci" }
    )
    // strength: 2 = case-insensitive
    // Serves: db.orders.find({ customer_email: "JOHN@EXAMPLE.COM" })
    //           .collation({ locale: "en", strength: 2 })
    // Without collation index: COLLSCAN with regex (/john@example.com/i)

OPTIMIZATION 5: BACKGROUND INDEX BUILDS
    
    // MongoDB 4.2+: ALL index builds are background by default
    // No lock on collection during build
    // Build can take minutes to hours on large collections
    // Monitor progress:
    db.currentOp({ "command.createIndexes": { $exists: true } })

    // Redshift has no index builds
    // Closest: VACUUM SORT (re-sorts data, can take hours)
    // Both are background operations that consume resources

✅ WHAT SHOULD DO (Best Practices)

✅ TARGET 3-8 INDEXES PER COLLECTION
    → 1 (_id — mandatory)
    → 2-5 compound indexes covering primary query patterns
    → 1-2 partial/sparse indexes for conditional queries
    → Multikey indexes only when array queries are critical
    → If you need 15+ → rethink your data model

✅ USE THE PREFIX RULE TO ELIMINATE REDUNDANCY
    → { a: 1, b: 1, c: 1 } replaces { a: 1 } and { a: 1, b: 1 }
    → Design compound indexes to serve MULTIPLE query patterns
    → One compound index can replace 2-3 single-field indexes

✅ APPLY ESR RULE FOR COMPOUND INDEX DESIGN
    → Equality fields first (highest selectivity)
    → Sort fields second (eliminates SORT stage)
    → Range fields last (scan bounded by equality + sort)
    → From Project 31: this maximizes index efficiency

✅ USE PARTIAL INDEXES FOR SPARSE OR CONDITIONAL DATA
    → Only 5% of orders have promo codes → partial index
    → Only 10% of users are premium → partial index
    → Saves 90-95% of index storage and write overhead

✅ USE HIDDEN INDEXES BEFORE DROPPING
    → Hide → monitor 24-48 hours → drop if safe
    → Safer than immediate drop → easy to unhide if needed

✅ MONITOR $indexStats REGULARLY (monthly minimum)
    → Unused indexes = free performance gain when dropped
    → Atlas Performance Advisor automates this
    → Set calendar reminder if not using Atlas

✅ CONSIDER WRITE-HEAVY VS READ-HEAVY WORKLOAD
    → Write-heavy (IoT, logs, events): FEWER indexes
    → Read-heavy (product catalog, user profiles): MORE indexes OK
    → Mixed (e-commerce orders): balance carefully
    → Time series collections: minimal secondary indexes

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT CREATE AN INDEX FOR EVERY SLOW QUERY
    → First check: can an EXISTING compound index serve this query?
    → Prefix rule: { a: 1, b: 1 } already serves { a: 1 } queries
    → Often: adjusting compound index field ORDER fixes the problem
    → Only create NEW index as last resort

❌ DO NOT INDEX LOW-CARDINALITY FIELDS ALONE
    → { status: 1 } where status ∈ {pending, shipped, delivered, cancelled}
    → Only 4 unique values across 10M documents
    → Each index entry points to 2.5M documents → not selective
    → BETTER: compound { status: 1, order_date: -1 } — status narrows, date refines
    → Like Redshift: DISTKEY on low-cardinality = uneven distribution

❌ DO NOT FORGET MULTIKEY INDEX OVERHEAD
    → { items.sku: 1 } on array with 5 elements → 5 index entries per document
    → 10M documents × 5 entries = 50M index entries (5x amplification!)
    → Multikey indexes are EXPENSIVE for writes
    → Use only when array element queries are critical
    → Consider Subset Pattern (Project 19) to limit array size

❌ DO NOT KEEP INDEXES "JUST IN CASE"
    → "Maybe someone will need this someday"
    → Unused index = storage cost + write cost + memory pressure + zero benefit
    → If $indexStats shows 0 ops for 30 days → DROP it
    → You can always recreate if needed (background build)

❌ DO NOT IGNORE INDEX BUILD IMPACT
    → Creating index on 10M documents takes minutes
    → During build: increased CPU + disk I/O + memory usage
    → On production: builds are background but still consume resources
    → Schedule large index builds during low-traffic periods
    → Like Redshift VACUUM: run during maintenance windows

❌ DO NOT CREATE IDENTICAL INDEXES WITH DIFFERENT NAMES
    → Developer A: createIndex({ email: 1 }, { name: "email_idx" })
    → Developer B: createIndex({ email: 1 }, { name: "user_email" })
    → MongoDB prevents this (duplicate key pattern error)
    → BUT: subtle differences can create near-duplicates
      → { email: 1 } vs { email: 1 } with collation → TWO indexes!
    → Review all indexes periodically for near-duplicates

📍 WHEN TO USE MORE INDEXES (MongoDB Wins)

MORE INDEXES JUSTIFIED WHEN:
    → Collection is READ-HEAVY (>90% reads)
    → Multiple distinct query patterns from different application features
    → Low write volume (daily batch import, not real-time streaming)
    → Large RAM available (indexes fit comfortably in cache)
    → Covered queries possible (eliminate FETCH stage entirely)
    → Product catalogs, reference data, user profiles

📍 WHEN TO USE FEWER INDEXES (Redshift Thinking Applies)

FEWER INDEXES JUSTIFIED WHEN:
    → Collection is WRITE-HEAVY (IoT, logs, events, time series)
    → Time series collections (internal index usually sufficient)
    → Limited RAM (indexes must compete for cache space)
    → Batch import performance is critical
    → Queries are primarily analytical (aggregation scans)
    → Accept slower reads for faster writes (OLAP-like pattern)
    → THIS is where MongoDB behaves more like Redshift
      → Fewer access paths, scan-oriented queries

⚙️ HOW IT ONLY WORKS INTERNALLY

B-Tree Index Maintenance — What Happens on Every Write

WHEN YOU INSERT ONE DOCUMENT:

    db.orders.insertOne({
        customer_id: "cust_new",
        order_date: new Date(),
        status: "pending",
        total: 99.99
    })

    FOR EACH INDEX — B-TREE INSERTION:
    
    Index: { customer_id: 1, order_date: -1 }
    ┌─────────────────────────────────────────────────────┐
    │  B-Tree Before Insert:                               │
    │                                                      │
    │           [cust_500]                                  │
    │          /          \                                 │
    │  [cust_100,cust_300] [cust_700,cust_900]             │
    │    /     |     \       /     |     \                  │
    │  leaf   leaf   leaf  leaf   leaf   leaf               │
    │                                                      │
    │  Insert "cust_new":                                   │
    │  1. Walk tree: root → right child → leaf              │
    │  2. Insert key in leaf page                           │
    │  3. If leaf full → SPLIT leaf → update parent         │
    │  4. If parent full → SPLIT parent → cascade up        │
    │  5. Write journal entry for crash recovery             │
    │                                                      │
    │  Cost: O(log n) per index                             │
    │  With 10M documents: ~24 comparisons per index        │
    │  With 23 indexes: 24 × 23 = 552 comparisons          │
    │  With 8 indexes:  24 × 8  = 192 comparisons          │
    │  ~2.9x fewer comparisons → ~2.9x faster writes       │
    └─────────────────────────────────────────────────────┘

    ADDITIONAL COSTS PER INDEX:
    → Memory: index pages loaded into WiredTiger cache
    → I/O: dirty index pages flushed to disk at checkpoint
    → Journal: index modifications written to WAL
    → Lock: document-level lock held during all index updates
    
    MULTIKEY INDEX (array field):
    Index: { items.sku: 1 }
    Document: { items: [{ sku: "A" }, { sku: "B" }, { sku: "C" }] }
    → THREE B-tree insertions for ONE document!
    → Each array element = separate index entry
    → Write cost multiplied by array length

COMPARED TO REDSHIFT:
    INSERT INTO orders VALUES (...);
    
    → Split row into column values
    → Append each value to its column block
    → Update zone map min/max for each affected block
    → Zone map update: compare 2 values (min/max) — trivial
    → NO B-tree maintenance
    → NO logarithmic walk per "index"
    → Write cost: O(columns) not O(indexes × log n)
    → THIS is why Redshift bulk loading is inherently simpler

WiredTiger Cache — Index Memory Pressure

HOW INDEXES COMPETE FOR CACHE:

    WiredTiger Cache (7 GB on 16GB server):
    ┌────────────────────────────────────────────────────┐
    │                                                     │
    │  ┌──────────┐ ┌──────────┐ ┌──────────┐           │
    │  │ Hot Data  │ │ Index    │ │ Index    │           │
    │  │ Pages     │ │ Pages    │ │ Pages    │ ...       │
    │  │ (docs)    │ │ (idx 1)  │ │ (idx 2)  │           │
    │  │ 2.5 GB    │ │ 1.2 GB   │ │ 0.8 GB   │           │
    │  └──────────┘ └──────────┘ └──────────┘           │
    │                                                     │
    │  With 23 indexes: ALL index pages compete for cache │
    │  Hot indexes evict cold indexes                     │
    │  Cold index needed → DISK READ → latency spike      │
    │                                                     │
    │  With 8 indexes: ALL indexes fit in cache           │
    │  No eviction → consistent performance               │
    │                                                     │
    └────────────────────────────────────────────────────┘
    
    MONITORING CACHE PRESSURE:
    db.serverStatus().wiredTiger.cache
    → "bytes currently in the cache"
    → "maximum bytes configured"
    → "pages evicted" ← if high → cache pressure!
    → "pages read into cache" ← if high → cache misses!
    
    IDEAL STATE:
    → eviction rate near 0
    → read-into-cache rate low (most reads from cache)
    → cache utilization < 80%
    
    DANGER STATE:
    → eviction rate high (pages being thrown out)
    → read-into-cache rate high (pages being reloaded)
    → cache utilization > 90%
    → CAUSE: too many indexes competing for limited cache

REDSHIFT COMPARISON:
    → Zone maps: always in memory (tiny footprint)
    → RA3: data in managed storage, hot data cached automatically
    → No "too many zone maps" problem
    → Cache managed transparently by Redshift
    → No manual tuning needed

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE WITH INDEX OPTIMIZATION:

PITFALL 1: "Indexes are free — add more for every query"
    ❌ Coming from Redshift where zone maps have zero cost
    → Every MongoDB index has REAL cost: storage, writes, memory
    → 23 indexes on a write-heavy collection = performance disaster
    ✅ Think of indexes as EXPENSIVE resources — budget them carefully

PITFALL 2: "I'll just drop all indexes and recreate the right ones"
    ❌ Dropping all indexes at once on production
    → Immediate query performance collapse
    → All queries become COLLSCAN until new indexes built
    → Index builds on 10M docs take minutes → downtime
    ✅ Drop ONE at a time, monitor, then drop next
    → Or use hideIndex() first to test safely

PITFALL 3: "Compound index field order doesn't matter"
    ❌ { order_date: -1, customer_id: 1 } for customer lookups
    → customer_id is NOT a prefix → index useless for customer queries
    ✅ { customer_id: 1, order_date: -1 } → customer_id is prefix
    → Field ORDER in compound index is CRITICAL
    → Same as Redshift: COMPOUND SORTKEY(a, b) — order matters!

PITFALL 4: "I need a unique index on every candidate key"
    ❌ Unique indexes on email, phone, username, social_security, account_number
    → Each unique index = full B-tree + uniqueness check on EVERY write
    → 5 unique indexes = 5 uniqueness checks per insert
    → Uniqueness check requires index lookup → adds latency
    ✅ Only add unique indexes where BUSINESS LOGIC requires uniqueness
    → email: ✅ unique (login identifier)
    → phone: ❓ maybe (not all systems require unique phones)
    → account_number: ✅ unique (business-critical)
    → social_security: ❌ don't index at all (security risk — Project 38)
    → Validate uniqueness in APPLICATION for non-critical fields

PITFALL 5: "I'll use index intersection instead of compound indexes"
    ❌ Relying on MongoDB to combine { customer_id: 1 } + { status: 1 }
    → Index intersection = TWO B-tree walks + merge operation
    → 2-10x slower than ONE compound index walk
    → MongoDB query planner may NOT choose intersection
    ✅ Create compound index { customer_id: 1, status: 1 }
    → ONE B-tree walk → faster and more predictable
    → Intersection = fallback for rare query combinations only

PITFALL 6: "Partial indexes are too complex — I'll skip them"
    ❌ Full index on { promo_code: 1 } for 10M docs (95% null)
    → 9.5M useless index entries → wasted storage + write overhead
    ✅ Partial index with { promo_code: { $exists: true } }
    → Only 500K entries → 95% smaller → 95% less write overhead
    → Partial indexes are MongoDB's MOST UNDERUSED optimization
    → Redshift users miss this because zone maps have no equivalent

PITFALL 7: "I dropped an index and queries are still fast"
    ❌ Concluding the index was unnecessary
    → Query planner may have cached the old plan → still using it briefly
    → OR: data is still in WiredTiger cache → reads from memory
    → Wait 24-48 hours and check again under load
    ✅ Clear plan cache after dropping: db.orders.getPlanCache().clear()
    → Re-run explain("executionStats") to confirm new plan
    → Monitor under PRODUCTION load, not just test queries

PITFALL 8: "reIndex() will fix performance problems"
    ❌ Running reIndex() as a general "fix everything" command
    → reIndex() drops ALL indexes and rebuilds from scratch
    → Takes VERY long on large collections (hours)
    → Blocks writes during parts of the process
    → Rarely needed — B-trees self-balance
    ✅ Only use reIndex() when:
    → Index corruption suspected (very rare)
    → After massive deletes (>50% of collection removed)

    → Normal operations: indexes self-maintain

📈 SCALABILITY CONSIDERATIONS

HOW INDEX STRATEGY CHANGES WITH SCALE:

SMALL COLLECTION (<100K documents):
    → Almost any index strategy works
    → COLLSCAN on 100K docs = ~50ms (acceptable for many use cases)
    → Over-indexing has minimal impact (small indexes fit in cache)
    → RECOMMENDATION: 2-4 indexes, don't over-optimize

MEDIUM COLLECTION (100K - 10M documents):
    → Index strategy MATTERS — COLLSCAN = seconds
    → Most indexes fit in cache if count is reasonable (5-8)
    → THIS is the sweet spot for index optimization
    → RECOMMENDATION: 4-8 indexes, apply ESR rule, use partials

LARGE COLLECTION (10M - 100M documents):
    → Index strategy is CRITICAL — wrong indexes = minutes
    → Indexes may NOT fit in cache → careful selection required
    → Write amplification directly impacts throughput
    → RECOMMENDATION: 3-6 indexes maximum, aggressive partial indexes
    → Consider: sharding (Project 21-22) to distribute index load

VERY LARGE COLLECTION (100M+ documents):
    → Index builds take HOURS (background, but resource-intensive)
    → Index size can exceed data size if not managed
    → Working set management (Project 33) becomes critical
    → RECOMMENDATION: minimum viable indexes (3-5)
    → Use sharding: each shard has SMALLER indexes → fit in cache

SHARDED COLLECTION (Project 21-22 connection):

📐 DATA MODELING IMPACT ON INDEX STRATEGY

HOW DOCUMENT DESIGN AFFECTS INDEX COUNT:

NORMALIZED MODEL (Redshift habit):
    orders:     { customer_id, status, order_date, total }
    items:      { order_id, sku, qty, price }
    customers:  { customer_id, email, name }

    → items needs indexes for: order_id, sku
    → customers needs indexes for: customer_id, email

    orders: {

    → Project 14 (embed vs reference) directly impacts index budget

    → MongoDB: embedded → fewer $lookups → fewer indexes needed
    → Document design IS index optimization

def find_redundant_indexes(indexes):
    """Find indexes that are prefixes of other indexes"""
    
    print(f"\n  REDUNDANCY ANALYSIS:")
    
    # Extract key patterns (skip _id)
    patterns = []
    for idx in indexes:
        if idx["name"] != "_id_":
            keys = list(idx["key"].keys())
            patterns.append((idx["name"], keys, dict(idx["key"])))
    
    redundant = []
    
    for i, (name_a, keys_a, pattern_a) in enumerate(patterns):
        for j, (name_b, keys_b, pattern_b) in enumerate(patterns):
            if i == j:
                continue
            
            # Check if A is a prefix of B
            if len(keys_a) < len(keys_b):
                is_prefix = True
                for k in range(len(keys_a)):
                    if keys_a[k] != keys_b[k]:
                        is_prefix = False
                        break
                    # Check direction too
                    if list(pattern_a.values())[k] != list(pattern_b.values())[k]:
                        is_prefix = False
                        break
                
                if is_prefix:
                    redundant.append((name_a, name_b))
                    print(f"  ├── ❌ {name_a}")
                    print(f"  │      PREFIX OF: {name_b}")
                    print(f"  │      {pattern_a} ⊂ {pattern_b}")
    
    # Check direction duplicates
    seen_fields = {}
    for name, keys, pattern in patterns:
        field_key = tuple(keys)
        if field_key in seen_fields:
            other_name = seen_fields[field_key]
            print(f"  ├── ❌ {name}")
            print(f"  │      DIRECTION DUPLICATE OF: {other_name}")
            redundant.append((name, other_name))
        else:
            seen_fields[field_key] = name
    
    if not redundant:
        print(f"  └── ✅ No redundant indexes found")
    else:
        print(f"  └── Found {len(redundant)} redundant indexes")
    
    return redundant

        # Efficiency check
        efficiency = docs_examined / max(n_returned, 1)
        
        status = "✅" if not has_collscan else "❌"
        sort_status = " ⚠️SORT" if has_sort else ""
        
        print(f"\n  {status} {q['name']}{sort_status}")
        print(f"     Stages: {' → '.join(stages)}")
        print(f"     Returned: {n_returned} | Examined: {docs_examined} | Time: {time_ms}ms | Ratio: {efficiency:.1f}:1")
        
        if has_collscan:
            print(f"     ❌ REGRESSION: COLLSCAN detected — need index for this query!")
            all_passed = False
    
    if all_passed:
        print(f"\n  ✅ ALL CRITICAL QUERIES VERIFIED — optimization successful")
    else:
        print(f"\n  ❌ SOME QUERIES REGRESSED — review and add missing indexes")
    
    return all_passed

def monitoring_report(collection):
    """Generate monitoring report for ongoing index health"""
    
    print(f"\n{'='*70}")
    print(f"ONGOING MONITORING REPORT")
    print(f"{'='*70}")
    
    stats = db.command("collstats", collection.name)
    
    # WiredTiger cache stats
    server_status = db.command("serverStatus")
    cache = server_status.get("wiredTiger", {}).get("cache", {})
    
    cache_size = cache.get("maximum bytes configured", 0)
    cache_used = cache.get("bytes currently in the cache", 0)
    cache_dirty = cache.get("tracked dirty bytes in the cache", 0)
    pages_evicted = cache.get("unmodified pages evicted", 0)
    pages_read = cache.get("pages read into cache", 0)
    
    cache_pct = (cache_used / max(cache_size, 1)) * 100
    
    print(f"\n  WIREDTIGER CACHE:")
    print(f"  ├── Cache size:      {cache_size / 1024 / 1024:.0f} MB")
    print(f"  ├── Cache used:      {cache_used / 1024 / 1024:.0f} MB ({cache_pct:.1f}%)")
    print(f"  ├── Dirty bytes:     {cache_dirty / 1024 / 1024:.0f} MB")
    print(f"  ├── Pages evicted:   {pages_evicted:,}")
    print(f"  └── Pages read in:   {pages_read:,}")
    
    if cache_pct > 90:
        print(f"  ❌ CRITICAL: Cache > 90% — consider reducing indexes or scaling up")
    elif cache_pct > 75:
        print(f"  ⚠️ WARNING: Cache > 75% — monitor closely")
    else:
        print(f"  ✅ HEALTHY: Cache utilization normal")
    
    # Index health summary
    total_idx_size = stats.get("totalIndexSize", 0)
    data_size = stats.get("size", 0)
    
    print(f"\n  INDEX HEALTH:")
    print(f"  ├── Data size:       {data_size / 1024 / 1024:.1f} MB")
    print(f"  ├── Total index size:{total_idx_size / 1024 / 1024:.1f} MB")
    print(f"  ├── Index/data ratio:{total_idx_size / max(data_size, 1):.2f}x")
    print(f"  ├── Index count:     {stats.get('nindexes', 0)}")
    
    # Check if indexes fit in cache
    if total_idx_size < cache_size * 0.5:
        print(f"  └── ✅ Indexes fit in <50% of cache")
    elif total_idx_size < cache_size * 0.8:
        print(f"  └── ⚠️ Indexes using >50% of cache — monitor")
    else:
        print(f"  └── ❌ Indexes may not fit in cache — reduce index count")
    
    print(f"\n  RECOMMENDATIONS:")
    print(f"  → Run $indexStats weekly to find unused indexes")
    print(f"  → Target index/data ratio < 0.5x")
    print(f"  → Target 3-8 indexes per collection")
    print(f"  → Use Atlas Performance Advisor for automatic suggestions")
