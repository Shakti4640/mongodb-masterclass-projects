KEY REDSHIFT TYPE RULES:
    → Every value in a column MUST be the same type
    → NULL is the ONLY way to represent "no value"
    → No distinction between "value is null" and "field doesn't exist"
    → Type conversion is explicit: CAST(x AS INTEGER)
    → Type mismatch on INSERT = ERROR (strict enforcement)
    → Once column type is set: CANNOT change without recreating table

🟢 MONGODB MINDSET — BSON Type System

What is BSON and Why Does It Exist

BSON = Binary JSON

JSON:
    → Text format: {"price": 29.99}
    → No native types beyond: string, number, boolean, null, array, object
    → "number" is just "number" — no int vs float vs decimal
    → No date type — dates are strings: "2024-01-15T10:30:00Z"
    → No binary type — must base64 encode
    → Slow to parse (text scanning)

BSON:
    → Binary format: \x10 price \x00 \x9A\x99... (type-tagged, length-prefixed)
    → Rich type system: 20+ distinct types
    → Separate int32, int64, double, decimal128
    → Native Date type (milliseconds since epoch)
    → Native Binary type (raw bytes)
    → Native ObjectId type (12 bytes, globally unique)
    → O(1) field access (length-prefixed, skip to field)
    → Traversal without parsing entire document

WHY MONGODB USES BSON INSTEAD OF JSON:
    → Performance: binary is faster to read/write than text
    → Types: richer type system for real-world data
    → Size: BSON can be smaller (binary numbers) or larger (field names stored)
    → Traversal: can skip fields without parsing them
    → Ordering: fields maintain insertion order (JSON objects are unordered)

CRITICAL INSIGHT:
    → You WRITE JSON (in mongosh/pymongo)
    → MongoDB STORES BSON (on disk)
    → You READ JSON (returned to your application)
    → The conversion is automatic and invisible
    → But understanding BSON types prevents SUBTLE BUGS

TYPES MONGODB HAS THAT REDSHIFT DOESN'T:
    → ObjectId (self-describing unique ID with embedded timestamp)
    → Nested Object (first-class, not a workaround)
    → Array (first-class, indexable, queryable)
    → Regular Expression (store and query with regex)
    → Decimal128 (IEEE 754 128-bit — more precise than Redshift DECIMAL)
    → Min/Max Key (comparison boundary markers)

TYPES REDSHIFT HAS THAT MONGODB DOESN'T:
    → SMALLINT (MongoDB has no 16-bit integer — use Int32)
    → CHAR(n) (fixed-length string — MongoDB only has variable-length)
    → DATE without time (MongoDB Date always includes time)
    → TIME without date (MongoDB has no time-only type)
    → GEOMETRY/GEOGRAPHY (MongoDB uses GeoJSON instead — different approach)
    → HLLSKETCH (HyperLogLog — MongoDB has no built-in probabilistic type)

🧩 HOW TO APPROACH IT IN MONGODB

THINKING FRAMEWORK FOR TYPES:

REDSHIFT THINKING (what you're used to):
    "What SQL type fits this column?"
    → Pick from menu → apply to column → ALL rows use it → done forever

MONGODB THINKING (what you need to learn):
    "What BSON type should this field be in THIS document?"
    → Pick type per VALUE → different documents CAN differ → discipline required
    → The SAME field name can hold different types in different documents
    → This is POWER and DANGER simultaneously

EXAMPLE OF THE DANGER:
    Document 1: { price: 29.99 }          ← Double (number)
    Document 2: { price: "29.99" }         ← String (text!)
    Document 3: { price: NumberDecimal("29.99") } ← Decimal128 (exact)

    ALL THREE ARE VALID IN THE SAME COLLECTION
    → Sorting will be WRONG (strings sort lexicographically)
    → Aggregation $sum will SKIP strings
    → Comparison operators behave differently per type
    
    THIS NEVER HAPPENS IN REDSHIFT (type enforced at column level)
    THIS IS THE #1 DATA QUALITY TRAP IN MONGODB

TYPE 1: ObjectId — The Crown Jewel MongoDB Type

// ═══════════════════════════════════════════════════════
// ObjectId — No Redshift Equivalent
// ═══════════════════════════════════════════════════════

// Redshift: BIGINT IDENTITY(1,1) — sequential, node-specific
// MongoDB:  ObjectId — 12 bytes, globally unique, timestamp-embedded

// ANATOMY OF AN ObjectId (12 bytes = 24 hex characters):
//
//  665a1b2c  3d4e5f  6a7b  8c9d0e
//  ────────  ──────  ────  ──────
//  │         │       │     │
//  │         │       │     └── 3 bytes: incrementing counter
//  │         │       │         (per process, starts random)
//  │         │       │
//  │         │       └──────── 2 bytes: process ID (PID)
//  │         │                 (which mongod/driver process)
//  │         │
//  │         └──────────────── 5 bytes: random value
//  │                           (per machine, generated once)
//  │
//  └────────────────────────── 4 bytes: UNIX timestamp (seconds)
//                               (when this ObjectId was created)

// WHY THIS DESIGN IS BRILLIANT:
//   → Globally unique WITHOUT coordination between servers
//   → Redshift IDENTITY needs a single sequence → bottleneck in distributed
//   → Contains creation timestamp → free metadata
//   → Roughly time-ordered → good default index behavior
//   → 12 bytes → compact compared to UUID (16 bytes)

// mongosh — ObjectId operations:

// Auto-generated on insert (Project 1 covered this)
db.sensors.insertOne({ name: "Temp-001" })

// Examine the ObjectId
let doc = db.sensors.findOne()
doc._id
// ObjectId('665a1b2c3d4e5f6a7b8c9d0e')

// Extract timestamp
doc._id.getTimestamp()
// ISODate('2024-05-31T14:30:22.000Z')

// Create ObjectId manually
let customId = ObjectId()                           // new, current time
let specificId = ObjectId("665a1b2c3d4e5f6a7b8c9d0e")  // from string

// Create ObjectId from specific timestamp
// (useful for time-range queries on _id!)
let fromTime = ObjectId(Math.floor(new Date("2024-01-01").getTime() / 1000).toString(16) + "[REDACTED:CREDIT_DEBIT_NUMBER]")

// QUERY BY TIME USING _id (no separate timestamp column needed!)
db.sensors.find({
    _id: { $gt: ObjectId("65912c000000000000000000") }  // after specific time
})

// ─── REDSHIFT EQUIVALENT ──────────────────────────────
// -- No equivalent to ObjectId timestamp extraction
// -- Closest: BIGINT IDENTITY(1,1) + created_at TIMESTAMP
// -- But IDENTITY gives you NO time info, NO global uniqueness
// -- Two Redshift clusters can generate same IDENTITY value
# pymongo — ObjectId operations

from bson import ObjectId
from datetime import datetime, timezone

# Auto-generated
oid = ObjectId()
print(f"ObjectId: {oid}")                  # 665a1b2c3d4e5f6a7b8c9d0e
print(f"Timestamp: {oid.generation_time}")  # 2024-05-31 14:30:22+00:00
print(f"As string: {str(oid)}")            # 665a1b2c3d4e5f6a7b8c9d0e
print(f"Is valid: {ObjectId.is_valid(str(oid))}")  # True

# Create from string (e.g., from API request)
oid_from_str = ObjectId("665a1b2c3d4e5f6a7b8c9d0e")

# Create from datetime (for time-range queries)
target_time = datetime(2024, 1, 1, tzinfo=timezone.utc)
oid_from_time = ObjectId.from_datetime(target_time)

# Query: "all documents created after 2024-01-01"
readings_after = db.sensor_readings.find({
    "_id": {"$gt": oid_from_time}
})
# This uses the _id index → FAST
# No separate timestamp index needed for creation-time queries

TYPE 2: Numbers — The Float Trap That Destroys Financial Data

// ═══════════════════════════════════════════════════════
// Numbers in MongoDB — THREE types where Redshift has FIVE
// ═══════════════════════════════════════════════════════

// REDSHIFT NUMBERS:
// SMALLINT → 2 bytes → -32768 to 32767
// INTEGER  → 4 bytes → -2.1B to 2.1B
// BIGINT   → 8 bytes → -9.2 quintillion to 9.2 quintillion
// DECIMAL  → variable → exact numeric up to 38 digits
// DOUBLE   → 8 bytes → approximate floating point

// MONGODB BSON NUMBERS:
// Int32    → 4 bytes → same as INTEGER
// Int64    → 8 bytes → same as BIGINT
// Double   → 8 bytes → same as DOUBLE PRECISION
// Decimal128→ 16 bytes → MORE precise than DECIMAL(38,s)

// ─── THE FLOAT TRAP ──────────────────────────────────
// THIS IS THE #1 MISTAKE EVERY DEVELOPER MAKES

// In mongosh: bare numbers are DOUBLES by default!
db.financial.insertOne({
    product: "Widget",
    price: 19.99            // ← THIS IS A DOUBLE, NOT DECIMAL!
})

// What's stored:
db.financial.findOne().price
// 19.99
// Looks fine, right? WRONG.

// The actual stored value (IEEE 754 double):
// 19.989999999999998 (approximately)
// For display it rounds to 19.99
// But MATH will expose the imprecision:

db.financial.insertMany([
    { item: "A", price: 0.1 },
    { item: "B", price: 0.2 }
])

db.financial.aggregate([
    { $group: { _id: null, total: { $sum: "$price" } } }
])
// total: 0.30000000000000004  ← NOT 0.3!
// THIS IS THE IEEE 754 FLOATING POINT PROBLEM
// Redshift DECIMAL(10,2) would give you exactly 0.30

// ─── THE SOLUTION: Decimal128 ─────────────────────────
db.financial_correct.insertMany([
    { item: "A", price: NumberDecimal("0.1") },    // ← STRING argument!
    { item: "B", price: NumberDecimal("0.2") }
])

db.financial_correct.aggregate([
    { $group: { _id: null, total: { $sum: "$price" } } }
])
// total: Decimal128("0.3")  ← EXACT!

// ─── INTEGER TYPES ────────────────────────────────────
// mongosh defaults:
//   42       → Double (not integer!)
//   42.0     → Double

// Force integer types:
NumberInt(42)       // → Int32 (4 bytes) — like Redshift INTEGER
NumberLong(42)      // → Int64 (8 bytes) — like Redshift BIGINT
NumberDecimal("42") // → Decimal128 (16 bytes) — like Redshift DECIMAL

// WHY THIS MATTERS:
// db.counters.insertOne({ count: 42 })         → stored as Double
// db.counters.insertOne({ count: NumberInt(42) }) → stored as Int32
// Both "look" the same but behave differently in:
//   → Storage size (8 bytes vs 4 bytes)
//   → Type comparison/sorting
//   → Schema validation checks
# pymongo — Number types handled correctly

from bson import Decimal128, Int64
from decimal import Decimal

# Python automatically maps types:
# Python int → BSON Int32 (if fits) or Int64 (if large)
# Python float → BSON Double
# Decimal128 → BSON Decimal128

# ─── CORRECT financial data insertion ──────────────────
db.financial.insert_one({
    "item": "Widget",
    "price": Decimal128("19.99"),          # EXACT decimal — like DECIMAL(10,2)
    "quantity": 42,                         # Python int → BSON Int32
    "total_revenue": Int64(1_000_000_000),  # Explicit Int64 → BSON Int64
    "tax_rate": Decimal128("0.0825"),       # EXACT — no float imprecision
})

# ─── WRONG (common mistake from Redshift developers) ──
db.financial.insert_one({
    "item": "Widget",
    "price": 19.99,          # Python float → BSON Double → IMPRECISE
    "tax_rate": 0.0825,      # Python float → BSON Double → IMPRECISE
})

# ─── Python Decimal → Decimal128 conversion ───────────
python_decimal = Decimal("19.99")
bson_decimal = Decimal128(python_decimal)   # Convert Python Decimal to BSON

# ─── REDSHIFT EQUIVALENT ──────────────────────────────
# -- Redshift: no ambiguity — DECIMAL(10,2) is always exact
# -- INSERT INTO products (price) VALUES (19.99);
# -- The column TYPE guarantees precision
# -- In MongoDB: YOU must choose the right type per insert

TYPE 3: Dates — Where Redshift Developers Get Burned

// ═══════════════════════════════════════════════════════
// Date/Time — MongoDB Date vs Redshift TIMESTAMP
// ═══════════════════════════════════════════════════════

// REDSHIFT DATE/TIME TYPES:
//   DATE          → date only (no time): '2024-01-15'
//   TIME          → time only (no date): '14:30:00'
//   TIMESTAMP     → date + time without timezone
//   TIMESTAMPTZ   → date + time with timezone (stored as UTC)
//   INTERVAL      → time duration: '3 days 4 hours'

// MONGODB DATE TYPE:
//   Date          → THAT'S IT — ONE type for everything
//                 → Milliseconds since Unix epoch (Jan 1, 1970 UTC)
//                 → 64-bit integer internally
//                 → ALWAYS includes both date AND time
//                 → ALWAYS stored in UTC
//                 → No DATE-only type
//                 → No TIME-only type
//                 → No INTERVAL type
//                 → Millisecond precision (not microsecond like Redshift)

// ─── CREATING DATES IN MONGOSH ───────────────────────

// Current date/time
new Date()                          // ISODate('2024-05-31T14:30:22.123Z')
ISODate()                           // Same thing — shorthand

// Specific date/time
new Date("2024-01-15")              // ISODate('2024-01-15T00:00:00.000Z')
new Date("2024-01-15T10:30:00Z")    // ISODate('2024-01-15T10:30:00.000Z')
ISODate("2024-01-15T10:30:00Z")     // Same thing

// From epoch milliseconds
new Date(1705312200000)             // ISODate('2024-01-15T10:30:00.000Z')

// ─── THE DATE-ONLY TRAP ──────────────────────────────
// Redshift: DATE '2024-01-15' has NO time component
// MongoDB:  new Date("2024-01-15") → 2024-01-15T00:00:00.000Z (midnight UTC!)

// This causes query issues:
// "Find all readings on January 15, 2024"

// REDSHIFT (simple):
// WHERE reading_date = DATE '2024-01-15'

// MONGODB (must use range):
db.readings.find({
    reading_time: {
        $gte: ISODate("2024-01-15T00:00:00Z"),
        $lt:  ISODate("2024-01-16T00:00:00Z")    // exclusive upper bound
    }
})
// You CANNOT just do: { reading_time: ISODate("2024-01-15") }
// That only matches documents at EXACTLY midnight!

// ─── TIMEZONE TRAP ───────────────────────────────────
// MongoDB stores ALL dates in UTC internally
// If you insert a local time without timezone:
new Date("2024-01-15T10:30:00")     // Interpreted as LOCAL timezone of server/client
new Date("2024-01-15T10:30:00Z")    // Explicitly UTC
new Date("2024-01-15T10:30:00-05:00") // EST → converted to UTC → stored as 15:30:00Z

// REDSHIFT comparison:
// TIMESTAMP: no timezone awareness (stored as-is)
// TIMESTAMPTZ: converted to UTC on storage (same as MongoDB)

// ─── DATE ARITHMETIC ─────────────────────────────────
// Redshift: DATEADD(day, 7, reading_time)
//           DATEDIFF(hour, start_time, end_time)
//           reading_time + INTERVAL '7 days'

// MongoDB aggregation:
db.readings.aggregate([
    {
        $project: {
            reading_time: 1,
            // Add 7 days
            next_week: {
                $dateAdd: {
                    startDate: "$reading_time",
                    unit: "day",
                    amount: 7
                }
            },
            // Difference in hours
            hours_ago: {
                $dateDiff: {
                    startDate: "$reading_time",
                    endDate: "$$NOW",           // current time
                    unit: "hour"
                }
            },
            // Extract parts
            year:  { $year: "$reading_time" },
            month: { $month: "$reading_time" },
            day:   { $dayOfMonth: "$reading_time" },
            hour:  { $hour: "$reading_time" },
            dow:   { $dayOfWeek: "$reading_time" }    // 1=Sunday
        }
    }
])

// ─── REDSHIFT EQUIVALENT ──────────────────────────────
// SELECT
//     reading_time,
//     DATEADD(day, 7, reading_time) AS next_week,
//     DATEDIFF(hour, reading_time, GETDATE()) AS hours_ago,
//     EXTRACT(YEAR FROM reading_time) AS year,
//     EXTRACT(MONTH FROM reading_time) AS month,
//     EXTRACT(DAY FROM reading_time) AS day,
//     EXTRACT(HOUR FROM reading_time) AS hour,
//     EXTRACT(DOW FROM reading_time) AS dow   -- 0=Sunday
// FROM iot.sensor_readings;
# pymongo — Date handling

from datetime import datetime, timezone, timedelta
from bson.codec_options import CodecOptions
import pytz

# ─── INSERTING DATES ──────────────────────────────────
# Python datetime → BSON Date automatically

db.readings.insert_one({

    "scheduled_maintenance": specific,
})

# DANGER: datetime without timezone
naive = datetime(2024, 1, 15, 10, 30, 0)   # NO timezone info!
# pymongo will store this but it's AMBIGUOUS
# Is this UTC? Local time? Server time?
# ALWAYS use timezone-aware datetimes:
aware = datetime(2024, 1, 15, 10, 30, 0, tzinfo=timezone.utc)

end = datetime(2024, 1, 16, tzinfo=timezone.utc)

readings = db.readings.find({

# ─── PRECISION DIFFERENCE ─────────────────────────────
# MongoDB Date: millisecond precision (3 decimal places)
# Redshift TIMESTAMP: microsecond precision (6 decimal places)
# If your data has microsecond precision → MongoDB truncates!
# Example:
#   Redshift:  2024-01-15 10:30:00.123456 (microseconds preserved)
#   MongoDB:   2024-01-15 10:30:00.123     (microseconds LOST)

TYPE 4: GeoJSON — Geospatial Data (MongoDB's Built-In Superpower)

// ═══════════════════════════════════════════════════════
// GeoJSON — Redshift GEOMETRY vs MongoDB Native Geo
// ═══════════════════════════════════════════════════════

// REDSHIFT GEOSPATIAL:
//   → GEOMETRY type: stores points, lines, polygons
//   → Limited functions: ST_Distance, ST_Contains, ST_Within
//   → No geospatial INDEX — must scan all rows
//   → Useful but bolted-on — not a core strength

// MONGODB GEOSPATIAL:
//   → GeoJSON format: standard format for geographic data
//   → 2dsphere index: enables geo queries with INDEX support
//   → $near, $geoWithin, $geoIntersects — rich query operators
//   → Built into the query engine — first-class citizen

// ─── STORING LOCATION ────────────────────────────────

// GeoJSON Point format (MUST follow this structure exactly):
db.sensors.insertOne({
    sensor_id: "TEMP-NYC-001",
    name: "Times Square Sensor",
    location: {
        type: "Point",                         // ← REQUIRED
        coordinates: [-73.9857, 40.7484]       // ← [longitude, latitude]
    },                                         //    NOT [lat, lng]!
    temperature_c: 22.5
})

// CRITICAL: coordinates are [LONGITUDE, LATITUDE]
//   → GeoJSON standard = [lng, lat]
//   → Google Maps shows [lat, lng]
//   → Swapping them = your queries return wrong results
//   → THIS IS THE #1 GEO MISTAKE

// ─── CREATE GEOSPATIAL INDEX ─────────────────────────
db.sensors.createIndex({ location: "2dsphere" })

// ─── GEO QUERIES ─────────────────────────────────────

// Find sensors within 5km of a point
db.sensors.find({
    location: {
        $near: {
            $geometry: {
                type: "Point",
                coordinates: [-73.9857, 40.7484]   // Times Square
            },
            $maxDistance: 5000       // 5000 meters = 5km
        }
    }
})
// Results sorted by distance (nearest first) — uses 2dsphere index

// Find sensors within a polygon (e.g., Manhattan boundary)
db.sensors.find({
    location: {
        $geoWithin: {
            $geometry: {
                type: "Polygon",
                coordinates: [[
                    [-74.0479, 40.6829],    // SW corner
                    [-73.9067, 40.6829],    // SE corner

// THE MOST CONFUSING DIFFERENCE FOR REDSHIFT DEVELOPERS
// ═══════════════════════════════════════════════════════

// REDSHIFT: Only TWO states for any column value
//   1. HAS a value:  humidity = 45.2
//   2. IS NULL:      humidity = NULL
//   That's it. Every row has every column. Period.

// MONGODB: THREE states for any field
//   1. HAS a value:        { humidity: 45.2 }
//   2. IS EXPLICITLY NULL: { humidity: null }
//   3. FIELD IS MISSING:   { } (humidity field doesn't exist at all)

⚡ BSON TYPE COMPARISON ORDER — Why Sorting Behaves Differently

// ═══════════════════════════════════════════════════════
// BSON COMPARISON ORDER — Critical for sort() and indexes
// ═══════════════════════════════════════════════════════

// REDSHIFT: Sorting is simple — each column has ONE type
//           Numbers sort numerically, strings lexicographically
//           NULL sorts as lowest (or highest with NULLS LAST)

// MONGODB: A field can contain DIFFERENT types across documents
//          MongoDB defines a STRICT ordering between types:

// BSON COMPARISON ORDER (lowest to highest):
//   1.  MinKey (internal)
//   2.  Null
//   3.  Numbers (Int32, Int64, Double, Decimal128 — compared as numbers)
//   4.  Symbol (deprecated)
//   5.  String
//   6.  Object (compared field by field)
//   7.  Array (compared element by element)
//   8.  BinData (by length, then bytes)
//   9.  ObjectId
//   10. Boolean (false < true)
//   11. Date
//   12. Timestamp (internal)
//   13. Regular Expression
//   14. MaxKey (internal)

// WHAT THIS MEANS IN PRACTICE:
db.mixed.insertMany([
    { val: 42 },              // number
    { val: "42" },            // string
    { val: null },            // null
    { val: true },            // boolean
    { val: new Date() },      // date
    { val: ObjectId() }       // objectid
])

db.mixed.find().sort({ val: 1 })
// ORDER:
//   null          ← nulls first (like Redshift NULLS FIRST)
//   42            ← numbers before strings
//   "42"          ← strings after numbers
//   ObjectId(...) ← objectid after strings
//   true          ← boolean after objectid
//   Date(...)     ← date after boolean

// THIS WOULD NEVER HAPPEN IN REDSHIFT
// Because a column cannot have mixed types!
// But in MongoDB it CAN happen if you're not careful
// This is why type discipline matters even without enforcement

# ═══════════════════════════════════════════════════════
# EQUIVALENT REDSHIFT SQL (for reference)
# ═══════════════════════════════════════════════════════
#
# -- Redshift: ALL types declared upfront, fixed per column
#
# CREATE TABLE iot.sensors (
#     sensor_id           VARCHAR(50) PRIMARY KEY,
#     name                VARCHAR(255),
#     type                VARCHAR(50),
#     -- No GeoJSON — must use separate lat/lng columns
#     longitude           DOUBLE PRECISION,
#     latitude            DOUBLE PRECISION,
#     -- OR use GEOMETRY:
#     -- location         GEOMETRY,
#     installed_at        TIMESTAMPTZ,
#     firmware_version    VARCHAR(20),
#     firmware_sig        VARBYTE(256),
#     is_active           BOOLEAN,
#     -- Cannot nest config as object — must flatten:
#     config_interval_sec INTEGER,
#     config_threshold_c  DOUBLE PRECISION,
#     config_humidity_en  BOOLEAN,            -- NULL when not applicable
#     -- Cannot store array of tags — must normalize:
#     -- tags would be a separate table: sensor_tags(sensor_id, tag)
#     maintenance_cost    DECIMAL(10,2),
#     deactivated_at      TIMESTAMPTZ,         -- NULL for active sensors
#     deactivation_reason VARCHAR(255)          -- NULL for active sensors
# )
# DISTSTYLE KEY DISTKEY(sensor_id) SORTKEY(installed_at);
#
# -- Separate table for tags (normalized)
# CREATE TABLE iot.sensor_tags (
#     sensor_id   VARCHAR(50) REFERENCES iot.sensors(sensor_id),
#     tag         VARCHAR(50)
# );
#
# CREATE TABLE iot.sensor_readings (
#     reading_id          BIGINT IDENTITY(1,1),
#     sensor_id           VARCHAR(50),
#     reading_time        TIMESTAMPTZ,          -- microsecond precision
#     temperature_c       DOUBLE PRECISION,
#     humidity_pct        DOUBLE PRECISION,      -- NULL = no value (no "missing" concept)
#     electricity_cost    DECIMAL(10,4),          -- exact decimal always
#     reading_quality     INTEGER,
#     anomaly_detected    BOOLEAN,
#     raw_payload         VARBYTE(1024),
#     -- Cannot nest metadata — must flatten:
#     battery_pct         INTEGER,
#     signal_strength_dbm INTEGER,
#     metadata_firmware   VARCHAR(20),
#     -- Cannot have optional anomaly_details — must flatten with NULLs:
#     anomaly_type        VARCHAR(50),            -- NULL for non-anomalies
#     anomaly_deviation   DOUBLE PRECISION,       -- NULL for non-anomalies
#     -- Cannot store error_codes array — must normalize:
#     error_msg           VARCHAR(500)
# )
# DISTSTYLE KEY DISTKEY(sensor_id) SORTKEY(reading_time);
#
# -- Separate table for error codes (normalized)
# CREATE TABLE iot.reading_errors (
#     reading_id  BIGINT REFERENCES iot.sensor_readings(reading_id),
#     error_code  INTEGER
# );
#
# -- NOTICE:
# -- Redshift needs 5 tables for what MongoDB does in 2 collections
# -- Redshift has ~15 NULL-able columns for polymorphic data
# -- Redshift cannot store nested objects or arrays natively
# -- Redshift SUPER type helps but is a workaround, not native
# -- Every query that needs tags or error codes requires a JOIN
#
# -- Geospatial query in Redshift (no index):
# SELECT sensor_id, name
# FROM iot.sensors
# WHERE ST_DistanceSphere(
#     ST_SetSRID(ST_Point(longitude, latitude), 4326),
#     ST_SetSRID(ST_Point(-73.9857, 40.7484), 4326)
# ) < 10000;
# -- Full table scan — no geospatial index available
#
# -- Date range in Redshift:
# SELECT sensor_id, reading_time, temperature_c
# FROM iot.sensor_readings
# WHERE reading_time >= '2024-05-31 14:00:00+00'
#   AND reading_time <  '2024-05-31 14:02:00+00';
# -- Sort key on reading_time → efficient range scan

⚙️ HOW IT ONLY WORKS INTERNALLY — Type Storage

MongoDB (WiredTiger BSON Storage)

WHEN YOU INSERT: { temperature_c: 28.5, electricity_cost: Decimal128("0.0234") }

BSON ENCODING ON DISK:

  Field: temperature_c
  ┌──────┬─────────────────┬──────────────┐
  │ 0x01 │ temperature_c\0 │ 8-byte IEEE  │
  │ type │ field name       │ 754 double   │
  │ =dbl │ (null-terminated)│ = 28.5       │
  └──────┴─────────────────┴──────────────┘
  Total: 1 + 14 + 8 = 23 bytes

  Field: electricity_cost
  ┌──────┬──────────────────────┬───────────────┐
  │ 0x13 │ electricity_cost_kwh │ 16-byte IEEE  │
  │ type │ \0                   │ 754 Decimal128│
  │ =d128│ (null-terminated)    │ = 0.0234      │
  └──────┴──────────────────────┴───────────────┘
  Total: 1 + 21 + 16 = 38 bytes

KEY INSIGHT:
  → BSON stores TYPE TAG (1 byte) per field per document
  → BSON stores FIELD NAME per field per document
  → This is why MongoDB documents are larger than Redshift rows
  → Field names are repeated in EVERY document
  → Short field names = less storage (but don't sacrifice readability)
  → WiredTiger compresses this on disk (snappy/zstd)
  → Compression eliminates most of the field name repetition

Redshift (Columnar Storage)

WHEN YOU INSERT: temperature_c = 28.5, electricity_cost = 0.0234

COLUMNAR ENCODING ON DISK:

  Column: temperature_c (DOUBLE PRECISION)
  ┌──────────────────────────────────────────────────────┐
  │ Block 001: [28.5, 19.2, 85.3, 22.1, ... 130,000 values] │
  │ Zone Map: min=19.2, max=85.3                          │
  │ Encoding: RAW (doubles don't compress well)            │
  │ Size: ~1MB per block                                   │
  └──────────────────────────────────────────────────────┘

  Column: electricity_cost (DECIMAL(10,4))
  ┌──────────────────────────────────────────────────────┐
  │ Block 001: [0.0234, 0.0312, 0.0234, ... 130,000 vals]│
  │ Zone Map: min=0.0234, max=0.0500                      │
  │ Encoding: DELTA (small variations → great compression) │
  │ Size: ~200KB compressed (from 1MB raw)                 │
  └──────────────────────────────────────────────────────┘

KEY INSIGHT:
  → Redshift stores NO field names per value (column = name)
  → Redshift stores NO type tags per value (column = type)
  → MUCH more compact for large datasets
  → Column-specific compression: each column gets optimal encoding
  → Zone maps: min/max per block → skip entire blocks
  → THIS is why Redshift scans billions of values faster
  → But: reconstructing ONE complete "document" requires reading ALL columns

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 8 SPECIFIC:

PITFALL 1: "I'll just use regular numbers for money"
    ❌ { price: 19.99 }                  → Double → imprecise
    ✅ { price: NumberDecimal("19.99") }  → Decimal128 → exact
    WHY: Redshift DECIMAL column auto-enforces precision
         MongoDB requires you to CHOOSE the right type every time
    SEVERITY: HIGH — financial calculations will be silently wrong

PITFALL 2: "NULL means the same thing everywhere"
    ❌ Treating { humidity: null } and missing humidity as identical
    ✅ Understanding THREE states: value, null, missing
    WHY: Redshift has only two states (value or NULL)
         MongoDB has three — and { field: null } matches TWO of them
    SEVERITY: HIGH — queries return unexpected results

PITFALL 3: "Dates are dates, what could go wrong?"
    ❌ Storing dates as strings: "2024-01-15"
    ❌ Using naive datetimes (no timezone)
    ❌ Expecting microsecond precision (MongoDB = millisecond)
    ❌ Querying { date: ISODate("2024-01-15") } expecting all-day match
    ✅ Always use ISODate/datetime with UTC timezone
    ✅ Use $gte/$lt range for "all records on a day"
    SEVERITY: MEDIUM — queries miss data or return wrong results

PITFALL 4: "Coordinates are [lat, lng] like Google Maps"
    ❌ coordinates: [40.7484, -73.9857]    → [lat, lng] = WRONG
    ✅ coordinates: [-73.9857, 40.7484]    → [lng, lat] = CORRECT
    WHY: GeoJSON standard uses [longitude, latitude]
         Google Maps and most UIs show [latitude, longitude]
    SEVERITY: HIGH — geo queries return completely wrong results

PITFALL 5: "Numbers in mongosh are integers by default"
    ❌ { count: 42 }           → actually stored as Double in mongosh!
    ✅ { count: NumberInt(42) } → stored as Int32
    WHY: mongosh (JavaScript) treats all numbers as doubles
         Python pymongo correctly maps int → Int32/Int64
    SEVERITY: LOW — usually works but wastes storage and fails type checks

PITFALL 6: "I'll use different types for the same field"
    ❌ Doc 1: { status: 1 }       → Int
       Doc 2: { status: "active" } → String
       Doc 3: { status: true }     → Boolean
    ✅ Pick ONE type per field and enforce via schema validation
    WHY: Sorting across types follows BSON comparison order (not intuitive)
         Indexes work per-type → mixed types = poor index utilization
    SEVERITY: HIGH — queries and sorting produce nonsensical results

📈 SCALABILITY CONSIDERATIONS — Type Impact

TYPE CHOICES AFFECT SCALABILITY:

1. DOCUMENT SIZE
   → Each document max 16MB
   → Field names stored in EVERY document
   → Short names save space: "t" vs "temperature_celsius"
   → But: readability matters more than bytes (compression handles it)
   → EXCEPTION: millions of tiny documents → short names matter

2. INDEX SIZE
   → Index entries include the indexed value
   → Decimal128 (16 bytes) vs Double (8 bytes) → 2x index size
   → String indexes: value length directly affects index size
   → ObjectId (12 bytes) is compact for _id index

3. WORKING SET (WiredTiger Cache)
   → Smaller documents = more documents fit in cache
   → More cache hits = faster queries
   → Choose smallest appropriate type:
     → Use Int32 (4B) instead of Int64 (8B) when values fit
     → Use Double (8B) instead of Decimal128 (16B) when precision doesn't matter
   → BUT: correctness > performance — never sacrifice precision for speed

4. ARRAY GROWTH
   → Arrays that grow unbounded = document growth = reallocation
   → WiredTiger must rewrite entire document when it grows beyond its page
   → Redshift doesn't have this problem (columnar — each column grows independently)
   → MongoDB solution: Bucket Pattern (Project 16), Outlier Pattern (Project 17)

5. GEOSPATIAL INDEX SIZE
   → 2dsphere indexes are larger than regular B-tree indexes
   → Only create geo indexes on collections that need geo queries
   → Redshift: no geo index overhead (also no geo index benefit)
