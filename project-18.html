PROJECT 18: Polymorphic Pattern — One Collection, Many Shapes

Problem: Insurance System Where Auto, Home, and Life Policies Have Completely Different Fields

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    You're building an insurance management platform
    
    THE BUSINESS ENTITIES:
    ├── AUTO policies:  vehicle info, VIN, driver history, liability limits
    ├── HOME policies:  property address, square footage, roof age, flood zone
    ├── LIFE policies:  beneficiaries, health conditions, term length, face value
    ├── TRAVEL policies: trip dates, destinations, coverage type, companions
    └── PET policies:   animal type, breed, age, pre-existing conditions
    
    SHARED FIELDS (all policies have):
    ├── policy_number, customer_id, status, premium, start_date, end_date
    ├── agent_id, created_at, updated_at
    └── coverage_amount, deductible
    
    TYPE-SPECIFIC FIELDS:
    ├── AUTO: vin, make, model, year, mileage, driver_license, liability_limit,
    │         collision_coverage, comprehensive, uninsured_motorist
    ├── HOME: property_address, square_feet, year_built, roof_type, roof_age,
    │         flood_zone, has_security_system, has_pool, dwelling_coverage
    ├── LIFE: beneficiaries[], face_value, term_years, health_class,
    │         smoker, medical_conditions[], cash_value
    ├── TRAVEL: trip_start, trip_end, destinations[], coverage_type,
    │           companions[], pre_existing_waiver, cancel_for_any_reason
    └── PET: animal_type, breed, pet_name, pet_age, species,
             pre_existing[], wellness_coverage, accident_only
    
    ACCESS PATTERNS:
    ├── PATTERN A: "Get ALL policies for customer X" (agent dashboard)
    ├── PATTERN B: "Get details of policy P-12345" (policy detail page)
    ├── PATTERN C: "Find all auto policies with VIN matching..." (fraud check)
    ├── PATTERN D: "All policies expiring in next 30 days" (renewal workflow)
    ├── PATTERN E: "Premium analytics by policy type by region" (management)
    └── PATTERN F: "Search across ALL policy types by customer name" (search)
    
    THE CORE QUESTION:
    → These entities share SOME fields but differ in MANY fields
    → How do you store them?
    → In Redshift: painful decision between multiple tables or one wide table
    → In MongoDB: polymorphic pattern — one collection, many shapes

🔴 REDSHIFT MINDSET — How You Would Solve This

Option A: Multiple Tables (Normalized — The "Correct" Approach)

-- PROBLEMS:
-- → LEFT JOIN with ALL type tables (5+ JOINs as types grow)
-- → Most columns are NULL (auto policy has NULL home/life fields)
-- → Adding new policy type = CREATE TABLE + modify ALL queries
-- → Every query must know about ALL policy types
-- → Result set has sparse columns (confusing for BI tools)
-- → Performance: ~200-500ms (multiple JOINs)

-- PROBLEMS:
-- → 40+ columns, most NULL for any given row
-- → Can't store arrays (beneficiaries, destinations, conditions)
--   → STILL need separate tables for those
-- → Adding new policy type = ALTER TABLE ADD COLUMN × N
-- → Confusing: which columns belong to which type?
-- → Wasted storage for NULL columns (Redshift handles NULLs OK but messy)
-- → No type safety: nothing prevents setting VIN on a life policy
-- → Schema documentation becomes critical (which fields for which type?)

Redshift Pain Points Summary

REDSHIFT STRUGGLES WITH POLYMORPHIC DATA BECAUSE:

1. RIGID SCHEMA
   → Every column defined upfront for ALL rows
   → NULL columns for inapplicable fields
   → ALTER TABLE for every new policy type or new field

2. NO NESTED OBJECTS
   → beneficiaries[], destinations[], medical_conditions[]
   → Each needs a SEPARATE TABLE + JOIN
   → 5 policy types × 2 array fields each = 10+ tables

3. NO TYPE-SPECIFIC VALIDATION
   → Nothing prevents setting VIN on a life policy
   → Application must enforce which columns apply to which type
   → Database schema doesn't express the type relationship

4. QUERY COMPLEXITY
   → "All policies for customer" = JOIN with ALL type tables
   → Every new policy type = modify ALL queries
   → BI tools see 40+ columns, most irrelevant for any given row

5. SCHEMA EVOLUTION
   → New policy type (e.g., "cyber") = ALTER TABLE ADD 10 columns
   → Or CREATE new type table + modify all queries
   → Either way: painful schema migration

🟢 MONGODB MINDSET — The Polymorphic Pattern

Core Concept

THE POLYMORPHIC PATTERN:
    → ONE collection stores ALL policy types
    → Each document has a DISCRIMINATOR FIELD (policy_type)
    → Shared fields present in ALL documents
    → Type-specific fields present ONLY in documents of that type
    → Each document carries EXACTLY the fields it needs — no more, no less
    
    THIS IS MONGODB'S NATURAL STRENGTH:
    → Flexible schema = different documents can have different fields
    → No NULLs for inapplicable fields
    → No ALTER TABLE for new types
    → No separate tables for nested data (arrays, objects)
    → No JOINs to reconstruct a complete policy
    
    THE DISCRIMINATOR FIELD:
    → policy_type: "auto" | "home" | "life" | "travel" | "pet"
    → Application reads policy_type first
    → Then knows which fields to expect
    → Like a "class" or "interface" in object-oriented programming
    → Redshift has NO equivalent — every row has same columns

       → Required fields for all types

       → Application-level validation for type-specific rules
    
    ✅ Embed type-specific nested data directly
       → beneficiaries[] inside life policy document
       → destinations[] inside travel policy document

    ✅ Design projection per type for list views
       → Dashboard shows summary: type-specific one-liner per policy
       → Detail page shows everything for one policy type

       → Keep schema validation for shared field requirements
       → Use app code for type-specific field validation

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "One Collection Per Type" (Redshift Multi-Table Habit)
    ❌ Creating: db.policies_auto, db.policies_home, db.policies_life
    SYMPTOMS:
        → "All policies for customer" needs 5+ separate queries
        → Cross-type analytics requires $unionWith (complex)
        → Index management multiplied across collections
        → Code duplication for shared field operations
    ROOT CAUSE:
        → Redshift brain: "one table per entity type"
        → MongoDB polymorphic: "one collection, many shapes"
    FIX: Single policies collection with policy_type discriminator
         → ONE query returns all types
         → Shared field indexes work across all types
         → Partial indexes handle type-specific fields

PITFALL 2: "Adding NULL Fields for Missing Type Data"
    ❌ Every policy document has ALL fields:
       {
           policy_type: "auto",
           vin: "1HGBH...",
           flood_zone: null,        // ← NOT an auto field
           face_value: null,         // ← NOT an auto field
           pet_name: null,           // ← NOT an auto field
           destinations: null        // ← NOT an auto field
       }
    SYMPTOMS:
        → Documents bloated with null fields
        → Confusing: which nulls are "field doesn't apply" vs "value unknown"?
        → Storage wasted on field names for null values
        → Basically recreating the Redshift wide table anti-pattern
    ROOT CAUSE:
        → Redshift habit: every row must have every column
        → Not trusting MongoDB's flexible schema
    FIX: Only include fields that APPLY to this document type
         → Missing field = "doesn't apply" (not stored at all)
         → null value = "applies but unknown" (explicitly stored)
         → This distinction is IMPOSSIBLE in Redshift

PITFALL 3: "No Discriminator Field"
    ❌ Documents with different shapes but no type indicator:
       { vin: "1HGBH...", make: "Honda" }          // is this auto?
       { square_feet: 2400, flood_zone: "X" }       // is this home?
    SYMPTOMS:
        → Application must guess type by checking which fields exist
        → if "vin" in doc → auto? But what if auto has no vin yet?
        → Fragile, error-prone code
        → Can't create type-specific partial indexes
    FIX: ALWAYS include discriminator field
         → policy_type: "auto" → application knows exactly what to expect
         → Index on policy_type enables type-specific queries
         → Schema validation can enforce discriminator values

PITFALL 4: "Inconsistent Shared Fields Across Types"
    ❌ Auto policy: { annual_premium: 1200 }
       Home policy: { premium_amount: 2400 }
       Life policy: { yearly_cost: 600 }
    SYMPTOMS:
        → Cross-type aggregation impossible without $switch
        → "Total premium across all types" requires mapping logic
        → BI tools see different field names for same concept
    FIX: Standardize shared fields with IDENTICAL names and types
         → ALL types: { premium: <number> }
         → Cross-type query: db.policies.aggregate([{ $group: { total: { $sum: "$premium" } } }])
         → Works instantly across all types

PITFALL 5: "Over-Engineering Schema Validation"
    ❌ Trying to encode complex type-specific rules in $jsonSchema:
       "if policy_type = 'auto' then vehicle.vin is required"
       "if policy_type = 'life' then beneficiaries must sum to 100%"
    REALITY:
        → $jsonSchema has limited conditional logic
        → Complex type-specific validation → application layer
        → $jsonSchema for: shared field requirements + enum values
        → Application code for: type-specific business rules
    FIX: Layer validation:
         Layer 1: $jsonSchema → required shared fields, enum types
         Layer 2: Application → type-specific field requirements
         Layer 3: Application → business logic (beneficiary % = 100)

PITFALL 6: "Forgetting Partial Indexes"
    ❌ Creating full indexes on type-specific fields:
       db.policies.createIndex({ "vehicle.vin": 1 })
       → Indexes ALL documents, not just auto policies
       → Index includes entries for home/life/pet/travel docs
         where vehicle.vin doesn't exist → wasted index space
    FIX: Always use partial indexes for type-specific fields:
       db.policies.createIndex(
           { "vehicle.vin": 1 },
           { partialFilterExpression: { policy_type: "auto" } }
       )
       → Only indexes auto policy documents
       → Smaller index → fits in RAM → faster lookups

PITFALL 7: "Not Planning for Type Evolution"
    ❌ Hardcoding type handling with no extensibility:
       if policy_type == "auto": render_auto_page()
       elif policy_type == "home": render_home_page()
       // ... no default case, no unknown type handling
    SYMPTOMS:
        → New type "cyber" added → application crashes on unknown type
        → No graceful degradation for unrecognized types
    FIX: Application should handle unknown types gracefully
         → Render shared fields for any type
         → Type-specific rendering as optional enhancement
         → Log unknown types for developer attention
         → Registry pattern: map type → handler function

📈 SCALABILITY CONSIDERATIONS

SCALING POLYMORPHIC COLLECTIONS:

─── SINGLE COLLECTION WITH MIXED TYPES ───

    ADVANTAGE: One collection = one set of indexes for shared fields
    → customer_id index serves ALL policy types
    → status + end_date index serves ALL renewal queries
    → No UNION-like operation across multiple collections

    POTENTIAL CONCERN: Collection gets very large with all types
    → 10M auto + 5M home + 3M life + 2M travel + 1M pet = 21M documents
    → ALL in one collection
    
    MITIGATION:
    → Partial indexes keep type-specific indexes small
    → Shared field indexes serve cross-type queries efficiently
    → Sharding distributes the full collection across nodes

    SHARDING STRATEGY:
    → Shard key: { customer_id: "hashed" }
      → All policies for one customer on same shard
      → "All customer policies" = targeted query (one shard)
    → Alternative: { policy_type: 1, customer_id: "hashed" }
      → Type-specific queries more targeted
      → BUT: "all customer policies" may hit multiple shards
    → RECOMMENDATION: { customer_id: "hashed" }
      → Primary access pattern is per-customer → optimize for that

─── INDEX SIZING WITH PARTIAL INDEXES ───

    WITHOUT partial indexes (full indexes):
    ┌──────────────────────────────┬──────────────────┐
    │ INDEX                         │ ENTRIES           │
    ├──────────────────────────────┼──────────────────┤
    │ { vehicle.vin: 1 }           │ 21M (all docs)    │
    │ { property.state: 1 }        │ 21M (all docs)    │
    │ { life_details.health: 1 }   │ 21M (all docs)    │
    │ TOTAL wasted entries          │ ~42M unnecessary   │
    └──────────────────────────────┴──────────────────┘
    
    WITH partial indexes:
    ┌──────────────────────────────────────────┬──────────────┐
    │ INDEX                                     │ ENTRIES       │
    ├──────────────────────────────────────────┼──────────────┤
    │ { vehicle.vin: 1 } WHERE type="auto"     │ 10M (auto)    │
    │ { property.state: 1 } WHERE type="home"  │ 5M (home)     │
    │ { life_details.health: 1 } WHERE type="life"│ 3M (life)  │
    │ TOTAL entries                              │ 18M (exact)   │
    │ SAVINGS                                   │ 70% fewer     │
    └──────────────────────────────────────────┴──────────────┘
    
    Partial indexes = 70% smaller → more fits in WiredTiger cache
    This is a MASSIVE advantage of polymorphic + partial indexes

─── VS REDSHIFT AT SCALE ───

    Redshift multi-table approach (21M policies):
    → policies_base: 21M rows → ~2 GB compressed
    → auto_details: 10M rows → ~1 GB
    → home_details: 5M rows → ~500 MB
    → life_details: 3M rows → ~300 MB
    → + beneficiaries, destinations, etc.: ~500 MB
    → TOTAL: ~4-5 GB compressed
    → JOIN cost: paid on every query
    → Schema changes: painful across multiple tables
    
    MongoDB polymorphic (21M policies):
    → policies collection: 21M documents → ~10-15 GB (BSON overhead)
    → Partial indexes: ~2-3 GB total
    → TOTAL: ~12-18 GB
    → No JOIN cost: everything in document
    → Schema changes: zero-cost (just insert new shape)
    
    TRADE-OFF:
    → Redshift: 3-4x more storage efficient (columnar compression)
    → MongoDB: 10-50x faster for OLTP operations (no JOINs)
    → Redshift: better for cross-type analytics at massive scale
    → MongoDB: better for per-policy and per-customer operations

📍 WHEN TO USE MONGODB (POLYMORPHIC) OVER REDSHIFT

USE MONGODB WITH POLYMORPHIC PATTERN WHEN:
    → Entities share SOME fields but differ in MANY fields
    → New types added frequently (monthly/quarterly)
    → Primary access: per-entity detail pages, per-customer dashboards
    → Need sub-10ms reads for individual policy lookups
    → Nested data varies by type (arrays, objects, different shapes)
    → Schema evolves rapidly per type (new fields added often)
    → Application-driven access patterns (not analyst-driven)

📍 WHEN TO STICK WITH REDSHIFT INSTEAD

USE REDSHIFT WHEN:
    → Primary access: cross-type analytics
      "Average premium by type by region by quarter"
    → Complex JOINs across entity types
      "Customers with both auto AND home policies"
    → BI tools need SQL access with predictable schema
    → Types are stable (rarely add new types)
    → Batch processing of all policies for regulatory reporting
    → Need columnar compression for cost efficiency at PB scale

# Demonstrates how application creates type-specific documents

class PolicyFactory:
    """
    Creates polymorphic policy documents.
    
    Each policy type has:
    - Shared fields (all types)
    - Type-specific fields (only that type)
    
    Redshift equivalent would require:
    - INSERT into policies_base table
    - INSERT into type-specific detail table
    - Two separate operations, two tables
    
    MongoDB: ONE insertOne() with all fields in ONE document
    """
    
    @staticmethod
    def _shared_fields(policy_number, policy_type, customer_id, 
                        customer_name, premium, coverage, deductible,
                        start_date, end_date, agent_id=None):
        """Fields present in ALL policy types"""
        return {
            "policy_number": policy_number,
            "policy_type": policy_type,        # DISCRIMINATOR FIELD
            "customer_id": customer_id,
            "customer_name": customer_name,     # embedded subset
            "agent_id": agent_id,
            "status": "active",
            "premium": premium,
            "coverage_amount": coverage,
            "deductible": deductible,
            "start_date": start_date,
            "end_date": end_date,
            "created_at": datetime.now(),
            "updated_at": datetime.now()
        }
    
    @classmethod
    def create_auto(cls, policy_number, customer_id, customer_name,
                     premium, coverage, deductible, start_date, end_date,
                     vehicle, driver, auto_coverage, agent_id=None):
        """Create auto insurance policy document"""
        doc = cls._shared_fields(
            policy_number, "auto", customer_id, customer_name,
            premium, coverage, deductible, start_date, end_date, agent_id
        )
        # AUTO-SPECIFIC fields
        doc["vehicle"] = vehicle
        doc["driver"] = driver
        doc["auto_coverage"] = auto_coverage
        return doc
    
    @classmethod
    def create_home(cls, policy_number, customer_id, customer_name,
                     premium, coverage, deductible, start_date, end_date,
                     property_info, home_details, home_risks, 
                     home_coverage, agent_id=None):
        """Create home insurance policy document"""
        doc = cls._shared_fields(
            policy_number, "home", customer_id, customer_name,
            premium, coverage, deductible, start_date, end_date, agent_id
        )
        # HOME-SPECIFIC fields
        doc["property"] = property_info
        doc["home_details"] = home_details
        doc["home_risks"] = home_risks
        doc["home_coverage"] = home_coverage
        return doc
    
    @classmethod
    def create_life(cls, policy_number, customer_id, customer_name,
                     premium, coverage, deductible, start_date, end_date,
                     life_details, beneficiaries, medical_history=None,
                     agent_id=None):
        """Create life insurance policy document"""
        doc = cls._shared_fields(
            policy_number, "life", customer_id, customer_name,
            premium, coverage, deductible, start_date, end_date, agent_id
        )
        # LIFE-SPECIFIC fields
        doc["life_details"] = life_details
        doc["beneficiaries"] = beneficiaries       # embedded array!
        if medical_history:
            doc["medical_history"] = medical_history
        return doc
    
    @classmethod
    def create_pet(cls, policy_number, customer_id, customer_name,
                    premium, coverage, deductible, start_date, end_date,
                    pet, pet_coverage, pet_health=None, agent_id=None):
        """Create pet insurance policy document"""
        doc = cls._shared_fields(
            policy_number, "pet", customer_id, customer_name,
            premium, coverage, deductible, start_date, end_date, agent_id
        )
        # PET-SPECIFIC fields
        doc["pet"] = pet
        doc["pet_coverage"] = pet_coverage
        if pet_health:
            doc["pet_health"] = pet_health
        return doc

# Application layer that handles different types transparently

class InsuranceService:
    """
    Query service that leverages the polymorphic pattern.
    
    Application checks policy_type (discriminator) to know
    which type-specific fields to expect and display.
    """
    
    def __init__(self, db):
        self.policies = db.policies
        self.customers = db.customers
    
    def get_customer_dashboard(self, customer_id):
        """
        Agent dashboard: all policies for a customer.
        
        ONE query, ONE collection, ALL types returned.
        
        Redshift equivalent:
            SELECT pb.*, ad.vin, ad.make, hd.property_city, ld.face_value, ...
            FROM policies_base pb
            LEFT JOIN auto_details ad ON ...
            LEFT JOIN home_details hd ON ...
            LEFT JOIN life_details ld ON ...
            LEFT JOIN pet_details pd ON ...
            WHERE pb.customer_id = ?;
            -- 4+ LEFT JOINs, sparse NULLs in result
        """
        return list(self.policies.find(
            {"customer_id": customer_id},
            {
                "policy_number": 1, "policy_type": 1, "status": 1,
                "premium": 1, "coverage_amount": 1, "start_date": 1, "end_date": 1,
                # Type-specific summary (only present in relevant docs)
                "vehicle.make": 1, "vehicle.model": 1, "vehicle.year": 1,
                "property.address.city": 1, "property.square_feet": 1,
                "life_details.face_value": 1, "life_details.term_years": 1,
                "pet.name": 1, "pet.breed": 1
            }
        ).sort("created_at", DESCENDING))
    
    def get_policy_detail(self, policy_number):
        """
        Full policy detail — complete document.
        
        ONE read returns EVERYTHING including nested arrays.
        No JOINs for beneficiaries, scheduled items, etc.
        """
        return self.policies.find_one({"policy_number": policy_number})
    
    def search_by_vin(self, vin):
        """VIN search using partial index (auto only)."""
        return self.policies.find_one({
            "policy_type": "auto",
            "vehicle.vin": vin
        })
    
    def get_expiring_policies(self, days=30):
        """All policies expiring within N days — cross-type."""
        cutoff = datetime.now() + timedelta(days=days)
        return list(self.policies.find(
            {"status": "active", "end_date": {"$lte": cutoff}},
            {"policy_number": 1, "policy_type": 1, "customer_name": 1,
             "premium": 1, "end_date": 1}
        ).sort("end_date", ASCENDING))
    
    def premium_analytics(self):
        """Cross-type premium analytics."""
        return list(self.policies.aggregate([
            {"$match": {"status": "active"}},
            {"$group": {
                "_id": "$policy_type",
                "count": {"$sum": 1},
                "total_premium": {"$sum": "$premium"},
                "avg_premium": {"$avg": "$premium"},
                "total_coverage": {"$sum": "$coverage_amount"},
                "avg_deductible": {"$avg": "$deductible"}
            }},
            {"$sort": {"total_premium": DESCENDING}},
            {"$project": {
                "type": "$_id", "_id": 0,
                "count": 1,
                "total_premium": {"$round": ["$total_premium", 2]},
                "avg_premium": {"$round": ["$avg_premium", 2]},
                "total_coverage": {"$round": ["$total_coverage", 2]},
                "avg_deductible": {"$round": ["$avg_deductible", 2]}
            }}
        ]))
    
    def format_policy_summary(self, policy):
        """Format type-specific summary line for dashboard."""
        t = policy.get("policy_type", "unknown")
        
        if t == "auto":
            v = policy.get("vehicle", {})
            return f"{v.get('year', '?')} {v.get('make', '?')} {v.get('model', '?')}"
        elif t == "home":
            p = policy.get("property", {}).get("address", {})
            sqft = policy.get("property", {}).get("square_feet", "?")
            return f"{p.get('city', '?')}, {sqft} sqft"
        elif t == "life":
            ld = policy.get("life_details", {})
            fv = ld.get("face_value", 0)
            ty = ld.get("term_years", "?")
            return f"${fv:,.0f} / {ty} year term"
        elif t == "pet":
            pet = policy.get("pet", {})
            return f"{pet.get('name', '?')} ({pet.get('breed', '?')})"
        else:
            return f"[{t} policy]"
