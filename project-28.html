PROJECT 28: Multi-Document Transactions

Problem: Transfer Money Between Two Different Customer Accounts, Place an Order While Decrementing Inventory Across Collections, and Maintain Referential Integrity Across Documents

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your platform now has operations that CANNOT fit in a single document:

    OPERATION 1 — Inter-Account Money Transfer
        → Transfer $500 from Alice's account (ACC-001) to Bob's account (ACC-002)
        → Must: debit Alice AND credit Bob atomically
        → Cannot: debit Alice but fail to credit Bob (money vanishes)
        → Cannot: credit Bob but fail to debit Alice (money created from nothing)
        → TWO different documents — single-document atomicity won't help

    OPERATION 2 — Order Placement + Inventory Decrement
        → Customer places order (orders collection)
        → Each product's stock_qty must be decremented (products collection)
        → Customer's stats must be updated (customers collection)
        → THREE different collections — must all succeed or all fail

    OPERATION 3 — Consistent Multi-Document Read
        → Reconciliation: read ALL accounts and verify total balance unchanged
        → Must see a CONSISTENT snapshot across all account documents
        → Cannot: read account A before a transfer, account B after
          (would show incorrect total)

    IN PROJECT 27:
        → We solved problems where data fits in ONE document
        → Embedding gave us free atomicity
        → No transactions needed

    NOW THE QUESTION:
        → What happens when embedding ISN'T possible?
        → Two customer accounts = two separate documents (can't embed)
        → Order + inventory = two separate collections (can't merge)
        → THIS is where multi-document transactions enter

    IN REDSHIFT:
        → BEGIN; ... COMMIT; — same tool for everything
        → No difference between single-row and multi-row transactions
        → Transactions are your DEFAULT, ONLY approach
        → No special syntax, no performance penalty distinction

    IN MONGODB:
        → Multi-document transactions exist (since 4.0)
        → BUT they have REAL performance costs
        → MongoDB's stance: "Design to avoid them. Use when you must."
        → Understanding WHEN to use them vs WHEN to redesign = mastery

🔴 REDSHIFT MINDSET — How You Would Solve This

Redshift: Transactions Are the Default Tool for Everything

IN REDSHIFT — THESE ARE COMPLETELY ROUTINE:

OPERATION 1: Inter-Account Transfer
    BEGIN;
    
    UPDATE accounts SET balance = balance - 500
    WHERE account_id = 'ACC-001' AND balance >= 500;
    
    -- Check if debit succeeded
    -- (In practice, check @@ROWCOUNT or use application logic)
    
    UPDATE accounts SET balance = balance + 500
    WHERE account_id = 'ACC-002';
    
    COMMIT;
    -- Both succeed or both roll back
    -- Standard SQL, standard approach, no special thinking

OPERATION 2: Order + Inventory
    BEGIN;
    
    INSERT INTO orders (order_id, customer_id, total, status)
    VALUES ('ORD-1001', 'CUST-001', 119.98, 'confirmed');
    
    INSERT INTO order_items (order_id, product_id, qty, price)
    VALUES ('ORD-1001', 'PROD-001', 1, 29.99),
           ('ORD-1001', 'PROD-002', 1, 89.99);
    
    UPDATE products SET stock_qty = stock_qty - 1
    WHERE product_id = 'PROD-001' AND stock_qty > 0;
    
    UPDATE products SET stock_qty = stock_qty - 1
    WHERE product_id = 'PROD-002' AND stock_qty > 0;
    
    UPDATE customers SET total_orders = total_orders + 1,
                         total_spent = total_spent + 119.98
    WHERE customer_id = 'CUST-001';
    
    COMMIT;
    -- 5 statements, 3 tables, all atomic

OPERATION 3: Consistent Read
    BEGIN;
    
    SELECT account_id, balance FROM accounts;
    -- Snapshot isolation: sees consistent point-in-time
    -- No concurrent transfers can affect this read
    
    COMMIT;
    -- Every query in Redshift is already snapshot-isolated
    -- BEGIN/COMMIT is optional for read-only operations

WHAT REDSHIFT DEVELOPERS NEVER THINK ABOUT:
    → "Is this transaction expensive?" — Not really, it's the default
    → "Should I redesign to avoid transactions?" — Why would I?
    → "How many documents does this touch?" — Not a concept in Redshift
    → "Will this lock block other operations?" — WLM handles concurrency
    
    Transactions in Redshift are CHEAP and UBIQUITOUS
    Transactions in MongoDB are EXPENSIVE and should be RARE
    THIS is the fundamental mindset shift for this project

🟢 MONGODB MINDSET — How This Same Problem Is Approached

The Cost of Multi-Document Transactions

BEFORE WRITING ANY CODE — UNDERSTAND THE COSTS:

    ┌──────────────────────────────────────────────────────────────┐
    │                                                               │
    │  SINGLE-DOCUMENT UPDATE (Project 27):                         │
    │  ├── Lock: ONE document for microseconds                      │
    │  ├── Overhead: ZERO transaction management                    │
    │  ├── Latency: 0.5-5ms                                         │
    │  ├── Concurrency impact: minimal (other docs unaffected)      │
    │  └── Failure mode: operation succeeds or fails (simple)       │
    │                                                               │
    │  MULTI-DOCUMENT TRANSACTION:                                  │
    │  ├── Lock: MULTIPLE documents for ENTIRE transaction duration │
    │  ├── Overhead: session, snapshot, commit protocol              │
    │  ├── Latency: 5-100ms (10-50x single-document)               │
    │  ├── Concurrency impact: locked docs block other writers      │
    │  ├── Failure modes: commit conflict, timeout, write conflict  │
    │  ├── WiredTiger cache pressure: snapshot held in memory       │
    │  ├── Oplog entries: single entry for entire transaction       │
    │  └── Max duration: 60 seconds default (configurable)          │
    │                                                               │
    │  COST RATIO: Multi-doc txn is ~10-50x more expensive          │
    │              than single-document operation                    │
    │                                                               │
    │  IN REDSHIFT: Transaction cost is ~1.1x single statement      │
    │               Almost negligible overhead                       │
    │                                                               │
    └──────────────────────────────────────────────────────────────┘

MONGODB'S OFFICIAL STANCE:
    "In most cases, multi-document transactions incur a greater
     performance cost over single-document writes. For most
     scenarios, the denormalized data model (embedded documents
     and arrays) will continue to be optimal for your data and
     use cases. That is, for most scenarios, modeling your data
     appropriately will minimize the need for multi-document
     transactions."
    — MongoDB Documentation

TRANSLATION:
    → Design your documents to avoid transactions (Project 27)
    → When you CANNOT avoid them: use them (this project)
    → But ALWAYS ask first: "Can I redesign the document instead?"

When Multi-Document Transactions Are REQUIRED

YOU CANNOT AVOID TRANSACTIONS WHEN:

    1. TWO INDEPENDENT ENTITIES must change atomically
       → Alice's account and Bob's account = two documents
       → Cannot embed Bob's balance inside Alice's document
       → Cannot merge all accounts into one document
       → MUST use transaction

    2. CROSS-COLLECTION consistency is required
       → orders collection + products collection + customers collection
       → Cannot merge all into one collection (different access patterns)
       → MUST use transaction (or accept eventual consistency)

    3. CONSISTENT MULTI-DOCUMENT READS
       → Read all accounts at same point-in-time
       → Without transaction: reads happen at different moments
       → With readConcern "snapshot" in transaction: consistent

    4. REFERENTIAL INTEGRITY across documents
       → Delete customer → must also delete their orders
       → Without transaction: orphaned orders possible
       → With transaction: all-or-nothing

YOU CAN OFTEN AVOID TRANSACTIONS BY:

    1. EMBEDDING (Project 27)
       → Move related data into one document
       → Cart items inside customer document → atomic move
    
    2. TWO-PHASE APPROACH with compensation
       → Debit account → create pending credit → confirm credit
       → If credit fails: reverse the debit (saga pattern)
       → More complex but no transaction overhead
    
    3. EVENTUAL CONSISTENCY acceptance
       → Inventory decremented slightly after order created
       → Acceptable lag for non-critical operations
       → Change streams (Project 25) for eventual sync
    
    4. OPTIMISTIC CONCURRENCY (Project 27)
       → Version field prevents conflicting updates
       → Retry on conflict
       → Works for many real-world scenarios

🔧 HOW TO APPROACH IT IN MONGODB

MULTI-DOCUMENT TRANSACTION DECISION FRAMEWORK:

    STEP 1: Do I ACTUALLY need a transaction?
    ┌──────────────────────────────────────────────────┐
    │ Can I embed the data in one document?             │
    │   YES → Use single-document atomicity (Project 27)│
    │   NO  → Continue to Step 2                        │
    └──────────────────────────────────────────────────┘
    
    STEP 2: Can I accept eventual consistency?
    ┌──────────────────────────────────────────────────┐
    │ Is it OK if changes propagate within seconds?     │
    │   YES → Use change streams + eventual sync        │
    │   NO  → Continue to Step 3                        │
    └──────────────────────────────────────────────────┘
    
    STEP 3: Can I use a saga/compensation pattern?
    ┌──────────────────────────────────────────────────┐
    │ Can I reverse a partial operation if later fails? │
    │   YES → Use saga pattern (more complex, no txn)   │
    │   NO  → Continue to Step 4                        │
    └──────────────────────────────────────────────────┘
    
    STEP 4: Use a multi-document transaction
    ┌──────────────────────────────────────────────────┐
    │ → session.start_transaction()                     │
    │ → Perform operations with session parameter       │
    │ → session.commit_transaction()                    │
    │ → Handle TransientTransactionError (retry)        │
    │ → Handle UnknownTransactionCommitResult (retry)   │
    │ → Keep transactions SHORT (< 5 seconds ideal)     │
    └──────────────────────────────────────────────────┘

print("=== OPERATION 1: Inter-Account Money Transfer ===\n")

// ─── WHY SINGLE-DOCUMENT WON'T WORK ─────────────────
// Alice's account and Bob's account are SEPARATE documents
// Cannot embed Bob's balance inside Alice's document
// Cannot merge all accounts into one document (millions of customers)
// MUST use multi-document transaction

// ─── THE TRANSACTION ─────────────────────────────────

var session = db.getMongo().startSession()

session.startTransaction({
    readConcern: { level: "snapshot" },
    // ↑ All reads in this transaction see the SAME point-in-time
    //   (equivalent to Redshift's default snapshot isolation)
    writeConcern: { w: "majority", j: true },
    // ↑ Commit is durable — survives failover
    //   (equivalent to Redshift's default durability)
    // readPreference: "primary"
    // ↑ Transactions MUST read from primary (automatic)
})

try {
    var transferAmount = 500.00
    var fromAccount = "ACC-001"
    var toAccount = "ACC-002"
    
    // ─── STEP 1: Debit the sender ───────────────────
    var debitResult = session.getDatabase("ecommerce").accounts.updateOne(
        {
            _id: fromAccount,
            balance: { $gte: transferAmount },  // sufficient funds check
            status: "active"                     // account must be active
        },
        {
            $inc: { balance: -transferAmount, version: 1 },
            $set: { updated_at: new Date() }
        },
        { session: session }  // ← CRITICAL: operation is part of transaction
    )
    
    if (debitResult.matchedCount === 0) {
        throw new Error("Insufficient funds or inactive account")
        // This will trigger the catch block → abort transaction
    }
    
    print(`  Debited $${transferAmount} from ${fromAccount}`)
    
    // ─── STEP 2: Credit the receiver ────────────────
    var creditResult = session.getDatabase("ecommerce").accounts.updateOne(
        {
            _id: toAccount,
            status: "active"  // receiver must be active
        },
        {
            $inc: { balance: transferAmount, version: 1 },
            $set: { updated_at: new Date() }
        },
        { session: session }
    )
    
    if (creditResult.matchedCount === 0) {
        throw new Error("Receiver account not found or inactive")
    }
    
    print(`  Credited $${transferAmount} to ${toAccount}`)
    
    // ─── STEP 3: Log the transfer ───────────────────
    session.getDatabase("ecommerce").transfer_log.insertOne(
        {
            from_account: fromAccount,
            to_account: toAccount,
            amount: transferAmount,
            currency: "USD",
            status: "completed",
            timestamp: new Date()
        },
        { session: session }
    )
    
    print(`  Transfer logged`)
    
    // ─── COMMIT ─────────────────────────────────────
    session.commitTransaction()
    print(`\n  ✅ Transaction COMMITTED`)
    
} catch (error) {
    // ─── ABORT ──────────────────────────────────────
    session.abortTransaction()
    print(`\n  ❌ Transaction ABORTED: ${error.message}`)
    
} finally {
    session.endSession()
}

print("\n=== OPERATION 1B: Transfer with Retry Logic ===\n")

// MongoDB transactions can fail with RETRYABLE errors:
// → TransientTransactionError: temporary issue (retry entire txn)
// → UnknownTransactionCommitResult: commit status unknown (retry commit)
//
// Redshift: retries are simpler (single cluster, no elections)
// MongoDB: distributed system = more failure modes = must handle retries

function transferWithRetry(fromAcct, toAcct, amount, maxRetries) {
    maxRetries = maxRetries || 3
    
    for (var attempt = 1; attempt <= maxRetries; attempt++) {
        var session = db.getMongo().startSession()
        
        try {
            session.startTransaction({
                readConcern: { level: "snapshot" },
                writeConcern: { w: "majority", j: true }
            })
            
            var dbTxn = session.getDatabase("ecommerce")
            
            // Debit
            var debit = dbTxn.accounts.updateOne(
                { _id: fromAcct, balance: { $gte: amount }, status: "active" },
                { $inc: { balance: -amount, version: 1 },
                  $set: { updated_at: new Date() } },
                { session: session }
            )
            
            if (debit.matchedCount === 0) {
                throw new Error("BUSINESS_ERROR: Insufficient funds")
                // Business errors: do NOT retry — abort immediately
            }
            
            // Credit
            var credit = dbTxn.accounts.updateOne(
                { _id: toAcct, status: "active" },
                { $inc: { balance: amount, version: 1 },
                  $set: { updated_at: new Date() } },
                { session: session }
            )
            
            if (credit.matchedCount === 0) {
                throw new Error("BUSINESS_ERROR: Receiver not found")
            }
            
            // Log
            dbTxn.transfer_log.insertOne({
                from_account: fromAcct,
                to_account: toAcct,
                amount: amount,
                status: "completed",
                attempt: attempt,
                timestamp: new Date()
            }, { session: session })
            
            // Commit (may throw UnknownTransactionCommitResult)
            session.commitTransaction()
            
            print(`  ✅ Transfer successful (attempt ${attempt})`)
            return { success: true, attempt: attempt }
            
        } catch (error) {
            var errorMsg = error.message || ""
            
            // BUSINESS ERROR: Do not retry
            if (errorMsg.startsWith("BUSINESS_ERROR")) {
                session.abortTransaction()
                print(`  ❌ Business error: ${errorMsg}`)
                return { success: false, error: errorMsg }
            }
            
            // TRANSIENT ERROR: Retry entire transaction
            if (error.hasOwnProperty("errorLabels") &&
                error.errorLabels.includes("TransientTransactionError")) {
                print(`  ⚠️ Transient error (attempt ${attempt}): retrying...`)
                // Don't abort — just retry from scratch
                continue
            }
            
            // UNKNOWN COMMIT RESULT: Retry just the commit
            if (error.hasOwnProperty("errorLabels") &&
                error.errorLabels.includes("UnknownTransactionCommitResult")) {
                print(`  ⚠️ Unknown commit result (attempt ${attempt}): ` +
                      `retrying commit...`)
                try {
                    session.commitTransaction()
                    print(`  ✅ Commit retry successful`)
                    return { success: true, attempt: attempt }
                } catch (commitError) {
                    print(`  ❌ Commit retry failed: ${commitError.message}`)
                    continue
                }
            }
            
            // OTHER ERROR: Abort and retry
            try { session.abortTransaction() } catch(e) {}
            
            if (attempt < maxRetries) {
                print(`  ⚠️ Error (attempt ${attempt}): ${errorMsg}. Retrying...`)
                sleep(100 * attempt)  // backoff
            } else {
                print(`  ❌ All ${maxRetries} attempts failed: ${errorMsg}`)
                return { success: false, error: errorMsg }
            }
            
        } finally {
            session.endSession()
        }
    }
}

function placeOrder(customerId, items) {
    var session = db.getMongo().startSession()
    
    try {
        session.startTransaction({
            readConcern: { level: "snapshot" },
            writeConcern: { w: "majority" }
        })
        
        var dbTxn = session.getDatabase("ecommerce")
        
        // ─── STEP 1: Verify and reserve inventory ───
        var orderItems = []
        var orderTotal = 0
        
        for (var i = 0; i < items.length; i++) {
            var item = items[i]
            
            // Atomic check-and-decrement within transaction
            var stockResult = dbTxn.products.updateOne(
                {
                    _id: item.product_id,
                    stock_qty: { $gte: item.qty }  // sufficient stock
                },
                {
                    $inc: {
                        stock_qty: -item.qty,
                        sold_count: item.qty
                    }
                },
                { session: session }
            )
            
            if (stockResult.matchedCount === 0) {
                // Product out of stock — abort ENTIRE order
                throw new Error(
                    `STOCK_ERROR: Insufficient stock for ${item.product_id}`
                )
            }
            
            // Get product details for order document
            var product = dbTxn.products.findOne(
                { _id: item.product_id },
                { session: session }
            )
            
            var lineTotal = product.price * item.qty
            orderItems.push({
                product_id: item.product_id,
                name: product.name,
                qty: item.qty,
                unit_price: product.price,
                line_total: lineTotal
            })
            orderTotal += lineTotal
            
            print(`    Reserved: ${item.qty}x ${product.name}`)
        }
        
        // ─── STEP 2: Create order ───────────────────
        var orderId = `ORD-${Date.now()}`
        dbTxn.orders.insertOne({
            _id: orderId,
            customer_id: customerId,
            items: orderItems,
            subtotal: Math.round(orderTotal * 100) / 100,
            tax: Math.round(orderTotal * 0.08 * 100) / 100,
            total: Math.round(orderTotal * 1.08 * 100) / 100,
            status: "confirmed",
            created_at: new Date()
        }, { session: session })
        
        print(`    Order created: ${orderId} ($${Math.round(orderTotal * 1.08 * 100) / 100})`)
        
        // ─── STEP 3: Update customer stats ──────────
        dbTxn.customers.updateOne(
            { _id: customerId },
            {
                $inc: {
                    "stats.total_orders": 1,
                    "stats.total_spent": Math.round(orderTotal * 1.08 * 100) / 100
                }
            },
            { session: session }
        )
        
        print(`    Customer stats updated`)
        
        // ─── COMMIT ─────────────────────────────────
        session.commitTransaction()
        print(`\n  ✅ Order transaction COMMITTED: ${orderId}`)
        return { success: true, order_id: orderId }
        
    } catch (error) {
        session.abortTransaction()
        print(`\n  ❌ Order transaction ABORTED: ${error.message}`)
        
        // If it was a stock error: inventory was never decremented
        // (transaction abort reverses all changes)
        return { success: false, error: error.message }
        
    } finally {
        session.endSession()

def run_transaction_with_retry(session, txn_func, *args, **kwargs):
    """
    Execute a transaction function with automatic retry
    for TransientTransactionError.
    
    This is the STANDARD pattern recommended by MongoDB.
    Every production transaction should use this wrapper.
    
    Redshift equivalent: not needed — Redshift handles retries internally
    MongoDB: distributed system = you must handle transient failures
    """
    while True:
        try:
            result = txn_func(session, *args, **kwargs)
            return result
        except (ConnectionFailure, OperationFailure) as exc:
            # TransientTransactionError: retry the entire transaction
            if exc.has_error_label("TransientTransactionError"):
                print(f"    ⚠️ TransientTransactionError — retrying...")
                continue
            else:
                raise


def commit_with_retry(session):
    """
    Commit a transaction with automatic retry
    for UnknownTransactionCommitResult.
    
    Commit can fail with unknown result due to:
    → Primary stepdown during commit
    → Network timeout during commit confirmation
    → Retrying commit is SAFE (idempotent)
    """
    while True:
        try:
            session.commit_transaction()
            return
        except (ConnectionFailure, OperationFailure) as exc:
            if exc.has_error_label("UnknownTransactionCommitResult"):
                print(f"    ⚠️ UnknownTransactionCommitResult — retrying commit...")
                continue
            else:
                raise

def transfer_funds(session, from_acct: str, to_acct: str, amount: float):
    """
    Transfer funds between two accounts atomically.
    
    This function is called WITHIN a transaction session.
    All operations use the session parameter.
    If any step fails, the entire transaction is aborted.
    
    Redshift equivalent:
        BEGIN;
        UPDATE accounts SET balance = balance - :amount WHERE ...;
        UPDATE accounts SET balance = balance + :amount WHERE ...;
        INSERT INTO transfer_log (...) VALUES (...);
        COMMIT;
    """
    accounts = session.client["ecommerce"]["accounts"]
    transfer_log = session.client["ecommerce"]["transfer_log"]
    
    # Start transaction with appropriate concerns
    session.start_transaction(
        read_concern=ReadConcern("snapshot"),
        write_concern=WriteConcern(w="majority", j=True)
    )
    
    try:
        # Step 1: Debit sender
        debit = accounts.update_one(
            {"_id": from_acct, "balance": {"$gte": amount}, "status": "active"},
            {"$inc": {"balance": -amount, "version": 1},
             "$set": {"updated_at": datetime.now()}},
            session=session
        )
        
        if debit.matched_count == 0:
            raise ValueError(f"Insufficient funds or inactive: {from_acct}")
        
        # Step 2: Credit receiver
        credit = accounts.update_one(
            {"_id": to_acct, "status": "active"},
            {"$inc": {"balance": amount, "version": 1},
             "$set": {"updated_at": datetime.now()}},
            session=session
        )
        
        if credit.matched_count == 0:
            raise ValueError(f"Receiver not found or inactive: {to_acct}")
        
        # Step 3: Log the transfer
        transfer_log.insert_one({
            "from_account": from_acct,
            "to_account": to_acct,
            "amount": amount,
            "currency": "USD",
            "status": "completed",
            "timestamp": datetime.now()
        }, session=session)
        
        # Step 4: Commit with retry
        commit_with_retry(session)
        
        print(f"  ✅ Transferred ${amount}: {from_acct} → {to_acct}")
        return {"success": True, "amount": amount}
        
    except ValueError as ve:
        # Business error — abort, don't retry
        session.abort_transaction()
        print(f"  ❌ Business error: {ve}")
        return {"success": False, "error": str(ve)}
        
    except Exception as e:
        # Technical error — abort (retry handled by wrapper)
        session.abort_transaction()
        raise  # Re-raise for retry wrapper to catch

def place_order(session, customer_id: str, items: list):
    """
    Place an order: create order + decrement inventory + update customer.
    Three collections modified atomically.
    
    If any product is out of stock: entire order is aborted.
    Inventory is NEVER decremented without a corresponding order.
    """
    orders = session.client["ecommerce"]["orders"]
    products = session.client["ecommerce"]["products"]
    customers = session.client["ecommerce"]["customers"]
    
    session.start_transaction(
        read_concern=ReadConcern("snapshot"),
        write_concern=WriteConcern(w="majority")
    )
    
    try:
        order_items = []
        order_total = 0.0
        
        # Step 1: Reserve inventory for each item
        for item in items:
            stock_result = products.update_one(
                {
                    "_id": item["product_id"],
                    "stock_qty": {"$gte": item["qty"]}
                },
                {
                    "$inc": {
                        "stock_qty": -item["qty"],
                        "sold_count": item["qty"]
                    }
                },
                session=session
            )
            
            if stock_result.matched_count == 0:
                raise ValueError(
                    f"Insufficient stock: {item['product_id']} "
                    f"(requested {item['qty']})"
                )
            
            # Get product details within the same snapshot
            product = products.find_one(
                {"_id": item["product_id"]},
                session=session
            )
            
            line_total = product["price"] * item["qty"]
            order_items.append({
                "product_id": item["product_id"],
                "name": product["name"],
                "qty": item["qty"],
                "unit_price": product["price"],
                "line_total": round(line_total, 2)
            })
            order_total += line_total
            
            print(f"    Reserved: {item['qty']}x {product['name']} "
                  f"(${line_total:.2f})")
        
        # Step 2: Create order document
        order_id = f"ORD-{int(datetime.now().timestamp() * 1000)}"
        tax = round(order_total * 0.08, 2)
        total_with_tax = round(order_total + tax, 2)
        
        orders.insert_one({
            "_id": order_id,
            "customer_id": customer_id,
            "items": order_items,
            "subtotal": round(order_total, 2),
            "tax": tax,
            "total": total_with_tax,
            "status": "confirmed",
            "created_at": datetime.now()
        }, session=session)
        
        print(f"    Order created: {order_id} (${total_with_tax})")
        
        # Step 3: Update customer statistics
        customers.update_one(
            {"_id": customer_id},
            {
                "$inc": {
                    "stats.total_orders": 1,
                    "stats.total_spent": total_with_tax
                }
            },
            session=session
        )
        
        # Commit
        commit_with_retry(session)
        
        print(f"  ✅ Order {order_id} committed")
        return {"success": True, "order_id": order_id, "total": total_with_tax}
        
    except ValueError as ve:
        session.abort_transaction()
        print(f"  ❌ Order failed: {ve}")
        return {"success": False, "error": str(ve)}
        
    except Exception as e:
        session.abort_transaction()
        raise

            return {"success": False, "error": f"Product not found: {item['product_id']}"}
        
        if product["stock_qty"] < item["qty"]:
            return {"success": False,
                    "error": f"Insufficient stock: {item['product_id']} "
                             f"(available: {product['stock_qty']}, "
                             f"requested: {item['qty']})"}
        
        line_total = product["price"] * item["qty"]
        order_items.append({
            "product_id": item["product_id"],
            "name": product["name"],
            "qty": item["qty"],
            "unit_price": product["price"],
            "line_total": round(line_total, 2)
        })
        order_total += line_total
    
    order_id = f"ORD-{int(datetime.now().timestamp() * 1000)}"
    tax = round(order_total * 0.08, 2)
    total_with_tax = round(order_total + tax, 2)
    
    order_doc = {
        "_id": order_id,
        "customer_id": customer_id,
        "items": order_items,
        "subtotal": round(order_total, 2),
        "tax": tax,
        "total": total_with_tax,
        "status": "confirmed",
        "created_at": datetime.now()
    }
    
    print(f"  Phase 1 (prepare): validated {len(items)} items, "
          f"total ${total_with_tax}")
    
    # ─── PHASE 2: EXECUTE (inside transaction) ────────
    # Only writes — no reads, no computation
    # Transaction duration: milliseconds, not seconds
    
    def execute_order(session):
        session.start_transaction(
            read_concern=ReadConcern("snapshot"),
            write_concern=WriteConcern(w="majority")
        )
        
        try:
            products_coll = session.client["ecommerce"]["products"]
            orders_coll = session.client["ecommerce"]["orders"]
            customers_coll = session.client["ecommerce"]["customers"]
            
            # Atomic inventory decrement (re-verify stock inside txn)
            for item in items:
                stock_result = products_coll.update_one(
                    {
                        "_id": item["product_id"],
                        "stock_qty": {"$gte": item["qty"]}
                    },
                    {
                        "$inc": {
                            "stock_qty": -item["qty"],
                            "sold_count": item["qty"]
                        }
                    },
                    session=session
                )
                
                if stock_result.matched_count == 0:
                    raise ValueError(
                        f"Stock changed since validation: {item['product_id']}"
                    )
            
            # Insert order
            orders_coll.insert_one(order_doc, session=session)
            
            # Update customer stats
            customers_coll.update_one(
                {"_id": customer_id},
                {"$inc": {
                    "stats.total_orders": 1,
                    "stats.total_spent": total_with_tax
                }},
                session=session
            )
            
            commit_with_retry(session)
            return {"success": True, "order_id": order_id}
            
        except ValueError as ve:
            session.abort_transaction()
            return {"success": False, "error": str(ve)}
        except Exception:
            session.abort_transaction()
            raise
    
    # Execute with retry wrapper
    with client.start_session() as session:
        result = run_transaction_with_retry(session, execute_order)
    
    if result["success"]:
        print(f"  Phase 2 (execute): committed {result['order_id']} ✅")
    else:
        print(f"  Phase 2 (execute): failed — {result['error']} ❌")
    
    return result

def transfer_saga(from_acct: str, to_acct: str, amount: float):
    """
    Transfer funds using Saga pattern — NO multi-document transaction.
    
    Instead of atomic transaction:
    1. Create a pending transfer record
    2. Debit sender (with reference to transfer)
    3. Credit receiver (with reference to transfer)
    4. Mark transfer complete
    
    If step 3 fails: COMPENSATE by reversing step 2
    
    TRADE-OFFS vs Transaction:
    ✓ No transaction overhead (faster per operation)
    ✓ No lock contention between documents
    ✓ Works across services/databases (distributed saga)
    ✗ More complex code
    ✗ Intermediate states visible (eventual consistency)
    ✗ Must implement compensation logic
    ✗ Must handle partial failure recovery
    
    USE WHEN:
    → Cross-service operations (MongoDB + external API)
    → Very high throughput (transaction overhead is bottleneck)
    → Eventual consistency is acceptable
    
    DON'T USE WHEN:
    → Strict atomicity required (financial regulations)
    → Simple enough for a transaction (don't over-engineer)
    """
    
    transfer_id = f"TXF-{int(datetime.now().timestamp() * 1000)}"
    
    # Step 1: Create pending transfer
    db.transfer_log.insert_one({
        "_id": transfer_id,
        "from_account": from_acct,
        "to_account": to_acct,
        "amount": amount,
        "status": "pending",
        "created_at": datetime.now(),
        "steps_completed": []
    })
    print(f"  Step 1: Transfer {transfer_id} created (pending)")
    
    try:
        # Step 2: Debit sender
        debit_result = db.accounts.update_one(
            {
                "_id": from_acct,
                "balance": {"$gte": amount},
                "status": "active"
            },
            {
                "$inc": {"balance": -amount},
                "$push": {"pending_transfers": transfer_id}
            }
        )
        
        if debit_result.matched_count == 0:
            # Sender can't be debited — mark transfer failed
            db.transfer_log.update_one(
                {"_id": transfer_id},
                {"$set": {"status": "failed_debit",
                          "failed_at": datetime.now()}}
            )
            print(f"  Step 2: FAILED — insufficient funds")
            return {"success": False, "error": "Insufficient funds"}
        
        db.transfer_log.update_one(
            {"_id": transfer_id},
            {"$push": {"steps_completed": "debit"},
             "$set": {"debit_at": datetime.now()}}
        )
        print(f"  Step 2: Debited ${amount} from {from_acct}")
        
        # Step 3: Credit receiver
        credit_result = db.accounts.update_one(
            {"_id": to_acct, "status": "active"},
            {
                "$inc": {"balance": amount},
                "$push": {"pending_transfers": transfer_id}
            }
        )
        
        if credit_result.matched_count == 0:
            # COMPENSATION: Reverse the debit
            print(f"  Step 3: FAILED — receiver not found")
            print(f"  COMPENSATING: Reversing debit...")
            
            db.accounts.update_one(
                {"_id": from_acct},
                {
                    "$inc": {"balance": amount},  # reverse debit
                    "$pull": {"pending_transfers": transfer_id}
                }
            )
            
            db.transfer_log.update_one(
                {"_id": transfer_id},
                {"$set": {"status": "compensated",
                          "compensated_at": datetime.now()}}
            )
            print(f"  COMPENSATED: Debit reversed ✓")
            return {"success": False, "error": "Receiver not found, compensated"}
        
        db.transfer_log.update_one(
            {"_id": transfer_id},
            {"$push": {"steps_completed": "credit"},
             "$set": {"credit_at": datetime.now()}}
        )
        print(f"  Step 3: Credited ${amount} to {to_acct}")
        
        # Step 4: Mark complete and clean up
        db.transfer_log.update_one(
            {"_id": transfer_id},
            {"$set": {"status": "completed",
                      "completed_at": datetime.now()}}
        )
        
        # Clean up pending_transfers arrays
        db.accounts.update_one(
            {"_id": from_acct},
            {"$pull": {"pending_transfers": transfer_id}}
        )
        db.accounts.update_one(
            {"_id": to_acct},
            {"$pull": {"pending_transfers": transfer_id}}
        )
        
        print(f"  Step 4: Transfer completed ✅")
        return {"success": True, "transfer_id": transfer_id}
        
    except Exception as e:
        # Unexpected error — attempt compensation
        print(f"  ERROR: {e}")
        print(f"  Attempting compensation...")
        
        transfer = db.transfer_log.find_one({"_id": transfer_id})
        
        if "debit" in transfer.get("steps_completed", []):
            if "credit" not in transfer.get("steps_completed", []):
                # Debit happened but credit didn't — reverse debit
                db.accounts.update_one(
                    {"_id": from_acct},
                    {"$inc": {"balance": amount},
                     "$pull": {"pending_transfers": transfer_id}}
                )
                print(f"  Debit reversed ✓")
        
        db.transfer_log.update_one(
            {"_id": transfer_id},
            {"$set": {"status": "error_compensated",
                      "error": str(e),
                      "compensated_at": datetime.now()}}
        )
        return {"success": False, "error": str(e)}

MongoDB Internals — Multi-Document Transaction Mechanics

HOW MULTI-DOCUMENT TRANSACTIONS WORK IN WIREDTIGER:

    STEP 1: session.start_transaction()
        → WiredTiger creates a SNAPSHOT of the data
        → Snapshot = point-in-time view (like Redshift query snapshot)
        → All reads in this transaction see THIS snapshot
        → Snapshot ID stored in session context
        → No locks acquired yet — just a snapshot marker

    STEP 2: First write operation within transaction
        → Document-level WRITE LOCK acquired on target document
        → Lock held for ENTIRE transaction duration
        → Other transactions trying to write same document:
          → Get WriteConflict error immediately
          → Must abort and retry (first-writer-wins)
        → Other NON-TRANSACTIONAL writes to same document:
          → Also get WriteConflict — must wait or retry

    STEP 3: Subsequent operations within transaction
        → Each read: uses the snapshot from Step 1
          → Sees data as of transaction start
          → Doesn't see other transactions' uncommitted writes
        → Each write: acquires additional document locks
          → More documents locked = more contention
          → More documents locked = higher abort probability

    STEP 4: session.commit_transaction()
        → All modifications written to oplog as SINGLE entry
          → One oplog entry for entire transaction (not per operation)
          → Replicated to secondaries as atomic unit
        → w:"majority": wait for majority to confirm
        → Journal entry written (if j:true)
        → ALL document locks released simultaneously
        → Snapshot discarded

    STEP 4 (alternative): session.abort_transaction()
        → All modifications DISCARDED (never written to storage)
        → All document locks released
        → Snapshot discarded
        → As if the transaction never happened

    INTERNAL DATA FLOW:

    ┌──────────────────────────────────────────────────────────┐
    │  start_transaction()                                      │
    │  ├── Snapshot created (T=100)                             │
    │  │                                                        │
    │  ├── updateOne(ACC-001) → lock ACC-001                    │
    │  │   ├── Read ACC-001 from snapshot T=100                 │
    │  │   ├── Apply $inc: balance -= 500 (in memory)           │
    │  │   └── Modified doc stored in transaction's write set   │
    │  │                                                        │
    │  ├── updateOne(ACC-002) → lock ACC-002                    │
    │  │   ├── Read ACC-002 from snapshot T=100                 │
    │  │   ├── Apply $inc: balance += 500 (in memory)           │
    │  │   └── Modified doc stored in transaction's write set   │
    │  │                                                        │
    │  ├── insertOne(transfer_log) → lock new doc               │
    │  │   └── New doc stored in transaction's write set        │
    │  │                                                        │
    │  └── commit_transaction()                                 │
    │      ├── Write ALL modified docs to storage               │
    │      ├── Create SINGLE oplog entry for all changes        │
    │      ├── Wait for w:"majority" confirmation               │
    │      ├── Release ALL locks (ACC-001, ACC-002, transfer)   │
    │      └── Discard snapshot                                 │
    │                                                           │
    │  DURING TRANSACTION (between start and commit):           │
    │  ├── Other readers: see OLD data (pre-transaction)        │
    │  ├── Other writers to ACC-001/ACC-002: WriteConflict      │
    │  ├── Other writers to OTHER documents: unaffected          │
    │  └── Transaction's reads: always see snapshot T=100       │
    │                                                           │
    │  AFTER COMMIT:                                            │
    │  ├── All readers see new balances                          │
    │  ├── All locks released                                   │
    │  └── Oplog entry replicated to secondaries                │
    └──────────────────────────────────────────────────────────┘

    WRITE CONFLICT DETECTION (optimistic concurrency):
    
    ┌────────────────────────────────────────────────────────┐
    │  Txn A starts (snapshot T=100)                          │
    │  Txn B starts (snapshot T=100)                          │
    │                                                         │
    │  Txn A: updateOne(ACC-001) → LOCK acquired ✓            │
    │  Txn B: updateOne(ACC-001) → CONFLICT! ✗                │
    │         → ACC-001 is locked by Txn A                    │
    │         → WriteConflict error returned immediately       │
    │         → Txn B must abort (no waiting, no blocking)     │
    │                                                         │
    │  Txn A: commit → LOCK released                           │
    │  Txn B: retry → new snapshot (T=101) → succeeds          │
    │                                                         │
    │  CONTRAST WITH REDSHIFT:                                │
    │  → Txn B would WAIT (block) for Txn A to commit         │
    │  → No WriteConflict error — just blocking                │
    │  → Can lead to deadlocks (Redshift detects & resolves)  │
    │                                                         │
    │  MongoDB: OPTIMISTIC — fail fast, retry                  │
    │  Redshift: PESSIMISTIC — wait, potentially deadlock      │
    └────────────────────────────────────────────────────────┘

    OPLOG ENTRY FOR TRANSACTION:
    
    Without transaction (3 separate operations):
        oplog entry 1: update ACC-001
        oplog entry 2: update ACC-002
        oplog entry 3: insert transfer_log
        → 3 separate oplog entries
        → Secondaries apply them independently
        → Brief window where ACC-001 is updated but ACC-002 isn't
    
    With transaction (1 atomic commit):
        oplog entry: {
            op: "c",  // command
            o: {
                applyOps: [
                    { op: "u", ns: "ecommerce.accounts", ... },  // ACC-001
                    { op: "u", ns: "ecommerce.accounts", ... },  // ACC-002
                    { op: "i", ns: "ecommerce.transfer_log", ... }
                ]
            }
        }
        → ONE oplog entry containing ALL operations
        → Secondaries apply ALL changes atomically
        → No intermediate state ever visible on secondaries

Redshift Internals — For Comparison

HOW REDSHIFT TRANSACTIONS WORK:

    STEP 1: BEGIN
        → Transaction ID assigned
        → Snapshot timestamp assigned (current commit state)
        → All reads will use this snapshot
    
    STEP 2: SQL statements execute
        → Each statement operates on the snapshot
        → Writes are buffered in transaction log
        → ROW-LEVEL locks acquired (within compute nodes)
        → If another transaction holds conflicting lock: WAIT
          (not fail — unlike MongoDB's WriteConflict)
    
    STEP 3: COMMIT
        → All buffered writes made permanent
        → Commit record written to transaction log
        → Locks released
        → New data visible to subsequent transactions
    
    KEY DIFFERENCES:
    ┌────────────────────────────────┬─────────────────┬─────────────────────┐
    │ BEHAVIOR                        │ MONGODB           │ REDSHIFT             │
    ├────────────────────────────────┼─────────────────┼─────────────────────┤
    │ Conflict handling               │ Fail immediately  │ Wait (block)         │
    │                                 │ (WriteConflict)   │ (or deadlock detect) │

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 28:

PITFALL 1: "I'll use transactions everywhere — just like Redshift"
    ❌ Every database operation wrapped in start_transaction/commit
    ✅ Only use transactions when modifying MULTIPLE documents atomically
    WHY: Redshift transactions are cheap (~1.1x overhead)
         MongoDB transactions are expensive (~10-50x overhead)
         Using transactions for single-document ops = massive waste
         Review Project 27: single-doc operations are already atomic

PITFALL 2: "I don't need retry logic — Redshift handles that"
    ❌ session.start_transaction()
       # ... operations ...
       session.commit_transaction()  # might throw TransientTransactionError
       # No retry — application crashes intermittently
    ✅ Use run_transaction_with_retry() wrapper for EVERY transaction
    WHY: MongoDB is distributed — transient failures are NORMAL
         Elections, network blips, write conflicts happen
         Redshift handles retries internally — MongoDB does NOT
         Without retry: random production failures guaranteed

PITFALL 3: "Long transaction with external API calls inside"
    ❌ session.start_transaction()
       db.accounts.update_one(..., session=session)  # lock acquired
       response = requests.post("https://payment-api/charge")  # 3 seconds
       db.orders.insert_one(..., session=session)
       session.commit_transaction()
       # Locks held for 3+ seconds during API call
    ✅ response = requests.post("https://payment-api/charge")  # BEFORE txn
       session.start_transaction()
       db.accounts.update_one(..., session=session)
       db.orders.insert_one(..., session=session)
       session.commit_transaction()
       # Locks held for milliseconds only
    WHY: Every millisecond the transaction is open = locks held
         External API call: 100ms to 10 seconds of unnecessary locking
         Other writers to same documents: blocked or conflicted
         PREPARE outside, EXECUTE inside

PITFALL 4: "Catching all errors the same way"
    ❌ except Exception as e:
           session.abort_transaction()
           print(f"Error: {e}")  # lost information about error type
    ✅ Separate handling for:
       → ValueError (business error): abort, don't retry, return error
       → TransientTransactionError: abort, retry entire transaction
       → UnknownTransactionCommitResult: retry just the commit
       → Other errors: abort, log, alert
    WHY: Different errors need different responses
         Retrying a business error = infinite loop
         Not retrying a transient error = unnecessary failure

PITFALL 5: "Using transactions on standalone MongoDB"
    ❌ client = MongoClient("mongodb://localhost:27017/")
       # Standalone mongod — no replica set
       session.start_transaction()  # FAILS
    ✅ Use replica set (even single-node for development)
       mongod --replSet rs0
       rs.initiate()
    WHY: Multi-document transactions REQUIRE replica set
         The oplog is needed for transaction atomicity
         Atlas: always a replica set (works automatically)
         Local dev: must configure replica set explicitly

PITFALL 6: "Assuming write conflicts are bugs"
    ❌ "WriteConflict errors in logs — something is broken!"
    ✅ WriteConflict is NORMAL in concurrent systems
       → Two transactions modify same document = expected conflict
       → First writer wins, second retries
       → run_transaction_with_retry handles this automatically
    WHY: MongoDB uses optimistic concurrency (fail fast, retry)
         Redshift uses pessimistic concurrency (wait/block)
         WriteConflict in MongoDB = normal concurrency behavior
         High WriteConflict rate = hot document problem (redesign)
