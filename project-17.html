PROJECT 17: Outlier Pattern — Handling Unbounded Arrays

Problem: Social Media Platform Where Most Users Have Few Followers, Celebrities Have Millions

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    You're building a social media platform's "following" feature
    
    THE DATA DISTRIBUTION (Power Law / Pareto):
    ├── 95% of users have 1-100 followers        → "normal" users
    ├── 4% of users have 100-10,000 followers     → "popular" users
    ├── 0.9% of users have 10,000-100,000         → "influencers"
    └── 0.1% of users have 100,000-50,000,000     → "celebrities"
    
    THE NUMBERS:
    ├── 10 million users on the platform
    ├── 9,500,000 users with ≤100 followers (embed easily)
    ├── 400,000 users with 100-10,000 followers (getting tight)
    ├── 90,000 users with 10,000-100,000 followers (won't fit)
    └── 10,000 users with 100,000+ followers (impossible to embed)
    
    THE EMBEDDING MATH:
    ├── Each follower reference: ObjectId (12 bytes) + metadata (~30 bytes) = ~42 bytes
    ├── 100 followers: 4.2 KB → ✅ fits easily in document
    ├── 10,000 followers: 420 KB → ⚠️ large document, cache pressure
    ├── 100,000 followers: 4.2 MB → ⚠️ approaching limit
    ├── 380,000 followers: ~16 MB → ❌ HITS 16MB DOCUMENT LIMIT
    └── 50,000,000 followers: 2.1 GB → ❌❌❌ absolutely impossible
    
    THE DILEMMA:
    → For 95% of users: embedding followers is PERFECT
      (fast reads, atomic updates, no $lookup)
    → For 5% of users: embedding BREAKS
      (document too large, write amplification, cache destruction)
    → For 0.1% of users: embedding is IMPOSSIBLE
      (exceeds 16MB hard limit)
    
    YOU CAN'T DESIGN FOR JUST THE COMMON CASE
    YOU CAN'T DESIGN FOR JUST THE EXTREME CASE
    YOU NEED A PATTERN THAT HANDLES BOTH
    
    THIS IS THE OUTLIER PATTERN

-- REDSHIFT ANALYSIS
-- ═══════════════════════════════════════════════════════

-- STRENGTHS:
-- ✅ Same design works for ALL users (1 follower or 50 million)
-- ✅ No document size concerns
-- ✅ SQL JOINs for complex social graph queries
-- ✅ Analytical queries: "follower growth trends by month"
-- ✅ DISTKEY(following_id) co-locates followers of same user
--
-- WEAKNESSES:
-- ❌ "Is A following B?" = JOIN + point lookup → 50-200ms
--    (MongoDB with embedding: 1-3ms)
-- ❌ "Get user profile" = always a separate table query
--    (MongoDB with embedding: profile + followers in 1 read)
-- ❌ High concurrency: celebrity gets 1000 new followers/minute
--    → 1000 INSERTs/minute to follows table
--    → 1000 UPDATEs to user.follower_count
--    → Redshift can't handle this write volume
-- ❌ Real-time "follow" operation = ~200-500ms latency

Why Redshift's Uniform Approach Has Hidden Costs

REDSHIFT TREATS ALL USERS THE SAME:
    → User with 5 followers: separate table, JOIN to read → overkill
    → User with 50M followers: separate table, JOIN to read → appropriate
    
    FOR 95% OF USERS:
    → The JOIN is unnecessary overhead
    → 5 follower IDs could easily be embedded in the user document
    → But Redshift has no concept of "embed" → always JOIN
    
    THIS IS ACCEPTABLE IN REDSHIFT because:
    → JOINs are cheap (DISTSTYLE ALL or co-located data)
    → Columnar storage handles it efficiently
    → No per-row overhead concerns
    
    BUT IN MONGODB:
    → $lookup (JOIN) is expensive (nested loop)
    → For 95% of users, embedding avoids $lookup entirely
    → We WANT to embed when possible
    → We NEED to reference when embedding breaks
    → THE OUTLIER PATTERN: embed for most, overflow for outliers

🟢 MONGODB MINDSET — The Outlier Pattern

Core Concept

THE OUTLIER PATTERN:
    1. Embed data in the parent document BY DEFAULT
    2. Set a THRESHOLD (max array size before overflow)
    3. When threshold is exceeded → mark document with "has_extras" flag
    4. Store overflow data in a separate collection
    5. Application checks has_extras flag → if true, query overflow collection
    
    FOR 95% OF READS:
    → has_extras = false
    → All data is in the document
    → ONE read, ZERO lookups
    → Same speed as full embedding
    
    FOR 5% OF READS (outliers):
    → has_extras = true
    → Embedded data + separate query for overflow
    → TWO reads (still fast — both indexed)
    → Handles unlimited growth
    
    RESULT:
    → 95% of users get MAXIMUM read performance (embedded)
    → 5% of users get GOOD read performance (embedded + overflow)
    → 100% of users fit within 16MB document limit
    → Storage is efficient (no overhead for normal users)

THE OUTLIER PATTERN — DESIGN DECISIONS:

DECISION 1: Threshold — When to Overflow
    ┌──────────────────────────────────────────────────────┐
    │ FACTOR                    │ CONSIDERATION              │
    ├───────────────────────────┼────────────────────────────┤
    │ Target document size      │ Keep under 500KB-1MB       │
    │                           │ (comfortable WiredTiger    │
    │                           │  cache citizen)            │
    ├───────────────────────────┼────────────────────────────┤
    │ Array item size           │ 42 bytes per follower ref  │
    │                           │ 1000 followers = ~42 KB    │
    │                           │ 10,000 followers = ~420 KB │
    ├───────────────────────────┼────────────────────────────┤
    │ 95th percentile           │ If 95% of users have ≤100  │
    │                           │ → threshold around 1000    │
    │                           │ (10x headroom above 95th)  │
    ├───────────────────────────┼────────────────────────────┤
    │ Write frequency           │ How often does array grow? │
    │                           │ Popular user: 100 follows/ │
    │                           │ day → hits 1000 in 10 days │
    └──────────────────────────┴────────────────────────────┘
    
    FOR OUR CASE: THRESHOLD = 1000 followers embedded
    → 1000 × 42 bytes = 42 KB (comfortable)
    → 95% of users never overflow
    → 5% overflow → has_extras flag + overflow collection

DECISION 2: Overflow Document Structure
    → Option A: One overflow doc per page (1000 followers each)
    → Option B: One overflow doc per user (unbounded array) ← defeats purpose
    → Option C: Use separate collection like Redshift (full reference)
    
    CHOICE: Option A — paged overflow documents
    → Each overflow page holds up to 1000 followers
    → Celebrity with 50M followers = 50,000 overflow pages
    → Each page independently readable

DECISION 3: What to Embed vs What to Reference in Followers
    → Just follower user_id? (12 bytes — minimal)
    → Follower user_id + username + avatar? (50 bytes — useful for display)
    → CHOICE: user_id + username + avatar (avoid $lookup for follower list)
    → Same subset embedding thinking from Project 14

// ═══════════════════════════════════════════════════════
// STEP 2: THE FOLLOW FUNCTION — Core of Outlier Pattern
// ═══════════════════════════════════════════════════════

function followUser(followerId, followingId) {
    /**
     * User followerId starts following user followingId.
     * 
     * LOGIC:
     * 1. Get follower's display info (for embedding)
     * 2. Check if target user's embedded array is under threshold
     *    → YES: $push to embedded followers array
     *    → NO:  Insert into overflow collection + set has_extras flag
     * 3. Update counts
     * 
     * REDSHIFT EQUIVALENT:
     *   INSERT INTO social.follows (follower_id, following_id) VALUES (...);
     *   UPDATE social.users SET follower_count = follower_count + 1 WHERE user_id = following_id;
     *   UPDATE social.users SET following_count = following_count + 1 WHERE user_id = follower_id;
     */
    
    // Get follower info for embedding
    const follower = db.users.findOne(
        { _id: followerId },
        { username: 1, avatar_url: 1 }
    )
    
    if (!follower) return { error: "Follower not found" }
    
    const followerEntry = {
        user_id: followerId,
        username: follower.username,
        avatar_url: follower.avatar_url,
        followed_at: new Date()
    }
    
    // Try to push to embedded array (only if under threshold)
    const embedResult = db.users.updateOne(
        {
            _id: followingId,
            // CRITICAL: only push if array hasn't hit threshold
            // This is the GATE that controls overflow
            $expr: { $lt: [{ $size: "$followers" }, FOLLOWER_THRESHOLD] }
        },
        {
            $push: { followers: followerEntry },
            $inc: { follower_count: 1 }
        }
    )
    
    if (embedResult.modifiedCount === 1) {
        // SUCCESS: follower embedded in user document
        // Update follower's following_count
        db.users.updateOne(
            { _id: followerId },
            { $inc: { following_count: 1 } }
        )
        return { embedded: true }
    }
    
    // OVERFLOW: embedded array is full → use overflow collection
    // First check: has this user overflowed before?
    const targetUser = db.users.findOne({ _id: followingId })
    
    if (!targetUser) return { error: "Target user not found" }
    
    // Find current overflow page (or create new one)
    const currentPage = db.followers_overflow.findOne(
        { user_id: followingId, is_full: false },
        { sort: { page: -1 } }
    )
    
    if (currentPage && currentPage.count < OVERFLOW_PAGE_SIZE) {
        // Push to existing overflow page
        db.followers_overflow.updateOne(
            { _id: currentPage._id },
            {
                $push: { followers: followerEntry },
                $inc: { count: 1 },
                $set: {
                    is_full: (currentPage.count + 1) >= OVERFLOW_PAGE_SIZE
                }
            }
        )
    } else {
        // Create new overflow page
        const newPageNum = currentPage ? currentPage.page + 1 : 1
        db.followers_overflow.insertOne({
            user_id: followingId,
            page: newPageNum,
            followers: [followerEntry],
            count: 1,
            is_full: false,
            created_at: new Date()
        })
    }
    
    // Update user document: mark has_extras, increment counts
    db.users.updateOne(
        { _id: followingId },
        {
            $set: { has_extras: true },
            $inc: {
                follower_count: 1,
                overflow_count: 1
            }
        }
    )
    
    // Update follower's following_count
    db.users.updateOne(
        { _id: followerId },
        { $inc: { following_count: 1 } }
    )
    
    return { embedded: false, overflow: true }
}

function getFollowers(userId, page, pageSize) {
    /**
     * Get paginated follower list for any user.
     * 
     * STRATEGY:
     * 1. Read user document (has embedded followers)
     * 2. If page 1 AND enough embedded → return from embedded array
     * 3. If has_extras AND need more → query overflow collection
     * 
     * This function handles BOTH normal and celebrity users
     * with the same API — the outlier pattern is transparent.
     */
    
    pageSize = pageSize || 20
    const skip = (page - 1) * pageSize
    
    const user = db.users.findOne(
        { _id: userId },
        { followers: 1, has_extras: 1, follower_count: 1 }
    )
    
    if (!user) return { error: "User not found" }
    
    const embeddedCount = user.followers.length
    
    // Can we serve this page entirely from embedded followers?
    if (skip + pageSize <= embeddedCount) {
        // YES — serve from embedded array (reverse for newest first)
        const reversed = [...user.followers].reverse()
        return {
            followers: reversed.slice(skip, skip + pageSize),
            total: user.follower_count,
            page: page,
            source: "embedded"
        }
    }
    
    // Do we need overflow data?
    if (!user.has_extras) {
        // No overflow — serve whatever we have from embedded
        const reversed = [...user.followers].reverse()
        return {
            followers: reversed.slice(skip, skip + pageSize),
            total: user.follower_count,
            page: page,
            source: "embedded (partial)"
        }
    }
    
    // OVERFLOW: need to query overflow collection
    // Calculate which overflow page(s) we need
    const overflowSkip = Math.max(0, skip - embeddedCount)
    
    const overflowFollowers = db.followers_overflow.aggregate([
        { $match: { user_id: userId } },
        { $sort: { page: -1 } },           // newest pages first
        { $unwind: "$followers" },
        { $sort: { "followers.followed_at": -1 } },
        { $skip: overflowSkip },
        { $limit: pageSize },
        { $replaceRoot: { newRoot: "$followers" } }
    ]).toArray()
    
    return {
        followers: overflowFollowers,
        total: user.follower_count,
        page: page,
        source: "overflow"
    }
}

function isFollowing(followerId, followingId) {
    /**
     * Check if followerId follows followingId.
     * 
     * STRATEGY:
     * 1. Check embedded followers array first (fast)
     * 2. If has_extras AND not found → check overflow collection
     * 
     * FOR NORMAL USERS: one query, instant answer
     * FOR CELEBRITIES: one query (likely found in embedded for recent followers)
     *                  OR two queries (check overflow if not found in embedded)
     * 
     * REDSHIFT EQUIVALENT:
     *   SELECT 1 FROM social.follows
     *   WHERE follower_id = ? AND following_id = ?;
     *   -- Always same cost: ~50-200ms
     */
    
    // Check embedded first
    const inEmbedded = db.users.findOne({
        _id: followingId,
        "followers.user_id": followerId
    })
    
    if (inEmbedded) return { following: true, source: "embedded" }
    
    // Check if overflow exists
    const user = db.users.findOne(
        { _id: followingId },
        { has_extras: 1 }
    )
    
    if (!user || !user.has_extras) return { following: false, source: "embedded" }
    
    // Check overflow collection
    const inOverflow = db.followers_overflow.findOne({
        user_id: followingId,
        "followers.user_id": followerId
    })
    
    return {
        following: !!inOverflow,
        source: "overflow"
    }
}

function unfollowUser(followerId, followingId) {
    /**
     * Remove follow relationship.
     * Must check both embedded array AND overflow collection.
     */
    
    // Try to remove from embedded array
    const embedResult = db.users.updateOne(
        { _id: followingId },
        {
            $pull: { followers: { user_id: followerId } },
            $inc: { follower_count: -1 }
        }
    )
    
    if (embedResult.modifiedCount === 1) {
        db.users.updateOne({ _id: followerId }, { $inc: { following_count: -1 } })
        return { removed_from: "embedded" }
    }
    
    // Try overflow collection
    const overflowResult = db.followers_overflow.updateOne(
        { user_id: followingId, "followers.user_id": followerId },
        {
            $pull: { followers: { user_id: followerId } },
            $inc: { count: -1 }
        }
    )
    
    if (overflowResult.modifiedCount === 1) {
        db.users.updateOne(
            { _id: followingId },
            { $inc: { follower_count: -1, overflow_count: -1 } }
        )
        db.users.updateOne({ _id: followerId }, { $inc: { following_count: -1 } })
        return { removed_from: "overflow" }
    }
    
    return { error: "Follow relationship not found" }
}

✅ WHAT SHOULD DO / ❌ WHAT SHOULD NOT DO

SHOULD DO:
    ✅ Choose threshold based on 95th percentile of your data
       → If 95% of users have ≤100 items: threshold = 1000 (10x headroom)
       → Keeps 95%+ of reads as single-document operations
    
    ✅ Use has_extras as a boolean flag (not a count)
       → Simple check: if has_extras → query overflow
       → Application logic stays clean
    
    ✅ Store overflow_count for accurate total without scanning overflow
       → follower_count = embedded.length + overflow_count
       → Display "50M followers" without counting overflow pages
    
    ✅ Keep most recent items in embedded array
       → Profile page shows recent followers → embedded = instant
       → Older followers in overflow → "load more" queries overflow
    
    ✅ Use same embedded structure in overflow pages
       → Same { user_id, username, avatar_url, followed_at } shape
       → Application code works identically for both sources
    
    ✅ Index the overflow collection on { user_id, page }
       → Paginated reads of overflow are fast
       → Also index { "followers.user_id" } for isFollowing checks

SHOULD NOT DO:
    ❌ Use different thresholds for different collections
       → Pick ONE threshold strategy, apply consistently
       → Reduces cognitive overhead
    
    ❌ Move ALL data to overflow when threshold is hit
       → Keep first 1000 embedded (hot data)
       → Only NEW followers go to overflow
       → Embedded data serves 95% of reads
    
    ❌ Forget to check overflow on unfollow/delete
       → Must check BOTH embedded and overflow for removal
       → Missing overflow check = data inconsistency
    
    ❌ Use outlier pattern when data is ALWAYS large
       → If ALL users have 10,000+ followers → just use reference
       → Outlier pattern is for BIMODAL distributions (few large, many small)
    
    ❌ Over-engineer: apply outlier pattern to every array
       → Tags (bounded to ~10) → just embed, no outlier needed
       → Cart items (bounded to ~50) → just embed
       → Outlier pattern for genuinely unbounded, power-law distributions

WIREDTIGER BEHAVIOR — WHY DOCUMENT SIZE MATTERS:

SCENARIO A: All users with full embedding (no outlier pattern)

    └────────────────────────────────────────────────────────┘

SCENARIO B: Write amplification

    FULL EMBEDDING — Adding follower to 2.5 MB document:
    1. Read 2.5 MB document from disk (or cache)
    2. Append 42 bytes to followers array
    3. Write ENTIRE 2.5 MB document back
    4. If document grew beyond page allocation:
       → Document migration: move to new location
       → Update ALL indexes pointing to this document
       → This is O(document_size) not O(change_size)
    
    WRITE AMPLIFICATION RATIO: 2.5 MB written for 42 bytes of new data
    = 60,000x amplification!
    
    OUTLIER PATTERN — Adding follower to overflow page:
    1. Overflow page is ~50 KB (1000 entries)
    2. Append 42 bytes to page's followers array
    3. Write ~50 KB back
    4. OR create new page: write ~42 bytes
    
    WRITE AMPLIFICATION RATIO: 50 KB or 42 bytes
    = 1,200x or 1x amplification
    = 50x improvement over full embedding


REDSHIFT COMPARISON:
    → Redshift has ZERO document size concerns
    → Each row is independent (no arrays, no embedding)
    → INSERT one row = write to column blocks
    → No write amplification from document growth
    → BUT: columnar write overhead for single rows (~50-200ms)
    → AND: no concept of embedding (always JOIN)

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I'll Just Increase the 16MB Limit" (Impossible)
    ❌ Searching for a MongoDB config to raise document size limit
    REALITY:
        → 16MB is a HARD LIMIT in the BSON specification
        → Cannot be changed, configured, or overridden
        → It exists for good reason: protects network, memory, and storage
    FIX: Design around it with outlier pattern

PITFALL 2: "Threshold Too Low" — Excessive Overflow
    ❌ Setting threshold to 50 when 60% of users have 50-500 items
    SYMPTOMS:
        → 60% of reads now require overflow queries
        → Lost the embedding performance advantage for majority
        → Overflow collection becomes massive
    FIX: Analyze data distribution FIRST
         → db.users.aggregate([
               { $bucket: { 
                   groupBy: "$follower_count",
                   boundaries: [0, 10, 100, 1000, 10000, 100000, Infinity],
                   output: { count: { $sum: 1 } }
               }}
           ])
         → Set threshold above 95th percentile

PITFALL 3: "Threshold Too High" — Cache Destruction
    ❌ Setting threshold to 50,000 to avoid overflow complexity
    SYMPTOMS:
        → Documents with 50,000 embedded items = ~2MB each
        → WiredTiger cache filled by few large documents
        → Normal user documents evicted constantly
        → Global read performance degrades
    FIX: Threshold should keep documents under ~500KB
         → 500KB / 42 bytes per entry ≈ 12,000 max threshold
         → But consider cache impact: 1000-5000 is usually optimal

PITFALL 4: "Forgot to Handle Overflow in Unfollow"
    ❌ Only checking embedded array when removing a follower
    SYMPTOMS:
        → Unfollow appears to succeed (no error)
        → But follower still exists in overflow collection
        → follower_count decremented but data remains
        → Inconsistency accumulates over time
    FIX: Always check BOTH embedded AND overflow for any removal
         → Wrap in application-level check:
           removed = removeFromEmbedded()
           if not removed and has_extras: removeFromOverflow()

PITFALL 5: "Not Updating has_extras When Overflow Empties"
    ❌ User's overflow pages are all empty but has_extras still true
    SYMPTOMS:
        → Every read for this user unnecessarily checks overflow
        → Wasted queries for empty overflow collection
    FIX: When overflow_count reaches 0, set has_extras = false
         → Add to unfollow logic:
           if overflow_count becomes 0: $set has_extras = false
         → Or periodic reconciliation job

PITFALL 6: "Using Outlier Pattern When Simple Reference Works"
    ❌ Applying outlier pattern to every one-to-many relationship
    REALITY:
        → Outlier pattern adds application complexity
        → Only justified when data has BIMODAL distribution
        → If ALL entities have large arrays → just use reference collection
        → If ALL entities have small arrays → just embed
    DECISION TREE:
        → Distribution uniform and small? → EMBED (Project 14)
        → Distribution uniform and large? → REFERENCE (Project 14)
        → Distribution bimodal (few large, many small)? → OUTLIER ✅

PITFALL 7: "Redshift Developer Skips Outlier Pattern Entirely"
    ❌ "I'll just use a separate collection for everything like Redshift tables"
    COST:
        → 95% of users now pay $lookup cost unnecessarily
        → Profile page: 5-20ms instead of 1-3ms
        → isFollowing check: 5-20ms instead of 1-3ms
        → At scale: millions of unnecessary $lookups per day
    FIX: Use outlier pattern to get embedding benefits for the majority
         while handling outliers gracefully

📈 SCALABILITY CONSIDERATIONS

SCALING THE OUTLIER PATTERN:

─── AT 10 MILLION USERS ───

    Users collection:
    → 10M documents
    → 95% with embedded followers (small docs ~5-50 KB)
    → 5% with has_extras=true (bounded docs ~50-500 KB)
    → Total: ~200 GB estimated
    → Working set (active users): fits in 32-64 GB RAM ✅
    
    Overflow collection:
    → ~500K users × avg 10 overflow pages = 5M overflow docs
    → Each page ~50 KB → total ~250 GB
    → Indexed by { user_id, page } → fast paginated reads
    → Indexed by { "followers.user_id" } → fast isFollowing checks
    
    SHARDING STRATEGY:
    → Users collection: shard by _id (hashed)
      → Uniform distribution across shards
      → Profile reads: targeted to one shard
    → Overflow collection: shard by user_id (hashed)
      → Co-locates overflow pages with same user's shard
      → Paginated reads: targeted to one shard
    
    ┌──────────────────────────────────────────────────┐
    │ CRITICAL: Shard overflow by user_id, NOT by _id   │
    │                                                    │
    │ If sharded by _id: overflow pages for same user   │
    │ scattered across shards → slow paginated reads     │
    │                                                    │
    │ If sharded by user_id: all pages for same user    │
    │ on same shard → fast paginated reads               │
    │                                                    │
    │ Redshift equivalent: DISTKEY(following_id) on      │
    │ follows table → same co-location principle         │
    └──────────────────────────────────────────────────┘


─── CELEBRITY PROBLEM AT SCALE ───

    Celebrity with 50M followers:
    → Embedded: 1000 followers (most recent)
    → Overflow: 49,999 pages × 1000 followers each
    
    "Load more followers" (page 500):
    → Query: db.followers_overflow.find({ user_id: x, page: 500 })
    → ONE indexed document read → ~2-5ms
    → Excellent performance even for deep pagination
    
    "New follower" for celebrity (1000 new followers/minute):
    → Each follow: updateOne on current overflow page
    → When page fills: create new page
    → User document update: $inc follower_count
    → All writes targeted to ONE shard (user_id)
    → Potential HOT SHARD problem for mega-celebrities
    
    HOT SHARD MITIGATION:
    → Option A: Separate "celebrity follows" queue
      → Buffer follows in memory/Redis
      → Batch-write to MongoDB every 10 seconds
    → Option B: Fan-out writes across multiple overflow docs
      → Random page assignment for new followers
      → Spreads writes across shard
    → Option C: Accept hot shard (if write volume manageable)
      → 1000 writes/minute = ~17/second → single shard handles this

class SocialGraph:
    """
    Social graph operations using the Outlier Pattern.
    
    Redshift equivalent would be simple SQL on follows table:
        INSERT INTO follows (follower_id, following_id) VALUES (?, ?);
        DELETE FROM follows WHERE follower_id = ? AND following_id = ?;
        SELECT 1 FROM follows WHERE follower_id = ? AND following_id = ?;
    
    MongoDB outlier pattern provides:
        → 95% of reads served from embedded array (1-3ms)
        → Overflow collection handles unbounded growth
        → has_extras flag avoids unnecessary overflow checks
    """
    
    def __init__(self, db, threshold=1000, page_size=1000):
        self.db = db
        self.users = db.users
        self.overflow = db.followers_overflow
        self.threshold = threshold
        self.page_size = page_size
    
    def follow(self, follower_id, following_id):
        """
        User follower_id starts following user following_id.
        
        Returns: { "status": "embedded" | "overflow", "success": bool }
        """
        # Get follower info for embedding
        follower = self.users.find_one(
            {"_id": follower_id},
            {"username": 1, "avatar_url": 1}
        )
        if not follower:
            return {"status": "error", "message": "Follower not found"}
        
        follower_entry = {
            "user_id": follower_id,
            "username": follower["username"],
            "avatar_url": follower.get("avatar_url", ""),
            "followed_at": datetime.now()
        }
        
        # Attempt 1: Push to embedded array (if under threshold)
        result = self.users.update_one(
            {
                "_id": following_id,
                "$expr": {"$lt": [{"$size": "$followers"}, self.threshold]}
            },
            {
                "$push": {"followers": follower_entry},
                "$inc": {"follower_count": 1}
            }
        )
        
        if result.modified_count == 1:
            self.users.update_one(
                {"_id": follower_id},
                {"$inc": {"following_count": 1}}
            )
            return {"status": "embedded", "success": True}
        
        # Attempt 2: Overflow to separate collection
        # Find current open page
        current_page = self.overflow.find_one(
            {"user_id": following_id, "is_full": False},
            sort=[("page", DESCENDING)]
        )
        
        if current_page and current_page["count"] < self.page_size:
            # Add to existing page
            new_count = current_page["count"] + 1
            self.overflow.update_one(
                {"_id": current_page["_id"]},
                {
                    "$push": {"followers": follower_entry},
                    "$inc": {"count": 1},
                    "$set": {"is_full": new_count >= self.page_size}
                }
            )
        else:
            # Create new page
            new_page_num = (current_page["page"] + 1) if current_page else 1
            self.overflow.insert_one({
                "user_id": following_id,
                "page": new_page_num,
                "followers": [follower_entry],
                "count": 1,
                "is_full": False,
                "created_at": datetime.now()
            })
        
        # Update user: set has_extras flag + increment counts
        self.users.update_one(
            {"_id": following_id},
            {
                "$set": {"has_extras": True},
                "$inc": {"follower_count": 1, "overflow_count": 1}
            }
        )
        self.users.update_one(
            {"_id": follower_id},
            {"$inc": {"following_count": 1}}
        )
        
        return {"status": "overflow", "success": True}
    
    def unfollow(self, follower_id, following_id):
        """
        Remove follow relationship.
        Must check both embedded and overflow.
        """
        # Try embedded first
        result = self.users.update_one(
            {"_id": following_id},
            {
                "$pull": {"followers": {"user_id": follower_id}},
                "$inc": {"follower_count": -1}
            }
        )
        
        if result.modified_count == 1:
            self.users.update_one(
                {"_id": follower_id},
                {"$inc": {"following_count": -1}}
            )
            return {"status": "removed_from_embedded"}
        
        # Try overflow
        result = self.overflow.update_one(
            {"user_id": following_id, "followers.user_id": follower_id},
            {
                "$pull": {"followers": {"user_id": follower_id}},
                "$inc": {"count": -1}
            }
        )
        
        if result.modified_count == 1:
            update_fields = {
                "$inc": {"follower_count": -1, "overflow_count": -1}
            }
            
            # Check if overflow is now empty
            user = self.users.find_one({"_id": following_id}, {"overflow_count": 1})
            if user and user.get("overflow_count", 1) - 1 <= 0:
                update_fields["$set"] = {"has_extras": False, "overflow_count": 0}
            
            self.users.update_one({"_id": following_id}, update_fields)
            self.users.update_one(
                {"_id": follower_id},
                {"$inc": {"following_count": -1}}
            )
            return {"status": "removed_from_overflow"}
        
        return {"status": "not_found"}
    
    def is_following(self, follower_id, following_id):
        """
        Check if follower_id follows following_id.
        
        95% of checks: resolved from embedded array (1 query)
        5% of checks: may need overflow query (2 queries)
        """
        # Check embedded
        found = self.users.find_one({
            "_id": following_id,
            "followers.user_id": follower_id
        })
        
        if found:
            return {"following": True, "source": "embedded"}
        
        # Check if overflow exists
        user = self.users.find_one(
            {"_id": following_id},
            {"has_extras": 1}
        )
        
        if not user or not user.get("has_extras", False):
            return {"following": False, "source": "embedded"}
        
        # Check overflow
        found_overflow = self.overflow.find_one({
            "user_id": following_id,
            "followers.user_id": follower_id
        })
        
        return {
            "following": bool(found_overflow),
            "source": "overflow"
        }
    
    def get_followers(self, user_id, page=1, page_size=20):
        """
        Get paginated follower list.
        Page 1 usually served from embedded array.
        Later pages may need overflow collection.
        """
        skip = (page - 1) * page_size
        
        user = self.users.find_one(
            {"_id": user_id},
            {"followers": 1, "has_extras": 1, "follower_count": 1}
        )
        
        if not user:
            return {"error": "User not found"}
        
        embedded = user.get("followers", [])
        embedded_count = len(embedded)
        
        # Can serve from embedded?
        if skip + page_size <= embedded_count:
            reversed_list = list(reversed(embedded))
            return {
                "followers": reversed_list[skip:skip + page_size],
                "total": user.get("follower_count", 0),
                "page": page,
                "source": "embedded"
            }
        
        # Need overflow?
        if not user.get("has_extras", False):
            reversed_list = list(reversed(embedded))
            return {
                "followers": reversed_list[skip:skip + page_size],
                "total": user.get("follower_count", 0),
                "page": page,
                "source": "embedded_partial"
            }
        
        # Query overflow
        overflow_skip = max(0, skip - embedded_count)
        overflow_followers = list(self.overflow.aggregate([
            {"$match": {"user_id": user_id}},
            {"$sort": {"page": DESCENDING}},
            {"$unwind": "$followers"},
            {"$sort": {"followers.followed_at": DESCENDING}},
            {"$skip": overflow_skip},
            {"$limit": page_size},
            {"$replaceRoot": {"newRoot": "$followers"}}
        ]))
        
        return {
            "followers": overflow_followers,
            "total": user.get("follower_count", 0),
            "page": page,
            "source": "overflow"
        }
    
    def get_profile(self, user_id, recent_followers=20):
        """Get user profile with recent followers."""
        return self.users.find_one(
            {"_id": user_id},
            {
                "username": 1, "display_name": 1, "bio": 1,
                "avatar_url": 1, "is_verified": 1,
                "follower_count": 1, "following_count": 1,
                "has_extras": 1,
                "followers": {"$slice": -recent_followers}
            }
        )
