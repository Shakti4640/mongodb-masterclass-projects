🔴 REDSHIFT MINDSET — "How You Already Secure Redshift"

Redshift Security Model (Your Existing Knowledge)

REDSHIFT SECURITY — YOUR MENTAL MODEL:

LAYER 1: NETWORK (who can connect)
    → VPC: Redshift cluster lives in private subnet
    → Security Groups: allow only specific IP ranges / security groups
    → Enhanced VPC Routing: all COPY/UNLOAD traffic stays in VPC
    → No public internet access (unless explicitly enabled)

LAYER 2: AUTHENTICATION (prove who you are)
    → Database users: CREATE USER analyst_user WITH PASSWORD '...'
    → IAM authentication: GetClusterCredentials API
    → Temporary credentials via IAM roles (preferred)
    → No multi-factor auth at database level (IAM handles MFA)

LAYER 3: AUTHORIZATION (what you can do)
    → Database-level: GRANT USAGE ON DATABASE
    → Schema-level: GRANT USAGE ON SCHEMA analytics
    → Table-level: GRANT SELECT ON analytics.fact_orders TO analyst_group
    → Column-level: GRANT SELECT (order_id, total, status) ON analytics.fact_orders
    → Row-level: CREATE RLS POLICY (Redshift row-level security)
    → Groups: CREATE GROUP analysts; ALTER GROUP analysts ADD USER john;

LAYER 4: ENCRYPTION
    → At rest: KMS encryption (AES-256) — always enabled for RA3
    → In transit: SSL/TLS — enforced via parameter group
    → No field-level encryption (all-or-nothing at cluster level)

LAYER 5: AUDITING
    → STL_CONNECTION_LOG: who connected when
    → STL_QUERY: what queries were run
    → STL_DDLTEXT: what DDL was executed
    → CloudTrail: API-level audit trail
    → User activity logging: enable_user_activity_logging = true

What Redshift Security CANNOT Do

LIMITATIONS OF REDSHIFT SECURITY MODEL:

❌ No field-level encryption (can't encrypt individual columns differently)
    → All data encrypted with same KMS key
    → DBA with cluster access can see everything

❌ No dynamic field masking at query time
    → Column-level GRANT is static (column visible or not)
    → Can't show "j***@email.com" instead of full email
    → Need views with masking functions (manual)

❌ No client-side encryption
    → Data is always decryptable by Redshift service
    → AWS engineers theoretically can access
    → No "zero knowledge" encryption

❌ Limited authentication methods
    → Password or IAM — no LDAP direct, no x.509 certificates
    → IAM federation provides SSO via SAML/OIDC (indirect)

❌ No database-level triggers for security events
    → Can't auto-alert on suspicious queries
    → Must use CloudTrail + CloudWatch alarms (external)

❌ No field-level audit (which fields were accessed)
    → STL_QUERY logs the SQL text but not field-level access
    → Can parse SQL text to infer column access (complex)

MONGODB ADDRESSES SEVERAL OF THESE LIMITATIONS
— especially field-level encryption and dynamic masking

Part 2: Authorization — Built-in Roles

// ═══════════════════════════════════════════════════════
// PART 2: AUTHORIZATION — Built-in Roles
//
// MongoDB has pre-defined roles (like Redshift built-in groups)
// But MongoDB roles are MORE GRANULAR than Redshift groups
//
// REDSHIFT AUTHORIZATION MODEL:
// → GRANT SELECT ON table TO group (table-level)
// → GRANT SELECT (col1, col2) ON table TO group (column-level)
// → Simple: SQL DCL commands
//
// MONGODB AUTHORIZATION MODEL:
// → Roles contain PRIVILEGES
// → Privileges = ACTIONS on RESOURCES
// → Actions: find, insert, update, remove, createIndex, etc.
// → Resources: database, collection, cluster
// → More granular than Redshift (action-level vs statement-level)
// ═══════════════════════════════════════════════════════

// ─── BUILT-IN ROLES REFERENCE ────────────────────────

// DATABASE-LEVEL ROLES:
// ┌────────────────────┬──────────────────────────────────────────┐
// │ MongoDB Role         │ Redshift Equivalent                       │
// ├────────────────────┼──────────────────────────────────────────┤
// │ read                │ GRANT SELECT ON ALL TABLES IN SCHEMA     │
// │ readWrite           │ GRANT SELECT, INSERT, UPDATE, DELETE     │
// │ dbAdmin             │ GRANT CREATE, ALTER on schema objects    │
// │ dbOwner             │ read + readWrite + dbAdmin combined      │
// │ userAdmin           │ GRANT/REVOKE on users (like superuser)  │
// ├────────────────────┼──────────────────────────────────────────┤
// │ CLUSTER-LEVEL:      │                                          │
// │ clusterAdmin        │ No direct equivalent (Redshift has no    │
// │                     │ cluster-level role concept)              │
// │ clusterMonitor      │ Like pg_monitor role                     │
// │ hostManager         │ No equivalent                            │
// │ root                │ Superuser (CREATE USER WITH CREATEDB)    │
// └────────────────────┴──────────────────────────────────────────┘

// ─── ASSIGN BUILT-IN ROLES ──────────────────────────

use admin

// Team 1: App developers — readWrite on operational database
db.grantRolesToUser("app_service", [
    { role: "readWrite", db: "shopstream" }
])
// This is like:
// GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES
// IN SCHEMA ops TO GROUP app_developers;

// Team 2: Data engineers — read on operational + write on pipeline db
db.grantRolesToUser("etl_service", [
    { role: "read", db: "shopstream" },
    { role: "readWrite", db: "pipeline_metadata" }
])
// This is like:
// GRANT SELECT ON ALL TABLES IN SCHEMA ops TO GROUP data_engineers;
// GRANT ALL ON ALL TABLES IN SCHEMA staging TO GROUP data_engineers;

// Team 5: Admin — already has root (assigned during creation)
// root = can do ANYTHING on ANY database
// Like Redshift superuser

// BUT: Built-in roles are TOO BROAD for Teams 3 and 4
// → "read" gives access to ALL collections in database
// → We need per-collection restrictions
// → We need to HIDE PII fields from analysts
// → This requires CUSTOM ROLES

Part 3: Authorization — Custom Roles (The Power Feature)

// ═══════════════════════════════════════════════════════
// PART 3: CUSTOM ROLES — Granular Access Control
//
// REDSHIFT APPROACH:
// → GRANT SELECT ON specific_table TO group
// → GRANT SELECT (col1, col2) ON table TO group
// → Simple but limited to SQL DCL
//
// MONGODB APPROACH:
// → Create custom role with specific ACTIONS on specific RESOURCES
// → Much more granular: can allow find but not aggregate
// → Can restrict to specific collections (not just database-wide)
// → BUT: no native field-level GRANT (solved with views + $redact)
// ═══════════════════════════════════════════════════════

use admin

// ─── CUSTOM ROLE: APP_DEVELOPER ──────────────────────
// Can read/write operational collections
// CANNOT access analytics or admin collections
// CANNOT drop collections or create indexes (leave that to admins)

db.createRole({
    role: "app_developer",
    privileges: [
        // Orders: full CRUD
        {
            resource: { db: "shopstream", collection: "orders" },
            actions: ["find", "insert", "update", "remove"]
        },
        // Carts: full CRUD
        {
            resource: { db: "shopstream", collection: "carts" },
            actions: ["find", "insert", "update", "remove"]
        },
        // Users/profiles: full CRUD
        {
            resource: { db: "shopstream", collection: "users" },
            actions: ["find", "insert", "update", "remove"]
        },
        // Inventory: read + update (no delete — safety)
        {
            resource: { db: "shopstream", collection: "inventory" },
            actions: ["find", "update"]
            // NO "remove" — can't delete inventory records
            // NO "insert" — inventory created by admin/import process
        },
        // Products: read only (catalog managed by content team)
        {
            resource: { db: "shopstream", collection: "products" },
            actions: ["find"]
        },
        // Allow listCollections (needed for Compass/drivers)
        {
            resource: { db: "shopstream", collection: "" },
            actions: ["listCollections"]
        }
    ],
    roles: []    // no inherited roles
})

// Assign to app service user
db.grantRolesToUser("app_service", [
    { role: "app_developer", db: "admin" }
])
// First remove the broad built-in role
db.revokeRolesFromUser("app_service", [
    { role: "readWrite", db: "shopstream" }
])

// WHAT APP_SERVICE CAN DO:
// ✅ db.orders.find({...})
// ✅ db.orders.insertOne({...})
// ✅ db.orders.updateOne({...})
// ✅ db.carts.deleteOne({...})
// ✅ db.products.find({...})
// ✅ db.inventory.find({...})
// ✅ db.inventory.updateOne({...})
//
// WHAT APP_SERVICE CANNOT DO:
// ❌ db.inventory.deleteOne({...})        → no "remove" on inventory
// ❌ db.inventory.insertOne({...})        → no "insert" on inventory
// ❌ db.products.updateOne({...})         → no "update" on products
// ❌ db.orders.createIndex({...})         → no "createIndex" action
// ❌ db.orders.drop()                     → no "dropCollection" action
// ❌ db.analytics_view.find({...})        → no access to other collections
// ❌ use admin                            → no access to admin database

// REDSHIFT EQUIVALENT:
// GRANT SELECT, INSERT, UPDATE, DELETE ON ops.orders TO GROUP app_developers;
// GRANT SELECT, INSERT, UPDATE, DELETE ON ops.carts TO GROUP app_developers;
// GRANT SELECT, UPDATE ON ops.inventory TO GROUP app_developers;
// GRANT SELECT ON ops.products TO GROUP app_developers;

// ═══════════════════════════════════════════════════════
// PART 4: FIELD-LEVEL SECURITY — The MongoDB Superpower
//
// THE PROBLEM:
// → Analysts need to query orders but CANNOT see:
//   - customer.email
//   - customer.phone
//   - shipping.address.street (full address)
//   - payment.card_last_four
//
// REDSHIFT APPROACH:
// → GRANT SELECT (order_id, total, status) ON orders TO analysts
// → Simple but rigid — column is visible or not
//
// MONGODB APPROACH:
// → Create a VIEW with $redact or $project that hides/masks fields
// → Grant analysts access to the VIEW (not the raw collection)
// → Views can DYNAMICALLY mask based on user role
// → More powerful than Redshift column-level GRANT
// ═══════════════════════════════════════════════════════

use shopstream

// ─── VIEW 1: ORDERS WITHOUT PII ──────────────────────
// Analysts see orders but PII fields are removed

db.createView(
    "v_orders_no_pii",          // view name
    "orders",                    // source collection
    [
        // $project: include safe fields, exclude PII
        {
            $project: {
                // KEEP these fields (safe for analysts)
                order_number: 1,
                status: 1,
                item_count: 1,
                unique_categories: 1,
                source: 1,
                marketing_campaign: 1,
                created_at: 1,
                updated_at: 1,
                
                // Payment: keep amounts, HIDE card info
                "payment.subtotal": 1,
                "payment.shipping": 1,
                "payment.tax": 1,
                "payment.total": 1,
                "payment.method": 1,
                // EXCLUDED: payment.card_last_four
                
                // Customer: keep segment/region, MASK PII
                "customer.segment": 1,
                "customer.region": 1,
                "customer.loyalty_tier": 1,
                // EXCLUDED: customer.email, customer.phone, customer.name
                
                // Customer ID: HASH it (pseudonymization)
                "customer_hash": {
                    $substr: [
                        { $toString: "$customer.customer_id" },
                        0, 8
                    ]
                },
                
                // Shipping: keep state/country, HIDE street address
                "shipping.address.state": 1,
                "shipping.address.country": 1,
                "shipping.address.zip": {
                    // Mask ZIP: show first 3 digits only
                    $concat: [
                        { $substr: ["$shipping.address.zip", 0, 3] },
                        "**"
                    ]
                },
                // EXCLUDED: shipping.address.street, shipping.address.city
                
                // Items: keep product info (not PII)
                "items.sku": 1,
                "items.product_name": 1,
                "items.category": 1,
                "items.quantity": 1,
                "items.subtotal": 1
                // EXCLUDED: items.unit_price (business sensitive? optional)
            }
        }
    ]
)

db.createView(
    "v_revenue_summary",
    "orders",
    [
        {
            $match: {
                status: { $in: ["delivered", "shipped", "confirmed"] }
            }
        },
        {
            $group: {
                _id: {
                    year: { $year: "$created_at" },
                    month: { $month: "$created_at" },
                    category: { $arrayElemAt: ["$unique_categories", 0] }
                },
                total_revenue: { $sum: { $toDouble: "$payment.total" } },
                order_count: { $sum: 1 },
                avg_order_value: { $avg: { $toDouble: "$payment.total" } },
                unique_customers: { $addToSet: "$customer.customer_id" }
            }
        },
        {
            $project: {
                _id: 0,
                year: "$_id.year",
                month: "$_id.month",
                category: "$_id.category",
                total_revenue: { $round: ["$total_revenue", 2] },
                order_count: 1,
                avg_order_value: { $round: ["$avg_order_value", 2] },
                unique_customers: { $size: "$unique_customers" }
            }
        },
        { $sort: { year: 1, month: 1 } }
    ]
)

// ─── ADVANCED: $redact FOR DYNAMIC FIELD MASKING ─────
// $redact goes DEEPER than $project:
// → Can conditionally include/exclude at ANY nesting level
// → Can make decisions based on field values
// → Like Redshift RLS but for FIELDS, not rows

db.createView(
    "v_orders_dynamic_redact",
    "orders",
    [
        {
            $redact: {
                $cond: {
                    // For each level of the document:
                    // If the sub-document has a field "pii_level"
                    // and it's "restricted" → prune (remove)
                    // Otherwise → descend (keep looking deeper)
                    if: { $eq: ["$pii_level", "restricted"] },
                    then: "$$PRUNE",      // remove this sub-document entirely
                    else: "$$DESCEND"     // keep this level, check children
                }
            }
        }
    ]
)

// To use this pattern: add pii_level markers to your documents:
// {
//   order_number: "ORD-001",
//   customer: {
//     pii_level: "restricted",    ← entire customer object hidden
//     name: "Sarah",
//     email: "sarah@email.com"
//   },
//   payment: {
//     pii_level: "restricted",    ← entire payment object hidden
//     total: 117.60,
//     card_last_four: "[REDACTED:CREDIT_DEBIT_NUMBER]"
//   },
//   items: [ ... ]                ← no pii_level → visible
// }
//
// Result through v_orders_dynamic_redact:
// {
//   order_number: "ORD-001",
//   items: [ ... ]
//   // customer and payment PRUNED entirely
// }
//
// REDSHIFT HAS NO EQUIVALENT TO $redact
// Closest: row-level security + column-level GRANT (but less flexible)

⚙️ HOW IT ONLY WORKS INTERNALLY

MONGODB AUTHENTICATION INTERNALS:

WHEN A CLIENT CONNECTS WITH SCRAM-SHA-256:

STEP 1: CLIENT SENDS username + authSource
    → Client: "I am app_service, authenticating against admin database"
    → Server: sends challenge (random nonce)

STEP 2: SCRAM CHALLENGE-RESPONSE
    → Client computes: HMAC(password, salt, iterations) + nonce
    → Sends proof to server
    → Server verifies against stored credential hash
    → PASSWORD NEVER SENT IN PLAINTEXT (even over TLS)
    → Unlike Redshift where password is sent over SSL tunnel
    
    SCRAM advantage over simple password:
    → Mutual authentication (client verifies server too)
    → No plaintext password on wire
    → Replay attack protection (unique nonce per session)

STEP 3: SERVER LOOKS UP USER
    → Checks admin.system.users collection
    → Finds user document with roles array
    → Caches resolved privileges for session duration

STEP 4: AUTHORIZATION CHECK (every operation)
    → Client sends: find on shopstream.orders
    → Server checks: does this user's resolved privileges
      include "find" action on { db: "shopstream", collection: "orders" }?
    → If yes: execute query
    → If no: return error code 13 (Unauthorized)
    
    EVERY SINGLE OPERATION is checked against privileges
    → Not just at connection time (like some databases)
    → Per-operation authorization

STEP 5: VIEW RESOLUTION
    → Client queries: v_orders_no_pii
    → Server recognizes: this is a VIEW on "orders" collection
    → Server checks: user has "find" on v_orders_no_pii
    → Server executes: view's aggregation pipeline on "orders"
    → Pipeline runs with SERVER'S privileges (not user's)
    → Result: user sees view output without needing access to "orders"
    
    THIS IS HOW VIEWS ENFORCE FIELD-LEVEL SECURITY:
    → User has access to VIEW only
    → View pipeline strips PII fields
    → User never has direct access to raw collection
    → Server executes view pipeline on user's behalf


REDSHIFT AUTHENTICATION INTERNALS (for comparison):

STEP 1: Client sends username + password over SSL
    → Password encrypted in transit by TLS
    → But password IS the authentication token

STEP 2: Redshift checks pg_user_info
    → Compares password hash (MD5 or SHA-256)
    → Creates session with user's privileges

STEP 3: Authorization check (per statement)
    → Checks: pg_class, pg_namespace for table/schema permissions
    → Column-level: checks pg_attribute_info
    → RLS: evaluates RLS policy predicates

NEW SECURITY TERMS IN THIS PROJECT:

    SCRAM-SHA-256:
        → Salted Challenge Response Authentication Mechanism
        → Password never sent over wire (even encrypted)
        → MongoDB default auth mechanism
        → More secure than Redshift's password-over-TLS

    Actions (MongoDB-specific):
        → Granular operations: find, insert, update, remove
        → Also: changeStream, createIndex, dropCollection, serverStatus
        → More granular than SQL's SELECT/INSERT/UPDATE/DELETE
        → Enables: "can read but can't watch changes"

    $redact:
        → Aggregation stage that prunes document branches
        → $$PRUNE: remove this sub-document
        → $$DESCEND: keep this level, check children
        → $$KEEP: keep this entire sub-document
        → No Redshift equivalent (unique to MongoDB)

    Authentication Database (authSource):
        → The database where a user's credentials are stored
        → Usually "admin" for service accounts
        → Specified in connection string: ?authSource=admin
        → Redshift has no equivalent (single user namespace)

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "AUTH ENABLED → PIPELINE BREAKS"
    SITUATION: Enable authentication on MongoDB cluster
    RESULT: Change stream listener can't connect → CDC stops → Redshift stale
    WHY: etl_service user missing changeStream or oplog.rs privileges
    ✅ FIX: Run verify_pipeline_security() BEFORE enabling auth (Part 6)

PITFALL 2: "VIEW DOESN'T HIDE PII FROM AGGREGATION"
    SITUATION: Analyst queries v_orders_no_pii with $lookup to raw users collection
    RESULT: PII visible through $lookup JOIN
    WHY: $lookup in user's aggregation can reference ANY collection
    ✅ FIX: 
    → Analyst should NOT have find access on raw collections
    → Only grant access to views
    → Views cannot be $lookup targets by unauthorized users
    → MongoDB checks permissions on $lookup target collection

PITFALL 3: "REDSHIFT DEVELOPER EXPECTS COLUMN-LEVEL GRANT"
    SITUATION: Developer tries: db.grantPrivileges("user", { field: "email" })
    RESULT: Error — no such command
    WHY: MongoDB has NO native field-level GRANT
    → Field security is achieved through VIEWS + $project/$redact
    ✅ FIX: Create view that excludes sensitive fields (Part 4)
    → Grant user access to view, not raw collection

PITFALL 4: "ADMIN ACCOUNT SHARED ACROSS TEAM"
    SITUATION: All 5 team members use same admin account
    RESULT: Can't audit who did what, can't revoke one person
    WHY: Shared credentials = no accountability
    ✅ FIX: Individual accounts per person + per service
    → Use SSO/OIDC for human accounts (auto-provisioning)

PITFALL 5: "FORGOT authSource IN CONNECTION STRING"
    SITUATION: User created in admin db, connects without ?authSource=admin
    RESULT: Authentication failure (user not found)
    WHY: MongoDB looks for user in the DATABASE specified in URI
    → If URI says /shopstream, looks in shopstream.system.users
    → User exists in admin.system.users → not found → auth fails
    ✅ FIX: Always specify authSource:
    → mongodb+srv://user:pass@host/shopstream?authSource=admin

PITFALL 6: "ROLES NOT UPDATED AFTER COLLECTION CHANGES"
    SITUATION: New collection "promotions" added to shopstream
    RESULT: app_service can't access it (not in custom role)
    WHY: Custom roles list SPECIFIC collections
    → New collections not automatically included
    → Unlike Redshift: GRANT ALL ON ALL TABLES includes future tables
    ✅ FIX:
    → Option A: Use { collection: "" } for all collections in database
    → Option B: Update role when new collections are created
    → Option C: Use DEFAULT privileges pattern (grant on database level)
    → Trade-off: broader access vs manual role maintenance
