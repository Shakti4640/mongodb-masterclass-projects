🔴 REDSHIFT MINDSET — How You Would Solve Multi-Tenancy

The Redshift-Only Multi-Tenant Architecture

YOUR INSTINCT (trained by Redshift):

    "Use schemas. One schema per tenant. Or one cluster per enterprise tenant.
     Row-level security for shared tables. Data sharing for cross-tenant analytics."

    THREE REDSHIFT APPROACHES:

    ─── APPROACH A: Shared Tables with tenant_id Column ────────
    
    CREATE TABLE shared.projects (
        tenant_id       VARCHAR(24) NOT NULL,     -- every row tagged
        project_id      BIGINT IDENTITY(1,1),
        name            VARCHAR(255),
        status          VARCHAR(20),
        created_at      TIMESTAMP
    )
    DISTSTYLE KEY DISTKEY(tenant_id)              -- co-locate tenant data
    SORTKEY(tenant_id, created_at);
    
    -- Every query MUST include WHERE tenant_id = 'xxx'
    -- Forgetting this = DATA BREACH
    SELECT * FROM shared.projects WHERE tenant_id = 'tenant_042';
    
    PROBLEMS:
    → One missing WHERE clause = cross-tenant data leak
    → No database-level isolation guarantee
    → All tenants compete for same WLM slots
    → Large tenant scans slow down small tenant queries
    → Custom fields per tenant: impossible without ALTER TABLE
    → Tenant deletion: DELETE WHERE tenant_id = 'xxx' across ALL tables
      → Leaves ghost data in sort blocks → VACUUM required
    
    ─── APPROACH B: Schema-per-Tenant ──────────────────────────
    
    CREATE SCHEMA tenant_042;
    CREATE TABLE tenant_042.projects (...);
    CREATE TABLE tenant_042.tasks (...);
    CREATE TABLE tenant_042.contacts (...);
    -- Repeat for all 500 tenants
    
    PROBLEMS:
    → 500 schemas × 10 tables = 5,000 tables
    → Redshift metadata overhead: significant at 5,000+ tables
    → DDL operations (schema changes) must run 500 times
    → Cross-tenant analytics: UNION ALL across 500 schemas → impractical
    → New tenant onboarding: CREATE SCHEMA + 10 CREATE TABLE statements
    → Tenant deletion: DROP SCHEMA CASCADE (cleaner but still DDL)
    → No per-schema resource isolation (shared compute)
    
    ─── APPROACH C: Cluster-per-Enterprise-Tenant ──────────────
    
    For 20 enterprise tenants: 20 separate Redshift clusters
    
    PROBLEMS:
    → 20 clusters × $5,000/month = $100,000/month just for enterprise
    → Cross-tenant analytics: Redshift data sharing (read-only, complex)
    → Operational overhead: manage 20+ clusters
    → Each cluster needs monitoring, patching, resizing independently
    → Not scalable to 500 tenants

NONE OF THESE APPROACHES WORK WELL:
    → Approach A: insecure, no isolation
    → Approach B: operational nightmare at scale
    → Approach C: prohibitively expensive
    
    Redshift was NOT designed for multi-tenant operational workloads
    It CAN serve as the analytical layer (like Projects 45-46)
    But the operational layer needs MongoDB's flexibility

Where Redshift Multi-Tenancy Specifically Fails

FAILURE POINTS:

1. CUSTOM FIELDS PER TENANT
   → Tenant A wants "department" field on projects
   → Tenant B wants "priority_score" (decimal) on projects
   → Tenant C wants "tags" (array) on projects
   → Redshift: ALTER TABLE ADD COLUMN affects ALL tenants
     OR: separate tables per tenant (schema explosion)
     OR: EAV pattern (terrible performance)
   → MongoDB: each tenant's documents have whatever fields they need
     → No ALTER TABLE, no schema explosion, no EAV

2. TENANT ONBOARDING IN 30 SECONDS
   → Redshift: CREATE SCHEMA + 10 CREATE TABLE + GRANT statements
     → DDL compilation: 5-15 seconds per statement
     → Total: 60-120 seconds minimum
   → MongoDB: insert one tenant config document → done
     → First data insert creates collections implicitly
     → Total: <1 second

3. TENANT DELETION (GDPR "RIGHT TO BE FORGOTTEN")
   → Redshift Approach A: DELETE across all tables → VACUUM needed → hours
   → Redshift Approach B: DROP SCHEMA CASCADE → seconds, but DDL lock
   → MongoDB shared collection: deleteMany({tenant_id: "xxx"}) → minutes
   → MongoDB database-per-tenant: db.dropDatabase() → instant, zero residue

4. NOISY NEIGHBOR
   → One enterprise tenant runs heavy analytics
   → Redshift: consumes WLM slots → other tenants queue
   → No per-tenant resource isolation within a cluster
   → MongoDB: read preference routing + per-tenant connection pools
     → Can throttle per tenant at application layer

5. PER-TENANT BILLING
   → Redshift: no native per-tenant storage/query tracking
     → Must build custom metering on top
   → MongoDB: db.stats() per database gives per-tenant storage
     → Atlas: built-in per-project metrics

# ═══════════════════════════════════════════════════════
# TENANT ROUTER — Routes operations to correct database
# This is the CORE multi-tenancy component
# ═══════════════════════════════════════════════════════

from pymongo import MongoClient
from functools import lru_cache
import threading

class TenantRouter:
    """
    Routes database operations to the correct MongoDB database
    based on tenant configuration
    
    SHARED tenants → clouddesk database (with tenant_id filter)
    DEDICATED tenants → tenant-specific database (no tenant_id needed)
    
    THIS IS THE PATTERN REDSHIFT CANNOT DO:
    → Redshift has no concept of routing queries to different schemas
      based on runtime context
    → Redshift row-level security exists but is fragile
    → MongoDB: the application layer controls routing completely
    """
    
    def __init__(self, shared_uri, enterprise_uris=None):
        self.shared_client = MongoClient(shared_uri)
        self.enterprise_clients = {}
        
        if enterprise_uris:
            for tenant_id, uri in enterprise_uris.items():
                self.enterprise_clients[tenant_id] = MongoClient(uri)
        
        # Cache tenant configs (refresh every 60 seconds)
        self._config_cache = {}
        self._cache_lock = threading.Lock()
        self._load_all_configs()
    
    def _load_all_configs(self):
        """Load all tenant configs into memory cache"""
        configs = self.shared_client["platform"]["tenant_configs"].find(
            {"status": {"$in": ["active", "trial"]}}
        )
        with self._cache_lock:
            for config in configs:
                self._config_cache[config["_id"]] = config
        print(f"Loaded {len(self._config_cache)} tenant configs")
    
    def get_tenant_config(self, tenant_id):
        """Get tenant configuration (cached)"""
        with self._cache_lock:
            return self._config_cache.get(tenant_id)
    
    def get_db(self, tenant_id):
        """
        Get the correct database for a tenant
        
        THIS IS THE KEY FUNCTION:
        → Shared tenants: returns shared database
        → Dedicated tenants: returns dedicated database
        → Caller doesn't need to know the difference
        """
        config = self.get_tenant_config(tenant_id)
        if not config:
            raise TenantNotFoundError(f"Tenant {tenant_id} not found")
        
        if config["status"] not in ("active", "trial"):
            raise TenantSuspendedError(f"Tenant {tenant_id} is {config['status']}")
        
        if config["isolation"]["model"] == "dedicated":
            # Enterprise: dedicated database (or cluster)
            if config["isolation"].get("connection_string"):
                # Separate Atlas cluster
                client = self.enterprise_clients.get(tenant_id)
                if not client:
                    client = MongoClient(config["isolation"]["connection_string"])
                    self.enterprise_clients[tenant_id] = client
                return client[config["isolation"]["database"]]
            else:
                # Dedicated database on shared cluster
                return self.shared_client[config["isolation"]["database"]]
        else:
            # Starter/Business: shared database
            return self.shared_client[config["isolation"]["database"]]
    
    def get_collection(self, tenant_id, collection_name):
        """
        Get a collection with appropriate tenant context
        
        For shared tenants: returns TenantScopedCollection
          (automatically adds tenant_id to all queries)
        For dedicated tenants: returns raw collection
          (no tenant_id needed — entire DB is theirs)
        """
        config = self.get_tenant_config(tenant_id)
        db = self.get_db(tenant_id)
        
        if config["isolation"]["model"] == "shared":
            return TenantScopedCollection(db[collection_name], tenant_id)
        else:
            return db[collection_name]


class TenantScopedCollection:
    """
    Wraps a MongoDB collection to AUTOMATICALLY inject tenant_id
    into all queries and documents
    
    THIS PREVENTS THE #1 MULTI-TENANCY BUG:
    → Forgetting WHERE tenant_id = 'xxx' in a query
    → With this wrapper: impossible to query without tenant_id
    
    COMPARISON TO REDSHIFT:
    → Redshift: row-level security policies (CREATE RLS POLICY)
      → Applied at database level, but fragile
      → Admin users can bypass
      → Complex to manage per-tenant
    → MongoDB TenantScopedCollection: application-level enforcement
      → Cannot be bypassed by query
      → Simple to understand and audit
      → But requires discipline (must use this wrapper, not raw collection)
    """
    
    def __init__(self, collection, tenant_id):
        self._collection = collection
        self._tenant_id = tenant_id
    
    def _inject_tenant(self, filter_doc):
        """Add tenant_id to any query filter"""
        if filter_doc is None:
            filter_doc = {}
        filter_doc["tenant_id"] = self._tenant_id
        return filter_doc
    
    def _inject_tenant_doc(self, document):
        """Add tenant_id to any document being inserted"""
        if isinstance(document, list):
            for doc in document:
                doc["tenant_id"] = self._tenant_id
            return document
        document["tenant_id"] = self._tenant_id
        return document
    
    def find(self, filter=None, *args, **kwargs):
        return self._collection.find(self._inject_tenant(filter), *args, **kwargs)
    
    def find_one(self, filter=None, *args, **kwargs):
        return self._collection.find_one(self._inject_tenant(filter), *args, **kwargs)
    
    def insert_one(self, document, *args, **kwargs):
        return self._collection.insert_one(
            self._inject_tenant_doc(document), *args, **kwargs
        )
    
    def insert_many(self, documents, *args, **kwargs):
        return self._collection.insert_many(
            self._inject_tenant_doc(documents), *args, **kwargs
        )
    
    def update_one(self, filter, update, *args, **kwargs):
        return self._collection.update_one(
            self._inject_tenant(filter), update, *args, **kwargs
        )
    
    def update_many(self, filter, update, *args, **kwargs):
        return self._collection.update_many(
            self._inject_tenant(filter), update, *args, **kwargs
        )
    
    def delete_one(self, filter, *args, **kwargs):
        return self._collection.delete_one(
            self._inject_tenant(filter), *args, **kwargs
        )
    
    def delete_many(self, filter=None, *args, **kwargs):
        return self._collection.delete_many(
            self._inject_tenant(filter), *args, **kwargs
        )
    
    def count_documents(self, filter=None, *args, **kwargs):
        return self._collection.count_documents(
            self._inject_tenant(filter), *args, **kwargs
        )
    
    def aggregate(self, pipeline, *args, **kwargs):
        # Prepend $match with tenant_id to pipeline
        tenant_match = {"$match": {"tenant_id": self._tenant_id}}
        pipeline = [tenant_match] + list(pipeline)
        return self._collection.aggregate(pipeline, *args, **kwargs)

// WHY tenant_id IS ALWAYS FIRST IN COMPOUND INDEX:
// → MongoDB uses index PREFIX for query matching
// → { tenant_id: 1, status: 1 } supports:
//   → find({ tenant_id: "x" })                    ✅ uses index
//   → find({ tenant_id: "x", status: "active" })  ✅ uses index
//   → find({ status: "active" })                   ❌ CANNOT use index
// → This GUARANTEES every query goes through tenant_id filter
// → A query without tenant_id = full collection scan = automatically slow
//   → This is a FEATURE: it prevents accidental cross-tenant queries
//
// COMPARISON TO REDSHIFT:
// → DISTKEY(tenant_id): co-locates tenant data on same slice
// → SORTKEY(tenant_id, ...): similar prefix concept for block skipping
// → But Redshift doesn't PREVENT queries without tenant_id
//   → A SELECT * FROM tasks WHERE status = 'done' scans ALL tenants
//   → In MongoDB with tenant_id-prefixed indexes: same query does COLLSCAN
//     → Performance penalty = automatic safety net

def offboard_tenant(tenant_id, reason="customer_request"):

    Complete tenant data deletion

      → Leaves dead rows → VACUUM required → hours
      → Sort blocks partially empty → scan performance degraded
    → Redshift schema-per-tenant: DROP SCHEMA CASCADE → clean but DDL lock
    → MongoDB shared: deleteMany({tenant_id: xxx}) → minutes, clean
    → MongoDB dedicated: db.dropDatabase() → instant, zero residue

    print(f"   Reason: {reason}")

        # DEDICATED: drop entire database — instant and clean
        db_name = config["isolation"]["database"]
        client.drop_database(db_name)
        print(f"   ✅ Dropped database: {db_name}")

        # SHARED: delete from all shared collections
        collections = ["projects", "tasks", "contacts", "deals",
                       "documents", "activity_logs", "users"]

        for coll_name in collections:

            if result.deleted_count > 0:
                print(f"   ✅ Deleted {result.deleted_count:,} from {coll_name}")
        
        print(f"   Total documents deleted: {total_deleted:,}")

                "status": "deleted",

                "deletion_reason": reason,
                "updated_at": datetime.now()
            }

# ─── USAGE TRACKING & NOISY NEIGHBOR PREVENTION ──────
def check_tenant_limits(tenant_id, operation, collection_name=None):
    """
    Check if tenant is within their plan limits BEFORE executing operation
    
    THIS IS HOW YOU PREVENT NOISY NEIGHBORS:
    → Each operation checks limits before executing
    → Over-limit operations are rejected or throttled
    → No single tenant can overwhelm the shared infrastructure
    
    COMPARISON TO REDSHIFT:
    → Redshift WLM: queue-level throttling (not per-tenant)
    → Redshift has NO concept of per-tenant resource limits
    → Must build external rate limiting for Redshift
    → MongoDB: enforce at application layer with this function
    """
    
    config = platform_db.tenant_configs.find_one(
        {"_id": tenant_id},
        {"limits": 1, "usage": 1, "plan": 1}
    )
    
    if not config:
        return False, "Tenant not found"
    
    limits = config["limits"]
    usage = config["usage"]
    
    # Check API rate limit
    if operation == "api_call":
        if usage.get("api_calls_today", 0) >= limits["max_api_calls_per_hour"] * 24:
            return False, f"Daily API limit exceeded ({limits['max_api_calls_per_hour'] * 24})"
        
        # Increment counter
        platform_db.tenant_configs.update_one(
            {"_id": tenant_id},
            {
                "$inc": {"usage.api_calls_today": 1, "usage.api_calls_this_month": 1},
                "$set": {"usage.last_active_at": datetime.now()}
            }
        )
        return True, "OK"
    
    # Check storage limit
    if operation == "storage_check":
        max_bytes = limits["max_storage_gb"] * 1024 * 1024 * 1024
        if usage.get("storage_bytes", 0) >= max_bytes:
            return False, f"Storage limit exceeded ({limits['max_storage_gb']}GB)"
        return True, "OK"
    
    # Check project limit
    if operation == "create_project":
        if usage.get("current_projects", 0) >= limits["max_projects"]:
            return False, f"Project limit exceeded ({limits['max_projects']})"
        return True, "OK"
    
    # Check user limit
    if operation == "add_user":
        if usage.get("current_users", 0) >= limits["max_users"]:
            return False, f"User limit exceeded ({limits['max_users']})"
        return True, "OK"
    
    return True, "OK"


# ─── PER-TENANT STORAGE METERING ─────────────────────
def calculate_tenant_storage(tenant_id):
    """
    Calculate storage usage for a tenant
    
    DEDICATED tenants: db.stats() gives exact database size
    SHARED tenants: aggregate document sizes per tenant
    
    THIS feeds into billing (per-tenant cost allocation)
    """
    
    config = platform_db.tenant_configs.find_one({"_id": tenant_id})
    
    if config["isolation"]["model"] == "dedicated":
        # Easy: database-level stats
        db = client[config["isolation"]["database"]]
        stats = db.command("dbStats")
        storage_bytes = stats.get("storageSize", 0)
        doc_count = stats.get("objects", 0)
        
    else:
        # Must aggregate per-tenant in shared collections
        storage_bytes = 0
        doc_count = 0
        
        for coll_name in ["projects", "tasks", "contacts", "deals",
                          "documents", "activity_logs", "users"]:
            pipeline = [
                {"$match": {"tenant_id": tenant_id}},
                {
                    "$group": {
                        "_id": None,
                        "count": {"$sum": 1},
                        "avg_size": {"$avg": {"$bsonSize": "$$ROOT"}}
                    }
                }
            ]
            result = list(shared_db[coll_name].aggregate(pipeline))
            if result:
                coll_count = result[0]["count"]
                coll_avg_size = result[0].get("avg_size", 500)
                doc_count += coll_count
                storage_bytes += int(coll_count * coll_avg_size)
    
    # Update usage in config
    platform_db.tenant_configs.update_one(
        {"_id": tenant_id},
        {"$set": {
            "usage.storage_bytes": storage_bytes,
            "updated_at": datetime.now()
        }}
    )
    
    return {
        "tenant_id": tenant_id,
        "storage_bytes": storage_bytes,
        "storage_mb": round(storage_bytes / 1024 / 1024, 2),
        "document_count": doc_count
    }

📦 COMPONENT 4: SHARDING STRATEGY — Multi-Tenant Specific

Shard Key Design (References: Projects 21-22)

// ═══════════════════════════════════════════════════════
// SHARDING STRATEGY — Multi-Tenant
// DIFFERENT from e-commerce (Project 45) and IoT (Project 46)
// ═══════════════════════════════════════════════════════

// ── SHARED COLLECTIONS SHARD KEY ─────────────────────
// Primary goal: co-locate all data for one tenant on same shard
// Secondary goal: distribute tenants evenly across shards

// Projects: tenant_id + _id
sh.shardCollection("clouddesk.projects", { "tenant_id": 1, "_id": 1 })
// WHY:
// → tenant_id first: all projects for one tenant on ONE shard
// → _id second: distributes within tenant (if tenant has millions of projects)
// → Query: find({ tenant_id: "x" }) → TARGETED to one shard
// → Query: find({ tenant_id: "x", _id: ObjectId("y") }) → TARGETED
// → Cross-tenant query (admin): find({}) → scatter-gather (acceptable for admin)

// Tasks: tenant_id + project_id
sh.shardCollection("clouddesk.tasks", { "tenant_id": 1, "project_id": 1 })
// WHY:
// → Same tenant on same shard as their projects
// → Tasks for one project co-located (frequent query pattern)

// Contacts: tenant_id + _id
sh.shardCollection("clouddesk.contacts", { "tenant_id": 1, "_id": 1 })

// Activity logs: tenant_id + timestamp
sh.shardCollection("clouddesk.activity_logs", { "tenant_id": 1, "timestamp": -1 })
// WHY:
// → Tenant-scoped time-range queries: "last 7 days of activity for tenant X"
// → Descending timestamp: recent data first within tenant's chunk

// Users: tenant_id + _id
sh.shardCollection("clouddesk.users", { "tenant_id": 1, "_id": 1 })

// ── WHY NOT HASHED tenant_id ─────────────────────────
// Hashed: distributes evenly but DESTROYS data locality
// → find({ tenant_id: "x", project_id: "y" }) → scatter-gather
// → We WANT all tenant data on same shard for:
//   → Fast per-tenant queries (no scatter-gather)
//   → Easy tenant deletion (delete from one shard)
//   → Per-tenant backup (snapshot one shard)

// ── HANDLING LARGE TENANTS (jumbo chunks) ────────────
// Problem: Enterprise tenant with 50GB of data on ONE shard
// → That shard becomes a hot spot
// Solution: compound shard key ensures sub-splitting
// → { tenant_id: "ent_001", _id: ObjectId("...") }
// → MongoDB can split chunks WITHIN one tenant_id by _id range
// → Large tenant's data spreads across MULTIPLE chunks on SAME shard
//   (or migrated to balance if needed)

// ┌──────────────┬─────────────────────┬────────────────────────────┐
// │ ARCHITECTURE  │ SHARD KEY            │ WHY                         │

def check_permission(tenant_id, user_id, resource, action):
    """
    Check if user has permission to perform action on resource
    within their tenant
    
    THREE LAYERS OF SECURITY:
    1. Tenant isolation: TenantScopedCollection ensures tenant_id in every query
    2. Role permissions: this function checks user role against action
    3. Resource ownership: optional check if user owns the specific resource
    
    COMPARISON TO REDSHIFT:
    → Redshift: GRANT SELECT ON TABLE TO user_group
      → Table-level or schema-level only
      → No row-level or field-level granularity natively
      → RLS policies exist but complex to manage per-tenant
    → MongoDB: application-level enforcement
      → Per-tenant, per-role, per-resource, per-action
      → Field-level redaction via $redact in aggregation pipeline
      → More flexible but requires application discipline
    """
    
    # Get user's role within this tenant
    router = TenantRouter("mongodb://localhost:27017/")
    users_coll = router.get_collection(tenant_id, "users")
    
    user = users_coll.find_one(
        {"_id": user_id},
        {"role": 1, "status": 1}
    )
    
    if not user:
        return False, "User not found in tenant"
    
    if user.get("status") != "active":
        return False, f"User is {user.get('status', 'unknown')}"
    
    role = user.get("role", "viewer")
    
    # Check role permissions
    role_perms = ROLE_PERMISSIONS.get(role, {})
    resource_perms = role_perms.get(resource, [])
    
    if action in resource_perms:
        return True, f"Allowed: {role} can {action} {resource}"
    else:
        return False, f"Denied: {role} cannot {action} {resource}"


def get_redacted_fields(tenant_id, user_role, collection_name):
    """
    Return list of fields to EXCLUDE based on user role
    
    Example: 'viewer' role cannot see deal.value on contacts
    Example: 'member' role cannot see billing information
    
    This is MongoDB's equivalent of Redshift column-level GRANT:
    → Redshift: GRANT SELECT(col1, col2) ON table TO role
    → MongoDB: projection in find() based on role
    """
    
    redaction_rules = {
        "viewer": {
            "contacts": ["deal.value", "deal.probability", "custom"],
            "projects": ["custom.budget"],
            "users": ["email", "phone"]
        },
        "member": {
            "contacts": ["deal.probability"],
            "users": ["phone"]
        },
        "manager": {},
        "admin": {}
    }
    
    role_rules = redaction_rules.get(user_role, {})
    return role_rules.get(collection_name, [])


def build_projection_for_role(user_role, collection_name, requested_fields=None):
    """
    Build MongoDB projection that excludes fields based on role
    
    Usage:
    projection = build_projection_for_role("viewer", "contacts")
    db.contacts.find({...}, projection)
    → Automatically excludes sensitive fields for viewer role
    """
    
    redacted = get_redacted_fields("", user_role, collection_name)
    
    if not redacted:
        return requested_fields or {}
    
    projection = {}
    for field in redacted:
        projection[field] = 0
    
    return projection

WITH activity_metrics AS (

    COALESCE(am.active_days_30d, 0)                       AS active_days,

    COALESCE(pp.total_actions_prev_30d, 0)                AS actions_prev_30d,
    -- Engagement trend

        WHEN pp.total_actions_prev_30d > 0
        THEN ROUND((am.total_actions_30d - pp.total_actions_prev_30d)::FLOAT
                    / pp.total_actions_prev_30d * 100, 1)
        ELSE NULL
    END                                                    AS engagement_change_pct,
    -- Churn risk score (0-100, higher = more risk)
    CASE
        WHEN am.active_users_30d IS NULL THEN 95
        WHEN am.active_days_30d < 5 THEN 80
        WHEN am.total_actions_30d < pp.total_actions_prev_30d * 0.5 THEN 70
        WHEN COALESCE(am.active_users_30d, 0)::FLOAT
             / NULLIF(du.total_users, 0) < 0.2 THEN 60
        WHEN am.active_days_30d < 15 THEN 40
        ELSE 10
    END                                                    AS churn_risk_score,
    am.last_activity
FROM platform.dim_tenants t

    GROUP BY 1

WHERE t.status = 'active'

✅ WHAT SHOULD DO (Multi-Tenant Best Practices)

SHOULD DO:

✅ USE HYBRID ISOLATION MODEL
    → Shared collections for small/medium tenants (cost efficient)
    → Dedicated databases for enterprise tenants (maximum isolation)
    → Routing layer abstracts the difference from application code
    → Business logic identical regardless of isolation model

✅ ALWAYS PREFIX INDEXES WITH tenant_id (shared collections)
    → Every compound index starts with tenant_id
    → Queries without tenant_id = collection scan = automatic safety
    → This is the MOST IMPORTANT multi-tenancy rule in MongoDB

✅ USE TenantScopedCollection WRAPPER (or equivalent)
    → Never let application code touch raw collections in shared model
    → Wrapper automatically injects tenant_id into every operation
    → One missing WHERE clause = data breach
    → The wrapper makes breaches structurally impossible

✅ IMPLEMENT RATE LIMITING PER TENANT
    → Check limits BEFORE executing operations
    → API calls per hour, storage, document count, user count
    → Return clear error messages when limits exceeded
    → Prevents one tenant from degrading shared infrastructure

✅ STORE CUSTOM FIELDS IN "custom" SUB-DOCUMENT
    → { ..., custom: { department: "Marketing", budget: 50000 } }
    → Each tenant defines their custom fields in tenant_config
    → Application validates custom fields against config on write
    → No schema changes needed when tenants add fields
    → Redshift users: this replaces ALTER TABLE ADD COLUMN entirely

✅ USE db.dropDatabase() FOR ENTERPRISE TENANT DELETION
    → Instant, complete, zero residue
    → No VACUUM needed (unlike Redshift DELETE)
    → Satisfies GDPR "right to be forgotten" immediately
    → For shared tenants: deleteMany is slower but still clean

✅ SEPARATE PLATFORM ANALYTICS FROM TENANT ANALYTICS
    → Platform analytics (cross-tenant): Redshift only
      → Billing, churn scoring, engagement metrics
    → Tenant analytics (per-tenant): MongoDB for real-time, Redshift for historical
    → NEVER let tenants see cross-tenant data (even in analytics)

❌ WHAT SHOULD NOT DO (Multi-Tenant Anti-Patterns)

DO NOT:

❌ USE COLLECTION-PER-TENANT IN SHARED DATABASE
    → 500 tenants × 6 collections = 3,000 collections
    → Each collection has its own indexes = 15,000+ indexes
    → MongoDB metadata overhead: significant at this scale
    → Connection pool exhaustion risk
    → Use shared collections with tenant_id field instead

❌ USE HASHED SHARD KEY ON tenant_id
    → Destroys data locality for per-tenant queries
    → find({ tenant_id: "x" }) becomes scatter-gather across ALL shards
    → Use range-based { tenant_id: 1, _id: 1 } instead
    → All data for one tenant on one shard = fast targeted queries

❌ TRUST APPLICATION CODE TO ALWAYS FILTER BY tenant_id
    → "We'll just remember to add WHERE tenant_id = 'x' everywhere"
    → One missed filter = cross-tenant data leak = lawsuit
    → Use TenantScopedCollection wrapper that FORCES tenant_id
    → Defense in depth: wrapper + index prefix + code review

❌ GIVE TENANTS DIRECT DATABASE ACCESS
    → "Enterprise tenant wants direct MongoDB connection"
    → Even with dedicated database: risks accidental data corruption
    → Always go through API layer with RBAC enforcement
    → If they need raw access: read-only replica with restricted user

❌ STORE TENANT SECRETS IN PLAIN TEXT
    → API keys, webhook secrets, SSO certificates
    → Use MongoDB CSFLE (Project 38) for sensitive tenant config fields
    → Redshift equivalent: never store secrets in Redshift tables

❌ RUN CROSS-TENANT ANALYTICS ON MONGODB
    → "Iterate all 500 tenant databases and aggregate"
    → Each iteration = separate query = network round trip
    → 500 queries × 100ms each = 50 seconds minimum
    → Use Redshift: all tenant data in one fact table
    → GROUP BY tenant_id → instant cross-tenant analytics

❌ USE SAME WRITE CONCERN FOR ALL TENANT TIERS
    → Starter tenants: w:1 acceptable (cost optimization)
    → Enterprise tenants: w:majority required (data durability SLA)
    → Configure write concern per tenant tier in the router

def demo_per_tenant_analytics(tenant_id):
    """Per-tenant dashboard metrics from MongoDB"""
    
    print(f"\n" + "=" * 60)
    print(f"  PER-TENANT ANALYTICS: {tenant_id}")
    print("=" * 60)
    
    config = platform_db.tenant_configs.find_one({"_id": tenant_id})
    is_dedicated = config["isolation"]["model"] == "dedicated"
    
    if is_dedicated:
        db = client[config["isolation"]["database"]]
        match_stage = {"$match": {}}
    else:
        db = shared_db
        match_stage = {"$match": {"tenant_id": tenant_id}}
    
    start = time.time()
    
    # Project status breakdown
    project_stats = list(db.projects.aggregate([
        match_stage,
        {"$group": {"_id": "$status", "count": {"$sum": 1}}}
    ]))
    
    # Task completion rate
    task_stats = list(db.tasks.aggregate([
        match_stage,
        {"$group": {
            "_id": None,
            "total": {"$sum": 1},
            "completed": {"$sum": {"$cond": [{"$eq": ["$status", "done"]}, 1, 0]}},
            "overdue": {"$sum": {"$cond": [
                {"$and": [
                    {"$ne": ["$status", "done"]},
                    {"$lt": ["$due_date", datetime.now()]}
                ]}, 1, 0
            ]}}
        }}
    ]))
    
    # Deal pipeline
    deal_stats = list(db.contacts.aggregate([
        match_stage,
        {"$match": {"deal.stage": {"$exists": True}}},
        {"$group": {
            "_id": "$deal.stage",
            "count": {"$sum": 1},
            "total_value": {"$sum": {"$toDouble": "$deal.value"}}
        }},
        {"$sort": {"total_value": -1}}
    ]))
    
    elapsed_ms = (time.time() - start) * 1000
    
    print(f"\n   📊 Project Status:")
    for ps in project_stats:
        print(f"     {ps['_id']}: {ps['count']}")
    
    if task_stats:
        ts = task_stats[0]
        completion_rate = round(ts["completed"] / max(ts["total"], 1) * 100, 1)
        print(f"\n   ✅ Task Metrics:")
        print(f"     Total: {ts['total']} | Completed: {ts['completed']} | "
              f"Overdue: {ts['overdue']}")
        print(f"     Completion rate: {completion_rate}%")
    
    print(f"\n   💰 Deal Pipeline:")
    total_pipeline = 0
    for ds in deal_stats:
        print(f"     {ds['_id']}: {ds['count']} deals, ${ds['total_value']:,.0f}")
        total_pipeline += ds["total_value"]
    print(f"     Total pipeline: ${total_pipeline:,.0f}")
    
    print(f"\n   ⏱ Query time: {elapsed_ms:.2f}ms")
    print(f"   (Real-time from MongoDB — no Redshift needed for this)")
