🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform (from Project 25) has grown.
    You now need FIVE automated reactions to data events:

    REACTION 1 — Welcome Flow
        → When new customer signs up → create welcome notification
        → Auto-generate a referral code
        → Add customer to onboarding email sequence
        → Must happen within 1 second of signup

    REACTION 2 — Audit Trail
        → Every change to an order (status, amount, items)
          must be logged to an immutable audit collection
        → Include: who changed, what changed, when, before/after values
        → Regulatory requirement — cannot miss ANY change

    REACTION 3 — Business Rule Enforcement
        → When order total exceeds $10,000 → auto-flag for review
        → When inventory drops below threshold → create restock alert
        → When customer makes 10th purchase → upgrade to Gold tier

    REACTION 4 — Scheduled Aggregation
        → Every hour: compute and cache "trending products" list
        → Every midnight: generate daily sales summary
        → Every Monday 9 AM: create weekly performance report

    REACTION 5 — Authentication Events
        → When user logs in → update last_login timestamp
        → When user fails login 5 times → lock account
        → When new device detected → send security alert

    IN PROJECT 25: You built custom consumer processes for change streams
        → YOU managed the consumer code
        → YOU managed the resume tokens
        → YOU managed the deployment (containers, servers)
        → YOU managed the error handling and retries
        → YOU managed the scaling

        → Atlas manages: execution, scaling, retries, resume tokens

        → Pay only for function executions

        → No triggers, no serverless functions, no event reactions

🔧 HOW TO APPROACH IT IN MONGODB

THINKING FRAMEWORK FOR ATLAS TRIGGERS:

    QUESTION 1: What TRIGGERS the reaction?
        → Data change?     → DATABASE TRIGGER
        → Time schedule?   → SCHEDULED TRIGGER
        → User auth event? → AUTHENTICATION TRIGGER

    QUESTION 2: What does the function NEED ACCESS to?
        → Same database?   → context.services.get("mongodb-atlas")
        → External API?    → context.http.get() or npm packages
        → Other Atlas data? → cross-cluster access
        → Secrets/keys?    → context.values.get("API_KEY")

    QUESTION 3: How critical is exactly-once processing?
        → Critical?        → Enable "event ordering" on trigger
        → Best-effort OK?  → Disable ordering (faster, parallel)

    QUESTION 4: How much latency is acceptable?
        → < 100ms?         → Trigger may be too slow — use change stream directly
        → < 1 second?      → Trigger is perfect
        → Minutes OK?      → Trigger or scheduled trigger both work

    DECISION TREE:
    ┌─────────────────────────────────────────────────┐
    │ Do you need to react to DATA CHANGES?            │
    │   YES → Do you need custom infrastructure?       │
    │          NO  → Atlas Database Trigger ✓           │
    │          YES → Change Stream (Project 25) ✓      │
    │   NO  → Do you need SCHEDULED execution?         │
    │          YES → Atlas Scheduled Trigger ✓          │
    │          NO  → Do you need AUTH event reaction?   │
    │                 YES → Atlas Authentication Trigger ✓│

// NOTE: "full_document_before_change": true
// → Requires changeStreamPreAndPostImages enabled on collection
// → Gives us BEFORE and AFTER state for every change
// → Critical for audit: "status changed FROM 'processing' TO 'shipped'"
// → This was enabled in Project 25 — same prerequisite applies


// ═══════════════════════════════════════════════════════
// ATLAS FUNCTION: onOrderChange
// ═══════════════════════════════════════════════════════

// File: functions/onOrderChange.js

exports = async function(changeEvent) {
    const db = context.services
        .get("mongodb-atlas")
        .db("ecommerce");
    
    const auditCollection = db.collection("order_audit_log");
    
    // ─── BUILD AUDIT ENTRY ───────────────────────────
    const auditEntry = {
        // What changed
        order_id: changeEvent.documentKey._id,
        operation: changeEvent.operationType,
        timestamp: new Date(),
        cluster_time: changeEvent.clusterTime,
        
        // Before state (pre-image)
        before: changeEvent.fullDocumentBeforeChange || null,
        
        // After state (post-image)
        after: changeEvent.fullDocument || null,
        
        // Delta (for updates)
        changes: null,
        
        // Metadata
        trigger_name: "orderAuditTrigger",
        resume_token: changeEvent._id
    };
    
    // ─── COMPUTE DELTA FOR UPDATES ───────────────────
    if (changeEvent.operationType === "update") {
        auditEntry.changes = {
            updated_fields: changeEvent.updateDescription.updatedFields,
            removed_fields: changeEvent.updateDescription.removedFields,
            truncated_arrays: changeEvent.updateDescription.truncatedArrays
        };
        
        // Human-readable change description
        const updatedFields = Object.keys(
            changeEvent.updateDescription.updatedFields || {}
        );
        auditEntry.change_summary = updatedFields.map(field => {
            const oldVal = changeEvent.fullDocumentBeforeChange
                ? changeEvent.fullDocumentBeforeChange[field]
                : "unknown";
            const newVal = changeEvent.updateDescription.updatedFields[field];
            return `${field}: ${JSON.stringify(oldVal)} → ${JSON.stringify(newVal)}`;
        });
    }
    
    // ─── COMPUTE DELTA FOR DELETES ───────────────────
    if (changeEvent.operationType === "delete") {
        auditEntry.change_summary = ["Document deleted"];
        // before image contains the deleted document (if pre-images enabled)
    }
    
    // ─── COMPUTE DELTA FOR INSERTS ───────────────────
    if (changeEvent.operationType === "insert") {
        auditEntry.change_summary = ["New order created"];
        auditEntry.changes = {
            initial_status: changeEvent.fullDocument.status,
            initial_total: changeEvent.fullDocument.total,
            item_count: (changeEvent.fullDocument.items || []).length
        };
    }
    
    // ─── WRITE AUDIT ENTRY ───────────────────────────
    // Use w:"majority" for audit entries — must be durable
    await auditCollection.insertOne(auditEntry);
    
    console.log(
        `AUDIT: ${changeEvent.operationType} on order ` +
        `${changeEvent.documentKey._id} — ` +
        `${(auditEntry.change_summary || []).join(", ")}`
    );
    
    // ─── ALERTING FOR SUSPICIOUS CHANGES ─────────────
    // Business rule: alert if order total changed by > 50%
    if (changeEvent.operationType === "update" &&
        changeEvent.updateDescription.updatedFields.total) {
        
        const oldTotal = changeEvent.fullDocumentBeforeChange?.total || 0;
        const newTotal = changeEvent.updateDescription.updatedFields.total;
        
        if (oldTotal > 0) {
            const changePercent = Math.abs((newTotal - oldTotal) / oldTotal * 100);
            
            if (changePercent > 50) {
                await db.collection("alerts").insertOne({
                    type: "suspicious_total_change",
                    order_id: changeEvent.documentKey._id,
                    old_total: oldTotal,
                    new_total: newTotal,
                    change_percent: changePercent,
                    severity: "high",
                    status: "open",
                    created_at: new Date()
                });
                
                console.log(
                    `⚠️ ALERT: Order total changed by ${changePercent.toFixed(1)}%`
                );
            }
        }
    }
};

// ═══════════════════════════════════════════════════════

// File: functions/enforceBusinessRules.js

exports = async function(changeEvent) {
    const db = context.services
        .get("mongodb-atlas")
        .db("ecommerce");
    
    const order = changeEvent.fullDocument;
    if (!order) return;  // safety check
    
    const orderId = order._id;
    const customerId = order.customer_id;
    
    // ─── RULE 1: High-value order review ─────────────
    // Orders > $10,000 must be flagged for manual review
    if (order.total > 10000 && order.status !== "flagged_for_review") {

            { _id: orderId },
            {
                $set: {
                    flagged_for_review: true,

            type: "high_value_order",
            order_id: orderId,

            severity: "medium",
            status: "open",
            created_at: new Date()
        });

    }
    
    // ─── RULE 2: Inventory check on new orders ───────
    if (changeEvent.operationType === "insert") {
        for (const item of (order.items || [])) {
            const product = await db.collection("products").findOne(
                { _id: item.product_id }
            );
            
            if (product && product.stock_qty < 10) {
                await db.collection("restock_alerts").updateOne(
                    { product_id: item.product_id, status: "open" },
                    {

                            product_name: product.name,

            }
        }
    }
    
    // ─── RULE 3: Customer tier upgrade ───────────────
    // On every new order: check if customer qualifies for upgrade
    if (changeEvent.operationType === "insert") {
        const orderCount = await db.collection("orders").countDocuments({
            customer_id: customerId,
            status: { $nin: ["cancelled", "refunded"] }
        });

        } else if (orderCount >= 10 && customer.tier === "Bronze") {

        }

        .get("mongodb-atlas")
        .db("ecommerce");

    // ─── AGGREGATE: Top products by order count in last hour ───
    const trending = await db.collection("orders").aggregate([
        {
            $match: {
                created_at: { $gte: oneHourAgo },
                status: { $nin: ["cancelled", "refunded"] }
            }
        },
        { $unwind: "$items" },
        {
            $group: {
                _id: "$items.product_id",
                product_name: { $first: "$items.name" },
                order_count: { $sum: 1 },
                total_qty: { $sum: "$items.qty" },
                total_revenue: {
                    $sum: { $multiply: ["$items.qty", "$items.price"] }
                }
            }
        },
        { $sort: { order_count: -1 } },
        { $limit: 20 }
    ]).toArray();
    
    // ─── CACHE: Store trending products for fast retrieval ───
    // Replace entire cached result (atomic swap)
    await db.collection("cached_trending").deleteMany({});
    
    if (trending.length > 0) {
        const cachedEntries = trending.map((item, rank) => ({
            product_id: item._id,
            product_name: item.product_name,
            rank: rank + 1,
            order_count: item.order_count,
            total_qty: item.total_qty,
            total_revenue: item.total_revenue,
            computed_at: new Date(),
            valid_until: new Date(Date.now() + 3600000)  // 1 hour TTL
        }));
        
        await db.collection("cached_trending").insertMany(cachedEntries);
    }

                created_at: { $gte: yesterday, $lt: today },
                status: { $nin: ["cancelled", "refunded"] }
            }
        },
        { $unwind: "$items" },
        {
            $lookup: {
                from: "products",
                localField: "items.product_id",
                foreignField: "_id",
                as: "product_info"
            }
        },
        { $unwind: "$product_info" },
        {
            $group: {
                _id: "$product_info.category",
                order_count: { $sum: 1 },
                units_sold: { $sum: "$items.qty" },
                revenue: {
                    $sum: { $multiply: ["$items.qty", "$items.price"] }
                }
            }
        },
        { $sort: { revenue: -1 } }

                _id: null,

                total_revenue: { $sum: "$total" },

                max_order: { $max: "$total" },
                unique_customers: { $addToSet: "$customer_id" }
            }
        },
        {
            $project: {
                total_orders: 1,
                total_revenue: 1,
                avg_order_value: { $round: ["$avg_order_value", 2] },
                max_order: 1,
                unique_customers: { $size: "$unique_customers" }

def audit_order_change(order_id: str, operation: str,
                       before: dict = None, after: dict = None,

        "order_id": order_id,
        "operation": operation,
        "timestamp": datetime.now(),
        "before": before,
        "after": after,
        "changes": changes,
        "change_summary": change_summary
    }
    
    db.order_audit_log.insert_one(audit_entry)
    print(f"  📋 AUDIT: {operation} on {order_id} — "
          f"{', '.join(change_summary)}")

def compute_trending_products():
    """
    Manual equivalent of Atlas Scheduled Trigger 'computeTrendingProducts'.
    
    WITH Atlas: runs automatically on schedule, no infrastructure needed.
    WITHOUT: need cron job, EventBridge+Lambda, or APScheduler.
    """
    one_hour_ago = datetime.now() - timedelta(hours=24)  # wider window for testing
    
    pipeline = [
        {"$match": {
            "created_at": {"$gte": one_hour_ago},
            "status": {"$nin": ["cancelled", "refunded"]}
        }},
        {"$unwind": "$items"},
        {"$group": {
            "_id": "$items.product_id",
            "product_name": {"$first": "$items.name"},
            "order_count": {"$sum": 1},
            "total_qty": {"$sum": "$items.qty"},
            "total_revenue": {
                "$sum": {"$multiply": ["$items.qty", "$items.price"]}
            }
        }},
        {"$sort": {"order_count": -1}},
        {"$limit": 20}
    ]
    
    trending = list(db.orders.aggregate(pipeline))
    
    # Cache results
    db.cached_trending.delete_many({})
    if trending:
        cached_entries = [{
            "product_id": item["_id"],
            "product_name": item["product_name"],
            "rank": rank + 1,
            "order_count": item["order_count"],
            "total_revenue": item["total_revenue"],
            "computed_at": datetime.now()
        } for rank, item in enumerate(trending)]
        
        db.cached_trending.insert_many(cached_entries)
    
    print(f"  Trending products computed: {len(trending)} products")
    for item in trending[:5]:
        print(f"    {item['product_name']}: "
              f"{item['order_count']} orders, "
              f"${item['total_revenue']:.2f} revenue")
    
    return trending

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

ATLAS TRIGGER OPTIMIZATIONS:

1. EVENT ORDERING CONTROL
   → "unordered": false → events processed in oplog order (slower, safe)
   → "unordered": true → events processed in parallel (faster, no order guarantee)
   → Audit trail: MUST be ordered (regulatory requirement)
   → Business rules: CAN be unordered (independent per document)
   → Welcome flow: CAN be unordered (one event per customer)

2. MATCH EXPRESSIONS (server-side filtering)
   → Filter BEFORE the trigger function executes
   → Reduces function invocations (cost savings)
   
   // Only fire for high-value orders
   "match": {
       "fullDocument.total": { "$gt": 10000 }
   }
   
   // Only fire for status changes
   "match": {
       "updateDescription.updatedFields.status": { "$exists": true }
   }
   
   → Filtering happens at the change stream level
   → Function NEVER called for non-matching events
   → Same concept as pipeline $match in Project 25

3. PROJECT EXPRESSIONS (reduce payload)
   → Include only fields the function needs
   → Reduces data transfer to function runtime
   
   "project": {
       "fullDocument.customer_id": 1,
       "fullDocument.total": 1,
       "fullDocument.status": 1
   }
   
   → Smaller payload = faster function execution

4. AWS EVENTBRIDGE INTEGRATION
   → Atlas Triggers can forward events to AWS EventBridge
   → Use when: you need to fan out to many AWS services
   → MongoDB change → Atlas Trigger → EventBridge → Lambda/SQS/SNS
   → Best of both worlds: MongoDB event source + AWS event routing

5. ERROR HANDLING CONFIGURATION
   → Atlas retries failed triggers automatically
   → Configurable: retry count, retry delay
   → Dead letter: failed events logged for investigation
   → Unlike manual change streams: no custom retry code needed

✅ WHAT SHOULD DO

SHOULD DO:
    ✅ Use Atlas Triggers for simple, reactive logic
       → Notifications, audit trails, business rules, caching
       → If function is < 50 lines and runs < 30 seconds: trigger is perfect
    
    ✅ Use match expressions to reduce trigger invocations
       → Every invocation has cost (compute + latency)
       → Filter at the trigger level, not in function code
       → Same principle as change stream pipeline filtering (Project 25)
    
    ✅ Keep trigger functions FAST and FOCUSED
       → One trigger = one responsibility
       → Don't combine audit + notification + business rules in one trigger
       → Multiple focused triggers > one monolithic trigger
    
    ✅ Enable pre/post images for audit triggers
       → changeStreamPreAndPostImages on the collection
       → Critical for "before and after" audit entries
       → Without: you only know WHAT changed, not FROM WHAT
    
    ✅ Use "unordered: true" when order doesn't matter
       → Parallel processing = faster
       → Welcome flows, notifications: order usually doesn't matter
       → Audit trails: order ALWAYS matters
    
    ✅ Test triggers with simple inserts/updates
       → Insert a document → check if trigger side-effects appear
       → Check Atlas Logs for execution details and errors
    
    ✅ Use context.values for secrets and configuration
       → API keys, webhook URLs, thresholds

       → Environment-specific values (dev/staging/prod)

REDSHIFT HABITS TO UNLEARN:
    🔄 Stop thinking "triggers are an application concern"
       → In MongoDB: triggers live NEXT TO the data
       → The database platform handles the reactive logic
    🔄 Stop thinking "I need Lambda for every automation"

       → One fewer service to manage
    🔄 Stop thinking "scheduling requires external infrastructure"
       → Atlas Scheduled Triggers = built-in cron

❌ WHAT SHOULD NOT DO

DO NOT:
    ❌ Use Atlas Triggers for heavy computation
       → 90-second default timeout (300s max)
       → No GPU, limited memory, limited npm packages
       → Heavy ML, large ETL: use dedicated compute (Project 25 approach)
    
    ❌ Create circular triggers
       → Trigger A modifies collection X → fires Trigger B
       → Trigger B modifies collection X → fires Trigger A
       → INFINITE LOOP — Atlas will throttle but still wastes resources
       → Solution: use match expressions to exclude trigger-generated changes
       → Or: use a "trigger_source" field to identify and skip

    ❌ Use triggers as your primary business logic layer
       → Triggers are REACTIVE — they respond to data changes
       → They should NOT contain core application logic
       → Core logic: application layer
       → Side effects: triggers (notifications, audit, sync)
    
    ❌ Ignore trigger error logs
       → Failed triggers don't throw errors to the writing application
       → The INSERT/UPDATE succeeds even if the trigger fails
       → Must monitor Atlas Logs for trigger failures
       → Set up Atlas alerts for trigger error rate
    
    ❌ Store large state in trigger function globals
       → Atlas Functions are STATELESS — no persistent memory
       → Each invocation is independent
       → Store state in the database, not in function variables
    
    ❌ Rely on trigger execution timing for business logic
       → Triggers execute AFTER the database operation
       → NOT transactional with the triggering write
       → If trigger fails: the original write STILL succeeded
       → Trigger is "best effort" — not ACID with the write

       → Self-managed: use change streams (Project 25) directly

KEY INTERNAL DETAILS:

    1. TRIGGER ≠ CHANGE STREAM (but built on it)
       → Atlas creates ONE change stream per trigger
       → Atlas manages the cursor, resume token, error handling
       → Your function receives pre-processed change events
       → You never see the raw oplog or manage the cursor

    2. FUNCTION RUNTIME IS ISOLATED
       → Each function invocation runs in its own sandbox
       → No shared state between invocations
       → Pre-authenticated MongoDB client provided via context
       → Limited npm packages available (not full Node.js)

    3. RESUME IS AUTOMATIC
       → If function fails: Atlas retries with same event
       → If Atlas App Services restarts: resumes from last processed event
       → Resume tokens stored in Atlas internal infrastructure
       → YOU never manage tokens (unlike Project 25)

    4. TRIGGER IS EVENTUALLY CONSISTENT WITH THE WRITE
       → Application writes document → write succeeds → returns to app
       → THEN (milliseconds later): trigger fires
       → Trigger failure does NOT roll back the original write
       → This is fundamentally different from SQL triggers which
         execute WITHIN the transaction
       → More like PostgreSQL AFTER trigger + DEFERRABLE

    5. ORDERING GUARANTEES
       → "unordered": false → one event at a time, in oplog order
         → Slower but deterministic
         → If function fails: blocks all subsequent events

         → Faster but no order guarantee

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 26:

PITFALL 1: "Triggers are ACID with the write — like SQL triggers"
    ❌ Assuming trigger failure rolls back the INSERT/UPDATE
    ✅ Understand: trigger fires AFTER write, independently
    WHY: SQL triggers (PostgreSQL) execute WITHIN the transaction
         Atlas triggers execute AFTER — asynchronously
         Original write ALWAYS succeeds regardless of trigger outcome
         If audit trigger fails: data is changed but audit is missing
         Must monitor trigger errors actively

PITFALL 2: "I'll put all my logic in one trigger"
    ❌ One trigger function with 200 lines doing audit + notification +
       business rules + analytics
    ✅ Separate triggers for separate concerns
    WHY: If one concern fails, it blocks others (in ordered mode)
         Harder to debug, monitor, and maintain
         Different concerns may need different ordering (ordered vs unordered)
         Separation of concerns applies to triggers too

PITFALL 3: "Atlas Triggers replace ALL my backend logic"
    ❌ Moving entire API business logic into trigger functions
    ✅ Triggers for SIDE EFFECTS only — core logic in application
    WHY: Trigger functions have limited runtime (300s)
         Limited npm packages available
         No debugging tools (console.log only)
         Cannot serve HTTP requests (not an API)
         Triggers are EVENT HANDLERS, not APPLICATION SERVERS

PITFALL 4: "I don't need to monitor trigger execution"
    ❌ Deploy triggers and forget about them
    ✅ Set up Atlas alerts for trigger error rate and latency
    WHY: Trigger failures are SILENT from the application's perspective
         Your app's insertOne() succeeds even if trigger crashes
         Without monitoring: audit gaps, missed notifications, stale caches
         All invisible until someone notices the downstream effects

PITFALL 5: "Change streams (Project 25) and triggers are the same thing"
    ❌ Using triggers when you need a custom consumer process
       Or building a custom consumer when a trigger would suffice
    ✅ Know the trade-offs:
       → Trigger: zero infrastructure, limited runtime, JS only
       → Change stream: full control, any language, unlimited runtime
    WHY: Triggers are MANAGED change stream consumers
         If your needs fit the constraints: trigger is simpler
         If they don't: change stream (Project 25) gives full flexibility

PITFALL 6: "I'll create a trigger that writes to the SAME collection"
    ❌ Trigger on orders.update → function updates orders → fires trigger again
    ✅ Use match expressions to exclude trigger-generated changes
       Or use a flag field: "trigger_processed": true
    WHY: Circular triggers waste resources and can cause unexpected behavior
         Atlas has loop detection but it's better to design it out
