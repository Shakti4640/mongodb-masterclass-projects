    THREE DISTRIBUTION STYLES:

    1. DISTSTYLE KEY (most common)
       → Choose one column as DISTKEY
       → Rows with same DISTKEY value → same slice
       → Goal: co-locate data that JOINs together
       → Example: DISTKEY(customer_id)
         → All orders for customer 123 → slice 2
         → JOIN orders ON customers → no data movement (co-located)

    2. DISTSTYLE EVEN
       → Round-robin distribution across all slices
       → Every slice gets roughly equal data
       → No co-location benefit → JOINs require redistribution
       → Good for: tables not involved in JOINs

    3. DISTSTYLE ALL
       → FULL COPY of table on EVERY slice
       → Small dimension tables (< 5 million rows)
       → JOINs never require data movement (table is everywhere)
       → Example: country_codes, product_categories

Why Redshift Scaling Feels "Automatic"

WHAT REDSHIFT HANDLES FOR YOU:

    1. Data placement: DISTKEY determines slice → automatic
    2. Data movement on resize: redistributed automatically
    3. Query routing: leader node knows where data lives → automatic
    4. Rebalancing: after resize, data rebalanced → automatic
    5. Sort order: maintained within each slice → automatic
    6. Zone maps: rebuilt automatically on new blocks → automatic

    YOU DECIDE: DISTKEY + SORTKEY + node count
    REDSHIFT HANDLES: everything else

    CHANGING DISTKEY:
    → Requires table recreation (CREATE TABLE AS SELECT)
    → But it's a single SQL command
    → Data redistributed in one operation
    → No ongoing management needed

    THIS IS WHAT YOU'RE USED TO:
    → Simple, declarative, one-time decision
    → System handles distribution transparently
    → Queries work regardless of distribution choice
    → Wrong DISTKEY = slower queries, not broken queries

🟢 MONGODB MINDSET — Sharding Is Distribution + Routing + Balancing

Why MongoDB Needs Sharding (Not Just "More RAM")

THE SCALING WALL — WHY VERTICAL SCALING FAILS:

    VERTICAL SCALING (bigger server):
        → 64 GB RAM → 128 GB → 256 GB → 512 GB
        → Each upgrade: diminishing returns, exponential cost
        → 512 GB RAM server: ~$20,000/month
        → Still ONE server: single point of failure
        → Still ONE thread for aggregation pipeline
        → Still ONE disk for I/O
        → MAX practical: ~2 TB RAM, ~50 TB storage
        → After that: NO bigger server exists

    HORIZONTAL SCALING (more servers — SHARDING):
        → Add 3 servers × 64 GB each = 192 GB total RAM
        → Add 3 more = 384 GB → 576 GB → no limit
        → Each server: independent CPU, RAM, disk, network
        → Parallel I/O across all shards
        → Parallel aggregation across all shards
        → Cost scales linearly, not exponentially
        → No upper limit on total capacity

    THIS IS THE SAME PRINCIPLE AS REDSHIFT MPP:
        → Redshift: add compute nodes → data distributed across slices
        → MongoDB: add shards → data distributed across shards
        → SAME GOAL, DIFFERENT MECHANICS

🧩 HOW TO APPROACH IT IN MONGODB

Core Concepts: Chunks, Balancer, Routing

THE THREE MECHANISMS OF SHARDING:

MECHANISM 1: CHUNK SPLITTING
    → Data is divided into CHUNKS based on shard key RANGES
    → Each chunk: contiguous range of shard key values
    → Default chunk size: 128 MB (configurable, was 64 MB before 6.0)
    → When a chunk exceeds max size → auto-split into two chunks
    
    EXAMPLE (shard key = order_date):
    ┌─────────────────────────────────────────────────────────────┐
    │ Chunk 1: order_date [MinKey → 2023-06-30]                   │
    │ Chunk 2: order_date [2023-07-01 → 2023-12-31]              │
    │ Chunk 3: order_date [2024-01-01 → 2024-03-31]              │
    │ Chunk 4: order_date [2024-04-01 → 2024-05-31]              │
    │ Chunk 5: order_date [2024-06-01 → MaxKey]                   │
    └─────────────────────────────────────────────────────────────┘
    
    REDSHIFT EQUIVALENT:
    → Redshift doesn't have "chunks" — it uses hash-based slot assignment
    → Row → hash(DISTKEY) → slice number → permanent assignment
    → MongoDB: document → shard key range → chunk → shard (movable)

MECHANISM 2: BALANCER
    → Background process that monitors chunk distribution
    → Goal: equal number of chunks across all shards
    → Moves chunks from overloaded shards to underloaded shards
    → Runs continuously (configurable window)
    → Transparent to application — reads/writes work during migration
    
    BEFORE BALANCING:
        Shard 1: 40 chunks  ← overloaded
        Shard 2: 10 chunks  ← underloaded
        Shard 3: 15 chunks  ← underloaded
    
    AFTER BALANCING:
        Shard 1: 22 chunks  ← balanced
        Shard 2: 22 chunks  ← balanced
        Shard 3: 21 chunks  ← balanced
    
    REDSHIFT EQUIVALENT:
    → Redshift rebalances during elastic resize (one-time)
    → MongoDB balancer runs CONTINUOUSLY (ongoing)
    → MongoDB can add a shard and data migrates automatically over time

MECHANISM 3: QUERY ROUTING (mongos)
    → mongos knows the chunk-to-shard mapping (from config servers)
    → For every query, mongos decides:
        a) TARGETED: shard key in query → route to specific shard(s)
        b) SCATTER-GATHER: no shard key → send to ALL shards, merge results
    
    TARGETED QUERY (fast):
        db.orders.find({ customer_id: "cust_123" })
        → Shard key = customer_id
        → mongos looks up: "cust_123" falls in chunk range [cust_100, cust_200]
        → That chunk lives on Shard 2
        → Query sent ONLY to Shard 2
        → Like Redshift: WHERE on DISTKEY → only relevant slice scanned
    
    SCATTER-GATHER QUERY (slow):
        db.orders.find({ product_id: "prod_456" })
        → product_id is NOT the shard key
        → mongos doesn't know which shard has this data
        → Query sent to ALL shards
        → Each shard returns matching documents
        → mongos merges results
        → Like Redshift: query without DISTKEY filter → all slices scanned

Range Sharding vs Hashed Sharding

TWO SHARDING STRATEGIES (vs Redshift's Three Distribution Styles):

STRATEGY 1: RANGE SHARDING
    → Shard key values divided into contiguous ranges
    → Each range = one chunk → assigned to a shard
    → Good for: range queries (date ranges, alphabetical)
    → Risk: monotonically increasing keys → all writes to ONE shard (hot shard)
    
    Shard Key: order_date
    ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │ Shard 1       │ │ Shard 2       │ │ Shard 3       │
    │ Jan-Apr 2024  │ │ May-Aug 2024  │ │ Sep-Dec 2024  │
    │ (historical)  │ │ (recent)      │ │ (current)     │
    │ Low traffic   │ │ Medium traffic│ │ ALL new writes │ ← HOT SHARD!
    └──────────────┘ └──────────────┘ └──────────────┘
    
    REDSHIFT EQUIVALENT: DISTSTYLE KEY
    → Hash of DISTKEY → specific slice
    → But Redshift uses HASH, not RANGE
    → So monotonic keys don't cause hot slices in Redshift

STRATEGY 2: HASHED SHARDING
    → Hash of shard key value → chunk assignment
    → Distributes writes evenly (random distribution)
    → Good for: high write throughput, even distribution
    → Loses: range query efficiency (ranges become scatter-gather)
    
    Shard Key: { customer_id: "hashed" }
    ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │ Shard 1       │ │ Shard 2       │ │ Shard 3       │
    │ hash ranges   │ │ hash ranges   │ │ hash ranges   │
    │ [MinKey, -30] │ │ [-30, +30]    │ │ [+30, MaxKey] │
    │ ~33% of data  │ │ ~33% of data  │ │ ~33% of data  │
    │ ~33% of writes│ │ ~33% of writes│ │ ~33% of writes│
    └──────────────┘ └──────────────┘ └──────────────┘
    
    REDSHIFT EQUIVALENT: DISTSTYLE EVEN (closest match)
    → Even distribution across slices
    → No co-location benefit
    → But Redshift EVEN doesn't use a key at all — truly random

function simpleHash(str) {
    let hash = 0
    for (let i = 0; i < str.length; i++) {
        let char = str.charCodeAt(i)
        hash = ((hash << 5) - hash) + char
        hash = hash & hash  // Convert to 32-bit integer
    }
    return Math.abs(hash) % 3  // 3 shards
}

def analyze_shard_key(collection, key_field, num_shards=3):
    """
    Analyze how data would be distributed for a given shard key.
    
    Evaluates:
    1. Cardinality — how many unique values?
    2. Frequency — are values evenly distributed?
    3. Monotonicity — do values always increase? (hot shard risk)
    
    Redshift equivalent thinking:
        → Cardinality = how many distinct DISTKEY values
        → Frequency = how even is data across slices
        → Monotonicity = N/A for Redshift (hash-based, not range-based)
    """
    
    # 1. CARDINALITY: distinct values
    distinct_count = len(collection.distinct(key_field))
    total_docs = collection.count_documents({})
    
    # 2. FREQUENCY: distribution of values
    freq_pipeline = [
        {"$group": {"_id": f"${key_field}", "count": {"$sum": 1}}},
        {"$group": {
            "_id": None,
            "avg_per_key": {"$avg": "$count"},
            "max_per_key": {"$max": "$count"},
            "min_per_key": {"$min": "$count"},
            "stddev": {"$stdDevPop": "$count"}
        }}
    ]
    freq_stats = list(collection.aggregate(freq_pipeline))
    freq = freq_stats[0] if freq_stats else {}
    
    # 3. SIMULATE SHARD DISTRIBUTION
    # For range sharding: split key space into num_shards ranges
    all_keys = sorted(collection.distinct(key_field))
    
    if not all_keys:
        return {"error": "No data"}
    
    chunk_size = len(all_keys) // num_shards
    shard_counts = []
    
    for s in range(num_shards):
        start_idx = s * chunk_size
        end_idx = (s + 1) * chunk_size if s < num_shards - 1 else len(all_keys)
        
        start_key = all_keys[start_idx]
        end_key = all_keys[min(end_idx, len(all_keys) - 1)]
        
        if s < num_shards - 1:
            count = collection.count_documents({
                key_field: {"$gte": start_key, "$lt": all_keys[end_idx]}
            })
        else:
            count = collection.count_documents({
                key_field: {"$gte": start_key}
            })
        
        shard_counts.append({
            "shard": f"Shard {s}",
            "range": f"[{start_key} → {end_key}]",
            "docs": count,
            "pct": round(count / total_docs * 100, 1)
        })
    
    # 4. MONOTONICITY CHECK
    # If the key is always increasing (like dates or auto-increment)
    # then all NEW writes go to the LAST shard = hot shard
    sample = list(collection.find(
        {}, {key_field: 1, "_id": 0}
    ).sort("_id", ASCENDING).limit(100))
    
    values = [doc[key_field] for doc in sample if key_field in doc]
    is_monotonic = all(values[i] <= values[i+1] for i in range(len(values)-1)) if len(values) > 1 else False
    
    # 5. SKEW RATIO
    max_docs = max(s["docs"] for s in shard_counts)
    min_docs = min(s["docs"] for s in shard_counts) or 1
    skew_ratio = round(max_docs / min_docs, 2)
    
    return {
        "key_field": key_field,
        "cardinality": distinct_count,
        "total_docs": total_docs,
        "avg_per_key": round(freq.get("avg_per_key", 0), 1),
        "max_per_key": freq.get("max_per_key", 0),
        "min_per_key": freq.get("min_per_key", 0),
        "stddev": round(freq.get("stddev", 0), 2),
        "shard_distribution": shard_counts,
        "skew_ratio": skew_ratio,
        "is_monotonic": is_monotonic,
        "cardinality_ratio": round(distinct_count / total_docs * 100, 1)
    }

def score_shard_key(analysis, supports_primary_query=True):
    """
    Score a shard key candidate on the three critical factors.
    
    MongoDB shard key factors:
        1. Cardinality — HIGH is good (many possible values)
        2. Frequency — EVEN is good (each value appears similar number of times)
        3. Monotonicity — NOT monotonic is good (avoids hot shard)
    
    Plus: query targeting — does it support the most common query?
    
    Redshift DISTKEY factors (for comparison):
        1. Cardinality — HIGH is good (even distribution across slices)
        2. JOIN frequency — distribute on the JOIN column
        3. Query pattern — filter on DISTKEY for single-slice scans
        Both systems optimize for: even distribution + query targeting
    """
    
    scores = {}
    
    # Cardinality score (0-5)
    ratio = analysis["cardinality_ratio"]
    if ratio > 50:
        scores["cardinality"] = 5
    elif ratio > 20:
        scores["cardinality"] = 4
    elif ratio > 5:
        scores["cardinality"] = 3
    elif ratio > 1:
        scores["cardinality"] = 2
    else:
        scores["cardinality"] = 1
    
    # Frequency score (0-5) based on skew ratio
    skew = analysis["skew_ratio"]
    if skew < 1.2:
        scores["frequency"] = 5
    elif skew < 1.5:
        scores["frequency"] = 4
    elif skew < 2.0:
        scores["frequency"] = 3
    elif skew < 5.0:
        scores["frequency"] = 2
    else:
        scores["frequency"] = 1
    
    # Monotonicity score (0-5)
    scores["monotonicity"] = 1 if analysis["is_monotonic"] else 5
    
    # Query targeting score (0-5)
    scores["query_targeting"] = 5 if supports_primary_query else 2
    
    # Total
    scores["total"] = sum(scores.values())
    scores["max_possible"] = 20
    
    return scores

def simulate_routing(query_filter, shard_key_fields, description):
    """
    Simulate how mongos would route a query.
    
    TARGETED: shard key (or prefix) in query filter → single shard
    SCATTER-GATHER: no shard key in filter → all shards
    
    Redshift equivalent:
        TARGETED = WHERE includes DISTKEY → single slice
        SCATTER-GATHER = WHERE doesn't include DISTKEY → all slices
    """
    
    # Check if shard key prefix is in the query filter
    filter_keys = set(query_filter.keys())
    shard_prefix = shard_key_fields[0]  # first field of compound key
    
    is_targeted = shard_prefix in filter_keys
    
    # Execute and time the query
    start = time.time()
    explain = orders.find(query_filter).explain()
    results = list(orders.find(query_filter))
    elapsed_ms = (time.time() - start) * 1000
    
    exec_stats = explain.get("executionStats", {})
    docs_examined = exec_stats.get("totalDocsExamined", "N/A")
    keys_examined = exec_stats.get("totalKeysExamined", "N/A")
    
    routing = "TARGETED (single shard)" if is_targeted else "SCATTER-GATHER (all shards)"
    
    print(f"\n  Query: {description}")
    print(f"  Filter: {query_filter}")
    print(f"  Routing: {routing}")
    print(f"  Results: {len(results)} documents")
    print(f"  Docs examined: {docs_examined}")
    print(f"  Time: {elapsed_ms:.2f}ms")
    
    if is_targeted:
        print(f"  ✅ Efficient: shard key prefix '{shard_prefix}' in filter")
        print(f"     → Only 1 shard processes this query")
    else:
        print(f"  ⚠️ Expensive: shard key prefix '{shard_prefix}' NOT in filter")
        print(f"     → ALL shards must process this query")
        print(f"     → mongos merges results from all shards")
    
    return {"targeted": is_targeted, "results": len(results), "time_ms": elapsed_ms}

✅ WHAT SHOULD DO (Best Practices)

✅ CHOOSE SHARD KEY BASED ON PRIMARY ACCESS PATTERN
    → "What query runs most often?"
    → "What filter is in 80% of queries?"
    → That field (or its prefix) should be in the shard key
    → Same principle as Redshift DISTKEY for JOIN co-location

✅ USE COMPOUND SHARD KEYS FOR HIGH CARDINALITY
    → { customer_id: 1 } alone = moderate cardinality
    → { customer_id: 1, order_date: 1 } = very high cardinality
    → High cardinality = more chunks = better distribution
    → More granular balancing possible

✅ CREATE THE SHARD KEY INDEX BEFORE SHARDING
    → sh.shardCollection() requires a supporting index
    → Create it first: db.collection.createIndex({ key: 1 })
    → On existing collection with data: index build takes time
    → Plan for this during maintenance window

✅ MONITOR CHUNK DISTRIBUTION REGULARLY
    → sh.status() — check chunk counts per shard
    → db.collection.getShardDistribution() — check data balance
    → Alert if skew ratio exceeds 2:1
    → Balancer should keep things even, but verify

✅ USE HASHED SHARDING FOR WRITE-HEAVY COLLECTIONS WITH POINT LOOKUPS
    → events collection with random access → hashed
    → Guarantees even write distribution
    → Point queries still targeted (hash lookup)
    → Sacrifice: range queries become scatter-gather

✅ DON'T SHARD SMALL COLLECTIONS
    → < 1 GB: definitely don't shard
    → 1-10 GB: probably don't shard
    → > 10 GB: consider sharding
    → > 100 GB: almost certainly should shard
    → Redshift equivalent: small tables → DISTSTYLE ALL

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT SHARD ON A MONOTONICALLY INCREASING FIELD (range sharding)
    WRONG: sh.shardCollection("db.events", { created_at: 1 })
    WHY: all new events → latest chunk → ONE shard gets all writes
    FIX: use hashed sharding OR compound key with non-monotonic prefix
    
    REDSHIFT COMPARISON:
    → Redshift uses HASH of DISTKEY → monotonic values still distributed evenly
    → MongoDB range sharding: monotonic = guaranteed hot shard
    → This is the #1 sharding mistake

❌ DO NOT SHARD ON LOW-CARDINALITY FIELDS
    WRONG: sh.shardCollection("db.orders", { status: 1 })
    WHY: only 5 statuses → max 5 chunks → can't balance across 3+ shards
    → "Jumbo chunks" that can't be split → one shard gets disproportionate data
    
    REDSHIFT COMPARISON:
    → DISTKEY on low-cardinality column → data skew across slices
    → Same problem in both systems → avoid low-cardinality distribution keys

❌ DO NOT CHOOSE SHARD KEY BASED ON WRITE PATTERN ALONE
    WRONG: "Hashed _id gives perfect write distribution — done!"
    WHY: if 90% of READS are by customer_id → every read is scatter-gather
    → Write distribution is ONE factor — query targeting matters more
    → Balance: write distribution + read targeting + cardinality

❌ DO NOT CHANGE SHARD KEY FREQUENTLY
    → Before MongoDB 5.0: shard key was IMMUTABLE (could never change)
    → MongoDB 5.0+: resharding possible but EXPENSIVE (hours to days)
    → Choose carefully upfront — treat like Redshift DISTKEY decision
    → Changing Redshift DISTKEY: recreate table (also expensive)
    → Both systems: distribution key is a major architectural decision

❌ DO NOT SHARD EVERYTHING "JUST IN CASE"
    → Sharding adds complexity: mongos routing, config servers, balancer
    → Sharding adds latency: extra network hop through mongos
    → Sharding complicates operations: backups, migrations, upgrades
    → Only shard when you NEED to (capacity or throughput limits)

❌ DO NOT CONNECT DIRECTLY TO SHARDS
    → Always connect through mongos (query router)
    → Direct shard connection sees only that shard's data
    → Like connecting to a single Redshift compute node instead of leader
    → Application connection string must point to mongos

📍 WHEN TO USE MONGODB SHARDING OVER REDSHIFT SCALING

USE MONGODB SHARDING WHEN:
    → Operational workload (OLTP): high-concurrency reads/writes
    → Need zero-downtime scaling (add shards while serving traffic)
    → Data is naturally partitioned (multi-tenant, per-customer)
    → Need independent scaling per collection
    → Write throughput is the bottleneck (MongoDB handles writes better)
    → Geographic distribution needed (zone sharding — Project 39)

📍 WHEN TO STICK WITH REDSHIFT SCALING INSTEAD

USE REDSHIFT RESIZE WHEN:
    → Analytical workload (OLAP): complex aggregations, JOINs, window functions
    → Simpler operational model (one command to resize)
    → Brief downtime is acceptable (elastic resize: minutes)
    → Query complexity matters more than write throughput
    → BI tools need SQL access to all data
    → Columnar compression keeps storage manageable

MongoDB: Chunk Migration Mechanics

WHAT HAPPENS WHEN THE BALANCER MOVES A CHUNK:

1. BALANCER DECISION:
   → Checks chunk count per shard every 10 seconds
   → If difference between most-loaded and least-loaded > threshold (8 chunks)
   → Selects a chunk from overloaded shard
   → Targets underloaded shard as destination

2. CHUNK MIGRATION PROCESS:
   Step 1: Source shard starts sending documents to destination shard
           → Reads documents in chunk range
           → Streams to destination over internal network
           → Application reads/writes CONTINUE on source shard
   
   Step 2: Destination shard receives and indexes documents
           → Builds local indexes on received data
           → Does NOT serve queries for this range yet
   
   Step 3: Source shard sends any changes made DURING migration
           → "Catch-up" phase: transfer deltas
           → Repeat until delta is very small
   
   Step 4: COMMIT — atomic metadata update in config servers
           → Config servers update chunk→shard mapping
           → mongos routers refresh their routing tables
           → Destination shard starts serving this chunk
           → Source shard stops serving this chunk
           → Source shard deletes migrated documents (cleanup)
   
   Step 5: Cleanup
           → Source shard: range deleter removes orphaned documents
           → Config servers: updated chunk ownership recorded

3. DURING MIGRATION — WHAT APPLICATIONS SEE:
   → Reads: served by SOURCE shard until commit (no interruption)
   → Writes: accepted by SOURCE shard, forwarded if needed
   → After commit: reads/writes served by DESTINATION shard
   → Application notices NOTHING — mongos handles routing transparently

4. PERFORMANCE IMPACT DURING MIGRATION:
   → Source shard: increased I/O (reading + sending data)
   → Destination shard: increased I/O (receiving + indexing)
   → Network: chunk data transferred between shards
   → Latency: slight increase during heavy migration
   → WHY balancer windows exist: schedule migrations during off-peak

Redshift: Resize Mechanics (For Comparison)

WHAT HAPPENS DURING REDSHIFT ELASTIC RESIZE:

1. NEW NODES PROVISIONED:
   → AWS spins up new compute nodes
   → New nodes join the cluster
   → Takes: 2-5 minutes

2. DATA REDISTRIBUTION:
   → ALL data redistributed across old + new nodes
   → Based on DISTKEY hash: recalculated for new slice count
   → Parallel copy: all nodes participate
   → Takes: 10-60 minutes depending on data volume

3. SWITCHOVER:
   → Brief period of unavailability (seconds to minutes)
   → Connections dropped and reconnected
   → New node configuration becomes active
   → Old nodes decommissioned

4. DIFFERENCES FROM MONGODB:
   → Redshift: ALL data moves at once (big bang)
   → MongoDB: chunks move gradually (incremental)
   → Redshift: brief downtime during switchover
   → MongoDB: zero downtime (transparent routing)
   → Redshift: faster total migration (parallel, all at once)
   → MongoDB: slower total migration (chunk by chunk, background)
   → Redshift: simpler (one command)
   → MongoDB: more complex (ongoing balancer management)

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I'll shard on _id — it's always unique"
    REDSHIFT THINKING: "IDENTITY column is a natural distribution key"
    MONGODB TRAP:
        → ObjectId is monotonically increasing (timestamp-based)
        → Range sharding on _id → HOT SHARD (all new docs → last chunk)
        → Hashed _id → even distribution BUT no query targeting
        → Most queries are NOT by _id (they're by customer_id, product_id)
    FIX: Shard on the field you QUERY by most often

PITFALL 2: "I sharded and now my queries are slower"
    CAUSE: queries don't include shard key → scatter-gather on every query
    EXAMPLE:
        → Shard key: customer_id
        → Common query: db.orders.find({ status: "pending" })
        → status NOT in shard key → ALL shards queried
        → Before sharding: 1 server, 45ms
        → After sharding: 3 servers × 45ms + merge overhead = 60ms
        → SLOWER because of scatter-gather overhead
    FIX: ensure primary queries include shard key
         add secondary indexes for non-shard-key queries

PITFALL 3: "Jumbo chunks won't split"
    CAUSE: all documents in chunk have same shard key value
    EXAMPLE:
        → Shard key: { country: 1 }
        → 60% of customers in "US"
        → All US orders in ONE chunk → exceeds 128 MB → can't split
        → This chunk can't be moved → permanent imbalance
    FIX: use higher-cardinality shard key
         compound key: { country: 1, customer_id: 1 }

PITFALL 4: "I changed my application queries but not the shard key"
    SCENARIO:
        → Originally: queries by customer_id (shard key = customer_id)
        → New feature: search orders by product → scatter-gather
        → Performance degrades as product-based queries increase
    FIX: anticipate ALL major query patterns when choosing shard key
         or plan for resharding (Project 40)

PITFALL 5: "Treating shard key like it's easily changeable"
    REDSHIFT: ALTER TABLE → change DISTKEY by recreating table (painful but doable)
    MONGODB: Before 5.0 → shard key is PERMANENT (cannot change)
             After 5.0 → resharding possible but takes hours/days
    LESSON: shard key choice is THE most consequential MongoDB decision
