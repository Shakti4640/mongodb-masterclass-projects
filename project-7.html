PROJECT 7: Flexible Schema — No ALTER TABLE Ever Again

Problem: Product Catalog Where Electronics Have Specs, Clothing Has Sizes, Books Have ISBN — All in One Collection

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform sells DIFFERENT types of products:
    
    ELECTRONICS:
        → Wireless Mouse: voltage, battery_life, connectivity, dpi
        → Mechanical Keyboard: switch_type, backlit, key_count
        → Monitor: resolution, panel_type, refresh_rate, ports
    
    CLOTHING:
        → T-Shirt: sizes, colors, fabric, fit_type
        → Running Shoes: sizes, width, terrain_type
        → Winter Jacket: sizes, insulation_type, waterproof_rating
    
    BOOKS:
        → Python Crash Course: isbn, pages, edition, publisher, language
        → MongoDB in Action: isbn, pages, edition, format (ebook/print)
    
    FOOD:
        → Organic Coffee: weight, origin, roast_level, expiry_date
        → Protein Bars: weight, flavor, calories, allergens[]
    
    ALL share some common fields: name, price, category, stock_qty
    But each type has COMPLETELY DIFFERENT additional attributes
    
    New product types appear regularly (furniture, digital downloads, subscriptions)
    Each type brings NEW attributes that don't apply to other types
    
    How do you store this?
    
    In Redshift: painful choice between wide sparse tables, 
                 multiple tables, or SUPER type
    In MongoDB:  each document carries exactly the fields IT needs
                 — no schema change, no ALTER TABLE, ever

🔴 REDSHIFT MINDSET — How You Would Solve This

Option A: One Wide Table (Most Common but Painful)

-- REDSHIFT OPTION A: One table with ALL possible columns
-- The "wide sparse table" approach

    product_type    VARCHAR(50),             -- discriminator
    created_at      TIMESTAMP DEFAULT GETDATE(),

    voltage         VARCHAR(20),
    battery_life    VARCHAR(50),

    switch_type     VARCHAR(50),

    resolution      VARCHAR(20),
    panel_type      VARCHAR(20),
    refresh_rate    INTEGER,
    ports           VARCHAR(255),            -- comma-separated? JSON string?
    
    -- Clothing-specific (NULL for non-clothing)
    sizes           VARCHAR(255),            -- how to store array? CSV? JSON?
    colors          VARCHAR(255),            -- same problem
    fabric          VARCHAR(100),
    fit_type        VARCHAR(50),
    width           VARCHAR(10),

DISTSTYLE KEY DISTKEY(category) SORTKEY(created_at);

-- PROBLEMS WITH THIS APPROACH:
-- ┌──────────────────────────────────────────────────────────────┐
-- │ 1. ~30 columns, but each product uses only ~8-10             │
-- │    → 60-70% of columns are NULL for any given row            │
-- │    → Wasted storage (Redshift compresses NULLs well, but...) │
-- │                                                               │
-- │ 2. Adding a new product type = ALTER TABLE ADD COLUMN         │
-- │    → ALTER TABLE is a DDL change — requires planning          │
-- │    → ALL existing rows get the new column (as NULL)           │
-- │    → Schema migrations become frequent and risky              │
-- │                                                               │
-- │ 3. Array data (sizes, colors, allergens) stored as strings   │
-- │    → "S,M,L,XL" or '["S","M","L","XL"]' as VARCHAR           │
-- │    → Cannot query: "find products available in size M"        │
-- │    → Must use LIKE '%M%' (incorrect: matches "Medium" too)   │
-- │                                                               │
-- │ 4. Self-documenting problem                                   │
-- │    → Looking at the table, you can't tell which columns       │
-- │      belong to which product type                             │
-- │    → Developer must KNOW that "dpi" is only for mice         │
-- │                                                               │
-- │ 5. Queries must always filter by product_type                 │
-- │    → SELECT * WHERE product_type = 'electronics' AND dpi > 0 │
-- │    → Without type filter, NULL checks are needed everywhere  │
-- └──────────────────────────────────────────────────────────────┘

Option B: Multiple Tables (Normalized but Complex)

-- REDSHIFT OPTION B: Base table + type-specific tables

-- Shared attributes
CREATE TABLE store.products_base (
    product_id      BIGINT IDENTITY(1,1) PRIMARY KEY,
    name            VARCHAR(255) NOT NULL,
    price           DECIMAL(10,2) NOT NULL,
    category        VARCHAR(100) NOT NULL,
    product_type    VARCHAR(50) NOT NULL,
    stock_qty       INTEGER DEFAULT 0,
    created_at      TIMESTAMP DEFAULT GETDATE()
) DISTSTYLE KEY DISTKEY(product_id) SORTKEY(created_at);

-- Electronics extension
CREATE TABLE store.products_electronics (
    product_id      BIGINT PRIMARY KEY,      -- FK to products_base
    voltage         VARCHAR(20),
    battery_life    VARCHAR(50),
    connectivity    VARCHAR(50),
    dpi             INTEGER
) DISTSTYLE KEY DISTKEY(product_id);

-- Clothing extension
CREATE TABLE store.products_clothing (
    product_id      BIGINT PRIMARY KEY,
    fabric          VARCHAR(100),
    fit_type        VARCHAR(50)
) DISTSTYLE KEY DISTKEY(product_id);

-- Clothing sizes (many-to-many)
CREATE TABLE store.product_sizes (
    product_id      BIGINT,
    size            VARCHAR(10)
) DISTSTYLE KEY DISTKEY(product_id);

-- Books extension
CREATE TABLE store.products_books (
    product_id      BIGINT PRIMARY KEY,
    isbn            VARCHAR(20),
    pages           INTEGER,
    edition         INTEGER,
    publisher       VARCHAR(200)
) DISTSTYLE KEY DISTKEY(product_id);

-- PROBLEMS:
-- → 6+ tables for ONE conceptual entity (products)
-- → JOINs needed for EVERY query
-- → Adding a new product type = creating new tables
-- → Queries must UNION or LEFT JOIN across type tables
-- → Application must know which type table to JOIN
-- → Very complex query for "show me all products":

SELECT b.*, e.voltage, e.dpi, c.fabric, bk.isbn
FROM store.products_base b
LEFT JOIN store.products_electronics e ON b.product_id = e.product_id
LEFT JOIN store.products_clothing c ON b.product_id = c.product_id
LEFT JOIN store.products_books bk ON b.product_id = bk.product_id;
-- → Gets worse with every new product type added

Option C: SUPER Type (Redshift's Semi-Structured Answer)

-- REDSHIFT OPTION C: SUPER type for flexible attributes

CREATE TABLE store.products_flex (
    product_id      BIGINT IDENTITY(1,1) PRIMARY KEY,
    name            VARCHAR(255) NOT NULL,
    price           DECIMAL(10,2) NOT NULL,
    category        VARCHAR(100) NOT NULL,
    product_type    VARCHAR(50) NOT NULL,
    stock_qty       INTEGER DEFAULT 0,
    attributes      SUPER,                   -- semi-structured JSON
    created_at      TIMESTAMP DEFAULT GETDATE()
) DISTSTYLE KEY DISTKEY(category) SORTKEY(created_at);

-- Insert electronics
INSERT INTO store.products_flex (name, price, category, product_type, stock_qty, attributes)
VALUES (
    'Wireless Mouse', 29.99, 'Electronics', 'mouse', 150,
    JSON_PARSE('{"voltage":"3V","battery_life":"12 months","connectivity":"bluetooth","dpi":1600}')
);

-- Insert clothing
INSERT INTO store.products_flex (name, price, category, product_type, stock_qty, attributes)
VALUES (
    'Running Shoes', 129.99, 'Footwear', 'shoes', 200,
    JSON_PARSE('{"sizes":[7,8,9,10,11,12],"width":"standard","terrain":"road"}')
);

-- Querying SUPER type:
SELECT name, price, attributes.dpi
FROM store.products_flex
WHERE product_type = 'mouse' AND attributes.dpi > 1000;

-- PROBLEMS:
-- → SUPER queries are SLOWER than native column queries
-- → No columnar compression on SUPER internals
-- → Limited indexing on SUPER fields (no sort key benefit)
-- → JSON_PARSE required for every insert
-- → Querying nested SUPER is verbose and error-prone
-- → SUPER max size: 16MB (same as MongoDB documents!)
-- → Basically: you're using Redshift as a document store (poorly)

The Redshift Reality

ALL THREE OPTIONS HAVE SIGNIFICANT TRADE-OFFS:

Option A (Wide Table):
    ✗ Sparse NULLs, constant ALTER TABLE, no array support
    ✓ Simple queries, familiar pattern

Option B (Multiple Tables):
    ✗ Complex JOINs, many tables to maintain, new tables per type
    ✓ Clean normalization, type-specific constraints

Option C (SUPER Type):
    ✗ Slower queries, limited optimization, verbose syntax
    ✓ Flexible, no ALTER TABLE
    → But if you're using SUPER heavily... you probably want MongoDB

NONE of these options are GOOD for polymorphic data in Redshift
Redshift was designed for UNIFORM data (every row has same columns)

🟢 MONGODB MINDSET — How This Same Problem Is Approached

The Fundamental Insight

MONGODB'S ANSWER: EACH DOCUMENT CARRIES ITS OWN STRUCTURE

    There is no table definition that constrains what fields a document has
    
    Electronics document:
    { name: "Wireless Mouse", price: 29.99, dpi: 1600, connectivity: "bluetooth" }
    
    Clothing document:
    { name: "Running Shoes", price: 129.99, sizes: [7,8,9,10,11,12], terrain: "road" }
    
    Book document:
    { name: "MongoDB in Action", price: 39.99, isbn: "978-1617291609", pages: 480 }
    
    ALL IN THE SAME COLLECTION
    
    No NULLs for missing fields — fields simply DON'T EXIST
    No ALTER TABLE — new fields just appear in new documents
    No arrays-as-strings — arrays are NATIVE
    No JOINs — each document is self-contained
    
    THIS IS THE "POLYMORPHIC PATTERN" — one of MongoDB's most powerful features

🧩 HOW TO APPROACH IT IN MONGODB

THE POLYMORPHIC DOCUMENT PATTERN:

    COMMON FIELDS (shared by ALL products):
        name, price, category, product_type, stock_qty, created_at
    
    TYPE-SPECIFIC FIELDS (only present when relevant):
        Electronics: specs.voltage, specs.battery_life, specs.connectivity
        Clothing: sizes[], colors[], fabric, fit_type
        Books: isbn, pages, edition, publisher
        Food: weight, origin, expiry_date, allergens[]
    
    DISCRIMINATOR FIELD:
        product_type: "electronics" | "clothing" | "book" | "food"
        → Tells the application HOW to interpret the document
        → Enables type-specific queries and indexes
        → Like Redshift's product_type column but MORE powerful

    DOCUMENT STRUCTURE PATTERN:
    {
        // COMMON FIELDS (every product has these)
        name: "...",
        price: NumberDecimal("29.99"),
        category: "...",
        product_type: "electronics",      ← DISCRIMINATOR
        stock_qty: 150,
        
        // TYPE-SPECIFIC FIELDS (only this type has these)
        specs: {                           ← grouped under "specs" for clarity
            voltage: "3V",
            battery_life: "12 months",
            connectivity: "bluetooth"
        },
        
        // SHARED OPTIONAL FIELDS (some products have, some don't)
        tags: ["wireless", "ergonomic"],
        images: ["url1.jpg", "url2.jpg"],
        
        // METADATA
        created_at: ISODate("..."),
        updated_at: ISODate("...")
    }

✅ WHAT SHOULD DO

BEST PRACTICES FOR POLYMORPHIC COLLECTIONS:

    ✅ USE A DISCRIMINATOR FIELD (product_type)
       → Every document should indicate what TYPE it is
       → Enables type-specific queries and indexes
       → Application uses this to determine how to render/process
       → Like Redshift's type column but fields actually VARY
    
    ✅ ENFORCE COMMON FIELDS WITH SCHEMA VALIDATION
       → name, price, category, product_type → REQUIRED
       → Use $jsonSchema validator on the collection
       → Let type-specific fields be FREE (no additionalProperties restriction)
       → This gives structure where it matters, flexibility where you need it
    
    ✅ GROUP TYPE-SPECIFIC FIELDS IN NESTED OBJECTS
       → Electronics: specs: { voltage, dpi, connectivity }
       → Not: voltage, dpi, connectivity at top level
       → Clearer document structure
       → Easier to understand which fields belong to which concern
    
    ✅ USE PARTIAL INDEXES FOR TYPE-SPECIFIC QUERIES
       → Index on specs.dpi ONLY for electronics:
         db.products.createIndex(
             { "specs.dpi": 1 },
             { partialFilterExpression: { product_type: "electronics" } }
         )
       → Smaller index, only contains relevant documents
       → We'll cover this in Project 11 (Index Types)
    
    ✅ EMBRACE FIELD ABSENCE (vs NULLs)
       → If a book doesn't have "specs" → field simply doesn't exist
       → Not: specs: null (which takes storage and appears in queries)
       → Use $exists to check for field presence
       → This is MongoDB's natural way — don't fight it

REDSHIFT HABITS TO UNLEARN:
    🔄 "Every record must have the same columns"
       → Documents in same collection can have completely different fields
       → This is a FEATURE, not a bug
    
    🔄 "I need ALTER TABLE for new fields"
       → Just include new fields in new documents
       → Old documents are unaffected
       → Zero downtime, zero migration
    
    🔄 "Arrays must be stored as strings"
       → MongoDB arrays are native, queryable, indexable
       → sizes: [7, 8, 9, 10] NOT sizes: "7,8,9,10"
    
    🔄 "Schema changes require a migration plan"
       → MongoDB schema evolves naturally with your application
       → New fields appear as business needs change
       → Old documents coexist with new ones
       → Application handles both shapes (backward compatibility)

❌ WHAT SHOULD NOT DO

ANTI-PATTERNS:

    ❌ ZERO VALIDATION (complete anarchy)
       → "MongoDB has no schema so anything goes!"
       → Result: price stored as string in some docs, number in others
       → Result: "product_type" misspelled as "productType" in some docs
       → FIX: Use schema validation for common fields at minimum
       
    ❌ SEPARATE COLLECTION PER PRODUCT TYPE
       # WRONG
       db.electronics.insertOne({...})

       db.products.insertOne({ product_type: "electronics", ... })
       db.products.insertOne({ product_type: "clothing", ... })
       
       → One collection, one query for cross-type operations
       → Exception: if types have COMPLETELY different access patterns

       { electronics_voltage: "3V", electronics_dpi: 1600 }
       { clothing_fabric: "Cotton", clothing_fit: "Regular" }

HOW MONGODB HANDLES DIFFERENT DOCUMENT SHAPES IN SAME COLLECTION:

    WIREDTIGER STORAGE:
    → WiredTiger stores each document as a BSON blob
    → It does NOT know or care about document structure
    → Document A with 8 fields and Document B with 15 fields

    → No column alignment, no NULL padding, no wasted space

    │                                                          │
    │ [Doc A: electronics, 350 bytes BSON]                     │
    │ [Doc B: clothing, 280 bytes BSON]                        │
    │ [Doc C: book, 420 bytes BSON]                            │
    │ [Doc D: food, 310 bytes BSON]                            │
    │ [Doc E: digital, 500 bytes BSON]                         │
    │                                                          │
    │ → Each document is independently sized                   │
    │ → No wasted space for absent fields                      │
    │ → No alignment to fixed column widths                    │
    └─────────────────────────────────────────────────────────┘
    
    REDSHIFT COMPARISON:
    ┌─────────────────────────────────────────────────────────┐
    │ Redshift columnar blocks (wide table approach):          │
    │                                                          │
    │ name column:     [Mouse]    [Shoes]    [Book]    [Coffee]│
    │ price column:    [29.99]    [129.99]   [35.99]   [18.99] │
    │ voltage column:  [3V]       [NULL]     [NULL]    [NULL]  │
    │ dpi column:      [1600]     [NULL]     [NULL]    [NULL]  │
    │ fabric column:   [NULL]     [NULL]     [NULL]    [NULL]  │
    │ isbn column:     [NULL]     [NULL]     [978...]  [NULL]  │
    │ sizes column:    [NULL]     ["7,8.."]  [NULL]    [NULL]  │
    │ expiry column:   [NULL]     [NULL]     [NULL]    [2025..]│
    │                                                          │
    │ → NULLs everywhere (compressed, but still column blocks) │
    │ → 30 columns × all rows = 30 column block files          │
    │ → Most blocks are mostly NULL                            │
    │ → Query "SELECT isbn WHERE type='book'" still reads      │
    │   isbn column block for ALL rows (zone maps help but...) │
    └─────────────────────────────────────────────────────────┘

    WHY MONGODB IS BETTER FOR POLYMORPHIC DATA:
    → Zero storage for absent fields (not even NULL markers)
    → Document size proportional to actual data (not max possible columns)
    → No column blocks filled with NULLs
    → No ALTER TABLE when new fields appear
    → Native arrays — no string serialization hacks
    
    WHY REDSHIFT IS STILL BETTER FOR ANALYTICS ON THIS DATA:
    → "Average price across ALL products" reads one column block
    → MongoDB must read ALL documents to extract price field
    → Columnar compression on price column → extremely efficient scan
    → MongoDB document scan → reads name, specs, everything (wasteful)

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 7:

PITFALL 1: "I need to define all possible fields somewhere"
    REDSHIFT TRAINED YOU: Table DDL is the source of truth for structure
    MONGODB TRAP: Trying to pre-define all fields in a config or schema doc
    REALITY: The documents themselves ARE the schema
    
    FIX: Use schema validation for COMMON required fields only
         Document type-specific fields in application code or documentation
         Use tools like MongoDB Compass "Schema" tab to analyze field presence
    SEVERITY: LOW — wastes time but doesn't break anything

PITFALL 2: "Inconsistent field naming across documents"
    REDSHIFT TRAINED YOU: Column names enforced by DDL — can't misspell
    MONGODB TRAP: 
        Some docs: { "product_type": "electronics" }
        Other docs: { "productType": "electronics" }     ← camelCase
        Other docs: { "product-type": "electronics" }    ← kebab-case
        Other docs: { "type": "electronics" }            ← shortened
    
    CONSEQUENCE: Queries miss documents with wrong field names
    
    FIX: Establish naming conventions (snake_case recommended)
         Use schema validation to enforce required field names
         Code review for insert/update operations
         Use an ODM like mongoengine or motor for field name enforcement
    SEVERITY: CRITICAL — silently returns incomplete query results

PITFALL 3: "Inconsistent field TYPES across documents"
    REDSHIFT TRAINED YOU: Column types enforced by DDL
    MONGODB TRAP:
        Doc 1: { "price": 29.99 }         ← double
        Doc 2: { "price": "29.99" }        ← STRING (from CSV import)
        Doc 3: { "price": 2999 }           ← integer (cents)
        Doc 4: { "price": NumberDecimal("29.99") }  ← Decimal128
    
    CONSEQUENCE: 
        db.products.find({ price: { $gt: 25 } })
        → Misses Doc 2 (string "29.99" can't compare with number 25)
        → Includes Doc 3 (2999 > 25 → but it's cents, not dollars!)
    
    FIX: Schema validation with bsonType enforcement on critical fields
         Application-level type casting before insert
         Periodic data quality checks:
         db.products.find({ price: { $not: { $type: "double" } } })
    SEVERITY: CRITICAL — silent data corruption and wrong query results

PITFALL 4: "Not using a discriminator field"
    MONGODB TRAP: Inserting different types without any type indicator
    
    # WRONG — no way to know what type each document is
    { name: "Mouse", dpi: 1600 }
    { name: "Shoes", sizes: [7,8,9] }
    { name: "Book", isbn: "978..." }
    
    # RIGHT — discriminator field tells you the type
    { name: "Mouse", product_type: "electronics", dpi: 1600 }
    { name: "Shoes", product_type: "clothing", sizes: [7,8,9] }
    { name: "Book", product_type: "book", isbn: "978..." }
    
    WHY: Without discriminator, queries must GUESS type by field existence:
         { dpi: { $exists: true } } → probably electronics (fragile!)
         With discriminator: { product_type: "electronics" } → certain
    SEVERITY: HIGH — makes type-specific queries fragile and unreliable

PITFALL 5: "Assuming all documents have the same fields"
    REDSHIFT TRAINED YOU: Every row has every column
    MONGODB TRAP: Accessing a field without checking existence
    
    # PYTHON — crashes on clothing products
    for product in products.find():
        print(product["specs"]["dpi"])     # KeyError for clothing/books!
    
    # SAFE — check first
    for product in products.find():
        dpi = product.get("specs", {}).get("dpi", "N/A")
        print(f"{product['name']}: DPI={dpi}")
    
    # BETTER — filter in query
    for product in products.find(
        {"specs.dpi": {"$exists": True}},
        {"name": 1, "specs.dpi": 1}
    ):
        print(f"{product['name']}: DPI={product['specs']['dpi']}")
    
    SEVERITY: MEDIUM — runtime errors in application code

PITFALL 6: "Over-relying on $exists for type-specific queries"
    # FRAGILE — field existence as proxy for type
    db.products.find({ "isbn": { $exists: true } })  # "must be a book"
    
    # What if a non-book product accidentally gets an isbn field?
    # Or what if isbn is temporarily missing from a book?
    
    # ROBUST — use discriminator field
    db.products.find({ product_type: "book" })        # explicitly a book
    
    SEVERITY: MEDIUM — works until edge cases break it

PITFALL 7: "Changing validation on a collection with existing non-conforming data"
    # You add strict validation to existing collection:
    db.command("collMod", "products", validator={...}, validationLevel="strict")
    
    # Existing documents that DON'T match → CANNOT be updated
    # They're "grandfathered in" but any update triggers validation → ERROR
    
    FIX: Use validationLevel: "moderate" first
         → Only validates inserts and updates to already-conforming docs
         → Gradually migrate non-conforming docs
         → Then switch to "strict" when all docs conform
    SEVERITY: HIGH — existing documents become "stuck" (can't update)

📍 WHEN TO USE MONGODB OVER REDSHIFT

FOR POLYMORPHIC / VARYING-SCHEMA DATA:

USE MONGODB WHEN:
    → Entities have different attributes by type (product catalog)
    → New entity types appear frequently (startup iteration)
    → Schema evolves rapidly (agile development)
    → Array data is central to the model (sizes, colors, tags)
    → Application serves entity pages (product page, user profile)
    → Low-latency single-entity reads are critical
    → Schema flexibility is a business requirement

📍 WHEN TO STICK WITH REDSHIFT INSTEAD

FOR THE SAME DATA:

USE REDSHIFT WHEN:
    → All products have IDENTICAL structure (uniform catalog)
    → Primary queries are cross-product analytics:
      "Average price per category across all products"
      "Top 10 products by revenue this quarter"
    → Data is loaded in batches (nightly catalog sync)
    → BI tools need flat table access (Tableau, QuickSight)
    → Schema changes are infrequent and planned
