PROJECT 27: Single-Document Transactions (Implicit ACID)

Problem: Transfer Items Between Cart and Wishlist, Update Order with Line Items and Totals, and Maintain Consistent Counters — All Atomically Without Explicit Transactions

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform has five operations that MUST be atomic:

    OPERATION 1 — Cart Item Move
        → Customer moves item from wishlist to cart
        → Must: remove from wishlist array AND add to cart array
        → Cannot: have item in BOTH or NEITHER (inconsistent state)

    OPERATION 2 — Order Placement
        → Customer places order from cart
        → Must: create order line items, calculate total,
                set status, record payment info, set timestamp
        → ALL fields must be set together — partial order is invalid

    OPERATION 3 — Inventory + Stats Update
        → Product purchased: decrement stock AND increment sold_count
                             AND update last_sold_at
        → Cannot: decrement stock but forget sold_count (counter drift)

    OPERATION 4 — Review Submission
        → Customer submits review: add review to product's review array
                                   AND update avg_rating
                                   AND increment review_count
        → Cannot: add review but show wrong avg_rating

    OPERATION 5 — Account Balance Transfer (within single account doc)
        → Move $100 from savings_balance to checking_balance
        → Must: decrement savings AND increment checking atomically
        → Cannot: decrement savings but fail to increment checking

    IN REDSHIFT:
        → Every one of these requires BEGIN → multiple SQL statements → COMMIT
        → Even within ONE table, updating multiple columns needs a transaction
          to guarantee atomicity against concurrent readers

    IN MONGODB:
        → If all the data lives in ONE DOCUMENT:
          a single updateOne() is AUTOMATICALLY atomic
        → No BEGIN, no COMMIT, no transaction overhead
        → This is MongoDB's MOST POWERFUL design principle
        → And it's the reason document modeling matters SO much

🔴 REDSHIFT MINDSET — How You Would Solve This

Redshift Always Needs Explicit Transactions for Multi-Step Atomicity

OPERATION 1 IN REDSHIFT — Cart Item Move:

    -- Normalized tables: wishlists, cart_items
    -- MUST use explicit transaction
    
    BEGIN;
    
    -- Step 1: Read the item from wishlist
    SELECT product_id, qty, added_at
    FROM wishlist_items
    WHERE customer_id = 'CUST-001' AND product_id = 'PROD-001';
    
    -- Step 2: Delete from wishlist
    DELETE FROM wishlist_items
    WHERE customer_id = 'CUST-001' AND product_id = 'PROD-001';
    
    -- Step 3: Insert into cart
    INSERT INTO cart_items (customer_id, product_id, qty, added_at)
    VALUES ('CUST-001', 'PROD-001', 1, GETDATE());
    
    COMMIT;
    
    -- If ANY step fails: ROLLBACK — item stays in wishlist
    -- If COMMIT succeeds: item is atomically moved
    -- THREE separate SQL statements
    -- ONE transaction wrapping them
    -- If you FORGET the BEGIN/COMMIT: race conditions possible


OPERATION 3 IN REDSHIFT — Inventory + Stats:

    BEGIN;
    
    -- Must update multiple columns atomically
    UPDATE products
    SET stock_qty = stock_qty - 1,
        sold_count = sold_count + 1,
        last_sold_at = GETDATE()
    WHERE product_id = 'PROD-001'
      AND stock_qty > 0;  -- prevent overselling
    
    -- Check if update affected a row (stock was available)
    -- If not: rollback
    
    COMMIT;
    
    -- In Redshift: even updating multiple columns in ONE row
    -- benefits from transaction wrapping for consistency
    -- against concurrent readers


OPERATION 5 IN REDSHIFT — Balance Transfer (within one account):

    -- Even though it's ONE row in ONE table:
    BEGIN;
    
    UPDATE accounts
    SET savings_balance = savings_balance - 100,
        checking_balance = checking_balance + 100,
        updated_at = GETDATE()
    WHERE account_id = 'ACC-001'
      AND savings_balance >= 100;  -- sufficient funds check
    
    COMMIT;
    
    -- Redshift: single UPDATE statement IS atomic
    -- But explicit transaction still used for:
    -- → Read-then-write patterns
    -- → Multi-statement operations
    -- → Consistency across concurrent connections


THE REDSHIFT PATTERN:
    → Normalize data into separate tables
    → Use JOINs to query related data
    → Use transactions (BEGIN/COMMIT) for multi-table atomicity
    → Even single-table updates use transactions for safety
    → Transaction overhead: lock management, WAL entries, commit processing
    → Limited concurrency: transactions hold locks, WLM slots consumed

Why Single-Document Atomicity Works

WHAT HAPPENS DURING updateOne():

    db.customers.updateOne(
        { _id: "CUST-001" },
        {
            $pull: { wishlist: { product_id: "PROD-001" } },
            $push: { cart: { product_id: "PROD-001", qty: 1, added_at: new Date() } }
        }
    )

    INSIDE MONGODB:
    1. WiredTiger acquires document-level lock on CUST-001
    2. Document is read from storage (or cache)
    3. BOTH modifications applied to the in-memory document:
       → wishlist array: PROD-001 removed
       → cart array: PROD-001 added
    4. Modified document written atomically
    5. Lock released
    
    AT NO POINT can another operation see:
    → Item in BOTH wishlist and cart
    → Item in NEITHER wishlist nor cart
    → Wishlist modified but cart not yet modified
    
    IT'S ONE OPERATION. ONE LOCK. ONE WRITE.
    
    COMPARED TO REDSHIFT:
    → THREE SQL statements (SELECT, DELETE, INSERT)
    → ONE explicit transaction (BEGIN/COMMIT)
    → Table-level or row-level locks held for duration
    → If connection drops between DELETE and INSERT: 
      data could be inconsistent (without transaction)

🔧 HOW TO APPROACH IT IN MONGODB

DESIGN DECISION FRAMEWORK:

    FOR EVERY SET OF RELATED DATA, ASK:

    "Must these fields be updated ATOMICALLY together?"
    
    YES → Embed in the SAME document
        → Single updateOne() gives you free ACID
        → No transactions needed
        → Maximum performance
    
    NO  → Can be in separate documents/collections
        → Reference via ID
        → Eventual consistency acceptable
        → Or use multi-document transaction (Project 28) if needed

    EXAMPLES:

    ┌──────────────────────────────────┬─────────────────────────────────┐
    │ DATA RELATIONSHIP                 │ DECISION                         │
    ├──────────────────────────────────┼─────────────────────────────────┤
    │ Customer + cart + wishlist         │ EMBED — must move items atomically│
    │ Order + line items + total         │ EMBED — order is one unit         │
    │ Product + stock_qty + sold_count   │ EMBED — must update together      │
    │ Product + reviews array + avg_rating│ EMBED — must be consistent       │
    │ Account + balances (savings/check) │ EMBED — transfer is atomic       │
    │ Customer + order history (1000s)   │ REFERENCE — unbounded growth     │
    │ Product + full review text (large) │ REFERENCE — subset pattern (P19) │
    └──────────────────────────────────┴─────────────────────────────────┘

    THIS IS WHY PROJECT 14 (Embedding vs Referencing) MATTERS:
    → Your embedding decision IS your transaction decision
    → Embed = free atomicity
    → Reference = need explicit transactions or accept eventual consistency

        _id: "PROD-001",

        $inc: {
            stock_qty: -1,        // decrement stock
            sold_count: 1          // increment sold
        },
        $set: {
            last_sold_at: new Date()  // update timestamp
        }
    }
)

print(`\nUpdate: matched=${inventoryResult.matchedCount}, ` +
      `modified=${inventoryResult.modifiedCount}`)

} else {

    print(`  Last sold: ${afterProd.last_sold_at}`)

// → Update applies changes
// → Both in ONE operation — no race condition
// → In Redshift: SELECT to check, then UPDATE — race condition window

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

SINGLE-DOCUMENT ATOMICITY OPTIMIZATIONS:

1. COMBINE MULTIPLE $set/$inc/$push IN ONE updateOne()
   → NEVER do this:
     db.products.updateOne({_id: "P1"}, {$inc: {stock_qty: -1}})
     db.products.updateOne({_id: "P1"}, {$inc: {sold_count: 1}})
     db.products.updateOne({_id: "P1"}, {$set: {last_sold_at: new Date()}})
     → THREE operations — NOT atomic across all three
     → Between first and second: stock decremented but sold_count wrong
   
   → ALWAYS do this:
     db.products.updateOne({_id: "P1"}, {
         $inc: {stock_qty: -1, sold_count: 1},
         $set: {last_sold_at: new Date()}
     })
     → ONE operation — ALL changes atomic

2. USE PIPELINE UPDATES FOR COMPUTED FIELDS (MongoDB 4.2+)
   → Instead of read → compute in app → write (NOT atomic):
     doc = db.products.find_one({_id: "P1"})
     new_avg = (doc["sum"] + 5) / (doc["count"] + 1)  # app-side compute
     db.products.update_one({_id: "P1"}, {"$set": {"avg": new_avg}})
     → RACE CONDITION: another write between read and write
   
   → Use pipeline update (atomic):
     db.products.update_one({_id: "P1"}, [
         {"$set": {"count": {"$add": ["$count", 1]},
                   "sum":   {"$add": ["$sum", 5]}}},
         {"$set": {"avg": {"$divide": ["$sum", "$count"]}}}
     ])
     → ONE atomic operation — no race condition

3. USE findOneAndUpdate FOR READ-THEN-WRITE PATTERNS
   → Instead of find_one() + update_one() (TWO operations):
     doc = db.coupons.find_one({_id: "C1", remaining: {"$gt": 0}})
     if doc:
         db.coupons.update_one({_id: "C1"}, {"$inc": {"remaining": -1}})
     → RACE: another thread claims between find and update
   
   → Use find_one_and_update (ONE atomic operation):
     doc = db.coupons.find_one_and_update(
         {_id: "C1", remaining: {"$gt": 0}},
         {"$inc": {"remaining": -1}},
         return_document=ReturnDocument.AFTER
     )
     → Atomic: filter + update + return in one step

4. USE CONDITIONAL FILTERS AS LOCKS
   → Instead of explicit locking:
     → Put the precondition IN the filter
     → {stock_qty: {"$gte": 1}} — only matches if stock available
     → {status: "processing"} — only matches if not already shipped
     → {version: 5} — optimistic concurrency (only if unchanged)
   → If filter doesn't match: operation is a no-op (matched_count = 0)
   → Check matched_count to detect failure — no exception thrown

5. OPTIMISTIC CONCURRENCY CONTROL (VERSION FIELD)
   → Add a "version" field to documents
   → Read: get document with version = N
   → Write: filter includes version = N, update sets version = N+1
   → If another writer changed it: version ≠ N → no match → retry
   
   result = db.orders.update_one(
       {"_id": "ORD-001", "version": 5},
       {"$set": {"status": "shipped"}, "$inc": {"version": 1}}
   )
   if result.matched_count == 0:
       # Someone else modified the document — retry
       raise ConcurrencyConflict("Document was modified by another process")
   
   → No locks, no transactions — pure atomic conditional update
   → Same pattern as DynamoDB conditional writes
   → Redshift: would need SELECT FOR UPDATE (pessimistic locking)

6. $push WITH $slice FOR BOUNDED ARRAYS
   → Keep only N most recent items (prevents unbounded growth)
   → $push + $each + $sort + $slice = atomic bounded insert
   
   db.products.updateOne({_id: "P1"}, {
       $push: {
           recent_reviews: {
               $each: [newReview],
               $sort: {created_at: -1},  // sort newest first
               $slice: 10                // keep only 10
           }
       }
   })
   → Array never grows beyond 10 — automatic cleanup
   → In Redshift: DELETE + INSERT + subquery for windowed keep

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Internals — Document-Level Atomicity

HOW SINGLE-DOCUMENT ATOMICITY WORKS IN WIREDTIGER:

    STEP 1: updateOne() arrives at mongod
        → Query planner identifies target document by _id (or index)
        → Single document located in WiredTiger B-tree

    STEP 2: Document-level INTENT LOCK acquired
        → WiredTiger uses DOCUMENT-LEVEL locking (not table-level)
        → Only THIS document is locked
        → Other documents in same collection: fully accessible
        → Lock type: exclusive write lock on document's storage record
        → Lock granularity: MUCH finer than Redshift table locks

    STEP 3: Document read into WiredTiger's in-memory buffer
        → If in cache: direct access (no disk I/O)
        → If not in cache: read from disk into cache
        → Document is in BSON format (complete unit)

    STEP 4: ALL update operators applied IN MEMORY
        → $set: modify field values in the BSON buffer
        → $inc: read current value, add increment, write new value
        → $push: append to array in the BSON buffer
        → $pull: remove from array in the BSON buffer
        → ALL changes applied to the SAME in-memory buffer
        → This is ONE operation — no partial state exists

    STEP 5: Modified document written atomically
        → WiredTiger writes the ENTIRE modified document
        → Write-ahead log (journal) entry created FIRST
        → Then document page updated
        → If crash between journal and page update:
          → Recovery replays journal → document is consistent

    STEP 6: Lock released
        → Other operations can now read/write this document
        → They see the FULLY updated version
        → No intermediate state was EVER visible

    ┌──────────────────────────────────────────────────────────┐
    │                                                           │
    │  TIME →                                                   │
    │                                                           │
    │  T1: Lock acquired                                        │
    │  T2: Document read into memory                            │
    │  T3: $pull wishlist (in memory)                           │
    │  T3: $push cart (in memory)     ← SAME memory operation  │
    │  T4: Write modified document to journal                   │
    │  T5: Write modified document to data file                 │
    │  T6: Lock released                                        │
    │                                                           │
    │  Between T1 and T6: NO other operation can see or modify  │
    │  this document. The lock ensures exclusive access.         │
    │                                                           │
    │  If crash at T3: journal has no entry → document unchanged│
    │  If crash at T4: journal has entry → replayed on recovery │
    │  If crash at T5: journal has entry → replayed on recovery │
    │                                                           │
    │  RESULT: Document is ALWAYS in a consistent state         │
    │                                                           │
    └──────────────────────────────────────────────────────────┘

WHY DOCUMENT STORAGE ENABLES THIS:
    → Entire document is ONE storage unit (unlike columnar)
    → Reading ONE document = ONE I/O operation
    → Writing ONE document = ONE I/O operation
    → Lock ONE document = lock ONE storage record
    → No cross-file coordination needed (unlike Redshift columns)

    CONTRAST WITH REDSHIFT COLUMNAR STORAGE:
    → One "row" spans MULTIPLE column files
    → Updating 3 columns = writing to 3 different files
    → Atomicity requires: transaction log + multi-file coordination
    → More complex internally, more overhead per operation
    → BUT: columnar is superior for analytical scans (different trade-off)

Redshift Internals — For Comparison

HOW REDSHIFT HANDLES ATOMIC UPDATES:

    STEP 1: UPDATE statement arrives at leader node
        → Parsed, optimized, sent to relevant compute node(s)

    STEP 2: Compute node identifies target row
        → Row is logically across multiple column files
        → name.col, price.col, stock_qty.col, sold_count.col, ...

    STEP 3: Redshift's UPDATE = DELETE old row + INSERT new row
        → Old row MARKED as deleted (not physically removed)
        → New row with ALL columns written to NEW blocks
        → This is because columnar storage can't update in-place
        → VACUUM later reclaims deleted row space

    STEP 4: Transaction log entry
        → Commit log records the operation
        → If crash: replay log to recover consistent state
        → Transaction isolation: snapshot-based (Project 24 comparison)

    STEP 5: Commit
        → Transaction log entry finalized
        → New row visible to subsequent queries
        → Old row invisible to subsequent queries

    WHY REDSHIFT IS SLOWER FOR SINGLE-ROW UPDATES:
    → DELETE + INSERT across multiple column files
    → More I/O than MongoDB's single document write
    → Transaction log overhead for every operation
    → VACUUM needed to reclaim space (ongoing maintenance)
    → Designed for BULK operations, not single-row mutations

    ┌────────────────────────────────┬─────────────────┬──────────────────────┐
    │ METRIC                          │ MONGODB           │ REDSHIFT              │
    ├────────────────────────────────┼─────────────────┼──────────────────────┤
    │ Single-document/row update      │ ~0.5-5ms          │ ~10-100ms             │
    │ Lock granularity                │ Document-level    │ Table or block-level  │
    │ Concurrent updates (same coll)  │ Thousands/sec     │ Limited by WLM        │
    │ Update mechanism                │ In-place modify   │ Delete + re-insert    │
    │ Space reclamation               │ Automatic         │ VACUUM required       │
    │ Transaction overhead            │ Zero (single doc) │ Always present        │
    │ Bulk update (1M rows)           │ ~30-60 seconds    │ ~5-30 seconds         │
    │ Analytical scan after update    │ Full doc scan     │ Columnar scan (fast)  │
    └────────────────────────────────┴─────────────────┴──────────────────────┘

    ROOT INSIGHT:
    → MongoDB: optimized for single-document mutations (OLTP)
    → Redshift: optimized for bulk analytical operations (OLAP)
    → Single-document atomicity is MongoDB's NATURAL strength
    → It's not a feature — it's a consequence of document storage

└────────────────────────────────┴──────────────────────────────────────┘

HOT DOCUMENT PROBLEM:
    → If ONE document gets thousands of updates per second
    → Document-level lock creates serialization bottleneck
    → Example: global counter, viral product, popular event
    → Solutions:
      → Bucket pattern (Project 16): distribute across multiple documents
      → Computed pattern (Project 20): pre-aggregate in multiple docs
      → Sharding: ensure hot keys are distributed (Project 22)
    → Redshift: same problem exists with hot rows, but less impactful
      because Redshift isn't designed for high-frequency point updates

📐 DATA MODELING COMPARISON

HOW SINGLE-DOCUMENT ATOMICITY DRIVES DATA MODELING:

REDSHIFT (Normalize First, Transact Later):
    ┌──────────────────────────────────────────────┐
    │  customers table     (customer_id, name, ...) │
    │  wishlist_items table(customer_id, product_id)│
    │  cart_items table    (customer_id, product_id)│
    │  orders table        (order_id, customer_id)  │
    │  order_items table   (order_id, product_id)   │
    │  payments table      (payment_id, order_id)   │
    │  shipping table      (shipping_id, order_id)  │
    │                                                │
    │  7 tables for one customer's e-commerce data   │
    │  Every cross-table operation needs a transaction│
    │  JOINs everywhere for complete picture          │
    └──────────────────────────────────────────────┘

MONGODB (Embed for Atomicity, Reference for Independence):
    ┌──────────────────────────────────────────────┐
    │  customers collection:                        │
    │  {                                            │
    │    _id, name, email,                          │
    │    wishlist: [...],     ← embedded (atomic)   │
    │    cart: [...],         ← embedded (atomic)   │
    │    stats: {...}         ← embedded (atomic)   │
    │  }                                            │
    │                                                │
    │  orders collection:                           │
    │  {                                            │
    │    _id, customer_id,                          │
    │    items: [...],        ← embedded (atomic)   │
    │    payment: {...},      ← embedded (atomic)   │
    │    shipping: {...}      ← embedded (atomic)   │
    │  }                                            │
    │                                                │
    │  2 collections for same e-commerce data        │
    │  Cart ↔ wishlist move: ONE updateOne()         │
    │  Order creation: ONE insertOne()               │
    │  No transactions needed for common operations  │
    └──────────────────────────────────────────────┘

THE TRADE-OFF:
    Redshift (normalized):
    ✓ No data duplication
    ✓ Easy to update shared data (change category name once)
    ✓ Flexible querying across entities (complex JOINs)
    ✗ Transactions required for cross-table consistency
    ✗ JOINs for every complete entity read
    
    MongoDB (embedded):
    ✓ Single-document atomicity (free ACID)
    ✓ Single read for complete entity
    ✓ No JOIN overhead for common access patterns
    ✗ Potential data duplication (customer name in order)
    ✗ 16MB document limit (must manage array growth)
    ✗ Updating shared data requires updating multiple documents
