🔴 REDSHIFT MINDSET — "How Redshift Handles Encryption"

What Redshift Offers (And Where It Stops)

REDSHIFT ENCRYPTION — YOUR EXISTING KNOWLEDGE:

LAYER 1: ENCRYPTION AT REST ✅
    → KMS (Key Management Service) encryption
    → AES-256 encryption of all data blocks on disk
    → Automatic: always enabled for RA3 nodes
    → Key rotation: automatic or manual via KMS
    → YOU configured this when creating the cluster
    
    HOW IT WORKS:
    → Redshift encrypts every 1MB columnar block before writing to disk
    → Decrypts transparently when reading
    → Key hierarchy: Master Key → Cluster Key → Database Key → Block Key
    → Performance impact: <5% (hardware-accelerated AES)
    
    WHAT IT PROTECTS:
    ✅ Someone steals the physical disk → data unreadable
    ✅ S3 snapshots are encrypted
    ✅ Cross-region copies maintain encryption
    
    WHAT IT DOES NOT PROTECT:
    ❌ Anyone who can connect to Redshift sees ALL data in plain text
    ❌ DBA can SELECT * and see every credit card number
    ❌ A compromised application sees everything
    ❌ SQL injection attack reveals all data

LAYER 2: ENCRYPTION IN TRANSIT ✅
    → SSL/TLS between client and Redshift
    → Configured via parameter group: require_ssl = true
    → Also: COPY/UNLOAD to S3 encrypted in transit
    
    WHAT IT PROTECTS:
    ✅ Network sniffer can't read queries or results
    ✅ Man-in-the-middle attack prevented
    
    WHAT IT DOES NOT PROTECT:
    ❌ Same as Layer 1 — once connected, all data is plain text

LAYER 3: FIELD-LEVEL ENCRYPTION ❌ DOES NOT EXIST
    → Redshift has NO capability to encrypt individual columns differently
    → Every column uses the same cluster-level KMS key
    → You CANNOT:
      - Encrypt SSN column with Key A
      - Encrypt credit_card column with Key B
      - Leave order_total unencrypted
    → You CANNOT prevent a DBA from reading any column
    → You CANNOT give different decryption keys to different applications
    
    WORKAROUNDS (all inadequate):
    → Application-level encryption: encrypt before INSERT, decrypt after SELECT
      Problem: can't query on encrypted columns (WHERE ssn = '...' fails)
    → Redshift UDF with encryption: custom function for encrypt/decrypt
      Problem: key must be in UDF code or parameter → DBA can see it
    → Column-level GRANT (Project 37): hides column from users
      Problem: DBA/superuser bypasses all GRANTs → sees everything
    → External tokenization service: replace PII with tokens
      Problem: adds latency, complexity, separate token vault

    BOTTOM LINE:
    → Redshift encryption is ALL-OR-NOTHING at the storage level
    → Once you're authenticated, you see everything
    → No way to encrypt field X with key A and field Y with key B
    → MongoDB CSFLE solves this problem completely

🧩 HOW TO APPROACH IT IN MONGODB

THREE-LAYER IMPLEMENTATION PLAN:

LAYER 1 — AT REST (same concept as Redshift KMS):
    → Atlas: already enabled (nothing to do)
    → Self-managed: enable WiredTiger encryption in mongod config
    → Optional: BYOK with AWS KMS for key management
    → RESULT: all data files encrypted on disk

LAYER 2 — IN TRANSIT (same concept as Redshift SSL):
    → Atlas: already enforced (TLS 1.2+ mandatory)
    → Self-managed: configure TLS certificates
    → RESULT: all network traffic encrypted

LAYER 3 — FIELD-LEVEL (MongoDB UNIQUE):
    → Choose encryption approach:
      a) CSFLE (Client-Side Field Level Encryption) — MongoDB 4.2+
         → Deterministic: same input → same ciphertext (queryable with equality)
         → Random: same input → different ciphertext each time (more secure)
      b) Queryable Encryption — MongoDB 7.0+
         → Next generation: supports equality AND range queries on encrypted data
         → More secure than deterministic CSFLE
    → Define encryption schema (which fields, which keys, which algorithm)
    → Configure key management (AWS KMS / Azure KV / local)
    → Application encrypts BEFORE sending to MongoDB
    → MongoDB stores ciphertext
    → Application decrypts AFTER receiving from MongoDB
    → Server NEVER sees plaintext

Layer 3: Client-Side Field Level Encryption (THE GAME-CHANGER)

# ═══════════════════════════════════════════════════════
# LAYER 3: CLIENT-SIDE FIELD LEVEL ENCRYPTION (CSFLE)
#
# THIS IS WHAT REDSHIFT CANNOT DO
#
# HOW CSFLE WORKS:
#
#   ┌──────────────┐                    ┌──────────────┐
#   │ APPLICATION   │                    │ MONGODB       │
#   │               │                    │ SERVER        │
#   │ ssn = "123"   │                    │               │
#   │      │        │                    │               │
#   │      ▼        │                    │               │
#   │ ENCRYPT(ssn)  │    encrypted       │               │
#   │ = "Zk3x7..."  │ ──────────────────►│ Stores:       │
#   │               │                    │ ssn: "Zk3x7.."│
#   │               │    encrypted       │               │
#   │ DECRYPT(resp) │ ◄──────────────────│ Returns:      │
#   │ = "123"       │                    │ ssn: "Zk3x7.."│
#   │               │                    │               │
#   └──────────────┘                    └──────────────┘
#
#   KEY POINT:
#   → MongoDB server NEVER sees "[REDACTED:SSN]"
#   → Server only sees "Zk3x7..." (ciphertext)
#   → Encryption/decryption happens in the CLIENT (pymongo driver)
#   → Even if server is compromised, encrypted fields are safe
#   → Even MongoDB Atlas engineers cannot read encrypted fields
#   → THIS is "zero-knowledge" encryption
#
# TWO ENCRYPTION ALGORITHMS:
#
#   DETERMINISTIC:
#   → Same plaintext → same ciphertext (always)
#   → Allows equality queries: find({ ssn: "[REDACTED:SSN]" })
#   → Driver encrypts the query value too → server matches ciphertext
#   → Trade-off: attacker can detect duplicate values
#   → Use for: fields you need to query (SSN, email, patient_id)
#
#   RANDOM:
#   → Same plaintext → different ciphertext (each time)
#   → CANNOT query on this field (ciphertext always different)
#   → More secure: no duplicate detection possible
#   → Use for: fields you don't query (credit card, diagnosis, notes)
#
# ═══════════════════════════════════════════════════════

# pip install pymongo pymongocrypt

from pymongo import MongoClient
from pymongo.encryption_options import AutoEncryptionOpts
from pymongo.encryption import ClientEncryption
from bson.codec_options import CodecOptions
from bson.binary import STANDARD, UUID_SUBTYPE
from bson import Binary
import os
from datetime import datetime

# ─── KEY MANAGEMENT SETUP ─────────────────────────────
#
# CSFLE uses a TWO-TIER key system:
#
# TIER 1: Customer Master Key (CMK)
#   → Lives in AWS KMS (or Azure KV, GCP KMS, or local file)
#   → NEVER leaves the KMS service
#   → Used to encrypt/decrypt the Data Encryption Keys
#   → You control this key → you control all encryption
#
# TIER 2: Data Encryption Keys (DEK)
#   → Created by MongoDB driver
#   → Stored in MongoDB (encrypted by CMK)
#   → Each DEK can encrypt different fields
#   → Multiple DEKs = different access for different teams
#
# KEY HIERARCHY:
#   AWS KMS (CMK: "ssn-master-key")
#       └── encrypts → DEK for SSN fields
#       └── encrypts → DEK for credit card fields
#   AWS KMS (CMK: "medical-master-key")
#       └── encrypts → DEK for diagnosis fields
#
# DIFFERENT CMKs = DIFFERENT TEAMS CAN DECRYPT DIFFERENT FIELDS
# → Payment team has access to credit card CMK
# → Medical team has access to diagnosis CMK
# → Neither can decrypt the other's fields
# → THIS IS IMPOSSIBLE IN REDSHIFT

# ─── OPTION A: LOCAL MASTER KEY (for development/testing) ──

# Generate a 96-byte local master key (DEVELOPMENT ONLY)
# In production: ALWAYS use AWS KMS / Azure KV / GCP KMS
local_master_key = os.urandom(96)

# Save for reuse (in real app: store in secrets manager)
LOCAL_KEY_PATH = "/tmp/mongodb_local_master_key"
with open(LOCAL_KEY_PATH, "wb") as f:
    f.write(local_master_key)

# KMS provider configuration
kms_providers_local = {
    "local": {
        "key": local_master_key
    }
}

# ─── OPTION B: AWS KMS MASTER KEY (for production) ────

kms_providers_aws = {
    "aws": {
        "accessKeyId": os.environ.get("AWS_ACCESS_KEY_ID"),
        "secretAccessKey": os.environ.get("AWS_SECRET_ACCESS_KEY")
    }
}

# AWS KMS master key specification
aws_master_key = {
    "provider": "aws",
    "key": "arn:aws:kms:us-east-1:123456789012:key/abcd-1234-efgh-5678",
    "region": "us-east-1"
}

# For this demo, we'll use LOCAL key (simpler setup)
# In production: replace with AWS KMS
kms_providers = kms_providers_local

# ─── STEP 1: CREATE DATA ENCRYPTION KEYS ─────────────

# Connection to the key vault database
# Key vault = MongoDB collection that stores encrypted DEKs
KEY_VAULT_NAMESPACE = "encryption.__keyVault"
KEY_VAULT_DB = "encryption"
KEY_VAULT_COLL = "__keyVault"

# Connect to MongoDB (regular connection for key management)
key_vault_client = MongoClient(
    "mongodb+srv://admin:AdminP@ss@cluster0.xxxxx.mongodb.net/"
)
key_vault_db = key_vault_client[KEY_VAULT_DB]
key_vault_coll = key_vault_db[KEY_VAULT_COLL]

# Create unique index on key vault (required)
key_vault_coll.create_index(
    "keyAltNames",
    unique=True,
    partialFilterExpression={"keyAltNames": {"$exists": True}}
)

# Create ClientEncryption instance for key management
client_encryption = ClientEncryption(
    kms_providers=kms_providers,
    key_vault_namespace=KEY_VAULT_NAMESPACE,
    key_vault_client=key_vault_client,
    codec_options=CodecOptions(uuid_representation=STANDARD)
)

# Create Data Encryption Keys (one per data category)

# DEK for PII (SSN, email) — queryable fields
pii_key_id = client_encryption.create_data_key(
    "local",                    # kms provider ("aws" in production)
    key_alt_names=["pii_key"],  # human-readable name
    # For AWS KMS: master_key=aws_master_key
)
print(f"PII Data Key ID: {pii_key_id}")

# DEK for Payment data (credit card numbers)
payment_key_id = client_encryption.create_data_key(
    "local",
    key_alt_names=["payment_key"]
)
print(f"Payment Data Key ID: {payment_key_id}")

# DEK for Medical data (diagnosis codes, prescriptions)
medical_key_id = client_encryption.create_data_key(
    "local",
    key_alt_names=["medical_key"]
)
print(f"Medical Data Key ID: {medical_key_id}")

# NOW: three DEKs exist in encryption.__keyVault collection
# Each DEK is encrypted by the master key (local or AWS KMS)
# Each DEK will encrypt different fields in our documents

# VERIFY: What the key vault looks like
for key_doc in key_vault_coll.find():
    print(f"  Key: {key_doc.get('keyAltNames', ['unnamed'])} "
          f"  ID: {key_doc['_id']} "
          f"  Status: {key_doc.get('status', 'active')}")


# ─── STEP 2: DEFINE ENCRYPTION SCHEMA ────────────────
#
# The encryption schema tells the driver:
# → WHICH fields to encrypt
# → WHICH key to use for each field
# → WHICH algorithm (deterministic or random)
#
# THIS IS THE CORE CONFIGURATION
# Think of it as a "security blueprint" for your documents

encryption_schema = {
    "shopstream.patient_orders": {
        "bsonType": "object",
        "properties": {
            # ─── SSN: Deterministic (queryable) ──────
            # Can do: find({ ssn: "[REDACTED:SSN]" })
            # Driver encrypts "123-45-6789" → matches encrypted value in DB
            "patient_ssn": {
                "encrypt": {

⚡ WHAT COULD BE DONE (Encryption-Specific Optimizations)

OPTIMIZATION 1: QUERYABLE ENCRYPTION (MongoDB 7.0+)
    → Next-generation of CSFLE
    → Supports equality AND range queries on encrypted fields
    → More secure than deterministic CSFLE
    → Server NEVER sees plaintext and NEVER sees query patterns
    → Use: find({ age: { $gt: 18, $lt: 65 } }) on encrypted age field!
    → CSFLE deterministic: only equality queries
    → Queryable Encryption: equality + range + more
    → Trade-off: higher storage overhead (encrypted indexes)

OPTIMIZATION 2: AUTOMATIC ENCRYPTION WITH JSON SCHEMA VALIDATION
    → Combine encryption schema with validation schema
    → If someone tries to insert SSN without encryption → rejected
    → Defense in depth: even if client misconfigured, server rejects

OPTIMIZATION 3: KEY ROTATION
    → Periodically rotate Data Encryption Keys
    → ClientEncryption.rewrap_many_data_key() → re-encrypts DEKs with new CMK
    → Documents don't need to be re-encrypted (only DEK wrapper changes)
    → Automatic via Atlas Encryption Key Management
    → Redshift: KMS key rotation is cluster-level (simpler but less granular)

OPTIMIZATION 4: PER-CUSTOMER ENCRYPTION KEYS
    → Instead of per-field-category keys
    → Create a DEK per customer
    → Each customer's data encrypted with their own key
    → GDPR crypto-shredding: delete one customer's key
    → Other customers unaffected
    → Trade-off: many keys to manage (thousands)

OPTIMIZATION 5: CRYPT_SHARED LIBRARY
    → mongocryptd (encryption daemon) replaced by crypt_shared lib
    → Linked directly into pymongo driver
    → No separate process needed
    → Better performance: in-process encryption
    → Configure: crypt_shared_lib_path in AutoEncryptionOpts

✅ WHAT SHOULD DO (Best Practices)

✅ USE AWS KMS (NOT LOCAL KEY) IN PRODUCTION
    → Local key is for development/testing ONLY
    → AWS KMS: key never leaves HSM hardware
    → AWS KMS: automatic key rotation
    → AWS KMS: audit trail via CloudTrail
    → AWS KMS: IAM policies control key access
    → Same KMS you already use for Redshift encryption

✅ ENCRYPT ONLY WHAT NEEDS ENCRYPTION
    → DON'T encrypt every field (massive performance hit)
    → Encrypt: SSN, credit card, diagnosis, email, phone, address
    → DON'T encrypt: order_number, status, dates, amounts
    → Rule: encrypt PII and sensitive data; leave operational fields plain
    → Each encrypted field adds ~20-30% overhead for that field

✅ USE DETERMINISTIC ONLY WHEN YOU NEED TO QUERY
    → Deterministic: queryable but less secure (duplicate detection possible)
    → Random: not queryable but more secure
    → SSN lookup: deterministic (need to find patient by SSN)
    → Credit card storage: random (never search by full card number)
    → Choose per-field based on access patterns

✅ SEPARATE KEYS BY DATA CATEGORY
    → PII key, payment key, medical key (as shown)
    → If one key is compromised, other data categories remain safe
    → Maps to organizational responsibility: payment team, medical team
    → Redshift habit to UNLEARN: one key for everything

✅ IMPLEMENT KEY ROTATION SCHEDULE
    → CMK rotation: annually (AWS KMS can auto-rotate)
    → DEK re-wrapping: quarterly
    → Document the rotation procedure and test it
    → Same discipline as Redshift KMS key rotation

✅ TEST ENCRYPTION END-TO-END BEFORE PRODUCTION
    → Insert with encrypted client → verify ciphertext in raw read
    → Read with encrypted client → verify plaintext returned
    → Read with unencrypted client → verify ciphertext visible
    → Query on deterministic field → verify match works
    → Query on random field → verify it correctly fails/returns nothing

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT STORE MASTER KEY IN APPLICATION CODE
    WRONG: local_master_key = b'\x01\x02\x03...' in source code
    WHY: anyone with code access has the master key
    ✅ FIX: AWS KMS, Azure Key Vault, GCP KMS (key never leaves HSM)

❌ DO NOT ENCRYPT FIELDS YOU NEED TO SORT OR RANGE-QUERY (with CSFLE)
    WRONG: encrypting "created_at" with deterministic encryption
    → Can do: find({ created_at: exact_date }) — but USELESS
    → Cannot do: find({ created_at: { $gt: date } }) — range query fails
    WHY: deterministic CSFLE only supports equality
    ✅ FIX: 
    → Don't encrypt operational fields (dates, status, amounts)
    → OR: use Queryable Encryption (MongoDB 7.0+) for range queries

❌ DO NOT USE DETERMINISTIC ENCRYPTION FOR LOW-CARDINALITY FIELDS
    WRONG: encrypting "gender" (M/F/Other) with deterministic
    WHY: only 3 possible ciphertexts → attacker easily maps them
    ✅ FIX: use random encryption for low-cardinality fields
    → Or: don't encrypt gender (is it truly sensitive for your use case?)

❌ DO NOT ENCRYPT THE SAME FIELD WITH DIFFERENT KEYS IN DIFFERENT DOCUMENTS
    WRONG: document 1 uses key_A for SSN, document 2 uses key_B for SSN
    WHY: deterministic queries won't work (different keys → different ciphertext)
    ✅ FIX: same field → same key (consistency required for queries)

❌ DO NOT SKIP ENCRYPTION BECAUSE "WE HAVE RBAC"
    WRONG: "Project 37 RBAC is enough — we don't need encryption"
    WHY: RBAC controls WHO can connect, encryption controls WHAT they see
    → Compromised admin account bypasses RBAC
    → Compromised database server bypasses RBAC
    → CSFLE protects even when RBAC fails (defense in depth)
    ✅ FIX: RBAC (Project 37) + Encryption (Project 38) = defense in depth

❌ DO NOT ENCRYPT FIELDS IN REDSHIFT PIPELINE WITHOUT HANDLING
    WRONG: CDC pipeline (Project 34) sends encrypted fields to Redshift as-is
    → Redshift receives Binary ciphertext → stores as VARBYTE → useless for analytics
    WHY: encrypted fields are meaningless in Redshift
    ✅ FIX: 
    → Option A: pipeline decrypts before sending to Redshift (has keys)
    → Option B: exclude encrypted fields from pipeline (don't send to Redshift)
    → Option C: send tokenized version (hash of SSN, not actual SSN)
    → Choose based on: does Redshift NEED the plaintext for analytics?

⚙️ HOW IT ONLY WORKS INTERNALLY

CSFLE ENCRYPTION FLOW — STEP BY STEP:

INSERT OPERATION:
    
    APPLICATION CODE:
    patient_orders.insert_one({
        "patient_ssn": "[REDACTED:SSN]",
        "status": "confirmed"
    })
        │
        ▼
    PYMONGO DRIVER (with auto-encryption):
    1. Reads encryption schema
    2. Identifies: patient_ssn needs encryption
    3. Fetches DEK from key vault (encryption.__keyVault)
    4. DEK is stored ENCRYPTED → decrypts DEK using CMK (KMS call)
    5. Encrypts "[REDACTED:SSN]" using DEK:
       → Deterministic: HMAC-SHA-512 for consistent ciphertext
       → Prepends metadata: key ID + algorithm marker
       → Result: Binary(0x06, subtype 6, "Zk3x7...encrypted...")
    6. Replaces plaintext with ciphertext in document
    7. Sends to server: { patient_ssn: Binary("Zk3x7..."), status: "confirmed" }
        │
        ▼
    MONGODB SERVER:
    → Receives document with Binary field (ciphertext)
    → Stores as-is in WiredTiger
    → Server NEVER saw "[REDACTED:SSN]"
    → Server CANNOT decrypt (doesn't have CMK)
    → Index on patient_ssn indexes the CIPHERTEXT (deterministic = consistent)

QUERY OPERATION:

    APPLICATION CODE:
    patient_orders.find_one({ "patient_ssn": "[REDACTED:SSN]" })
        │
        ▼
    PYMONGO DRIVER:
    1. Identifies: patient_ssn is encrypted (deterministic)
    2. Encrypts query value: "[REDACTED:SSN]" → Binary("Zk3x7...")
    3. Sends to server: find({ patient_ssn: Binary("Zk3x7...") })
        │
        ▼
    MONGODB SERVER:
    → Matches Binary("Zk3x7...") against stored ciphertext
    → Deterministic: same plaintext → same ciphertext → MATCH
    → Returns document with encrypted fields
        │
        ▼
    PYMONGO DRIVER:
    1. Receives document with Binary fields
    2. Identifies encrypted fields by Binary subtype 6
    3. Extracts key ID from ciphertext metadata
    4. Fetches DEK → decrypts DEK using CMK (KMS call)
    5. Decrypts each encrypted field
    6. Returns document with plaintext to application

    APPLICATION CODE:
    → Receives: { patient_ssn: "[REDACTED:SSN]", status: "confirmed" }
    → Transparent! Application doesn't know encryption happened

SECURITY BOUNDARY:

    ┌──────────────────────────────────────────────────────┐
    │  TRUST BOUNDARY                                       │
    │                                                        │
    │  TRUSTED (sees plaintext):                             │

    │  → Application memory (briefly, during encrypt/decrypt)│

    └──────────────────────────────────────────────────────┘

    REDSHIFT COMPARISON:
    ┌──────────────────────────────────────────────────────┐
    │  REDSHIFT TRUST BOUNDARY                              │
    │                                                        │
    │  TRUSTED (sees plaintext):                             │

NEW ENCRYPTION TERMS IN THIS PROJECT:

    CSFLE (Client-Side Field Level Encryption):
        → Encryption performed by the MongoDB driver (client)
        → Server never sees plaintext
        → Available since MongoDB 4.2
        → Supports deterministic and random algorithms
        → Redshift has NO equivalent

    Queryable Encryption:
        → Next-generation CSFLE (MongoDB 7.0+)
        → Supports equality AND range queries on encrypted fields
        → More secure than deterministic CSFLE
        → Server learns nothing about plaintext or query patterns
        → Uses encrypted indexes for query matching
        → Redshift has NO equivalent

    Customer Master Key (CMK):
        → Top-level key stored in external KMS (AWS KMS, Azure KV, GCP)
        → NEVER leaves the KMS service (hardware security module)
        → Used to encrypt/decrypt Data Encryption Keys
        → Equivalent to Redshift's KMS master key BUT can have MULTIPLE

    Data Encryption Key (DEK):
        → Per-category key that actually encrypts document fields
        → Stored in MongoDB (encrypted by CMK)
        → Multiple DEKs = different access for different teams
        → Redshift has NO equivalent (single cluster key)

    Key Vault:
        → MongoDB collection (encryption.__keyVault) storing encrypted DEKs
        → Accessible only to authorized clients
        → Like a KMS key store but inside MongoDB
        → Redshift stores keys externally only (KMS)

    Deterministic Encryption:
        → Same plaintext → same ciphertext (always)
        → Enables equality queries on encrypted fields
        → Trade-off: attacker can detect duplicate values
        → Algorithm: AEAD_AES_256_CBC_HMAC_SHA_512-Deterministic

    Random Encryption:
        → Same plaintext → different ciphertext (each time)
        → Cannot query on these fields
        → More secure: no pattern leakage
        → Algorithm: AEAD_AES_256_CBC_HMAC_SHA_512-Random

    Crypto-Shredding:
        → Delete encryption key → all data encrypted with it becomes unreadable
        → Satisfies GDPR Right to Erasure without touching documents
        → Works even on backups and replicas (key is gone everywhere)
        → Redshift has NO equivalent approach

    Binary Subtype 6:
        → BSON binary subtype indicating encrypted data
        → Driver recognizes this subtype → triggers decryption
        → Unencrypted client sees Binary(subtype=6) → knows it's encrypted
        → Contains metadata: key ID, algorithm, ciphertext

    Trust Boundary:
        → The line between components that see plaintext vs ciphertext
        → CSFLE: trust boundary is the APPLICATION (narrow = secure)
        → Redshift: trust boundary is the CLUSTER (wide = less secure)
        → Narrower trust boundary = smaller attack surface

PITFALL 5: "PERFORMANCE DEGRADED AFTER ENABLING CSFLE"

    → Insert latency: 2ms → 45ms (22x slower)
    → Read latency: 1ms → 30ms (30x slower)

      - KMS call to decrypt DEK (network round-trip to AWS KMS)

    → KMS call latency: ~5-20ms each

    → Encrypt ONLY truly sensitive fields (3-5 fields, not 15)
    → DEK caching: driver caches decrypted DEKs for 60 seconds

    → KMS proximity: KMS in same region as application
    → Batch operations: insertMany with encryption amortizes KMS calls

    WHY:
    → Queryable Encryption creates encrypted indexes
    → Each encrypted index stores encrypted tokens for query matching
    → More encrypted queryable fields = more encrypted index data
    
    ✅ FIX:
    → Only use Queryable Encryption on fields you ACTUALLY query
    → Use random encryption for fields you never query (no index overhead)

KEY PERFORMANCE INSIGHTS:

1. DEK CACHING IS CRITICAL
    → First operation: ~20ms (KMS round-trip to decrypt DEK)
    → Subsequent operations: ~3-5ms (DEK cached in driver memory)
    → Cache TTL: configurable (default 60 seconds)
    → Implication: sustained workloads barely feel the overhead
    → Cold starts (Lambda, new connections) pay the full KMS cost

2. KMS REGION MATTERS
    → KMS in same region as application: ~5ms round-trip
    → KMS in different region: ~50-100ms round-trip
    → ALWAYS co-locate KMS with application servers
    → Same principle as co-locating Redshift with S3

3. NUMBER OF ENCRYPTED FIELDS MATTERS
    → 1-3 encrypted fields: ~5-15ms overhead (acceptable)
    → 5-10 encrypted fields: ~15-30ms overhead (noticeable)
    → 15+ encrypted fields: ~30-60ms overhead (may violate SLAs)
    → Rule of thumb: encrypt ≤5 fields for operational workloads

4. QUERYABLE ENCRYPTION STORAGE OVERHEAD
    → Per encrypted queryable field: ~2x storage for that field
    → Collection with 3 queryable encrypted fields: ~30-50% total storage increase
    → Budget accordingly for Atlas tier sizing

SCALING ENCRYPTION ACROSS THE ARCHITECTURE:

    MongoDB (operational):
    → CSFLE on 3-5 sensitive fields per collection
    → Operational latency: 5-15ms (within SLA)
    → DEK caching handles sustained throughput
    
    Redshift (analytical):
    → KMS encryption at rest (automatic, no overhead)
    → No field-level encryption (not needed — RBAC sufficient)
    → Encrypted fields from MongoDB excluded from pipeline
    → Or: pipeline decrypts before Redshift (if analytics needs PII)
    
    S3 (data lake):
    → Server-side encryption (SSE-S3 or SSE-KMS)
    → If $out from MongoDB includes encrypted fields → stays encrypted
    → If pipeline writes decrypted data → S3 encryption protects at rest
    
    Pipeline:
    → Choose: decrypt in pipeline OR exclude encrypted fields
    → If decrypting: pipeline service needs KMS access (IAM role)
    → If excluding: simplest and most secure approach

COMPARISON WITH REDSHIFT ENCRYPTION SCALING:
    → Redshift: <5% overhead (hardware AES, no per-field cost)
    → MongoDB CSFLE: 5-30% overhead (per-field, KMS calls)
    → Redshift: simpler (all-or-nothing)
    → MongoDB: more complex but MUCH more granular
    → Trade-off: simplicity vs security granularity

📐 DATA MODELING COMPARISON — Encryption Impact

ENCRYPTION CHANGES HOW YOU MODEL DATA:

WITHOUT CSFLE (normal MongoDB — Projects 1-37):
    {
        patient_ssn: "[REDACTED:SSN]",           ← queryable, indexable, plain text
        patient_email: "john@example.com",   ← queryable, indexable, plain text
        credit_card: "[REDACTED:CREDIT_DEBIT_NUMBER]",         ← queryable (but shouldn't be)
        diagnosis_code: "I10",               ← queryable, aggregatable
        status: "confirmed",                 ← queryable
        total: 89.99                         ← aggregatable
    }
    
    → ALL fields queryable and indexable
    → ANY authenticated user sees everything
    → Security relies entirely on RBAC (Project 37)

WITH CSFLE (Project 38):
    {
        patient_ssn: Binary(subtype 6),      ← deterministic: equality query ONLY
        patient_email: Binary(subtype 6),    ← deterministic: equality query ONLY
        credit_card: Binary(subtype 6),      ← random: NOT queryable at all
        diagnosis_code: Binary(subtype 6),   ← random: NOT queryable at all
        status: "confirmed",                 ← plain text: fully queryable
        total: 89.99                         ← plain text: fully aggregatable
    }
    
    → Encrypted fields: limited queryability
    → Deterministic: equality only (find by exact SSN)
    → Random: no queries (store and retrieve only)
    → Plain fields: unchanged (full queryability)
    → Security: RBAC + encryption (defense in depth)

DESIGN IMPLICATIONS:

1. CHOOSE CAREFULLY WHAT TO ENCRYPT
    → Encrypt: data that causes harm if exposed (PII, PHI, PCI)
    → Don't encrypt: data needed for analytics, sorting, ranging
    → Gray area: email (query by email? → deterministic. Never query? → random)

2. INDEX STRATEGY CHANGES
    → Can't create normal index on random-encrypted fields
    → Deterministic fields: index works on ciphertext (equality only)
    → Queryable Encryption: creates encrypted indexes (more flexible)
    → Unencrypted fields: indexes work normally (Projects 9-11)

3. AGGREGATION STRATEGY CHANGES
    → $group on encrypted fields: only with authorized client
    → $sort on encrypted fields: NOT possible (ciphertext sort ≠ plaintext sort)
    → $match on deterministic fields: equality only
    → $match on random fields: NOT possible
    → For analytics: aggregate on unencrypted fields (status, date, amount)

4. EMBEDDING DECISIONS AFFECTED (Project 14)
    → Encrypted fields inside embedded objects: works fine
    → Encrypted fields inside arrays: supported but complex
    → Each array element's encrypted field = separate encryption
    → Example: items[].patient_notes encrypted → each item's notes encrypted separately

REDSHIFT COMPARISON:
    → Redshift encryption has ZERO impact on data modeling
    → All queries, indexes, aggregations work identically
    → Because: encryption is at storage level (transparent to SQL)
    → MongoDB CSFLE: encryption impacts query capabilities
    → This is the TRADE-OFF: more security = some query restrictions
    → Worth it for truly sensitive data (healthcare, finance, government)

# ─── APPROACH C: PIPELINE SENDS TOKENIZED VALUES ─────
# Pipeline creates non-reversible tokens for encrypted fields
# Redshift can aggregate by token without seeing plaintext

def pipeline_with_tokenization():
    """
    CDC pipeline that TOKENIZES encrypted fields
    
    WHEN TO USE:
    → Need to GROUP BY patient in Redshift without revealing SSN
    → Need to COUNT DISTINCT patients without seeing PII
    → Token is consistent (same SSN → same token) but not reversible
    """
    
    import hashlib
    
    TOKENIZATION_SALT = "shopstream_analytics_2024"  # secret salt
    
    def tokenize(value):
        """Create non-reversible token from sensitive value"""
        if value is None:
            return None
        salted = f"{TOKENIZATION_SALT}:{value}".encode()
        return hashlib.sha256(salted).hexdigest()[:16]
    
    # Regular client (encrypted fields are ciphertext)
    client = MongoClient(
        "mongodb+srv://etl_service:pass@cluster0.xxxxx.mongodb.net/"
    )
    
    # BUT: we need an ENCRYPTED client to read and tokenize
    # Actually: we need decryption to tokenize
    # So this approach ALSO requires keys (like Approach A)
    # BUT: tokens are one-way (can't reverse to get SSN)
    # Redshift sees tokens, not plaintext
