PROJECT 19: Subset Pattern — Reducing Working Set

Problem: Product Page Loads 50,000 Reviews but Only Shows 10

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform has matured
    Popular products now have 10,000 to 50,000 reviews each
    
    THE PRODUCT PAGE needs:
        → Product details (name, price, description, images)
        → Average rating + rating distribution (5-star breakdown)
        → 10 MOST RECENT reviews (shown by default)
        → "Load more" button for older reviews (paginated)
    
    THE ANALYTICS DASHBOARD needs:
        → All reviews for sentiment analysis
        → Review trends over time
        → Reviewer demographics
    
    CURRENT STATE (naive design from early days):
        → Each product document EMBEDS all 50,000 reviews
        → Product document size: 15-25 MB each
        → 100,000 products × 25 MB = 2.5 TB of product documents
        → Application is SLOW — pages take 3-8 seconds to load
        → MongoDB server RAM usage is critical — constant cache evictions
    
    YOU MUST FIX THIS
    
    Your Redshift brain says: "Just query with LIMIT 10 — what's the problem?"
    MongoDB says: "The ENTIRE document loads into memory — even for 10 reviews"
    
    THIS is the problem the Subset Pattern solves

🔴 REDSHIFT MINDSET — How You Would Solve This

The Normalized Approach (Redshift's Natural Design)

-- REDSHIFT: This problem barely exists because of normalization

-- Table 1: Products (small rows, fast scans)
CREATE TABLE store.products (
    product_id      BIGINT IDENTITY(1,1) PRIMARY KEY,
    name            VARCHAR(500) NOT NULL,
    price           DECIMAL(10,2),
    description     VARCHAR(10000),
    avg_rating      DECIMAL(3,2),
    review_count    INTEGER DEFAULT 0,
    category        VARCHAR(100),
    created_at      TIMESTAMP DEFAULT GETDATE()
)
DISTSTYLE KEY DISTKEY(product_id)
SORTKEY(product_id);

-- Table 2: Reviews (separate table, millions of rows)
CREATE TABLE store.reviews (
    review_id       BIGINT IDENTITY(1,1) PRIMARY KEY,
    product_id      BIGINT NOT NULL,        -- FK to products
    user_id         BIGINT NOT NULL,
    rating          SMALLINT NOT NULL,       -- 1-5
    title           VARCHAR(500),
    body            VARCHAR(10000),
    helpful_votes   INTEGER DEFAULT 0,
    verified        BOOLEAN DEFAULT false,
    created_at      TIMESTAMP DEFAULT GETDATE()
)
DISTSTYLE KEY DISTKEY(product_id)    -- co-locate with products for JOINs
SORTKEY(product_id, created_at);     -- compound: filter by product, sort by date

-- Query: Product page (product + 10 recent reviews)
SELECT p.*, r.rating, r.title, r.body, r.created_at as review_date
FROM store.products p
LEFT JOIN store.reviews r ON p.product_id = r.product_id
WHERE p.product_id = 12345
ORDER BY r.created_at DESC
LIMIT 10;

-- Query: All reviews for analytics
SELECT product_id, rating, body, created_at
FROM store.reviews
WHERE product_id = 12345
ORDER BY created_at DESC;
-- No LIMIT — full scan of product's reviews
-- Redshift handles this efficiently: columnar scan + sort key

Why This "Just Works" in Redshift

REDSHIFT'S ADVANTAGES HERE:

1. SEPARATION BY DEFAULT
   → Products table: small rows (~500 bytes each)
   → Reviews table: separate, only loaded when queried
   → LIMIT 10 on reviews = Redshift reads only 10 rows
     (sort key on created_at means it reads from the END)

2. COLUMNAR EFFICIENCY
   → "Show 10 reviews" only reads: rating, title, body, created_at columns
   → Other columns (helpful_votes, verified, user_id) NOT read
   → Redshift skips unneeded columns entirely

3. ZONE MAPS + SORT KEY
   → SORTKEY(product_id, created_at) means:
     → Zone maps know which blocks contain product_id = 12345
     → Within those blocks, data is sorted by created_at
     → Getting "10 most recent" = read last few blocks only

4. NO MEMORY PRESSURE FROM DOCUMENT SIZE
   → Each row is independent — no "loading the whole product"
   → Query optimizer decides exactly which rows to fetch
   → Memory used = only what the query needs

REDSHIFT DEVELOPER'S CONCLUSION:
    "Normalization + LIMIT + sort key = problem solved"
    "Why would MongoDB struggle with this?"

🟢 MONGODB MINDSET — Why This Problem EXISTS in MongoDB

The Root Cause: Documents Load as Whole Units

THE FUNDAMENTAL ISSUE:

REDSHIFT (Columnar):
    → SELECT rating, title FROM reviews WHERE product_id = 12345 LIMIT 10
    → Reads ONLY rating + title columns
    → Reads ONLY 10 rows
    → Memory used: ~5 KB
    
MONGODB (Document):
    → db.products.findOne({ _id: 12345 })
    → Reads ENTIRE document including ALL 50,000 embedded reviews
    → BSON document loaded into WiredTiger cache as ONE unit
    → Memory used: 15-25 MB per product lookup
    → Even with projection { reviews: { $slice: -10 } }
      the FULL document must be read from disk first
      then $slice is applied IN MEMORY

    THIS IS THE CRITICAL INSIGHT:
    ┌──────────────────────────────────────────────────────┐
    │ MongoDB reads DOCUMENTS, not FIELDS                   │
    │ Projection filters AFTER loading, not BEFORE          │
    │ (Unlike Redshift where column pruning happens at I/O) │
    │                                                        │
    │ A 25 MB document costs 25 MB of cache space            │
    │ even if you only need 5 KB from it                     │
    └──────────────────────────────────────────────────────┘

The Working Set Problem — Why This Kills Performance

WIREDTIGER CACHE — THE HEART OF THE PROBLEM:

WiredTiger cache = MongoDB's in-memory data store
    → Default size: 50% of (RAM - 1 GB)
    → Server with 16 GB RAM → ~7.5 GB WiredTiger cache
    → ALL frequently accessed documents should fit here
    → If they don't → CACHE EVICTIONS → DISK READS → SLOW

SCENARIO WITH NAIVE DESIGN:
    100,000 products × 20 MB average = 2 TB total data
    Cache size: 7.5 GB
    
    Products accessed per hour: ~10,000 unique products
    Cache needed: 10,000 × 20 MB = 200 GB
    Cache available: 7.5 GB
    
    RESULT: constant eviction → every product page = disk read
    LATENCY: 50-200ms per page (disk) instead of 1-5ms (cache)

SCENARIO WITH SUBSET PATTERN:
    Product documents (without bulk reviews): ~2 KB each
    Products accessed per hour: ~10,000
    Cache needed: 10,000 × 2 KB = 20 MB
    Cache available: 7.5 GB
    
    RESULT: ALL hot products fit in cache → zero disk reads
    LATENCY: 1-5ms per page (cache hit)

    │ NAIVE:   10,000 products × 20 MB = 200 GB needed    │

    │ THAT'S A 10,000x REDUCTION IN CACHE PRESSURE          │

🧩 HOW TO APPROACH IT IN MONGODB

The Subset Pattern — Core Concept

PATTERN DEFINITION:
    Split frequently accessed data (HOT) from rarely accessed data (COLD)
    EMBED the hot subset in the main document
    STORE the full dataset in a separate collection
    
    ┌─────────────────────────────────────────────────────────────┐
    │                                                              │
    │   BEFORE (Naive Embedding):                                  │
    │   ┌──────────────────────────────────────────┐               │
    │   │ Product Document (20 MB)                  │               │
    │   │ ├── name, price, description              │               │
    │   │ └── reviews: [50,000 review objects]      │  ← ALL here  │
    │   └──────────────────────────────────────────┘               │
    │                                                              │
    │   AFTER (Subset Pattern):                                    │
    │   ┌──────────────────────────┐  ┌──────────────────────────┐ │
    │   │ Product Document (2 KB)   │  │ Reviews Collection        │ │
    │   │ ├── name, price, desc     │  │ ├── review for prod_123   │ │
    │   │ ├── avg_rating            │  │ ├── review for prod_123   │ │
    │   │ ├── review_count: 50000   │  │ ├── review for prod_456   │ │
    │   │ └── recent_reviews: [10]  │  │ ├── ... 50,000 reviews    │ │
    │   │     (HOT subset only)     │  │ └── (FULL archive)        │ │
    │   └──────────────────────────┘  └──────────────────────────┘ │
    │        ↑ Fits in cache              ↑ Queried on demand      │
    │        ↑ Single read for page       ↑ Paginated access       │
    │                                                              │
    └─────────────────────────────────────────────────────────────┘

ANALOGY FOR REDSHIFT DEVELOPERS:
    Think of it as a MATERIALIZED VIEW embedded inside the document
    → The product document IS the materialized "product page" view
    → The reviews collection IS the full fact table
    → You pre-compute what the application needs most frequently
    → But the full data is always available for deeper queries

Decision Framework: What Goes in the Subset?

ASK THESE QUESTIONS:

1. WHAT DOES THE APPLICATION SHOW FIRST?
   → Product page: 10 most recent reviews
   → EMBED: 10 most recent reviews
   
2. WHAT DOES THE USER REQUEST ON DEMAND?
   → "Load more reviews" button → paginated full list
   → SEPARATE COLLECTION: all reviews
   
3. HOW OFTEN DOES THE SUBSET CHANGE?
   → New review arrives → update top 10 in product doc
   → Frequency: maybe 1-10 new reviews per day per product
   → Acceptable write overhead for massive read benefit
   
4. WHAT IS THE SIZE IMPACT?
   → 10 embedded reviews × ~200 bytes = ~2 KB
   → 50,000 embedded reviews × ~200 bytes = ~10 MB
   → 5000x size reduction in the product document

SUBSET SELECTION CRITERIA:
┌────────────────────────────────────┬──────────────────────────┐
│ EMBED IN MAIN DOCUMENT (HOT)       │ SEPARATE COLLECTION (COLD)│
├────────────────────────────────────┼──────────────────────────┤
│ 10 most recent reviews              │ All 50,000 reviews        │
│ Rating distribution (5-star counts) │ Full review text + metadata│
│ Average rating (pre-computed)       │ Reviewer profile links    │
│ Total review count                  │ Helpful vote history      │
│ Top 3 "most helpful" reviews        │ Review edit history       │
└────────────────────────────────────┴──────────────────────────┘

Step 2: Size Comparison — The Proof

DOCUMENT SIZE ANALYSIS:

BEFORE (Naive Embedding):
    Product document with 50,000 reviews embedded
    → Product fields: ~1 KB
    → 50,000 reviews × 400 bytes avg = ~20 MB
    → TOTAL: ~20 MB per product document
    
    10,000 active products in cache:
    → 10,000 × 20 MB = 200 GB needed
    → Cache available: 7.5 GB
    → Cache hit ratio: ~3.75%  ← TERRIBLE
    → Almost every read = DISK I/O

AFTER (Subset Pattern):
    Product document with 10 embedded reviews + stats
    → Product fields: ~1 KB
    → 10 recent reviews × 200 bytes = ~2 KB
    → 3 top helpful reviews × 200 bytes = ~0.6 KB
    → Review stats: ~0.2 KB
    → TOTAL: ~4 KB per product document
    
    10,000 active products in cache:
    → 10,000 × 4 KB = 40 MB needed
    → Cache available: 7.5 GB
    → Cache hit ratio: ~100%  ← PERFECT
    → Almost every read = CACHE HIT

// Generate 100 reviews for a product (simulating 50,000)
let reviews = []
let productId = "prod_12345"

for (let i = 0; i < 100; i++) {
    reviews.push({

        user_id: ObjectId(),
        user_name: `user_${String(i).padStart(4, '0')}`,
        rating: Math.floor(Math.random() * 5) + 1,
        title: `Review title ${i}`,
        body: `This is the detailed review body for review number ${i}. ` +
              `It contains the user's full thoughts about the product ` +
              `including pros, cons, and recommendations. `.repeat(3),
        date: new Date(2024, 0, 1 + Math.floor(i * 1.5)),  // spread across dates
        helpful_votes: Math.floor(Math.random() * 200),
        verified: Math.random() > 0.3,
        images: i % 5 === 0 ? [`review_img_${i}.jpg`] : [],
        edit_history: [],
        user_device: ["iPhone", "Android", "Desktop", "iPad"][Math.floor(Math.random() * 4)],
        purchase_date: new Date(2023, 6 + Math.floor(Math.random() * 6), 1)
    })
}

db.reviews.insertMany(reviews)
print(`Inserted ${db.reviews.countDocuments({ product_id: productId })} reviews`)

// ─── INDEX for reviews collection ────────────────────
// Compound index: find by product, sort by date (like Redshift SORTKEY)
db.reviews.createIndex(
    { product_id: 1, date: -1 },
    { name: "product_date_idx" }
)

// Index for "most helpful" queries
db.reviews.createIndex(
    { product_id: 1, helpful_votes: -1 },
    { name: "product_helpful_idx" }
)

print("Review indexes created")


// ═══════════════════════════════════════════════════════
// STEP 2: BUILD THE SUBSET — Compute What Goes in Product
// ═══════════════════════════════════════════════════════

// Get 10 most recent reviews (the HOT subset)
let recentReviews = db.reviews.find(
    { product_id: productId },
    {
        // Projection: only fields needed for product page display
        _id: 1,
        user_name: 1,
        rating: 1,
        title: 1,
        body: { $substrBytes: ["$body", 0, 200] },  // first 200 chars only
        date: 1,
        helpful_votes: 1,
        verified: 1
        // EXCLUDE: images, edit_history, user_device, purchase_date
        // Those are COLD data — only in reviews collection
    }
).sort({ date: -1 }).limit(10).toArray()

// Get top 3 most helpful reviews
let topHelpful = db.reviews.find(
    { product_id: productId }
).sort({ helpful_votes: -1 }).limit(3).toArray().map(r => ({
    review_id: r._id,
    user_name: r.user_name,
    rating: r.rating,
    title: r.title,
    body: r.body.substring(0, 200),
    date: r.date,
    helpful_votes: r.helpful_votes,
    verified: r.verified
}))

// Compute review statistics using aggregation (from Project 12)
let stats = db.reviews.aggregate([
    { $match: { product_id: productId } },
    { $group: {
        _id: null,
        count: { $sum: 1 },
        avg_rating: { $avg: "$rating" },
        rating_1: { $sum: { $cond: [{ $eq: ["$rating", 1] }, 1, 0] } },
        rating_2: { $sum: { $cond: [{ $eq: ["$rating", 2] }, 1, 0] } },
        rating_3: { $sum: { $cond: [{ $eq: ["$rating", 3] }, 1, 0] } },
        rating_4: { $sum: { $cond: [{ $eq: ["$rating", 4] }, 1, 0] } },
        rating_5: { $sum: { $cond: [{ $eq: ["$rating", 5] }, 1, 0] } }
    }},
    { $project: {
        _id: 0,
        count: 1,
        avg_rating: { $round: ["$avg_rating", 1] },
        distribution: {
            "1": "$rating_1",
            "2": "$rating_2",
            "3": "$rating_3",
            "4": "$rating_4",
            "5": "$rating_5"
        }
    }}
]).toArray()[0]

print("\nComputed review stats:")
printjson(stats)

def submit_review(product_id: str, review_data: dict):
    """
    Atomic-ish operation to add a review and update the subset.
    
    In Redshift: INSERT INTO reviews + UPDATE products (2 SQL statements)
    In MongoDB:  insertOne into reviews + updateOne on product (2 operations)
    Both require 2 operations — MongoDB's update is more expressive
    """
    
    # STEP A: Insert full review into reviews collection
    result = reviews.insert_one(review_data)
    review_id = result.inserted_id
    print(f"  Inserted review: {review_id}")
    
    # STEP B: Update product document's subset
    # $push with $each + $sort + $slice = bounded sorted array maintenance
    subset_entry = {
        "_id": review_id,
        "user_name": review_data["user_name"],
        "rating": review_data["rating"],
        "title": review_data["title"],
        "body_preview": review_data["body"][:150] + "...",
        "date": review_data["date"],
        "helpful_votes": review_data["helpful_votes"],
        "verified": review_data["verified"]
    }
    
    # Rating key for distribution update
    rating_key = f"review_stats.distribution.{review_data['rating']}"
    
    update_result = products.update_one(
        {"_id": product_id},
        {
            "$push": {
                "recent_reviews": {
                    "$each": [subset_entry],
                    "$sort": {"date": -1},     # newest first
                    "$slice": 10               # keep only 10
                }
            },
            "$inc": {
                "review_stats.count": 1,
                rating_key: 1
            },
            "$set": {
                "updated_at": datetime.now()
            }
        }
    )
    print(f"  Product subset updated: {update_result.modified_count} doc modified")
    
    # STEP C: Recalculate average rating
    # (In production: use $set with expression or do this less frequently)
    updated_product = products.find_one(
        {"_id": product_id},
        {"review_stats": 1}
    )
    dist = updated_product["review_stats"]["distribution"]
    total = sum(int(v) for v in dist.values())
    weighted = sum(int(k) * int(v) for k, v in dist.items())
    new_avg = round(weighted / total, 1) if total > 0 else 0
    
    products.update_one(
        {"_id": product_id},
        {"$set": {"review_stats.avg_rating": new_avg}}
    )
    print(f"  New avg rating: {new_avg}")
    
    return review_id

def upvote_review(review_id: ObjectId, product_id: str):
    """
    When user clicks "helpful" on a review:
    
    1. Update reviews collection (always — source of truth)
    2. If review is in the product's embedded subset → update there too
    3. If not in subset → no product update needed
    
    Redshift equivalent:
        UPDATE store.reviews SET helpful_votes = helpful_votes + 1
        WHERE review_id = :id;
        -- One table, one update, done
        -- No need to update product table
    
    MongoDB trade-off:
        -- Denormalization means updating in multiple places
        -- But READ performance is dramatically better
    """
    
    # Step 1: Update the review in reviews collection (source of truth)
    reviews.update_one(
        {"_id": review_id},
        {"$inc": {"helpful_votes": 1}}
    )
    print(f"  Updated review {review_id} in reviews collection")
    
    # Step 2: Check if this review is in the product's recent_reviews subset
    # Use array filter to update the embedded copy IF it exists
    result = products.update_one(
        {
            "_id": product_id,
            "recent_reviews._id": review_id     # only match if review is embedded
        },
        {
            "$inc": {"recent_reviews.$.helpful_votes": 1}   # $ = matched array element
        }
    )
    
    if result.modified_count > 0:
        print(f"  Also updated embedded copy in product document")
    else:
        print(f"  Review not in embedded subset — no product update needed")
    
    # Step 3: Same for top_helpful_reviews subset
    result2 = products.update_one(
        {
            "_id": product_id,
            "top_helpful_reviews._id": review_id
        },
        {
            "$inc": {"top_helpful_reviews.$.helpful_votes": 1}
        }
    )
    
    if result2.modified_count > 0:
        print(f"  Also updated top_helpful embedded copy")

def batch_refresh_subsets(batch_size: int = 100):
    """
    Scheduled job (cron/Atlas Trigger) to keep subsets fresh.
    
    WHY: In high-traffic systems, updating subset on every review 
    might be too expensive. Instead:
        → recent_reviews: update on each new review (real-time freshness)
        → top_helpful_reviews: batch refresh every hour (eventual consistency)
        → review_stats: batch refresh every 15 minutes
    
    Redshift equivalent:
        REFRESH MATERIALIZED VIEW product_review_stats;
        -- One command refreshes all products
        -- MongoDB: must iterate and update each product document
    """
    
    # Get all distinct product IDs that have reviews
    product_ids = reviews.distinct("product_id")
    refreshed = 0
    
    for pid in product_ids:
        # Recompute stats
        stats_result = list(reviews.aggregate([
            {"$match": {"product_id": pid}},
            {"$group": {
                "_id": None,
                "count": {"$sum": 1},
                "avg_rating": {"$avg": "$rating"},
                "r1": {"$sum": {"$cond": [{"$eq": ["$rating", 1]}, 1, 0]}},
                "r2": {"$sum": {"$cond": [{"$eq": ["$rating", 2]}, 1, 0]}},
                "r3": {"$sum": {"$cond": [{"$eq": ["$rating", 3]}, 1, 0]}},
                "r4": {"$sum": {"$cond": [{"$eq": ["$rating", 4]}, 1, 0]}},
                "r5": {"$sum": {"$cond": [{"$eq": ["$rating", 5]}, 1, 0]}}
            }}
        ]))
        
        if not stats_result:
            continue
        
        s = stats_result[0]
        
        # Get fresh recent reviews
        recent = list(
            reviews.find(
                {"product_id": pid},
                {"user_name": 1, "rating": 1, "title": 1,
                 "body": 1, "date": 1, "helpful_votes": 1, "verified": 1}
            ).sort("date", DESCENDING).limit(10)
        )
        for r in recent:
            r["body_preview"] = r.pop("body", "")[:150] + "..."
        
        # Get fresh top helpful
        helpful = list(
            reviews.find(
                {"product_id": pid},
                {"user_name": 1, "rating": 1, "title": 1,
                 "body": 1, "date": 1, "helpful_votes": 1, "verified": 1}
            ).sort("helpful_votes", DESCENDING).limit(3)
        )
        for r in helpful:
            r["body_preview"] = r.pop("body", "")[:150] + "..."
        
        # Atomic update of entire subset
        products.update_one(
            {"_id": pid},
            {"$set": {
                "review_stats": {
                    "count": s["count"],
                    "avg_rating": round(s["avg_rating"], 1),
                    "distribution": {
                        "1": s["r1"], "2": s["r2"], "3": s["r3"],
                        "4": s["r4"], "5": s["r5"]
                    }
                },
                "recent_reviews": recent,
                "top_helpful_reviews": helpful,
                "updated_at": datetime.now()
            }}
        )
        refreshed += 1
    
    print(f"  Refreshed subsets for {refreshed} products")
    return refreshed

batch_refresh_subsets()

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

OPTIMIZATION 1: $slice PROJECTION (without subset pattern)
    → If you can't redesign documents:
    → db.products.findOne({ _id: "prod_12345" }, { reviews: { $slice: -10 } })
    → Returns only last 10 reviews in the array
    → BUT: full document still loaded from disk into memory
    → Projection happens AFTER disk read, not before
    → Saves network transfer, does NOT save cache pressure
    → VERDICT: band-aid, not a solution

OPTIMIZATION 2: CHANGE STREAM FOR REAL-TIME SUBSET REFRESH
    → Instead of updating subset in submit_review() function:
    → Set up change stream on reviews collection
    → On every insert → automatically update product subset
    → Decouples review submission from subset maintenance
    → More resilient (if subset update fails, review still saved)
    → We'll build this in Project 25

OPTIMIZATION 3: $merge FOR BATCH REFRESH
    → Use aggregation pipeline with $merge to refresh all product subsets
    → Single pipeline replaces the Python loop in batch_refresh_subsets()
    
    db.reviews.aggregate([
        { $sort: { product_id: 1, date: -1 } },
        { $group: {
            _id: "$product_id",
            recent_reviews: { $push: { /* fields */ } },
            count: { $sum: 1 },
            avg_rating: { $avg: "$rating" }
        }},
        { $set: {
            recent_reviews: { $slice: ["$recent_reviews", 10] }
        }},
        { $merge: {
            into: "products_subset",
            on: "_id",
            whenMatched: "merge"
        }}
    ])
    → Server-side execution — no data leaves MongoDB
    → Much faster than client-side loop for thousands of products

OPTIMIZATION 4: PARTIAL INDEX ON REVIEWS
    → If 90% of review queries are for recent reviews:
    → db.reviews.createIndex(
        { product_id: 1, date: -1 },
        { partialFilterExpression: { date: { $gte: ISODate("2024-01-01") } } }
      )
    → Smaller index → fits in RAM → faster queries for recent data
    → Old reviews still queryable, just with full collection scan
    → Concept from Project 11 (partial indexes)

OPTIMIZATION 5: SEPARATE HOT AND COLD REVIEW COLLECTIONS
    → Beyond subset pattern: actually split the collection
    → reviews_recent: last 90 days (small, fast, indexed)
    → reviews_archive: older than 90 days (large, cold storage)
    → TTL-like rotation: move reviews from recent → archive periodically
    → Product subset still embeds top 10 from reviews_recent
    → Extreme optimization for very high volume (millions of reviews/day)

✅ WHAT SHOULD DO (Best Practices)

✅ CHOOSE SUBSET SIZE BASED ON UI REQUIREMENTS
    → UI shows 10 reviews by default? → Embed 10
    → UI shows 5 reviews + "show all"? → Embed 5
    → NEVER embed more than the UI needs on first load
    → The number should match your application's default view

✅ KEEP SUBSET ENTRIES MINIMAL
    → Full review body: 500-2000 characters (in reviews collection)
    → Subset body_preview: 150 characters (in product document)
    → Full images array: [url1, url2, url3] (in reviews collection)
    → Subset: no images (loaded on demand)
    → EVERY BYTE in the subset = multiplied by every product in cache

✅ USE $push WITH $sort AND $slice FOR MAINTENANCE
    → Atomic operation to maintain bounded, sorted array
    → No race conditions — MongoDB handles concurrency
    → db.products.updateOne(
        { _id: pid },
        { $push: { recent_reviews: {
            $each: [newReview],
            $sort: { date: -1 },
            $slice: 10
        }}}
      )

✅ HAVE A SOURCE OF TRUTH
    → reviews collection = SOURCE OF TRUTH (complete data)
    → product.recent_reviews = CACHED SUBSET (derived data)
    → If subset gets corrupted → rebuild from reviews collection
    → NEVER delete from reviews just because subset was updated

✅ INDEX THE REVIEWS COLLECTION PROPERLY
    → { product_id: 1, date: -1 } for paginated access
    → { product_id: 1, helpful_votes: -1 } for "most helpful" queries
    → { product_id: 1, rating: 1, date: -1 } for filtered views
    → These indexes serve the "Load More" and analytics queries

✅ MONITOR DOCUMENT SIZES
    → db.products_subset.aggregate([
        { $project: { size: { $bsonSize: "$$ROOT" } } },
        { $group: { _id: null, avg: { $avg: "$size" }, max: { $max: "$size" } } }
      ])
    → Alert if avg document size exceeds your target (e.g., 10 KB)
    → Growing document = subset not bounded properly

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT EMBED THEN $slice AT QUERY TIME AS A PERMANENT SOLUTION
    WRONG APPROACH:
        → Keep all 50,000 reviews embedded
        → Use { reviews: { $slice: -10 } } on every query
    WHY IT FAILS:
        → $slice is a PROJECTION — applied AFTER disk read
        → Full 20 MB document loaded into WiredTiger cache
        → Full 20 MB deserialized by driver
        → $slice reduces NETWORK transfer but NOT disk I/O or cache usage
        → The 200 GB cache problem (from our analysis) remains unchanged
    
    THIS IS THE #1 TRAP REDSHIFT DEVELOPERS FALL INTO:
        → In Redshift: LIMIT 10 prevents reading extra rows (sort key + zone maps)
        → In MongoDB: $slice / limit does NOT prevent reading extra document data
        → Document = atomic unit of I/O — it's all or nothing from disk

❌ DO NOT UPDATE SUBSET ON EVERY MINOR EVENT
    WRONG:
        → Update product.recent_reviews on every helpful vote
        → Update product.top_helpful_reviews on every new review
    WHY:
        → helpful_votes change frequently → too many writes to product doc
        → Separate update frequencies by data volatility:
          → recent_reviews: on every new review (low frequency per product)
          → top_helpful_reviews: hourly batch (votes change frequently)
          → review_stats: every 15 min or on every review (your choice)

❌ DO NOT STORE FULL REVIEW TEXT IN THE SUBSET
    WRONG:
        → recent_reviews: [{ body: "2000 characters of text..." }]
    WHY:
        → 10 reviews × 2000 chars = 20 KB just for review text
        → Product document bloats from 4 KB to 24 KB
        → 6x more cache pressure
    RIGHT:
        → body_preview: "First 150 characters..." (truncated)
        → Full body loaded on demand from reviews collection
        → User clicks "Read more" → separate query

❌ DO NOT DUPLICATE MUTABLE DATA WITHOUT A REFRESH STRATEGY
    WRONG:
        → Embed user_name in subset but never update when user changes name
        → user_name: "bob_2020" in subset, but user renamed to "bob_builder"
    WHY:
        → Denormalization requires maintenance discipline
        → EVERY duplicated field needs a defined refresh strategy
    RIGHT:
        → user_name rarely changes → acceptable staleness (refresh daily)

        → Project 14 covered this decision framework

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB: WiredTiger and the Working Set

WHAT HAPPENS WHEN A PRODUCT PAGE LOADS (SUBSET PATTERN):

1. APPLICATION SENDS: db.products.findOne({ _id: "prod_12345" })

2. WIREDTIGER CHECKS CACHE
   → Is document for _id="prod_12345" in the WiredTiger cache?
   
   IF YES (CACHE HIT — the common case with subset pattern):
       → Document already in memory (uncompressed)
       → Return immediately
       → Latency: ~0.1-1ms
       → NO DISK I/O
   
   IF NO (CACHE MISS — happens when cache is full):
       → Read from disk (compressed BSON in WiredTiger data file)
       → Decompress (snappy/zstd/zlib → uncompressed BSON)
       → Place in cache (evicting LRU document if needed)
       → Return to application
       → Latency: ~5-50ms (depends on disk speed)

3. WHY DOCUMENT SIZE MATTERS FOR CACHE:

   WiredTiger cache is a FIXED SIZE (e.g., 7.5 GB)
   
   NAIVE DESIGN (20 MB per product):
   ┌─────────────────────────────────────────┐
   │ WiredTiger Cache: 7.5 GB                 │
   │ ┌───────────────────┐                    │
   │ │ Product A (20 MB)  │                   │
   │ ├───────────────────┤                    │
   │ │ Product B (20 MB)  │                   │
   │ ├───────────────────┤                    │
   │ │ Product C (20 MB)  │                   │
   │ ├───────────────────┤                    │
   │ │ ... 375 products   │ ← ONLY 375 FIT!  │
   │ │     total          │                   │
   │ └───────────────────┘                    │
   │ Product 376 arrives → EVICT Product A    │
   │ Next request for A → DISK READ (slow)    │
   └─────────────────────────────────────────┘
   
   SUBSET DESIGN (4 KB per product):
   ┌─────────────────────────────────────────┐
   │ WiredTiger Cache: 7.5 GB                 │
   │ ┌───┬───┬───┬───┬───┬───┬───┬───┬───┐  │
   │ │ A │ B │ C │ D │ E │...│   │   │   │  │
   │ └───┴───┴───┴───┴───┴───┴───┴───┴───┘  │
   │ ~1,875,000 products fit!                 │
   │ ALL hot products cached → ALL cache hits │
   │ Zero evictions → zero disk reads         │
   └─────────────────────────────────────────┘

4. CACHE EVICTION MECHANICS:
   → WiredTiger uses LRU (Least Recently Used) eviction
   → When cache reaches ~80% full → background eviction starts
   → When cache reaches ~95% full → foreground eviction (blocks operations!)
   → Foreground eviction = application stalls = user-visible latency spikes
   
   SUBSET PATTERN PREVENTS THIS:
   → Small documents → more fit in cache → less eviction pressure
   → No foreground eviction → no latency spikes
   → Predictable performance

5. DOCUMENT-LEVEL LOCKING:
   → When updating product subset (new review arrives):
   → WiredTiger locks ONLY that one product document
   → Other products readable/writable simultaneously
   → Small documents → lock held for less time → better concurrency
   → 20 MB document update → longer lock → worse concurrency

Redshift: Why This Problem Doesn't Exist There

REDSHIFT'S COLUMNAR ARCHITECTURE AVOIDS THIS ENTIRELY:

1. SELECT p.name, p.price FROM products WHERE product_id = 12345
   → Reads ONLY name column block + price column block
   → Skips: description, images, all other columns
   → Zone maps: block min/max on product_id → skip irrelevant blocks
   → Data read: ~100 bytes (just the values needed)

2. SELECT r.rating, r.title FROM reviews 
   WHERE product_id = 12345 ORDER BY created_at DESC LIMIT 10
   → Sort key on (product_id, created_at) → data physically sorted
   → Zone maps skip to blocks containing product_id = 12345
   → Within those blocks: data sorted by created_at
   → LIMIT 10: read last 10 rows from sorted data → STOP
   → Data read: ~2 KB (just 10 rows × 2 columns)

3. NO "WORKING SET" PROBLEM:
   → Redshift doesn't keep individual rows in memory cache
   → It caches COLUMN BLOCKS (1 MB chunks)
   → Each block serves MANY rows
   → Block-level caching is naturally efficient for columnar access
   → Result cache: identical queries return instantly

WHY THE INTERNALS CAUSE DIFFERENT BEHAVIOR:

┌────────────────────────────────────────────────────────────────┐
│                                                                 │
│ MongoDB WiredTiger:                                             │
│   → I/O unit = DOCUMENT (entire document read/written)          │
│   → Cache unit = DOCUMENT (entire document cached)              │
│   → Large documents = waste I/O + waste cache + slow updates    │
│   → SOLUTION: make documents small (Subset Pattern)             │
│                                                                 │
│ Redshift Columnar:                                              │
│   → I/O unit = COLUMN BLOCK (1 MB of one column)               │
│   → Cache unit = COLUMN BLOCK                                   │
│   → Row size doesn't matter — columns read independently        │
│   → 50,000 reviews in separate table = separate column blocks   │
│   → LIMIT 10 = read only needed blocks                          │
│   → NO subset pattern needed — architecture handles it          │
│                                                                 │
│ CORE INSIGHT:                                                   │
│   Redshift's architecture solves this problem automatically     │
│   MongoDB requires explicit design (Subset Pattern) to solve it │
│   This is NOT a MongoDB weakness — it's a DIFFERENT TRADE-OFF   │
│   MongoDB's document model gives other advantages               │
│   (single-read page loads, flexible schema, real-time writes)   │
│   but requires thoughtful document design                       │
│                                                                 │
└────────────────────────────────────────────────────────────────┘

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I'll just use $slice — same as LIMIT"
    REDSHIFT THINKING:
        → LIMIT 10 prevents reading more than 10 rows
        → Zone maps + sort key → minimal I/O
    MONGODB REALITY:
        → $slice: { reviews: { $slice: -10 } }
        → ENTIRE document read from disk first
        → $slice applied in memory AFTER full document load
        → 20 MB read → 10 reviews returned → 19.998 MB wasted
    FIX: Use subset pattern — embed only what you need

PITFALL 2: "My subset is growing unbounded"
    SYMPTOM:
        → Used $push without $slice
        → recent_reviews array grows with every new review
        → After 6 months: 5,000 "recent" reviews embedded
    FIX:
        → ALWAYS use $push with $slice: { $push: { $each: [...], $slice: 10 } }
        → Bounded arrays (from Project 17 Outlier Pattern)
        → Monitor with: db.products.aggregate([
            { $project: { reviewCount: { $size: "$recent_reviews" } } },
            { $group: { _id: null, max: { $max: "$reviewCount" }, avg: { $avg: "$reviewCount" } } }
          ])

PITFALL 3: "Subset is stale — showing yesterday's top review"
    SYMPTOM:
        → A review got 500 helpful votes today
        → But top_helpful_reviews still shows yesterday's ranking
    CAUSE:
        → top_helpful_reviews only refreshed in batch job (hourly)
        → Between refreshes, embedded data is stale
    FIX:
        → ACCEPT eventual consistency for non-critical data
        → OR increase refresh frequency for important subsets
        → OR use change streams for real-time refresh (Project 25)
        → Document your staleness tolerance per subset type

PITFALL 4: "I denormalized everything — now updates are everywhere"
    SYMPTOM:
        → user_name in recent_reviews, top_helpful_reviews, AND reviews collection
        → User changes name → must update ALL three places
        → Forgot one → inconsistent data
    FIX:
        → Minimize mutable fields in subsets
        → user_name rarely changes → acceptable in subset (refresh on batch)
        → OR store user_id only, resolve name at display time
        → Map every duplicated field to its refresh strategy

PITFALL 5: "I'm doing Redshift-style pagination on large offsets"
    REDSHIFT THINKING:
        → OFFSET 10000 LIMIT 10 (skip 10,000 rows)
        → Redshift handles this with sort key efficiency
    MONGODB REALITY:
        → skip(10000).limit(10) → MongoDB must iterate through 10,000 documents
        → Performance degrades linearly with skip value
    FIX:
        → Use range-based pagination instead:
        → First page:  find({ product_id: pid }).sort({ date: -1 }).limit(10)
        → Next page:   find({ product_id: pid, date: { $lt: lastSeenDate } })
                        .sort({ date: -1 }).limit(10)
        → Uses index efficiently — no skip overhead regardless of page depth

📈 SCALABILITY CONSIDERATIONS

SCALING THE SUBSET PATTERN:

HORIZONTAL (MongoDB Sharding):
    products collection:
        → Shard key: { _id: "hashed" } or { category: 1, _id: 1 }
        → Each shard holds a subset of products
        → Product page query: targeted to single shard (shard key = _id)
        → Small documents (4 KB) → each shard's cache holds millions
    
    reviews collection:
        → Shard key: { product_id: "hashed" }
        → Reviews for same product co-located on same shard
        → "Load More" query: targeted to single shard
        → Analytics across ALL reviews: scatter-gather (slower)

VERTICAL (Redshift Resize):
    → Add more nodes or bigger nodes
    → products table: small, rarely the bottleneck
    → reviews table: can grow to billions of rows
    → Columnar compression keeps storage manageable
    → Sort key on (product_id, created_at) → efficient at any scale

CACHE SCALING COMPARISON:
    MongoDB:
        → More products → need more cache OR smaller documents
        → Subset pattern: documents stay small → cache scales linearly
        → 16 GB RAM server → ~2M products cached (at 4 KB each)
        → Need more? → shard → each shard has its own cache
    
    Redshift:
        → More products → more column blocks → managed automatically
        → Result cache handles repeated queries
        → RA3 managed storage → compute/storage scale independently
        → No "document size" concern — columnar blocks are fixed 1 MB

AT EXTREME SCALE (100M+ reviews):
    MongoDB:
        → reviews collection: sharded by product_id
        → products collection: subset pattern keeps pages fast
        → Analytics queries: use aggregation on sharded reviews
        → OR: pipe to Redshift for heavy analytics (Project 34)
    
    Redshift:
        → 100M review rows: well within Redshift comfort zone
        → Columnar scan on rating column: fast even at 100M rows
        → Complex JOINs with orders, users: Redshift excels here
        → Product page serving: NOT Redshift's strength (latency too high)

HOW TO MONITOR SUBSET PATTERN HEALTH:

1. DOCUMENT SIZE MONITORING:
   db.products_subset.aggregate([
       { $project: {
           name: 1,
           doc_size_bytes: { $bsonSize: "$$ROOT" },
           review_subset_count: { $size: { $ifNull: ["$recent_reviews", []] } }
       }},
       { $group: {
           _id: null,
           avg_size: { $avg: "$doc_size_bytes" },
           max_size: { $max: "$doc_size_bytes" },
           min_size: { $min: "$doc_size_bytes" },
           avg_subset_count: { $avg: "$review_subset_count" },
           max_subset_count: { $max: "$review_subset_count" }
       }}
   ])

   → If growing: subset not bounded properly
   → max_subset_count should never exceed your $slice value (10)

2. CACHE HIT RATIO:
   db.serverStatus().wiredTiger.cache
   → "pages read into cache" (cache misses)
   → "pages requested from the cache" (total requests)
   → Hit ratio = 1 - (misses / total)
   → Target: > 95% for product page workload
   → If low: documents too large OR cache too small

3. SUBSET FRESHNESS:
   // Find products where subset might be stale
   db.products_subset.aggregate([
       { $project: {
           name: 1,
           last_subset_update: "$updated_at",
           newest_embedded_review: { $arrayElemAt: ["$recent_reviews.date", 0] }
       }},
       { $match: {
           // Subset older than 1 hour
           updated_at: { $lt: new Date(Date.now() - 3600000) }
       }},
       { $limit: 10 }
   ])

4. REDSHIFT EQUIVALENT MONITORING:
   -- Redshift: no subset pattern → different metrics
   -- Monitor: query latency, queue wait time, scan efficiency
   -- SVL_QUERY_SUMMARY: execution time per query
   -- STL_WLM_QUERY: WLM queue wait times
