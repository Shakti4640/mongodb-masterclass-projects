PROJECT 22: Choosing the Right Shard Key — The Hardest Decision

Problem: Multi-Tenant SaaS Platform — 500 Tenants, Billions of Documents, One Wrong Key Ruins Everything

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    You're building a B2B SaaS analytics platform
    Each tenant (company) sends event data for their users
    
    CURRENT STATE:
        → 500 tenants (companies)
        → Tenant sizes vary WILDLY:
            → Top 5 tenants: 500M events each (enterprise customers)
            → Next 45 tenants: 50M events each
            → Next 200 tenants: 5M events each
            → Bottom 250 tenants: 100K events each
        → Total: ~5.5 billion events, ~4 TB
        → Growing: 50M new events/day
    
    ACCESS PATTERNS:
        1. MOST COMMON (80%): Query events for ONE tenant
           → "Show me all events for tenant_042 in the last 7 days"
           → Must be fast: <100ms
        
        2. COMMON (15%): Aggregate within one tenant
           → "Count events by type for tenant_042 today"
           → Must be fast: <500ms
        
        3. RARE (5%): Cross-tenant analytics (admin dashboard)
           → "Total events across all tenants today"
           → Acceptable: <5 seconds
    
    THE DILEMMA:
        → Shard by tenant_id? → Top 5 tenants = MASSIVE chunks (hot shards)
        → Shard by event_date? → Monotonic = hot shard for writes
        → Shard by _id (hashed)? → Every tenant query = scatter-gather
        → Compound key? → Which combination? Which order?
    
    YOU MUST CHOOSE ONE SHARD KEY
    → It determines your system's performance ceiling
    → It was PERMANENT before MongoDB 5.0
    → Even now, resharding takes days on 4 TB
    → Getting it wrong = rebuild the entire cluster
    
    Your Redshift brain says: "DISTKEY on tenant_id, SORTKEY on event_date — done"
    MongoDB says: "That's the starting point — but the DETAILS matter enormously"

🔴 REDSHIFT MINDSET — How You Would Solve This

The Redshift Decision (Relatively Simple)

-- REDSHIFT: Distribution + Sort Key Decision

-- The decision tree in Redshift is straightforward:
-- 1. What column do you JOIN on most? → DISTKEY
-- 2. What column do you filter by time? → SORTKEY
-- 3. Is the table small? → DISTSTYLE ALL

CREATE TABLE analytics.events (
    event_id        BIGINT IDENTITY(1,1),
    tenant_id       BIGINT NOT NULL,
    user_id         VARCHAR(64),
    event_type      VARCHAR(50),
    event_data      SUPER,              -- semi-structured JSON in Redshift
    session_id      VARCHAR(64),
    device          VARCHAR(20),
    country         VARCHAR(5),
    amount          DECIMAL(10,2),
    created_at      TIMESTAMP NOT NULL
)
DISTSTYLE KEY DISTKEY(tenant_id)        -- co-locate all tenant data
COMPOUND SORTKEY(tenant_id, created_at);  -- fast range scans within tenant

-- WHY THIS WORKS IN REDSHIFT:

-- 1. DISTKEY(tenant_id):
--    → Hash of tenant_id → deterministic slice assignment
--    → All events for tenant_042 → same slice(s)
--    → JOIN events ON tenants → no data redistribution
--    → EVEN with "top 5 tenants have 500M rows each":
--      → Hash distributes across slices (not 1:1 mapping)
--      → Tenant_042 events might span 3-4 slices
--      → Redshift handles skew with work stealing

-- 2. COMPOUND SORTKEY(tenant_id, created_at):
--    → Within each slice: data sorted by tenant_id then date
--    → WHERE tenant_id = 42 AND created_at >= '2024-05-01'
--      → Zone maps skip blocks with wrong tenant_id
--      → Within tenant_id blocks: skip old dates
--      → Extremely efficient scan

-- 3. SKEW HANDLING:
--    → Top 5 tenants: more rows → more blocks on their slices
--    → Redshift WLM: work-stealing across slices balances query time
--    → Not perfect but acceptable for analytics workload
--    → Alternative: DISTSTYLE EVEN if skew is extreme

-- THE KEY INSIGHT FOR REDSHIFT USERS:
-- Redshift separates DISTRIBUTION (DISTKEY) from ORDERING (SORTKEY)
-- You optimize TWO things independently
-- MongoDB shard key serves BOTH purposes simultaneously
-- This makes the MongoDB decision MUCH harder

Why Redshift's Approach Doesn't Translate Directly

REDSHIFT vs MONGODB — The Fundamental Difference:

REDSHIFT:
    DISTKEY = which SLICE stores the row (distribution)
    SORTKEY = physical ORDER within the slice (access efficiency)
    → Two independent knobs to tune
    → DISTKEY wrong? JOINs are slow, but queries still work
    → SORTKEY wrong? Scans are slow, but queries still work
    → Change either: recreate table (hours, but one command)

MONGODB:
    Shard Key = which SHARD stores the document (distribution)
    Shard Key = which queries are TARGETED vs SCATTER-GATHER (routing)
    Shard Key = determines CHUNK boundaries (balancing granularity)
    → ONE decision controls THREE things
    → Shard key wrong? Queries slow AND writes unbalanced AND chunks uneven
    → Change: resharding (MongoDB 5.0+) or rebuild from scratch
    → Much higher stakes than Redshift DISTKEY choice

🟢 MONGODB MINDSET — The Three Factors Deep Dive

Factor 1: CARDINALITY — How Many Unique Values?

CARDINALITY = number of distinct shard key values
    → Determines MAXIMUM number of chunks
    → More chunks = finer-grained balancing
    → Fewer chunks = "jumbo chunks" that can't be split

WHY IT MATTERS:
    → Each chunk contains documents with a RANGE of shard key values
    → Chunk splits when it exceeds max size (128 MB)
    → Split point = a shard key value BETWEEN two documents
    → If all documents have SAME shard key value → CAN'T SPLIT

CARDINALITY ANALYSIS FOR OUR PROBLEM:

    Candidate: tenant_id
    → 500 unique values
    → Maximum chunks: 500 (one per tenant)
    → 4 TB / 500 = 8 GB average per chunk
    → Top tenant: 500M events ≈ 400 GB in ONE chunk
    → 400 GB chunk on ONE shard → that shard is overwhelmed
    → VERDICT: cardinality too low for data volume

    Candidate: tenant_id + event_date (compound)
    → 500 tenants × 365 days/year × 3 years = ~547,500 combinations
    → Maximum chunks: 547,500
    → 4 TB / 547,500 = ~7.5 MB per chunk average
    → Top tenant, one day: ~1.4M events ≈ ~1 GB per chunk
    → Manageable — chunks can split further if needed
    → VERDICT: excellent cardinality

    Candidate: tenant_id + user_id (compound)
    → 500 tenants × millions of users = hundreds of millions of combinations
    → Extremely high cardinality
    → Each chunk: very small
    → VERDICT: very high cardinality (almost too high — many tiny chunks)

    Candidate: event_type
    → 10-20 unique values (click, pageview, purchase, signup, etc.)
    → Maximum chunks: 20
    → 4 TB / 20 = 200 GB per chunk
    → JUMBO CHUNKS guaranteed — can never balance properly
    → VERDICT: ❌ catastrophically low cardinality

    │ SHARD KEY                      │ CARDINALITY  │ ASSESSMENT            │
    ├───────────────────────────────┼─────────────┼──────────────────────┤
    │ event_type                     │ ~15          │ ❌ Disastrous          │
    │ tenant_id                      │ 500          │ ⚠️ Low for 4 TB       │
    │ tenant_id + event_date         │ ~550K        │ ✅ Excellent           │
    │ tenant_id + user_id            │ ~100M+       │ ✅ Very high           │
    │ _id (ObjectId)                 │ ~5.5B        │ ✅ Maximum (but no     │
    │                                │              │    query targeting)    │
    └───────────────────────────────┴─────────────┴──────────────────────┘

REDSHIFT COMPARISON:
    → Redshift DISTKEY: hash-based → cardinality determines skew
    → Low cardinality DISTKEY in Redshift: data skew across slices
    → Same problem, same root cause, different manifestation
    → Redshift: slow queries due to skew
    → MongoDB: jumbo chunks + hot shards + inability to balance

Factor 2: FREQUENCY — How Evenly Are Values Distributed?

FREQUENCY = how many documents share each shard key value
    → Even frequency = balanced chunks = balanced shards
    → Uneven frequency = some chunks MUCH larger = hot shards

WHY IT MATTERS EVEN WITH HIGH CARDINALITY:
    → { tenant_id: 1, event_date: 1 } has high cardinality (550K)
    → But tenant_001 generates 5M events/day vs tenant_400 generates 100/day
    → Chunk for (tenant_001, 2024-05-31): 5M events ≈ 4 GB
    → Chunk for (tenant_400, 2024-05-31): 100 events ≈ 80 KB
    → 50,000x difference in chunk sizes!
    → Shard hosting tenant_001's chunks: overwhelmed
    → Shard hosting tenant_400's chunks: nearly idle

FREQUENCY ANALYSIS:

    Our tenant distribution:
    ┌────────────────┬──────────────┬──────────────┬──────────────────┐
    │ TENANT TIER     │ COUNT         │ EVENTS/TENANT │ % OF TOTAL DATA   │
    ├────────────────┼──────────────┼──────────────┼──────────────────┤
    │ Enterprise      │ 5             │ 500M each     │ 45% (2.5B events) │
    │ Large           │ 45            │ 50M each      │ 41% (2.25B)       │
    │ Medium          │ 200           │ 5M each       │ 18% (1B)          │
    │ Small           │ 250           │ 100K each     │ <1% (25M)         │
    └────────────────┴──────────────┴──────────────┴──────────────────┘

    KEY INSIGHT: 
        → 5 tenants generate 45% of ALL data
        → 250 tenants generate <1% of data
        → 10,000:1 ratio between largest and smallest tenant
        
    WITH SHARD KEY { tenant_id: 1 }:
        → Tenant_001 (500M events) → one chunk range → ONE shard
        → That shard has 500M events = 400 GB
        → Other shards share the remaining 250 tenants
        → EXTREME skew: one shard has 10x more data
    
    WITH SHARD KEY { tenant_id: 1, _id: 1 }:
        → Tenant_001's 500M events split into MANY chunks
        → Each chunk: range of (tenant_001, ObjectId_X) to (tenant_001, ObjectId_Y)
        → Chunks can be distributed across MULTIPLE shards
        → Large tenant's data spreads across cluster
        → MUCH better balance

    ┌────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │ THE FREQUENCY FIX: Add a high-cardinality second field         │
    │                                                                 │
    │ { tenant_id: 1 }                                                │
    │   → 500M events locked to one chunk range per tenant            │
    │   → Can't split within a tenant → hot shard                    │
    │                                                                 │
    │ { tenant_id: 1, _id: 1 }                                       │
    │   → 500M events splittable within each tenant                   │
    │   → Chunks split at _id boundaries → distributable              │
    │   → Large tenant's data spans multiple shards                   │
    │                                                                 │
    └────────────────────────────────────────────────────────────────┘

REDSHIFT COMPARISON:
    → DISTKEY(tenant_id) with skewed tenant sizes:
      → Some slices get 10x more data → query skew
      → Redshift mitigation: work-stealing between slices
      → MongoDB mitigation: compound shard key for sub-tenant splitting
    → Both systems suffer from frequency skew
    → MongoDB has MORE tools to address it (compound keys, zone sharding)

Factor 3: MONOTONICITY — Does the Value Always Increase?

MONOTONICITY = shard key values that always increase (or decrease)
    → New documents always get the HIGHEST shard key value
    → That value falls in the LAST chunk range
    → ALL writes go to ONE shard → hot shard for writes

EXAMPLES OF MONOTONIC FIELDS:
    → created_at (timestamp) — always increases ← MOST COMMON TRAP
    → _id (ObjectId) — contains timestamp, always increases
    → auto_increment_id — always increases
    → version_number — always increases

EXAMPLES OF NON-MONOTONIC FIELDS:
    → tenant_id — new events spread across existing tenants
    → user_id — new events for random users
    → hashed anything — hash output is random
    → customer_id — not correlated with time

MONOTONICITY ANALYSIS FOR OUR PROBLEM:

    { created_at: 1 } — Range sharding on timestamp
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ Shard 0   │ │ Shard 1   │ │ Shard 2   │
    │ Jan-Feb   │ │ Mar-Apr   │ │ May-NOW   │
    │           │ │           │ │ ← ALL new │
    │ 0 writes  │ │ 0 writes  │ │   writes! │
    │ old data  │ │ old data  │ │ HOT SHARD │
    └──────────┘ └──────────┘ └──────────┘
    → 100% of new writes → Shard 2 → bottleneck
    → Shard 0 and 1 are idle for writes
    → ❌ NEVER use monotonic field as range shard key

    { tenant_id: 1, created_at: 1 } — Compound with tenant prefix
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ Shard 0   │ │ Shard 1   │ │ Shard 2   │
    │ tenant    │ │ tenant    │ │ tenant    │
    │ 001-166   │ │ 167-333   │ │ 334-500   │
    │ writes    │ │ writes    │ │ writes    │
    │ from 166  │ │ from 166  │ │ from 166  │
    │ tenants   │ │ tenants   │ │ tenants   │
    └──────────┘ └──────────┘ └──────────┘
    → New writes spread across tenants → spread across shards
    → tenant_id is NOT monotonic (events for existing tenants)
    → created_at is monotonic BUT it's the SECOND field
    → Within each tenant's chunk range: new events go to end
    → But chunks can split on created_at boundary → ok
    → ✅ tenant_id prefix breaks the monotonicity problem

    { created_at: 1, tenant_id: 1 } — WRONG ORDER!
    → created_at is PREFIX → monotonic → HOT SHARD
    → tenant_id as second field doesn't help
    → ALL new events have latest timestamp → last chunk → one shard
    → ❌ Order of compound shard key matters enormously

    ┌────────────────────────────────────────────────────────────────┐
    │ CRITICAL RULE:                                                  │
    │                                                                 │
    │ In a compound shard key: { A: 1, B: 1 }                        │
    │ → A is the PREFIX — determines initial routing                  │
    │ → B is the SUFFIX — enables sub-splitting within A's range     │
    │                                                                 │
    │ If A is monotonic → hot shard (regardless of B)                │
    │ If A is non-monotonic but B is → ok (A distributes writes)     │
    │                                                                 │
    │ FIELD ORDER = EVERYTHING in compound shard keys                 │
    │                                                                 │
    │ Redshift parallel:                                              │
    │ → DISTKEY = the distribution field (like A)                    │
    │ → SORTKEY = the ordering field (like B)                        │
    │ → Redshift separates them → no ordering concern for DISTKEY    │
    │ → MongoDB combines them → ordering is critical                  │
    └────────────────────────────────────────────────────────────────┘

    │       │
    │       ▼

    │
    ├── "We filter by date range across all tenants"
    │       │
    │       ▼
    │   [Is this an OLAP or OLTP workload?]
    │       │
    │       ├── OLAP (analytical)

    │       │       → Or: { date: "hashed" } if must use MongoDB
    │       │

    │               → { date: 1 } is MONOTONIC → ❌ never range shard on date
    │               → { date: "hashed" } → loses date range targeting
    │               → Better: compound { tenant_id: 1, date: 1 }
    │
    └── "We mostly do point lookups by _id"
            │
            ▼
        USE: { _id: "hashed" }
        → Even distribution
        → Every _id lookup is targeted
        → All other queries are scatter-gather
        → Only if _id lookups are 80%+ of queries

function evaluateShardKey(collection, keyFields, numShards, description) {
    /**
     * Comprehensive evaluation of a shard key candidate.
     * 
     * Tests all three factors:
     * 1. CARDINALITY: how many unique key combinations?
     * 2. FREQUENCY: how evenly distributed?
     * 3. MONOTONICITY: does the key always increase?
     * 
     * Plus: query targeting analysis
     */
    
    let totalDocs = collection.countDocuments()
    
    // Build group key expression
    let groupId = {}
    keyFields.forEach(f => {
        if (f.hashed) {
            groupId[f.field] = `$${f.field}`  // can't hash in aggregation, approximate
        } else {
            groupId[f.field] = `$${f.field}`
        }
    })
    
    // CARDINALITY: count distinct key combinations
    let cardPipeline = [
        { $group: { _id: groupId, count: { $sum: 1 } } },
        { $group: {
            _id: null,
            cardinality: { $sum: 1 },
            avg_frequency: { $avg: "$count" },
            max_frequency: { $max: "$count" },
            min_frequency: { $min: "$count" },
            stddev: { $stdDevPop: "$count" },
            frequencies: { $push: "$count" }
        }}
    ]
    
    let cardResult = collection.aggregate(cardPipeline).toArray()[0]
    if (!cardResult) return null
    
    // FREQUENCY: coefficient of variation (stddev / mean)
    let cv = cardResult.stddev / cardResult.avg_frequency
    
    // SKEW: ratio of max to average
    let skewRatio = cardResult.max_frequency / cardResult.avg_frequency
    
    // SIMULATE SHARD DISTRIBUTION
    // Sort frequencies descending, assign to shards round-robin
    let freqs = cardResult.frequencies.sort((a, b) => b - a)
    let shardLoads = Array(numShards).fill(0)
    freqs.forEach((freq, i) => {
        // Simple round-robin assignment (approximate)
        shardLoads[i % numShards] += freq
    })
    
    let maxShardLoad = Math.max(...shardLoads)
    let minShardLoad = Math.min(...shardLoads)
    let shardSkew = minShardLoad > 0 ? maxShardLoad / minShardLoad : Infinity
    
    // MONOTONICITY: check if first key field is time-based
    let firstField = keyFields[0].field
    let isTimeField = ["event_date", "created_at", "timestamp", "_id"].includes(firstField)
    let isMonotonic = isTimeField && !keyFields[0].hashed
    
    return {
        description: description,
        keySpec: keyFields.map(f => f.hashed ? `${f.field}: "hashed"` : `${f.field}: 1`).join(", "),
        cardinality: cardResult.cardinality,
        cardinalityRatio: (cardResult.cardinality / totalDocs * 100).toFixed(1),
        avgFrequency: Math.round(cardResult.avg_frequency),
        maxFrequency: cardResult.max_frequency,
        minFrequency: cardResult.min_frequency,
        cv: cv.toFixed(3),
        skewRatio: skewRatio.toFixed(1),
        shardLoads: shardLoads,
        shardSkew: shardSkew.toFixed(2),
        isMonotonic: isMonotonic,
        totalDocs: totalDocs
    }
}

function scoreCandidate(e) {

    else if (skew < 10) scores.frequency = 3
    else if (skew < 50) scores.frequency = 2
    else scores.frequency = 1

    return scores
}

// Simulate chunk sizes for different shard key choices
function simulateChunks(collection, groupFields, chunkSizeThresholdMB) {
    /**
     * Simulates how chunks would form for a given shard key.
     * 
     * A "jumbo chunk" is one that exceeds the max chunk size
     * AND cannot be split because all documents have the same key value.
     * 
     * Redshift doesn't have this problem:
     * → Hash-based distribution → no "chunks"
     * → Each row independently assigned to a slice
     * → No concept of "range that can't be split"
     */
    
    let groupId = {}
    groupFields.forEach(f => { groupId[f] = `$${f}` })
    
    let chunks = collection.aggregate([
        { $group: {
            _id: groupId,
            doc_count: { $sum: 1 },
            // Approximate size: avg doc ~300 bytes
            approx_size_mb: { $sum: { $literal: 0.0003 } }  // 300 bytes per doc
        }},
        { $sort: { doc_count: -1 } }
    ]).toArray()
    
    let jumboCount = 0
    let totalChunks = chunks.length
    let maxChunkDocs = 0
    
    chunks.forEach(c => {
        if (c.doc_count > maxChunkDocs) maxChunkDocs = c.doc_count
        // A chunk is "jumbo" if it exceeds threshold AND has single key value
        // (single key = groupFields produce one group = can't split further)
        if (c.approx_size_mb > chunkSizeThresholdMB) {
            jumboCount++
        }
    })
    
    return {
        total_chunks: totalChunks,
        jumbo_chunks: jumboCount,
        max_chunk_docs: maxChunkDocs,
        top_5: chunks.slice(0, 5).map(c => ({
            key: JSON.stringify(c._id),
            docs: c.doc_count,
            size_mb: c.approx_size_mb.toFixed(1)
        }))
    }
}

# STEP 2: COMPREHENSIVE SHARD KEY EVALUATOR

class ShardKeyEvaluator:
    """
    Evaluates shard key candidates on all three factors:
    1. Cardinality
    2. Frequency (evenness)
    3. Monotonicity
    
    Plus: query targeting, write distribution, jumbo chunk risk
    
    Redshift equivalent:
        No direct equivalent — Redshift DISTKEY is simpler
        → Choose JOIN column → done
        → No cardinality/frequency/monotonicity analysis needed
        → Hash-based distribution handles most concerns automatically
    """
    
    def __init__(self, collection, num_shards=3):
        self.collection = collection
        self.num_shards = num_shards
        self.total_docs = collection.count_documents({})
    
    def evaluate(self, key_fields: list, description: str, is_hashed: bool = False) -> dict:
        """
        Full evaluation of a shard key candidate.
        
        key_fields: list of field names in the compound key
        description: human-readable description
        is_hashed: whether the first field is hashed
        """
        
        # Build $group _id expression
        group_id = {f: f"${f}" for f in key_fields}
        
        # 1. CARDINALITY + FREQUENCY
        pipeline = [
            {"$group": {"_id": group_id, "count": {"$sum": 1}}},
            {"$group": {
                "_id": None,
                "cardinality": {"$sum": 1},
                "avg_freq": {"$avg": "$count"},
                "max_freq": {"$max": "$count"},
                "min_freq": {"$min": "$count"},
                "stddev": {"$stdDevPop": "$count"},
                "p95_freq": {"$percentile": {
                    "input": "$count",
                    "p": [0.95],
                    "method": "approximate"
                }} if False else {"$max": "$count"},  # fallback
                "frequencies": {"$push": "$count"}
            }}
        ]
        
        try:
            result = list(self.collection.aggregate(pipeline))
        except Exception:
            # Fallback for older MongoDB versions
            pipeline[1]["$group"].pop("p95_freq", None)
            result = list(self.collection.aggregate(pipeline))
        
        if not result:
            return {"error": "No data"}
        
        stats = result[0]
        
        # 2. COEFFICIENT OF VARIATION (evenness measure)
        cv = stats["stddev"] / stats["avg_freq"] if stats["avg_freq"] > 0 else 0
        
        # 3. SKEW RATIO
        skew_ratio = stats["max_freq"] / stats["avg_freq"] if stats["avg_freq"] > 0 else float('inf')
        
        # 4. MONOTONICITY CHECK
        first_field = key_fields[0]
        time_fields = {"event_date", "created_at", "timestamp", "updated_at"}
        is_monotonic = first_field in time_fields and not is_hashed
        
        # Also check _id (ObjectId is monotonic)
        if first_field == "_id" and not is_hashed:
            is_monotonic = True
        
        # 5. SIMULATE SHARD DISTRIBUTION
        frequencies = sorted(stats["frequencies"], reverse=True)
        shard_loads = [0] * self.num_shards
        
        for i, freq in enumerate(frequencies):
            # Assign to least-loaded shard (greedy balancing)
            min_shard = shard_loads.index(min(shard_loads))
            shard_loads[min_shard] += freq
        
        max_load = max(shard_loads)
        min_load = min(shard_loads) or 1
        shard_skew = max_load / min_load
        
        # 6. JUMBO CHUNK RISK
        # Approximate: avg doc size ~300 bytes, chunk limit 128 MB
        chunk_limit_docs = (128 * 1024 * 1024) / 300  # ~447,000 docs per chunk
        jumbo_risk = stats["max_freq"] > chunk_limit_docs
        
        # 7. SCORING
        scores = self._score(stats["cardinality"], skew_ratio, is_monotonic, 
                            shard_skew, key_fields)
        
        return {
            "description": description,
            "key_fields": key_fields,
            "key_spec": ", ".join(f'{f}: "hashed"' if (is_hashed and f == key_fields[0]) else f"{f}: 1" for f in key_fields),
            "is_hashed": is_hashed,
            "cardinality": stats["cardinality"],
            "cardinality_pct": round(stats["cardinality"] / self.total_docs * 100, 1),
            "avg_frequency": round(stats["avg_freq"], 1),
            "max_frequency": stats["max_freq"],
            "min_frequency": stats["min_freq"],
            "stddev": round(stats["stddev"], 2),
            "cv": round(cv, 3),
            "skew_ratio": round(skew_ratio, 1),
            "is_monotonic": is_monotonic,
            "shard_loads": shard_loads,
            "shard_skew": round(shard_skew, 2),
            "jumbo_risk": jumbo_risk,
            "scores": scores,
            "total_score": sum(scores.values())
        }
    
    def _score(self, cardinality, skew_ratio, is_monotonic, shard_skew, key_fields):
        """Score each factor 1-5."""
        scores = {}
        
        # Cardinality (1-5)
        if cardinality > 50000: scores["cardinality"] = 5
        elif cardinality > 10000: scores["cardinality"] = 4
        elif cardinality > 1000: scores["cardinality"] = 3
        elif cardinality > 100: scores["cardinality"] = 2
        else: scores["cardinality"] = 1
        
        # Frequency evenness (1-5)
        if skew_ratio < 2: scores["frequency"] = 5
        elif skew_ratio < 5: scores["frequency"] = 4
        elif skew_ratio < 10: scores["frequency"] = 3
        elif skew_ratio < 50: scores["frequency"] = 2
        else: scores["frequency"] = 1
        
        # Monotonicity (1-5)
        scores["monotonicity"] = 1 if is_monotonic else 5
        
        # Shard balance (1-5)
        if shard_skew < 1.2: scores["balance"] = 5
        elif shard_skew < 1.5: scores["balance"] = 4
        elif shard_skew < 2.0: scores["balance"] = 3
        elif shard_skew < 5.0: scores["balance"] = 2
        else: scores["balance"] = 1
        
        # Query targeting (1-5)
        scores["targeting"] = 5 if key_fields[0] == "tenant_id" else 2
        
        return scores
    
    def print_evaluation(self, eval_result):
        """Pretty print a shard key evaluation."""
        e = eval_result
        
        print(f"\n  ┌── {{ {e['key_spec']} }} ──")
        print(f"  │  {e['description']}")
        print(f"  │")
        
        # Cardinality
        card_label = ("✅ HIGH" if e["cardinality"] > 10000 
                     else "⚠️ MODERATE" if e["cardinality"] > 100 
                     else "❌ LOW")
        print(f"  │  Cardinality:  {e['cardinality']:>10,} unique values ({e['cardinality_pct']}%) — {card_label}")
        
        # Frequency
        freq_label = ("✅ EVEN" if e["skew_ratio"] < 3 
                     else "⚠️ SKEWED" if e["skew_ratio"] < 10 
                     else "❌ HIGHLY SKEWED")
        print(f"  │  Frequency:    avg={e['avg_frequency']}, max={e['max_frequency']}, "
              f"skew={e['skew_ratio']}x — {freq_label}")
        
        # Monotonicity
        mono_label = "❌ HOT SHARD RISK" if e["is_monotonic"] else "✅ SAFE"
        print(f"  │  Monotonicity: {mono_label}")
        
        # Jumbo chunks
        jumbo_label = "❌ JUMBO RISK" if e["jumbo_risk"] else "✅ SPLITTABLE"
        print(f"  │  Jumbo chunks: {jumbo_label}")
        
        # Shard distribution
        total_load = sum(e["shard_loads"])
        print(f"  │  Shard balance ({self.num_shards} shards):")
        for i, load in enumerate(e["shard_loads"]):
            pct = load / total_load * 100
            bar = "█" * int(pct / 2)
            print(f"  │    Shard {i}: {load:>8,} docs ({pct:>5.1f}%) {bar}")
        
        balance_label = ("✅ BALANCED" if e["shard_skew"] < 1.5 
                        else "⚠️ SOME SKEW" if e["shard_skew"] < 3 
                        else "❌ IMBALANCED")
        print(f"  │    Skew ratio: {e['shard_skew']}x — {balance_label}")
        
        # Scores
        s = e["scores"]
        print(f"  │")
        print(f"  │  SCORES:")
        print(f"  │    Cardinality:  {'★' * s['cardinality']}{'☆' * (5 - s['cardinality'])}")
        print(f"  │    Frequency:    {'★' * s['frequency']}{'☆' * (5 - s['frequency'])}")
        print(f"  │    Monotonicity: {'★' * s['monotonicity']}{'☆' * (5 - s['monotonicity'])}")
        print(f"  │    Balance:      {'★' * s['balance']}{'☆' * (5 - s['balance'])}")
        print(f"  │    Targeting:    {'★' * s['targeting']}{'☆' * (5 - s['targeting'])}")
        print(f"  │    TOTAL:        {e['total_score']}/25")
        print(f"  └──{'─' * 50}")
    
    def compare_all(self, candidates: list):
        """Evaluate and rank all candidates."""
        results = []
        
        for c in candidates:
            result = self.evaluate(
                c["fields"], c["description"],
                c.get("hashed", False)
            )
            if "error" not in result:
                results.append(result)
                self.print_evaluation(result)
        
        # Ranking
        results.sort(key=lambda x: x["total_score"], reverse=True)
        
        print(f"\n{'=' * 60}")
        print("FINAL RANKING")
        print(f"{'=' * 60}")
        
        print(f"\n  {'Rank':<6} {'Shard Key':<35} {'Score':>7} {'Verdict':>10}")
        print(f"  {'─' * 60}")
        
        for i, r in enumerate(results):
            verdict = "🏆 WINNER" if i == 0 else ("🥈 GOOD" if i == 1 else "")
            print(f"  {i+1:<6} {{ {r['key_spec']:<33}}} {r['total_score']:>4}/25 {verdict:>10}")
        
        return results

def simulate_write_distribution(events_list, key_extractor, num_shards, label):

        shard_writes[shard] += 1
    
    total = sum(shard_writes)
    max_writes = max(shard_writes)
    min_writes = min(shard_writes) or 1

    print(f"    Skew: {skew:.2f}x — {skew_label}")
    
    return skew

✅ WHAT SHOULD DO (Best Practices)

✅ ANALYZE YOUR DATA DISTRIBUTION BEFORE CHOOSING
    → Run the ShardKeyEvaluator (or equivalent analysis)
    → Know your cardinality, frequency distribution, growth pattern
    → Don't guess — MEASURE
    → Redshift developers: you'd check SVV_TABLE_INFO for skew
    → Same discipline needed for MongoDB, but MORE critical

✅ USE COMPOUND SHARD KEYS FOR SKEWED DATA
    → { tenant_id: 1, _id: 1 } — handles ANY level of skew
    → _id suffix guarantees: every document has unique compound key
    → Chunks can ALWAYS be split → no jumbo chunks ever
    → Large tenants automatically spread across multiple shards

✅ PUT THE QUERY FILTER FIELD FIRST IN COMPOUND KEY
    → { tenant_id: 1, _id: 1 } → tenant queries are TARGETED
    → { _id: 1, tenant_id: 1 } → tenant queries are SCATTER-GATHER
    → First field = routing prefix = determines targeting
    → Same logic as Redshift: DISTKEY on JOIN/filter column

✅ PROTOTYPE WITH REALISTIC DATA BEFORE PRODUCTION
    → Generate data matching production distribution
    → Run your actual query patterns against it
    → Measure: query latency, write throughput, shard balance
    → THIS is your dress rehearsal — production is the show

✅ DOCUMENT YOUR SHARD KEY DECISION AND REASONING
    → Future engineers will ask "why this shard key?"
    → Document: access patterns analyzed, candidates evaluated, winner rationale
    → Include: what queries are TARGETED vs SCATTER-GATHER
    → This documentation saves weeks of debugging later

✅ PLAN FOR GROWTH WHEN CHOOSING SHARD KEY
    → Today: 500 tenants, 4 TB
    → Next year: 2000 tenants, 20 TB
    → Does your shard key still work at 10x scale?
    → { tenant_id: 1, _id: 1 } scales infinitely (always splittable)
    → { tenant_id: 1 } might hit jumbo chunks at 10x

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT CHOOSE SHARD KEY BASED ON A SINGLE FACTOR
    WRONG: "tenant_id has good cardinality — ship it!"
    WHY: cardinality is ONE of three factors
         frequency skew could make it terrible
         always evaluate ALL THREE: cardinality + frequency + monotonicity

❌ DO NOT USE TIMESTAMPS AS SHARD KEY PREFIX (RANGE)
    WRONG: { created_at: 1 } or { event_date: 1 }
    WHY: monotonic → ALL new writes → last chunk → one shard
    EXCEPTION: { created_at: "hashed" } is OK (but loses time range targeting)
    EXCEPTION: { tenant_id: 1, created_at: 1 } is OK (tenant prefix distributes)

❌ DO NOT CHOOSE SHARD KEY THAT MOST QUERIES DON'T USE
    WRONG: shard key = { country: 1 } but 80% of queries filter by tenant_id
    WHY: 80% of queries become scatter-gather → worse than no sharding
    RULE: shard key prefix MUST be in your most common query filter

❌ DO NOT ASSUME REDSHIFT DISTKEY INTUITION TRANSFERS DIRECTLY
    TRAP: "In Redshift I'd use tenant_id as DISTKEY, so I'll use tenant_id as shard key"
    WHY THIS CAN FAIL:
    → Redshift: hash-based → skewed tenant sizes don't cause jumbo chunks
    → MongoDB: range-based → skewed tenant = jumbo chunks
    → Redshift: SORTKEY handles time ordering independently
    → MongoDB: shard key must handle BOTH distribution AND query targeting
    → ADD _id AS SUFFIX to handle what Redshift's hash handles automatically

❌ DO NOT SHARD A COLLECTION AND THEN IGNORE MONITORING
    → Sharding is not "set and forget"
    → Monitor: sh.status(), chunk counts, balancer activity
    → Alert on: chunk imbalance > 2x, jumbo chunks, balancer errors
    → Redshift: cluster metrics in CloudWatch (automatic)
    → MongoDB: requires intentional monitoring setup

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB: How Shard Key Determines Document Placement

DOCUMENT INSERTION ON A SHARDED COLLECTION:

1. APPLICATION:
   db.events.insertOne({ tenant_id: "tenant_001", _id: ObjectId("abc...") })

2. MONGOS (query router):
   → Extract shard key value: { tenant_id: "tenant_001", _id: ObjectId("abc...") }
   → Look up routing table (from config servers):
     "tenant_001" + this _id falls in chunk range:
       [{ tenant_id: "tenant_001", _id: MinKey }, 
        { tenant_id: "tenant_001", _id: ObjectId("def...") }]
     → This chunk lives on Shard 2
   → Route insertOne to Shard 2

3. SHARD 2 (Primary of replica set):
   → Receive the document
   → Write to WiredTiger (B-tree insert)
   → Replicate to secondaries (oplog)
   → Return success to mongos → return to application

4. CHUNK SPLITTING (if needed):
   → After insert: chunk size checked
   → If chunk > 128 MB:
     → Find median _id value within chunk
     → Split into two chunks at that _id
     → Update config servers with new chunk boundaries
     → Both chunks remain on same shard (initially)
   → Balancer later moves one chunk to less-loaded shard

5. WHY COMPOUND KEY MATTERS FOR SPLITTING:
   
   Simple key { tenant_id: 1 }:
   → All docs with tenant_id: "tenant_001" have same key value
   → Chunk boundaries can ONLY be between different tenant_id values
   → If tenant_001 has 500M docs → ONE chunk → CANNOT SPLIT
   → This is a JUMBO CHUNK → permanently stuck on one shard
   
   Compound key { tenant_id: 1, _id: 1 }:
   → Each document has unique compound key value
   → Chunk boundaries can be ANYWHERE within tenant_001's data
   → Split at: { tenant_id: "tenant_001", _id: ObjectId("median...") }
   → 500M docs → thousands of chunks → distributed across ALL shards
   → NO JUMBO CHUNKS POSSIBLE
   
   THIS IS THE CRITICAL INTERNAL MECHANISM:
   ┌──────────────────────────────────────────────────────────────┐
   │ Simple key:   chunk boundaries between KEY VALUES             │
   │               → limited by cardinality of key                │
   │                                                               │
   │ Compound key: chunk boundaries between COMPOUND VALUE COMBOS  │
   │               → limited by cardinality of COMBINATION          │
   │               → { tenant_id, _id } = infinite cardinality     │
   │               → chunks can always split → always balanceable  │
   └──────────────────────────────────────────────────────────────┘

Redshift: Why Hash Distribution Avoids These Problems

REDSHIFT ROW PLACEMENT:

1. INSERT INTO events VALUES (tenant_id=42, ...)

2. LEADER NODE:
   → Hash tenant_id: hash(42) = 0x7A3F...
   → Modulo number of slices: 0x7A3F % 8 = 3
   → Route to Slice 3

3. WHY REDSHIFT DOESN'T HAVE JUMBO CHUNKS:
   → No "chunks" concept — each row independently assigned
   → hash(tenant_id) distributes rows even if tenant_id has low cardinality
   → 500 tenants × hash → spread across 8 slices
   → Some slices get more rows (skew) but no "unsplittable" problem
   → Worst case: uneven slice sizes → slower parallel queries
   → NOT: stuck data that can't be rebalanced

4. WHY REDSHIFT DOESN'T HAVE HOT SLICES FROM MONOTONIC KEYS:
   → hash(timestamp_1) → slice 5
   → hash(timestamp_2) → slice 2  (different slice!)
   → hash(timestamp_3) → slice 7  (different again!)
   → Hash function RANDOMIZES the slice assignment
   → Sequential timestamps → random slice placement
   → MongoDB range sharding: sequential → same chunk → same shard

5. WHAT REDSHIFT CAN'T DO THAT MONGODB CAN:
   → MongoDB: zone sharding (data locality by region)
   → MongoDB: targeted range queries on shard key
   → MongoDB: add shards online with zero downtime
   → MongoDB: per-collection shard keys (different strategy per collection)
   → Redshift: one distribution style per table, applied uniformly

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I chose tenant_id without _id suffix and now I have jumbo chunks"
    SYMPTOM:
        → sh.status() shows chunks marked as "jumbo"
        → Balancer can't move these chunks
        → One shard has 80% of data, others have 10% each
        → Adding new shards doesn't help (data can't migrate)
    CAUSE:
        → Shard key: { tenant_id: 1 }
        → Enterprise tenant: 500M events → all in one chunk range
        → Chunk can't split (all docs have same tenant_id value)
    FIX:
        → Resharding to { tenant_id: 1, _id: 1 } (MongoDB 5.0+)
        → Or: dump collection, recreate with compound key, reload
        → PREVENTION: always add high-cardinality suffix (_id)

PITFALL 2: "I put event_date first in compound key and writes are stuck on one shard"
    SYMPTOM:
        → One shard at 100% CPU, others at 10%
        → All inserts queue on one shard
        → Read performance fine, write performance terrible
    CAUSE:
        → Shard key: { event_date: 1, tenant_id: 1 }
        → event_date is monotonic → all new events → latest chunk → one shard
        → tenant_id as SECOND field doesn't help (routing based on first field)
    FIX:
        → Reverse the order: { tenant_id: 1, event_date: 1 }
        → Or: resharding (if MongoDB 5.0+)
    LESSON:
        → Field ORDER in compound shard key is CRITICAL
        → First field determines routing → must NOT be monotonic

PITFALL 3: "My Redshift DISTKEY intuition led me to shard on the JOIN column"
    SCENARIO:
        → In Redshift: DISTKEY on tenant_id → co-locate for JOINs
        → In MongoDB: shard on tenant_id → seems equivalent
        → BUT: MongoDB has no JOINs in the Redshift sense
        → $lookup is rare and different from SQL JOIN
        → Shard key should optimize for QUERY FILTERS, not JOINs
    LESSON:
        → Redshift DISTKEY: optimize for JOIN co-location
        → MongoDB shard key: optimize for query filter targeting
        → Similar but NOT identical optimization goals

PITFALL 4: "I didn't test with production-scale data"
    SYMPTOM:
        → Development: 10,000 events → shard key works great
        → Production: 5 billion events → jumbo chunks, hot shards
        → Skew only visible at scale
    CAUSE:
        → Small test data doesn't reveal frequency distribution
        → Power-law skew only appears with realistic data volumes
    FIX:
        → Generate test data matching production distribution
        → Use the ShardKeyEvaluator with production-like data
        → Test with at least 1% of expected production volume

PITFALL 5: "I need to change the shard key but we're on MongoDB 4.x"
    SITUATION:
        → MongoDB 4.x: shard key is IMMUTABLE (cannot change)
        → Wrong shard key chosen → system degrading
    OPTIONS (all painful):
        a) Upgrade to MongoDB 5.0+ and use resharding
        b) Create new collection with correct shard key
           → Migrate data in background
           → Switch application to new collection
           → Delete old collection
        c) Accept the performance hit and add more shards
           → Doesn't fix jumbo chunks but adds capacity
        d) Live with it and plan for next major version
    LESSON:
        → The cost of wrong shard key in MongoDB < 5.0 was ENORMOUS
        → Even in 5.0+: resharding is days, not minutes
        → INVEST TIME UPFRONT in the decision
