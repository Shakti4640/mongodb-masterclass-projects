-- Table 1: Orders (header information)
CREATE TABLE store.orders (
    order_id        VARCHAR(20) PRIMARY KEY,
    customer_id     BIGINT NOT NULL,         -- FK to customers
    order_date      TIMESTAMP DEFAULT GETDATE(),
    status          VARCHAR(20) DEFAULT 'processing',
    subtotal        DECIMAL(10,2),
    tax             DECIMAL(10,2),
    shipping_cost   DECIMAL(10,2),
    total           DECIMAL(10,2),
    
    -- Shipping address (flattened into columns)
    ship_street     VARCHAR(255),
    ship_apt        VARCHAR(50),
    ship_city       VARCHAR(100),
    ship_state      VARCHAR(2),
    ship_zip        VARCHAR(10)
)
DISTSTYLE KEY DISTKEY(customer_id) SORTKEY(order_date);

-- Table 2: Order Items (line items — one row per item)
CREATE TABLE store.order_items (
    item_id         BIGINT IDENTITY(1,1),
    order_id        VARCHAR(20) NOT NULL,    -- FK to orders
    product_name    VARCHAR(255),
    quantity        INTEGER,
    unit_price      DECIMAL(10,2),
    line_total      DECIMAL(10,2)
)
DISTSTYLE KEY DISTKEY(order_id) SORTKEY(order_id);
-- DISTKEY on order_id → co-locate items with their order for JOIN

-- Table 3: Payments (one row per payment method used)
CREATE TABLE store.payments (
    payment_id      BIGINT IDENTITY(1,1),
    order_id        VARCHAR(20) NOT NULL,    -- FK to orders
    method          VARCHAR(50),
    card_last_four  VARCHAR(4),
    amount          DECIMAL(10,2),
    status          VARCHAR(20)
)
DISTSTYLE KEY DISTKEY(order_id) SORTKEY(order_id);

-- Optional Table 4: Order Status History
CREATE TABLE store.order_status_history (
    history_id      BIGINT IDENTITY(1,1),
    order_id        VARCHAR(20) NOT NULL,
    status          VARCHAR(20),
    changed_at      TIMESTAMP DEFAULT GETDATE(),
    changed_by      VARCHAR(100)
)
DISTSTYLE KEY DISTKEY(order_id) SORTKEY(changed_at);

The Cost of Normalization

REDSHIFT NORMALIZED APPROACH — THE COSTS:

    DESIGN TIME:
    → 4 tables to design, create, maintain
    → Distribution keys must align for co-located JOINs
    → Sort keys chosen for each table
    → Referential integrity is YOUR responsibility (no FK enforcement)

    WRITE TIME:
    → 4 INSERT statements for one order
    → Must be wrapped in a transaction
    → 4 tables → 4 sets of columnar blocks updated
    → 4 sets of zone maps updated
    → 4 VACUUM operations needed eventually

    READ TIME:
    → 3-way JOIN to reconstruct one order
    → Even with co-located distribution: JOIN overhead
    → Result has redundant data (order header repeated per item)
    → Application must reshape the flat rows into a hierarchical object

    MAINTENANCE TIME:
    → Schema changes affect multiple tables
    → Adding "gift_wrap" to order items → ALTER TABLE
    → Adding "loyalty_points_used" to payment → ALTER TABLE
    → Each change requires DDL on the specific table

WHY REDSHIFT DOES IT THIS WAY:
    → Normalization eliminates data duplication → saves columnar storage
    → JOINs are what SQL databases DO
    → Analytical queries benefit: "total revenue per product" is easy
    → GROUP BY across items, SUM across payments — powerful
    
BUT FOR THE APPLICATION:
    → "Show me order ORD-2024-1001" requires 3 JOINs
    → Every page load for an order = multi-table query
    → High latency for a simple "display this order" operation

🟢 MONGODB MINDSET — How This Same Problem Is Approached

The Core Philosophy

MONGODB'S FUNDAMENTAL PRINCIPLE:

    "Data that is accessed together should be stored together"
    
    When does the application need order data?
    → "Show me order ORD-2024-1001"
    → It needs: header + items + address + payment — ALL AT ONCE
    
    Therefore: store ALL of it in ONE document
    
    This is called EMBEDDING (or denormalization)

🧩 HOW TO APPROACH IT IN MONGODB

THE EMBEDDING DECISION FRAMEWORK:

    For EVERY piece of related data, ask these questions:
    
    QUESTION 1: "Is this data accessed TOGETHER with the parent?"
        YES → Embed (items are ALWAYS shown with the order)
        NO  → Reference (product catalog is independent of orders)
    
    QUESTION 2: "Does this data BELONG TO the parent exclusively?"
        YES → Embed (this shipping address is FOR this order only)
        NO  → Reference (customer profile is shared across orders)
    
    QUESTION 3: "Is this data BOUNDED in size?"
        YES → Embed (an order has 1-50 items — bounded)
        NO  → Reference (a product could have millions of reviews — unbounded)
    
    QUESTION 4: "Does this data change INDEPENDENTLY?"
        YES → Reference (product price changes independently of past orders)
        NO  → Embed (order items don't change after order is placed)

    APPLYING TO OUR ORDER:
    ┌─────────────────────────┬───────────┬──────────────────────────────┐
    │ DATA                     │ DECISION   │ REASONING                     │
    ├─────────────────────────┼───────────┼──────────────────────────────┤
    │ Order items              │ EMBED      │ Always accessed with order    │
    │                          │            │ Belongs to this order only    │
    │                          │            │ Bounded (1-50 items)          │
    │                          │            │ Don't change independently    │
    ├─────────────────────────┼───────────┼──────────────────────────────┤
    │ Shipping address         │ EMBED      │ Accessed with order           │
    │                          │            │ Snapshot at order time        │
    │                          │            │ One address per order         │
    │                          │            │ Doesn't change after order    │
    ├─────────────────────────┼───────────┼──────────────────────────────┤
    │ Payment info             │ EMBED      │ Accessed with order           │
    │                          │            │ Belongs to this order         │
    │                          │            │ 1-2 payments per order        │
    │                          │            │ Doesn't change after charge   │
    ├─────────────────────────┼───────────┼──────────────────────────────┤
    │ Status history           │ EMBED      │ Accessed with order           │
    │                          │            │ Belongs to this order only    │
    │                          │            │ Bounded (5-10 status changes) │
    │                          │            │ Append-only                   │
    ├─────────────────────────┼───────────┼──────────────────────────────┤
    │ Customer reference       │ PARTIAL    │ Customer ID + name embedded   │
    │                          │ EMBED      │ (snapshot for display)        │
    │                          │            │ Full profile NOT embedded     │
    │                          │            │ (changes independently)       │
    ├─────────────────────────┼───────────┼──────────────────────────────┤
    │ Product catalog details  │ REFERENCE  │ Product details change        │
    │ (full description, etc.) │            │ independently of orders       │
    │                          │            │ Shared across all orders      │
    │                          │            │ Only SKU + name + price       │
    │                          │            │ embedded (snapshot at order)  │
    └─────────────────────────┴───────────┴──────────────────────────────┘

    NOTICE THE "SNAPSHOT" CONCEPT:
    → We embed the PRODUCT NAME and PRICE at order time
    → Even if the product price changes later, the ORDER keeps the original price
    → This is CORRECT business logic — the order records what was charged
    → In Redshift: you'd JOIN to products table → shows CURRENT price (wrong!)
    → Or: store price in order_items table (denormalization in Redshift too!)
    → MongoDB makes this natural — embed the snapshot

def create_order(customer_info, items, shipping_address, payment_method):
    """Create a new order with all embedded data."""
    subtotal = sum(item["unit_price"] * item["quantity"] for item in items)
    tax = round(subtotal * 0.08, 2)
    shipping = 0 if subtotal > 100 else 9.99
    total = round(subtotal + tax + shipping, 2)
    
    for item in items:
        item["line_total"] = round(item["unit_price"] * item["quantity"], 2)
    
    order = {
        "_id": f"ORD-{datetime.now().strftime('%Y')}-{orders.count_documents({}) + 1001:04d}",
        "customer": customer_info,
        "items": items,
        "shipping_address": shipping_address,
        "payment": {
            "method": payment_method["method"],
            "card_last_four": payment_method["card_last_four"],
            "amount": total,
            "status": "charged"
        },
        "pricing": {
            "subtotal": round(subtotal, 2),
            "tax": tax,
            "shipping": shipping,
            "total": total
        },
        "status": "processing",
        "status_history": [{
            "status": "processing",
            "changed_at": datetime.now(),
            "changed_by": "system"
        }],
        "order_date": datetime.now(),
        "updated_at": datetime.now()
    }
    
    result = orders.insert_one(order)
    return result.inserted_id


def update_order_status(order_id, new_status, changed_by, note=None):
    """Update order status and append to history."""
    history_entry = {
        "status": new_status,
        "changed_at": datetime.now(),
        "changed_by": changed_by
    }
    if note:
        history_entry["note"] = note
    
    result = orders.update_one(
        {"_id": order_id},
        {
            "$set": {"status": new_status, "updated_at": datetime.now()},
            "$push": {"status_history": history_entry}
        }
    )
    return result.modified_count

    ✅ EMBED DATA THAT IS ACCESSED TOGETHER
       → Order items belong with the order
       → Shipping address belongs with the order
       → One read returns everything the application needs
    
    ✅ USE SNAPSHOT EMBEDDING FOR REFERENCED ENTITIES
       → Embed customer NAME in order (not full profile)
       → Embed product NAME and PRICE in order items (not full catalog)
       → Snapshots capture point-in-time values (correct for orders)
    
    ✅ USE ARRAYS FOR ONE-TO-MANY WITHIN THE DOCUMENT
       → items: [...] for order line items
       → status_history: [...] for status changes
       → Arrays are first-class in MongoDB — leverage them
    
    ✅ USE CUSTOM _id WHEN YOU HAVE A NATURAL KEY
       → _id: "ORD-2024-1001" instead of auto-generated ObjectId
       → More readable in queries and logs
       → Still unique, still indexed automatically
       → Redshift equivalent: using business key as PRIMARY KEY
    
    ✅ PRE-CALCULATE VALUES STORED IN THE DOCUMENT
       → line_total = quantity × unit_price (calculated before insert)
       → pricing.total = subtotal + tax + shipping (calculated before insert)
       → Avoids runtime computation on every read
       → Redshift: you'd compute in SQL queries — MongoDB: compute on write

REDSHIFT HABITS TO UNLEARN:
    🔄 "I need separate tables for each entity"
       → Embed related data in one document
       → Think "what does the application need?" not "what are the entities?"
    
    🔄 "Duplication is bad"
       → Strategic duplication (snapshots) is GOOD in MongoDB
       → Customer name in order = correct point-in-time record
       → Product price in order item = historical accuracy
       → Redshift normalizes to avoid duplication → requires JOINs
       → MongoDB denormalizes intentionally → eliminates JOINs
       → Trade-off: more storage vs faster reads + simpler code
    
    🔄 "JOINs are how I connect related data"
       → Embedding IS the connection in MongoDB
       → Data that belongs together lives together
       → $lookup exists (Project 13) but should be the EXCEPTION
       → If you're using $lookup frequently → your document design is wrong
    
    🔄 "I need referential integrity enforced by the database"
       → MongoDB has NO foreign key constraints
       → Embedding provides natural referential integrity
       → Delete a document → all embedded data gone (automatic cascade)
       → For referenced data → enforce in application code
       → This feels unsafe but works well in practice with good document design
    
    🔄 "ALTER TABLE ADD COLUMN for new fields"
       → Just add the field to new documents
       → Old documents don't need the field
       → Want to add "gift_message" to some orders?
         Just include it in those orders — no schema change

❌ WHAT SHOULD NOT DO

ANTI-PATTERNS — ESPECIALLY TRAPS FOR REDSHIFT USERS:

    ❌ EMBEDDING EVERYTHING (the opposite extreme)
       # WRONG — embedding unbounded data
       {
           _id: "customer_001",
           name: "Alice",
           orders: [
               { ... order 1 with items ... },
               { ... order 2 with items ... },
               // ... 10,000 orders over 5 years
               // Document grows to 16MB → FAILS
           ]
       }
       
       # RIGHT — orders as separate collection, reference customer
       # customers collection: { _id: "c1", name: "Alice" }
       # orders collection: { _id: "ORD-001", customer_id: "c1", items: [...] }
       
       RULE: Embed BOUNDED data (1-50 items per order ✅)
             Reference UNBOUNDED data (unlimited orders per customer ❌)
       
       THE 16MB DOCUMENT LIMIT IS YOUR GUARDRAIL
       If you're approaching it → your design is wrong

    ❌ CREATING SEPARATE COLLECTIONS FOR EVERYTHING (Redshift reflex)
       # WRONG — mimicking Redshift normalized tables
       db.orders.insertOne({ order_id: "ORD-001", customer_id: 1, total: 214 })
       db.order_items.insertOne({ order_id: "ORD-001", sku: "WM-001", qty: 2 })
       db.order_items.insertOne({ order_id: "ORD-001", sku: "MK-001", qty: 1 })
       db.order_payments.insertOne({ order_id: "ORD-001", method: "visa" })
       db.order_addresses.insertOne({ order_id: "ORD-001", city: "Denver" })
       
       → You just recreated Redshift's normalized model in MongoDB
       → Now you need $lookup (JOIN) to reconstruct the order
       → You've lost every advantage MongoDB offers
       → THIS IS THE #1 MISTAKE REDSHIFT DEVELOPERS MAKE IN MONGODB
       
       # RIGHT — one document per order with embedded data
       db.orders.insertOne({
           _id: "ORD-001",
           customer: { name: "Alice" },
           items: [ { sku: "WM-001", qty: 2 }, { sku: "MK-001", qty: 1 } ],
           payment: { method: "visa" },
           shipping_address: { city: "Denver" }
       })

    ❌ EMBEDDING DATA THAT CHANGES INDEPENDENTLY
       # WRONG — embedding full product catalog in order
       {
           items: [{
               sku: "WM-001",
               product_name: "Wireless Mouse",
               price: 29.99,
               // DON'T embed these — they change independently:
               description: "Ergonomic wireless mouse with 12-month battery...",
               manufacturer: "Logitech",
               weight: "0.15 kg",
               dimensions: "10.5 x 6.1 x 3.8 cm",
               warranty: "2 years",
               category_path: ["Electronics", "Peripherals", "Mice"],
               reviews_count: 4523,        // changes constantly
               average_rating: 4.7,        // changes constantly
               current_stock: 1547         // changes constantly
           }]
       }
       
       # RIGHT — embed only the SNAPSHOT needed for this order
       {
           items: [{
               sku: "WM-001",
               product_name: "Wireless Mouse",
               unit_price: 29.99,          // price AT TIME OF ORDER
               quantity: 2,
               line_total: 59.98
           }]
       }
       
       → Embed: name + price (snapshot for order record)
       → Reference: full product details (look up from products collection)
       → Reviews, stock, ratings change constantly → NEVER embed in orders

    ❌ DEEPLY NESTING MORE THAN 3-4 LEVELS
       # PROBLEMATIC — too deep
       {
           order: {
               customer: {
                   address: {
                       location: {
                           coordinates: {
                               latitude: 39.7392
                           }
                       }
                   }
               }
           }
       }
       
       # BETTER — flatten where possible
       {
           customer_name: "Alice",
           shipping_address: {
               city: "Denver",
               state: "CO",
               coordinates: [39.7392, -104.9903]
           }
       }
       
       WHY: Deep nesting makes queries verbose:
       "order.customer.address.location.coordinates.latitude"
       vs: "shipping_address.coordinates"
       
       → Dot notation queries get unwieldy past 3 levels
       → Index definitions become long and error-prone
       → Keep nesting meaningful but not excessive

    ❌ USING ARRAYS WHERE OBJECTS ARE BETTER
       # WRONG — array for single-valued data
       {
           payment: [
               { method: "visa", amount: 214.04 }    // only ever one payment
           ]
       }
       
       # RIGHT — use object for single-valued data
       {
           payment: { method: "visa", amount: 214.04 }
       }
       
       # RIGHT — use array for multi-valued data
       {
           items: [                                    // multiple items per order
               { sku: "WM-001", qty: 2 },
               { sku: "MK-001", qty: 1 }
           ]
       }
       
       RULE: Array = zero-to-many | Object = exactly one

    ❌ FORGETTING TO HANDLE ARRAY GROWTH
       → $push to status_history without $slice → unbounded growth
       → An order that gets updated 1000 times → huge document
       
       # ALWAYS use $slice for potentially growing arrays
       { "$push": {
           "status_history": {
               "$each": [new_entry],
               "$slice": -50            // keep last 50 entries max
           }
       }}

📍 WHEN TO USE MONGODB OVER REDSHIFT

FOR HIERARCHICAL/NESTED DATA:

USE MONGODB WHEN:
    → Data is naturally hierarchical (orders → items → options)
    → Application reads complete entities (full order in one call)
    → Entities have varying structure (different item types)
    → Real-time single-entity operations (view/edit one order)
    → High concurrency on different entities
    → Schema evolves frequently (new fields added to orders)
    → Embedded data has bounded growth

📍 WHEN TO STICK WITH REDSHIFT INSTEAD

FOR THE SAME DATA:

USE REDSHIFT WHEN:
    → Primary use case is cross-entity analytics:
      "Total revenue per product per month across all orders"
      "Average order value by customer segment"
      "Top 10 products by quantity sold"
    → Need to JOIN orders with external datasets (marketing, shipping logs)
    → Need window functions across orders (running totals, rankings)
    → Data pipeline feeds from many sources into analytical models
    → Reporting tools expect flat table structure (Tableau, Looker)

MongoDB Nested Document Storage — BSON Internals

HOW NESTED DOCUMENTS ARE STORED IN BSON:

    DOCUMENT:
    {
        _id: "ORD-2024-1001",
        customer: { name: "Alice", email: "alice@ex.com" },
        items: [
            { sku: "WM-001", qty: 2, price: 29.99 },
            { sku: "MK-001", qty: 1, price: 89.99 }
        ]
    }
    
    BSON BINARY LAYOUT (simplified):
    ┌─────────────────────────────────────────────────────────────┐
    │ [total_doc_size: 4 bytes]                                    │
    │                                                              │
    │ [type: 0x02][_id\0]["ORD-2024-1001"]                        │  ← string field
    │                                                              │
    │ [type: 0x03][customer\0]                                     │  ← embedded doc start
    │   [sub_doc_size: 4 bytes]                                    │
    │   [type: 0x02][name\0]["Alice"]                              │
    │   [type: 0x02][email\0]["alice@ex.com"]                      │
    │   [0x00]                                                     │  ← embedded doc end
    │                                                              │
    │ [type: 0x04][items\0]                                        │  ← array start
    │   [array_size: 4 bytes]                                      │
    │   [type: 0x03][0\0]                                          │  ← array index 0
    │     [sub_doc_size]                                           │
    │     [type: 0x02][sku\0]["WM-001"]                            │
    │     [type: 0x10][qty\0][2]                                   │  ← int32
    │     [type: 0x01][price\0][29.99]                             │  ← double
    │     [0x00]                                                   │  ← end of item 0
    │   [type: 0x03][1\0]                                          │  ← array index 1
    │     [sub_doc_size]                                           │
    │     [type: 0x02][sku\0]["MK-001"]                            │
    │     [type: 0x10][qty\0][1]                                   │
    │     [type: 0x01][price\0][89.99]                             │
    │     [0x00]                                                   │  ← end of item 1
    │   [0x00]                                                     │  ← array end
    │                                                              │
    │ [0x00]                                                       │  ← document end
    └─────────────────────────────────────────────────────────────┘

    KEY INSIGHTS:
    
    1. ENTIRE DOCUMENT IS ONE CONTIGUOUS BSON BLOB
       → customer, items, everything = one binary block
       → Stored in ONE B-tree leaf page in WiredTiger
       → Reading the order = reading ONE block (sequential I/O)
       → FAST for "get the whole order"
    
    2. ARRAYS ARE STORED AS INDEXED DOCUMENTS
       → items[0], items[1] stored with index keys "0", "1"
       → This is why you can query "items.0.sku" (first item's sku)
       → Array indexes are strings internally (not integers)
    
    3. NESTED DOCUMENTS ARE RECURSIVE BSON
       → customer is type 0x03 (embedded document)
       → Contains its own sub-document with its own fields
       → Dot notation traverses these nested structures
       → "customer.name" → enter customer sub-doc → find name field

Document Size and the 16MB Limit

THE 16MB DOCUMENT SIZE LIMIT:

    MongoDB documents cannot exceed 16MB (16,777,216 bytes)
    
    FOR OUR ORDER DOCUMENT:
    → Order header: ~200 bytes
    → Each item: ~100-200 bytes
    → Shipping address: ~150 bytes
    → Payment: ~100 bytes
    → Status history entry: ~100 bytes
    → Total for typical order: ~1-2 KB
    
    HOW MANY ITEMS BEFORE 16MB?
    → ~100-200 bytes per item
    → 16MB / 200 bytes = ~80,000 items per order
    → Real-world orders: 1-50 items → NOWHERE NEAR the limit
    → This is why items are safe to embed
    
    WHAT APPROACHES THE LIMIT?
    → Embedding ALL orders in a customer document (unbounded)
    → Embedding ALL reviews in a product document (unbounded)
    → Embedding ALL log entries in a session document (unbounded)
    → Storing large binary data (images, files) in a document
    
    SOLUTIONS FOR LARGE DATA:
    → Bucket Pattern (Project 16): group into fixed-size chunks
    → Outlier Pattern (Project 17): overflow into linked documents
    → GridFS: MongoDB's file storage system for files > 16MB
    → Reference Pattern (Project 14): separate collection for large arrays
    
    REDSHIFT COMPARISON:
    → Redshift: no per-row size limit (columns stored separately)
    → But: VARCHAR max 65,535 bytes, SUPER max 16MB
    → Redshift limits are per-column, not per-row
    → MongoDB limit is per-document (entire entity)

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 6:

PITFALL 1: "Normalizing in MongoDB like Redshift" (THE #1 MISTAKE)
    REDSHIFT TRAINED YOU: One entity per table, JOINs connect them
    MONGODB TRAP: Creating order_items, payments, addresses collections
    CONSEQUENCE: Need $lookup everywhere, lose all embedding advantages
    FIX: Embed bounded, co-accessed, owned data inside the parent document
    SEVERITY: CRITICAL — defeats the entire purpose of using MongoDB

PITFALL 2: "Embedding everything in one mega-document"
    REACTION TO PITFALL 1: "Okay, I'll embed EVERYTHING!"
    MONGODB TRAP: All customer orders embedded in customer document
    CONSEQUENCE: Document grows unbounded → hits 16MB limit
    FIX: Embed bounded data, reference unbounded data
    SEVERITY: CRITICAL — application crashes when limit reached

PITFALL 3: "Not understanding the snapshot concept"
    REDSHIFT TRAINED YOU: JOIN to get current product price
    MONGODB TRAP: Referencing product collection for price in orders
    CONSEQUENCE: "What was the price when ordered?" becomes impossible
    FIX: Embed price snapshot in order item (point-in-time value)
    SEVERITY: HIGH — incorrect business logic (showing current vs historical price)

PITFALL 4: "Using $lookup for every query" 
    REDSHIFT TRAINED YOU: JOINs are natural and efficient
    MONGODB TRAP: Using $lookup like SQL JOINs on every query
    CONSEQUENCE: $lookup is MUCH slower than SQL JOINs in Redshift
                 MongoDB isn't optimized for cross-collection operations
    FIX: Redesign documents to eliminate $lookup need
         If you need $lookup frequently → your document model is wrong
    SEVERITY: HIGH — severe performance degradation

PITFALL 5: "Flat document structure (no nesting)"
    REDSHIFT TRAINED YOU: All columns at one level (flat row)
    MONGODB TRAP: Storing address fields at top level instead of nested
    { street: "...", city: "...", state: "...", zip: "..." }
    
    BETTER: Group related fields as nested object
    { shipping_address: { street: "...", city: "...", state: "...", zip: "..." } }
    
    WHY: Nested objects are logically grouped
         $set: { shipping_address: entire_new_address } updates all at once
         Cleaner document structure, easier to reason about
    SEVERITY: LOW — works but messy, harder to maintain

PITFALL 6: "Expecting queries on nested arrays to behave like SQL"
    // Find orders where item price > 50 AND quantity > 1
    
    // WRONG — could match different array elements
    db.orders.find({
        "items.unit_price": { $gt: 50 },
        "items.quantity": { $gt: 1 }
    })
    // Could match: price>50 on item A, quantity>1 on item B → false positive
    
    // RIGHT — use $elemMatch for same-element conditions
    db.orders.find({
        items: { $elemMatch: { unit_price: { $gt: 50 }, quantity: { $gt: 1 } } }
    })
    
    SEVERITY: HIGH — silently returns incorrect results

📐 DATA MODELING COMPARISON — The Deep Dive

THE SPECTRUM FROM FULLY NORMALIZED TO FULLY EMBEDDED:

LEVEL 1: FULLY NORMALIZED (Redshift default)
    5 collections (mimicking Redshift tables)
    → Maximum data integrity
    → Maximum JOIN overhead
    → Maximum write complexity
    → WRONG for MongoDB

LEVEL 2: REFERENCED (MongoDB with ObjectId references)
    orders: { customer_id: ObjectId("..."), items: [item_id, item_id] }
    → Like Redshift FKs but without enforcement
    → Still needs $lookup (JOIN) for complete reads
    → Appropriate for: truly independent entities

LEVEL 3: PARTIALLY EMBEDDED (RECOMMENDED for most cases)
    orders: {
        customer: { id: ObjectId("..."), name: "Alice" },  ← snapshot
        items: [ { sku, name, price, qty } ],                ← embedded
        shipping_address: { ... },                            ← embedded
        payment: { ... }                                      ← embedded
    }
    → Best balance of read performance and data consistency
    → Snapshots for display, references for full lookups
    → THIS IS WHAT WE BUILT IN THIS PROJECT

LEVEL 4: FULLY EMBEDDED (everything in one document)
    customers: {
        name: "Alice",
        orders: [ { items: [...], payment: {...} }, ... ],  ← ALL orders embedded
        reviews: [ ... ],
        activity_log: [ ... ]
    }
    → Fastest reads (everything in one document)
    → Dangerous: unbounded growth → 16MB limit
    → WRONG for most cases

THE RIGHT ANSWER IS ALMOST ALWAYS LEVEL 3
    → Embed what's accessed together and bounded
    → Reference what's independent or unbounded
    → Snapshot for display, reference for full lookup
    → This is the MongoDB sweet spot
