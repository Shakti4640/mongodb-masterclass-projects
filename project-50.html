🔴 REDSHIFT MINDSET — The "Everything in Redshift" Temptation

YOUR INITIAL INSTINCT (before this guide):

    "I know Redshift. Let me put everything there."
    
    ATTEMPT: Redshift as the center of everything
    
    ┌───────────────────────────────────────────────────┐
    │              REDSHIFT-CENTRIC DESIGN               │
    │                                                     │
    │  Application → Redshift → Analytics                 │
    │                    ↑                                │
    │              Everything in one place                 │
    └───────────────────────────────────────────────────┘
    
    WHERE THIS BREAKS:
    
    ❌ Transaction processing at 50ms?
       → Redshift: single-row operations take 50-500ms minimum
       → Leader node overhead, columnar reconstruction, WLM queuing
       → At 500K transactions/day = ~6 TPS average, 60 TPS peak
       → Redshift WLM has ~50 concurrent query slots
       → IMPOSSIBLE to serve real-time operations from Redshift
    
    ❌ 50K API requests/sec?
       → Redshift max connections: 500
       → Even with connection pooling: maybe 200 concurrent queries
       → Each query: 50-500ms minimum
       → Throughput ceiling: ~400-4000 requests/sec MAXIMUM
       → Need 50,000 → 10x+ beyond Redshift's capability
    
    ❌ Real-time fraud detection at <100ms?
       → Redshift query latency floor: ~50ms (trivial queries)
       → Complex model scoring: 500ms-5s
       → Plus network round-trip to Redshift
       → CANNOT meet 100ms SLA
    
    ❌ Flexible merchant schemas?
       → Each merchant type has different fields
       → ALTER TABLE for every new merchant type?
       → Wide sparse table with hundreds of NULLable columns?
       → Schema migration = downtime + VACUUM
    
    ❌ Multi-region with <50ms reads?
       → Redshift is single-region (even RA3)
       → Cross-region data sharing: read-only, with latency
       → Cannot serve local reads to US + EU + APAC simultaneously
    
    CONCLUSION: Redshift alone CANNOT solve this problem
    But Redshift is ESSENTIAL for the analytical portion
    
    YOU NEED A POLYGLOT DATA ARCHITECTURE

🟢 MONGODB MINDSET — The "Everything in MongoDB" Temptation

YOUR NEW INSTINCT (after 49 projects):

    "MongoDB is amazing. Let me put everything there."
    
    ATTEMPT: MongoDB as the center of everything
    
    ┌───────────────────────────────────────────────────┐
    │              MONGODB-CENTRIC DESIGN                 │
    │                                                     │
    │  Application → MongoDB → Analytics via Aggregation  │
    │                    ↑                                │
    │              Everything in one place                 │
    └───────────────────────────────────────────────────┘
    
    WHERE THIS BREAKS:
    
    ❌ Revenue reporting across 500M transactions over 3 years?
       → MongoDB aggregation on 500M documents: minutes to hours
       → No columnar compression: scans entire documents
       → RAM-intensive: aggregation pipeline memory limits
       → Redshift: same query in 5-30 seconds (columnar + MPP)
    
    ❌ Complex multi-table JOINs for financial reconciliation?
       → 10-way $lookup pipeline: terrible performance
       → No cost-based query optimizer for complex joins
       → Redshift: optimized JOIN execution with distribution keys
    
    ❌ 200 analysts running ad-hoc SQL?
       → Analysts know SQL, not MQL
       → BI tools (Tableau, Looker) expect SQL interfaces
       → MongoDB's Atlas SQL interface: limited, not full SQL
       → Redshift: native SQL, full BI tool ecosystem
    
    ❌ Window functions for cohort analysis?
       → MongoDB $setWindowFields works but limited
       → Complex multi-level window functions: painful in MQL
       → Redshift: RANK, DENSE_RANK, LAG, LEAD, NTILE natively
    
    ❌ Petabyte-scale historical analysis?
       → MongoDB storage costs: ~$0.25/GB/month (Atlas)
       → Redshift RA3 + Spectrum: ~$0.023/GB/month on S3
       → 10x cost difference for cold analytical data
    
    CONCLUSION: MongoDB alone CANNOT solve this problem
    But MongoDB is ESSENTIAL for the operational portion
    
    YOU NEED BOTH — AND MORE

    # ─── TRANSACTION PROCESSING ───────────────────────
    def process_payment(self, user_id, merchant_id,
                        amount, currency, items):
        """
        Process a payment transaction
        
        Uses: single-document atomicity (Project 27)
        + multi-document transaction for balance update (Project 28)
        
        Redshift equivalent: this would be an INSERT + UPDATE
        in a SQL transaction — but at 50ms, not 500ms
        """
        
        # ── Prepare transaction document ──────────────
        # (Project 6 — nested documents)
        # (Project 19 — subset pattern for merchant)
        merchant = self.merchants.find_one(
            {"_id": merchant_id},
            {"name": 1, "category": 1}     # subset projection
        )
        
        txn_doc = {
            "user_id": user_id,
            "type": "payment",
            "amount": Decimal128(str(amount)),
            "currency": currency,
            "status": "pending",
            "merchant": {                    # embedded subset
                "id": merchant_id,
                "name": merchant["name"],
                "category": merchant.get("category", "other")
            },
            "items": items,                  # embedded array
            "fraud_score": None,             # set by Flink async
            "created_at": datetime.utcnow(),
            "updated_at": datetime.utcnow()
        }
        
        # ── Multi-document transaction ────────────────
        # (Project 28 — when atomicity across docs needed)
        with self.client.start_session() as session:
            with session.start_transaction():
                # Insert transaction
                result = self.transactions.insert_one(
                    txn_doc, session=session
                )
                txn_id = result.inserted_id
                
                # Update user's last_transaction timestamp
                self.users.update_one(
                    {"_id": user_id},
                    {
                        "$set": {
                            "last_transaction_at": datetime.utcnow()
                        },
                        "$inc": {
                            "transaction_count": 1,     # computed pattern (P20)
                            "total_spent": float(amount)
                        }
                    },
                    session=session
                )
                
                # Write audit trail (immutable)
                self.audit.insert_one({
                    "entity_type": "transaction",
                    "entity_id": txn_id,
                    "action": "created",
                    "actor_id": user_id,
                    "actor_type": "user",
                    "details": {
                        "amount": Decimal128(str(amount)),
                        "currency": currency,
                        "merchant_id": merchant_id
                    },
                    "created_at": datetime.utcnow()
                }, session=session)
        
        return txn_id
    
    # ─── REAL-TIME FRAUD SCORE UPDATE ─────────────────
    def update_fraud_score(self, txn_id, score):
        """
        Called by Kafka consumer when Flink produces fraud score
        
        This is the MongoDB Sink Connector writing back
        from the streaming layer (Project 44)
        """
        self.transactions.update_one(
            {"_id": txn_id},
            {
                "$set": {
                    "fraud_score": score,
                    "updated_at": datetime.utcnow()
                }
            }
        )
        
        # If high fraud score, auto-flag for review
        if score > 0.85:
            self.transactions.update_one(
                {"_id": txn_id},
                {"$set": {"status": "review"}}
            )
            
            # Audit the flag
            self.audit.insert_one({
                "entity_type": "transaction",
                "entity_id": txn_id,
                "action": "flagged_fraud",
                "actor_type": "system",
                "details": {"fraud_score": score},
                "created_at": datetime.utcnow()
            })

    def onboard_merchant(self, merchant_data):
        """
        Different merchant types have different fields
        
        MongoDB's polymorphic pattern handles this naturally
        Redshift would need: wide table OR multiple tables
        """
        base = {
            "name": merchant_data["name"],
            "type": merchant_data["type"],
            "region": merchant_data["region"],
            "status": "pending_review",
            "created_at": datetime.utcnow(),
            "stats": {                       # computed pattern (P20)
                "total_transactions": 0,
                "total_revenue": Decimal128("0"),
                "avg_transaction": Decimal128("0")
            }
        }
        
        # Type-specific fields (polymorphic)
        if merchant_data["type"] == "restaurant":
            base["cuisine"] = merchant_data.get("cuisine")
            base["delivery"] = merchant_data.get("delivery", False)
            base["seating_capacity"] = merchant_data.get("seats")
        
        elif merchant_data["type"] == "ecommerce":
            base["website"] = merchant_data.get("website")
            base["shipping_countries"] = merchant_data.get(
                "shipping", [])
            base["return_policy_days"] = merchant_data.get(
                "return_days", 30)
        
        elif merchant_data["type"] == "subscription":
            base["billing_cycles"] = merchant_data.get(
                "cycles", ["monthly", "annual"])
            base["trial_days"] = merchant_data.get("trial", 14)
            base["churn_rate"] = None
        
        # All types get inserted into SAME collection
        # No ALTER TABLE, no schema migration, no downtime
        result = self.merchants.insert_one(base)
        return result.inserted_id

    def bronze_to_silver_transactions(self, date_str):
        """
        Transform raw CDC events into clean transaction table
        
        SOURCE: s3://novapay-datalake/bronze/novapay.novapay.transactions/
        TARGET: Iceberg table novapay_catalog.silver.transactions
        
        TRANSFORMATION:
        MongoDB document (nested, flexible)
            → Flat columnar table (typed, validated)
        """
        
        # ── Read Bronze (raw CDC from Kafka S3 Sink) ──
        bronze_df = (self.spark.read
            .parquet(
                f"s3://novapay-datalake/bronze/"
                f"novapay.novapay.transactions/"
                f"year={date_str[:4]}/"
                f"month={date_str[5:7]}/"
                f"day={date_str[8:10]}/")
        )
        
        # ── Transform: flatten nested documents ───────
        # MongoDB nested document → flat columns
        silver_df = (bronze_df
            # Extract top-level fields
            .select(
                F.col("_id").alias("transaction_id"),
                F.col("user_id"),
                F.col("type").alias("txn_type"),
                
                # Decimal128 → Decimal (Redshift-compatible)
                F.col("amount.$numberDecimal")
                    .cast(DecimalType(18, 2))
                    .alias("amount"),
                
                F.col("currency"),
                F.col("status"),
                
                # Flatten embedded merchant (Project 6 → flat)
                F.col("merchant.id").alias("merchant_id"),
                F.col("merchant.name").alias("merchant_name"),
                F.col("merchant.category")
                    .alias("merchant_category"),
                
                F.col("fraud_score")
                    .cast(DecimalType(5, 4))
                    .alias("fraud_score"),
                
                # Timestamps
                F.col("created_at.$date")
                    .cast(TimestampType())
                    .alias("created_at"),
                F.col("updated_at.$date")
                    .cast(TimestampType())
                    .alias("updated_at"),
                
                # CDC metadata
                F.col("_cdc_operation"),
                F.col("_cdc_timestamp")
            )
            
            # Deduplicate: keep latest version per transaction_id
            # (CDC may have multiple updates for same document)
            .withColumn("row_num", F.row_number().over(
                F.Window
                .partitionBy("transaction_id")
                .orderBy(F.col("_cdc_timestamp").desc())
            ))
            .filter(F.col("row_num") == 1)
            .drop("row_num")
            
            # Filter out deletes (keep only active records)
            .filter(F.col("_cdc_operation") != "delete")
            
            # Add derived columns
            .withColumn("transaction_date",
                        F.to_date("created_at"))
            .withColumn("transaction_hour",
                        F.hour("created_at"))
            .withColumn("region",
                        F.lit("US"))  # derived from user lookup
            
            # Data quality checks
            .filter(F.col("amount").isNotNull())
            .filter(F.col("amount") > 0)
            .filter(F.col("user_id").isNotNull())
        )
        
        # ── Write to Silver (Iceberg table) ───────────
        (silver_df
            .writeTo("novapay_catalog.silver.transactions")
            .option("merge-schema", "true")  # handle new fields
            .partitionedBy(
                F.days("transaction_date"),
                F.bucket(16, "user_id")
            )
            .append()  # or .overwritePartitions() for reprocessing
        )
        
        row_count = silver_df.count()
        print(f"✅ Silver transactions: {row_count} rows "
              f"for {date_str}")
        return row_count
    
    def silver_to_gold_daily_aggregates(self, date_str):
        """
        Create Gold layer aggregates for Redshift consumption
        
        These aggregates are what analysts actually query
        Pre-computed in Spark → loaded to Redshift → fast BI queries
        """
        
        silver = self.spark.table(
            "novapay_catalog.silver.transactions"
        ).filter(F.col("transaction_date") == date_str)
        
        # ── Daily revenue by merchant ─────────────────
        daily_revenue = (silver

            .groupBy(
                "transaction_date",
                "merchant_id",
                "merchant_name",
                "merchant_category",
                "currency"
            )
            .agg(
                F.sum("amount").alias("total_revenue"),

    → Still queryable via Atlas Data Federation
    → Reduces MongoDB storage cost by 10x for cold data
    → Hot (MongoDB) → Warm (Atlas Archive/S3) → Cold (Glacier)

    → MongoDB server NEVER sees plaintext

OPTIMIZATION 5: REDSHIFT SERVERLESS FOR VARIABLE WORKLOADS

✅ WHAT SHOULD DO (Architecture Best Practices)

SHOULD DO — ARCHITECTURE LEVEL:

    ✅ SEPARATION OF CONCERNS
       → MongoDB: operational data store (OLTP)
       → Kafka: event streaming backbone
       → S3 + Iceberg: unified data lake
       → Spark/Flink: data processing
       → Redshift: analytical warehouse (OLAP)
       → Each component does ONE thing well

    ✅ SINGLE SOURCE OF TRUTH PER DOMAIN
       → User profiles: MongoDB is source of truth
       → Revenue aggregates: Redshift is source of truth
       → Historical events: S3 data lake is source of truth
       → Real-time features: MongoDB is source of truth
       → Trained ML models: SageMaker is source of truth

    ✅ IDEMPOTENT PIPELINES EVERYWHERE
       → CDC replay must not create duplicates
       → Redshift loads use MERGE (upsert), not INSERT
       → Spark writes use overwritePartitions (date-bounded)
       → Kafka consumers track offsets with exactly-once semantics

    ✅ SCHEMA EVOLUTION AS FIRST-CLASS CONCERN
       → MongoDB adds field → Schema Registry captures change
       → Schema Registry → Kafka consumers handle gracefully
       → S3 Parquet/Iceberg: schema evolution built-in
       → Redshift: automated ALTER TABLE ADD COLUMN in ETL
       → NEVER break downstream when upstream schema changes

    ✅ MONITOR THE ENTIRE PIPELINE (Project 49)
       → Not just MongoDB, not just Redshift
       → DATA FRESHNESS is the #1 metric
       → "How old is the newest Redshift data vs MongoDB?"
       → Alert when freshness > 10 minutes (pipeline stall)

    ✅ COST-TIER YOUR DATA
       → Hot (0-30 days):     MongoDB Atlas (fast, expensive)
       → Warm (1-6 months):   Redshift RA3 (queryable, moderate)
       → Cold (6-24 months):  S3 + Spectrum (cheap, queryable)
       → Archive (2+ years):  S3 Glacier (cheapest, slow retrieval)
       → Project 48 cost model applied at every tier

❌ WHAT SHOULD NOT DO (Architecture Anti-Patterns)

SHOULD NOT DO — MISTAKES THAT KILL ARCHITECTURES:

    ❌ MONGODB AS YOUR ONLY DATABASE
       → "We'll do analytics in MongoDB aggregation pipelines"
       → At 500M documents: aggregation takes hours
       → Analysts can't use Tableau/Looker on MongoDB natively
       → You'll rebuild a warehouse inside MongoDB (badly)

    ❌ REDSHIFT AS YOUR ONLY DATABASE
       → "We'll serve API reads from Redshift"
       → At 50K requests/sec: Redshift collapses
       → Single-row lookups: 50-500ms (unacceptable for APIs)
       → You'll build caching layers to compensate (badly)

    ❌ DIRECT MongoDB-to-Redshift COPY (skipping Kafka/S3)
       → "We'll just mongoexport → S3 → COPY every hour"
       → Misses incremental changes between exports
       → Full exports are expensive at scale
       → No replay capability if load fails
       → No other consumers can use the data stream

    ❌ TIGHT COUPLING BETWEEN LAYERS
       → MongoDB schema change breaks Redshift COPY → outage
       → Must have Schema Registry as buffer
       → Must have dead letter queues for bad records
       → Must have data quality checks at every boundary

    ❌ PREMATURE OPTIMIZATION
       → Don't shard MongoDB until you need to (>1TB or >10K ops/s)
       → Don't add Spark until simple transforms outgrow Kafka Connect
       → Don't add Flink until batch processing can't meet latency SLA
       → Start simple → add complexity only when proven necessary

    ❌ IGNORING DATA GRAVITY
       → Data has gravity: once stored, it attracts processing
       → Don't copy data to 5 places — route processing TO the data
       → S3 is the gravitational center (cheapest storage)
       → Bring compute to S3 (Spark, Athena, Spectrum, Flink)
       → Only copy to Redshift what analysts ACTIVELY query

📍 THE ULTIMATE DECISION FRAMEWORK

WHEN TO USE MONGODB:
    ✅ Application-facing data store (APIs, web, mobile)
    ✅ Flexible/evolving schema (product catalogs, user profiles)
    ✅ Single-document operations at <10ms latency
    ✅ High write throughput (>10K writes/sec)
    ✅ Real-time event processing (change streams)
    ✅ Geospatial queries (store locator, delivery routing)
    ✅ Full-text search (product search, document search)
    ✅ Multi-region with local reads/writes
    ✅ Document-oriented data (nested, polymorphic)
    ✅ Horizontal scaling (add shards for capacity)

WHEN TO USE REDSHIFT:
    ✅ Analytical queries (GROUP BY, JOIN, window functions)
    ✅ BI tool serving (Tableau, Looker, QuickSight)
    ✅ SQL-native analysts (200+ users running ad-hoc queries)
    ✅ Historical analysis (years of data, trend detection)
    ✅ Complex multi-table JOINs (star/snowflake schema)
    ✅ Petabyte-scale cold data analysis (via Spectrum)
    ✅ Regulatory reporting (standardized, repeatable)
    ✅ Cost-efficient storage for large analytical datasets
    ✅ Materialized views for dashboard acceleration
    ✅ Data sharing across teams/organizations

WHEN TO USE BOTH:
    ✅ ALWAYS in production systems of meaningful scale
    ✅ Operational + Analytical = different access patterns
    ✅ Real-time serving + batch analysis = different latency needs
    ✅ Application engineers + data analysts = different skill sets
    ✅ OLTP + OLAP = fundamentally different workloads
    ✅ The question is not "which one" but "how they connect"

WHEN TO ADD DATA LAKE (S3 + Iceberg):
    ✅ Data volume exceeds cost-effective Redshift local storage
    ✅ Multiple compute engines need same data (Spark, Athena, RS)
    ✅ Long-term retention requirements (years of history)
    ✅ ML training data needs (Spark/SageMaker read from S3)
    ✅ Schema evolution is frequent (Iceberg handles gracefully)
    ✅ Decoupling storage from compute is a priority

WHEN TO ADD KAFKA:
    ✅ Multiple consumers need the same data stream
    ✅ CDC latency requirement < 15 minutes
    ✅ Event replay capability needed (reprocess historical events)
    ✅ Decoupling producers from consumers (independence)
    ✅ Stream processing (Flink/Spark Streaming) is required
    ✅ Dead letter queue / error handling sophistication needed

⚠️ COMMON PITFALLS AND FAILURE MODES

ARCHITECTURE-LEVEL PITFALLS:

PITFALL 1: "We'll add the data lake later"
    → Start with MongoDB → Redshift direct sync
    → Works at small scale → breaks at medium scale
    → No replay capability, no multi-consumer, no ML access
    FIX: Design with S3 data lake from day 1 — even if simple

PITFALL 2: "Let's normalize everything in MongoDB like Redshift"
    → 15 collections with $lookup JOINs everywhere
    → Performance collapses under load
    → You've built a slow relational database
    FIX: Denormalize in MongoDB, normalize in Redshift
         DIFFERENT models for DIFFERENT purposes

PITFALL 3: "Schema evolution will just work"
    → MongoDB adds field → Kafka serialization breaks
    → Kafka message → Redshift COPY fails (unknown column)
    → Pipeline stalls → data freshness degrades → analysts complain
    FIX: Schema Registry mandatory between every boundary
         Automated ALTER TABLE in Redshift ETL
         Dead letter queues for every connector

PITFALL 4: "We don't need Kafka — MongoDB change streams go direct"
    → Change stream consumer processes + writes to Redshift
    → Consumer crashes → resume token may have expired
    → No replay, no buffering, no multi-consumer
    → Single point of failure in the pipeline
    FIX: Kafka provides durability, replay, fan-out
         Change streams PRODUCE to Kafka, not directly to sinks

PITFALL 5: "Let's put all historical data in MongoDB"
    → 3 years of transactions: 2 billion documents
    → MongoDB storage: $250/TB/month
    → S3 storage: $23/TB/month (10x cheaper)
    → MongoDB working set (cache) cannot fit all this
    → Performance degrades for operational queries
    FIX: Hot in MongoDB (6 months) → Cold in S3 (years)
         Query cold data via Spectrum or Atlas Data Federation

PITFALL 6: "Real-time means everything must be streaming"
    → Flink jobs for simple transformations that batch handles fine
    → Massive Kafka infrastructure for 100 messages/second
    → Complexity explosion with minimal latency benefit
    FIX: Stream only what NEEDS <5 min latency (fraud, notifications)
         Batch everything else (daily aggregates, ML training, reports)

📐 DATA MODELING COMPARISON — Architecture Level

THE SAME DATA — THREE DIFFERENT MODELS:

═══ IN MONGODB (Operational) ═══
{
    _id: ObjectId("..."),
    user_id: ObjectId("..."),
    type: "payment",
    amount: Decimal128("49.99"),
    currency: "USD",
    status: "completed",
    merchant: {                          ← EMBEDDED (denormalized)
        id: ObjectId("..."),
        name: "Coffee Shop",
        category: "food"
    },
    items: [                             ← EMBEDDED ARRAY
        { sku: "LATTE", qty: 2, price: Decimal128("4.99") },
        { sku: "MUFFIN", qty: 1, price: Decimal128("3.49") }
    ],
    fraud_score: 0.02,
    created_at: ISODate("2024-06-15T14:30:22Z")
}

WHY THIS SHAPE:
→ One read returns EVERYTHING needed to display transaction
→ No JOINs needed → fast API response
→ Embedded merchant = no $lookup needed
→ Embedded items = complete transaction in one document
→ Optimized for: "Show me this transaction" (point lookup)


═══ IN S3/ICEBERG (Data Lake) ═══
Parquet file with columnar layout:

transaction_id | user_id | type    | amount | currency | status    |
─────────────────────────────────────────────────────────────────────
abc123         | usr456  | payment | 49.99  | USD      | completed |

merchant_id | merchant_name | merchant_category |
──────────────────────────────────────────────────
mrc789      | Coffee Shop   | food              |

+ Separate nested structure for items:
transaction_id | item_sku | item_qty | item_price |
──────────────────────────────────────────────────
abc123         | LATTE    | 2        | 4.99       |
abc123         | MUFFIN   | 1        | 3.49       |

WHY THIS SHAPE:
→ Columnar format → efficient for analytical scans
→ Partitioned by date → skip irrelevant partitions
→ Schema-on-read → any engine can query
→ Optimized for: "Process all transactions for ML training"


═══ IN REDSHIFT (Analytical) ═══
warehouse.fact_transactions:
    transaction_id  BIGINT    ← DISTKEY: user_id
    user_id         BIGINT    ← SORTKEY: (transaction_date, merchant_id)
    txn_type        VARCHAR
    amount          DECIMAL(18,2)
    currency        VARCHAR(3)
    status          VARCHAR(20)
    merchant_id     BIGINT    ← FK to dim_merchants
    fraud_score     DECIMAL(5,4)
    transaction_date DATE
    created_at      TIMESTAMP

warehouse.fact_transaction_items:   ← SEPARATE TABLE (normalized)
    item_id         BIGINT
    transaction_id  BIGINT    ← FK to fact_transactions
    sku             VARCHAR
    quantity        INTEGER
    price           DECIMAL(10,2)

warehouse.dim_merchants:            ← SEPARATE TABLE (normalized)
    merchant_id     BIGINT
    name            VARCHAR
    category        VARCHAR

WHY THIS SHAPE:
→ Normalized → no data duplication for merchants
→ DISTKEY on user_id → co-locate user's transactions per slice
→ SORTKEY on date → zone maps skip irrelevant blocks
→ Columnar storage → SUM(amount) reads ONLY amount column
→ Optimized for: "Revenue by merchant category for Q2 2024"

    def validate_change_streams(self, timeout_seconds=30):
        """
        Validate Change Streams are working
        (Foundation for Kafka CDC — Project 25)
        """
        print("\n" + "=" * 60)
        print("VALIDATING: Change Streams (CDC Foundation)")
        print("=" * 60)
        
        events_captured = []
        
        # ── Open change stream on transactions ────────
        pipeline = [
            {"$match": {
                "operationType": {"$in": ["insert", "update"]},
                "ns.coll": "transactions"
            }}
        ]
        
        print("  📡 Opening change stream on transactions...")
        
        # Use a separate thread to insert while watching
        import threading
        
        def insert_test_transaction():
            """Insert a test transaction after 2 seconds"""
            time.sleep(2)
            self.db.transactions.insert_one({
                "user_id": ObjectId(),
                "type": "payment",
                "amount": Decimal128("25.00"),
                "currency": "USD",
                "status": "pending",
                "merchant": {
                    "id": ObjectId(),
                    "name": "Stream Test Merchant",
                    "category": "test"
                },
                "items": [],
                "fraud_score": None,
                "created_at": datetime.utcnow(),
                "updated_at": datetime.utcnow(),
                "_test_marker": "change_stream_validation"
            })
            print("  📝 Test transaction inserted")
            
            time.sleep(1)
            
            # Now update it (simulating fraud score from Flink)
            self.db.transactions.update_one(
                {"_test_marker": "change_stream_validation"},
                {"$set": {
                    "fraud_score": 0.15,
                    "status": "completed",
                    "updated_at": datetime.utcnow()
                }}
            )
            print("  📝 Test transaction updated (fraud score)")
        
        # Start the inserter thread
        inserter = threading.Thread(target=insert_test_transaction)
        inserter.start()
        
        # Watch for changes
        try:
            with self.db.transactions.watch(
                pipeline,
                max_await_time_ms=5000
            ) as stream:
                deadline = time.time() + timeout_seconds
                while time.time() < deadline and len(events_captured) < 2:
                    if stream.try_next() is not None:
                        change = stream.try_next()
                        # We already consumed one in the check
                        # Let's use the iterator properly
                        pass
                
                # Simpler approach: iterate with timeout
                stream_iter = self.db.transactions.watch(
                    pipeline,
                    max_await_time_ms=3000
                )
                
                for change in stream_iter:
                    events_captured.append({
                        "operation": change["operationType"],
                        "collection": change["ns"]["coll"],
                        "document_key": str(
                            change.get("documentKey", {}).get("_id")),
                        "timestamp": change["clusterTime"]
                    })
                    print(f"  🔔 Change detected: "
                          f"{change['operationType']} on "
                          f"{change['ns']['coll']}")
                    
                    if len(events_captured) >= 2:
                        break
                
                stream_iter.close()
        except Exception as e:
            print(f"  ℹ️  Stream ended: {e}")
        
        inserter.join(timeout=10)
        
        if len(events_captured) >= 1:
            print(f"  ✅ Change stream captured "
                  f"{len(events_captured)} events")
            for evt in events_captured:
                print(f"      {evt['operation']}: {evt['document_key']}")
        else:
            print(f"  ⚠️  No events captured in {timeout_seconds}s")
            print(f"      (This is OK in standalone mode — "
                  f"change streams require replica set)")
        
        # Cleanup test data
        self.db.transactions.delete_many({
            "_test_marker": "change_stream_validation"
        })
        
        return {"events_captured": len(events_captured), "status": "PASS"}
    
    def validate_collection_health(self):
        """
        Validate all collections are properly configured
        (Project 49 — monitoring applied)
        """
        print("\n" + "=" * 60)
        print("VALIDATING: Collection Health")
        print("=" * 60)
        
        collections_to_check = [
            "users", "transactions", "merchants",
            "audit_trail", "notifications"
        ]
        
        results = {}
        
        for coll_name in collections_to_check:
            coll = self.db[coll_name]
            
            # Check if collection exists
            if coll_name not in self.db.list_collection_names():
                print(f"  ⚠️  {coll_name}: does not exist yet")
                results[coll_name] = {"status": "MISSING"}
                continue
            
            # Get stats
            stats = self.db.command("collStats", coll_name)
            indexes = list(coll.list_indexes())
            
            # Index usage
            try:
                idx_stats = list(coll.aggregate([
                    {"$indexStats": {}}
                ]))
            except Exception:
                idx_stats = []
            
            unused_indexes = [
                idx["name"] for idx in idx_stats
                if idx["accesses"]["ops"] == 0
                and idx["name"] != "_id_"
            ]
            
            result = {
                "status": "OK",
                "doc_count": stats.get("count", 0),
                "avg_doc_size_bytes": stats.get("avgObjSize", 0),
                "data_size_mb": round(
                    stats.get("size", 0) / (1024**2), 2),
                "storage_size_mb": round(
                    stats.get("storageSize", 0) / (1024**2), 2),
                "index_count": stats.get("nindexes", 0),
                "total_index_size_mb": round(
                    stats.get("totalIndexSize", 0) / (1024**2), 2),
                "unused_indexes": unused_indexes
            }
            
            # Warnings
            warnings = []
            if stats.get("nindexes", 0) > 10:
                warnings.append(
                    f"High index count: {stats['nindexes']}")
            if stats.get("avgObjSize", 0) > 50000:
                warnings.append(
                    f"Large avg doc: {stats['avgObjSize']}B "
                    f"— consider subset pattern")
            if len(unused_indexes) > 0:
                warnings.append(
                    f"Unused indexes: {unused_indexes}")
            
            if warnings:
                result["status"] = "WARN"
                result["warnings"] = warnings
            
            results[coll_name] = result
            
            status_emoji = "✅" if result["status"] == "OK" else "🟡"
            print(f"  {status_emoji} {coll_name}:")
            print(f"      Docs: {result['doc_count']} | "
                  f"Avg size: {result['avg_doc_size_bytes']}B | "
                  f"Data: {result['data_size_mb']}MB | "
                  f"Indexes: {result['index_count']}")
            
            for warning in warnings:
                print(f"      ⚠️  {warning}")
        
        return results
    
    def validate_sharding_readiness(self):
        """
        Validate sharding configuration
        (Projects 21-22 — shard key validation)
        """
        print("\n" + "=" * 60)
        print("VALIDATING: Sharding Readiness")
        print("=" * 60)
        
        try:
            shard_status = self.mongo.admin.command("listShards")
            shards = shard_status.get("shards", [])
            
            if len(shards) == 0:
                print("  ℹ️  Not sharded (standalone/replica set)")
                print("  📋 Recommended shard keys when ready:")
                print("      transactions: { user_id: 'hashed' }")
                print("      users: { region: 1, _id: 1 }")
                print("      merchants: { region: 1, _id: 1 }")
                print("      audit_trail: { entity_type: 1, created_at: 1 }")
                return {"status": "NOT_SHARDED", "shards": 0}
            
            print(f"  ✅ Sharded cluster: {len(shards)} shards")
            for shard in shards:
                print(f"      {shard['_id']}: {shard['host']}")
            
            # Check chunk distribution
            config_db = self.mongo["config"]
            collections = list(config_db.collections.find(
                {"dropped": False}
            ))
            
            for coll in collections:
                ns = coll["_id"]
                chunks = list(config_db.chunks.aggregate([
                    {"$match": {"ns": ns}},
                    {"$group": {
                        "_id": "$shard",
                        "count": {"$sum": 1}
                    }},
                    {"$sort": {"count": -1}}
                ]))
                
                if chunks:
                    max_chunks = chunks[0]["count"]
                    min_chunks = chunks[-1]["count"]
                    ratio = max_chunks / max(min_chunks, 1)
                    
                    status = "✅" if ratio < 1.5 else "⚠️"
                    print(f"\n  {status} {ns}: "
                          f"shard key = {json.dumps(coll.get('key', {}))}")
                    for chunk_info in chunks:
                        bar = "█" * int(
                            chunk_info["count"] / max_chunks * 20)
                        print(f"      {chunk_info['_id']}: "
                              f"{chunk_info['count']} chunks {bar}")
                    print(f"      Balance ratio: {ratio:.2f} "
                          f"({'BALANCED' if ratio < 1.5 else 'SKEWED'})")
            
            return {"status": "SHARDED", "shards": len(shards)}
        
        except Exception as e:
            print(f"  ℹ️  Cannot check sharding: {e}")
            return {"status": "UNKNOWN", "error": str(e)}
