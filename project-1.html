PROJECT 1: Your First Database & Collection

Problem: Store a List of Products for an E-Commerce Platform

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    You just joined a startup building an e-commerce platform
    Your first task: set up a database to store products
    Each product has: name, price, category, stock quantity, created date

    You have done this 100 times in Redshift
    Now you must do it in MongoDB for the first time

    The question isn't "how" — it's "why does everything feel different?"

🔴 REDSHIFT MINDSET — How You Would Solve This

Step 1: Plan the Schema FIRST (Redshift demands this)

YOUR INSTINCT (trained by Redshift):
    → Think about ALL the columns you'll ever need
    → Choose data types carefully (storage = cost)
    → Think about distribution key (how will data be spread?)
    → Think about sort key (what will queries filter on?)
    → Think about encoding/compression per column
    → THEN create the table
    → THEN insert data
    
YOU CANNOT INSERT DATA WITHOUT A TABLE DEFINITION IN REDSHIFT
This is the fundamental rule you've internalized

Step 2: Create the Cluster (if not already running)

IN REDSHIFT:
    → Go to AWS Console → Amazon Redshift → Create Cluster
    → Choose node type (dc2.large, ra3.xlarge, etc.)
    → Choose number of nodes
    → Set master username/password
    → Configure VPC, subnet, security group
    → Wait 5-10 minutes for cluster provisioning
    → Connect via Query Editor v2 or psql

    COST STARTS IMMEDIATELY — even with zero data
    Cluster runs 24/7 unless paused (serverless is different)

Step 3: Create the Database and Table

-- REDSHIFT: Create database
CREATE DATABASE ecommerce;

-- Connect to ecommerce database
-- (In psql: \c ecommerce)

-- REDSHIFT: Create schema (namespace)
CREATE SCHEMA store;

-- REDSHIFT: Create table with FULL schema definition upfront
CREATE TABLE store.products (
    product_id      BIGINT IDENTITY(1,1),    -- auto-increment PK
    name            VARCHAR(255) NOT NULL,     -- must define max length
    price           DECIMAL(10,2) NOT NULL,    -- must define precision
    category        VARCHAR(100),
    stock_qty       INTEGER DEFAULT 0,
    created_at      TIMESTAMP DEFAULT GETDATE(),
    
    -- Redshift-specific: compression encoding
    -- Redshift auto-assigns, but you CAN specify
    PRIMARY KEY (product_id)
)
DISTSTYLE KEY                    -- how rows are distributed across nodes
DISTKEY (category)               -- distribute by category for co-located joins
SORTKEY (created_at)             -- sort by date for range scan efficiency
;

Step 4: What Redshift Did Behind the Scenes

WHEN YOU RAN CREATE TABLE:
    → Redshift LOCKED the schema definition permanently
    → Every column's type, size, encoding = FIXED
    → Distribution key decided how data spreads across compute slices
    → Sort key decided physical sort order on disk
    → Zone maps will be auto-created per 1MB block per column
    → Columnar storage: each column stored SEPARATELY on disk
    → If you need a new column later: ALTER TABLE ADD COLUMN
    → If you need to change DISTKEY/SORTKEY: recreate entire table
    
    THE TABLE EXISTS — empty but fully structured — BEFORE any data arrives

🟢 MONGODB MINDSET — How This Same Problem Is Approached

Step 1: There Is No "Plan Schema First" Step

YOUR NEW REALITY:
    → You do NOT define columns (fields) upfront
    → You do NOT choose data types in advance
    → You do NOT create the collection (table) explicitly
    → You just START INSERTING DATA
    → The database creates itself when you first reference it
    → The collection creates itself when you first insert into it
    
    THIS WILL FEEL WRONG — your Redshift brain will scream
    "But where's the schema? Where's the DDL? Where's the contract?"
    
    MongoDB's answer: "The document IS the schema"

Step 2: Connect to MongoDB (No Cluster Provisioning Needed for Local)

TWO WAYS TO START:

OPTION A — Local MongoDB (for learning)
    → Install MongoDB Community Server
    → Start mongod process (the database server)
    → Connect via mongosh (the shell — equivalent to psql)

OPTION B — MongoDB Atlas (cloud — equivalent to Redshift on AWS)
    → Go to cloud.mongodb.com → Create Free Cluster
    → Choose cloud provider (AWS/GCP/Azure)
    → Choose region
    → FREE TIER EXISTS (M0) — 512MB storage — no cost
    → Get connection string
    → Connect via mongosh or Compass (GUI)

Step 3: "Create" Database and Collection

// MONGODB SHELL (mongosh):

// "Switch" to a database — this CREATES it if it doesn't exist
// But it doesn't actually exist on disk until you insert data
use ecommerce

// That's it. No CREATE DATABASE command.
// No schema definition. No table creation.
// The database "ecommerce" is now your context
// but NOTHING has been written to disk yet.

// To "create" a collection, you just... insert into it:
db.products.insertOne({
    name: "Wireless Mouse",
    price: 29.99,
    category: "Electronics",
    stock_qty: 150,
    created_at: new Date()
})

Step 4: What MongoDB Did Behind the Scenes

WHEN YOU RAN insertOne():
    1. Database "ecommerce" was CREATED (directory on disk)
    2. Collection "products" was CREATED (files on disk)
    3. The document was assigned an _id field (ObjectId) automatically
    4. Document was written as BSON (Binary JSON) to WiredTiger storage
    5. A default index on _id was auto-created
    6. Journal entry was written (for crash recovery)
    
    NO SCHEMA WAS LOCKED
    → Next document can have COMPLETELY different fields
    → No ALTER TABLE ever needed
    → No column type enforcement (unless you add validation later)
    
    THE COLLECTION CAME INTO EXISTENCE BECAUSE DATA ARRIVED
    — opposite of Redshift where structure exists before data

🧩 HOW TO APPROACH IT IN MONGODB

THINKING FRAMEWORK (rewire your brain):

IN REDSHIFT YOU THINK:
    1. What columns will I need?         → FIRST
    2. What types are they?              → SECOND
    3. How will I distribute the data?   → THIRD
    4. How will I sort the data?         → FOURTH
    5. Create the table                  → FIFTH
    6. Load the data                     → LAST

IN MONGODB YOU THINK:
    1. What does ONE product look like?  → FIRST (think in documents)
    2. How will the application ACCESS   → SECOND (read/write patterns)
       this product?
    3. Insert the document               → THIRD (collection auto-created)
    4. Add indexes LATER based on        → FOURTH (optimize after you know
       actual query patterns                  actual access patterns)

THE ORDER IS REVERSED:
    Redshift: Structure → Data → Queries
    MongoDB:  Data → Queries → Structure (indexes/sharding later)

🔧 HOW CAN WE SOLVE IT IN MONGODB

Method 1: Implicit Collection Creation (Most Common)

// Local:
mongosh

// Atlas:
mongosh "mongodb+srv://cluster0.xxxxx.mongodb.net/" --apiVersion 1 --username myUser

// Step 2: Switch to database (creates it implicitly)
use ecommerce

// Step 3: Insert a product (creates collection implicitly)
db.products.insertOne({
    name: "Wireless Mouse",
    price: 29.99,
    category: "Electronics",
    stock_qty: 150,
    created_at: new Date()
})

// OUTPUT:
// {
//   acknowledged: true,
//   insertedId: ObjectId('665a1b2c3d4e5f6a7b8c9d0e')
// }

Method 2: Explicit Collection Creation (When You Need Options)

// mongosh — When you want to set options at creation time

use ecommerce

// Explicitly create collection with options
db.createCollection("products", {
    // Optional: capped collection (fixed-size, auto-delete oldest)
    // capped: true,
    // size: 1048576,  // 1MB max
    
    // Optional: schema validation (we'll cover deeply in Project 7)
    validator: {
        $jsonSchema: {
            bsonType: "object",
            required: ["name", "price"],
            properties: {
                name:      { bsonType: "string" },
                price:     { bsonType: "double" },
                category:  { bsonType: "string" },
                stock_qty: { bsonType: "int" }
            }
        }
    },
    validationLevel: "moderate",    // apply to inserts and updates
    validationAction: "warn"        // warn but don't reject (vs "error")
})

Method 3: Python (pymongo) — The Application Way

# Equivalent to using psycopg2 or SQLAlchemy for Redshift

from pymongo import MongoClient
from datetime import datetime

# ─── CONNECTION ───────────────────────────────────────────────

# ─── "CREATE" COLLECTION ─────────────────────────────────────

# ─── INSERT FIRST DOCUMENT (NOW everything gets created) ─────

    "price": 29.99,

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

AT THIS STAGE (Project 1), optimizations are minimal
But here's what MongoDB ALLOWS that Redshift doesn't:

1. CUSTOM _id
   → Instead of auto-generated ObjectId, use your own:
     { _id: "PROD-001", name: "Wireless Mouse", ... }
   → Like choosing your own primary key vs IDENTITY column
   → Useful when you have a natural unique key (SKU, product code)

2. WRITE CONCERN ON INSERT
   → db.products.insertOne({...}, { writeConcern: { w: 1 } })
   → Control durability guarantee per operation
   → Redshift has no equivalent — every write is fully durable (slower)

3. ORDERED vs UNORDERED INSERTS
   → insertMany([...], { ordered: false })
   → Unordered: continues on error (like Redshift COPY MAXERROR)
   → Ordered: stops at first error (default)

✅ WHAT SHOULD DO (Best Practices vs Redshift Habits to Unlearn)

SHOULD DO:
    ✅ Think about your DOCUMENT SHAPE before inserting
       → Not schema columns — but what ONE entity looks like
       → "What does a product look like as a JSON object?"
    
    ✅ Use meaningful database names
       → use ecommerce  (not use db1)
    
    ✅ Use plural collection names by convention
       → db.products (not db.product)
       → Like Redshift table naming conventions
    
    ✅ Let MongoDB generate _id (ObjectId) unless you have a strong reason
       → ObjectId contains timestamp — free created_at information
       → ObjectId is globally unique — no sequence collision across shards
    
    ✅ Use the pymongo driver for application code
       → mongosh for exploration (like psql for Redshift)
       → pymongo for production code (like psycopg2/SQLAlchemy for Redshift)
    
    ✅ Use Compass (GUI) alongside mongosh
       → Visual document exploration
       → Equivalent to Redshift Query Editor v2 but for documents

REDSHIFT HABITS TO UNLEARN:
    🔄 Stop thinking "I must define ALL columns before inserting"
    🔄 Stop thinking "I need to choose distribution and sort keys now"
    🔄 Stop thinking "schema migration = ALTER TABLE"
    🔄 Stop thinking "every row must have the same fields"

❌ WHAT SHOULD NOT DO (Anti-Patterns)

DO NOT:
    ❌ Create empty databases/collections "just to have structure"
       → They don't persist until data is written
       → Waste of time — opposite of Redshift where empty tables are normal
    
    ❌ Try to enforce column types at insert time manually
       → "if type(price) != float: raise Error" in your app code
       → USE schema validation instead (Project 7) if you need type safety
    
    ❌ Store everything as strings
       → price: "29.99" ← WRONG (can't do math, can't sort numerically)
       → price: 29.99   ← CORRECT (stored as BSON double)
       → Redshift enforces types so you never think about this
       → MongoDB lets you store ANYTHING — you must be disciplined
    
    ❌ Use SQL-like naming with dots in collection names
       → db["store.products"] ← technically works but DON'T
       → Use separate databases instead: use store → db.products
    
    ❌ Create hundreds of small collections (one per entity subtype)
       → Coming from Redshift: one table per entity = normal
       → In MongoDB: polymorphic documents in ONE collection = better
       → We'll cover this deeply in Project 18
    
    ❌ Assume "no schema" means "no design"
       → BIGGEST TRAP for Redshift developers
       → MongoDB requires MORE design thinking, not less
       → You're designing DOCUMENTS, not TABLES
       → The design just lives in your application, not in DDL

📍 WHEN TO USE MONGODB OVER REDSHIFT

FOR THIS SPECIFIC PROBLEM (product catalog):

USE MONGODB WHEN:
    → Products have VARYING attributes
      (electronics: voltage, watts | clothing: size, color, fabric)
    → You need sub-millisecond single-product lookups
    → You need real-time inventory updates (high write concurrency)
    → Product schema changes frequently (new attributes added weekly)
    → You need to serve an API directly from the database
    → Your team iterates fast (startup, agile, rapid prototyping)

📍 WHEN TO STICK WITH REDSHIFT INSTEAD

FOR THIS SPECIFIC PROBLEM (product catalog):

USE REDSHIFT WHEN:
    → All products have IDENTICAL structure (same columns)
    → Primary use case is ANALYTICAL queries
      ("average price per category per quarter across 5 years")
    → You need to JOIN products with 10+ other tables for reports
    → You're loading data in batches (nightly ETL)
    → Your queries scan millions of products for aggregation
    → You need SQL compatibility for BI tools (Tableau, Looker)

📍 WHEN TO USE BOTH TOGETHER

THE REAL-WORLD ANSWER (most common architecture):

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Internals — WiredTiger Storage

WHEN YOU RAN: db.products.insertOne({ name: "Wireless Mouse", ... })

HERE'S WHAT HAPPENED INSIDE:

1. BSON ENCODING
   → Your JSON document was converted to BSON (Binary JSON)
   → BSON is a binary format — more compact, typed, fast to parse
   → JSON:  { "price": 29.99 }          ← text, no type info
   → BSON:  \x10 price \x00 \x29.99...  ← binary, typed as double
   
   WHY BSON AND NOT JSON?
   → JSON is text → slow to parse, no native types
   → BSON adds: Date, Decimal128, ObjectId, Binary, int32/int64
   → BSON allows O(1) field traversal (length-prefixed)

2. WIREDTIGER STORAGE ENGINE
   → WiredTiger is MongoDB's default storage engine (since 3.2)
   → It uses a B-tree structure for document storage
   → Documents stored in PAGES (like Redshift blocks but different)
   
   CRITICAL DIFFERENCE:
   → Redshift: each COLUMN stored in separate 1MB block
     → "price" values for ALL rows stored together → great for SUM(price)
   → MongoDB: each DOCUMENT stored as complete unit
     → ALL fields of ONE product stored together → great for "get product by id"

3. JOURNAL (Write-Ahead Log)
   → BEFORE writing to data files, write to journal
   → Journal = WAL (Write-Ahead Log) — same concept as Redshift's commit log
   → Crash recovery: replay journal to recover uncommitted writes
   → Journal checkpoints every 60 seconds (configurable)

4. _id INDEX AUTO-CREATION
   → A unique B-tree index on _id was automatically created
   → This is like having an automatic PRIMARY KEY index
   → Redshift primary keys are NOT enforced (informational only!)
   → MongoDB _id index IS enforced — duplicates rejected

5. MEMORY (WiredTiger Cache)
   → Document loaded into WiredTiger internal cache
   → Default: 50% of (RAM - 1GB)
   → Documents are stored in cache in UNCOMPRESSED format
   → On disk: compressed (snappy/zlib/zstd)
   → Redshift equivalent: query result cache + block cache

PHYSICAL FILES CREATED:
    /data/db/
    ├── ecommerce/              ← database directory
    │   ├── collection-xxx.wt   ← products collection data
    │   └── index-xxx.wt        ← _id index data
    ├── journal/                ← write-ahead log
    │   └── WiredTigerLog.xxx
    └── WiredTiger.wt           ← metadata

Redshift Internals — For Comparison

WHEN YOU RAN: INSERT INTO store.products VALUES (...)

HERE'S WHAT HAPPENED INSIDE REDSHIFT:

1. LEADER NODE received the query
   → Parsed SQL, created execution plan
   → Determined which compute slice owns this row (based on DISTKEY)

2. COMPUTE SLICE received the row
   → Split the row into COLUMNS
   → Each column value stored in separate 1MB block
   → name → block_name_001.col
   → price → block_price_001.col
   → category → block_category_001.col
   
3. ZONE MAPS updated
   → Min/max values per block per column recorded
   → WHERE price > 100 → skip blocks where max(price) < 100
   → THIS is why Redshift is fast for analytical scans

4. COMPRESSION applied per column
   → Each column compressed with appropriate encoding
   → LZO, ZSTD, Delta, Bytedict, etc.
   → price (decimal) might use Delta encoding
   → category (low cardinality) might use Bytedict

WHY THIS MATTERS:
    → Redshift stores COLUMNS separately → fast for "scan one column across all rows"
    → MongoDB stores DOCUMENTS together → fast for "get all fields of one document"
    
    → SELECT AVG(price) FROM products  → Redshift reads ONLY price column → FAST
    → db.products.findOne({_id: x})    → MongoDB reads ONE document → FAST
    
    → SELECT AVG(price) FROM products  → MongoDB must read ALL documents → SLOW
    → db.products.findOne({_id: x})    → Redshift must reconstruct full row → SLOWER

WHY THE INTERNALS CAUSE DIFFERENT BEHAVIOR

SAME OPERATION — DIFFERENT PERFORMANCE:

┌─────────────────────────────┬────────────────┬────────────────┐
│ OPERATION                    │ REDSHIFT        │ MONGODB         │
├─────────────────────────────┼────────────────┼────────────────┤
│ Get one product by ID        │ ~10-50ms        │ ~1-5ms          │
│ (point lookup)               │ (overkill)      │ (designed for)  │
├─────────────────────────────┼────────────────┼────────────────┤
│ AVG(price) across 10M rows   │ ~2-5 seconds    │ ~30-60 seconds  │
│ (full column scan)           │ (designed for)  │ (reads all docs)│
├─────────────────────────────┼────────────────┼────────────────┤
│ Insert 1 product             │ ~50-200ms       │ ~1-5ms          │
│ (single write)               │ (columnar write)│ (document write)│
├─────────────────────────────┼────────────────┼────────────────┤
│ Insert 1M products           │ ~30s (COPY)     │ ~30s (bulk)     │
│ (bulk load)                  │ (parallel load) │ (insertMany)    │
├─────────────────────────────┼────────────────┼────────────────┤
│ Add new field to one product │ Impossible      │ ~1ms            │
│ (schema change per row)      │ (must ALTER)    │ (just update)   │
└─────────────────────────────┴────────────────┴────────────────┘

THE ROOT CAUSE:
    Columnar (Redshift) = optimize for scanning MANY rows, FEW columns
    Document (MongoDB)  = optimize for accessing FEW documents, ALL fields

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE IN MONGODB — PROJECT 1:

PITFALL 1: "Let me create all my collections first"
    ❌ db.createCollection("products")
       db.createCollection("orders")
       db.createCollection("customers")
       db.createCollection("order_items")
       ... then insert data later
    ✅ Just start inserting — collections appear automatically
    WHY: Redshift trained you to build structure first
         MongoDB is data-first, structure-second

PITFALL 2: "Where's my auto-increment ID?"
    ❌ Trying to implement auto-increment sequence in MongoDB
       (creating a counters collection, using findAndModify)
    ✅ Use ObjectId — it's BETTER than auto-increment
    WHY: ObjectId is globally unique (works across shards)
         Auto-increment requires coordination (bottleneck)
         ObjectId contains timestamp (free metadata)

PITFALL 3: "I need to define column types somewhere"
    ❌ Creating a "schema document" or config file listing types
    ✅ Types are inferred from the data you insert
       price: 29.99 → stored as BSON double
       price: NumberDecimal("29.99") → stored as Decimal128
    WHY: MongoDB is typed at the VALUE level, not COLUMN level
         Same field can have different types in different documents
         (though you SHOULDN'T do this — it's just possible)

PITFALL 4: "I'll connect with SQL"
    ❌ Trying to use JDBC SQL driver to talk to MongoDB
    ✅ Use native driver: pymongo, mongosh, or MongoDB JDBC (Atlas SQL)
    WHY: MongoDB speaks MQL (MongoDB Query Language), not SQL
         There IS an Atlas SQL interface, but it's for BI tools
         For application code: ALWAYS use native driver

PITFALL 5: "The database disappeared!"
    → You ran `use mydb` but inserted nothing
    → Disconnected and reconnected
    → `show dbs` — mydb is gone!
    WHY: Empty databases are not persisted
         Must have at least one document in one collection
         Redshift: empty databases persist forever

📈 SCALABILITY CONSIDERATIONS

AT PROJECT 1 LEVEL — what you should KNOW (not implement yet):

MONGODB SCALING PATH:
    Single Server (what we just set up)
        ↓
    Replica Set (3+ servers for high availability) → Project 23
        ↓
    Sharded Cluster (data split across multiple replica sets) → Project 21
        ↓
    Global Cluster (shards in multiple regions) → Project 41

    HORIZONTAL SCALING: add more servers
    Each step adds servers, not bigger servers

REDSHIFT SCALING PATH:
    Single Node (dc2.large)
        ↓
    Multi-Node (add compute nodes)
        ↓
    RA3 (managed storage, decouple compute/storage)
        ↓
    Serverless (auto-scaling)
        ↓
    Cross-region Data Sharing (read replicas)

    VERTICAL + HORIZONTAL: bigger nodes AND more nodes

KEY DIFFERENCE:
    MongoDB: designed for horizontal scale from day 1
             add servers → data automatically redistributed
    
    Redshift: designed for vertical scale primarily
              resize operations have downtime (classic)
              RA3/Serverless improved this significantly

📐 DATA MODELING COMPARISON

FOR THIS PROBLEM (product catalog):

REDSHIFT (Normalized Default):
    products table     → product_id, name, price, category_id, stock_qty
    categories table   → category_id, category_name, description
    
    WHY: normalized to avoid data duplication
         JOIN categories ON products.category_id = categories.category_id
         One source of truth for category names

MONGODB (Denormalized Default):
    products collection → each document contains ALL product info
    {
        name: "Wireless Mouse",
        price: 29.99,
        category: "Electronics",     ← category NAME stored directly
        stock_qty: 150               ← no separate categories collection
    }
    
    WHY: avoid JOIN ($lookup) overhead
         single query returns everything needed
         category name duplicated across products (accepted trade-off)

WHEN TO BREAK CONVENTION:
    Redshift denormalized: when JOINs are too expensive (star schema)
    MongoDB normalized: when referenced data changes frequently
                        (category renamed → update ONE place vs MANY)
    
    WE'LL EXPLORE THIS DEEPLY IN PROJECT 14
