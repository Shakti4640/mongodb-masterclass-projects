The Query-Time Computation Approach

-- REDSHIFT: Compute everything at query time
-- This is Redshift's STRENGTH — scanning millions of rows fast

-- Table: events (append-only, massive)
CREATE TABLE analytics.events (
    event_id        BIGINT IDENTITY(1,1),
    event_type      VARCHAR(20),        -- 'page_view', 'add_to_cart', 'purchase'
    product_id      BIGINT,
    user_id         BIGINT,
    session_id      VARCHAR(64),
    amount          DECIMAL(10,2),       -- only for purchases
    quantity        INTEGER,
    created_at      TIMESTAMP DEFAULT GETDATE()
)
DISTSTYLE KEY DISTKEY(product_id)
SORTKEY(created_at);
-- Sort key on created_at: recent data at the end of disk
-- Zone maps: "today's data" query skips all historical blocks

-- ═══════════════════════════════════════════════════════
-- DASHBOARD QUERY 1: Today's totals
-- ═══════════════════════════════════════════════════════
SELECT 
    COUNT(*) FILTER (WHERE event_type = 'purchase') AS total_orders,
    SUM(amount) FILTER (WHERE event_type = 'purchase') AS total_revenue,
    AVG(amount) FILTER (WHERE event_type = 'purchase') AS avg_order_value,
    COUNT(DISTINCT user_id) AS active_users,
    COUNT(*) FILTER (WHERE event_type = 'page_view') AS total_views,
    COUNT(*) FILTER (WHERE event_type = 'add_to_cart') AS total_carts
FROM analytics.events
WHERE created_at >= CURRENT_DATE;
-- Redshift execution: ~2-5 seconds on 30M rows
-- Sort key skips all data before today
-- Columnar: reads only event_type, amount, user_id columns
-- Parallel across all slices

-- ═══════════════════════════════════════════════════════
-- DASHBOARD QUERY 2: Orders per minute (last 60 minutes)
-- ═══════════════════════════════════════════════════════
SELECT 
    DATE_TRUNC('minute', created_at) AS minute,
    COUNT(*) AS order_count,
    SUM(amount) AS minute_revenue
FROM analytics.events
WHERE event_type = 'purchase'
  AND created_at >= DATEADD(hour, -1, GETDATE())
GROUP BY 1
ORDER BY 1;
-- Execution: ~1-3 seconds

-- ═══════════════════════════════════════════════════════
-- DASHBOARD QUERY 3: Top 5 products by revenue today
-- ═══════════════════════════════════════════════════════
SELECT 
    p.name,
    COUNT(*) AS purchases,
    SUM(e.amount) AS revenue
FROM analytics.events e
JOIN store.products p ON e.product_id = p.product_id
WHERE e.event_type = 'purchase'
  AND e.created_at >= CURRENT_DATE
GROUP BY p.name
ORDER BY revenue DESC
LIMIT 5;
-- Execution: ~3-8 seconds (JOIN + aggregation)

-- ═══════════════════════════════════════════════════════
-- PRODUCT PAGE QUERY: Lifetime stats for one product
-- ═══════════════════════════════════════════════════════
SELECT 
    COUNT(*) FILTER (WHERE event_type = 'page_view') AS view_count,
    COUNT(*) FILTER (WHERE event_type = 'purchase') AS purchase_count,
    COUNT(*) FILTER (WHERE event_type = 'add_to_cart') AS cart_count,
    SUM(amount) FILTER (WHERE event_type = 'purchase') AS total_revenue
FROM analytics.events
WHERE product_id = 12345;
-- Execution: ~1-5 seconds (scan all events for this product)
-- DISTKEY on product_id: all events co-located on one slice

-- ═══════════════════════════════════════════════════════
-- MATERIALIZED VIEW: Pre-compute for faster access
-- ═══════════════════════════════════════════════════════
CREATE MATERIALIZED VIEW analytics.daily_product_stats AS
SELECT 
    product_id,
    DATE_TRUNC('day', created_at) AS day,
    COUNT(*) FILTER (WHERE event_type = 'page_view') AS views,
    COUNT(*) FILTER (WHERE event_type = 'add_to_cart') AS carts,
    COUNT(*) FILTER (WHERE event_type = 'purchase') AS purchases,
    SUM(amount) FILTER (WHERE event_type = 'purchase') AS revenue
FROM analytics.events
GROUP BY 1, 2;

-- Refresh periodically
REFRESH MATERIALIZED VIEW analytics.daily_product_stats;
-- Takes: 30-120 seconds depending on data volume
-- Stale between refreshes

Why Query-Time Works in Redshift (But Not MongoDB)

REDSHIFT'S ADVANTAGES FOR QUERY-TIME COMPUTATION:

1. COLUMNAR SCAN EFFICIENCY
   → COUNT(*) WHERE event_type = 'purchase'
   → Reads ONLY event_type column (not amount, user_id, product_id, etc.)
   → 30M rows × 1 byte (compressed event_type) ≈ 5 MB scanned
   → NOT 30M rows × 200 bytes per row = 6 GB
   → 1000x less I/O than row-store

2. SORT KEY + ZONE MAPS
   → SORTKEY(created_at) → today's data in last blocks
   → WHERE created_at >= CURRENT_DATE → skip 99% of blocks
   → Zone maps: each 1 MB block has min/max created_at
   → Blocks from last month? max(created_at) < today → SKIP

3. MPP PARALLELISM
   → 30M rows distributed across 8 slices → each scans ~3.75M
   → All slices scan simultaneously
   → Leader node aggregates partial results
   → Wall-clock time: ~1/8 of single-node time

4. RESULT CACHE
   → Same dashboard query within 15 minutes → cached result
   → No re-scan needed
   → Cache invalidated when underlying table changes

REDSHIFT CONCLUSION:
    → "Query-time computation on 30M rows: 2-5 seconds"
    → "Materialized view refresh: every 5 minutes"
    → "Acceptable for analytics dashboard"
    → "No need to pre-compute on every event"

Why Query-Time FAILS in MongoDB

MONGODB'S CHALLENGES FOR QUERY-TIME COMPUTATION:

1. DOCUMENT STORAGE = FULL DOCUMENT READS
   → db.events.aggregate([{ $match: { event_type: "purchase" } }, { $group: ... }])
   → Even with index on event_type: must read each matching DOCUMENT
   → Each document: ~200 bytes (all fields loaded)
   → 10M purchases × 200 bytes = 2 GB of document reads
   → vs Redshift: reads only event_type column = 5 MB

2. NO COLUMNAR OPTIMIZATION
   → MongoDB cannot read "just the amount field" across all documents
   → SUM(amount) requires loading every matching document
   → Then extracting the amount field from each BSON document
   → Dramatically more I/O than columnar scan

3. SINGLE-THREADED AGGREGATION (per shard)
   → Aggregation pipeline runs on ONE thread per shard
   → No MPP parallelism within a single shard
   → 30M documents on one shard → sequential processing
   → Sharding helps (parallel across shards) but not same as MPP

4. LATENCY REQUIREMENT MISMATCH
   → Dashboard needs: < 1 second refresh
   → MongoDB aggregation on 30M docs: 30-120 seconds
   → 30-120x too slow for real-time dashboard
   → IMPOSSIBLE to serve real-time dashboard from raw aggregation

🟢 MONGODB MINDSET — The Computed Pattern

Core Concept: Compute on Write, Read for Free

THE FUNDAMENTAL SHIFT:

REDSHIFT (Query-Time Compute):
    EVENT HAPPENS → store raw event → done
    DASHBOARD LOADS → scan all events → compute stats → return
    
    WRITE COST:  LOW (just append a row)
    READ COST:   HIGH (scan millions of rows every time)
    BEST WHEN:   Reads are infrequent, data volume manageable for scans

MONGODB (Write-Time Compute):
    EVENT HAPPENS → store raw event → UPDATE pre-computed stats → done
    DASHBOARD LOADS → read one document with stats → return immediately
    
    WRITE COST:  HIGHER (insert event + update computed fields)
    READ COST:   NEAR-ZERO (read one small document)
    BEST WHEN:   Reads are frequent, real-time required, data volume huge

ANALOGY:
    Redshift approach: Library with no catalog
        → "How many science books?" → count every shelf → slow
        → But adding a book is fast (just put it on a shelf)
    
    MongoDB approach: Library with a LIVE catalog board
        → Adding a book: put on shelf + update the board (+1 science)
        → "How many science books?" → look at the board → instant
        → Adding is slightly slower, but asking is instant

THE TRADE-OFF:
    ┌────────────────────────────────────────────────────┐
    │ Query-Time (Redshift):                              │
    │   WRITES: O(1) — just append                        │
    │   READS:  O(N) — scan all matching rows             │
    │                                                      │
    │ Write-Time (MongoDB Computed Pattern):               │
    │   WRITES: O(1) + O(1) — append + update counter     │
    │   READS:  O(1) — read one document                  │
    │                                                      │
    │ When N is large and reads are frequent:              │
    │   Write-Time wins dramatically                       │
    └────────────────────────────────────────────────────┘

🧩 HOW TO APPROACH IT IN MONGODB

Design the Computed Documents

WHAT TO PRE-COMPUTE — Decision Framework:

ASK THESE QUESTIONS:
    1. How often is this value READ?
       → Dashboard refreshes every second → READ 1/second
    2. How often is this value's source data WRITTEN?
       → 10 purchases/second → WRITE 10/second
    3. What's the aggregation cost at query time?
       → COUNT 30M rows → 30+ seconds
    4. Is the computation SIMPLE (count, sum, avg)?
       → Yes → perfect for $inc, $set atomic operations
       → Complex (percentiles, moving averages) → harder to pre-compute

DECISION MATRIX:
┌───────────────────────┬──────────────────┬──────────────────────────────┐
│ METRIC                 │ READ FREQUENCY    │ APPROACH                      │
├───────────────────────┼──────────────────┼──────────────────────────────┤
│ Total orders today     │ 1/second          │ PRE-COMPUTE ($inc on order)   │
│ Revenue today          │ 1/second          │ PRE-COMPUTE ($inc on order)   │
│ Avg order value        │ 1/second          │ DERIVE (revenue / orders)     │
│ Product view count     │ Every page load   │ PRE-COMPUTE ($inc on view)    │
│ Product purchase count │ Every page load   │ PRE-COMPUTE ($inc on purchase)│
│ Orders per minute      │ 1/second          │ BUCKET + PRE-COMPUTE          │
│ Top 5 products         │ 1/second          │ PERIODIC REFRESH (every min)  │
│ 90th percentile latency│ 1/minute          │ QUERY-TIME (complex math)     │
│ User cohort analysis   │ 1/day             │ QUERY-TIME or Redshift        │
└───────────────────────┴──────────────────┴──────────────────────────────┘

RULE OF THUMB:
    → Read >> Write frequency → PRE-COMPUTE
    → Write >> Read frequency → QUERY-TIME
    → Simple aggregation (count, sum) → PRE-COMPUTE
    → Complex aggregation (percentile, window) → QUERY-TIME or Redshift

The Three-Layer Architecture

LAYER 1: RAW EVENTS (Source of Truth)
    → events collection: every event stored as-is
    → Like Redshift's analytics.events table
    → Used for: recovery, ad-hoc analysis, debugging
    → Can be aged out (TTL index) or archived to S3

LAYER 2: COMPUTED DOCUMENTS (Real-Time Stats)
    → daily_stats collection: one document per day
    → product_stats: computed fields embedded in product documents
    → Updated atomically on every event via $inc
    → Used for: dashboards, product pages, APIs

LAYER 3: PERIODIC AGGREGATIONS (Complex Stats)
    → Hourly/daily batch aggregation for complex metrics
    → Rankings, percentiles, trends
    → Like Redshift materialized views
    → Rebuilt from Layer 1 (raw events)

// STEP 3: THE CORE — Event Processing with Computed Updates
// ═══════════════════════════════════════════════════════

function processEvent(event) {
    /**
     * THIS IS THE HEART OF THE COMPUTED PATTERN
     * 
     * Every event triggers:
     *   1. Insert raw event (source of truth)
     *   2. Update daily_stats (dashboard counters)
     *   3. Update product_stats (product page counters)
     * 
     * All updates use $inc → ATOMIC, no read-modify-write race conditions
     * 
     * Redshift equivalent:
     *   INSERT INTO analytics.events VALUES (...);
     *   -- That's it. No pre-computation. Dashboard runs full scan.
     */
    
    let eventDate = event.created_at || new Date()
    let dateKey = eventDate.toISOString().split('T')[0]
    let hourKey = String(eventDate.getHours()).padStart(2, '0')
    let device = (event.metadata && event.metadata.device) || "desktop"
    
    // ─── LAYER 1: Store raw event ────────────────────
    event.created_at = eventDate
    db.events.insertOne(event)
    
    // ─── LAYER 2A: Update daily_stats ────────────────
    let dailyUpdate = {
        $set: { updated_at: new Date() }
    }
    let dailyInc = {}
    
    switch (event.event_type) {
        case "page_view":
            dailyInc["views.count"] = 1
            dailyInc[`by_device.${device}.views`] = 1
            break
            
        case "add_to_cart":
            dailyInc["carts.count"] = 1
            break
            
        case "purchase":
            dailyInc["orders.count"] = 1
            dailyInc["orders.revenue"] = event.amount
            dailyInc["orders.quantity"] = event.quantity || 1
            dailyInc[`orders_by_hour.${hourKey}.count`] = 1
            dailyInc[`orders_by_hour.${hourKey}.revenue`] = event.amount
            dailyInc[`by_device.${device}.purchases`] = 1
            break
    }
    
    dailyUpdate["$inc"] = dailyInc
    
    db.daily_stats.updateOne(
        { _id: dateKey },
        dailyUpdate,
        { upsert: true }    // create if doesn't exist (midnight rollover)
    )
    
    // ─── LAYER 2B: Update product_stats ──────────────
    if (event.product_id) {
        let productInc = {}
        
        switch (event.event_type) {
            case "page_view":
                productInc["stats.lifetime.views"] = 1
                productInc["stats.today.views"] = 1
                break
                
            case "add_to_cart":
                productInc["stats.lifetime.carts"] = 1
                productInc["stats.today.carts"] = 1
                break
                
            case "purchase":
                productInc["stats.lifetime.purchases"] = 1
                productInc["stats.lifetime.revenue"] = event.amount
                productInc["stats.today.purchases"] = 1
                productInc["stats.today.revenue"] = event.amount
                break
        }
        
        db.product_stats.updateOne(
            { _id: event.product_id },
            {
                $inc: productInc,
                $set: { updated_at: new Date() }
            },
            { upsert: true }
        )
    }
}

function rebuildDailyStats(dateKey) {
    /**
     * If computed values get corrupted or out of sync:
     * → Rebuild from raw events (source of truth)
     * → This is like REFRESH MATERIALIZED VIEW in Redshift
     * → But manual and explicit
     * 
     * ALWAYS keep raw events — they're your safety net
     */
    
    let startDate = new Date(dateKey + "T00:00:00Z")
    let endDate = new Date(dateKey + "T23:59:59.999Z")
    
    let rebuilt = db.events.aggregate([
        { $match: { created_at: { $gte: startDate, $lte: endDate } } },
        { $facet: {
            orders: [
                { $match: { event_type: "purchase" } },
                { $group: {
                    _id: null,
                    count: { $sum: 1 },
                    revenue: { $sum: "$amount" },
                    quantity: { $sum: { $ifNull: ["$quantity", 1] } }
                }}
            ],
            views: [
                { $match: { event_type: "page_view" } },
                { $group: {
                    _id: null,
                    count: { $sum: 1 },
                    unique_users: { $addToSet: "$user_id" }
                }},
                { $project: {
                    count: 1,
                    unique_users: { $size: "$unique_users" }
                }}
            ],
            carts: [
                { $match: { event_type: "add_to_cart" } },
                { $count: "count" }
            ],
            orders_by_hour: [
                { $match: { event_type: "purchase" } },
                { $group: {
                    _id: { $hour: "$created_at" },
                    count: { $sum: 1 },
                    revenue: { $sum: "$amount" }
                }},
                { $sort: { _id: 1 } }
            ]
        }}
    ]).toArray()[0]
    
    // Build the hourly breakdown
    let hourlyMap = {}
    rebuilt.orders_by_hour.forEach(h => {
        let hourStr = String(h._id).padStart(2, '0')
        hourlyMap[hourStr] = { count: h.count, revenue: h.revenue }
    })
    
    let ordersData = rebuilt.orders[0] || { count: 0, revenue: 0, quantity: 0 }
    let viewsData = rebuilt.views[0] || { count: 0, unique_users: 0 }
    let cartsData = rebuilt.carts[0] || { count: 0 }
    
    db.daily_stats.updateOne(
        { _id: dateKey },
        {
            $set: {
                orders: {
                    count: ordersData.count,
                    revenue: ordersData.revenue,
                    quantity: ordersData.quantity
                },
                views: {
                    count: viewsData.count,
                    unique_users: viewsData.unique_users
                },
                carts: { count: cartsData.count },
                orders_by_hour: hourlyMap,
                rebuilt_at: new Date(),
                updated_at: new Date()
            }
        },
        { upsert: true }
    )
    
    print(`  Rebuilt daily_stats for ${dateKey}`)
    print(`    Orders: ${ordersData.count}, Revenue: $${ordersData.revenue.toFixed(2)}`)
    print(`    Views: ${viewsData.count}, Unique users: ${viewsData.unique_users}`)
    print(`    Carts: ${cartsData.count}`)
    
    return ordersData
}

class EventProcessor:
    """
    Implements the Computed Pattern for real-time statistics.
    
    REDSHIFT EQUIVALENT:
        In Redshift, you would:
        1. INSERT INTO events (raw event)
        2. Dashboard runs: SELECT COUNT(*), SUM(amount) FROM events WHERE ...
        3. Materialized view refreshed every N minutes
        
    MONGODB COMPUTED PATTERN:
        1. insertOne into events (raw event)
        2. updateOne on daily_stats with $inc (atomic counter update)
        3. updateOne on product_stats with $inc (atomic counter update)
        4. Dashboard reads ONE document — instant
    """
    
    PRODUCTS = {
        "prod_001": {"name": "Wireless Headphones", "price": 299.99},
        "prod_002": {"name": "Mechanical Keyboard", "price": 89.99},
        "prod_003": {"name": "Gaming Mouse", "price": 49.99},
        "prod_004": {"name": "4K Monitor", "price": 499.99},
        "prod_005": {"name": "USB-C Hub", "price": 39.99},
        "prod_006": {"name": "Webcam HD", "price": 79.99},
        "prod_007": {"name": "Desk Lamp", "price": 34.99},
        "prod_008": {"name": "Mouse Pad XL", "price": 24.99},
    }
    
    def __init__(self, db):
        self.db = db
        self.events = db["events"]
        self.daily_stats = db["daily_stats"]
        self.product_stats = db["product_stats"]
        self._init_products()
    
    def _init_products(self):
        """Initialize product_stats documents."""
        for pid, info in self.PRODUCTS.items():
            self.product_stats.update_one(
                {"_id": pid},
                {"$setOnInsert": {
                    "name": info["name"],
                    "price": info["price"],
                    "stats": {
                        "lifetime": {"views": 0, "carts": 0, "purchases": 0, "revenue": 0.0},
                        "today": {"views": 0, "carts": 0, "purchases": 0, "revenue": 0.0}
                    },
                    "updated_at": datetime.now()
                }},
                upsert=True
            )
    
    def process_event(self, event: dict) -> str:
        """
        Process a single event through all three layers.
        
        Returns: inserted event ID
        """
        event_time = event.get("created_at", datetime.now())
        date_key = event_time.strftime("%Y-%m-%d")
        hour_key = event_time.strftime("%H")
        event_type = event["event_type"]
        product_id = event.get("product_id")
        device = event.get("metadata", {}).get("device", "desktop")
        
        # ─── LAYER 1: Raw event storage ──────────────
        event["created_at"] = event_time
        result = self.events.insert_one(event)
        
        # ─── LAYER 2A: Daily stats update ────────────
        daily_inc = {}
        
        if event_type == "page_view":
            daily_inc["views.count"] = 1
            daily_inc[f"by_device.{device}.views"] = 1
        
        elif event_type == "add_to_cart":
            daily_inc["carts.count"] = 1
        
        elif event_type == "purchase":
            amount = event.get("amount", 0)
            quantity = event.get("quantity", 1)
            daily_inc["orders.count"] = 1
            daily_inc["orders.revenue"] = amount
            daily_inc["orders.quantity"] = quantity
            daily_inc[f"orders_by_hour.{hour_key}.count"] = 1
            daily_inc[f"orders_by_hour.{hour_key}.revenue"] = amount
            daily_inc[f"by_device.{device}.purchases"] = 1
        
        if daily_inc:
            self.daily_stats.update_one(
                {"_id": date_key},
                {
                    "$inc": daily_inc,
                    "$set": {"updated_at": datetime.now()},
                    "$setOnInsert": {"created_at": datetime.now()}
                },
                upsert=True
            )
        
        # ─── LAYER 2B: Product stats update ──────────
        if product_id:
            product_inc = {}
            
            if event_type == "page_view":
                product_inc["stats.lifetime.views"] = 1
                product_inc["stats.today.views"] = 1
            
            elif event_type == "add_to_cart":
                product_inc["stats.lifetime.carts"] = 1
                product_inc["stats.today.carts"] = 1
            
            elif event_type == "purchase":
                amount = event.get("amount", 0)
                product_inc["stats.lifetime.purchases"] = 1
                product_inc["stats.lifetime.revenue"] = amount
                product_inc["stats.today.purchases"] = 1
                product_inc["stats.today.revenue"] = amount
            
            if product_inc:
                self.product_stats.update_one(
                    {"_id": product_id},
                    {
                        "$inc": product_inc,
                        "$set": {"updated_at": datetime.now()}
                    },
                    upsert=True
                )
        
        return str(result.inserted_id)
    
    def process_batch(self, event_list: list) -> int:
        """
        Process multiple events efficiently.
        
        NOTE: Even in batch, each event updates computed docs individually.
        Could optimize with bulk_write, but $inc operations are already fast.
        """
        count = 0
        for event in event_list:
            self.process_event(event)
            count += 1
        return count
    
    def get_dashboard(self, date_key: str = None) -> dict:
        """
        Read dashboard data — ONE document read, instant response.
        
        Redshift equivalent:
            SELECT COUNT(*), SUM(amount), AVG(amount), ...
            FROM analytics.events
            WHERE created_at >= CURRENT_DATE;
            -- 2-5 seconds for 30M rows
        
        MongoDB computed:
            findOne({ _id: "2024-05-31" })
            -- <1ms always, regardless of event count
        """
        if date_key is None:
            date_key = datetime.now().strftime("%Y-%m-%d")
        
        raw = self.daily_stats.find_one({"_id": date_key})
        
        if not raw:
            return {"error": f"No data for {date_key}"}
        
        orders = raw.get("orders", {})
        views = raw.get("views", {})
        carts = raw.get("carts", {})
        
        order_count = orders.get("count", 0)
        revenue = orders.get("revenue", 0)
        view_count = views.get("count", 0)
        cart_count = carts.get("count", 0)
        
        # DERIVED values — computed at read time from pre-computed components
        # These are cheap to derive (division) vs expensive to aggregate (scan)
        avg_order = revenue / order_count if order_count > 0 else 0
        cart_to_purchase = (order_count / cart_count * 100) if cart_count > 0 else 0
        view_to_purchase = (order_count / view_count * 100) if view_count > 0 else 0
        
        return {
            "date": date_key,
            "orders": order_count,
            "revenue": round(revenue, 2),
            "avg_order_value": round(avg_order, 2),
            "views": view_count,
            "carts": cart_count,
            "conversion_rate": round(view_to_purchase, 2),
            "cart_conversion": round(cart_to_purchase, 2),
            "by_device": raw.get("by_device", {}),
            "orders_by_hour": raw.get("orders_by_hour", {}),
            "top_products": raw.get("top_products_revenue", [])
        }
    
    def get_product_stats(self, product_id: str) -> dict:
        """
        Read product stats — ONE document read.
        
        Redshift equivalent:
            SELECT COUNT(*) FILTER (WHERE event_type = 'page_view') AS views,
                   COUNT(*) FILTER (WHERE event_type = 'purchase') AS purchases,
                   SUM(amount) FILTER (WHERE event_type = 'purchase') AS revenue
            FROM analytics.events
            WHERE product_id = :pid;
            -- 1-5 seconds per product
        
        MongoDB computed:
            findOne({ _id: product_id })
            -- <1ms always
        """
        raw = self.product_stats.find_one({"_id": product_id})
        if not raw:
            return {"error": f"No stats for {product_id}"}
        
        lt = raw.get("stats", {}).get("lifetime", {})
        td = raw.get("stats", {}).get("today", {})
        
        views = lt.get("views", 0)
        purchases = lt.get("purchases", 0)
        carts = lt.get("carts", 0)
        
        return {
            "product_id": product_id,
            "name": raw.get("name", ""),
            "price": raw.get("price", 0),
            "lifetime": {
                "views": views,
                "carts": carts,
                "purchases": purchases,
                "revenue": round(lt.get("revenue", 0), 2),
                "view_to_cart": round(carts / views * 100, 1) if views > 0 else 0,
                "cart_to_purchase": round(purchases / carts * 100, 1) if carts > 0 else 0,
                "view_to_purchase": round(purchases / views * 100, 1) if views > 0 else 0
            },
            "today": {
                "views": td.get("views", 0),
                "carts": td.get("carts", 0),
                "purchases": td.get("purchases", 0),
                "revenue": round(td.get("revenue", 0), 2)
            }
        }
    
    def refresh_top_products(self, date_key: str = None, top_n: int = 5):
        """
        Periodic refresh for ranked data (can't use $inc for rankings).
        
        Called by: scheduled job every 1-5 minutes
        
        Redshift equivalent:
            REFRESH MATERIALIZED VIEW analytics.top_products_daily;
        """
        if date_key is None:
            date_key = datetime.now().strftime("%Y-%m-%d")
        
        top = list(
            self.product_stats.find(
                {"stats.today.revenue": {"$gt": 0}},
                {"name": 1, "stats.today.revenue": 1, "stats.today.purchases": 1}
            ).sort("stats.today.revenue", DESCENDING).limit(top_n)
        )
        
        top_list = [{
            "product_id": p["_id"],
            "name": p.get("name", ""),
            "revenue": p["stats"]["today"]["revenue"],
            "count": p["stats"]["today"]["purchases"]
        } for p in top]
        
        self.daily_stats.update_one(
            {"_id": date_key},
            {"$set": {
                "top_products_revenue": top_list,
                "updated_at": datetime.now()
            }}
        )
        
        return top_list
    
    def midnight_rollover(self):
        """
        Reset 'today' counters at midnight.
        
        THIS IS THE COST OF WRITE-TIME COMPUTATION:
            → Redshift doesn't need this (WHERE created_at >= CURRENT_DATE)
            → MongoDB must explicitly reset counters
            → Trade-off: slightly more operational complexity for massively faster reads
        
        Run via: Atlas Scheduled Trigger or cron job at 00:00:01
        """
        # Reset all product "today" stats
        result = self.product_stats.update_many(
            {},
            {"$set": {
                "stats.today": {
                    "views": 0, "carts": 0,
                    "purchases": 0, "revenue": 0.0
                }
            }}
        )
        
        # Initialize new daily_stats document
        new_day = datetime.now().strftime("%Y-%m-%d")
        self.daily_stats.update_one(
            {"_id": new_day},
            {"$setOnInsert": {
                "orders": {"count": 0, "revenue": 0, "quantity": 0},
                "views": {"count": 0, "unique_users": 0},
                "carts": {"count": 0},
                "orders_by_hour": {},
                "top_products_revenue": [],
                "by_device": {
                    "mobile": {"views": 0, "purchases": 0},
                    "desktop": {"views": 0, "purchases": 0},
                    "tablet": {"views": 0, "purchases": 0}
                },
                "created_at": datetime.now(),
                "updated_at": datetime.now()
            }},
            upsert=True
        )
        
        return result.modified_count
    
    def rebuild_daily_stats(self, date_key: str):
        """
        RECOVERY: Rebuild computed values from raw events.
        
        When to use:
            → Suspected data corruption in computed docs
            → After a bug fix in processEvent logic
            → Periodic validation (compare computed vs aggregated)
        
        Redshift equivalent:
            REFRESH MATERIALIZED VIEW analytics.daily_product_stats;
            -- One command, Redshift handles the rest
        
        MongoDB:
            → Must manually aggregate and replace
            → More code, but more control over the process
        """
        start_date = datetime.strptime(date_key, "%Y-%m-%d")
        end_date = start_date + timedelta(days=1)
        
        pipeline = [
            {"$match": {
                "created_at": {"$gte": start_date, "$lt": end_date}
            }},
            {"$facet": {
                "orders": [
                    {"$match": {"event_type": "purchase"}},
                    {"$group": {
                        "_id": None,
                        "count": {"$sum": 1},
                        "revenue": {"$sum": "$amount"},
                        "quantity": {"$sum": {"$ifNull": ["$quantity", 1]}}
                    }}
                ],
                "views": [
                    {"$match": {"event_type": "page_view"}},
                    {"$group": {
                        "_id": None,
                        "count": {"$sum": 1},
                        "unique_users": {"$addToSet": "$user_id"}
                    }},
                    {"$project": {
                        "count": 1,
                        "unique_users": {"$size": "$unique_users"}
                    }}
                ],
                "carts": [
                    {"$match": {"event_type": "add_to_cart"}},
                    {"$count": "count"}
                ],
                "by_hour": [
                    {"$match": {"event_type": "purchase"}},
                    {"$group": {
                        "_id": {"$hour": "$created_at"},
                        "count": {"$sum": 1},
                        "revenue": {"$sum": "$amount"}
                    }},
                    {"$sort": {"_id": 1}}
                ],
                "by_device": [
                    {"$group": {
                        "_id": {
                            "device": {"$ifNull": ["$metadata.device", "desktop"]},
                            "type": "$event_type"
                        },
                        "count": {"$sum": 1}
                    }}
                ]
            }}
        ]
        
        result = list(self.events.aggregate(pipeline))
        if not result:
            return None
        
        facets = result[0]
        
        # Extract facet results with defaults
        orders = facets["orders"][0] if facets["orders"] else {"count": 0, "revenue": 0, "quantity": 0}
        views = facets["views"][0] if facets["views"] else {"count": 0, "unique_users": 0}
        carts = facets["carts"][0] if facets["carts"] else {"count": 0}
        
        # Build hourly map
        hourly = {}
        for h in facets["by_hour"]:
            hour_str = str(h["_id"]).zfill(2)
            hourly[hour_str] = {"count": h["count"], "revenue": h["revenue"]}
        
        # Build device map
        device_map = {
            "mobile": {"views": 0, "purchases": 0},
            "desktop": {"views": 0, "purchases": 0},
            "tablet": {"views": 0, "purchases": 0}
        }
        for d in facets["by_device"]:
            device = d["_id"]["device"]
            etype = d["_id"]["type"]
            if device not in device_map:
                device_map[device] = {"views": 0, "purchases": 0}
            if etype == "page_view":
                device_map[device]["views"] = d["count"]
            elif etype == "purchase":
                device_map[device]["purchases"] = d["count"]
        
        # Replace computed document
        self.daily_stats.update_one(
            {"_id": date_key},
            {"$set": {
                "orders": {
                    "count": orders["count"],
                    "revenue": orders["revenue"],
                    "quantity": orders.get("quantity", 0)
                },
                "views": {
                    "count": views["count"],
                    "unique_users": views.get("unique_users", 0)
                },
                "carts": {"count": carts["count"]},
                "orders_by_hour": hourly,
                "by_device": device_map,
                "rebuilt_at": datetime.now(),
                "updated_at": datetime.now()
            }},
            upsert=True
        )
        
        return {
            "orders": orders["count"],
            "revenue": orders["revenue"],
            "views": views["count"],
            "carts": carts["count"]
        }
    
    def rebuild_product_stats(self, product_id: str):
        """Rebuild a single product's computed stats from raw events."""
        
        pipeline = [
            {"$match": {"product_id": product_id}},
            {"$group": {
                "_id": "$event_type",
                "count": {"$sum": 1},
                "total_amount": {"$sum": {"$ifNull": ["$amount", 0]}}
            }}
        ]
        
        results = {r["_id"]: r for r in self.events.aggregate(pipeline)}
        
        views = results.get("page_view", {}).get("count", 0)
        carts = results.get("add_to_cart", {}).get("count", 0)
        purchases = results.get("purchase", {}).get("count", 0)
        revenue = results.get("purchase", {}).get("total_amount", 0)
        
        # Rebuild today's stats separately
        today_start = datetime.now().replace(hour=0, minute=0, second=0, microsecond=0)
        today_pipeline = [
            {"$match": {"product_id": product_id, "created_at": {"$gte": today_start}}},
            {"$group": {
                "_id": "$event_type",
                "count": {"$sum": 1},
                "total_amount": {"$sum": {"$ifNull": ["$amount", 0]}}
            }}
        ]
        
        today_results = {r["_id"]: r for r in self.events.aggregate(today_pipeline)}
        
        self.product_stats.update_one(
            {"_id": product_id},
            {"$set": {
                "stats.lifetime": {
                    "views": views,
                    "carts": carts,
                    "purchases": purchases,
                    "revenue": round(revenue, 2)
                },
                "stats.today": {
                    "views": today_results.get("page_view", {}).get("count", 0),
                    "carts": today_results.get("add_to_cart", {}).get("count", 0),
                    "purchases": today_results.get("purchase", {}).get("count", 0),
                    "revenue": round(today_results.get("purchase", {}).get("total_amount", 0), 2)
                },
                "rebuilt_at": datetime.now(),
                "updated_at": datetime.now()
            }},
            upsert=True
        )
        
        return {"views": views, "carts": carts, "purchases": purchases, "revenue": revenue}

✅ WHAT SHOULD DO (Best Practices)

✅ USE $inc FOR ALL SIMPLE COUNTERS
    → $inc is ATOMIC — no read-modify-write race conditions
    → Two concurrent purchases both $inc orders.count by 1
    → MongoDB guarantees both increments apply correctly
    → No locking, no transactions needed for counters
    → This is MongoDB's KILLER FEATURE for the Computed Pattern

✅ DERIVE RATIOS AT READ TIME, NOT WRITE TIME
    → STORE: orders.count, orders.revenue
    → DERIVE: avg_order_value = revenue / count (at read time)
    → WHY: division is cheap, storing avoids recomputation of averages
    → WRONG: storing avg_order_value and recalculating on every event
      (recalculating running average requires knowing previous count)
    
    WHAT TO STORE vs WHAT TO DERIVE:
    ┌────────────────────────┬────────────────────────────────┐
    │ STORE (via $inc)        │ DERIVE (at read time)           │
    ├────────────────────────┼────────────────────────────────┤
    │ count                   │ rate (count / time)             │
    │ sum                     │ average (sum / count)           │
    │ min (via $min)          │ range (max - min)               │
    │ max (via $max)          │ percentage (part / whole * 100) │
    │ count_by_category       │ distribution (cat / total)      │
    └────────────────────────┴────────────────────────────────┘

✅ ALWAYS KEEP RAW EVENTS (Source of Truth)
    → Computed values are DERIVED — can be rebuilt
    → Raw events are ORIGINAL — cannot be recreated
    → If computed docs get corrupted → rebuild from raw
    → If business logic changes → recompute from raw
    → Use TTL index to auto-delete old events (90 days, 1 year)
    → Archive to S3 before TTL expiry for long-term analytics

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT COMPUTE EVERYTHING AT QUERY TIME IN MONGODB
    REDSHIFT THINKING:
        → "Just run SELECT COUNT(*) — it's fast enough"
    MONGODB REALITY:
        → COUNT on 30M documents: 30-120 seconds
        → Columnar engines skip irrelevant columns
        → Document engines read ENTIRE documents
        → MUST pre-compute for any metric read frequently

❌ DO NOT STORE RUNNING AVERAGES VIA $inc
    WRONG:
        → $inc: { "avg_order_value": (newAmount - currentAvg) / newCount }
        → Requires READING current value first → race condition
        → Two concurrent updates → incorrect average
    RIGHT:
        → $inc: { "orders.count": 1, "orders.revenue": amount }
        → Derive average at read time: revenue / count
        → Both $inc operations are atomic and independent

❌ DO NOT USE TRANSACTIONS FOR COUNTER UPDATES
    WRONG:

        events.insert_one(event, session=session)
        daily_stats.update_one({...}, {"$inc": {...}}, session=session)
        product_stats.update_one({...}, {"$inc": {...}}, session=session)
        session.commit_transaction()
    WHY:
        → Transactions add latency and lock contention
        → $inc is already atomic per document
        → If product_stats update fails, it's recoverable (rebuild)
        → Eventual consistency between raw and computed is ACCEPTABLE
    RIGHT:
        → Insert raw event first (source of truth)
        → Update computed docs without transaction
        → If update fails → retry or catch in periodic rebuild
        → ONLY use transactions for financial accuracy (Project 28)

❌ DO NOT PRE-COMPUTE EVERYTHING
    WRONG:
        → Pre-compute revenue by product by country by device by hour
        → Combinatorial explosion: 8 products × 7 countries × 3 devices × 24 hours = 4032 counters
        → Every event updates dozens of computed fields
    RIGHT:
        → Pre-compute only what the DASHBOARD needs
        → For ad-hoc slicing → query raw events or use Redshift
        → The Computed Pattern is for KNOWN access patterns
        → Unknown queries → aggregation pipeline or Redshift

❌ DO NOT FORGET MIDNIGHT ROLLOVER
    REDSHIFT THINKING:
        → WHERE created_at >= CURRENT_DATE handles "today" automatically
        → No rollover logic needed
    MONGODB TRAP:
        → "today" counters in product_stats must be RESET at midnight
        → If you forget → "today" shows cumulative lifetime stats
        → Must have a scheduled job (cron / Atlas Trigger)
        → Test your rollover logic across timezone boundaries!

❌ DO NOT USE float FOR FINANCIAL AMOUNTS
    WRONG:
        → $inc: { "orders.revenue": 29.99 }
        → JavaScript floating point: 0.1 + 0.2 = 0.30000000000000004
        → After millions of $inc operations → significant drift
    RIGHT (for critical accuracy):
        → Use Decimal128: NumberDecimal("29.99")
        → Or store cents as integers: $inc: { revenue_cents: 2999 }
        → Divide by 100 at display time
        → For dashboards: float is often acceptable (tiny drift)
        → For financial reporting: use Decimal128 or integer cents

MongoDB: Why $inc Is Perfect for Computed Pattern

$inc OPERATION INTERNALS:

1. APPLICATION SENDS:
   db.daily_stats.updateOne(
       { _id: "2024-05-31" },
       { $inc: { "orders.count": 1, "orders.revenue": 299.99 } }
   )

2. WIREDTIGER RECEIVES UPDATE:
   → Find document with _id = "2024-05-31" (B-tree lookup via _id index)
   → Load document into WiredTiger cache (if not already there)
   → Apply $inc IN PLACE: orders.count = 14832 + 1 = 14833
   → Apply $inc IN PLACE: orders.revenue = 847291.44 + 299.99 = 847591.43
   → Mark page as DIRTY in cache
   → Write to journal (WAL) for durability
   → Return success to client

3. WHY THIS IS FAST:
   → _id lookup: O(log N) via B-tree = ~microseconds
   → Document already in cache (hot document) = no disk I/O
   → $inc modifies bytes IN PLACE = no document rewrite
   → No read-modify-write cycle = no race conditions
   → Journal write: sequential append = fast
   
   TOTAL LATENCY: ~0.1-1ms per $inc operation

4. CONCURRENCY — THE KEY ADVANTAGE:
   → Two concurrent purchases at the same instant:
   
   Thread A: $inc { "orders.count": 1, "orders.revenue": 299.99 }
   Thread B: $inc { "orders.count": 1, "orders.revenue": 89.99 }
   
   WiredTiger handles this with DOCUMENT-LEVEL LOCKING:
   → Thread A acquires intent lock on document
   → Thread A applies $inc: count = 14832 → 14833
   → Thread A releases lock
   → Thread B acquires lock
   → Thread B applies $inc: count = 14833 → 14834
   → Thread B releases lock
   
   RESULT: count = 14834 (both increments applied correctly)
   
   NO TRANSACTIONS NEEDED — $inc is atomic per document
   
   Compare to Redshift:
   → UPDATE daily_stats SET orders = orders + 1
   → Redshift uses MVCC (Multi-Version Concurrency Control)
   → Serializable isolation by default
   → Two concurrent UPDATEs may cause serialization failure
   → Must handle retries in application code
   → MongoDB $inc: no retries needed, always succeeds

5. WHEN $inc FAILS:
   → Document doesn't exist AND no upsert → no change (modified: 0)
   → Network timeout → unknown state (check or retry)
   → Write concern not met (replica lag) → write concern error
   → NONE of these cause data corruption
   → Worst case: counter slightly behind (catch up on next event)

Redshift: Why Query-Time Computation Works There

REDSHIFT QUERY-TIME COMPUTATION INTERNALS:

1. QUERY ARRIVES:
   SELECT COUNT(*), SUM(amount)
   FROM events WHERE event_type = 'purchase' AND created_at >= CURRENT_DATE

2. LEADER NODE:
   → Parse SQL → create execution plan
   → Determine which slices have relevant data
   → Push predicate to compute nodes

3. EACH COMPUTE SLICE:
   → Check zone maps for event_type column blocks
     → Block min/max: if max(event_type) < 'purchase' → SKIP
   → Check zone maps for created_at column blocks
     → Block min/max: if max(created_at) < today → SKIP entire block
   → For matching blocks:
     → Read event_type column block (1 MB compressed → ~50 KB)
     → Read amount column block (only for purchase events)
     → Apply predicate filter
     → Compute partial COUNT and SUM
     → Return partial results to leader

4. LEADER NODE:
   → Receive partial results from all slices
   → Merge: total_count = sum(partial_counts)
   → Merge: total_sum = sum(partial_sums)
   → Return to client

5. WHY THIS IS FAST (DESPITE SCANNING MILLIONS OF ROWS):
   → Zone maps skip 99%+ of blocks (today's data = last blocks only)
   → Columnar: reads ONLY event_type + amount columns (~2 of 12 columns)
   → Compressed: 1 MB block decompresses from ~50 KB
   → Parallel: 8 slices scan simultaneously
   → Effective scan: ~5 MB of data for 30M row table
   
   COMPARED TO MONGODB:
   → MongoDB must read ENTIRE documents (all 12 fields per doc)
   → No columnar skip → reads ~6 GB instead of ~5 MB
   → Single-threaded per shard → no MPP parallelism
   → 1000x more I/O → 100x slower

6. RESULT CACHE:
   → First query: 2-5 seconds (full computation)
   → Same query within 15 minutes: <100ms (cached result)
   → Cache invalidated when table data changes
   → Good for dashboards that refresh periodically
   → NOT good for sub-second real-time requirements

PITFALL 1: "I'll just run aggregation on every dashboard refresh"
    REDSHIFT THINKING:

        → Aggregation from raw events shows 14,835

        → Rebuild function ready to run on demand
        → Accept small temporary drift (<0.1%) for non-financial metrics

📐 DATA MODELING COMPARISON

THIS PROBLEM — FOUR APPROACHES:

1. REDSHIFT PURE QUERY-TIME:
    events table → SELECT COUNT/SUM at dashboard load
    ✓ Simple writes, simple schema
    ✓ Any ad-hoc query works
    ✗ 2-10 second latency per dashboard refresh
    ✗ Can't serve real-time (<100ms) requirements
    BEST FOR: Analytics dashboards, BI tools

2. REDSHIFT + MATERIALIZED VIEW:
    events table → MV refreshed every 5 minutes
    ✓ Faster reads from MV (~200ms)
    ✓ Still supports ad-hoc queries on raw table
    ✗ Stale between refreshes (5-min lag)
    ✗ MV refresh is expensive at scale
    BEST FOR: Near-real-time analytics dashboards

3. MONGODB COMPUTED PATTERN:
    events collection + computed docs (daily_stats, product_stats)
    ✓ Sub-millisecond dashboard reads
    ✓ Always fresh (updated on every event)
    ✗ More complex write logic
    ✗ Limited to pre-defined metrics
    ✗ Must maintain rollover/rebuild logic
    BEST FOR: Real-time operational dashboards, product pages

4. MONGODB + REDSHIFT HYBRID:
    MongoDB: events + computed docs → real-time dashboard
    Redshift: events (batch loaded) → ad-hoc analytics
    ✓ Sub-millisecond for operational dashboard (MongoDB)
    ✓ Any ad-hoc query for analysts (Redshift)
    ✓ Each system does what it's best at
    ✗ Data pipeline complexity (sync MongoDB → Redshift)
    ✗ Higher infrastructure cost (two systems)
    BEST FOR: Production systems at scale (most common in real world)

VISUAL COMPARISON:

    APPROACH              WRITE COST    READ LATENCY    FLEXIBILITY
    ─────────────────────────────────────────────────────────────────
    Redshift query-time   ●○○○○         ●●●○○ (2-10s)   ●●●●●
    Redshift + MV         ●●○○○         ●●●●○ (200ms)   ●●●●○
    MongoDB computed      ●●●○○         ●●●●● (<1ms)    ●●○○○
    Hybrid (both)         ●●●●○         ●●●●● (<1ms)    ●●●●●

    ● = more is better for read latency and flexibility
    ● = less is better for write cost

WHEN TO BREAK CONVENTION:
    → MongoDB computed but need ad-hoc → add aggregation pipeline queries on events
    → Redshift query-time but need real-time → add Redis counters in front
    → Hybrid too expensive → choose based on PRIMARY use case
    → If dashboard is internal (10 users) → Redshift query-time is fine
    → If dashboard is customer-facing (10,000 users) → MongoDB computed is necessary

⚠️ COMMON PITFALLS AND FAILURE MODES (Additional)

PITFALL 6: "Using $set instead of $inc for counters"
    WRONG:
        // Read current value
        let current = db.daily_stats.findOne({ _id: today })
        let newCount = current.orders.count + 1
        // Write new value
        db.daily_stats.updateOne(
            { _id: today },
            { $set: { "orders.count": newCount } }
        )
    WHY IT FAILS:
        → Read-modify-write = RACE CONDITION
        → Thread A reads count = 100
        → Thread B reads count = 100
        → Thread A writes count = 101
        → Thread B writes count = 101 ← SHOULD BE 102!
        → Lost update: one event disappeared
    RIGHT:
        db.daily_stats.updateOne(
            { _id: today },
            { $inc: { "orders.count": 1 } }
        )
        → $inc is ATOMIC — MongoDB handles concurrency internally
        → Both threads: count goes 100 → 101 → 102 correctly

PITFALL 7: "Not handling the cold start / first event of the day"
    SYMPTOM:
        → First purchase at 00:00:01 → updateOne with $inc
        → Document doesn't exist yet → modified: 0
        → Counter never starts
    FIX:
        → ALWAYS use upsert: true with $setOnInsert for defaults
        → db.daily_stats.updateOne(
            { _id: dateKey },
            {
                $inc: { "orders.count": 1 },
                $setOnInsert: {
                    views: { count: 0 },
                    carts: { count: 0 },
                    created_at: new Date()
                }
            },
            { upsert: true }
          )
        → First event creates document WITH defaults + applies $inc
        → Subsequent events just $inc (upsert has no effect)

PITFALL 8: "Redshift developer tries to normalize computed data"
    WRONG THINKING:
        → "I should have separate collections: order_counts, revenue_sums, view_counts"
        → "Each with their own documents, indexed independently"
        → "Like separate fact tables in Redshift"
    WHY IT FAILS IN MONGODB:
        → Dashboard needs ALL metrics → 5 separate queries instead of 1
        → 5 round-trips × network latency = 5-25ms instead of <1ms
        → More collections = more index memory = more cache pressure
    RIGHT:
        → ONE document per time period (daily_stats)
        → Contains ALL dashboard metrics
        → ONE read = entire dashboard
        → This is DENORMALIZATION — the MongoDB way
        → Opposite of Redshift instinct but correct for document model

PITFALL 9: "Forgetting that $inc creates fields automatically"
    SURPRISE BEHAVIOR:
        → $inc: { "orders_by_hour.14.count": 1 }
        → If orders_by_hour doesn't exist → CREATED as { "14": { "count": 1 } }
        → If orders_by_hour.14 doesn't exist → CREATED as { "count": 1 }
        → $inc auto-creates the entire nested path
    WHY THIS MATTERS:
        → You DON'T need to pre-initialize every hour slot
        → Hours with zero events simply don't have entries
        → At read time: missing hour = zero (handle in application)
        → This is GOOD — reduces initialization complexity
    TRAP:
        → Don't assume all 24 hours exist in orders_by_hour
        → Application code must handle: orders_by_hour["03"] || { count: 0, revenue: 0 }

📈 SCALABILITY CONSIDERATIONS (Extended)

SCALING COMPUTED PATTERN — ADVANCED STRATEGIES:

STRATEGY 1: SHARDED COUNTERS (for extreme write throughput)
    PROBLEM:
        → At 50,000 events/second, ALL events update ONE daily_stats document
        → Single document = single shard = write bottleneck (hot shard)
    
    SOLUTION:
        → Split into N counter documents (e.g., 10 shards)
        → daily_stats_0, daily_stats_1, ..., daily_stats_9
        → Each event: random shard → $inc on that shard's document
        → Dashboard read: aggregate all N shards
        
        // Write (distributed across 10 documents)
        let shard = Math.floor(Math.random() * 10)
        db.daily_stats.updateOne(
            { _id: `${dateKey}_shard_${shard}` },
            { $inc: { "orders.count": 1, "orders.revenue": amount } },
            { upsert: true }
        )
        
        // Read (aggregate all shards — still fast, just 10 docs)
        let totals = db.daily_stats.aggregate([
            { $match: { _id: { $regex: `^${dateKey}_shard_` } } },
            { $group: {
                _id: null,
                total_orders: { $sum: "$orders.count" },
                total_revenue: { $sum: "$orders.revenue" }
            }}
        ]).toArray()[0]
    
    TRADE-OFF:
        → Writes: distributed (no hot document) → higher throughput
        → Reads: aggregate 10 docs instead of 1 → still <5ms
        → Complexity: moderate increase

STRATEGY 2: TIME-BUCKETED COMPUTED DOCUMENTS
    → Instead of one daily_stats per day:
    → One document per 5-minute bucket
    
    {
        _id: "2024-05-31T14:30",    // 5-minute bucket
        orders: { count: 47, revenue: 2841.50 },
        views: { count: 892 },
        // ...
    }
    
    → Dashboard: aggregate last 12 buckets (1 hour) = 12 doc reads
    → Sparkline: each bucket = one data point
    → Natural TTL: old buckets auto-deleted
    → No midnight rollover problem (each bucket is self-contained)
    → Builds on: Project 16 Bucket Pattern

STRATEGY 3: TIERED COMPUTATION
    → Tier 1 (real-time): minute-level counters via $inc
    → Tier 2 (near-real-time): hourly aggregation from Tier 1 (cron every 5 min)
    → Tier 3 (batch): daily summary from Tier 2 (cron at midnight)
    → Tier 4 (analytical): monthly/yearly from Tier 3 → Redshift
    
    ┌─────────────┐   aggregate   ┌──────────────┐   aggregate
    │ Minute-level │ ───────────→ │ Hourly-level  │ ──────────→ ...
    │ (real-time)  │   every 5m   │ (near-real)   │   midnight
    │ via $inc     │              │ via agg pipe   │
    └─────────────┘              └──────────────┘
    
    → Each tier is smaller and cheaper to query than raw events
    → Dashboard reads from appropriate tier based on time range
    → Last hour → minute-level (Tier 1)
    → Last week → hourly-level (Tier 2)
    → Last year → daily-level (Tier 3) or Redshift (Tier 4)

REDSHIFT SCALING COMPARISON:
    → Redshift handles all tiers with ONE query
    → Sort key + zone maps automatically optimize time ranges
    → No tiered architecture needed
    → Just add nodes for more compute power
    → Simpler architecture, higher per-query latency
    → Trade-off: simplicity (Redshift) vs speed (MongoDB computed)
