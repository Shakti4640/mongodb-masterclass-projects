PROJECT 34: MongoDB as Operational Store Feeding Redshift Warehouse

THE Hybrid Architecture — Where Both Systems Work Together

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    You are the lead data engineer at "ShopStream" — a mid-size 
    e-commerce platform processing 50,000 orders per day
    
    THE BUSINESS HAS TWO FUNDAMENTALLY DIFFERENT NEEDS:
    
    NEED 1 — OPERATIONAL (real-time, user-facing):
        → Product catalog with flexible attributes (Project 7, 18)
        → Shopping cart: add/remove/update items sub-millisecond
        → User sessions: track browsing behavior in real-time
        → Order processing: create, update status, track shipments
        → Inventory: decrement stock atomically on purchase (Project 27)
        → Personalization: "customers who bought X also bought Y"
        → 10,000 concurrent users, <10ms response time
        → Schema changes weekly (new product types, new features)
    
    NEED 2 — ANALYTICAL (batch/near-real-time, analyst-facing):
        → Daily revenue by category, region, customer segment
        → Cohort analysis: retention curves by signup month
        → Funnel analysis: browse → cart → checkout → purchase conversion
        → Inventory forecasting: predict stockouts by SKU
        → Customer lifetime value (CLV) calculations
        → Marketing attribution: which campaigns drive revenue
        → Complex JOINs across 15+ dimension tables
        → BI dashboards in Tableau/QuickSight refreshed hourly
    
    THE QUESTION:
        Can ONE database handle both?
        
    THE ANSWER:
        No. And trying to force one system to do both = the #1 
        architecture mistake in modern data engineering.
    
    THE SOLUTION:
        MongoDB for NEED 1 + Redshift for NEED 2
        Connected by a real-time data pipeline
        
    THIS IS THE MOST COMMON PRODUCTION ARCHITECTURE IN THE INDUSTRY

🔴 REDSHIFT MINDSET — "Can't We Just Use Redshift for Everything?"

Why Teams Try This (And Why It Fails)

THE TEMPTATION:
    "We already have Redshift. Why add MongoDB?"
    "SQL is universal. Everyone knows it."
    "One database = simpler architecture."
    
WHAT HAPPENS WHEN YOU TRY:

    ┌────────────────────────────────────────────────────────────┐
    │  REDSHIFT AS BOTH OPERATIONAL + ANALYTICAL                  │
    │                                                              │
    │  Problem 1: LATENCY                                         │
    │  → Cart "add item" operation                                │
    │  → Redshift: INSERT single row → 50-200ms                   │
    │  → User expects: <10ms                                      │
    │  → Columnar storage penalizes single-row operations          │
    │  → (We proved this in Project 1 internals)                   │
    │                                                              │
    │  Problem 2: CONCURRENCY                                      │
    │  → 10,000 users simultaneously updating carts                │
    │  → Redshift WLM: ~50 concurrent queries (even with scaling)  │
    │  → Queue depth grows → timeouts → angry users                │
    │  → (We discussed WLM limits in Project 15)                   │
    │                                                              │
    │  Problem 3: SCHEMA RIGIDITY                                  │
    │  → New product type needs new attributes                     │
    │  → ALTER TABLE ADD COLUMN → blocks writes                    │
    │  → Wide sparse tables → wasted storage + slow scans          │
    │  → (We saw this problem in Project 7)                        │
    │                                                              │
    │  Problem 4: WRITE AMPLIFICATION                              │
    │  → Every cart update → columnar write across all columns     │
    │  → Redshift designed for bulk COPY, not single-row writes    │
    │  → VACUUM required frequently → maintenance overhead         │
    │                                                              │
    │  Problem 5: COST                                             │
    │  → Keeping Redshift cluster large enough for OLTP concurrency│
    │  → Paying for compute 24/7 even during low-traffic hours     │
    │  → Concurrency Scaling charges add up quickly                │
    └────────────────────────────────────────────────────────────┘
    
    VERDICT: Redshift alone CANNOT serve operational workloads
             It was never designed to. That's not a weakness — it's a focus.

What "Redshift for Everything" Architecture Looks Like

-- THE PAINFUL APPROACH: Everything in Redshift

-- Operational tables (fighting against Redshift's nature)
CREATE TABLE ops.shopping_carts (
    cart_id         BIGINT IDENTITY(1,1),
    user_id         BIGINT NOT NULL,
    status          VARCHAR(20) DEFAULT 'active',
    items           SUPER,              -- Redshift SUPER type for semi-structured
    updated_at      TIMESTAMP DEFAULT GETDATE()
)
DISTSTYLE KEY DISTKEY(user_id) SORTKEY(updated_at);

-- Every cart update = single-row UPDATE on columnar storage = SLOW
UPDATE ops.shopping_carts 
SET items = JSON_PARSE('[{"sku":"ABC","qty":2},{"sku":"DEF","qty":1}]'),
    updated_at = GETDATE()
WHERE cart_id = 98765;
-- Latency: 50-200ms (unacceptable for user-facing)

-- Analytical tables (what Redshift is ACTUALLY good at)
CREATE TABLE analytics.daily_revenue (
    order_date      DATE,
    category        VARCHAR(100),
    region          VARCHAR(50),
    total_revenue   DECIMAL(15,2),
    order_count     INTEGER
)
DISTSTYLE KEY DISTKEY(category) SORTKEY(order_date);

-- THIS query is fast — Redshift was born for this
SELECT category, DATE_TRUNC('month', order_date), SUM(total_revenue)
FROM analytics.daily_revenue
WHERE order_date >= '2024-01-01'
GROUP BY 1, 2
ORDER BY 3 DESC;
-- Latency: 2-5 seconds across billions of rows = EXCELLENT

WHAT HAPPENS WHEN YOU TRY:

    ┌────────────────────────────────────────────────────────────┐
    │  MONGODB AS BOTH OPERATIONAL + ANALYTICAL                   │
    │                                                              │
    │  Operational workloads: ✅ EXCELLENT                         │
    │  → Cart operations: <5ms ✅                                  │
    │  → 10,000 concurrent users: handled ✅                       │
    │  → Flexible schema: native ✅                                │
    │  → Single-document atomicity: free ✅ (Project 27)           │
    │                                                              │
    │  Analytical workloads: ⚠️ STRUGGLES                          │
    │                                                              │
    │  Problem 1: FULL COLLECTION SCANS                            │
    │  → "Revenue by category by month for past 3 years"           │
    │  → MongoDB reads ENTIRE documents (all fields)               │
    │  → Redshift reads ONLY revenue + category + date columns     │
    │  → MongoDB: scan 500GB of documents for 3 columns            │
    │  → Redshift: scan 15GB of 3 columnar files                   │
    │  → 33x more I/O in MongoDB for same analytical query         │
    │  → (Project 1 internals: document vs columnar storage)       │
    │                                                              │
    │  Problem 2: COMPLEX JOINS                                    │
    │  → "Revenue by customer segment by marketing channel"        │
    │  → Requires: orders + customers + marketing_events + segments│
    │  → MongoDB $lookup: nested loop join (slow at scale)          │
    │  → Redshift: hash join across distributed data (fast)         │
    │  → (Project 13: $lookup limitations)                         │
    │                                                              │
    │  Problem 3: WORKING SET PRESSURE                             │
    │  → Analytical queries scan cold data (historical)            │
    │  → Evicts hot operational data from WiredTiger cache          │
    │  → Cart operations slow down during report generation         │
    │  → (Project 33: working set management)                      │
    │                                                              │
    │  Problem 4: NO COLUMNAR COMPRESSION                          │
    │  → 3 years of order history: 2TB in MongoDB                  │
    │  → Same data in Redshift: ~200GB (10x compression)           │
    │  → Storage cost difference is massive                        │
    │                                                              │
    │  Problem 5: BI TOOL COMPATIBILITY                            │
    │  → Tableau, Looker, QuickSight speak SQL natively             │
    │  → MongoDB BI Connector exists but limited                    │
    │  → Analysts know SQL, not aggregation pipelines               │
    └────────────────────────────────────────────────────────────┘
    
    VERDICT: MongoDB alone CANNOT serve heavy analytical workloads efficiently
             It was never designed to. That's not a weakness — it's a focus.

🧩 HOW TO APPROACH IT IN MONGODB (+ REDSHIFT)

ARCHITECTURE DESIGN THINKING:

STEP 1: CLASSIFY EVERY DATA ENTITY
    For each entity, ask:
    → Is it READ/WRITTEN by the application in real-time? → MongoDB
    → Is it only QUERIED for analysis/reporting? → Redshift
    → Is it BOTH? → MongoDB (source of truth) → pipeline → Redshift (analytics copy)

    ┌────────────────────┬─────────────────┬──────────────────────┐
    │ ENTITY              │ OPERATIONAL?     │ ANALYTICAL?           │
    ├────────────────────┼─────────────────┼──────────────────────┤
    │ Shopping Cart       │ ✅ (real-time)   │ ⚠️ (funnel analysis)  │
    │ Product Catalog     │ ✅ (serve pages) │ ✅ (category reports)  │
    │ Orders              │ ✅ (processing)  │ ✅ (revenue analysis)  │
    │ User Profiles       │ ✅ (auth/prefs)  │ ✅ (cohort analysis)   │
    │ Inventory           │ ✅ (stock mgmt)  │ ✅ (forecasting)       │
    │ User Sessions       │ ✅ (tracking)    │ ✅ (behavior analysis) │
    │ Marketing Campaigns │ ❌               │ ✅ (attribution)       │
    │ Date Dimensions     │ ❌               │ ✅ (time analysis)     │
    └────────────────────┴─────────────────┴──────────────────────┘
    
    RESULT:
    → MongoDB: carts, products, orders, users, inventory, sessions
    → Redshift: ALL of the above (analytics copies) + marketing + dimensions
    → Pipeline: MongoDB → transform → Redshift (for dual-purpose entities)

STEP 2: DESIGN MONGODB DOCUMENTS (source of truth)
    → Optimized for APPLICATION access patterns (Projects 6, 14, 16-20)
    → Denormalized for fast reads
    → Indexed for query patterns (Projects 9-11)

STEP 3: DESIGN REDSHIFT STAR SCHEMA (analytics destination)
    → Optimized for ANALYTICAL query patterns
    → Normalized into facts and dimensions
    → Distribution/sort keys for JOIN and scan patterns

STEP 4: DESIGN THE PIPELINE
    → Change Streams capture every insert/update/delete (Project 25)
    → Kafka/Kinesis buffers and distributes events
    → Transformer flattens documents into relational rows
    → Loader writes to Redshift (COPY or streaming ingestion)

STEP 5: HANDLE THE IMPEDANCE MISMATCH
    → MongoDB document (denormalized, nested, arrays)
    → Must become multiple Redshift rows (normalized, flat, relational)
    → THIS is the hardest part of the pipeline

Part 2: Redshift Star Schema Design (Analytical Layer)

-- ═══════════════════════════════════════════════════════
-- REDSHIFT ANALYTICAL SCHEMA DESIGN
-- Optimized for: complex queries, JOINs, aggregation, BI tools
-- ═══════════════════════════════════════════════════════

-- ─── FACT TABLE: ORDER ITEMS (grain = one line item) ──
-- ONE MongoDB order document → MULTIPLE Redshift rows
-- This is the key transformation: denormalized → normalized

CREATE TABLE analytics.fact_order_items (
    order_item_id       BIGINT IDENTITY(1,1),
    order_id            VARCHAR(24) NOT NULL,      -- MongoDB ObjectId as string
    order_number        VARCHAR(50) NOT NULL,
    order_date          TIMESTAMP NOT NULL,
    order_status        VARCHAR(20),
    
    -- Flattened from items[] array
    sku                 VARCHAR(50) NOT NULL,
    product_name        VARCHAR(255),
    unit_price          DECIMAL(10,2),
    quantity            INTEGER,
    subtotal            DECIMAL(10,2),
    
    -- Foreign keys to dimensions
    customer_id         VARCHAR(24),               -- MongoDB ObjectId
    product_category    VARCHAR(100),
    shipping_state      VARCHAR(2),
    shipping_country    VARCHAR(2),
    payment_method      VARCHAR(50),
    source_channel      VARCHAR(20),
    campaign_id         VARCHAR(100),
    
    -- Dates for partitioning/sorting
    created_at          TIMESTAMP,
    updated_at          TIMESTAMP,
    
    -- Pipeline metadata
    _mongo_doc_id       VARCHAR(24),               -- traceability back to MongoDB
    _pipeline_loaded_at TIMESTAMP DEFAULT GETDATE()
)
DISTSTYLE KEY
DISTKEY(customer_id)                                -- co-locate customer data
COMPOUND SORTKEY(created_at, product_category);     -- optimize date + category queries


-- ─── DIMENSION: CUSTOMERS ─────────────────────────────
CREATE TABLE analytics.dim_customers (
    customer_id         VARCHAR(24) PRIMARY KEY,    -- MongoDB ObjectId
    name                VARCHAR(255),

def save_resume_token(collection, token):
    """Persist resume token to disk for crash recovery"""

# ─── BSON TO JSON SERIALIZER ──────────────────────────
# MongoDB documents contain BSON types (ObjectId, ISODate, Decimal128)
# Kafka needs JSON-serializable data
# This is the IMPEDANCE MISMATCH between MongoDB and the rest of the world

def serialize_for_kafka(document):
    """
    Convert MongoDB document to Kafka-friendly JSON
    
    Key transformations:
    - ObjectId → string
    - ISODate → ISO format string
    - Decimal128 → string (preserve precision)
    - Binary → base64 string
    """
    def convert(obj):
        if hasattr(obj, '__str__') and type(obj).__name__ == 'ObjectId':
            return str(obj)
        if isinstance(obj, datetime):
            return obj.isoformat()
        if type(obj).__name__ == 'Decimal128':
            return str(obj)
        if isinstance(obj, bytes):
            import base64
            return base64.b64encode(obj).decode('utf-8')
        if isinstance(obj, dict):
            return {k: convert(v) for k, v in obj.items()}
        if isinstance(obj, list):
            return [convert(item) for item in obj]
        return obj
    
    return convert(document)

def watch_collection(collection_name):
    """
    Watch a MongoDB collection for changes and publish to Kafka
    
    This is the CORE of the CDC pipeline:
    - MongoDB oplog (internal replication log) tracks ALL changes
    - Change Stream (Project 25) provides a clean API to read oplog
    - Each change event contains: operationType, fullDocument, updateDescription
    - We serialize and publish to Kafka for downstream consumers
    """
    
    client = MongoClient(MONGO_URI)
    db = client[DATABASE_NAME]
    collection = db[collection_name]
    topic = COLLECTION_TOPIC_MAP[collection_name]
    
    # Resume from last position (exactly-once semantics)
    resume_token = load_resume_token(collection_name)
    
    # Pipeline filter: only watch insert, update, replace, delete
    # (ignore internal MongoDB operations)
    pipeline = [
        {
            "$match": {
                "operationType": {
                    "$in": ["insert", "update", "replace", "delete"]
                }
            }
        }
    ]
    
    # Change stream options
    watch_options = {
        "pipeline": pipeline,
        "full_document": "updateLookup",    # include full doc on updates
        "full_document_before_change": "whenAvailable"  # include before-state (MongoDB 6.0+)
    }
    
    if resume_token:
        watch_options["resume_after"] = resume_token
        logger.info(f"Resuming {collection_name} from saved token")
    else:
        logger.info(f"Starting {collection_name} from current position")
    
    # ─── THE MAIN LOOP ────────────────────────────────
    try:
        with collection.watch(**watch_options) as stream:
            logger.info(f"Watching {collection_name} → {topic}")
            
            for change in stream:
                # Extract key information
                operation = change["operationType"]
                doc_id = str(change["documentKey"]["_id"])
                timestamp = change["clusterTime"].as_datetime()
                
                # Build Kafka message
                kafka_message = {
                    "metadata": {
                        "source": "mongodb",
                        "database": DATABASE_NAME,
                        "collection": collection_name,
                        "operation": operation,
                        "document_id": doc_id,
                        "timestamp": timestamp.isoformat(),
                        "pipeline_captured_at": datetime.utcnow().isoformat()
                    },
                    "payload": {}
                }
                
                # Add document content based on operation type
                if operation in ("insert", "replace", "update"):
                    full_doc = change.get("fullDocument")
                    if full_doc:
                        kafka_message["payload"]["after"] = serialize_for_kafka(full_doc)
                    
                    # Include before-image for updates (if available)
                    before_doc = change.get("fullDocumentBeforeChange")
                    if before_doc:
                        kafka_message["payload"]["before"] = serialize_for_kafka(before_doc)
                    
                    # Include specific update fields (for partial updates)
                    if operation == "update" and "updateDescription" in change:
                        kafka_message["payload"]["updated_fields"] = serialize_for_kafka(
                            change["updateDescription"].get("updatedFields", {})
                        )
                        kafka_message["payload"]["removed_fields"] = (
                            change["updateDescription"].get("removedFields", [])
                        )
                
                elif operation == "delete":
                    kafka_message["payload"]["deleted_id"] = doc_id
                    before_doc = change.get("fullDocumentBeforeChange")
                    if before_doc:
                        kafka_message["payload"]["before"] = serialize_for_kafka(before_doc)
                
                # ─── PUBLISH TO KAFKA ─────────────────
                producer.send(
                    topic=topic,
                    key=doc_id,          # partition by document ID (ordering guarantee)
                    value=kafka_message
                )
                
                # ─── SAVE RESUME TOKEN ────────────────
                # Save AFTER successful Kafka publish
                # This ensures at-least-once delivery
                save_resume_token(collection_name, change["_id"])
                
                logger.info(
                    f"{operation.upper()} on {collection_name}/{doc_id} → {topic}"
                )
    
    except Exception as e:
        logger.error(f"Error watching {collection_name}: {e}")
        raise

# ─── S3 WRITER (MICRO-BATCH) ─────────────────────────
# Accumulate transformed rows, then write as gzipped CSV to S3
# Redshift COPY performs best with multiple compressed files

class S3BatchWriter:
    """
    Accumulates transformed rows and writes to S3 in micro-batches
    Redshift COPY reads from S3 — this is the standard loading pattern
    
    Why S3 and not direct Redshift INSERT?
    → Redshift COPY from S3 is 10-100x faster than individual INSERTs
    → S3 provides durability buffer (if Redshift is temporarily unavailable)
    → Multiple files enable parallel COPY across Redshift slices
    → (Same reason you use COPY from S3 instead of INSERT in normal Redshift work)
    """
    
    def __init__(self, bucket, prefix="cdc-events"):
        self.bucket = bucket
        self.prefix = prefix
        self.buffers = defaultdict(list)   # table_name → [row_dicts]
        self.last_flush = datetime.utcnow()
    
    def add_rows(self, table_name, rows):
        """Add transformed rows to buffer"""
        self.buffers[table_name].extend(rows)
    
    def should_flush(self):
        """Check if we should write to S3"""
        total_rows = sum(len(rows) for rows in self.buffers.values())
        time_elapsed = (datetime.utcnow() - self.last_flush).total_seconds()
        
        return total_rows >= BATCH_SIZE or time_elapsed >= BATCH_TIMEOUT_SEC
    
    def flush(self):
        """Write all buffered rows to S3 as gzipped CSV files"""
        
        if not any(self.buffers.values()):
            return
        
        timestamp = datetime.utcnow().strftime("%Y%m%d_%H%M%S_%f")
        
        for table_name, rows in self.buffers.items():
            if not rows:
                continue
            
            # Convert to CSV
            output = io.StringIO()
            if rows:
                writer = csv.DictWriter(
                    output,
                    fieldnames=rows[0].keys(),
                    quoting=csv.QUOTE_MINIMAL
                )
                writer.writeheader()
                writer.writerows(rows)
            
            # Gzip compress
            csv_content = output.getvalue().encode("utf-8")
            compressed = gzip.compress(csv_content)
            
            # Write to S3
            # Path structure enables Redshift COPY with manifest
            s3_key = (
                f"{self.prefix}/"
                f"table={table_name}/"
                f"year={datetime.utcnow().year}/"
                f"month={datetime.utcnow().month:02d}/"
                f"day={datetime.utcnow().day:02d}/"
                f"{table_name}_{timestamp}.csv.gz"
            )
            
            s3.put_object(
                Bucket=self.bucket,
                Key=s3_key,
                Body=compressed,
                ContentType="text/csv",
                ContentEncoding="gzip"
            )
            
            logger.info(
                f"Flushed {len(rows)} rows → s3://{self.bucket}/{s3_key}"
            )
        
        # Reset buffers
        self.buffers = defaultdict(list)
        self.last_flush = datetime.utcnow()

# ─── MAIN CONSUMER LOOP ──────────────────────────────

def run_consumer():
    """
    Main processing loop:
    1. Consume change events from Kafka
    2. Transform MongoDB documents to Redshift rows
    3. Batch-write to S3
    4. Redshift COPY picks up from S3
    """
    
    consumer = KafkaConsumer(
        "shopstream.orders.changes",
        "shopstream.products.changes",
        "shopstream.customers.changes",
        bootstrap_servers=KAFKA_BROKERS,
        group_id="redshift-loader",
        auto_offset_reset="earliest",
        value_deserializer=lambda m: json.loads(m.decode("utf-8")),
        
        # Consumer tuning
        max_poll_records=500,
        session_timeout_ms=30000,
        enable_auto_commit=False          # manual commit after S3 write
    )
    
    writer = S3BatchWriter(bucket=S3_BUCKET)
    
    logger.info("Consumer started. Waiting for events...")
    
    try:
        while True:
            # Poll for messages (non-blocking with timeout)
            messages = consumer.poll(timeout_ms=1000)
            
            for topic_partition, records in messages.items():
                for record in records:
                    event = record.value
                    metadata = event.get("metadata", {})
                    payload = event.get("payload", {})
                    
                    collection = metadata.get("collection", "")
                    operation = metadata.get("operation", "")
                    
                    # Skip deletes for now (handle separately)
                    # In production: soft-delete in Redshift or DELETE + re-COPY
                    if operation == "delete":
                        logger.info(
                            f"DELETE on {collection}/{metadata.get('document_id')}"
                            f" — marking for Redshift soft-delete"
                        )
                        continue
                    
                    # Get the document (after-image)
                    document = payload.get("after", {})
                    
                    if not document:
                        continue
                    
                    # Transform using appropriate transformer
                    transformer = TRANSFORMERS.get(collection)
                    if transformer:
                        transformed = transformer(document)
                        
                        for table_name, rows in transformed.items():
                            writer.add_rows(table_name, rows)
                    else:
                        logger.warning(f"No transformer for collection: {collection}")
            
            # Flush to S3 if batch is ready
            if writer.should_flush():
                writer.flush()
                consumer.commit()      # commit Kafka offsets AFTER S3 write
                logger.info("Batch committed")
    
    except KeyboardInterrupt:
        logger.info("Shutting down consumer...")
        writer.flush()                 # flush remaining buffer
        consumer.commit()
        consumer.close()

if __name__ == "__main__":
    run_consumer()

# ─── S3 FILE DISCOVERY ───────────────────────────────

def discover_new_files(table_name, since_hours=1):
    """
    Find new S3 files that haven't been loaded yet
    
    Strategy: look at files created in the last N hours
    In production: use a manifest table or S3 event notifications
    
    S3 path convention (set in Component 2):
    s3://bucket/cdc-events/table=fact_orders/year=2024/month=05/day=31/file.csv.gz
    """
    s3_client = boto3.client("s3")
    prefix = f"{S3_PREFIX}/table={table_name}/"
    
    # List objects in the table's S3 prefix
    paginator = s3_client.get_paginator("list_objects_v2")
    files = []
    
    cutoff = datetime.utcnow() - timedelta(hours=since_hours)
    
    for page in paginator.paginate(Bucket=S3_BUCKET, Prefix=prefix):
        for obj in page.get("Contents", []):
            # Only include files modified after cutoff
            if obj["LastModified"].replace(tzinfo=None) > cutoff:
                files.append(obj["Key"])
    
    return files


def create_manifest(files):
    """
    Create a Redshift COPY manifest file
    
    Manifest = JSON file listing exactly which S3 files to load
    WHY: prevents double-loading, enables selective loading
    
    MongoDB equivalent concept: resume token (Project 25)
    Both solve the same problem: "where did I leave off?"
    """
    manifest = {
        "entries": [
            {
                "url": f"s3://{S3_BUCKET}/{f}",
                "mandatory": True
            }
            for f in files
        ]
    }
    
    manifest_key = f"{S3_PREFIX}/manifests/{datetime.utcnow().strftime('%Y%m%d_%H%M%S')}.manifest"
    
    s3_client = boto3.client("s3")
    s3_client.put_object(
        Bucket=S3_BUCKET,
        Key=manifest_key,
        Body=json.dumps(manifest).encode("utf-8")
    )
    
    return f"s3://{S3_BUCKET}/{manifest_key}"

# ─── LOADING STRATEGIES ──────────────────────────────

def load_append(conn, table_config, manifest_url):
    """
    APPEND strategy: simply COPY new rows into the table
    Used for: fact_order_items (immutable events)
    
    This is the simplest pattern
    Like insertMany() in MongoDB — just add data
    """
    redshift_table = table_config["redshift_table"]
    copy_options = table_config["copy_options"]
    
    copy_sql = f"""
        COPY {redshift_table}
        FROM '{manifest_url}'
        IAM_ROLE '{IAM_ROLE}'
        MANIFEST
        {copy_options}
    ;"""
    
    with conn.cursor() as cur:
        logger.info(f"COPY APPEND → {redshift_table}")
        cur.execute(copy_sql)
        
        # Check how many rows were loaded
        cur.execute(f"SELECT pg_last_copy_count();")
        row_count = cur.fetchone()[0]
        logger.info(f"  Loaded {row_count} rows into {redshift_table}")
    
    conn.commit()
    return row_count


def load_upsert(conn, table_config, manifest_url):
    """
    UPSERT strategy: stage → delete matching → insert new
    Used for: fact_orders, dim_customers, dim_products
    
    WHY UPSERT:
    → MongoDB documents get UPDATED (order status changes, customer edits profile)
    → Change Stream captures every update as a new event
    → Redshift needs to REPLACE the old row with the new version
    → Redshift has no native UPSERT/MERGE (unlike PostgreSQL)
    → Standard pattern: staging table → DELETE + INSERT in transaction
    
    MongoDB equivalent: updateOne with upsert:true (Project 4)
    But Redshift requires this multi-step dance
    """
    redshift_table = table_config["redshift_table"]
    merge_key = table_config["merge_key"]
    copy_options = table_config["copy_options"]
    
    # Staging table name
    staging_table = f"staging.stg_{redshift_table.split('.')[-1]}"
    
    with conn.cursor() as cur:
        # Step 1: Create staging table (temporary clone of target)
        logger.info(f"UPSERT Step 1: Create staging table {staging_table}")
        cur.execute(f"DROP TABLE IF EXISTS {staging_table};")
        cur.execute(f"CREATE TABLE {staging_table} (LIKE {redshift_table});")
        
        # Step 2: COPY new data into staging
        logger.info(f"UPSERT Step 2: COPY into staging")
        copy_sql = f"""
            COPY {staging_table}
            FROM '{manifest_url}'
            IAM_ROLE '{IAM_ROLE}'
            MANIFEST
            {copy_options}
        ;"""
        cur.execute(copy_sql)
        
        cur.execute(f"SELECT COUNT(*) FROM {staging_table};")
        staged_count = cur.fetchone()[0]
        logger.info(f"  Staged {staged_count} rows")
        
        # Step 3: Deduplicate staging (keep latest per merge key)
        # MongoDB might send multiple updates for the same document
        # We only want the LATEST version in Redshift
        logger.info(f"UPSERT Step 3: Deduplicate staging")
        cur.execute(f"""
            DELETE FROM {staging_table}
            WHERE {merge_key} IN (
                SELECT {merge_key}
                FROM {staging_table}
                GROUP BY {merge_key}
                HAVING COUNT(*) > 1
            )
            AND ctid NOT IN (
                SELECT MAX(ctid)
                FROM {staging_table}
                GROUP BY {merge_key}
            );
        """)
        
        # Step 4: BEGIN TRANSACTION — Delete old + Insert new (atomic)
        # This ensures analysts never see partial state
        logger.info(f"UPSERT Step 4: Atomic DELETE + INSERT")
        cur.execute("BEGIN TRANSACTION;")
        
        # Delete rows in target that exist in staging (will be replaced)
        cur.execute(f"""
            DELETE FROM {redshift_table}
            WHERE {merge_key} IN (
                SELECT {merge_key} FROM {staging_table}
            );
        """)
        
        # Insert all rows from staging into target
        cur.execute(f"""
            INSERT INTO {redshift_table}
            SELECT * FROM {staging_table};
        """)
        
        cur.execute("END TRANSACTION;")
        
        # Step 5: Cleanup staging
        cur.execute(f"DROP TABLE IF EXISTS {staging_table};")
        
        logger.info(f"  Upserted {staged_count} rows into {redshift_table}")
    
    conn.commit()
    return staged_count

# ─── LOAD TRACKING TABLE ─────────────────────────────
# Track what has been loaded to prevent double-loading
# This is the Redshift equivalent of MongoDB's resume token

def ensure_tracking_table(conn):
    """Create load tracking table if it doesn't exist"""
    with conn.cursor() as cur:
        cur.execute("""
            CREATE TABLE IF NOT EXISTS staging.pipeline_load_log (
                load_id         BIGINT IDENTITY(1,1),
                table_name      VARCHAR(100),
                manifest_url    VARCHAR(500),
                rows_loaded     INTEGER,
                load_status     VARCHAR(20),      -- success, failed, in_progress
                started_at      TIMESTAMP DEFAULT GETDATE(),
                completed_at    TIMESTAMP,
                error_message   VARCHAR(1000)
            )
            DISTSTYLE AUTO
            SORTKEY(started_at);
        """)
    conn.commit()


def log_load(conn, table_name, manifest_url, rows_loaded, status, error=None):
    """Record load event for auditability and idempotency"""
    with conn.cursor() as cur:
        cur.execute("""
            INSERT INTO staging.pipeline_load_log
            (table_name, manifest_url, rows_loaded, load_status, completed_at, error_message)
            VALUES (%s, %s, %s, %s, GETDATE(), %s);
        """, (table_name, manifest_url, rows_loaded, status, error))
    conn.commit()


def already_loaded(conn, manifest_url):
    """Check if this manifest was already loaded successfully"""
    with conn.cursor() as cur:
        cur.execute("""
            SELECT COUNT(*) FROM staging.pipeline_load_log
            WHERE manifest_url = %s AND load_status = 'success';
        """, (manifest_url,))
        return cur.fetchone()[0] > 0

# ─── MAIN LOADER ORCHESTRATOR ─────────────────────────

def run_loader():
    """
    Main loader function — runs on schedule (cron, Airflow, Step Functions)
    
    Flow:
    1. For each table: discover new S3 files
    2. Create manifest
    3. Load using appropriate strategy (append or upsert)
    4. Log the load
    5. Run post-load maintenance (ANALYZE)
    """
    
    conn = get_redshift_connection()
    ensure_tracking_table(conn)
    
    total_loaded = 0
    
    for table_name, config in TABLE_CONFIGS.items():
        logger.info(f"\n{'='*60}")
        logger.info(f"Processing table: {table_name}")
        logger.info(f"{'='*60}")
        
        try:
            # Step 1: Discover new files
            files = discover_new_files(table_name, since_hours=2)
            
            if not files:
                logger.info(f"  No new files for {table_name}")
                continue
            
            logger.info(f"  Found {len(files)} new files")
            
            # Step 2: Create manifest
            manifest_url = create_manifest(files)
            
            # Step 3: Check idempotency
            if already_loaded(conn, manifest_url):
                logger.info(f"  Manifest already loaded — skipping")
                continue
            
            # Step 4: Load based on strategy
            strategy = config["merge_strategy"]
            
            if strategy == "append":
                rows = load_append(conn, config, manifest_url)
            elif strategy == "upsert":
                rows = load_upsert(conn, config, manifest_url)
            else:
                raise ValueError(f"Unknown strategy: {strategy}")
            
            # Step 5: Log success
            log_load(conn, table_name, manifest_url, rows, "success")
            total_loaded += rows
            
        except Exception as e:
            logger.error(f"  FAILED loading {table_name}: {e}")
            log_load(conn, table_name, manifest_url, 0, "failed", str(e))
            conn.rollback()
            continue
    
    # ─── POST-LOAD MAINTENANCE ────────────────────────
    # Redshift needs ANALYZE after significant loads
    # Updates table statistics for query planner
    # MongoDB equivalent: none needed (WiredTiger auto-manages stats)
    
    if total_loaded > 0:
        logger.info(f"\n{'='*60}")
        logger.info(f"Running post-load ANALYZE")
        logger.info(f"{'='*60}")
        
        with conn.cursor() as cur:
            for table_name, config in TABLE_CONFIGS.items():
                redshift_table = config["redshift_table"]
                logger.info(f"  ANALYZE {redshift_table}")
                cur.execute(f"ANALYZE {redshift_table};")
        conn.commit()
    
    logger.info(f"\nLoader complete. Total rows loaded: {total_loaded}")
    conn.close()


# ─── ENTRY POINT ──────────────────────────────────────
if __name__ == "__main__":
    run_loader()

def reconcile_counts():
    """
    Compare document/row counts between MongoDB and Redshift
    
    IMPORTANT: exact match is NOT expected because:
    → Pipeline has latency (minutes behind)
    → MongoDB has newer data not yet in Redshift
    → We check that Redshift count is CLOSE to MongoDB count
    → And that the gap is within expected pipeline latency
    """
    
    mongo_db = get_mongo()
    rs_conn = get_redshift()
    rs_cur = rs_conn.cursor()
    
    # Allow for pipeline lag (data up to 10 minutes old may not be in Redshift yet)
    lag_cutoff = datetime.utcnow() - timedelta(minutes=10)
    
    checks = [
        {
            "name": "Orders",
            "mongo_collection": "orders",
            "mongo_filter": {"created_at": {"$lte": lag_cutoff}},
            "redshift_query": f"""
                SELECT COUNT(DISTINCT order_id)
                FROM analytics.fact_orders
                WHERE created_at <= '{lag_cutoff.isoformat()}'
            """
        },
        {
            "name": "Customers",
            "mongo_collection": "customers",
            "mongo_filter": {"created_at": {"$lte": lag_cutoff}},
            "redshift_query": f"""
                SELECT COUNT(*)
                FROM analytics.dim_customers
            """
        }
    ]
    
    logger.info(f"{'='*60}")
    logger.info(f"PIPELINE RECONCILIATION REPORT")
    logger.info(f"Cutoff: {lag_cutoff.isoformat()} (10min lag allowed)")
    logger.info(f"{'='*60}")
    
    all_passed = True
    
    for check in checks:
        # MongoDB count
        mongo_count = mongo_db[check["mongo_collection"]].count_documents(
            check["mongo_filter"]
        )
        
        # Redshift count
        rs_cur.execute(check["redshift_query"])
        redshift_count = rs_cur.fetchone()[0]
        
        # Calculate drift
        if mongo_count > 0:
            drift_pct = abs(mongo_count - redshift_count) / mongo_count * 100
        else:
            drift_pct = 0
        
        status = "✅ PASS" if drift_pct < 5 else "❌ FAIL"
        if drift_pct >= 5:
            all_passed = False
        
        logger.info(f"\n  {check['name']}:")
        logger.info(f"    MongoDB:   {mongo_count:>10,}")
        logger.info(f"    Redshift:  {redshift_count:>10,}")
        logger.info(f"    Drift:     {drift_pct:>9.2f}%")
        logger.info(f"    Status:    {status}")
    
    rs_cur.close()
    rs_conn.close()
    
    return all_passed

# ─── VALUE RECONCILIATION ─────────────────────────────

def reconcile_values():
    """
    Compare actual aggregated values between MongoDB and Redshift
    
    This catches transformation bugs:
    → MongoDB total revenue (via aggregation pipeline)
    → vs Redshift total revenue (via SQL)
    → Should match within rounding tolerance
    """
    
    mongo_db = get_mongo()
    rs_conn = get_redshift()
    rs_cur = rs_conn.cursor()
    
    # Pick a completed date range (both systems should agree)
    yesterday = (datetime.utcnow() - timedelta(days=1)).replace(
        hour=0, minute=0, second=0, microsecond=0
    )
    day_before = yesterday - timedelta(days=1)
    
    # ─── MONGODB: Revenue for yesterday ───────────────
    mongo_pipeline = [
        {
            "$match": {
                "created_at": {
                    "$gte": day_before,
                    "$lt": yesterday
                },
                "status": {"$nin": ["cancelled", "refunded"]}
            }
        },
        {
            "$group": {
                "_id": None,
                "total_revenue": {"$sum": "$payment.total"},
                "order_count": {"$sum": 1},
                "total_items": {"$sum": "$item_count"}
            }
        }
    ]
    
    mongo_result = list(mongo_db.orders.aggregate(mongo_pipeline))
    mongo_revenue = mongo_result[0]["total_revenue"] if mongo_result else 0
    mongo_orders = mongo_result[0]["order_count"] if mongo_result else 0
    
    # ─── REDSHIFT: Revenue for yesterday ──────────────
    rs_cur.execute(f"""
        SELECT 
            SUM(total) AS total_revenue,
            COUNT(*) AS order_count
        FROM analytics.fact_orders
        WHERE created_at >= '{day_before.isoformat()}'
          AND created_at < '{yesterday.isoformat()}'
          AND status NOT IN ('cancelled', 'refunded')
    """)
    rs_row = rs_cur.fetchone()
    redshift_revenue = float(rs_row[0]) if rs_row[0] else 0
    redshift_orders = rs_row[1] if rs_row[1] else 0
    
    # ─── COMPARE ──────────────────────────────────────
    revenue_diff = abs(mongo_revenue - redshift_revenue)
    revenue_tolerance = 0.01  # $0.01 tolerance (rounding)
    
    logger.info(f"\n  Revenue Reconciliation ({day_before.date()}):")
    logger.info(f"    MongoDB Revenue:   ${mongo_revenue:>12,.2f}  ({mongo_orders} orders)")
    logger.info(f"    Redshift Revenue:  ${redshift_revenue:>12,.2f}  ({redshift_orders} orders)")
    logger.info(f"    Difference:        ${revenue_diff:>12,.2f}")
    logger.info(f"    Status:            {'✅ MATCH' if revenue_diff < revenue_tolerance else '❌ MISMATCH'}")
    
    rs_cur.close()
    rs_conn.close()
    
    return revenue_diff < revenue_tolerance

def check_pipeline_lag():
    """
    Measure how far behind Redshift is from MongoDB
    
    Method: find the most recent record in each system
    Difference = pipeline lag
    
    Target: <5 minutes for near-real-time
    Acceptable: <1 hour for hourly batch
    Alert: >1 hour
    """
    
    mongo_db = get_mongo()
    rs_conn = get_redshift()
    rs_cur = rs_conn.cursor()
    
    # Latest order in MongoDB
    latest_mongo = mongo_db.orders.find_one(
        sort=[("created_at", -1)],
        projection={"created_at": 1}
    )
    mongo_latest_ts = latest_mongo["created_at"] if latest_mongo else None
    
    # Latest order in Redshift
    rs_cur.execute("""
        SELECT MAX(created_at) FROM analytics.fact_orders;
    """)
    redshift_latest_ts = rs_cur.fetchone()[0]
    
    if mongo_latest_ts and redshift_latest_ts:
        lag = mongo_latest_ts - redshift_latest_ts
        lag_minutes = lag.total_seconds() / 60
        
        if lag_minutes < 5:
            status = "✅ REAL-TIME"
        elif lag_minutes < 60:
            status = "⚠️ ACCEPTABLE"
        else:
            status = "❌ ALERT: HIGH LAG"
        
        logger.info(f"\n  Pipeline Lag:")
        logger.info(f"    MongoDB latest:   {mongo_latest_ts}")
        logger.info(f"    Redshift latest:  {redshift_latest_ts}")
        logger.info(f"    Lag:              {lag_minutes:.1f} minutes")
        logger.info(f"    Status:           {status}")
    
    rs_cur.close()
    rs_conn.close()

-- 1. Revenue by category by month (classic BI dashboard)
-- Scans millions of rows across columnar storage
-- Only touches 3 columns out of 20+ (columnar advantage)
SELECT 
    dp.category,
    DATE_TRUNC('month', foi.created_at) AS month,
    SUM(foi.subtotal)                   AS revenue,
    COUNT(DISTINCT foi.order_id)        AS orders,
    SUM(foi.quantity)                   AS units_sold,
    AVG(foi.unit_price)                 AS avg_price
FROM analytics.fact_order_items foi
JOIN analytics.dim_products dp ON foi.sku = dp.sku
WHERE foi.created_at >= DATEADD(year, -1, GETDATE())
GROUP BY 1, 2
ORDER BY 1, 2;
-- Redshift: ~3 seconds across 100M rows ✅
-- MongoDB: ~60+ seconds (scan all documents, $lookup) ❌

-- 2. Customer cohort retention analysis
-- Classic analytical query impossible in MongoDB efficiently
SELECT
    DATE_TRUNC('month', dc.signup_date)                          AS cohort,
    DATEDIFF(month, DATE_TRUNC('month', dc.signup_date),
             DATE_TRUNC('month', fo.order_date))                 AS months_since,
    COUNT(DISTINCT fo.customer_id)                               AS active,
    COUNT(DISTINCT fo.customer_id)::FLOAT /
        FIRST_VALUE(COUNT(DISTINCT fo.customer_id)) 
        OVER (PARTITION BY DATE_TRUNC('month', dc.signup_date) 
              ORDER BY DATEDIFF(month, DATE_TRUNC('month', dc.signup_date),
                                DATE_TRUNC('month', fo.order_date)))  AS retention_rate
FROM analytics.fact_orders fo
JOIN analytics.dim_customers dc ON fo.customer_id = dc.customer_id
WHERE dc.signup_date >= '2023-01-01'
GROUP BY 1, 2
ORDER BY 1, 2;
-- Redshift: ~5 seconds ✅ (hash join + window function on MPP)
-- MongoDB: PRACTICALLY IMPOSSIBLE efficiently ❌❌

-- 3. Marketing attribution: which campaigns drive the most revenue
SELECT
    fo.campaign_id,
    fo.source_channel,
    COUNT(*)                    AS orders,
    SUM(fo.total)              AS total_revenue,
    AVG(fo.total)              AS avg_order_value,
    COUNT(DISTINCT fo.customer_id) AS unique_customers,
    SUM(fo.total) / COUNT(DISTINCT fo.customer_id) AS revenue_per_customer
FROM analytics.fact_orders fo
WHERE fo.created_at >= DATEADD(month, -3, GETDATE())
  AND fo.campaign_id IS NOT NULL
GROUP BY 1, 2
ORDER BY total_revenue DESC
LIMIT 20;
-- Redshift: ~2 seconds ✅
-- MongoDB: ~30 seconds (no columnar optimization) ⚠️

-- 4. Funnel analysis: browse → cart → checkout → purchase
-- Requires joining multiple event types across time
WITH funnel AS (
    SELECT
        DATE_TRUNC('week', event_date) AS week,
        COUNT(DISTINCT CASE WHEN event_type = 'page_view' 
              THEN user_id END)         AS browsers,
        COUNT(DISTINCT CASE WHEN event_type = 'add_to_cart' 
              THEN user_id END)         AS cart_adders,
        COUNT(DISTINCT CASE WHEN event_type = 'checkout_start' 
              THEN user_id END)         AS checkout_starters,
        COUNT(DISTINCT CASE WHEN event_type = 'purchase' 
              THEN user_id END)         AS purchasers
    FROM analytics.fact_events
    WHERE event_date >= DATEADD(month, -3, GETDATE())
    GROUP BY 1
)
SELECT
    week,
    browsers,
    cart_adders,
    checkout_starters,
    purchasers,
    ROUND(cart_adders::FLOAT / NULLIF(browsers, 0) * 100, 1)           AS browse_to_cart_pct,
    ROUND(checkout_starters::FLOAT / NULLIF(cart_adders, 0) * 100, 1)  AS cart_to_checkout_pct,
    ROUND(purchasers::FLOAT / NULLIF(checkout_starters, 0) * 100, 1)   AS checkout_to_purchase_pct
FROM funnel
ORDER BY week;
-- Redshift: ~4 seconds (conditional aggregation on sorted data) ✅
-- MongoDB: Would require multiple aggregation passes or $facet ❌

-- 5. Inventory forecasting: predict stockouts
SELECT
    dp.sku,
    dp.product_name,
    dp.category,
    SUM(CASE WHEN foi.created_at >= DATEADD(day, -7, GETDATE())
             THEN foi.quantity ELSE 0 END)   AS units_sold_7d,
    SUM(CASE WHEN foi.created_at >= DATEADD(day, -30, GETDATE())
             THEN foi.quantity ELSE 0 END)   AS units_sold_30d,
    ROUND(SUM(CASE WHEN foi.created_at >= DATEADD(day, -30, GETDATE())
                   THEN foi.quantity ELSE 0 END) / 30.0, 1) AS daily_run_rate,
    -- Would need inventory snapshot join for days_of_stock_remaining
    ROUND(SUM(CASE WHEN foi.created_at >= DATEADD(day, -30, GETDATE())
                   THEN foi.quantity ELSE 0 END) / 30.0, 1) AS daily_velocity
FROM analytics.fact_order_items foi
JOIN analytics.dim_products dp ON foi.sku = dp.sku
WHERE foi.created_at >= DATEADD(day, -30, GETDATE())
GROUP BY 1, 2, 3
HAVING SUM(CASE WHEN foi.created_at >= DATEADD(day, -7, GETDATE())
                THEN foi.quantity ELSE 0 END) > 0
ORDER BY daily_run_rate DESC;
-- Redshift: ~3 seconds (columnar scan + conditional aggregation) ✅
-- MongoDB: ~45 seconds (document scan + $group + $project) ❌

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT REPLICATE MONGODB DOCUMENT STRUCTURE IN REDSHIFT
    MongoDB:
    {
        customer: { name: "Sarah", segment: "premium" },
        items: [ { sku: "X", qty: 2 }, { sku: "Y", qty: 1 } ]
    }
    
    WRONG in Redshift:
    CREATE TABLE orders (
        order_id VARCHAR,
        customer_json SUPER,     ← storing nested JSON in Redshift
        items_json SUPER          ← defeats the purpose of columnar
    );
    
    RIGHT in Redshift:
    → Normalize into fact_orders + fact_order_items + dim_customers
    → Flatten nested objects into columns
    → Explode arrays into rows
    → THIS is why the transformer (Component 2) exists

❌ DO NOT QUERY MONGODB FOR ANALYTICAL WORKLOADS
    WRONG: Running nightly reports against MongoDB production cluster
    → Evicts operational data from WiredTiger cache (Project 33)
    → Slows down application for end users
    → Full collection scans compete with point lookups
    
    RIGHT: All analytical queries go to Redshift

❌ DO NOT USE REDSHIFT FOR REAL-TIME OPERATIONAL QUERIES
    WRONG: Application API reads from Redshift for product pages
    → Latency too high (50-200ms vs 1-5ms)
    → Concurrency too low (WLM limits vs MongoDB thousands)
    → Cost too high (Redshift compute for simple lookups)
    
    RIGHT: All operational queries go to MongoDB

❌ DO NOT SKIP THE TRANSFORMATION STEP
    WRONG: mongoexport → CSV → Redshift COPY (dump raw documents)
    → Nested objects become JSON strings in VARCHAR columns
    → Arrays not exploded into rows
    → No dimensional modeling
    → BI tools can't work with nested data
    
    RIGHT: Proper transformer that normalizes documents into star schema

❌ DO NOT BUILD BIDIRECTIONAL SYNC
    WRONG: MongoDB ↔ Redshift (data flowing both directions)
    → Creates circular dependencies
    → Conflict resolution nightmare
    → Which system is source of truth?
    
    RIGHT: Unidirectional: MongoDB → pipeline → Redshift (always)
    → If analysts need to push results back, use a SEPARATE pathway
      (Redshift → ML model → MongoDB, NOT Redshift → MongoDB direct sync)

❌ DO NOT USE THE SAME MONGODB INDEXES FOR PIPELINE AND APPLICATION
    WRONG: Creating indexes on MongoDB to optimize change stream filtering
           that hurt application write performance
    
    RIGHT: Change streams use the oplog (no special indexes needed)
           Keep MongoDB indexes optimized for APPLICATION queries only

📍 WHEN TO USE THIS HYBRID ARCHITECTURE

USE MONGODB + REDSHIFT TOGETHER WHEN:
    ✅ You have BOTH operational AND analytical workloads
    ✅ Application needs <10ms response time for reads/writes
    ✅ Analysts need complex JOINs across dimensions
    ✅ Data volume exceeds what one system handles well for both
    ✅ Schema evolves frequently on the operational side
    ✅ BI tools (Tableau, QuickSight) need SQL access
    ✅ You need real-time operational + batch/near-real-time analytical

USE MONGODB ALONE WHEN:
    → Analytics are simple (aggregation pipeline suffices)
    → Data volume is small (<100GB)
    → No BI tool requirement
    → Team is small and can't maintain pipeline

USE REDSHIFT ALONE WHEN:
    → No real-time operational workload
    → All data arrives via batch ETL
    → All access patterns are analytical
    → Application doesn't interact with the warehouse directly

STEP-BY-STEP: What happens when a user places an order

SECOND 0.000 — USER CLICKS "PLACE ORDER"
    │
    ▼
SECOND 0.001 — APPLICATION SERVER
    → Validates order data
    → Calls MongoDB driver (pymongo)
    
SECOND 0.002 — MONGODB: insertOne()
    → WiredTiger receives document
    → Write to journal (WAL) — crash safety (Project 1 internals)
    → Document written to collection's B-tree pages
    → _id index updated
    → Document stored as BSON in WiredTiger page
    → Write concern: w:"majority" — waits for 2/3 replica acknowledgment
    │
    ▼
SECOND 0.005 — APPLICATION RESPONDS TO USER
    → "Order confirmed!" — user sees confirmation
    → Total latency: 5ms
    → User experience: instant ✅
    │
    ▼
SECOND 0.005 — MONGODB OPLOG (simultaneously)
    → Every write is recorded in the oplog (operations log)
    → oplog is a capped collection on each replica set member
    → Entry: { op: "i", ns: "shopstream.orders", o: {full document} }
    → Oplog is the foundation of:
      a) Replication (secondaries read oplog to stay in sync)
      b) Change Streams (our pipeline reads oplog for CDC)
    │
    ▼
SECOND 0.010 — CHANGE STREAM LISTENER (Component 1)
    → Tailing the oplog via change stream API
    → Receives: { operationType: "insert", fullDocument: {...} }
    → Serializes document to JSON (BSON → JSON conversion)
    → Publishes to Kafka topic: shopstream.orders.changes
    │
    ▼
SECOND 0.050 — KAFKA
    → Message arrives in shopstream.orders.changes topic
    → Partitioned by order_id (key) — ordering guaranteed per order
    → Replicated across Kafka brokers (durability)
    → Retained for 7 days (replay window)
    │
    ▼
SECOND 0.100 — TRANSFORMER/CONSUMER (Component 2)
    → Consumes message from Kafka
    → Extracts fullDocument from change event
    → TRANSFORMATION (the critical step):
      
      MongoDB document (1 document):
      { _id, customer: {...}, items: [{...}, {...}], payment: {...} }
                │
                ▼
      Redshift rows (multiple rows across multiple tables):
      fact_orders:      1 row  (order header)
      fact_order_items: 2 rows (one per line item)
      dim_customers:    1 row  (customer snapshot)
    
    → Adds to micro-batch buffer
    │
    ▼
SECOND 1.000 — S3 WRITE (micro-batch fills or timeout triggers)
    → Batch of ~1000 transformed rows written as gzipped CSV
    → S3 path: s3://bucket/cdc-events/table=fact_orders/2024/05/31/file.csv.gz
    → Multiple files per table (enables parallel COPY)
    │
    ▼
SECOND 60-300 — REDSHIFT COPY LOADER (Component 3, scheduled)
    → Discovers new S3 files via listing
    → Creates manifest pointing to new files
    → For fact_order_items: COPY APPEND (new rows)
    → For fact_orders: UPSERT (stage → delete old → insert new)
    → For dim_customers: UPSERT (update existing or insert new)
    │
    ▼
SECOND 300 — REDSHIFT: COPY EXECUTES
    → Leader node reads manifest
    → Distributes files to compute slices (parallel)
    → Each slice reads its portion of CSV
    → Data converted to columnar format:
      order_id values → one column block (compressed)
      customer_id values → one column block (compressed)
      total values → one column block (compressed)
    → Zone maps updated per block (min/max values)
    → ANALYZE updates table statistics
    │
    ▼
SECOND 300+ — REDSHIFT: DATA AVAILABLE FOR ANALYTICS
    → Analyst runs: SELECT category, SUM(revenue) ...
    → Query planner uses zone maps to skip irrelevant blocks
    → MPP engine distributes work across slices
    → Columnar scan reads only needed columns
    → Hash joins across dimension tables
    → Result returned in 2-5 seconds
    
TOTAL PIPELINE LATENCY: ~1-5 minutes (near-real-time)
    → MongoDB operational response: 5ms (user-facing)
    → Redshift analytical availability: 1-5 minutes later
    → Both systems working at peak efficiency for their workload

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "JUST DUMP MONGODB TO REDSHIFT WITH MONGOEXPORT"
    ❌ mongoexport → CSV → S3 → COPY
    WHY IT FAILS:
    → mongoexport flattens only top-level fields
    → Nested objects become JSON strings in VARCHAR columns
    → Arrays not exploded into rows
    → No incremental capture — full dump every time
    → No handling of updates/deletes
    ✅ FIX: Use change streams + proper transformer (Components 1-2)

PITFALL 2: "STORING MONGODB ObjectId AS BIGINT IN REDSHIFT"
    ❌ Converting ObjectId to integer (losing information)
    WHY IT FAILS:
    → ObjectId is 12 bytes — doesn't fit in BIGINT (8 bytes)
    → Losing the timestamp embedded in ObjectId
    ✅ FIX: Store as VARCHAR(24) in Redshift
    → Still joinable, still indexable, preserves traceability

PITFALL 3: "RUNNING ANALYTICS ON MONGODB READ REPLICAS"
    ❌ "We'll use secondaryPreferred reads for reports"
    WHY IT FAILS:
    → Still document storage — still reads ALL fields for each doc
    → Competes with replication traffic
    → Stale data (replication lag)
    → No columnar compression benefit
    → Slower than Redshift by 10-30x for analytical queries
    ✅ FIX: Send data to Redshift for analytics (this project)

PITFALL 4: "REAL-TIME SYNC = INSERT ONE ROW AT A TIME INTO REDSHIFT"
    ❌ For each MongoDB change: INSERT INTO redshift_table VALUES (...)
    WHY IT FAILS:
    → Redshift single-row INSERT: 50-200ms each
    → At 50,000 orders/day: constant INSERT pressure
    → No columnar compression benefit on tiny inserts
    → VACUUM needed constantly
    → WLM queue contention
    ✅ FIX: Micro-batch to S3 → COPY (Component 2-3)
    → COPY 1000 rows at once: ~500ms total
    → vs INSERT 1000 rows one at a time: ~100 seconds total
    → 200x performance improvement

PITFALL 5: "NOT HANDLING SCHEMA EVOLUTION"
    ❌ MongoDB adds a new field → transformer crashes
    WHY IT FAILS:
    → MongoDB documents evolve over time (new fields, changed types)
    → Rigid transformer code breaks on unexpected fields
    → Redshift table doesn't have the new column
    ✅ FIX:
    → Transformer uses .get("field", default) (shown in Component 2)
    → Monitor for new fields → ALTER TABLE ADD COLUMN in Redshift
    → Or use Redshift SUPER type for truly dynamic fields

PITFALL 6: "NOT HANDLING MONGODB DOCUMENT UPDATES"
    ❌ Pipeline only captures inserts, ignores updates
    WHY IT FAILS:
    → Order status changes (pending → confirmed → shipped → delivered)
    → Redshift fact_orders row stays as "pending" forever
    → Dashboards show incorrect order status distribution
    ✅ FIX: 
    → Change stream with full_document: "updateLookup" (Component 1)
    → Upsert loading strategy in Redshift (Component 3)
    → Every update replaces the old Redshift row with current state

PITFALL 7: "NOT PLANNING FOR PIPELINE FAILURES"
    ❌ No retry logic, no dead-letter queue, no monitoring
    WHAT HAPPENS:
    → Kafka consumer crashes → events pile up in Kafka
    → S3 writer fails → transformed data lost
    → Redshift COPY fails → data gap in analytics
    ✅ FIX:
    → Resume tokens (Component 1) → restart from last position
    → Kafka retention (7 days) → replay window
    → Idempotent loading (Component 3) → safe to retry
    → Monitoring (Component 4) → detect problems early

TRANSFORMATION RULES:
    1. Top-level fields     → fact_orders columns (1:1 mapping)
    2. items[] array        → fact_order_items rows (1:N explosion)
    3. customer{} object    → dim_customers row (extract + upsert)
    4. shipping.address{}   → flat columns in fact_orders (dot notation → underscore)
    5. payment{} object     → flat columns in fact_orders
    6. Computed fields      → derived columns or pre-computed in MongoDB

Why The Transformation Direction Matters

CRITICAL UNDERSTANDING:

MONGODB → REDSHIFT (denormalized → normalized):
    → ONE document becomes MANY rows across MANY tables
    → Arrays EXPLODE into rows (items[] → fact_order_items)
    → Nested objects FLATTEN into columns (customer.name → customer_name)
    → Embedded snapshots EXTRACT into dimensions (customer → dim_customers)
    → Redundant data DEDUPLICATED (customer appears once in dim, not in every order)
    
    THIS DIRECTION IS NATURAL:
    → Denormalized data CONTAINS all the information
    → You're just REORGANIZING it for analytical efficiency
    → No data loss — just structural change

REDSHIFT → MONGODB (normalized → denormalized) — Project 35:
    → MANY rows across MANY tables become ONE document
    → JOINed rows NEST into objects and arrays
    → Dimension lookups EMBED into documents
    → This direction requires JOINs BEFORE inserting
    → More complex — we'll cover in Project 35

THE MENTAL MODEL:
    Think of MongoDB documents as COMPRESSED relational data:
    → All the JOINs are PRE-COMPUTED and stored together
    → The pipeline DECOMPRESSES them back into relational form
    → Redshift then applies columnar COMPRESSION for analytics
    
    MongoDB compression: structural (everything in one document)
    Redshift compression: physical (each column compressed separately)

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO DEBUG AND UNDERSTAND THE PIPELINE:

LAYER 1: MONGODB OPERATIONAL MONITORING
    Tool: Atlas UI / mongosh
    
    // How many orders created today?
    db.orders.countDocuments({
        created_at: { $gte: ISODate("2024-05-31T00:00:00Z") }
    })
    
    // What does the latest order look like?
    db.orders.find().sort({ created_at: -1 }).limit(1).pretty()
    
    // Is the oplog healthy? (change streams depend on this)
    db.getReplicationInfo()
    // Check: oplog window should be > 24 hours
    // If oplog is too small, change stream may miss events

LAYER 2: CHANGE STREAM MONITORING
    Tool: Application logs from Component 1
    
    Key metrics to track:
    → Events captured per second (throughput)
    → Resume token age (how current is the listener)
    → Serialization errors (BSON→JSON conversion failures)
    → Kafka publish failures (network issues)

LAYER 3: KAFKA MONITORING
    Tool: Kafka Manager / Confluent Control Center / AWS MSK Console
    
    Key metrics:
    → Consumer lag per partition (are consumers keeping up?)
    → Messages per second per topic
    → Partition distribution (is data evenly spread?)
    → Consumer group status (all consumers active?)

LAYER 4: TRANSFORMER MONITORING
    Tool: Application logs from Component 2
    
    Key metrics:
    → Documents transformed per second
    → Transformation errors (missing fields, type mismatches)
    → S3 writes per batch (micro-batch efficiency)
    → Buffer size over time (is batching working?)

LAYER 5: REDSHIFT LOADING MONITORING
    Tool: Redshift Query Editor / System Tables
    
    -- Check recent COPY operations
    SELECT query, filename, lines_scanned, 
           num_files, num_files_unknown
    FROM stl_load_commits
    ORDER BY curtime DESC
    LIMIT 20;
    
    -- Check for COPY errors
    SELECT starttime, filename, colname, 
           type, col_length, raw_field_value, err_reason
    FROM stl_load_errors
    ORDER BY starttime DESC
    LIMIT 20;
    
    -- Check pipeline load log (our custom tracking table)
    SELECT * FROM staging.pipeline_load_log
    ORDER BY started_at DESC
    LIMIT 20;

LAYER 6: END-TO-END RECONCILIATION
    Tool: Component 4 (monitoring script)
    
    → Count reconciliation: MongoDB count ≈ Redshift count
    → Value reconciliation: MongoDB SUM ≈ Redshift SUM
    → Lag monitoring: time between MongoDB write and Redshift availability
    
    ALL SIX LAYERS MUST BE MONITORED
    Pipeline failures are SILENT — data just stops flowing
    Without monitoring, analysts see stale dashboards and don't know why
