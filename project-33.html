PROJECT 33: Working Set & Memory Management

Problem: MongoDB Cluster Performance Degrades Unpredictably — Some Queries Return in 2ms, Others Take 500ms for the Same Operation

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform (optimized in Projects 31-32) has a new problem:
    
    SYMPTOMS:
    → Same query: sometimes 2ms, sometimes 500ms (250x variance!)
    → No index changes, no code changes, no schema changes
    → Problem worsens during peak hours (10am-2pm)
    → Problem worsens after nightly batch import
    → Application team says: "MongoDB is unreliable"
    → DevOps team says: "Server metrics look fine — CPU 40%, disk OK"
    
    THE MYSTERY:
    → explain() shows IXSCAN (index is being used — Project 31 ✅)
    → Index count is optimal (Project 32 ✅)
    → Query plan is perfect — so WHY the latency variance?
    
    THE ANSWER: WORKING SET EXCEEDS WIREDTIGER CACHE
    
    → When data/index pages are IN cache → 0.01ms access (RAM speed)
    → When data/index pages NOT in cache → 1-10ms access (disk speed)
    → 100-1000x difference between cache hit and cache miss
    → This is THE #1 cause of unpredictable MongoDB performance
    → And Redshift users NEVER think about this because RA3 handles it
    
    YOUR CLUSTER:
    → Atlas M50 instance: 32GB RAM → ~15GB WiredTiger cache
    → Data size: 28GB (across all collections)
    → Index size: 4.2GB (after Project 32 optimization)
    → Total: 32.2GB of data + indexes
    → Cache: 15GB
    → PROBLEM: 32.2GB doesn't fit in 15GB
    → SOME data must live on disk → cache misses → latency spikes
    
    THE QUESTION: "How do I make the RIGHT data stay in cache?"

🔴 REDSHIFT MINDSET — Why This Problem Is Different in Redshift

Redshift's Memory Architecture

WHY REDSHIFT USERS DON'T THINK ABOUT WORKING SET:

REDSHIFT MEMORY MODEL:
    ┌──────────────────────────────────────────────────┐
    │  Redshift RA3 Node                                │
    │                                                    │
    │  ┌────────────────┐    ┌────────────────────────┐ │
    │  │ Local SSD Cache │    │ Managed Storage (S3)    │ │
    │  │ (hot data)      │←──→│ (all data)              │ │
    │  │ Automatic!      │    │ Virtually unlimited     │ │
    │  │ No tuning needed│    │                          │ │
    │  └────────────────┘    └────────────────────────┘ │
    │                                                    │
    │  REDSHIFT HANDLES EVERYTHING:                      │
    │  → Automatically caches frequently accessed blocks  │
    │  → LRU eviction of cold blocks                     │
    │  → Prefetching for sequential scans                │
    │  → Result cache for repeated queries               │
    │  → NO user tuning knobs for cache                  │
    │  → "It just works" (mostly)                        │
    └──────────────────────────────────────────────────┘

REDSHIFT'S ADVANTAGES FOR MEMORY:
    1. Columnar storage → reads ONLY needed columns
       → 10-column table, query needs 2 columns → reads 20% of data
       → Effective working set is MUCH smaller than total data
    
    2. Compression → data smaller in memory
       → 10:1 compression ratio common
       → 100GB logical data = 10GB physical = fits in cache easily
    
    3. MPP distribution → data split across nodes
       → 100GB data across 4 nodes = 25GB per node
       → Each node's cache handles its portion
    
    4. RA3 managed storage → automatic hot/cold tiering
       → Frequently accessed blocks stay in local SSD
       → Rarely accessed blocks in S3 (slower but unlimited)
       → No manual management required

REDSHIFT'S HIDDEN MEMORY PROBLEMS (they exist but are less visible):
    → WLM memory allocation: queries compete for memory slots
    → Large sort operations: spill to disk if insufficient memory
    → Hash joins on large tables: temp space needed
    → BUT: Redshift handles these internally — users rarely tune them
    → Worst case: query takes longer, doesn't become unpredictable

WHY MONGODB IS DIFFERENT:
    → Document storage → reads ENTIRE document (all fields)
    → Even if you only need 2 fields → entire document loaded into cache
    → No columnar column pruning (except time series — Project 30)
    → Single-node (or per-shard) → all data on one server
    → WiredTiger cache is a HARD limit → must manage manually
    → THIS is the fundamental architectural difference

What Redshift Users Must Unlearn

REDSHIFT HABITS THAT HURT IN MONGODB:

HABIT 1: "Just add more storage"
    → Redshift RA3: add storage → data goes to S3 → no performance impact
    → MongoDB: add storage → data on disk → cache miss rate INCREASES
    → More data with same RAM = WORSE performance (opposite of Redshift!)
    → MongoDB fix: add RAM (scale up) or shard (scale out)

HABIT 2: "Wide tables are fine"
    → Redshift: 100-column table → query reads only needed columns
    → MongoDB: 100-field document → entire document loaded for ANY query
    → Wider documents = more cache space per document = fewer documents in cache
    → MongoDB fix: Subset Pattern (Project 19) — split hot and cold fields

HABIT 3: "Historical data doesn't affect performance"
    → Redshift: old data in cold S3 storage → doesn't affect hot queries
    → MongoDB: old data in same collection → competes for cache space
    → Scanning historical data evicts current hot data from cache
    → MongoDB fix: separate collections, TTL, or archive to cold storage

HABIT 4: "One big table for everything"
    → Redshift: one large fact table, sort key optimizes scans → fine
    → MongoDB: one large collection, everything in WiredTiger cache → cache pressure
    → MongoDB fix: separate collections by access pattern (hot/warm/cold)

🟢 MONGODB MINDSET — Understanding WiredTiger Cache

The Cache Architecture

WIREDTIGER CACHE — THE CORE CONCEPT:

    ┌──────────────────────────────────────────────────────────┐
    │  MongoDB Server (32GB RAM)                                │
    │                                                           │
    │  ┌────────────────────────────────────────────────────┐  │
    │  │ OS + MongoDB Process: ~2GB                          │  │
    │  └────────────────────────────────────────────────────┘  │
    │                                                           │
    │  ┌────────────────────────────────────────────────────┐  │
    │  │ WiredTiger Cache: ~15GB                             │  │
    │  │ (50% of RAM - 1GB = (32 × 0.5) - 1 = 15GB)        │  │
    │  │                                                      │  │
    │  │ ┌──────────┐ ┌──────────┐ ┌──────────┐             │  │
    │  │ │ Document  │ │ Index    │ │ Internal │             │  │
    │  │ │ Pages     │ │ Pages    │ │ Overhead │             │  │
    │  │ │           │ │          │ │ (~20%)   │             │  │
    │  │ │ HOT docs  │ │ B-tree   │ │ metadata │             │  │
    │  │ │ from      │ │ nodes    │ │ cursors  │             │  │
    │  │ │ active    │ │ from     │ │ sessions │             │  │
    │  │ │ queries   │ │ active   │ │          │             │  │
    │  │ │           │ │ indexes  │ │          │             │  │
    │  │ └──────────┘ └──────────┘ └──────────┘             │  │
    │  │                                                      │  │
    │  │ When FULL: evict least-recently-used pages            │  │
    │  │ Evicted page needed again → DISK READ → 100x slower  │  │
    │  └────────────────────────────────────────────────────┘  │
    │                                                           │
    │  ┌────────────────────────────────────────────────────┐  │
    │  │ OS File System Cache: ~15GB                         │  │
    │  │ (remaining RAM after WT cache + OS)                 │  │
    │  │ → Caches disk pages read by WiredTiger              │  │
    │  │ → Second-level cache (slower than WT cache)         │  │
    │  │ → Helps but NOT a substitute for WT cache           │  │
    │  └────────────────────────────────────────────────────┘  │
    │                                                           │
    │  ┌────────────────────────────────────────────────────┐  │
    │  │ Disk (SSD/NVMe):                                    │  │
    │  │ → All data + all indexes stored here                │  │
    │  │ → WiredTiger reads pages FROM disk INTO cache       │  │
    │  │ → Access time: ~0.1-1ms (vs ~0.001ms from cache)    │  │
    │  └────────────────────────────────────────────────────┘  │
    └──────────────────────────────────────────────────────────┘

CACHE SIZE FORMULA:
    Default: max(256MB, 50% of (RAM - 1GB))
    
    Examples:
    → 8GB RAM  → (8 × 0.5) - 0.5 = 3.5GB cache
    → 16GB RAM → (16 × 0.5) - 0.5 = 7.5GB cache
    → 32GB RAM → (32 × 0.5) - 0.5 = 15.5GB cache
    → 64GB RAM → (64 × 0.5) - 0.5 = 31.5GB cache
    
    Atlas M-series:
    → M10: 2GB RAM → ~0.75GB cache (very limited!)
    → M30: 8GB RAM → ~3.5GB cache
    → M50: 32GB RAM → ~15GB cache
    → M60: 64GB RAM → ~31GB cache
    → M80: 128GB RAM → ~63GB cache

KEY INSIGHT:
    → Your WORKING SET must fit in the WT cache
    → Working set ≠ total data size
    → Working set = data + indexes ACTIVELY ACCESSED
    → If working set > cache → performance cliff

What Is "Working Set" — The Critical Concept

WORKING SET DEFINITION:

    Working Set = all data pages + index pages that your application
                  accesses within a given time window (typically minutes)

    EXAMPLES:
    
    E-COMMERCE PLATFORM:
    → Total data: 28GB
    → Total indexes: 4.2GB
    → But in any given hour, the application accesses:
      → Product catalog (browsed products): ~2GB
      → Active carts (current shoppers): ~500MB
      → Recent orders (order status checks): ~1GB
      → User sessions (logged-in users): ~300MB
      → Index pages for above queries: ~2GB
      → WORKING SET: ~5.8GB
    → 15GB cache easily holds 5.8GB → ✅ FAST
    
    SAME PLATFORM DURING NIGHTLY BATCH:
    → Batch import reads/writes ALL orders: ~12GB
    → Analytics queries scan ALL products: ~8GB
    → Index pages for batch operations: ~3GB
    → WORKING SET: ~23GB
    → 15GB cache CANNOT hold 23GB → ❌ THRASHING
    → Hot product/cart data evicted → morning users hit cold cache
    
    THIS IS WHY:
    → Daytime: 2ms queries (data in cache)
    → After batch: 500ms queries (data evicted, must reload from disk)
    → Gradually recovers as hot data reloads → "self-healing" appearance
    → But EVERY NIGHT the cycle repeats

🧩 HOW TO APPROACH IT IN MONGODB

THE MEMORY MANAGEMENT METHODOLOGY — 5 STEPS:

STEP 1: MEASURE — What is the current cache state?
    → db.serverStatus().wiredTiger.cache
    → Key metrics: bytes in cache, eviction rate, read-into-cache rate
    → Atlas: Monitoring → Cache metrics dashboard

STEP 2: CALCULATE — What is the working set size?
    → Data accessed in typical hour: sum of hot collection sizes
    → Index pages for hot queries: estimate from index sizes
    → Internal overhead: ~20% of cache
    → Working set = hot data + hot indexes + overhead

STEP 3: COMPARE — Does working set fit in cache?
    → Working set < 80% of cache → ✅ healthy headroom
    → Working set 80-95% of cache → ⚠️ monitor closely
    → Working set > 95% of cache → ❌ thrashing imminent

STEP 4: OPTIMIZE — Reduce working set or increase cache
    → Reduce: Subset Pattern, archive old data, smaller documents
    → Increase: scale up (more RAM), shard (distribute data)
    → Isolate: separate hot/cold collections, read preferences

STEP 5: MONITOR — Set up alerts for cache pressure
    → Alert when eviction rate exceeds threshold
    → Alert when cache utilization > 90%
    → Atlas: built-in cache alerts
    → Self-hosted: custom monitoring with serverStatus

function cacheSnapshot() {
    let s = db.serverStatus().wiredTiger.cache
    return {
        timestamp: new Date(),
        used: s["bytes currently in the cache"],
        max: s["maximum bytes configured"],
        dirty: s["tracked dirty bytes in the cache"],
        readIn: s["bytes read into cache"],
        evictedClean: s["unmodified pages evicted"],
        evictedDirty: s["modified pages evicted"]

// ── MONITORING SCRIPT (run periodically via cron/Atlas trigger) ──

function cacheHealthCheck() {
    let s = db.serverStatus().wiredTiger.cache
    let cacheMax = s["maximum bytes configured"]
    let cacheUsed = s["bytes currently in the cache"]
    let cacheDirty = s["tracked dirty bytes in the cache"]
    let evictedDirty = s["modified pages evicted"]
    
    let usedPct = (cacheUsed / cacheMax * 100)
    let dirtyPct = (cacheDirty / cacheMax * 100)
    
    let alerts = []
    
    // Alert 1: Cache utilization > 90%
    if (usedPct > 90) {
        alerts.push({
            level: "CRITICAL",
            metric: "cache_utilization",
            value: usedPct.toFixed(1) + "%",
            message: "WiredTiger cache > 90% — eviction thrashing likely"
        })
    } else if (usedPct > 80) {
        alerts.push({
            level: "WARNING",
            metric: "cache_utilization",
            value: usedPct.toFixed(1) + "%",
            message: "WiredTiger cache > 80% — approaching limit"
        })
    }
    
    // Alert 2: Dirty cache > 20%
    if (dirtyPct > 20) {
        alerts.push({
            level: "WARNING",
            metric: "dirty_cache",
            value: dirtyPct.toFixed(1) + "%",
            message: "Dirty cache > 20% — checkpoint pressure"
        })
    }
    
    // Alert 3: Dirty page eviction occurring
    if (evictedDirty > 0) {
        alerts.push({
            level: "CRITICAL",
            metric: "dirty_eviction",
            value: evictedDirty.toString(),
            message: "Dirty pages being evicted — severe cache pressure"
        })
    }
    
    // Log results
    let healthDoc = {
        timestamp: new Date(),
        cache_used_pct: usedPct,
        cache_dirty_pct: dirtyPct,
        cache_max_gb: (cacheMax / 1024 / 1024 / 1024).toFixed(2),
        cache_used_gb: (cacheUsed / 1024 / 1024 / 1024).toFixed(2),
        alerts: alerts,
        status: alerts.length === 0 ? "HEALTHY" : 
                alerts.some(a => a.level === "CRITICAL") ? "CRITICAL" : "WARNING"
    }
    
    // Store in monitoring collection
    db.cache_health_log.insertOne(healthDoc)
    
    return healthDoc
}

⚙️ HOW IT ONLY WORKS INTERNALLY

WiredTiger Cache — Deep Internals

WHAT HAPPENS ON A CACHE MISS:

    Query: db.orders.findOne({ customer_id: "cust_5000" })
    Index: { customer_id: 1, order_date: -1 }

    STEP 1: INDEX PAGE LOOKUP
    → B-tree root page: check if in cache
      → IN CACHE: read from RAM (~0.001ms) ← cache hit
      → NOT IN CACHE: read from disk (~0.5ms) ← cache miss
    → Walk B-tree: root → internal → leaf
    → Each page: check cache → if miss → disk read
    → Worst case: 3-4 page reads for deep B-tree

    STEP 2: DOCUMENT PAGE LOOKUP
    → Leaf node contains: document location (file offset)
    → Read document page from collection file
      → IN CACHE: ~0.001ms
      → NOT IN CACHE: ~0.5-2ms (disk seek + read)
    → Decompress page (if loaded from disk)
    → Extract document from page

    STEP 3: CACHE INSERTION (if miss)
    → Loaded page placed in WiredTiger cache
    → If cache full → EVICTION triggered
    → LRU (Least Recently Used): oldest page evicted
    → If evicted page is DIRTY → write to disk first (expensive!)
    → If evicted page is CLEAN → just discard (cheap)

    TOTAL LATENCY:
    ALL CACHE HITS:
    → Index: 3 pages × 0.001ms = 0.003ms
    → Document: 1 page × 0.001ms = 0.001ms
    → Total: ~0.004ms → rounds to <1ms
    
    ALL CACHE MISSES:
    → Index: 3 pages × 0.5ms = 1.5ms
    → Document: 1 page × 1ms = 1ms
    → Eviction: 1 page × 0.5ms = 0.5ms (if cache full)
    → Total: ~3ms → but can spike to 10-50ms under pressure
    
    MIXED (common in practice):
    → Index root: in cache (hot)
    → Index internal: in cache (warm)
    → Index leaf: MISS (cold)
    → Document: MISS (cold)
    → Total: ~1-3ms

    THIS EXPLAINS THE VARIANCE:
    → Same query, same index, same plan
    → But different latency depending on cache state
    → 2ms (all hits) vs 500ms (misses + eviction + disk contention)

EVICTION MECHANICS:

    ┌───────────────────────────────────────────────────────┐
    │  WiredTiger Cache (15GB)                               │
    │                                                        │
    │  ┌──────────────────────────────────────────────┐     │
    │  │ HOT PAGES (recently accessed)                  │     │
    │  │ → Product listings, active carts, recent orders│     │
    │  │ → Protected from eviction (recently used)      │     │
    │  ├──────────────────────────────────────────────┤     │
    │  │ WARM PAGES (accessed recently but not just now) │     │
    │  │ → Index internal nodes, session data            │     │
    │  │ → May be evicted if cache pressure increases    │     │
    │  ├──────────────────────────────────────────────┤     │
    │  │ COLD PAGES (haven't been accessed in a while)  │     │
    │  │ → Old order data, inactive user profiles       │     │
    │  │ → FIRST to be evicted when cache full          │     │
    │  └──────────────────────────────────────────────┘     │
    │                                                        │
    │  EVICTION TRIGGERS:                                    │
    │  → Cache utilization > 80% → background eviction starts│
    │  → Cache utilization > 95% → aggressive eviction       │
    │  → Application threads may be BLOCKED doing eviction   │
    │  → This is the "performance cliff" — threads stall     │
    │                                                        │
    │  EVICTION TARGETS (in order):                          │
    │  1. Clean cold pages (cheapest — just discard)         │
    │  2. Clean warm pages (still cheap)                     │
    │  3. Dirty cold pages (must write to disk first!)       │
    │  4. Dirty warm pages (expensive + stalls writes)       │
    │  → Stage 3-4 = "dirty eviction" = CRITICAL alert      │
    └───────────────────────────────────────────────────────┘

CHECKPOINT PROCESS:
    → Every 60 seconds (default): WiredTiger checkpoints
    → Writes ALL dirty pages to disk
    → After checkpoint: dirty pages become clean
    → Clean pages can be evicted cheaply
    → Frequent checkpoints reduce dirty page accumulation
    → But checkpoints themselves cause I/O spikes

Redshift vs MongoDB — Memory Architecture Comparison

FUNDAMENTAL DIFFERENCE:

REDSHIFT (Columnar + MPP + Managed Storage):
    
    Query: SELECT price, name FROM products WHERE category = 'Electronics'
    
    Memory path:
    1. Read ONLY category column block → check zone map → skip/read
    2. Read ONLY price column block → for matching rows
    3. Read ONLY name column block → for matching rows
    4. Total memory: (category + price + name) × matching rows
    → Maybe 10% of total table data loaded into memory
    → Even without cache: columnar reads are inherently efficient
    
    ADVANTAGE: Column pruning reduces effective working set by 5-10x

MONGODB (Document + Single-Thread + WiredTiger):
    
    Query: db.products.find({ category: "Electronics" }, { price: 1, name: 1 })
    
    Memory path:
    1. Index lookup: IXSCAN on { category: 1 } → find matching doc IDs
    2. For EACH matching document:
       a. Load ENTIRE document page into cache (ALL fields!)
       b. Extract price and name from document
       c. Return only price and name to client
    3. Total memory: ALL fields × ALL matching documents in cache
    → 100% of document data loaded for any field access
    → Projection only reduces NETWORK transfer, not CACHE usage
    
    DISADVANTAGE: No column pruning → working set = full documents

    EXCEPTION — TIME SERIES (Project 30):
    → Columnar storage within buckets
    → Can read ONLY needed measurement arrays
    → Working set reduced for time series queries
    → THIS is why time series collections are more memory-efficient

    WHY GRADUAL:
    → RA3: automatic tiering (hot SSD → cold S3)
    → S3 reads: ~5-50ms (slower but not catastrophic)
    → Columnar: less data to cache per query
    → MPP: data distributed → each node handles less
    → Result: performance degrades smoothly as data grows

    MONGODB DEGRADATION CURVE:
    Performance │
    ████████████│████████████████
    ████████████│████████████████
    ████████████│████████████████
    ████████████│███████████████
    ████████████│█
    ████████████│                   ← CLIFF!
    ────────────┼──────────────── Working Set Size →
                │ Cache Limit
    
    WHY CLIFF:
    → Working set < cache: ALL reads from RAM → consistently fast
    → Working set > cache: SOME reads from disk → unpredictable
    → Eviction cascades: evicting page A to load B,
      then evicting B to reload A → THRASHING
    → Dirty eviction: must write before evict → write stalls
    → Application threads blocked doing eviction work
    → Result: sudden, dramatic performance collapse
    
    THE CLIFF POINT:
    → Typically at 85-95% cache utilization
    → Below 80%: smooth, fast, predictable
    → 80-90%: occasional spikes, mostly OK
    → 90-95%: frequent spikes, degrading
    → >95%: thrashing, dirty eviction, severe degradation
    → THIS is why 80% is the target utilization threshold

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE WITH MEMORY MANAGEMENT:

PITFALL 1: "Disk size = memory needed"
    ❌ "Data is 4GB on disk, so 8GB RAM should be plenty"
    → WiredTiger cache stores UNCOMPRESSED data
    → 4GB on disk (compressed) = 10-20GB in cache (uncompressed)
    → Plus: index pages, internal overhead (~20%)
    → ACTUAL need: 12-25GB cache for 4GB disk data
    ✅ Multiply disk size by 2-5x for cache sizing
    → Then add index sizes + 20% overhead
    → Choose instance where cache > estimated working set × 1.2

PITFALL 2: "Projection reduces cache usage"
    ❌ db.orders.find({ status: "active" }, { name: 1, total: 1 })
       "I'm only reading 2 fields, so cache impact is minimal"
    → MongoDB loads ENTIRE document page into cache regardless
    → Projection only reduces what's sent to the CLIENT
    → Cache still holds full documents
    ✅ Use COVERED QUERIES for true cache reduction
    → Design index to include projected fields
    → PROJECTION_COVERED stage = zero document page reads
    → Or use Subset Pattern (separate small-document collection)

PITFALL 3: "Adding RAM always fixes slow queries"
    ❌ "Queries are slow → upgrade from M30 to M50"
    → If working set is 3GB and cache is 3.5GB (M30)
    → Upgrading to M50 (15GB cache) helps
    → BUT: if the problem is a missing index (Project 31)
    → Or too many indexes (Project 32)
    → Or a bad query pattern
    → Adding RAM masks the problem, doesn't fix root cause
    ✅ Diagnose in order: explain() → index audit → THEN memory
    → Projects 31 → 32 → 33 = correct diagnostic sequence
    → RAM upgrade is the LAST resort, not the first

PITFALL 4: "Nightly batch won't affect daytime performance"
    ❌ Running large ETL/import at 2am, assuming it recovers by 9am
    → Batch writes 5M documents → 5M × N index updates
    → Cache completely churned → all hot operational data evicted
    → 9am: first user queries → ALL cache misses → cold start
    → Takes 30-60 minutes for cache to warm up with hot data
    ✅ Mitigations:
    → Run batch on secondary (separate cache — don't pollute primary)
    → Use ordered inserts (better cache locality during batch)
    → Pre-warm cache after batch: run common queries programmatically
    → Schedule batch to finish 1+ hour before peak traffic

PITFALL 5: "One big collection is fine — it works in Redshift"
    ❌ 50M documents in one collection (10GB data, 3GB indexes)
    → Any full collection scan (analytics) loads everything into cache
    → Evicts operational hot data
    → Redshift handles this: columnar reads only needed columns
    → MongoDB cannot: reads entire documents
    ✅ Split by access pattern:
    → Hot collection: recent/active data (small, cache-friendly)
    → Cold collection: historical data (large, disk-resident)
    → Archive: very old data (S3 via Atlas Data Federation)
    → Each collection's hot data fits in cache independently

PITFALL 6: "db.collection.stats().size tells me cache usage"
    ❌ stats().size = 2GB → "only 2GB in cache"
    → stats().size reports COMPRESSED on-disk size
    → Cache holds UNCOMPRESSED data
    → Actual cache footprint: 2GB × compression ratio (2-5x)
    → Could be 4-10GB in cache!
    ✅ Use stats().size × estimated_compression_ratio for cache sizing
    → Or check db.serverStatus().wiredTiger.cache for actual usage
    → Atlas: check Cache Utilization metric directly

PITFALL 7: "I'll just set cacheSizeGB to 90% of RAM"
    ❌ 32GB server → cacheSizeGB: 28
    → OS has only 4GB for: filesystem cache, kernel, network, mongod process
    → Filesystem cache = second-level cache for MongoDB (important!)
    → Too little OS RAM → kernel starts swapping → CATASTROPHIC
    → Swapping is 1000x slower than cache miss
    ✅ Keep default (50% of RAM - 1GB) unless carefully analyzed
    → Maximum safe: 70% of RAM for WiredTiger
    → Leave 30% for OS + filesystem cache + mongod process
    → Monitor OS swap usage: if swapping → reduce cacheSizeGB

def cache_health_report():
    """Complete WiredTiger cache analysis"""
    
    print(f"\n{'='*70}")
    print("WIREDTIGER CACHE HEALTH REPORT")
    print(f"{'='*70}")
    
    status = db.command("serverStatus")
    cache = status.get("wiredTiger", {}).get("cache", {})
    
    if not cache:
        print("  ⚠️ WiredTiger cache stats not available")
        print("  (May be using a different storage engine or insufficient permissions)")
        return None
    
    # Extract metrics
    cache_max = cache.get("maximum bytes configured", 0)
    cache_used = cache.get("bytes currently in the cache", 0)
    cache_dirty = cache.get("tracked dirty bytes in the cache", 0)
    read_into = cache.get("bytes read into cache", 0)
    written_from = cache.get("bytes written from cache", 0)
    evict_clean = cache.get("unmodified pages evicted", 0)
    evict_dirty = cache.get("modified pages evicted", 0)
    pages_in = cache.get("pages currently held in the cache", 0)
    pages_read = cache.get("pages read into cache", 0)
    
    # Calculate
    used_pct = (cache_used / max(cache_max, 1)) * 100
    dirty_pct = (cache_dirty / max(cache_max, 1)) * 100
    free_gb = (cache_max - cache_used) / 1024 / 1024 / 1024
    
    # Display
    print(f"\n  CACHE CAPACITY:")
    print(f"  ├── Maximum:        {cache_max / 1024 / 1024 / 1024:.2f} GB")
    print(f"  ├── Used:           {cache_used / 1024 / 1024 / 1024:.2f} GB ({used_pct:.1f}%)")
    print(f"  ├── Dirty:          {cache_dirty / 1024 / 1024:.0f} MB ({dirty_pct:.1f}%)")
    print(f"  ├── Free:           {free_gb:.2f} GB")
    print(f"  └── Pages in cache: {pages_in:,}")
    
    print(f"\n  CACHE ACTIVITY (cumulative since start):")
    print(f"  ├── Bytes read in:     {read_into / 1024 / 1024 / 1024:.2f} GB")
    print(f"  ├── Bytes written out: {written_from / 1024 / 1024 / 1024:.2f} GB")
    print(f"  ├── Pages read in:     {pages_read:,}")
    print(f"  ├── Clean evictions:   {evict_clean:,}")
    print(f"  └── Dirty evictions:   {evict_dirty:,}")
    
    # Health assessment
    print(f"\n  HEALTH ASSESSMENT:")
    
    issues = []
    
    if used_pct > 95:
        issues.append(("CRITICAL", f"Cache {used_pct:.1f}% full — active thrashing"))
    elif used_pct > 85:
        issues.append(("WARNING", f"Cache {used_pct:.1f}% full — approaching limit"))
    
    if dirty_pct > 20:
        issues.append(("WARNING", f"Dirty cache {dirty_pct:.1f}% — checkpoint pressure"))
    
    if evict_dirty > 1000:
        issues.append(("CRITICAL", f"{evict_dirty:,} dirty evictions — severe pressure"))
    
    if not issues:
        print(f"  ✅ HEALTHY — cache utilization {used_pct:.1f}%, no pressure detected")
    else:
        for level, msg in issues:
            icon = "❌" if level == "CRITICAL" else "⚠️"
            print(f"  {icon} {level}: {msg}")
    
    # Recommendations
    print(f"\n  RECOMMENDATIONS:")
    if used_pct > 85:
        print(f"  → Scale up instance (more RAM → larger cache)")
        print(f"  → Or shard collection to distribute working set")
        print(f"  → Or apply Subset Pattern to reduce document sizes")
        print(f"  → Or archive old data to separate collection")
    elif used_pct > 60:
        print(f"  → Monitor during peak hours — may approach limit")
        print(f"  → Plan capacity for data growth")
    else:
        print(f"  → Cache healthy — consider if instance is over-provisioned")
        print(f"  → Potential cost savings by downsizing")
    
    return {
        "cache_max_gb": cache_max / 1024 / 1024 / 1024,
        "cache_used_gb": cache_used / 1024 / 1024 / 1024,
        "used_pct": used_pct,
        "dirty_pct": dirty_pct,
        "evict_dirty": evict_dirty,
        "issues": issues
    }

def working_set_analysis():

    total_data = 0

    print(f"  {'-'*30} {'-'*10} {'-'*10} {'-'*12} {'-'*8} {'-'*12}")
    
    for name in db.list_collection_names():
        if name.startswith("system."):
            continue
        
        try:
            stats = db.command("collstats", name)
            data_mb = stats.get("size", 0) / 1024 / 1024
            index_mb = stats.get("totalIndexSize", 0) / 1024 / 1024
            count = stats.get("count", 0)
            avg_doc = stats.get("avgObjSize", 0)
            storage_mb = stats.get("storageSize", 0) / 1024 / 1024
            
            # Estimate compression ratio
            if storage_mb > 0 and data_mb > 0:
                compression_ratio = data_mb / storage_mb
            else:
                compression_ratio = 2.5  # default estimate
            
            # Cache holds UNCOMPRESSED data
            # stats.size is already uncompressed logical size
            # But WiredTiger internal format adds ~10-20% overhead
            cache_estimate_mb = (data_mb * 1.15) + (index_mb * 1.1)
            
            total_data += data_mb
            total_index += index_mb
            total_uncompressed_estimate += cache_estimate_mb
            
            collections_data.append({
                "name": name,
                "data_mb": data_mb,
                "index_mb": index_mb,
                "count": count,
                "avg_doc": avg_doc,
                "cache_mb": cache_estimate_mb
            })
            
            print(f"  {name:<30s} {data_mb:>10.1f} {index_mb:>10.1f} {count:>12,} {avg_doc:>8.0f} {cache_estimate_mb:>12.1f}")

            print(f"  {name:<30s} Error: {e}")
    
    print(f"  {'-'*84}")
    print(f"  {'TOTAL':<30s} {total_data:>10.1f} {total_index:>10.1f} {'':>12s} {'':>8s} {total_uncompressed_estimate:>12.1f}")

    print(f"\n  CACHE CAPACITY: {cache_max_gb:.2f} GB ({cache_max_gb * 1024:.0f} MB)")
    print(f"  TOTAL DATA + INDEXES (cache estimate): {total_uncompressed_estimate:.0f} MB ({total_uncompressed_estimate/1024:.2f} GB)")
    
    # Internal overhead
    overhead_mb = cache_max_gb * 1024 * 0.20
    usable_cache_mb = cache_max_gb * 1024 * 0.80
    
    print(f"  Internal overhead (~20%): {overhead_mb:.0f} MB")
    print(f"  Usable cache: {usable_cache_mb:.0f} MB ({usable_cache_mb/1024:.2f} GB)")
    
    if total_uncompressed_estimate < usable_cache_mb:
        print(f"\n  ✅ ALL DATA FITS IN CACHE")
        print(f"     Headroom: {usable_cache_mb - total_uncompressed_estimate:.0f} MB ({(1 - total_uncompressed_estimate/usable_cache_mb)*100:.0f}%)")
    else:
        print(f"\n  ❌ DATA EXCEEDS CACHE CAPACITY")
        print(f"     Overflow: {total_uncompressed_estimate - usable_cache_mb:.0f} MB")
        print(f"     Need: {total_uncompressed_estimate/1024/0.8:.1f} GB cache (accounting for 20% overhead)")
        
        # Estimate hot data percentage
        for hot_pct in [0.2, 0.3, 0.4, 0.5]:
            hot_mb = total_uncompressed_estimate * hot_pct
            fits = "✅" if hot_mb < usable_cache_mb else "❌"
            print(f"     If {hot_pct*100:.0f}% is hot: {hot_mb:.0f} MB → {fits}")
    
    return collections_data, total_uncompressed_estimate, cache_max_gb

            evict_rate = (snapshot["evicted"] - prev_snapshot["evicted"]) / elapsed
            
            status = "✅ OK"
            if used_pct > 90:
                status = "❌ CRITICAL"
            elif used_pct > 80:
                status = "⚠️ WARNING"
            elif evict_rate > 100:
                status = "⚠️ EVICTING"
            
            print(f"  {snapshot['time'].strftime('%H:%M:%S'):>8s} "
                  f"{used_pct:>6.1f}% {dirty_pct:>6.1f}% "
                  f"{read_rate_mb:>11.1f} {evict_rate:>8.0f} {status}")
        
        snapshots.append(snapshot)
        prev_snapshot = snapshot
        
        if i < iterations:
            time.sleep(interval_seconds)
    
    # Summary
    used_pcts = [(s["used"]/s["max"])*100 for s in snapshots]
    print(f"\n  SUMMARY:")
    print(f"  ├── Min utilization: {min(used_pcts):.1f}%")
    print(f"  ├── Max utilization: {max(used_pcts):.1f}%")
    print(f"  ├── Avg utilization: {sum(used_pcts)/len(used_pcts):.1f}%")
    
    if max(used_pcts) > 90:
        print(f"  └── ❌ Cache pressure detected during monitoring window")
    else:
        print(f"  └── ✅ Cache stable during monitoring window")
    
    return snapshots

def implement_subset_pattern(source_collection, summary_collection,
                              hot_fields, index_specs=None):
    """Create a cache-friendly summary collection from a large collection"""
    
    print(f"\n{'='*70}")
    print(f"SUBSET PATTERN: {source_collection} → {summary_collection}")
    print(f"{'='*70}")
    
    source = db[source_collection]
    summary = db[summary_collection]
    
    # Before stats
    source_stats = db.command("collstats", source_collection)
    source_avg = source_stats.get("avgObjSize", 0)
    source_count = source_stats.get("count", 0)
    source_size_mb = source_stats.get("size", 0) / 1024 / 1024
    
    print(f"\n  SOURCE: {source_collection}")
    print(f"  ├── Documents:    {source_count:,}")
    print(f"  ├── Avg doc size: {source_avg:,} bytes")
    print(f"  └── Data size:    {source_size_mb:.1f} MB")
    
    # Build projection
    projection = {field: 1 for field in hot_fields}
    projection["_id"] = 1
    
    print(f"\n  HOT FIELDS: {hot_fields}")
    print(f"  Creating summary collection...")
    
    # Create summary via aggregation
    summary.drop()
    source.aggregate([
        {"$project": projection},
        {"$out": summary_collection}
    ])
    
    # Create indexes on summary
    if index_specs:
        for spec in index_specs:
            keys, name = spec
            summary.create_index(keys, name=name)
            print(f"  Index created: {name}")
    
    # After stats
    summary_stats = db.command("collstats", summary_collection)
    summary_avg = summary_stats.get("avgObjSize", 0)
    summary_count = summary_stats.get("count", 0)
    summary_size_mb = summary_stats.get("size", 0) / 1024 / 1024
    
    print(f"\n  SUMMARY: {summary_collection}")
    print(f"  ├── Documents:    {summary_count:,}")
    print(f"  ├── Avg doc size: {summary_avg:,} bytes")
    print(f"  └── Data size:    {summary_size_mb:.1f} MB")
    
    # Improvement
    size_reduction = (1 - summary_size_mb / max(source_size_mb, 0.01)) * 100
    doc_reduction = (1 - summary_avg / max(source_avg, 1)) * 100
    
    print(f"\n  CACHE IMPROVEMENT:")
    print(f"  ├── Doc size reduction: {doc_reduction:.0f}% ({source_avg} → {summary_avg} bytes)")
    print(f"  ├── Data size reduction: {size_reduction:.0f}% ({source_size_mb:.1f} → {summary_size_mb:.1f} MB)")
    print(f"  └── {size_reduction:.0f}% less cache pressure for listing queries")
    
    # Estimate uncompressed cache savings
    cache_saved_mb = (source_size_mb - summary_size_mb) * 2.5  # uncompressed estimate
    print(f"\n  ESTIMATED CACHE SAVINGS: ~{cache_saved_mb:.0f} MB (uncompressed)")
    
    return {
        "source_size_mb": source_size_mb,
        "summary_size_mb": summary_size_mb,
        "reduction_pct": size_reduction,
        "cache_saved_mb": cache_saved_mb
    }

def archive_old_data(collection_name, date_field, cutoff_days,
                      archive_suffix="_archive"):
    """Move old data to archive collection to reduce working set"""
    
    print(f"\n{'='*70}")
    print(f"DATA ARCHIVAL: {collection_name} (>{cutoff_days} days old)")
    print(f"{'='*70}")
    
    collection = db[collection_name]
    archive_name = collection_name + archive_suffix
    
    cutoff_date = datetime.now() - timedelta(days=cutoff_days)
    
    # Count documents to archive
    archive_count = collection.count_documents({date_field: {"$lt": cutoff_date}})
    total_count = collection.count_documents({})
    keep_count = total_count - archive_count
    
    print(f"\n  Total documents:     {total_count:,}")
    print(f"  To archive (>{cutoff_days}d): {archive_count:,} ({archive_count/max(total_count,1)*100:.0f}%)")
    print(f"  To keep (≤{cutoff_days}d):   {keep_count:,} ({keep_count/max(total_count,1)*100:.0f}%)")
    
    if archive_count == 0:
        print(f"\n  ℹ️ No documents older than {cutoff_days} days — nothing to archive")
        return
    
    # Before stats
    before_size = db.command("collstats", collection_name).get("size", 0) / 1024 / 1024
    
    # Step 1: Copy old docs to archive
    print(f"\n  Step 1: Copying to {archive_name}...")
    collection.aggregate([
        {"$match": {date_field: {"$lt": cutoff_date}}},
        {"$out": archive_name}
    ])
    
    archived = db[archive_name].count_documents({})
    print(f"  Archived: {archived:,} documents")
    
    # Step 2: Delete from main collection
    print(f"  Step 2: Deleting from {collection_name}...")
    delete_result = collection.delete_many({date_field: {"$lt": cutoff_date}})
    print(f"  Deleted: {delete_result.deleted_count:,} documents")
    
    # After stats
    after_size = db.command("collstats", collection_name).get("size", 0) / 1024 / 1024
    archive_size = db.command("collstats", archive_name).get("size", 0) / 1024 / 1024
    
    print(f"\n  RESULTS:")
    print(f"  ├── {collection_name}: {before_size:.1f} MB → {after_size:.1f} MB")
    print(f"  ├── {archive_name}: {archive_size:.1f} MB")
    print(f"  ├── Active collection reduction: {(1-after_size/max(before_size,0.01))*100:.0f}%")
    print(f"  └── Estimated cache savings: ~{(before_size - after_size) * 2.5:.0f} MB (uncompressed)")
    
    # -- REDSHIFT EQUIVALENT:
    # -- DELETE FROM orders WHERE order_date < DATEADD(day, -30, GETDATE());
    # -- VACUUM orders;  ← can take HOURS on large tables
    # -- MongoDB archival: faster, more flexible, no VACUUM needed
