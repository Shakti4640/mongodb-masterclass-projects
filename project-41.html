PROJECT 41: Global Clusters — Multi-Region Deployments

Problem: Global App — Users in US, EU, Asia Need <50ms Reads to Local Data

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your fintech platform has gone global
    
    USER BASE:
        → 12 million users in North America (US-EAST, US-WEST)
        → 8 million users in Europe (EU-WEST: Ireland, EU-CENTRAL: Frankfurt)
        → 6 million users in Asia-Pacific (AP-SOUTHEAST: Singapore, AP-NORTHEAST: Tokyo)
        → Total: 26 million active users
    
    COLLECTIONS:
        → users: 26M documents (profile, preferences, KYC status)
        → accounts: 40M documents (balances, account types)
        → transactions: 2.1B documents (payment history, transfers)
        → notifications: 800M documents (alerts, messages)
    
    REQUIREMENTS:
        → Users MUST read their own data in <50ms (local region)
        → Writes MUST be durable and consistent (financial data)
        → EU user data MUST stay in EU (GDPR — Project 39)
        → US user data MUST stay in US (regulatory)
        → APAC data MUST stay in APAC (data sovereignty)
        → Global admin dashboard needs to query ALL regions
        → 99.999% availability (five nines — <5.26 minutes downtime/year)
        → Disaster recovery: survive entire region failure
    
    CURRENT STATE (from Project 39-40):
        → 5-shard cluster in US-EAST
        → Zone sharding by region on single cluster
        → Cross-region latency: US→EU = 80-120ms, US→APAC = 150-250ms
        → Users in EU experiencing 200-300ms reads (unacceptable)
    
    THE QUESTION:
        → How do you make ONE database span the entire planet?
        → How do you guarantee local reads AND global consistency?
        → How does this compare to Redshift's cross-region story?

-- 1. Read-only on consumer side

-- 7. RA3 node type required (not dc2, not Serverless)

🟢 MONGODB MINDSET — How This Same Problem Is Approached

MongoDB Atlas Global Clusters — Native Multi-Region

MONGODB'S ANSWER: Global Clusters (Atlas)

    ONE logical database → MULTIPLE physical regions
    → Data automatically routed to correct region
    → Reads served from LOCAL region (<10ms)
    → Writes routed to region that OWNS the data (<50ms for local)
    → GDPR compliance built into the architecture
    → Survive entire region failure automatically

HOW IT WORKS:
    1. Location field in every document: { country: "DE" }
    2. Shard key includes location: { country: 1, user_id: 1 }
    3. Zone tags map countries to regions:
       → "DE", "FR", "GB" → EU shard
       → "US", "CA" → US shard
       → "JP", "SG", "AU" → APAC shard
    4. mongos reads zone config → routes to correct region
    5. Application connects to NEAREST mongos (global endpoint)
    6. Each region has its own mongos, config server, and shard(s)

The Three Global Cluster Patterns

PATTERN 1: GEOGRAPHICALLY PARTITIONED DATA (our fintech use case)
    → Each user's data lives in ONE region only
    → Writes go to the owning region
    → Reads served from the owning region
    → Cross-region queries possible but slower ($lookup across regions)
    
    BEST FOR:
    → Regulatory compliance (GDPR, data sovereignty)
    → User-facing applications with regional affinity
    → When users rarely access data from other regions
    
    TRADE-OFF:
    → Global admin queries must scatter-gather across all regions
    → A user traveling from US to EU still reads from US shard

PATTERN 2: GEOGRAPHICALLY DISTRIBUTED REPLICAS
    → Data written to PRIMARY region
    → Replicas in EVERY region for local reads
    → All writes go to one region (like Redshift data sharing)
    → Reads served from local replica
    
    ┌──────────────────────────────────────────────────────────┐
    │ US-EAST (Primary)     EU-WEST (Replica)   APAC (Replica) │
    │ ┌──────────┐         ┌──────────┐        ┌──────────┐   │
    │ │ P  S  S  │ ──rep──→│ S  S     │        │ S  S     │   │
    │ │ (writes) │         │ (reads)  │        │ (reads)  │   │
    │ └──────────┘         └──────────┘        └──────────┘   │
    │                                                          │
    │ ALL writes → US-EAST                                     │
    │ Reads → nearest replica (local region)                   │
    │ Replication lag: 50-200ms (cross-region)                 │
    └──────────────────────────────────────────────────────────┘
    
    BEST FOR:
    → Read-heavy workloads (90%+ reads)
    → Reference data (product catalogs, config)
    → When write locality doesn't matter
    
    TRADE-OFF:
    → Writes always cross-region (unless user is in primary region)
    → Replication lag → eventual consistency for reads in non-primary

PATTERN 3: HYBRID (Most Real-World Deployments)
    → User-specific data: geographically partitioned (Pattern 1)
    → Reference data: geographically replicated (Pattern 2)
    → Global aggregation data: computed locally, merged globally

🧩 HOW TO APPROACH IT IN MONGODB

GLOBAL CLUSTER DESIGN FRAMEWORK:

STEP 1 — CLASSIFY YOUR DATA
    Every collection falls into one of three categories:
    
    ┌─────────────────────────────────────────────────────────────────┐
    │ CATEGORY          │ DESCRIPTION              │ STRATEGY          │
    ├─────────────────────────────────────────────────────────────────┤
    │ REGIONAL           │ Belongs to ONE region     │ Zone-sharded      │
    │ (users, accounts,  │ Must stay in that region  │ by location field │
    │  transactions)     │ GDPR / sovereignty        │ Local R/W         │
    │                    │                           │                   │
    │ GLOBAL-READ        │ Same everywhere           │ Replicated to     │
    │ (product catalog,  │ Written rarely            │ all regions       │
    │  exchange rates,   │ Read from anywhere        │ Write to primary, │
    │  config)           │                           │ read local replica│
    │                    │                           │                   │
    │ GLOBAL-WRITE       │ Written from anywhere     │ Most complex      │
    │ (audit logs,       │ Read from anywhere        │ Careful conflict  │
    │  analytics events) │ Eventual consistency OK   │ resolution needed │
    └─────────────────────────────────────────────────────────────────┘

STEP 2 — DESIGN LOCATION-AWARE SHARD KEYS
    Regional data needs a location field in the shard key:
    → { country: 1, user_id: 1 }
    → OR { region: 1, account_id: 1 }
    → The FIRST field in shard key = the routing field
    → Zones map location values to physical regions

STEP 3 — CONFIGURE ZONE MAPPINGS
    Map country/region codes to physical cloud regions:
    → "US", "CA" → zone "NA" → shards in us-east-1 / us-west-2
    → "DE", "FR", "GB", "ES" → zone "EU" → shards in eu-west-1
    → "JP", "SG", "AU", "IN" → zone "APAC" → shards in ap-southeast-1

STEP 4 — DEPLOY AND VERIFY
    → Atlas handles physical deployment
    → Verify zone assignments
    → Test local read latency from each region
    → Test write routing from each region
    → Verify data sovereignty with compliance audit

STEP 5 — CONFIGURE READ/WRITE CONCERNS FOR GLOBAL
    → Writes: w: "majority" (within local region's replica set)
    → Reads: readPreference: "nearest" (closest replica)
    → Causal consistency for cross-region reads when needed

def configure_zone_ranges():
    """
    Map every country to its zone for every regional collection
    This is the global routing table
    
    No Redshift equivalent — Redshift has no zone concept
    """
    
    print("\n" + "=" * 70)
    print("CONFIGURING ZONE RANGES")
    print("=" * 70)
    
    regional_collections = {
        "fintech.users": "user_id",
        "fintech.accounts": "account_id",
        "fintech.transactions": "timestamp",
        "fintech.notifications": "user_id"
    }
    
    total_ranges = 0
    
    for ns, second_field in regional_collections.items():
        for zone, countries in ZONE_MAP.items():
            for country in countries:
                # Build min/max for this country's range
                # The shard key is { country: 1, <second_field>: 1, ... }
                min_key = {"country": country}
                max_key = {"country": country + "\uffff"}
                
                # Add remaining shard key fields with MinKey/MaxKey
                if "transactions" in ns:
                    min_key["timestamp"] = MinKey()
                    min_key["_id"] = MinKey()
                    max_key["timestamp"] = MaxKey()
                    max_key["_id"] = MaxKey()
                elif "notifications" in ns:
                    min_key[second_field] = MinKey()
                    min_key["_id"] = MinKey()
                    max_key[second_field] = MaxKey()
                    max_key["_id"] = MaxKey()
                else:
                    min_key[second_field] = MinKey()
                    max_key[second_field] = MaxKey()
                
                try:
                    admin_db.command(
                        "updateZoneKeyRange",
                        ns,
                        min=min_key,
                        max=max_key,
                        zone=zone
                    )
                    total_ranges += 1
                except Exception as e:
                    print(f"  ⚠️ Zone range error for {country} in {ns}: {e}")
        
        print(f"  ✅ Zone ranges configured for {ns}")
    
    print(f"\n  Total zone ranges created: {total_ranges}")
    print(f"  Zones: {list(ZONE_MAP.keys())}")
    print(f"  Countries mapped: {sum(len(v) for v in ZONE_MAP.values())}")

        zone_dist = list(config_db.tags.aggregate([
            {"$match": {"ns": ns}},
            {"$group": {"_id": "$tag", "count": {"$sum": 1}}},
            {"$sort": {"_id": 1}}

        print(f"\n  {ns}: {zone_count} zone ranges")

            print(f"    {z['_id']}: {z['count']} country ranges")

    print("\nChunk distribution per shard per zone:")

        if not coll_info:

        pipeline = [
            {"$match": {"uuid": coll_info.get("uuid", coll_info.get("_id"))}},

                "_id": "$shard",

            {"$sort": {"_id": 1}}
        ]
        
        chunks = list(config_db.chunks.aggregate(pipeline))
        print(f"\n  {ns}:")
        for c in chunks:

            zones = shard_info.get("tags", ["untagged"])

def local_read_demo(country, user_id):
    """
    Demonstrate local read — targeted to single shard in correct region
    
    Redshift equivalent:
        -- From EU consumer cluster:
        SELECT * FROM fintech_datashare.public.users
        WHERE country = 'DE' AND user_id = 'USR-EU-001';
        -- BUT: reads cross-region from producer in US (80-120ms)
        
    MongoDB:
        → mongos sees country='DE' in query
        → Routes to EU shard directly
        → Reads from local replica
        → <10ms
    """
    
    zone = COUNTRY_TO_ZONE.get(country, "UNKNOWN")
    print(f"\n--- Local Read: {country} (zone: {zone}) ---")
    
    # Use nearest read preference — reads from closest replica
    read_pref_coll = db.get_collection(
        "users",
        read_preference=ReadPreference.NEAREST,
        read_concern=ReadConcern("local")
    )
    
    start = time.time()
    user = read_pref_coll.find_one(
        {"country": country, "user_id": user_id}
    )
    elapsed_ms = round((time.time() - start) * 1000, 1)
    
    if user:
        print(f"  Found: {user.get('name')} ({user.get('email')})")
        print(f"  Latency: {elapsed_ms}ms")
        print(f"  Expected: <10ms (local shard in {zone} region)")
    else:
        print(f"  User not found: {user_id} in {country}")
    
    # Verify routing with explain
    explain_result = db.command(
        "explain",
        {"find": "users", "filter": {"country": country, "user_id": user_id}},
        verbosity="queryPlanner"
    )
    
    winning_plan = explain_result.get("queryPlanner", {}).get("winningPlan", {})
    targeted_shards = winning_plan.get("shards", [])
    
    if targeted_shards:
        shard_names = [s.get("shardName", "unknown") for s in targeted_shards]
        print(f"  Routed to shard(s): {shard_names}")
        print(f"  Stage: {winning_plan.get('stage', 'unknown')}")
        if winning_plan.get("stage") == "SINGLE_SHARD":
            print(f"  ✅ TARGETED query — single shard (optimal)")
        else:
            print(f"  ⚠️  Multi-shard query (may indicate zone issue)")
    
    return elapsed_ms

def cross_region_transaction_demo():
    """
    International money transfer: DE → US
    Multi-document transaction across regions
    
    Redshift equivalent:
        -- Redshift doesn't support real-time OLTP transactions
        -- This would be an application-level saga pattern
        -- Or a batch ETL operation
        BEGIN;
        UPDATE accounts SET balance = balance - 500 WHERE account_id = 'ACC-EU-001';
        UPDATE accounts SET balance = balance + 460 WHERE account_id = 'ACC-NA-001';
        INSERT INTO transactions (...) VALUES (...);
        INSERT INTO transactions (...) VALUES (...);
        COMMIT;
        -- All in one cluster (single region)
        -- Cross-region: impossible in single Redshift transaction
    """
    
    print("\n" + "=" * 70)
    print("CROSS-REGION TRANSACTION: Germany → United States")
    print("=" * 70)
    
    wc = WriteConcern(w="majority", j=True)
    rc = ReadConcern("snapshot")
    
    session = client.start_session()
    
    start = time.time()
    
    try:
        session.start_transaction(
            read_concern=rc,
            write_concern=wc,
            read_preference=ReadPreference.PRIMARY
        )
        
        session_db = session.client.fintech
        
        # ─── Step 1: Debit German account (EU shard) ──
        debit_result = session_db.accounts.update_one(
            {"country": "DE", "account_id": "ACC-EU-001"},
            {"$inc": {"balance": Decimal128("-500.00")}},

        print(f"    Matched: {debit_result.matched_count}, "
              f"Modified: {debit_result.modified_count}")

            {"country": "US", "account_id": "ACC-NA-001"},

        )
        print(f"  Step 2: Credited ACC-NA-001 (US): +$460.00")
        print(f"    Matched: {credit_result.matched_count}, "
              f"Modified: {credit_result.modified_count}")

            "txn_id": f"TXN-INTL-{int(time.time())}-SEND",
            "account_id": "ACC-EU-001",
            "country": "DE",
            "type": "international_transfer",
            "amount": Decimal128("-500.00"),
            "currency": "EUR",

                "account_id": "ACC-NA-001",

        SELECT country, COUNT(*) as users, SUM(balance) as total_balance
        FROM users u JOIN accounts a ON u.user_id = a.user_id
        GROUP BY country ORDER BY total_balance DESC;
        -- Works within single cluster but cross-region data share is read-only
        -- No JOINs across data shares from different producers
        
    MongoDB:
        → Scatter query to ALL shards in ALL regions
        → Each shard aggregates locally
        → mongos merges results
        → Higher latency (cross-region) but complete picture
    """
    
    print("\n" + "=" * 70)
    print("GLOBAL ADMIN DASHBOARD")
    print("=" * 70)

            "_id": "$country",
            "count": {"$sum": 1}
        }},

            "zone": {
                "$switch": {
                    "branches": [
                        {"case": {"$in": ["$_id", ZONE_MAP["NA"]]},
                         "then": "NA"},
                        {"case": {"$in": ["$_id", ZONE_MAP["EU"]]},
                         "then": "EU"},
                        {"case": {"$in": ["$_id", ZONE_MAP["APAC"]]},

            }

def benchmark_global_latency():
    """
    Measure latency for all operation types across regions
    
    Redshift equivalent:
        -- No equivalent benchmarking — all queries go to one cluster
        -- Cross-region: always 80-250ms via data share
        -- Local: 2-30 seconds (analytical query latency)
    """
    
    print("\n" + "=" * 70)
    print("GLOBAL LATENCY BENCHMARK")
    print("=" * 70)
    
    results = []
    
    # ─── Local reads per region ───────────────────────
    print("\n📖 Local Read Latency:")
    local_tests = [
        ("US", "USR-NA-001", "NA"),
        ("DE", "USR-EU-001", "EU"),
        ("JP", "USR-AP-001", "APAC"),
    ]
    
    for country, user_id, zone in local_tests:
        latencies = []
        for _ in range(5):  # 5 iterations for average
            coll = db.get_collection(
                "users",
                read_preference=ReadPreference.NEAREST
            )
            start = time.time()
            coll.find_one({"country": country, "user_id": user_id})
            latencies.append((time.time() - start) * 1000)
        
        avg = round(sum(latencies) / len(latencies), 1)

def verify_data_sovereignty():
    """
    Verify that data resides in the correct region
    Critical for GDPR, data sovereignty regulations
    
    Redshift equivalent:
        -- Check which cluster has the data
        -- If using separate clusters: trivial (each cluster = one region)
        -- If using data sharing: data is in producer region (NOT local)
        --   → GDPR violation if producer is outside EU
    """
    
    print("\n" + "=" * 70)
    print("DATA SOVEREIGNTY VERIFICATION")
    print("=" * 70)
    
    # ─── Check: EU data on EU shards only ─────────────
    print("\n🇪🇺 GDPR Compliance Check: EU user data location")
    
    eu_countries = ZONE_MAP["EU"]
    
    for ns in ["fintech.users", "fintech.accounts", "fintech.transactions"]:
        coll_name = ns.split(".")[1]
        
        # Get collection UUID
        coll_info = config_db.collections.find_one({"_id": ns})
        if not coll_info:
            print(f"  ⚠️  Collection {ns} not found in config")
            continue
        
        coll_uuid = coll_info.get("uuid")
        
        # Find chunks for EU countries
        # Check which shards hold these chunks
        eu_chunks = list(config_db.chunks.find({
            "uuid": coll_uuid,
            "$or": [
                {"min.country": {"$in": eu_countries}},
            ]
        }, {"shard": 1, "min.country": 1}))
        
        # Verify all EU chunks are on EU-tagged shards
        eu_shards = set()
        violations = []
        
        for chunk in eu_chunks:
            shard_name = chunk["shard"]
            eu_shards.add(shard_name)
            
            # Check shard's zone tags
            shard_info = config_db.shards.find_one({"_id": shard_name})
            shard_tags = shard_info.get("tags", [])
            
            if "EU" not in shard_tags:
                violations.append({
                    "collection": ns,
                    "country": chunk["min"].get("country", "?"),
                    "shard": shard_name,
                    "shard_tags": shard_tags
                })
        
        if violations:
            print(f"\n  ❌ {coll_name}: VIOLATIONS FOUND!")
            for v in violations:
                print(f"    Country {v['country']} on shard {v['shard']} "
                      f"(tags: {v['shard_tags']}) — NOT in EU zone!")
        else:
            print(f"  ✅ {coll_name}: All EU data on EU-tagged shards "
                  f"({eu_shards})")
    
    # ─── Repeat for other regions ─────────────────────
    for zone_name, countries in ZONE_MAP.items():
        if zone_name == "EU":
            continue  # Already checked
        
        print(f"\n🌍 {zone_name} data location check:")
        
        for ns in ["fintech.users", "fintech.accounts"]:
            coll_name = ns.split(".")[1]
            coll_info = config_db.collections.find_one({"_id": ns})
            if not coll_info:
                continue
            
            zone_chunks = list(config_db.chunks.find({
                "uuid": coll_info.get("uuid"),
                "min.country": {"$in": countries}
            }, {"shard": 1}))
            
            zone_shards = set(c["shard"] for c in zone_chunks)
            
            all_correct = True
            for shard_name in zone_shards:
                shard_info = config_db.shards.find_one({"_id": shard_name})
                if zone_name not in shard_info.get("tags", []):
                    all_correct = False
                    print(f"  ❌ {coll_name}: {shard_name} missing "
                          f"{zone_name} tag!")
            
            if all_correct:
                print(f"  ✅ {coll_name}: All {zone_name} data correctly placed")
    
    print("\n" + "-" * 70)
    print("SOVEREIGNTY AUDIT COMPLETE")

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

1. ANALYTICS NODES PER REGION
   → Atlas allows "analytics nodes" in each region
   → Isolated replica members that handle only analytics workloads
   → Prevents heavy aggregations from impacting operational traffic
   → readPreference: "secondary" with tag: { nodeType: "ANALYTICS" }

2. LOCAL READ PREFERENCE WITH MAXSTALENESS
   → For non-critical reads: allow slightly stale data for lower latency
   → readPreference: "nearest", maxStalenessSeconds: 120
   → Reads from closest node even if 2 minutes behind primary
   → Perfect for dashboards, activity feeds, notifications

3. PRE-COMPUTED REGIONAL SUMMARIES
   → Instead of scatter-gather for global dashboard:
   → Each region computes local summary → stored in global_stats collection
   → Dashboard reads pre-computed summaries → avoids scatter-gather
   → Uses change streams (Project 25) to trigger recomputation

4. COMPOUND LOCATION FIELD
   → Instead of country as routing field:
   → { location: { country: "DE", region: "EU", city: "Berlin" } }
   → Shard key: { "location.region": 1, user_id: 1 }
   → Fewer zone ranges (3 regions vs 50+ countries)
   → Simpler zone configuration

5. READ-ONLY SECONDARY IN EVERY REGION
   → Even if shard only serves NA data:
   → Place read-only secondary in EU and APAC
   → Cross-region reads hit local secondary (faster)
   → Trade-off: replication lag for cross-region reads

✅ WHAT SHOULD DO

SHOULD DO:

    ✅ USE ATLAS GLOBAL CLUSTERS (not self-managed)
       → Atlas manages cross-region replica placement
       → Automatic DNS routing to nearest mongos
       → Built-in monitoring per region
       → Self-managed global sharding is extremely complex

    ✅ DESIGN LOCATION FIELD CAREFULLY
       → This field is IMMUTABLE in practice (part of shard key)
       → Use ISO country codes (standard, unambiguous)
       → Consider: what happens when user moves countries?
       → Answer: create new user in new region, migrate data
       → DO NOT update country field (requires chunk migration)

    ✅ MINIMIZE CROSS-REGION TRANSACTIONS
       → Intra-region transaction: <20ms (fast)
       → Cross-region transaction: 200-500ms (slow, fragile)
       → Design data model so most operations are regional
       → Cross-region transfers: use async saga pattern when possible

    ✅ USE CAUSAL CONSISTENCY FOR USER SESSIONS
       → User writes then reads → must see own write
       → session.causalConsistency = true
       → Works even when read hits a different replica than write

    ✅ SET UP ALERTS PER REGION
       → Replication lag between regions
       → Per-region CPU, memory, IOPS
       → Cross-region network latency
       → Election events (indicates failover)

    ✅ TEST REGIONAL FAILOVER REGULARLY
       → Atlas allows "Test Failover" button
       → Verify: election time, application reconnection, data integrity
       → Document: which region's users are affected, for how long

❌ WHAT SHOULD NOT DO

DO NOT:

    ❌ SCATTER-GATHER FOR EVERY QUERY
       → Every query MUST include the country/region field
       → Without it: mongos sends to ALL shards in ALL regions
       → db.users.find({ email: "hans@example.de" })  ← SCATTER-GATHER
       → db.users.find({ country: "DE", email: "hans@example.de" })  ← TARGETED
       → Redshift trap: you're used to full-table scans being fine
         MongoDB global: full-cluster scan crosses CONTINENTS

    ❌ USE AUTO-INCREMENT OR SEQUENCE-BASED IDs GLOBALLY
       → Sequences require coordination across regions
       → ObjectId is globally unique without coordination
       → Or use UUID v4 / ULID for custom IDs

    ❌ ASSUME CROSS-REGION READS ARE FAST
       → Physics: speed of light US→EU ≈ 40ms one-way
       → Round trip: 80-120ms minimum
       → With processing: 100-200ms realistic
       → Design to avoid cross-region reads for user-facing operations

    ❌ PUT ALL MONGOS IN ONE REGION
       → Each region needs its own mongos instances
       → Atlas handles this automatically
       → Self-managed: deploy mongos in EVERY region

    ❌ CHANGE COUNTRY FIELD AFTER DOCUMENT CREATION
       → Country is part of shard key → effectively immutable
       → Changing it means: delete from old shard + insert on new shard
       → Must be done as application-level migration, not simple update
       → Redshift trap: you're used to UPDATE any column freely

    ❌ IGNORE REPLICATION LAG IN GLOBAL DEPLOYMENT
       → Cross-region replication: 50-200ms lag (normal)
       → Reading from secondary in remote region: may see stale data
       → For financial data: always read from primary (strong consistency)
       → For notifications: nearest secondary is fine (eventual consistency)

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 41:

PITFALL 1: "I'll just deploy separate MongoDB clusters per region"
    ❌ Three independent MongoDB clusters (mimicking Redshift approach)
    ✅ ONE Atlas Global Cluster with zone sharding
    WHY: Separate clusters = no unified query layer
         No cross-region transactions possible
         No automatic failover across regions
         Triple operational overhead
         Atlas Global Clusters solve all of this natively

PITFALL 2: "Cross-region reads will be fast because MongoDB is fast"
    ❌ Expecting <10ms for ALL reads regardless of data location
    ✅ Local reads: <10ms | Cross-region reads: 80-250ms
    WHY: Physics — speed of light is the bottleneck
         Same latency as Redshift cross-region data shares
         Design to keep reads LOCAL (zone-aware shard key)

PITFALL 3: "I don't need the country field in every document"
    ❌ Relying on application-level routing instead of shard key
    ✅ Country/region MUST be in the shard key for zone routing
    WHY: Without location in shard key, mongos can't route to correct zone
         Every query becomes scatter-gather across ALL regions
         Redshift parallel: like having no DISTKEY (EVEN distribution)
         But worse — EVEN across continents instead of across local nodes

PITFALL 4: "Global transactions are just like local transactions"
    ❌ Using multi-document transactions freely across regions
    ✅ Minimize cross-region transactions — design around them
    WHY: Cross-region two-phase commit: 200-500ms per transaction
         Network partition between regions → transaction timeout
         Redshift doesn't even support cross-cluster transactions
         MongoDB does but at significant latency cost

PITFALL 5: "I'll handle user migration between countries with an update"
    ❌ db.users.updateOne({ user_id: "X" }, { $set: { country: "US" } })
    ✅ Application-level migration: delete from old region, insert in new
    WHY: Shard key fields cannot be updated in-place
         Update would fail or cause orphaned document
         Must be explicit: delete old + insert new + migrate related data
         This is a BUSINESS PROCESS, not a database operation

PITFALL 6: "The global cluster handles everything automatically"
    ❌ Deploy and forget
    ✅ Monitor per-region: replication lag, latency, failover events
    WHY: Cross-region replication can fall behind during traffic spikes
         Regional outages require verification after recovery
         Zone violations can occur during resharding (Project 40)
         Atlas provides tools — but you must USE them

    ┌──────────────────────────────────────────────────────────┐
    │ 1. Client resolves SRV record:                            │
    │    _mongodb._tcp.fintech-global.xxxxx.mongodb.net         │
    │                                                           │
    │ 2. DNS returns MULTIPLE mongos endpoints:                 │
    │    mongos-us-east.xxxxx.mongodb.net:27017   (US)         │
    │    mongos-eu-west.xxxxx.mongodb.net:27017   (EU)         │
    │    mongos-ap-se.xxxxx.mongodb.net:27017     (APAC)       │
    │                                                           │
    │ 3. MongoDB driver measures latency to each:               │
    │    → From EU app server:                                  │
    │      mongos-eu-west: 2ms    ← NEAREST                    │
    │      mongos-us-east: 85ms                                 │
    │      mongos-ap-se: 180ms                                  │
    │                                                           │
    │ 4. Driver connects to mongos-eu-west (nearest)            │
    │                                                           │

Cross-Region Write Path (Transaction Internals)

CROSS-REGION TRANSACTION: DE user sends money to US user

    TIMELINE (two-phase commit across regions):

    T=0ms    Application (EU) → mongos-eu → "start transaction"
    
    T=5ms    mongos-eu → shard-eu-1: 
             updateOne({ country: "DE", account_id: "ACC-EU-001" },
                       { $inc: { balance: -500 } })
             → shard-eu-1 primary LOCKS document
             → Write to EU oplog
             → Acknowledged to mongos
    
    T=10ms   mongos-eu → shard-na-1: (CROSS-REGION: EU → US)
             updateOne({ country: "US", account_id: "ACC-NA-001" },
                       { $inc: { balance: +460 } })
             → Network transit EU → US: ~80ms
    
    T=90ms   shard-na-1 primary receives request
             → LOCKS document
             → Write to US oplog
             → Acknowledged to mongos-eu
    
    T=170ms  mongos-eu receives acknowledgment from shard-na-1
             → Network transit US → EU: ~80ms
    
    T=175ms  mongos-eu → insert sender transaction doc (EU shard)
             → shard-eu-1: ~5ms (local)
    
    T=180ms  mongos-eu → insert receiver transaction doc (US shard)
             → shard-na-1: ~160ms (cross-region round trip)
    
    T=340ms  ALL operations acknowledged
             mongos-eu initiates TWO-PHASE COMMIT:
    
    T=345ms  PHASE 1 (PREPARE):
             → mongos-eu → shard-eu-1: "prepare commit"
             → mongos-eu → shard-na-1: "prepare commit" (cross-region)
             → Both shards write prepare record to oplog
             → Both respond "prepared"
    
    T=430ms  PHASE 2 (COMMIT):
             → mongos-eu → shard-eu-1: "commit" (local: 5ms)
             → mongos-eu → shard-na-1: "commit" (cross-region: 160ms)
             → Both shards apply changes permanently
             → Both release document locks
             → Both replicate to secondaries
    
    T=600ms  Transaction fully committed
             → Total: ~600ms
             → Of which: ~480ms is cross-region network latency
             → Actual processing: ~120ms
             → PHYSICS IS THE BOTTLENECK, not MongoDB

    COMPARE WITH LOCAL (intra-EU) TRANSACTION:
    → DE user sends money to FR user
    → Both accounts on EU shard(s)
    → Total: ~15-25ms (all network hops within EU)
    → 25x FASTER than cross-region

    COMPARE WITH REDSHIFT:
    → Redshift: single cluster, all data local
    → BEGIN; UPDATE; UPDATE; INSERT; INSERT; COMMIT;
    → ~50-200ms (analytical overhead, not network)
    → BUT: Redshift can't serve this as OLTP (wrong workload)
    → AND: Redshift has no multi-region write capability
