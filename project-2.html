🔴 REDSHIFT MINDSET — How You Would Solve This

Step 1: Prepare the Table (already done — schema exists)

-- REDSHIFT: Table must exist BEFORE loading data
-- (You learned in Project 1 — schema first, data second)

CREATE TABLE store.customers (
    customer_id     BIGINT IDENTITY(1,1),
    name            VARCHAR(200) NOT NULL,
    email           VARCHAR(255) NOT NULL,
    age             SMALLINT,
    city            VARCHAR(100),
    state           VARCHAR(2),
    signup_date     DATE,

    PRIMARY KEY (customer_id)
)
DISTSTYLE KEY
DISTKEY(state)              -- distribute by state for regional analytics
SORTKEY(signup_date);       -- sort by date for time-range queries

Step 2: Upload CSV to S3

REDSHIFT BULK LOADING WORKFLOW:
    1. Export/generate CSV file
    2. Upload to S3 bucket: s3://my-bucket/data/customers.csv
    3. Ensure Redshift cluster has IAM role with S3 read access
    4. Run COPY command
    
    YOU CANNOT LOAD FILES DIRECTLY FROM LOCAL DISK
    → Must go through S3 (or DynamoDB, or EMR, or SSH)
    → This is a common friction point

Step 3: Run COPY Command

-- REDSHIFT: The primary bulk loading mechanism
COPY store.customers (name, email, age, city, state, signup_date, total_spent, is_premium)
FROM 's3://my-bucket/data/customers.csv'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftS3Access'
CSV
IGNOREHEADER 1
DATEFORMAT 'YYYY-MM-DD'
MAXERROR 10                  -- tolerate up to 10 bad rows
COMPUPDATE ON                -- auto-apply compression
STATUPDATE ON                -- auto-update statistics
;

-- Check for load errors
SELECT * FROM stl_load_errors ORDER BY starttime DESC LIMIT 10;

-- Verify
SELECT COUNT(*) FROM store.customers;  -- should be ~1000

What Redshift Did Behind the Scenes

COPY COMMAND INTERNALS:
    1. Leader node parsed the COPY command
    2. Leader identified the S3 files
    3. Files were SPLIT across compute slices (parallel read)
    4. Each slice read its portion and parsed CSV rows
    5. Data type validation per column per row
    6. Rows distributed across slices based on DISTKEY(state)
    7. Within each slice, data sorted by SORTKEY(signup_date)
    8. Columnar encoding applied per column
    9. Zone maps updated with min/max per 1MB block
    10. Statistics updated for query optimizer
    
    COPY IS THE FASTEST WAY TO LOAD DATA INTO REDSHIFT
    → Single-row INSERT is 100x slower
    → COPY parallelizes across all nodes
    → This is why "COPY from S3" is Redshift's golden rule

🟢 MONGODB MINDSET — How This Same Problem Is Approached

The Fundamental Difference

IN REDSHIFT:
    → Bulk loading is a SPECIAL operation (COPY)
    → Different from single INSERT
    → Requires external staging (S3)
    → Optimized for large batches (thousands to billions)
    → Single INSERT is discouraged (slow, generates lots of WAL)

IN MONGODB:
    → Bulk loading uses the SAME operations as single inserts
    → insertMany() = insertOne() called efficiently in batch
    → No external staging required (load from anywhere)
    → ALSO has mongoimport for file-based loading
    → Single insert is perfectly normal (designed for OLTP)
    
    THREE WAYS TO BULK LOAD:
    1. insertMany() — programmatic (pymongo/mongosh)
    2. mongoimport — command-line tool (CSV, JSON, TSV files)
    3. MongoDB Connector for Spark — big data pipelines

🧩 HOW TO APPROACH IT IN MONGODB

THINKING FRAMEWORK FOR BULK LOADING:

STEP 1: WHAT DOES ONE CUSTOMER DOCUMENT LOOK LIKE?
    → Not "what columns does the table have"
    → But "what does ONE customer look like as a JSON object?"
    {
        "name": "Alice Johnson",
        "email": "alice@example.com",
        "age": 34,
        "address": {              ← NESTED (city+state become embedded object)
            "city": "Austin",
            "state": "TX"
        },
        "signup_date": ISODate(),
        "total_spent": 1250.75,
        "is_premium": true
    }

STEP 2: HOW IS THE DATA ARRIVING?
    → CSV file from marketing? → mongoimport
    → API payload from another service? → insertOne() or insertMany()
    → Streaming from Kafka? → insertOne() in consumer loop
    → Migration from Redshift? → Export CSV → mongoimport

STEP 3: DO I NEED ERROR TOLERANCE?
    → Some records might have bad data
    → ordered: false → continue inserting despite errors
    → ordered: true (default) → stop at first error

STEP 4: DO I NEED ATOMICITY?
    → All 1000 must succeed or none? → Use transaction (Project 28)
    → Best-effort loading? → insertMany with ordered: false

NOTICE THE DIFFERENCE:
    Redshift: "How do I get data into S3 in the right format?"
    MongoDB:  "What does the document look like and how is data arriving?"

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

OPTIMIZATION 1: WRITE CONCERN TUNING FOR BULK LOADS
    → Default: w:1 (acknowledged by primary)
    → Faster:  w:0 (fire-and-forget — no acknowledgment)
    → Safer:   w:"majority" (acknowledged by majority of replica set)
    
    # mongosh
    db.customers.insertMany(docs, { writeConcern: { w: 0 } })
    
    # pymongo
    customers.with_options(
        write_concern=WriteConcern(w=0)
    ).insert_many(docs)
    
    TRADE-OFF:
    → w:0 is ~2x faster but you don't know if writes succeeded
    → Use for non-critical bulk loads (log data, analytics events)
    → Redshift equivalent: none — COPY always confirms

OPTIMIZATION 2: UNORDERED INSERTS FOR PARALLELISM
    → ordered=False allows MongoDB driver to batch writes internally
    → Driver can send multiple batches to server in parallel
    → ~20-30% faster than ordered=True for large insertMany()
    → Redshift equivalent: COPY automatically parallelizes

OPTIMIZATION 3: CHUNK LARGE INSERTS
    → MongoDB has a 48MB BSON document size limit per COMMAND
    → insertMany with 100K large documents may exceed this
    → Solution: chunk into batches of 1000-5000
    
    for i in range(0, len(docs), 1000):
        customers.insert_many(docs[i:i+1000], ordered=False)
    
    → Redshift equivalent: split large files into multiple S3 objects
      (COPY processes multiple files in parallel)

OPTIMIZATION 4: DISABLE INDEXES DURING BULK LOAD
    → If collection has many indexes, each insert updates ALL indexes
    → For initial bulk load: insert first, create indexes after
    → Redshift equivalent: COPY with COMPUPDATE OFF (defer compression)
    
    # Load data first
    customers.insert_many(huge_dataset, ordered=False)
    # Create indexes after
    customers.create_index({"email": 1}, unique=True)
    customers.create_index({"address.state": 1})

❌ WHAT SHOULD NOT DO

ANTI-PATTERNS — ESPECIALLY TRAPS FOR REDSHIFT USERS:

    ❌ LOOPING insertOne() FOR BULK DATA
       # WRONG — Redshift developer instinct: "one INSERT per row"
       for row in csv_reader:
           db.customers.insert_one(row)
       
       # RIGHT — batch them
       docs = list(csv_reader)
       db.customers.insert_many(docs, ordered=False)
       
       WHY: Each insertOne() = separate network round trip
            insertMany() = one round trip for entire batch
            Same reason Redshift COPY >> individual INSERTs

    ❌ STORING NUMBERS AS STRINGS
       # WRONG (from CSV import without type handling)
       { "age": "34", "total_spent": "1250.75" }
       
       # RIGHT
       { "age": 34, "total_spent": 1250.75 }
       
       WHY: mongoimport from CSV stores EVERYTHING as strings by default!
            Must use --columnsHaveTypes or post-process to fix types
            Redshift enforces types via DDL — MongoDB doesn't
            THIS IS THE #1 DATA QUALITY TRAP IN MONGODB

    ❌ FLATTENING NESTED DATA TO MATCH REDSHIFT THINKING
       # WRONG — "I'll store city and state as top-level fields like Redshift columns"
       { "name": "Alice", "city": "Austin", "state": "TX" }
       
       # RIGHT — group related fields as nested objects
       { "name": "Alice", "address": { "city": "Austin", "state": "TX" } }
       
       WHY: Nested objects group related data logically
            Easier to update entire address at once
            Easier to add zip, country later (just add to nested object)
            Redshift can't nest — MongoDB should when it makes sense

    ❌ INSERTING MILLIONS WITHOUT MONITORING MEMORY
       → WiredTiger cache has limits
       → Inserting faster than WiredTiger can flush = backpressure
       → Monitor: db.serverStatus().wiredTiger.cache
       → Add small delays between batches if cache pressure is high
       → Redshift equivalent: WLM queue full during heavy COPY loads

    ❌ USING mongoimport FOR PRODUCTION DATA PIPELINES
       → mongoimport is great for one-time loads and development
       → For production: use application-level inserts (pymongo)
       → Or use MongoDB Connector for Spark/Kafka
       → Redshift equivalent: COPY is production-grade; mongoimport is not

    ❌ IGNORING THE _id FIELD
       → Every inserted document gets _id (ObjectId) automatically
       → If your source data has a natural unique key (email, SKU)
         consider: { "_id": "alice@example.com", ... }
       → Saves a separate unique index
       → But be CAREFUL: _id is immutable after insert

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Write Path — What Happens During insertMany()

WHEN YOU RUN: db.customers.insertMany([doc1, doc2, ... doc1000])

STEP 1: DRIVER-LEVEL BATCHING (pymongo / mongosh)
    → Driver splits 1000 documents into batches
    → Each batch must be < 48MB (BSON command size limit)
    → Each batch max 100,000 documents (server limit)
    → For 1000 small docs: usually 1 batch

STEP 2: WIRE PROTOCOL
    → Driver sends OP_MSG to mongod server
    → Contains: database name, collection name, array of BSON documents
    → Each document individually BSON-encoded
    → Sent over TCP connection (persistent — connection pooling)

    REDSHIFT COMPARISON:
    → COPY sends S3 path → compute nodes read from S3 directly
    → MongoDB: driver sends actual data over network to mongod
    → For large loads: Redshift COPY is more efficient (S3 parallel read)
    → For small-medium loads: MongoDB is faster (no S3 staging)

STEP 3: MONGOD SERVER RECEIVES DOCUMENTS
    → Parse BSON documents
    → For EACH document:
        a. Generate _id (ObjectId) if not provided
        b. Validate against schema (if validator exists)
        c. Check unique index constraints (_id uniqueness)
        d. Pass to WiredTiger storage engine

STEP 4: WIREDTIGER WRITE PATH
    → Document written to IN-MEMORY buffer (WiredTiger cache)
    → Journal entry written to disk (WAL — for crash recovery)
    → Journal write = fsync to disk (durable)
    → Actual data file update happens LATER (checkpoint every 60s)
    
    WRITE FLOW:
    ┌────────────┐     ┌──────────────┐     ┌─────────────┐
    │ Application │ ──→ │ WT Cache      │ ──→ │ Data Files   │
    │ (insert)    │     │ (in memory)   │     │ (on disk)    │
    └────────────┘     └──────────────┘     └─────────────┘
                              │
                              ▼
                       ┌──────────────┐
                       │ Journal       │
                       │ (on disk,     │
                       │  immediate)   │
                       └──────────────┘
    
    REDSHIFT COMPARISON:
    → Redshift: data written to columnar blocks on managed storage
    → Each column stored separately → great for column scans
    → MongoDB: entire document stored as unit → great for document retrieval
    → Redshift: commit log (WAL equivalent) → MongoDB: journal

STEP 5: INDEX UPDATES
    → For EACH document inserted:
      → _id index (B-tree) updated with new ObjectId → pointer to document
      → Any other indexes on the collection also updated
    → This is WHY inserts get slower with more indexes
    → Each index = additional B-tree update per document
    
    REDSHIFT COMPARISON:
    → Redshift: zone maps updated per block (very cheap — just min/max)
    → MongoDB: B-tree index updated per document (more expensive)
    → Redshift has NO traditional indexes — zone maps are automatic metadata
    → MongoDB indexes are explicit B-trees you create and maintain

STEP 6: WRITE ACKNOWLEDGMENT
    → Server sends acknowledgment back to driver
    → w:1 → acknowledged after primary writes to journal
    → w:"majority" → acknowledged after majority of replica set journals
    → w:0 → acknowledged immediately (no waiting)
    
    REDSHIFT COMPARISON:
    → COPY always waits for full commit (no tunable write concern)
    → MongoDB gives you CHOICE of durability vs speed

The CSV-to-Document Transformation Problem

CRITICAL INSIGHT FOR REDSHIFT USERS:

CSV IN REDSHIFT:
    name,email,age,city,state,total_spent,is_premium
    Alice,alice@ex.com,34,Austin,TX,1250.75,true
    
    → COPY reads CSV → maps to columns → types enforced by DDL
    → age "34" → INTEGER 34 (automatic)
    → total_spent "1250.75" → DECIMAL(12,2) 1250.75 (automatic)
    → is_premium "true" → BOOLEAN true (automatic)
    → Type conversion guaranteed by table definition

CSV IN MONGODB (mongoimport):
    name,email,age,city,state,total_spent,is_premium
    Alice,alice@ex.com,34,Austin,TX,1250.75,true
    
    → mongoimport reads CSV → creates documents
    → DEFAULT BEHAVIOR: EVERYTHING IS A STRING!
    → age: "34"          ← STRING, not number!
    → total_spent: "1250.75"  ← STRING, not number!
    → is_premium: "true"      ← STRING, not boolean!
    
    → THIS IS A MAJOR TRAP FOR REDSHIFT USERS
    → You assume types will be inferred — THEY WON'T (from CSV)

SOLUTION 1: Use --columnsHaveTypes flag
    mongoimport --type=csv --headerline \
        --columnsHaveTypes \
        --fields="name.string(),email.string(),age.int32(),
                  city.string(),state.string(),
                  total_spent.double(),is_premium.boolean()" \
        --file=customers.csv

SOLUTION 2: Use JSON format instead of CSV
    → JSON preserves types naturally:
    {"name":"Alice","age":34,"total_spent":1250.75,"is_premium":true}
    → mongoimport with JSON → types preserved correctly

SOLUTION 3: Load via pymongo (application code)
    → You control types explicitly in Python:
    {"age": int(row["age"]), "total_spent": float(row["total_spent"])}

RECOMMENDATION: 
    → Always prefer JSON over CSV for MongoDB imports
    → Or use pymongo with explicit type casting
    → NEVER trust CSV import types without --columnsHaveTypes

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 2:

PITFALL 1: "CSV types will be inferred"
    REDSHIFT TRAINED YOU: COPY maps CSV columns to DDL types automatically
    MONGODB REALITY: mongoimport from CSV → everything is STRING by default
    FIX: Use JSON format, or --columnsHaveTypes, or pymongo with casting
    SEVERITY: HIGH — silently breaks all numeric queries and sorting

PITFALL 2: "I'll create the collection and indexes before loading"
    REDSHIFT TRAINED YOU: CREATE TABLE (with sort key) → then COPY
    MONGODB REALITY: Creating indexes before bulk load = slower inserts
    FIX: Insert data FIRST, create indexes AFTER for initial loads
         (indexes must be updated per document during insert)
    SEVERITY: MEDIUM — 2-5x slower bulk loads with many indexes

PITFALL 3: "ordered=True is fine" (default)
    REDSHIFT TRAINED YOU: COPY either succeeds or fails (then check stl_load_errors)
    MONGODB REALITY: ordered=True stops at FIRST error
                     1000 documents, error at #500 = only 500 inserted
    FIX: Use ordered=False for bulk loads (continue past errors)
    SEVERITY: HIGH — silently loses data after first error

PITFALL 4: "I'll store everything in separate collections like Redshift tables"
    REDSHIFT TRAINED YOU: customers table, customer_preferences table,
                          customer_referrals table → JOIN them
    MONGODB REALITY: Embed preferences and referral data IN the customer document
    FIX: Think "what does one customer look like?" not "what tables do I need?"
    SEVERITY: MEDIUM — works but defeats MongoDB's document model advantages

PITFALL 5: "null and missing field are the same thing"
    REDSHIFT: NULL is NULL — there's only one concept
    MONGODB: TWO different things:
        { "phone": null }    ← field EXISTS with null value
        { }                  ← field DOES NOT EXIST
        
        db.customers.find({ phone: null })     → matches BOTH
        db.customers.find({ phone: { $exists: false } }) → matches only missing
        db.customers.find({ phone: { $type: "null" } })  → matches only explicit null
    
    FIX: Be intentional about null vs absent
         Absent = customer never provided phone
         null = customer explicitly declined to provide phone
    SEVERITY: HIGH — incorrect query results if confused

PITFALL 6: "48MB command size limit"
    REDSHIFT: COPY has no practical row limit (processes S3 files of any size)
    MONGODB: insertMany() has 48MB BSON command size limit
             100,000 document limit per command
    FIX: Chunk large inserts into batches of 1000-5000
    SEVERITY: MEDIUM — causes cryptic errors with very large batches

def parse_customer_row(row):
    """
    Convert CSV string values to proper MongoDB types.
    
    THIS FUNCTION IS THE EQUIVALENT OF REDSHIFT'S DDL TYPE ENFORCEMENT.
    In Redshift, the table definition handles this automatically.
    In MongoDB, YOU must handle it in your application code.
    """
    doc = {
        "name": row["name"],                                    # string → string (ok)
        "email": row["email"],                                  # string → string (ok)
        "age": int(row["age"]),                                 # string → int
        "address": {                                            # FLATTEN → NEST
            "city": row["city"],                                # (MongoDB style)
            "state": row["state"]
        },
        "signup_date": datetime.strptime(                       # string → datetime
            row["signup_date"], "%Y-%m-%d"
        ),
        "total_spent": float(row["total_spent"]),               # string → float
        "is_premium": row["is_premium"].lower() == "true"       # string → boolean
    }
    return doc

📊 WRITE CONCERN DEEP DIVE — A Concept Redshift Doesn't Have

WRITE CONCERN = How many replicas must confirm a write before it's "done"

THIS CONCEPT DOES NOT EXIST IN REDSHIFT:
    → Redshift COPY/INSERT always fully commits to managed storage
    → No knob to trade durability for speed
    → Every write is fully durable, period

IN MONGODB — YOU CHOOSE PER OPERATION:

┌────────────────────┬────────────────────┬─────────────────────┬───────────────┐
│ WRITE CONCERN       │ WHAT IT MEANS       │ WHEN TO USE          │ SPEED         │
├────────────────────┼────────────────────┼─────────────────────┼───────────────┤
│ w: 0                │ Don't wait for      │ Log data, metrics    │ ★★★★★ Fastest │
│ (fire-and-forget)   │ any confirmation    │ Data you can afford  │               │
│                     │                     │ to lose              │               │
├────────────────────┼────────────────────┼─────────────────────┼───────────────┤
│ w: 1                │ Primary confirms    │ Most operations      │ ★★★★ Fast     │
│ (DEFAULT)           │ write to memory     │ Default behavior     │               │
│                     │ + journal           │ Good balance         │               │
├────────────────────┼────────────────────┼─────────────────────┼───────────────┤
│ w: "majority"       │ Majority of replica │ Financial data       │ ★★★ Medium    │
│                     │ set confirms write  │ Critical operations  │               │
│                     │                     │ Must survive failover│               │
├────────────────────┼────────────────────┼─────────────────────┼───────────────┤
│ w: "majority" +     │ Majority confirms   │ Audit logs           │ ★★ Slower     │
│ j: true             │ AND journaled to    │ Compliance data      │               │
│                     │ disk on each        │ Maximum durability   │               │
└────────────────────┴────────────────────┴─────────────────────┴───────────────┘

BSON TYPE MAPPING FROM REDSHIFT SQL TYPES:
(You'll use these every time you insert data)

┌──────────────────────┬──────────────────────┬──────────────────────────────┐
│ REDSHIFT TYPE         │ BSON TYPE             │ PYMONGO / MONGOSH             │
├──────────────────────┼──────────────────────┼──────────────────────────────┤
│ SMALLINT              │ Int32                 │ int (Python)                  │
│ INTEGER               │ Int32                 │ int (Python)                  │
│ BIGINT                │ Int64                 │ Int64(val) in mongosh         │
│                       │                       │ int (Python auto-handles)     │
├──────────────────────┼──────────────────────┼──────────────────────────────┤
│ REAL / FLOAT4         │ Double                │ float (Python)                │
│ DOUBLE / FLOAT8       │ Double                │ float (Python)                │
│ DECIMAL(p,s)          │ Decimal128            │ Decimal128("29.99")           │
│                       │                       │ from bson.decimal128 import   │
├──────────────────────┼──────────────────────┼──────────────────────────────┤
│ VARCHAR(n)            │ String                │ str (Python)                  │
│ CHAR(n)               │ String                │ str (no fixed length)         │
│ TEXT                   │ String                │ str (no size limit*)          │
├──────────────────────┼──────────────────────┼──────────────────────────────┤
│ BOOLEAN               │ Boolean               │ True / False (Python)         │
│                       │                       │ true / false (mongosh)        │
├──────────────────────┼──────────────────────┼──────────────────────────────┤
│ DATE                  │ Date                  │ datetime.date → datetime      │
│ TIMESTAMP             │ Date                  │ datetime.datetime (Python)    │
│ TIMESTAMPTZ           │ Date (always UTC      │ ISODate() in mongosh          │
│                       │  internally)          │ datetime (Python, UTC aware)  │
├──────────────────────┼──────────────────────┼──────────────────────────────┤
│ VARBYTE               │ BinData               │ bytes (Python) → Binary       │

CRITICAL DIFFERENCE — DECIMAL PRECISION:
    Redshift:  DECIMAL(38, 10) — exact decimal arithmetic
    MongoDB:   Decimal128 — IEEE 754 decimal128 (34 significant digits)
    
    FOR FINANCIAL DATA:
    # WRONG — float has rounding errors
    { "price": 29.99 }           # ← stored as Double (IEEE 754 binary)
                                  #    0.1 + 0.2 ≠ 0.3 in binary float!
    
    # RIGHT — exact decimal
    { "price": Decimal128("29.99") }  # ← stored as Decimal128 (exact)
    
    # pymongo
    from bson.decimal128 import Decimal128
    doc = {"price": Decimal128("29.99")}
    
    # mongosh
    doc = { price: NumberDecimal("29.99") }
    
    REDSHIFT USERS: you're used to DECIMAL(10,2) handling this automatically
    MONGODB USERS: you must EXPLICITLY use Decimal128 for financial precision
