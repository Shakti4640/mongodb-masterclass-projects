PROJECT 4: Updating Documents vs UPDATE Statements

Problem: A Customer Changes Their Shipping Address and Loyalty Tier Upgrades

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform is live with 1000 customers (Projects 1-3)
    Several things need to happen:
    
    1. Alice moved from Austin to Denver — update her address
    2. Bob's spending crossed $1000 — upgrade to premium
    3. Customer makes a purchase — increment total_spent by $149.99
    4. Customer adds "Sports" to preferred categories
    5. Customer removes "Clothing" from preferred categories
    6. Customer opts out of SMS — remove the sms_alerts field entirely
    7. New visitor makes a purchase — create customer if doesn't exist (upsert)
    8. All customers in Texas get a loyalty bonus — add $50 to total_spent
    
    In Redshift: UPDATE table SET column = value WHERE condition
    Every update REPLACES the entire row internally
    
    In MongoDB: Surgical, atomic, field-level operators
    You modify EXACTLY what changed — nothing else
    THIS is where MongoDB's operational nature truly shines

🔴 REDSHIFT MINDSET — How You Would Solve This

All 8 Updates in Redshift SQL

WHERE email = 'david@example.com';

-- Option B: SUPER type (better but still awkward)
-- UPDATE store.customers
-- SET preferences = JSON_PARSE(
--     JSON_SERIALIZE(preferences) ... complex manipulation

-- WHERE email = 'david@example.com';
-- → EXTREMELY PAINFUL in Redshift

-- ─── UPDATE 5: Remove category from preferences ─────
-- Even worse than adding — string/JSON manipulation hell
-- No clean way in Redshift

WHERE state = 'TX';

What Redshift Does Behind the Scenes for UPDATE

CRITICAL INSIGHT ABOUT REDSHIFT UPDATES:

    Redshift is COLUMNAR and APPEND-ONLY
    
    When you run: UPDATE customers SET city = 'Denver' WHERE email = 'alice@...'
    
    1. The OLD row is MARKED AS DELETED (soft delete)
    2. A completely NEW row is WRITTEN with all columns
       → Even columns that DIDN'T change are rewritten
       → city changed from 'Austin' to 'Denver'
       → But name, email, age, state, etc. are ALL rewritten too
    3. The old row's disk space is NOT reclaimed immediately
    4. You must run VACUUM to reclaim deleted row space
    5. You must run ANALYZE to update statistics
    
    THEREFORE:
    → UPDATE in Redshift = DELETE old row + INSERT new row
    → This is EXPENSIVE for frequent updates
    → This is WHY Redshift is bad for OLTP (many small updates)
    → This is WHY MongoDB exists (designed for frequent updates)
    
    COST OF ONE UPDATE IN REDSHIFT:
    → Read all columns of the row
    → Write all columns of the new row
    → Mark old row as deleted
    → Zone maps may need updating
    → Sort order may be violated (requires VACUUM SORT)
    → Statistics become stale (requires ANALYZE)
    
    COST OF ONE UPDATE IN MONGODB:
    → Read the document
    → Modify ONLY the changed field(s) in place (when possible)
    → Write the modified document
    → Update affected indexes only
    → No vacuum needed, no analyze needed

🟢 MONGODB MINDSET — How This Same Problem Is Approached

The Fundamental Difference

REDSHIFT UPDATE MODEL:
    UPDATE table SET column = value WHERE condition
    → REPLACES entire row (all columns rewritten)
    → ONE syntax for all modifications
    → Cannot add/remove columns per row
    → Cannot manipulate arrays natively
    → Row-level granularity only

MONGODB UPDATE MODEL:
    db.collection.updateOne( filter, update_operators )
    → MODIFIES specific fields (surgical precision)
    → MANY specialized operators for different modifications
    → Can add new fields ($set)
    → Can remove fields ($unset)
    → Can increment numbers ($inc)
    → Can manipulate arrays ($push, $pull, $addToSet)
    → Field-level granularity (sub-document level even)

THE KEY INSIGHT:
    Redshift: "Replace the row with a new version"
    MongoDB:  "Modify exactly what changed inside the document"
    
    This is like:
    Redshift: rewriting an entire book page because one word changed
    MongoDB:  erasing one word and writing the new one

📊 SIDE-BY-SIDE COMPARISON TABLE

┌──────────────────────┬─────────────────────────────┬──────────────────────────────────┐
│ OPERATION             │ REDSHIFT                     │ MONGODB                           │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Change a field value  │ UPDATE SET col = 'new'       │ updateOne({f}, {$set: {f:'new'}}) │
│                       │ (rewrites entire row)        │ (modifies only that field)        │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Increment a number    │ UPDATE SET col = col + 10    │ updateOne({f}, {$inc: {col: 10}}) │
│                       │ (read-modify-write)          │ (atomic increment)                │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Add a new field       │ ALTER TABLE ADD COLUMN        │ updateOne({f}, {$set: {new: v}})  │
│ (to one record)       │ then UPDATE (ALL rows get it)│ (only this document gets it)      │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Remove a field        │ SET col = NULL               │ updateOne({f}, {$unset: {col:1}}) │
│ (from one record)     │ (column still exists as NULL)│ (field completely removed)         │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Add to array          │ String/JSON manipulation     │ {$push: {arr: "new_item"}}         │
│                       │ (extremely painful)          │ (native array operation)           │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Remove from array     │ Virtually impossible cleanly │ {$pull: {arr: "old_item"}}         │
│                       │                              │ (native array operation)           │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Add to array          │ No equivalent                │ {$addToSet: {arr: "item"}}         │
│ (only if not present) │                              │ (prevents duplicates)              │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Upsert                │ DELETE + INSERT               │ updateOne({f}, {ops}, upsert:true)│
│ (insert or update)    │ (not atomic)                 │ (single atomic operation)          │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Update one row        │ UPDATE ... WHERE ... LIMIT 1 │ updateOne({filter}, {update})      │
│                       │ (no native LIMIT on UPDATE)  │ (exactly one document)             │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Update many rows      │ UPDATE ... WHERE ...          │ updateMany({filter}, {update})     │
│                       │ (all matching rows)          │ (all matching documents)           │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Replace entire record │ UPDATE SET all columns       │ replaceOne({filter}, {new_doc})    │
│                       │ (the only option)            │ (rarely needed — prefer operators) │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Atomicity             │ Single UPDATE is atomic       │ Single-document update is atomic   │
│                       │ Multi-table needs transaction │ Multi-document needs transaction   │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Post-update cleanup   │ VACUUM + ANALYZE required    │ None required                      │
│                       │                              │ (automatic compaction)             │
├──────────────────────┼─────────────────────────────┼──────────────────────────────────┤
│ Return updated doc    │ Not supported natively       │ findOneAndUpdate() returns doc     │
│                       │ (must SELECT after UPDATE)   │ (atomic find + update + return)    │
└──────────────────────┴─────────────────────────────┴──────────────────────────────────┘

🧩 HOW TO APPROACH IT IN MONGODB

THE UPDATE OPERATOR MENTAL MODEL:

Every MongoDB update has TWO parts:
    1. FILTER: which document(s) to update    → same as WHERE clause
    2. UPDATE: what to change                 → uses UPDATE OPERATORS

    db.collection.updateOne(
        { email: "alice@example.com" },       ← FILTER (find Alice)
        { $set: { "address.city": "Denver" } }← UPDATE (change city)
    )

UPDATE OPERATORS — THE COMPLETE TOOLKIT:

    FIELD OPERATORS:
    ┌─────────────┬────────────────────────────────────────────────────┐
    │ $set          │ Set field to value (create if doesn't exist)       │
    │ $unset        │ Remove field entirely from document                │
    │ $rename       │ Rename a field                                     │
    │ $inc          │ Increment numeric field by value (+ or -)          │
    │ $mul          │ Multiply numeric field by value                    │
    │ $min          │ Update field only if new value is LESS than current│
    │ $max          │ Update field only if new value is MORE than current│
    │ $currentDate  │ Set field to current date/timestamp                │
    │ $setOnInsert  │ Set field only during upsert INSERT (not update)   │
    └─────────────┴────────────────────────────────────────────────────┘

    ARRAY OPERATORS:
    ┌─────────────┬────────────────────────────────────────────────────┐
    │ $push         │ Add element to end of array                        │
    │ $pull         │ Remove all elements matching condition              │
    │ $addToSet     │ Add element only if not already in array            │
    │ $pop          │ Remove first (-1) or last (1) element               │
    │ $pullAll      │ Remove all matching values from array               │
    │ $each         │ Modifier: push/addToSet multiple elements           │
    │ $slice        │ Modifier: limit array size after push               │
    │ $sort         │ Modifier: sort array after push                     │
    │ $position     │ Modifier: insert at specific array position         │
    │ $             │ Positional: update first matching array element     │
    │ $[]           │ All positional: update ALL array elements           │
    │ $[<id>]       │ Filtered positional: update elements matching filter│
    └─────────────┴────────────────────────────────────────────────────┘

REDSHIFT HAS ZERO EQUIVALENTS FOR ARRAY OPERATORS
This entire category of operations is IMPOSSIBLE in Redshift
(unless using SUPER type with extremely complex JSON manipulation)

// PROJECT 4 — ALL 8 UPDATE SCENARIOS IN MONGOSH
// Using the 1000 customers from Project 2
// ═══════════════════════════════════════════════════════

use ecommerce

print("=" .repeat(60))
print("PROJECT 4: UPDATING DOCUMENTS")
print("=" .repeat(60))


// ═══════════════════════════════════════════════════════
// UPDATE 1: Alice moved — change her address
// Redshift: UPDATE SET city='Denver', state='CO' WHERE email='alice@...'
// ═══════════════════════════════════════════════════════

print("\n--- UPDATE 1: Change address ($set on nested fields) ---")

// BEFORE
let alice_before = db.customers.findOne(
    { email: "customer_0000@example.com" },
    { name: 1, address: 1, _id: 0 }
)
print("BEFORE:")
printjson(alice_before)

// THE UPDATE
let result1 = db.customers.updateOne(
    { email: "customer_0000@example.com" },     // FILTER (find Alice)
    {
        $set: {                                   // OPERATOR
            "address.city": "Denver",             // change city
            "address.state": "CO"                 // change state
        }
    }
)

print(`\nResult: matched=${result1.matchedCount}, modified=${result1.modifiedCount}`)

// AFTER
let alice_after = db.customers.findOne(
    { email: "customer_0000@example.com" },
    { name: 1, address: 1, _id: 0 }
)
print("AFTER:")
printjson(alice_after)

// WHAT $set DID:
// ┌──────────────────────────────────────────────────────────────┐
// │ $set: { "address.city": "Denver", "address.state": "CO" }   │
// │                                                               │
// │ → Navigated INTO the address nested object (dot notation)    │
// │ → Changed ONLY city and state                                │
// │ → Left ALL other fields untouched (name, email, age, etc.)   │
// │ → If "address.zip" existed, it would be untouched too        │
// │                                                               │
// │ REDSHIFT: UPDATE SET city='Denver', state='CO'               │
// │ → Internally: deleted old row, wrote ENTIRE new row          │
// │ → ALL columns rewritten even though only 2 changed           │
// │                                                               │
// │ MONGODB: modified ONLY the 2 fields within nested object     │
// │ → Rest of document untouched in storage                      │
// │ → (If document doesn't grow, WiredTiger updates in-place)    │
// └──────────────────────────────────────────────────────────────┘


// ═══════════════════════════════════════════════════════
// UPDATE 2: Bob crossed $1000 — upgrade to premium
// Redshift: UPDATE SET is_premium = TRUE WHERE email = 'bob@...'
// ═══════════════════════════════════════════════════════

print("\n--- UPDATE 2: Upgrade to premium ($set boolean) ---")

let result2 = db.customers.updateOne(
    { email: "customer_0001@example.com" },
    { $set: { is_premium: true } }
)
print(`matched=${result2.matchedCount}, modified=${result2.modifiedCount}`)

// If is_premium was ALREADY true → matchedCount=1, modifiedCount=0
// MongoDB tracks whether the value ACTUALLY changed
// Redshift: always reports rows affected regardless of actual change


// ═══════════════════════════════════════════════════════
// UPDATE 3: Customer makes purchase — increment spending
// Redshift: UPDATE SET total_spent = total_spent + 149.99
// ═══════════════════════════════════════════════════════

print("\n--- UPDATE 3: Increment spending ($inc) ---")

// BEFORE
let carol_before = db.customers.findOne(
    { email: "customer_0002@example.com" },
    { name: 1, total_spent: 1, _id: 0 }
)
print(`BEFORE: ${carol_before.name} spent $${carol_before.total_spent}`)

// THE UPDATE — $inc is ATOMIC
let result3 = db.customers.updateOne(
    { email: "customer_0002@example.com" },
    { $inc: { total_spent: 149.99 } }     // add 149.99 to current value
)

// AFTER
let carol_after = db.customers.findOne(
    { email: "customer_0002@example.com" },
    { name: 1, total_spent: 1, _id: 0 }
)
print(`AFTER:  ${carol_after.name} spent $${carol_after.total_spent}`)

// WHY $inc IS BETTER THAN $set FOR COUNTERS:
// ┌──────────────────────────────────────────────────────────────┐
// │ $set approach (WRONG for counters):                          │
// │   1. Read current value: total_spent = 1000                  │
// │   2. Calculate: 1000 + 149.99 = 1149.99                     │
// │   3. Write: $set: { total_spent: 1149.99 }                  │
// │   PROBLEM: If another request changes total_spent between    │
// │   step 1 and step 3 → RACE CONDITION → lost update          │
// │                                                               │
// │ $inc approach (CORRECT for counters):                        │
// │   1. Single atomic operation: $inc: { total_spent: 149.99 } │
// │   2. MongoDB atomically reads + adds + writes                │
// │   3. NO race condition — guaranteed correct                  │
// │                                                               │
// │ REDSHIFT:                                                    │
// │   UPDATE SET total_spent = total_spent + 149.99              │
// │   → Also atomic within a single UPDATE statement             │
// │   → But Redshift isn't designed for frequent small updates   │
// │   → Each UPDATE = delete + reinsert entire row               │
// └──────────────────────────────────────────────────────────────┘

// $inc with NEGATIVE value (decrement)
db.customers.updateOne(
    { email: "customer_0002@example.com" },
    { $inc: { total_spent: -10 } }      // subtract 10 (refund)
)

// $inc on NON-EXISTENT field → creates field with the increment value
db.customers.updateOne(
    { email: "customer_0002@example.com" },
    { $inc: { loyalty_points: 100 } }   // field doesn't exist → created as 100
)
let carol_points = db.customers.findOne(
    { email: "customer_0002@example.com" },
    { loyalty_points: 1, _id: 0 }
)
print(`Loyalty points (new field): ${carol_points.loyalty_points}`)


// ═══════════════════════════════════════════════════════
// UPDATE 4: Add "Sports" to preferred categories
// Redshift: IMPOSSIBLE cleanly — JSON string manipulation nightmare
// ═══════════════════════════════════════════════════════

print("\n--- UPDATE 4: Add to array ($push) ---")

// First, find a customer who HAS preferences
let target = db.customers.findOne(
    { "preferences.preferred_categories": { $exists: true } },
    { name: 1, email: 1, "preferences.preferred_categories": 1, _id: 0 }
)
print("BEFORE:")
printjson(target)

// $push — add element to end of array
let result4 = db.customers.updateOne(
    { email: target.email },
    { $push: { "preferences.preferred_categories": "Sports" } }
)

let target_after = db.customers.findOne(
    { email: target.email },
    { name: 1, "preferences.preferred_categories": 1, _id: 0 }
)
print("AFTER ($push):")
printjson(target_after)

// IMPORTANT: $push allows DUPLICATES
// If "Sports" was already in the array, it gets added AGAIN
// Use $addToSet to prevent duplicates (shown below)


// ═══════════════════════════════════════════════════════
// UPDATE 5: Remove "Clothing" from preferred categories
// Redshift: VIRTUALLY IMPOSSIBLE cleanly
// ═══════════════════════════════════════════════════════

print("\n--- UPDATE 5: Remove from array ($pull) ---")

// Find customer with "Clothing" in categories
let pullTarget = db.customers.findOne(
    { "preferences.preferred_categories": "Clothing" },
    { name: 1, email: 1, "preferences.preferred_categories": 1, _id: 0 }
)

if (pullTarget) {
    print("BEFORE:")
    printjson(pullTarget)

    // $pull — remove ALL occurrences matching the condition
    db.customers.updateOne(
        { email: pullTarget.email },
        { $pull: { "preferences.preferred_categories": "Clothing" } }
    )

    let pullAfter = db.customers.findOne(
        { email: pullTarget.email },
        { name: 1, "preferences.preferred_categories": 1, _id: 0 }
    )
    print("AFTER ($pull):")
    printjson(pullAfter)
} else {
    print("No customer with 'Clothing' found — demonstrating with manual setup")
    db.customers.updateOne(
        { email: "customer_0005@example.com" },
        { $set: { "preferences.preferred_categories": ["Books", "Clothing", "Home"] } }
    )
    db.customers.updateOne(
        { email: "customer_0005@example.com" },
        { $pull: { "preferences.preferred_categories": "Clothing" } }
    )
    printjson(db.customers.findOne(
        { email: "customer_0005@example.com" },
        { name: 1, "preferences.preferred_categories": 1, _id: 0 }
    ))
}


// ═══════════════════════════════════════════════════════
// UPDATE 6: Remove sms_alerts field entirely
// Redshift: SET sms_alerts = NULL (field still exists)
// MongoDB: $unset — field COMPLETELY REMOVED from document
// ═══════════════════════════════════════════════════════

print("\n--- UPDATE 6: Remove field entirely ($unset) ---")

// Find customer with preferences.sms_alerts
let unsetTarget = db.customers.findOne(
    { "preferences.sms_alerts": { $exists: true } },
    { name: 1, email: 1, preferences: 1, _id: 0 }
)

if (unsetTarget) {
    print("BEFORE:")
    printjson(unsetTarget)

    // $unset — completely remove the field
    db.customers.updateOne(
        { email: unsetTarget.email },
        { $unset: { "preferences.sms_alerts": "" } }
        // The value ("") doesn't matter for $unset — it's ignored
        // Convention: use "" or 1 or true — all work the same
    )

    let unsetAfter = db.customers.findOne(
        { email: unsetTarget.email },
        { name: 1, preferences: 1, _id: 0 }
    )
    print("AFTER ($unset):")
    printjson(unsetAfter)
    // sms_alerts field is GONE — not null, not empty — GONE
}

// THE DIFFERENCE: $unset vs $set to null
// ┌──────────────────────────────────────────────────────────────┐
// │ $set: { phone: null }                                        │
// │ → Document: { name: "Alice", phone: null, ... }              │
// │ → Field EXISTS with null value                               │
// │ → { phone: { $exists: true } } → MATCHES                    │
// │ → { phone: null } → MATCHES                                 │
// │                                                               │
// │ $unset: { phone: "" }                                        │
// │ → Document: { name: "Alice", ... }   (no phone field at all) │
// │ → Field DOES NOT EXIST                                       │
// │ → { phone: { $exists: true } } → DOES NOT MATCH             │
// │ → { phone: null } → MATCHES (null matches missing too!)     │
// │                                                               │
// │ REDSHIFT: Can only SET to NULL — column always exists         │
// │ MONGODB: Can truly REMOVE a field from a document             │
// │ This is a UNIQUE capability of document databases             │
// └──────────────────────────────────────────────────────────────┘


// ═══════════════════════════════════════════════════════
// UPDATE 7: Upsert — create if doesn't exist, update if does
// Redshift: DELETE + INSERT in a transaction (not atomic)
// MongoDB: Single atomic operation
// ═══════════════════════════════════════════════════════

print("\n--- UPDATE 7: Upsert (insert-or-update) ---")

// Attempt 1: Customer doesn't exist → INSERT happens
let result7a = db.customers.updateOne(
    { email: "newvisitor@example.com" },          // FILTER
    {
        $set: {                                     // UPDATE
            name: "New Visitor",
            "address.city": "Portland",
            "address.state": "OR"
        },
        $inc: { total_spent: 79.99 },              // increment (or create as 79.99)
        $setOnInsert: {                             // ONLY set these on INSERT
            signup_date: new Date(),
            is_premium: false,
            age: null                               // unknown age
        }
    },
    { upsert: true }                               // ← THE KEY: enable upsert
)

print(`Attempt 1 (new customer):`)
print(`  matched=${result7a.matchedCount}, modified=${result7a.modifiedCount}`)
print(`  upsertedId=${result7a.upsertedId}`)
// upsertedId has a value → document was INSERTED

printjson(db.customers.findOne({ email: "newvisitor@example.com" }))

// Attempt 2: Customer NOW exists → UPDATE happens
let result7b = db.customers.updateOne(
    { email: "newvisitor@example.com" },
    {
        $set: { name: "New Visitor (returned)" },
        $inc: { total_spent: 120.00 },              // add another purchase
        $setOnInsert: {                              // IGNORED because this is an update
            signup_date: new Date(),                 // won't overwrite
            is_premium: false                        // won't overwrite
        }
    },
    { upsert: true }
)

print(`\nAttempt 2 (existing customer):`)
print(`  matched=${result7b.matchedCount}, modified=${result7b.modifiedCount}`)
print(`  upsertedId=${result7b.upsertedId}`)
// upsertedId is null → document was UPDATED (not inserted)

printjson(db.customers.findOne({ email: "newvisitor@example.com" }))
// total_spent = 79.99 + 120.00 = 199.99 (accumulated correctly)
// signup_date unchanged (setOnInsert only runs on insert)

// $setOnInsert EXPLAINED:
// ┌──────────────────────────────────────────────────────────────┐
// │ $setOnInsert: { signup_date: new Date(), is_premium: false } │
// │                                                               │
// │ ON INSERT (document didn't exist):                           │
// │   → signup_date is SET to current date                       │
// │   → is_premium is SET to false                               │
// │   → These are "default values for new documents"             │
// │                                                               │
// │ ON UPDATE (document already existed):                        │
// │   → signup_date is IGNORED (keeps original value)            │
// │   → is_premium is IGNORED (keeps original value)             │
// │   → Prevents overwriting creation-time data                  │
// │                                                               │
// │ REDSHIFT: No equivalent — must code this logic in app layer  │
// └──────────────────────────────────────────────────────────────┘


// ═══════════════════════════════════════════════════════

    customers.update_one({"email": x}, {"$set": {"city": "Denver"}})
    customers.update_one({"email": x}, {"$inc": {"total_spent": 100}})
    customers.update_one({"email": x}, {"$push": {"tags": "vip"}})
    
    # RIGHT — 1 round trip
    customers.update_one({"email": x}, {
        "$set": {"city": "Denver"},
        "$inc": {"total_spent": 100},
        "$push": {"tags": "vip"}
    })
    → 3x fewer network round trips
    → Single atomic operation (all-or-nothing)

OPTIMIZATION 2: bulk_write FOR MIXED OPERATIONS
    # When updating many different documents with different operations
    from pymongo import UpdateOne, UpdateMany
    
    operations = [
        UpdateOne({"email": "a@ex.com"}, {"$inc": {"total_spent": 50}}),
        UpdateOne({"email": "b@ex.com"}, {"$set": {"is_premium": True}}),
        UpdateMany({"address.state": "TX"}, {"$inc": {"total_spent": 25}}),

    result = customers.bulk_write(operations, ordered=False)
    # Single network call for ALL operations
    # Same concept as Project 2 bulk_write — now with updates

✅ WHAT SHOULD DO

BEST PRACTICES FOR UPDATES IN MONGODB:

    ✅ USE UPDATE OPERATORS ($set, $inc, etc.) — NOT replaceOne()
       → Operators modify ONLY what changed
       → replaceOne replaces ENTIRE document (like Redshift UPDATE)
       → Operators are safer — can't accidentally delete fields
    
    ✅ USE $inc FOR COUNTERS — NEVER read-then-set
       # WRONG (race condition)
       doc = customers.find_one({"email": x})
       new_total = doc["total_spent"] + 100
       customers.update_one({"email": x}, {"$set": {"total_spent": new_total}})
       
       # RIGHT (atomic)
       customers.update_one({"email": x}, {"$inc": {"total_spent": 100}})
       
       → $inc is atomic at the storage engine level
       → No race condition even under high concurrency
       → Redshift equivalent (SET col = col + 100) is also atomic within single UPDATE
         but Redshift isn't designed for high-concurrency updates
    
    ✅ USE $addToSet INSTEAD OF $push WHEN DUPLICATES MATTER
       → $push always adds (can create duplicates)
       → $addToSet only adds if not already present
       → Choose based on your use case:
         Tags, categories → $addToSet (no duplicates)
         Activity log, events → $push (duplicates are fine)
    
    ✅ USE $setOnInsert WITH UPSERTS FOR CREATION-TIME FIELDS
       → signup_date, created_at, initial_tier → only set on creation
       → $setOnInsert is IGNORED during updates
       → Prevents overwriting original values
    
    ✅ USE findOneAndUpdate() WHEN YOU NEED THE RESULT
       → Atomic: find + update + return in one operation
       → No race condition between finding and updating
       → Use returnDocument: "after" to get the updated version
       → Perfect for: counters, queues, state machines
    
    ✅ COMBINE MULTIPLE OPERATORS IN ONE UPDATE CALL
       → { $set: {...}, $inc: {...}, $push: {...} }
       → One network round trip, one atomic operation
       → Never split into multiple update_one calls for the same document

REDSHIFT HABITS TO UNLEARN:
    🔄 "UPDATE replaces the row — I need to set ALL columns"
       → MongoDB $set touches ONLY the specified fields
       → Everything else stays untouched
       → replaceOne() exists but should RARELY be used
    
    🔄 "I need to run VACUUM after updates"
       → MongoDB WiredTiger handles space reuse automatically
       → No VACUUM, no ANALYZE needed after updates
    
    🔄 "Updating one column is cheap"
       → In Redshift: updating one column rewrites the ENTIRE row
       → In MongoDB: $set on one field is truly cheap (modifies in place when possible)
       → MongoDB updates are genuinely lighter than Redshift updates
    
    🔄 "I can't add fields to existing records without ALTER TABLE"
       → $set creates new fields on individual documents
       → No ALTER TABLE, no schema migration, no downtime
       → Different documents in same collection can have different fields
    
    🔄 "Arrays need separate tables"
       → MongoDB arrays are first-class: $push, $pull, $addToSet, $elemMatch
       → No separate table, no JOIN, no foreign key
       → Array operations are atomic within the document

❌ WHAT SHOULD NOT DO

ANTI-PATTERNS — ESPECIALLY TRAPS FOR REDSHIFT USERS:

    ❌ USING replaceOne() AS YOUR DEFAULT UPDATE METHOD
       # WRONG — Redshift developer instinct: "update = replace the row"
       doc = customers.find_one({"email": x})
       doc["address"]["city"] = "Denver"
       customers.replace_one({"email": x}, doc)
       
       # RIGHT — surgical update
       customers.update_one({"email": x}, {"$set": {"address.city": "Denver"}})
       
       WHY replaceOne IS DANGEROUS:
       → If another process added a field between find and replace → LOST
       → If doc variable is missing fields → those fields DELETED
       → If doc is stale (cached) → overwrites newer changes
       → replaceOne is the MongoDB equivalent of Redshift's row-rewrite
         You came to MongoDB to AVOID this pattern

    ❌ READING-THEN-WRITING FOR ATOMIC OPERATIONS
       # WRONG — race condition
       doc = customers.find_one({"email": x})
       if doc["total_spent"] > 1000:
           customers.update_one({"email": x}, {"$set": {"is_premium": True}})
       
       # RIGHT — atomic conditional update
       customers.update_one(
           {"email": x, "total_spent": {"$gt": 1000}},    # condition IN the filter
           {"$set": {"is_premium": True}}
       )
       
       WHY: Between your find and update, another process could change total_spent
       → Move the condition INTO the filter
       → MongoDB checks condition and applies update atomically
       → Redshift: UPDATE SET is_premium = TRUE WHERE total_spent > 1000
         (same atomic pattern — but Redshift users forget this in MongoDB)

    ❌ UPDATING THE _id FIELD
       # WRONG — _id is IMMUTABLE
       customers.update_one(
           {"email": x},
           {"$set": {"_id": "new_id"}}    # ERROR!
       )
       
       → _id cannot be changed after insertion
       → Must delete and reinsert if you need a different _id
       → Redshift: IDENTITY column also cannot be updated
       → Same restriction, same reason (primary key stability)

    ❌ USING $push WITHOUT $slice ON GROWING ARRAYS
       # WRONG — array grows unbounded
       customers.update_one(
           {"email": x},
           {"$push": {"activity_log": {"action": "login", "date": datetime.now()}}}
       )
       # After 100,000 logins → document approaches 16MB limit → FAILS
       
       # RIGHT — cap the array
       customers.update_one(
           {"email": x},
           {"$push": {
               "activity_log": {
                   "$each": [{"action": "login", "date": datetime.now()}],
                   "$slice": -100,        # keep only last 100 entries
                   "$sort": {"date": -1}  # newest first
               }
           }}
       )
       
       → $slice prevents unbounded growth
       → 16MB document limit is a HARD wall
       → We'll design proper solutions in Project 17 (Outlier Pattern)

    ❌ UPDATING MANY DOCUMENTS WITHOUT UNDERSTANDING SCOPE
       # DANGEROUS — missing filter = updates ALL documents
       customers.update_many(
           {},                                          # ← EMPTY FILTER = ALL DOCUMENTS
           {"$set": {"accidentally_set_on_everyone": True}}
       )
       
       → ALWAYS double-check your filter before updateMany
       → Use countDocuments(filter) first to verify scope
       → Redshift equivalent: UPDATE without WHERE clause
       → Same danger, but MongoDB has no ROLLBACK for completed writes
         (unless using transactions — Project 28)

    ❌ MIXING OPERATORS ON THE SAME FIELD
       # WRONG — cannot $set and $inc same field
       customers.update_one(
           {"email": x},
           {
               "$set": {"total_spent": 1000},
               "$inc": {"total_spent": 100}      # CONFLICT → error
           }
       )
       
       # Choose ONE: set to exact value OR increment
       # $set: "I know the final value should be 1000"
       # $inc: "I know it should increase by 100 from whatever it is now"

    ❌ FORGETTING THAT updateOne UPDATES ONLY THE FIRST MATCH
       # If multiple documents match the filter:
       customers.update_one(
           {"address.state": "TX"},        # matches many customers
           {"$set": {"bonus": True}}       # only FIRST match updated
       )
       
       → Use updateMany() when you intend to update ALL matches
       → updateOne() is for targeted single-document updates
       → Redshift: UPDATE always affects ALL matching rows (like updateMany)
       → This is a SUBTLE difference that catches Redshift users

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Update Path — What Happens During updateOne()

WHEN YOU RUN:
    db.customers.updateOne(
        { email: "alice@example.com" },
        { $set: { "address.city": "Denver" }, $inc: { total_spent: 149.99 } }
    )

STEP 1: FIND THE DOCUMENT
    → Query planner evaluates filter: { email: "alice@example.com" }
    → Checks indexes: if index on email exists → IXSCAN (fast)
    → If no index → COLLSCAN (scan all documents — slow)
    → Document located in WiredTiger B-tree

STEP 2: ACQUIRE DOCUMENT-LEVEL LOCK
    → WiredTiger uses DOCUMENT-LEVEL locking (not table-level!)
    → Only THIS document is locked
    → Other documents in the collection can be read/written concurrently
    → Redshift: table-level or block-level locking during UPDATE
    
    LOCK GRANULARITY COMPARISON:
    ┌──────────────────────┬──────────────────────────────────┐
    │ REDSHIFT               │ MONGODB                           │
    ├──────────────────────┼──────────────────────────────────┤
    │ Table-level write lock │ Document-level lock               │
    │ UPDATE blocks other    │ Update on doc A does NOT block    │
    │ writes to same table   │ updates on doc B in same collection│
    │ Concurrent UPDATEs     │ Concurrent updates on DIFFERENT   │
    │ are serialized         │ documents run in PARALLEL          │
    └──────────────────────┴──────────────────────────────────┘
    
    THIS is why MongoDB handles high-concurrency updates
    and Redshift does not

STEP 3: APPLY UPDATE OPERATORS
    → $set: navigate to address.city in BSON, change value
    → $inc: navigate to total_spent, read current value, add 149.99, write new value
    → Both operations applied atomically (all-or-nothing)
    → Document modified in WiredTiger cache (in-memory)

STEP 4: IN-PLACE UPDATE vs DOCUMENT RELOCATION
    ┌──────────────────────────────────────────────────────────────┐
    │ CASE A: Document size UNCHANGED or SMALLER after update      │
    │ → WiredTiger updates the document IN PLACE on the page       │
    │ → Very fast — no relocation needed                           │
    │ → $set: changing "Austin" to "Denver" (same length) = in-place│
    │ → $inc: number stays same size = in-place                    │
    │                                                               │
    │ CASE B: Document GROWS after update                          │
    │ → Document may need to RELOCATE to a larger page space       │
    │ → WiredTiger moves document to new location                  │
    │ → Index entries updated to point to new location             │
    │ → Slightly more expensive                                    │
    │ → $push to array = document grows = possible relocation      │
    │ → $set new field = document grows = possible relocation      │
    │                                                               │
    │ REDSHIFT: ALWAYS relocates (delete old row + insert new row) │
    │ MONGODB: relocates ONLY when document size grows beyond page │
    │ → MongoDB in-place updates are significantly cheaper         │
    └──────────────────────────────────────────────────────────────┘

STEP 5: JOURNAL WRITE
    → Modified document written to WiredTiger journal (WAL)
    → Journal is synced to disk (durability guarantee)
    → Based on write concern:
      w:1 → acknowledged after journal write on primary
      w:"majority" → acknowledged after journal write on majority
    → Actual data file update happens at next checkpoint (60s)

STEP 6: INDEX UPDATES
    → ONLY indexes affected by the changed fields are updated
    → $set address.city: if index on "address.city" exists → update index
    → $inc total_spent: if index on total_spent exists → update index
    → If changed field has NO index → no index update needed
    
    REDSHIFT COMPARISON:
    → Zone maps updated: min/max per block may change
    → But zone maps are much lighter than B-tree indexes
    → Redshift advantage: zone maps are cheaper to maintain
    → MongoDB advantage: indexes enable point lookups (zone maps don't)

STEP 7: OPLOG ENTRY
    → Update operation recorded in oplog (operations log)
    → Oplog is a capped collection that records all write operations
    → Secondary replica set members read oplog to replicate the update
    → Change streams (Project 25) also read from oplog
    
    OPLOG ENTRY LOOKS LIKE:
    {
        "op": "u",                              // u = update
        "ns": "ecommerce.customers",            // namespace
        "o": {                                   // the update operations
            "$v": 2,
            "diff": {
                "saddress": { "u": { "city": "Denver" } },
                "utotal_spent": 1399.98          // new value
            }
        },
        "o2": { "_id": ObjectId("...") },       // document identifier
        "ts": Timestamp(...)                     // operation timestamp
    }
    
    → Oplog records the DELTA (what changed), not the full document
    → This is efficient for replication
    → Redshift has NO equivalent — replication is at storage level (RA3)

Redshift UPDATE Internals — For Comparison

WHEN YOU RUN:
    UPDATE store.customers SET city = 'Denver', total_spent = total_spent + 149.99
    WHERE email = 'alice@example.com'

    1. Leader node creates plan → pushed to compute slices
    2. Slice with Alice's row (determined by DISTKEY):
       a. Reads ALL columns of Alice's row from columnar blocks
       b. Marks the old row as DELETED (ghost row)
       c. Creates a NEW row with ALL columns:
          → name: "Alice" (unchanged but rewritten)
          → email: "alice@example.com" (unchanged but rewritten)
          → age: 34 (unchanged but rewritten)
          → city: "Denver" (CHANGED)
          → state: "TX" (unchanged but rewritten)
          → total_spent: 1399.98 (CHANGED)
          → ... all other columns rewritten too
       d. Appends new row to unsorted region of each column block
       e. Zone maps updated for affected blocks
    3. Table now has "unsorted" rows → needs VACUUM SORT
    4. Statistics may be stale → needs ANALYZE
    5. Ghost rows waste disk space → needs VACUUM DELETE
    
    TOTAL COST: Read all columns + Write all columns + Ghost row overhead
    
    VS MONGODB:
    TOTAL COST: Read one document + Modify 2 fields + Write modified document
    
    FOR A SINGLE ROW UPDATE:
    → MongoDB is 10-100x more efficient than Redshift
    → Redshift is designed for BATCH updates on millions of rows
    → MongoDB is designed for INDIVIDUAL updates on single documents

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 4:

PITFALL 1: "replaceOne is my default UPDATE"
    REDSHIFT TRAINED YOU: UPDATE rewrites entire row → you think in terms of
                          "set ALL columns to their new values"
    MONGODB TRAP: Using replaceOne({}, entire_document) instead of $set
    CONSEQUENCE: Accidentally delete fields that weren't in replacement doc
    FIX: Use $set, $inc, $push — always prefer operators over replacement

PITFALL 2: "I'll read, modify in code, then write back"
    REDSHIFT TRAINED YOU: UPDATE SET col = col + 1 is atomic in SQL
    MONGODB TRAP: find → modify in Python → replace (three steps)
    CONSEQUENCE: Race condition under concurrent access
    FIX: Use $inc, $set, $push — server-side atomic operators
         Or use findOneAndUpdate for atomic read-modify-return

PITFALL 3: "updateOne updates ALL matching documents"
    REDSHIFT TRAINED YOU: UPDATE ... WHERE ... affects ALL matching rows
    MONGODB REALITY: updateOne affects ONLY the FIRST matching document
    FIX: Use updateMany() when you intend to update all matches
    SEVERITY: HIGH — silently updates only one document when you expected many

PITFALL 4: "I need transactions for every update"
    REDSHIFT TRAINED YOU: BEGIN/COMMIT around UPDATE statements
    MONGODB REALITY: Single-document updates are ALREADY atomic
                     No transaction needed for single-document operations
    FIX: Only use transactions (Project 28) for MULTI-document atomicity
    SEVERITY: MEDIUM — unnecessary transactions hurt performance

PITFALL 5: "Empty filter in updateMany"
    db.customers.updateMany({}, {"$set": {"oops": True}})
    → Updates EVERY document in collection
    → No confirmation prompt (unlike some SQL clients)
    → No easy ROLLBACK
    FIX: Always count_documents(filter) before updateMany
         Use a staging/test collection first

PITFALL 6: "$push without $slice grows documents forever"
    → Arrays grow with each $push
    → 16MB document limit is hard
    → Redshift: separate table rows have no equivalent limit
    FIX: Always use $slice with $push on potentially growing arrays
         Or use Bucket pattern (Project 16) for high-volume data

PITFALL 7: "Updating a field that's part of a shard key"
    → Shard key fields are immutable (in older MongoDB versions)
    → MongoDB 5.0+ allows shard key updates but with restrictions
    → Redshift: changing DISTKEY column value just moves the row to another slice
    → MongoDB: changing shard key may require document migration between shards
    FIX: Design shard keys carefully (Project 22)
    
PITFALL 8: "Expecting modifiedCount > 0 when value didn't change"
    result = customers.update_one(
        {"email": "alice@ex.com"},
        {"$set": {"is_premium": True}}    # already True
    )
    # matchedCount = 1, modifiedCount = 0
    # Document MATCHED but value didn't CHANGE
    
    → Redshift: reports rows affected even if value unchanged
    → MongoDB: distinguishes between matched and actually modified
    → Check matchedCount for "did the filter find a document?"
    → Check modifiedCount for "did the document actually change?"

│ Counter increments   │ ~50ms each (heavy)          │ ~1ms each (atomic $inc)       │

KEY INSIGHT:
    → MongoDB WINS for: high-frequency, small, single-document updates
    → Redshift WINS for: large batch updates across millions of rows
    → MongoDB WINS for: concurrent updates on different documents
    → Redshift WINS for: complex JOIN-based conditional updates

📐 DATA MODELING COMPARISON

HOW DOCUMENT DESIGN AFFECTS UPDATES:

SCENARIO: Customer makes a purchase → update spending + add order + update tier

REDSHIFT (3 separate tables):
    BEGIN;
    
    -- Update customer spending
    UPDATE customers SET total_spent = total_spent + 149.99
    WHERE customer_id = 1234;
    
    -- Insert order record
    INSERT INTO orders (customer_id, amount, date, status)
    VALUES (1234, 149.99, GETDATE(), 'processing');
    
    -- Update tier if threshold crossed
    UPDATE customers SET tier = 'Gold'
    WHERE customer_id = 1234 AND total_spent > 1000;
    
    COMMIT;
    
    → 3 operations across 2 tables
    → Requires explicit transaction for atomicity
    → Each UPDATE rewrites entire row

MONGODB (single document):
    db.customers.updateOne(
        { _id: 1234 },
        {
            $inc: { total_spent: 149.99 },
            $push: {
                recent_orders: {
                    $each: [{ amount: 149.99, date: new Date(), status: "processing" }],
                    $slice: -10                // keep last 10 orders embedded
                }
            },
            $currentDate: { last_purchase: true }
        }
    )
    // Then conditionally update tier (or do it in app logic)
    db.customers.updateOne(
        { _id: 1234, total_spent: { $gt: 1000 } },
        { $set: { tier: "Gold" } }
    )
    
    → 1-2 operations on single collection
    → Single-document updates are automatically atomic
    → Only changed fields modified
    → No transaction needed for the first update (single document)

TRADE-OFF:
    MongoDB: Faster for single-customer operations (no JOIN, no transaction)
    Redshift: Better for batch tier recalculation across ALL customers
    
    BEST PRACTICE: Use MongoDB for real-time updates
                   Sync to Redshift for batch analytics and recalculations
