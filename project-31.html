🔴 REDSHIFT MINDSET — How You Diagnose Slow Queries

EXPLAIN in Redshift — What You Already Know

-- REDSHIFT: EXPLAIN a slow query
EXPLAIN
SELECT o.order_id, o.total, c.name, c.email
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE c.email = 'john@example.com'
  AND o.order_date >= '2024-01-01'
ORDER BY o.order_date DESC
LIMIT 20;

-- OUTPUT (simplified):
-- XN Limit  (cost=0.00..100.00 rows=20)
--   -> XN Merge  (cost=0.00..500.00 rows=1000)
--     -> XN Network  (cost=0.00..500.00 rows=1000)
--       -> XN Sort  (cost=0.00..450.00 rows=1000)
--         -> XN Hash Join DS_DIST_ALL_NONE  (cost=0.00..400.00 rows=1000)
--           -> XN Seq Scan on orders  (cost=0.00..300.00 rows=50000)
--                Filter: (order_date >= '2024-01-01')
--           -> XN Hash  (cost=0.00..50.00 rows=1)
--             -> XN Seq Scan on customers  (cost=0.00..50.00 rows=1)
--                  Filter: (email = 'john@example.com')

-- WHAT YOU LOOK FOR IN REDSHIFT EXPLAIN:

-- 1. DISTRIBUTION TYPE (DS_DIST_*)
--    DS_DIST_ALL_NONE  → one table replicated, no redistribution needed (GOOD)
--    DS_DIST_BOTH      → both tables redistributed across network (BAD)
--    DS_DIST_INNER     → inner table redistributed (OK)
--    DS_BCAST_INNER    → inner table broadcast to all slices (OK for small)

-- 2. SCAN TYPE
--    Seq Scan          → reading all rows (might be OK or BAD)
--    Sort Key Scan     → using sort key to skip blocks (GOOD)

-- 3. COST ESTIMATES
--    cost=start..total  → relative cost units
--    rows=N             → estimated row count
--    Higher cost = slower

-- 4. JOIN TYPE
--    Hash Join          → build hash table, probe (most common)
--    Merge Join         → both inputs sorted (efficient for sorted data)
--    Nested Loop        → for very small tables (or BAD plan)

-- REDSHIFT ALSO HAS:
-- STL_QUERY      → actual execution time
-- STL_SCAN       → actual rows scanned per slice
-- STL_ALERT      → performance alerts (missing stats, etc.)
-- SVL_QUERY_SUMMARY → simplified query summary

What Redshift EXPLAIN Tells You vs What It Doesn't

REDSHIFT EXPLAIN STRENGTHS:
    ✅ Shows distribution strategy (network data movement)
    ✅ Shows estimated rows at each step
    ✅ Shows join strategies
    ✅ Shows sort key usage
    ✅ Cost-based optimizer with statistics

REDSHIFT EXPLAIN LIMITATIONS:
    ❌ EXPLAIN alone doesn't show ACTUAL execution stats
    ❌ Must query STL_ system tables for actual metrics
    ❌ No "rejected plans" concept (optimizer picks one, doesn't show alternatives)
    ❌ No index choice (Redshift doesn't have traditional indexes)
    ❌ Zone map filtering not visible in EXPLAIN output
    ❌ Must use SVL_QUERY_SUMMARY for actual vs estimated comparison

🟢 MONGODB MINDSET — How explain() Works Differently

The Three Verbosity Levels — Your Diagnostic Toolkit

MONGODB explain() HAS THREE MODES:

1. "queryPlanner" (default)
    → Shows the PLAN without executing the query
    → Like Redshift EXPLAIN (plan only)
    → Fast — doesn't actually run the query
    → Shows: winning plan, rejected plans, index selection

2. "executionStats"
    → EXECUTES the query and shows actual metrics
    → Like Redshift EXPLAIN + STL_QUERY combined
    → Shows: actual docs examined, keys examined, execution time
    → THIS IS THE MODE YOU'LL USE 90% OF THE TIME

3. "allPlansExecution"
    → Executes ALL candidate plans (not just the winner)
    → Shows performance of rejected plans too
    → Expensive — use only for deep debugging
    → No Redshift equivalent (Redshift doesn't expose alternative plans)

The Key Metrics — What Separates Good from Bad

THE FOUR NUMBERS THAT MATTER MOST:

1. totalDocsExamined
   → How many documents MongoDB READ from disk/cache
   → Like: actual rows scanned in Redshift STL_SCAN
   → LOWER IS BETTER
   → If this equals totalDocsReturned → PERFECT (no wasted reads)
   → If this is 100x totalDocsReturned → TERRIBLE (scanning everything)

2. totalKeysExamined
   → How many index entries MongoDB traversed
   → Like: sort key entries checked in Redshift zone map scan
   → Should be close to totalDocsExamined (or equal for covered queries)

3. nReturned
   → How many documents were returned to the client
   → Like: rows returned in Redshift query results

4. executionTimeMillis
   → Total wall-clock time for the query
   → Like: elapsed time in STL_QUERY

THE GOLDEN RATIO:
    nReturned ≈ totalDocsExamined ≈ totalKeysExamined

    PERFECT: nReturned=20, docsExamined=20, keysExamined=20
    → Every document read was returned — zero waste

    BAD: nReturned=20, docsExamined=1000000, keysExamined=0
    → Read 1M documents to return 20 — COLLECTION SCAN (no index!)

    CONCERNING: nReturned=20, docsExamined=5000, keysExamined=5000
    → Index used but 5000 docs examined for 20 returned
    → Index not selective enough — wrong index or missing compound index

🧩 HOW TO APPROACH IT IN MONGODB

THE DIAGNOSTIC METHODOLOGY — 5 STEPS:

STEP 1: IDENTIFY THE SLOW QUERY
    → Application logs, Atlas Performance Advisor, Profiler
    → "Which query is slow?" before "Why is it slow?"

STEP 2: RUN explain("executionStats")
    → Get the ACTUAL execution metrics
    → Not just the plan — the REAL numbers

STEP 3: CHECK THE GOLDEN RATIO
    → nReturned vs totalDocsExamined vs totalKeysExamined
    → Is there waste? How much?

STEP 4: IDENTIFY THE STAGE TYPE
    → COLLSCAN = no index = almost always the problem
    → IXSCAN = index used = check if RIGHT index
    → SORT = in-memory sort = check if index can provide sort

STEP 5: FIX AND VERIFY
    → Create/modify index
    → Re-run explain() to confirm improvement
    → Compare before/after metrics

REDSHIFT EQUIVALENT METHODOLOGY:
    1. Find slow query (STL_QUERY)
    2. Run EXPLAIN
    3. Check distribution (DS_DIST_*)
    4. Check sort key usage
    5. Fix (change DISTKEY/SORTKEY, VACUUM, ANALYZE)

// (Project 10 compound index knowledge applied)
// ── STEP 2: FIX — Compound index ─────────────────────

// Equality: customer_id = "cust_5000"

//       inputStage: {
//         stage: "FETCH",               ← NO SORT STAGE! ✅
//         inputStage: {
//           stage: "IXSCAN",
//           keyPattern: { customer_id: 1, order_date: -1 },
//           indexName: "customer_date_compound",
//           indexBounds: {

//           }
//         }

// → No need to fetch all 50 orders, sort, then take 20

// REDSHIFT EQUIVALENT:
// Like having COMPOUND SORTKEY(customer_id, order_date)
// → Zone maps skip to customer's data
// → Data already in date order → no sort needed
// → Limit stops after first 20 rows

Case 3: Covered Query — Index-Only Scan (Fastest Possible)

// ═══════════════════════════════════════════════════════
// CASE 3: Quick inventory check — need only category + stock count
// Query: "How many items in Electronics category?"
// ═══════════════════════════════════════════════════════

// ── THE QUERY ────────────────────────────────────────
// Redshift: SELECT category, SUM(stock_qty) FROM products
//           WHERE category = 'Electronics' GROUP BY 1;

// First without covered index:
db.products.find(
    { category: "Electronics" },
    { _id: 0, category: 1, stock_qty: 1 }  // projection
).explain("executionStats")

// OUTPUT: IXSCAN on category → FETCH → project
// The FETCH stage means: after finding via index,
// go back to the FULL DOCUMENT to get stock_qty
// This is an extra disk/cache read per document

// ── FIX: Create index that COVERS the query ──────────
// A "covered query" = all fields in query + projection are IN the index
// MongoDB reads ONLY the index — never touches the document

db.products.createIndex(

AGGREGATION explain() — IMPORTANT DIFFERENCES:

1. SYNTAX IS DIFFERENT:
    find():      db.coll.find({...}).explain("executionStats")
    aggregate(): db.coll.explain("executionStats").aggregate([...])
    
    Note: explain() BEFORE aggregate(), not after
    (Chaining .explain() after aggregate() doesn't work the same way)

2. OUTPUT STRUCTURE IS DIFFERENT:
    find(): flat executionStats at top level
    aggregate(): nested in "stages" array
    → First stage = "$cursor" (how data enters pipeline)
    → Subsequent stages = pipeline stages ($group, $sort, etc.)

3. ONLY FIRST STAGE IS OPTIMIZED BY QUERY PLANNER:
    → $match at beginning → query planner can use indexes
    → $group, $sort, $project → pipeline engine (no index usage)
    → THIS IS WHY $match SHOULD BE FIRST STAGE
    → Like Redshift: WHERE clause processed before GROUP BY

4. $match + $sort AT START CAN BE COALESCED:
    → If $match followed by $sort, and index covers both:
    → MongoDB may combine into single IXSCAN
    → Like Redshift: sort key covers both filter and order

┌──────────────────────────────────────────────────────────┐
│  QUERY PLANNER — PLAN COMPETITION                         │
│                                                           │
│  Step 1: Identify candidate indexes                       │
│    → { customer_id: 1 }                                   │
│    → { order_date: -1 }                                   │
│    → { status: 1, order_date: -1 }                        │
│    → { customer_id: 1, order_date: -1 }                   │
│                                                           │

│    → Plan A: IXSCAN(customer_id) → FETCH → SORT           │
│    → Plan B: IXSCAN(order_date) → FETCH(filter customer)  │
│    → Plan C: IXSCAN(status+date) → FETCH(filter customer) │
│    → Plan D: IXSCAN(customer+date) → FETCH(filter status) │
│                                                           │
│  Step 3: RACE the plans (run all simultaneously)          │
│    → Each plan runs until one produces 101 results         │
│      (or all documents examined)                          │
│    → First plan to reach 101 results WINS                 │
│    → Plan D wins: most selective, no sort needed           │
│                                                           │
│  Step 4: CACHE the winning plan                           │
│    → Same query shape → same plan (cached for reuse)      │

│                                                           │
│  REDSHIFT COMPARISON:                                     │
│  → Redshift uses COST-BASED optimizer                     │
│    → Estimates row counts, I/O costs per plan              │
│    → Picks lowest-cost plan (no actual trial execution)   │
│  → MongoDB uses EMPIRICAL plan competition                │
│    → Actually RUNS plans briefly                          │
│    → Picks fastest in practice (not estimated)            │
│  → MongoDB approach: slower first-time, accurate choice   │
│  → Redshift approach: fast always, sometimes wrong        │

✅ WHAT SHOULD DO (Best Practices)

✅ ALWAYS USE explain("executionStats") — NOT JUST explain()
    → "queryPlanner" shows the plan but NOT actual performance
    → "executionStats" shows REAL numbers: time, docs examined
    → You need the REAL numbers to make decisions
    → Like Redshift: EXPLAIN alone is insufficient — need STL_ tables too

✅ CHECK THE GOLDEN RATIO FIRST
    → nReturned / totalDocsExamined should be close to 1.0
    → If ratio is < 0.01 (returning 1% of examined) → index problem
    → If ratio is ~1.0 → index is working perfectly
    → This single metric tells you 80% of the story

✅ LOOK FOR SORT STAGES
    → SORT in explain = in-memory sort = potential memory problem
    → If data is already sorted by index → no SORT stage needed
    → Compound index that covers filter + sort = eliminates SORT stage
    → Like Redshift: if ORDER BY matches SORTKEY → no sort needed

✅ LOOK FOR FETCH STAGES
    → FETCH means: go from index to full document
    → If query only needs indexed fields → covered query → no FETCH
    → PROJECTION_COVERED = fastest possible (no document reads)
    → Add frequently projected fields to index for covered queries

✅ RUN explain() AFTER EVERY INDEX CHANGE
    → Created new index → verify it's being used
    → Dropped index → verify no query regressed
    → Changed query → verify plan didn't change unexpectedly
    → Like Redshift: EXPLAIN after every schema change

✅ USE ATLAS PERFORMANCE ADVISOR IN PRODUCTION
    → Automatically suggests missing indexes
    → Shows slow queries with explain analysis
    → Like Redshift Advisor recommendations
    → Saves hours of manual explain() analysis

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT IGNORE COLLSCAN ON LARGE COLLECTIONS
    → COLLSCAN on 100 documents → fine (too small for index overhead)
    → COLLSCAN on 100K+ documents → almost always a problem
    → Exception: analytical queries that MUST scan everything
      (like Redshift full-column scans for aggregation — by design)

❌ DO NOT TRUST explain("queryPlanner") ALONE
    → Shows what MongoDB PLANS to do — not what ACTUALLY happened
    → Plan might look good but actual execution reveals problems
    → ALWAYS use "executionStats" for real diagnostics
    → Like Redshift: estimated rows vs actual rows can differ 100x

❌ DO NOT USE hint() AS A PERMANENT SOLUTION
    → hint() forces a specific index
    → Data distribution changes → forced index becomes suboptimal
    → Let query planner choose — fix the INDEX, not the PLAN
    → Use hint() only for debugging and temporary workarounds

❌ DO NOT FORGET: $match ORDER IN PIPELINE MATTERS
    → $match first → query planner can use indexes
    → $match after $group → NO index possible (data already transformed)
    → ALWAYS push $match as early as possible in pipeline
    → Like Redshift: WHERE before GROUP BY is fundamental optimization

❌ DO NOT ADD INDEXES WITHOUT CHECKING WRITE IMPACT
    → Every index = overhead on EVERY insert/update/delete
    → 20 indexes on one collection → 20x write amplification
    → explain() shows READ optimization — you must also consider WRITE cost
    → Project 32 covers this in depth

❌ DO NOT DIAGNOSE ON SMALL DATASETS
    → explain() on 100 documents → everything is fast
    → explain() on 10M documents → real problems visible
    → Test with PRODUCTION-SCALE data (or representative subset)
    → Like Redshift: EXPLAIN on empty table = meaningless

❌ DO NOT CONFUSE STAGE NAMES
    → IXSCAN ≠ fast (might scan entire index — check keysExamined)
    → COLLSCAN ≠ always bad (analytics queries on small collections)
    → FETCH ≠ bad (just means going from index to document)
    → SORT ≠ always bad (small sorts in memory are fine)
    → READ THE NUMBERS, not just the stage names

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Query Execution Engine — Deep Dive

WHEN YOU RUN: db.orders.find({ customer_id: "cust_5000" }).sort({ order_date: -1 })

THE COMPLETE INTERNAL PROCESS:

PHASE 1: QUERY PARSING
    → mongosh sends BSON command to mongod
    → Command: { find: "orders", 
                  filter: { customer_id: "cust_5000" },
                  sort: { order_date: -1 } }
    → mongod parses into internal query representation
    
    REDSHIFT EQUIVALENT:
    → psql sends SQL text to leader node
    → Leader node parses SQL → creates parse tree → logical plan

PHASE 2: QUERY PLANNING (Index Selection)
    → Query planner examines:
      a. All indexes on the collection
      b. Query filter fields
      c. Sort fields
      d. Projection fields
    
    → Generates CANDIDATE PLANS:
      Plan A: COLLSCAN (always a candidate — baseline)
      Plan B: IXSCAN on { customer_id: 1 } + SORT
      Plan C: IXSCAN on { customer_id: 1, order_date: -1 }
      Plan D: IXSCAN on { order_date: -1 } + FILTER
    
    → CHECK PLAN CACHE:
      → Same query shape seen before?
      → If cached plan exists AND still valid → skip competition
      → Cache key = query shape (field names, operators, sort)
    
    REDSHIFT EQUIVALENT:
    → Optimizer generates candidate plans
    → Uses STATISTICS (row counts, distinct values) for cost estimation
    → Picks lowest-cost plan (cost-based optimization)
    
    MONGODB DIFFERENCE:
    → NO cost-based optimization (no statistics-based estimates)
    → Instead: EMPIRICAL trial execution (actually run each plan briefly)

PHASE 3: PLAN COMPETITION (Empirical)
    → If plan cache miss: run ALL candidate plans simultaneously
    → Each plan races to produce results
    → First plan to reach 101 results (or finish) WINS
    → Winner cached for future queries with same shape
    
    ┌───────────────────────────────────────────────────┐
    │  Plan Competition (conceptual timeline):           │
    │                                                    │
    │  Plan A (COLLSCAN):  ████████████████████░░░░░░░░  │
    │  Plan B (IXSCAN+SORT): ████████░░░░░░░░░░░░░░░░░  │
    │  Plan C (IXSCAN compound): ███░ WINNER! (101 results)│
    │  Plan D (IXSCAN+FILTER): ██████████░░░░░░░░░░░░░  │
    │                                                    │
    │  Plan C reached 101 results first → selected       │
    │  Other plans STOPPED → saved as rejectedPlans      │
    └───────────────────────────────────────────────────┘

PHASE 4: EXECUTION (Winning Plan)
    → Execute winning plan to completion
    → For IXSCAN: walk B-tree index
    → For FETCH: load documents from WiredTiger cache (or disk)
    → For SORT: sort in memory (or spill to disk if > 100MB)
    → For LIMIT: stop after N results
    
    EXECUTION STAGES — BOTTOM TO TOP:
    
    LIMIT (stop after 20)
      ↑ receives documents
    FETCH (load full document from collection)
      ↑ receives document IDs from index
    IXSCAN (walk B-tree index, find matching keys)
      ↑ starts here — reads index from disk/cache
    
    REDSHIFT EQUIVALENT:
    → Execute plan across all compute slices in parallel
    → Each slice processes its portion (based on DISTKEY)
    → Leader node merges results from all slices
    
    KEY DIFFERENCE:
    → Redshift: PARALLEL execution across multiple nodes
    → MongoDB: SINGLE-THREAD execution (per query)
    → MongoDB sharded: scatter-gather across shards (like Redshift MPP)

PHASE 5: RESULTS
    → Documents streamed back to client via cursor
    → Cursor batches: default 101 documents first batch, then 16MB batches
    → Client iterates cursor to get more results
    
    REDSHIFT EQUIVALENT:
    → Results sent from compute slices to leader node
    → Leader node merges and returns to client
    → Large results: client uses DECLARE CURSOR / FETCH
    
    KEY DIFFERENCE:
    → Redshift: leader node is bottleneck for result merging

Redshift Internals — For Direct Comparison

REDSHIFT QUERY EXECUTION — SAME QUERY:

SELECT * FROM orders
WHERE customer_id = 'cust_5000'
ORDER BY order_date DESC
LIMIT 20;

PHASE 1: LEADER NODE — Parse + Optimize
    → Parse SQL → logical plan
    → Cost-based optimizer:
      → Check table statistics (pg_statistic)
      → Estimate row counts per step
      → Evaluate distribution strategy (DISTKEY routing)
      → Evaluate sort key usage (zone map filtering)
      → Generate SINGLE optimal plan (no competition)
    → Create physical plan with node assignments

PHASE 2: DISTRIBUTE — Send Plan to Compute Slices
    → DISTKEY(customer_id) = "cust_5000" routes to ONE slice
    → Only that slice processes the query (no broadcast!)
    → Like MongoDB: index targets specific data → skip the rest

PHASE 3: COMPUTE SLICE — Execute
    → Scan customer_id column blocks
    → Zone maps: skip blocks where max(customer_id) < "cust_5000"
                  or min(customer_id) > "cust_5000"
    → For matching blocks: decompress customer_id values
    → Filter: keep rows where customer_id = "cust_5000"
    → For matching rows: read order_date column (for sort)
    → Sort by order_date DESC
    → Limit to 20 rows
    → Read remaining columns for those 20 rows

PHASE 4: RETURN — Send Results to Leader
    → Compute slice sends 20 rows to leader node
    → Leader returns to client

WHY The Internals Cause Different Behavior

THE FUNDAMENTAL OPTIMIZATION DIFFERENCE:

REDSHIFT OPTIMIZES BY:
    1. Statistics → accurate row count estimates
    2. Distribution → minimize data movement
    3. Sort key → skip blocks via zone maps
    4. Columnar → read only needed columns
    5. MPP → parallel execution across slices
    
    → STRENGTHS: large analytical scans, complex JOINs
    → WEAKNESS: no B-tree indexes for point lookups
    → WEAKNESS: single sort key limits access patterns

MONGODB OPTIMIZES BY:
    1. B-tree indexes → direct key lookup (O(log n))
    2. Multiple indexes → multiple access patterns
    3. Compound indexes → cover filter + sort + projection
    4. Covered queries → skip document reads entirely
    5. Plan competition → empirical best-plan selection
    
    → STRENGTHS: point lookups, range queries, multiple access patterns
    → WEAKNESS: full-document reads (no columnar column pruning)
    → WEAKNESS: single-thread execution per query (no MPP)

THIS IS WHY:
    → MongoDB: add index → 1000x improvement for point queries
    → Redshift: change DISTKEY → 10x improvement for JOIN queries
    → Different optimization levers for different workloads
    → explain()/EXPLAIN reveals WHICH lever to pull

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE WITH explain():

PITFALL 1: "IXSCAN means my query is optimized"
    ❌ Seeing IXSCAN and assuming the query is fast
    → IXSCAN with keysExamined: 500000, nReturned: 10
    → Index USED but not SELECTIVE — scanning 500K keys for 10 results
    → Check the RATIO, not just the stage name
    ✅ Look at keysExamined vs nReturned ratio
    → If keysExamined >> nReturned → wrong index or need compound index

PITFALL 2: "I'll just run EXPLAIN like in Redshift"
    ❌ db.orders.find({...}).explain()  ← default = "queryPlanner" only
    → Shows plan but NOT actual execution stats
    → You see IXSCAN and think "great!" but don't see 500K docs examined
    ✅ ALWAYS use explain("executionStats")
    → Shows the REAL numbers: time, docs examined, keys examined
    → This is like Redshift EXPLAIN + STL_QUERY combined

PITFALL 3: "My query is fast on test data, so it's fine"
    ❌ explain() on 100 documents → COLLSCAN in 0ms → "looks fine"
    → Production has 10M documents → COLLSCAN in 30 seconds
    → COLLSCAN scales LINEARLY with collection size
    → IXSCAN scales LOGARITHMICALLY (O(log n))
    ✅ Test with production-scale data or extrapolate
    → 100 docs COLLSCAN in 0ms → 10M docs ≈ 100,000ms (100 seconds)
    → 100 docs IXSCAN in 0ms → 10M docs ≈ still 0-1ms

PITFALL 4: "I don't need to explain aggregation pipelines"
    ❌ Assuming aggregation is always efficient because it's a "pipeline"
    → Pipeline with COLLSCAN on first $match → scans everything
    → $group on unindexed field → processes all documents
    → $lookup without index on foreign collection → nested COLLSCAN
    ✅ db.coll.explain("executionStats").aggregate([...])
    → Check $cursor stage for index usage
    → Check each pipeline stage for document counts

PITFALL 5: "Redshift doesn't need indexes, so I'll use few in MongoDB"
    ❌ Redshift mentality: "sort key + distribution key = sufficient"
    → MongoDB has NO columnar column pruning
    → MongoDB has NO zone maps (only on time series buckets)
    → Indexes are THE optimization mechanism — there is nothing else
    ✅ Every query pattern needs an appropriate index
    → Unlike Redshift where one sort key serves many queries
    → MongoDB needs specific indexes for specific query shapes

PITFALL 6: "The query was fast yesterday, something changed"
    ❌ Assuming server issue when query suddenly slows down
    POSSIBLE CAUSES:
    → Plan cache was cleared → new plan competition → different winner
    → Data distribution changed → index less selective
    → Working set exceeded WiredTiger cache → disk reads
    → Index was dropped by another developer
    ✅ Re-run explain("executionStats") and compare with previous results
    → Check: same index used? Same docs examined? Same execution time?
    → Check: db.coll.getPlanCache().list() for plan changes

PITFALL 7: "I'll add an index for every slow query"
    ❌ 30 slow queries → 30 new indexes
    → Each index: 
      → Storage overhead (index files on disk + in memory)
      → Write amplification (every insert updates every index)
      → Memory pressure (indexes compete for WiredTiger cache)
    ✅ Analyze query patterns → find COMPOUND indexes that serve MULTIPLE queries
    → { customer_id: 1, order_date: -1, status: 1 }
    → Serves: customer lookup, customer+date range, customer+date+status
    → ONE compound index instead of THREE separate indexes

📈 SCALABILITY CONSIDERATIONS

HOW explain() BEHAVIOR CHANGES WITH SCALE:

SINGLE NODE (what we practiced):
    → explain() shows local execution plan
    → All stages run on one mongod process
    → Straightforward: COLLSCAN vs IXSCAN vs SORT

REPLICA SET (Project 23):
    → explain() runs on the node you're connected to
    → Primary: shows write + read plans
    → Secondary: shows read plans (with readPreference)
    → Plan is SAME across replicas (same indexes, same data)
    → BUT: secondary may have stale data → slightly different execution stats

SHARDED CLUSTER (Projects 21-22):
    → explain() output gets MORE COMPLEX
    → New stages appear:
      
      SINGLE_SHARD → query targets ONE shard (shard key in filter)
        → Like Redshift: DISTKEY in WHERE → one slice
        → BEST CASE for sharded queries
      
      SHARD_MERGE → query hits MULTIPLE shards, results merged
        → Like Redshift: query without DISTKEY → all slices
        → Results from each shard merged at mongos
      
      SHARD_MERGE_SORT → multi-shard with sort
        → Each shard sorts locally → mongos merge-sorts
        → Like Redshift: each slice sorts → leader merges
    
    → explain() shows PER-SHARD execution stats
    → You can see which shard was slow (like Redshift per-slice stats)

    SHARDED explain() OUTPUT:
    {
        queryPlanner: {
            winningPlan: {
                stage: "SINGLE_SHARD",  // or "SHARD_MERGE"
                shards: [
                    {
                        shardName: "shard-0",
                        winningPlan: {
                            stage: "IXSCAN",
                            // per-shard plan details
                        },
                        executionStats: {
                            nReturned: 20,
                            docsExamined: 20,
                            executionTimeMillis: 2
                        }
                    }
                    // more shards if SHARD_MERGE
                ]
            }
        }
    }

SCALING INSIGHT:
    → On single node: COLLSCAN on 10M docs = 30 seconds
    → On 10 shards: COLLSCAN on 1M docs/shard = 3 seconds (parallel)
    → BUT: scatter-gather overhead + network latency added
    → BETTER: proper index → IXSCAN on any scale = milliseconds
    → Sharding helps COLLSCAN but doesn't fix the root cause
    → Indexes fix the root cause regardless of topology

            from: "products",
            localField: "product_ids",
            foreignField: "_id",
            as: "products"

    → Embedding = pre-joined data = no $lookup = no JOIN cost

def analyze_explain(explain_result, label="Query"):
    """Extract and display key metrics from explain output"""
    
    print(f"\n{'='*60}")
    print(f"ANALYSIS: {label}")
    print(f"{'='*60}")
    
    # Handle find() explain output
    if "executionStats" in explain_result:
        stats = explain_result["executionStats"]
        planner = explain_result.get("queryPlanner", {})
        
        n_returned = stats.get("nReturned", 0)
        docs_examined = stats.get("totalDocsExamined", 0)
        keys_examined = stats.get("totalKeysExamined", 0)
        exec_time = stats.get("executionTimeMillis", 0)
        
        # Extract winning plan stage
        winning = planner.get("winningPlan", {})
        stages = extract_stages(winning)
        
        # Extract rejected plans count
        rejected = planner.get("rejectedPlans", [])
        
        print(f"\n  EXECUTION METRICS:")
        print(f"  ├── nReturned:          {n_returned:,}")
        print(f"  ├── totalDocsExamined:  {docs_examined:,}")
        print(f"  ├── totalKeysExamined:  {keys_examined:,}")
        print(f"  └── executionTimeMillis: {exec_time}ms")
        
        # Efficiency ratio
        if n_returned > 0:
            doc_ratio = docs_examined / n_returned
            key_ratio = keys_examined / n_returned if keys_examined > 0 else 0
            
            print(f"\n  EFFICIENCY:")
            print(f"  ├── Docs/Returned ratio: {doc_ratio:.1f}:1", end="")
            if doc_ratio <= 1.1:
                print(" ✅ PERFECT")
            elif doc_ratio <= 10:
                print(" ⚠️ ACCEPTABLE")
            else:
                print(f" ❌ SCANNING {doc_ratio:.0f}x MORE THAN NEEDED")
            
            print(f"  └── Keys/Returned ratio: {key_ratio:.1f}:1", end="")
            if key_ratio <= 1.5:
                print(" ✅ PERFECT")
            elif key_ratio <= 10:
                print(" ⚠️ ACCEPTABLE")
            else:
                print(f" ❌ INDEX NOT SELECTIVE ENOUGH")
        
        print(f"\n  PLAN STAGES: {' → '.join(stages)}")
        
        # Check for problems
        if "COLLSCAN" in stages:
            print(f"  ⚠️  WARNING: COLLSCAN detected — no index used!")
            print(f"       Add index for the filter fields")
        if "SORT" in stages:
            print(f"  ⚠️  WARNING: In-memory SORT detected")
            print(f"       Consider compound index covering filter + sort")
        if "PROJECTION_COVERED" in stages:
            print(f"  ✅ COVERED QUERY — zero document reads!")
        if keys_examined == 0 and docs_examined > 0:
            print(f"  ❌ NO INDEX USED — full collection scan")
        
        print(f"\n  REJECTED PLANS: {len(rejected)}")
        for i, plan in enumerate(rejected):
            rej_stages = extract_stages(plan)
            print(f"    Plan {i+1}: {' → '.join(rej_stages)}")
    
    # Handle aggregate() explain output
    elif "stages" in explain_result:
        print(f"\n  AGGREGATION PIPELINE EXPLAIN:")
        for i, stage in enumerate(explain_result["stages"]):
            stage_name = list(stage.keys())[0] if stage else "unknown"
            print(f"    Stage {i}: {stage_name}")
            if "$cursor" in stage:
                cursor_stats = stage["$cursor"].get("executionStats", {})
                print(f"      nReturned: {cursor_stats.get('nReturned', 'N/A')}")
                print(f"      docsExamined: {cursor_stats.get('totalDocsExamined', 'N/A')}")
                print(f"      keysExamined: {cursor_stats.get('totalKeysExamined', 'N/A')}")
                
                cursor_plan = stage["$cursor"].get("queryPlanner", {})
                winning = cursor_plan.get("winningPlan", {})
                stages_list = extract_stages(winning)
                print(f"      Plan: {' → '.join(stages_list)}")


def extract_stages(plan, stages=None):
    """Recursively extract stage names from explain plan tree"""
    if stages is None:
        stages = []
    
    if isinstance(plan, dict):
        stage = plan.get("stage", "")
        if stage:
            stages.append(stage)
        
        # Recurse into inputStage (single child)
        if "inputStage" in plan:
            extract_stages(plan["inputStage"], stages)
        
        # Recurse into inputStages (multiple children — e.g., OR)
        if "inputStages" in plan:
            for child in plan["inputStages"]:
                extract_stages(child, stages)
    
    return stages
