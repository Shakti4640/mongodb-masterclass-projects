PROJECT 16: Bucket Pattern — Time Series Grouping

Problem: Store Millions of IoT Sensor Readings Per Hour Efficiently

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    You're building an IoT monitoring platform for a smart factory
    
    THE NUMBERS:
    ├── 500 sensors on the factory floor
    ├── Each sensor sends a reading every 1 second
    ├── That's 500 readings/second = 30,000/minute = 1.8M/hour
    ├── That's 43.2 MILLION readings per day
    ├── That's 1.3 BILLION readings per month
    └── You need to store 12 months of data = ~15.7 BILLION readings
    
    EACH READING CONTAINS:
    ├── sensor_id: which sensor
    ├── timestamp: when the reading was taken
    ├── temperature: degrees Celsius
    ├── humidity: percentage
    ├── pressure: hPa
    └── vibration: mm/s
    
    ACCESS PATTERNS:
    ├── PATTERN A: "Show me sensor #42's readings for the last hour" (OLTP — real-time)
    ├── PATTERN B: "Average temperature per sensor per hour for last 24h" (dashboard)
    ├── PATTERN C: "Alert if any sensor exceeds threshold" (real-time monitoring)
    ├── PATTERN D: "Historical trend: daily averages for sensor #42 over 6 months" (analytics)
    └── PATTERN E: "Which sensors had the highest vibration variance this week?" (diagnostics)
    
    YOUR CHALLENGE:
    → At 1 document per reading: 15.7 BILLION documents
    → MongoDB index overhead: ~100 bytes per index entry
    → 15.7B × 100 bytes = 1.57 TB just for ONE index
    → WiredTiger overhead per document: ~50-100 bytes metadata
    → 15.7B × 75 bytes overhead = 1.18 TB just for document metadata
    → THE DATA ITSELF might only be 200 bytes per reading
    → OVERHEAD EXCEEDS DATA — something is very wrong
    
    THE BUCKET PATTERN SOLVES THIS

🔴 REDSHIFT MINDSET — How You Would Solve This

Step 1: One Row Per Reading (The Natural Approach)

-- ═══════════════════════════════════════════════════════
-- REDSHIFT: IoT Sensor Data Schema
-- One row per reading — the standard approach
-- ═══════════════════════════════════════════════════════

CREATE TABLE iot.sensor_readings (
    reading_id      BIGINT IDENTITY(1,1),
    sensor_id       INTEGER NOT NULL,
    reading_time    TIMESTAMP NOT NULL,
    temperature     DECIMAL(5,2),
    humidity        DECIMAL(5,2),
    pressure        DECIMAL(7,2),
    vibration       DECIMAL(6,3),
    
    -- No explicit PRIMARY KEY (Redshift PKs are informational only)
    -- Distribution + Sort keys are what matter
)
DISTSTYLE KEY
DISTKEY(sensor_id)                 -- co-locate all readings from same sensor
COMPOUND SORTKEY(sensor_id, reading_time)  -- optimize for sensor + time range queries
;

-- WHY THIS WORKS IN REDSHIFT:
-- 1. Columnar storage: each column stored separately
--    → SUM(temperature) reads ONLY temperature column
--    → 15.7B readings × 4 bytes (DECIMAL) = 63 GB for one column
--    → With compression (DELTA on temperature): ~15-20 GB
--    → FAST full-column scan
--
-- 2. Sort key on (sensor_id, reading_time):
--    → WHERE sensor_id = 42 AND reading_time BETWEEN ... 
--    → Zone maps skip 99%+ of blocks
--    → Reads only relevant sensor's time range
--
-- 3. Distribution key on sensor_id:
--    → All readings for sensor 42 on ONE compute slice
--    → No data movement for per-sensor queries
--    → GROUP BY sensor_id = local operation per slice
--
-- 4. Row overhead: minimal in columnar format
--    → No per-row metadata in column blocks
--    → Zone maps per 1MB block (not per row)
--    → 15.7B rows is perfectly fine for Redshift


-- ═══════════════════════════════════════════════════════
-- QUERY A: Last hour of readings for sensor 42
-- ═══════════════════════════════════════════════════════
SELECT reading_time, temperature, humidity, pressure, vibration
FROM iot.sensor_readings
WHERE sensor_id = 42
  AND reading_time >= DATEADD(hour, -1, GETDATE())
ORDER BY reading_time DESC;

-- PERFORMANCE: ~1-3 seconds
-- → Sort key skip: jumps directly to sensor 42's recent data
-- → Zone maps skip all blocks outside time range
-- → Reads ~3600 rows (1 per second × 3600 seconds)
-- → Columnar: reads only 5 columns needed


-- ═══════════════════════════════════════════════════════
-- QUERY B: Hourly averages per sensor for last 24h
-- ═══════════════════════════════════════════════════════
SELECT 
    sensor_id,
    DATE_TRUNC('hour', reading_time) AS hour,
    AVG(temperature) AS avg_temp,
    AVG(humidity) AS avg_humidity,
    MIN(temperature) AS min_temp,
    MAX(temperature) AS max_temp,
    COUNT(*) AS reading_count
FROM iot.sensor_readings
WHERE reading_time >= DATEADD(hour, -24, GETDATE())
GROUP BY sensor_id, DATE_TRUNC('hour', reading_time)
ORDER BY sensor_id, hour;

-- PERFORMANCE: ~3-10 seconds
-- → Scans 43.2M rows (24 hours of data)
-- → Columnar: reads only sensor_id, reading_time, temperature, humidity
-- → Parallel across all compute slices
-- → GROUP BY sensor_id is local (DISTKEY)


-- ═══════════════════════════════════════════════════════
-- QUERY D: Daily averages for 6 months (historical trend)
-- ═══════════════════════════════════════════════════════
SELECT 
    DATE_TRUNC('day', reading_time) AS day,
    AVG(temperature) AS avg_temp,
    STDDEV(temperature) AS temp_stddev,
    AVG(vibration) AS avg_vibration,
    MAX(vibration) AS max_vibration
FROM iot.sensor_readings
WHERE sensor_id = 42
  AND reading_time >= DATEADD(month, -6, GETDATE())
GROUP BY DATE_TRUNC('day', reading_time)
ORDER BY day;

-- PERFORMANCE: ~5-15 seconds
-- → Scans ~15.5M rows (180 days × 86400 readings/day for one sensor)
-- → Sort key skip: only reads sensor 42's blocks
-- → Columnar: reads only needed columns
-- → THIS IS WHAT REDSHIFT IS BUILT FOR

Step 2: What Redshift Does Well Here

REDSHIFT STRENGTHS FOR TIME SERIES IoT:
    ✅ Columnar storage → reads only needed columns from billions of rows
    ✅ Sort key on (sensor_id, timestamp) → perfect range scan efficiency
    ✅ Zone maps → skip 99%+ of data for targeted queries
    ✅ Parallel execution → 43M row scan distributed across all slices
    ✅ Compression → DELTA encoding on timestamps is extremely compact
    ✅ No per-row overhead → 15.7B rows is normal operational size
    ✅ SQL GROUP BY + aggregate functions → standard, well-optimized
    ✅ Window functions → LAG, LEAD, running averages trivially

REDSHIFT WEAKNESSES FOR THIS USE CASE:
    ❌ Individual INSERT per reading = terrible (columnar write penalty)
       → Must batch into COPY loads (every 5-15 minutes from S3/Kinesis)
    ❌ Real-time alerting = impossible (data arrives in batches)
    ❌ Single-sensor latest reading = overkill (MPP overhead for 1 row)
    ❌ High-frequency ingestion (500 writes/sec) = not designed for this
    ❌ VACUUM needed constantly as new data arrives in unsorted region
    ❌ Concurrent read + write = WLM contention

🟢 MONGODB MINDSET — The Naive Approach First, Then Bucket Pattern

Naive Approach: One Document Per Reading (DON'T DO THIS)

// NAIVE MONGODB: One document per sensor reading

// TOTAL useful data per document: ~56 bytes
// ACTUAL storage per document:
//   → BSON encoding overhead: ~120 bytes (field names repeated in every doc)
//   → WiredTiger metadata per document: ~50-100 bytes
//   → _id index entry: ~100 bytes
//   → compound index entry: ~80 bytes
//   → TOTAL per document: ~400-500 bytes
//
// AT SCALE (15.7 BILLION documents):
//   → Useful data: 15.7B × 56 bytes = 880 GB
//   → Actual storage: 15.7B × 450 bytes = 7.1 TB
//   → Index on (sensor_id, timestamp): 15.7B × 80 bytes = 1.26 TB
//   → _id index: 15.7B × 100 bytes = 1.57 TB
//   → TOTAL: ~10 TB for 880 GB of useful data
//   → OVERHEAD RATIO: 11:1 (!!!)
//
// COMPARED TO REDSHIFT:
//   → Columnar + compression: ~500 GB - 1 TB for same data
//   → MongoDB naive approach: 10 TB
//   → 10-20x MORE storage for same data


// ADDITIONAL PROBLEMS:

// PROBLEM 1: WiredTiger cache pressure
// → 15.7B documents × 450 bytes = can't fit working set in RAM
// → Constant cache eviction → disk reads → high latency
// → Need MASSIVE RAM (or accept slow queries)

// PROBLEM 2: Index size exceeds RAM
// → 1.26 TB compound index cannot fit in memory
// → Index lookups hit disk → 10-100x slower than in-memory
// → Redshift zone maps: ~1 MB per 1 million rows = 15.7 GB total

// PROBLEM 3: Document count overhead
// → MongoDB must track 15.7 BILLION documents
// → Collection metadata, free list management, B-tree depth
// → B-tree with 15.7B entries = very deep tree → more I/O per lookup

// PROBLEM 4: Write amplification
// → 500 insertOne() calls per second
// → Each insert: write document + update _id index + update compound index
// → 500 × 3 = 1500 write operations per second
// → Journal writes for each → high I/O

// VERDICT: ❌ DO NOT USE ONE-DOCUMENT-PER-READING AT THIS SCALE

The Bucket Pattern — THE Solution

CORE IDEA:
    Instead of ONE document per reading...
    → Store ONE document per sensor per TIME BUCKET
    → Each bucket document contains an ARRAY of readings
    → Bucket size: 1 minute, 1 hour, or 1 day (depends on volume)
    
    For our use case (1 reading/second per sensor):
    → 1-HOUR buckets: 3600 readings per bucket document
    → 500 sensors × 24 hours = 12,000 documents per day
    → vs naive: 500 × 86,400 = 43,200,000 documents per day
    → REDUCTION: 3,600x fewer documents!

🔧 HOW TO APPROACH IT IN MONGODB

BUCKET PATTERN DESIGN DECISIONS:

DECISION 1: Bucket Size (time granularity)
    ┌─────────────────────────────────────────────────────────┐
    │ Reading Frequency  │ Recommended Bucket │ Docs per Bucket│
    ├────────────────────┼────────────────────┼────────────────┤
    │ 1/second           │ 1 hour             │ 3,600          │
    │ 1/minute           │ 1 day              │ 1,440          │
    │ 1/hour             │ 1 week or month    │ 168 / 720      │
    │ 10/second          │ 10 minutes         │ 6,000          │
    └────────────────────┴────────────────────┴────────────────┘
    
    RULE OF THUMB:
    → Target 200-3000 readings per bucket
    → Keep bucket document under 1-2 MB
    → Each reading ~50-80 bytes in array
    → 3000 readings × 70 bytes = ~210 KB per bucket doc (good)
    → 3600 readings × 70 bytes = ~250 KB per bucket doc (good)
    
    For our case: 1 reading/second → 1-HOUR bucket = 3600 readings ✅

DECISION 2: Bucket Key
    → What identifies a unique bucket?
    → { sensor_id + bucket_start_time }
    → Compound: one bucket per sensor per hour
    → This becomes the index key AND shard key candidate

DECISION 3: Pre-computed Statistics
    → Include min/max/avg/count INSIDE the bucket document
    → Updated on every $push with $inc and $min/$max
    → Dashboard queries read stats DIRECTLY — no aggregation needed
    → Like Redshift materialized views but updated in real-time

DECISION 4: Bucket Lifecycle
    → "Open" bucket: current hour, still receiving readings
    → "Closed" bucket: past hour, no more writes
    → Closed buckets are IMMUTABLE — great for caching and archival

// STEP 2: The Bucket Write Operation
// THE KEY FUNCTION — how data enters the bucket

function getHourBoundary(date) {
    // Round down to hour boundary
    return new Date(
        date.getFullYear(),
        date.getMonth(),
        date.getDate(),
        date.getHours(),
        0, 0, 0
    )
}

function addReading(sensorId, timestamp, temperature, humidity, pressure, vibration) {
    /**
     * Add a sensor reading to the appropriate hourly bucket.
     * 
     * This uses upsert: 
     *   → If bucket exists: $push reading + update stats
     *   → If bucket doesn't exist: create it with first reading
     * 
     * SINGLE ATOMIC OPERATION — no race conditions
     * 
     * Redshift equivalent:
     *   INSERT INTO iot.sensor_readings 
     *   (sensor_id, reading_time, temperature, humidity, pressure, vibration)
     *   VALUES (42, '2024-07-20 14:30:01', 23.45, 65.2, 1013.25, 0.342);
     *   -- Simple INSERT but: columnar write overhead, needs VACUUM, no real-time
     */
    
    const bucketStart = getHourBoundary(timestamp)
    const bucketEnd = new Date(bucketStart.getTime() + 60 * 60 * 1000)  // +1 hour
    
    const reading = {
        ts: timestamp,
        t: temperature,
        h: humidity,
        p: pressure,
        v: vibration
    }
    
    db.sensor_readings.updateOne(
        {
            // FIND: the bucket for this sensor and hour
            sensor_id: sensorId,
            bucket_start: bucketStart,
            // Only update if bucket isn't "full" (safety check)
            count: { $lt: 3600 }
        },
        {
            // UPDATE: push reading + update stats
            $push: { readings: reading },
            $inc: { 
                count: 1,
                "stats.temperature.sum": temperature,
                "stats.humidity.sum": humidity,
                "stats.pressure.sum": pressure,
                "stats.vibration.sum": vibration
            },
            $min: {
                "stats.temperature.min": temperature,
                "stats.humidity.min": humidity,
                "stats.pressure.min": pressure,
                "stats.vibration.min": vibration
            },
            $max: {
                "stats.temperature.max": temperature,
                "stats.humidity.max": humidity,
                "stats.pressure.max": pressure,
                "stats.vibration.max": vibration
            },
            $set: {
                bucket_end: bucketEnd,
                last_reading_at: timestamp
            },
            $setOnInsert: {
                // These fields set ONLY when creating a new bucket
                sensor_id: sensorId,
                bucket_start: bucketStart,
                is_full: false,
                created_at: new Date()
            }
        },
        {
            upsert: true  // Create bucket if it doesn't exist
        }
    )
}

def get_hour_boundary(dt):
    """Round datetime down to hour boundary"""
    return dt.replace(minute=0, second=0, microsecond=0)


def add_reading(sensor_id, timestamp, temperature, humidity, pressure, vibration):
    """
    Add a sensor reading to the appropriate hourly bucket.
    
    Uses upsert + atomic operators:
    → If bucket exists: $push reading + update stats
    → If bucket doesn't exist: create with $setOnInsert
    
    SINGLE ATOMIC OPERATION — thread-safe, no race conditions
    
    Redshift equivalent:
        INSERT INTO iot.sensor_readings 
        (sensor_id, reading_time, temperature, humidity, pressure, vibration)
        VALUES (%s, %s, %s, %s, %s, %s);
    """
    bucket_start = get_hour_boundary(timestamp)
    bucket_end = bucket_start + timedelta(hours=1)
    
    reading = {
        "ts": timestamp,
        "t": round(temperature, 2),
        "h": round(humidity, 2),
        "p": round(pressure, 2),
        "v": round(vibration, 3)
    }
    
    db.sensor_readings.update_one(
        {
            "sensor_id": sensor_id,
            "bucket_start": bucket_start,
            "count": {"$lt": 3600}  # safety: don't exceed bucket capacity
        },
        {
            "$push": {"readings": reading},
            "$inc": {
                "count": 1,
                "stats.temperature.sum": temperature,
                "stats.humidity.sum": humidity,
                "stats.pressure.sum": pressure,
                "stats.vibration.sum": vibration
            },
            "$min": {
                "stats.temperature.min": temperature,
                "stats.humidity.min": humidity,
                "stats.pressure.min": pressure,
                "stats.vibration.min": vibration
            },
            "$max": {
                "stats.temperature.max": temperature,
                "stats.humidity.max": humidity,
                "stats.pressure.max": pressure,
                "stats.vibration.max": vibration
            },
            "$set": {
                "bucket_end": bucket_end,
                "last_reading_at": timestamp
            },
            "$setOnInsert": {
                "sensor_id": sensor_id,
                "bucket_start": bucket_start,
                "is_full": False,
                "created_at": datetime.now()
            }
        },
        upsert=True
    )


def finalize_bucket_stats():
    """
    Compute averages for all buckets.
    Run periodically or when bucket closes.

            }}

Redshift Internals — Why Columnar Handles This Differently

REDSHIFT: 15.7B rows, columnar storage

    Column: sensor_id (INTEGER, 4 bytes)
    ┌────────────────────────────────────────────┐
    │ Block 1: [42, 42, 42, ... 42]  min=42 max=42│  ← 1MB block
    │ Block 2: [42, 42, 42, ... 42]  min=42 max=42│     all same sensor
    │ ...                                          │
    │ Block N: [43, 43, 43, ... 43]  min=43 max=43│  ← next sensor
    └────────────────────────────────────────────┘
    Sort key: (sensor_id, reading_time) → blocks are sorted
    Zone map: min/max per block → skip blocks for wrong sensor
    
    Column: temperature (DECIMAL 5,2)
    ┌────────────────────────────────────────────┐
    │ Block 1: [23.45, 23.48, 23.44, ... ]       │
    │ Block 2: [23.51, 23.49, 23.52, ... ]       │
    │ Compression: DELTA encoding                  │
    │   → Store differences: [23.45, +0.03, -0.04]│
    │   → 2-3 bytes per value instead of 4         │
    │   → 15.7B × 2.5 bytes = ~39 GB (compressed) │
    └────────────────────────────────────────────┘
    
    AVG(temperature) query:
    → Reads ONLY temperature column blocks
    → Skips sensor_id, humidity, pressure, vibration columns entirely
    → ~39 GB scan instead of full table
    → Distributed across 32 slices = ~1.2 GB per slice
    → Parallel scan: completes in seconds
    
    THIS IS WHY REDSHIFT WINS FOR ANALYTICAL SCANS
    → Columnar storage reads only needed columns
    → MongoDB (even with buckets) reads entire documents
    → For SUM/AVG across billions of values: Redshift is fundamentally better

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "One Document Per Reading" (Redshift Developer Default)
    ❌ Directly translating Redshift rows to MongoDB documents
    SYMPTOMS:
        → Billions of tiny documents
        → Index size exceeds RAM
        → Queries slow down as data grows
        → Storage 5-10x larger than expected
    FIX: Apply bucket pattern — group readings by time window

PITFALL 2: Bucket Too Large (Exceeding 16MB)
    ❌ Using 1-day buckets with 1 reading/second = 86,400 readings
       86,400 × ~70 bytes = ~6 MB (OK)
       But with 10 readings/second = 864,000 × 70 = 60 MB (EXCEEDS 16MB!)
    FIX: Choose bucket size based on reading frequency
         Target 200-3000 readings per bucket
         For 10/sec: use 5-minute buckets (3000 readings)

PITFALL 3: Not Pre-Computing Statistics
    ❌ Storing raw readings in buckets but computing stats at query time
    SYMPTOMS:
        → Dashboard queries still slow (must $unwind + $group)
        → Bucket pattern benefits partially lost
    FIX: Maintain stats with $inc, $min, $max on every write
         Dashboard reads stats directly — no aggregation needed

PITFALL 4: Long Field Names in Readings Array
    ❌ { temperature: 23.45, humidity: 65.2, pressure: 1013.25 }
       × 3600 readings = enormous field name overhead
    FIX: Use short names: { t: 23.45, h: 65.2, p: 1013.25 }
         Document application-level mapping: t = temperature
         Save 100+ KB per bucket document

PITFALL 5: Not Using TTL for Data Lifecycle
    ❌ Manually deleting old data with scheduled scripts
    FIX: TTL index on bucket_start automatically removes old buckets
         Redshift equivalent: partition management (more complex)

PITFALL 6: Querying Individual Readings Across Buckets
    ❌ "Find all readings where temperature > 90" across all buckets
       Must $unwind every bucket → defeats the purpose
    FIX: Use pre-computed stats.temperature.max to find candidate buckets
         THEN drill into only those buckets for specific readings
         Two-phase query: filter buckets → drill into matches
