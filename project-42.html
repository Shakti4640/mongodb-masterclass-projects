-- REDSHIFT: The analytical queries you'd write

-- 1. Transaction volume per region per minute (last hour)
SELECT
    DATE_TRUNC('minute', timestamp) AS minute,
    country,
    COUNT(*) AS txn_count,
    SUM(CAST(amount AS DECIMAL(18,2))) AS total_volume,
    AVG(CAST(amount AS DECIMAL(18,2))) AS avg_amount
FROM transactions
WHERE timestamp >= DATEADD(hour, -1, GETDATE())
GROUP BY 1, 2
ORDER BY 1 DESC, 2;
-- Execution time: 2-8 seconds (scanning millions of recent rows)
-- Data freshness: 15-30 minutes behind real-time (ETL lag)

-- 2. Rolling 24-hour revenue trend (1-minute buckets)
SELECT
    DATE_TRUNC('minute', timestamp) AS minute,
    SUM(CASE WHEN amount > 0 THEN CAST(amount AS DECIMAL(18,2)) ELSE 0 END) AS revenue,
    COUNT(*) AS txn_count,
    SUM(SUM(CAST(amount AS DECIMAL(18,2)))) OVER (
        ORDER BY DATE_TRUNC('minute', timestamp)
        ROWS BETWEEN 59 PRECEDING AND CURRENT ROW
    ) AS rolling_60min_revenue
FROM transactions
WHERE timestamp >= DATEADD(hour, -24, GETDATE())
  AND type IN ('purchase', 'deposit', 'transfer')
GROUP BY 1
ORDER BY 1;
-- Execution time: 5-15 seconds (24 hours of data, 1440 buckets)
-- Window function: rolling 60-minute sum

-- 3. Fraud detection (transactions > $10,000)
-- THIS IS WHERE REDSHIFT FAILS FOR REAL-TIME:
SELECT *
FROM transactions
WHERE ABS(CAST(amount AS DECIMAL(18,2))) > 10000
  AND timestamp >= DATEADD(minute, -5, GETDATE())

🟢 MONGODB MINDSET — How This Same Problem Is Approached

The MongoDB Real-Time Analytics Philosophy

MONGODB'S FUNDAMENTAL ADVANTAGE FOR REAL-TIME:
    → Your operational data IS your analytical data
    → No ETL pipeline between "source" and "analytics"
    → Query the LIVE collection directly
    → Data freshness: ZERO latency (you're reading live data)

    BUT THE RISK:
    → Analytical queries on operational collections can degrade OLTP performance
    → Aggregation pipeline scanning millions of docs = heavy load
    → This is WHY Redshift exists — purpose-built for analytical scans

    MONGODB'S SOLUTIONS:
    1. Secondary reads — route analytics to replica secondaries
    2. Analytics nodes — dedicated replica members for analytics only
    3. Pre-aggregation — compute metrics at write time (Project 20)
    4. Change Streams — push-based real-time (Project 25)
    5. MongoDB Charts — native visualization with live connection
    6. Time Series Collections — optimized for time-based analytics (Project 30)

    THE ARCHITECTURE:
    ┌──────────────┐
    │ Application   │ ──writes──→ ┌──────────────────┐
    │ (OLTP traffic)│              │ MongoDB Primary   │
    └──────────────┘              │ (handles writes)  │
                                  └────────┬─────────┘
                                           │ replication
                          ┌────────────────┼────────────────┐
                          ▼                ▼                ▼
                   ┌──────────┐    ┌──────────┐    ┌────────────┐
                   │ Secondary │    │ Secondary │    │ Analytics  │
                   │ (OLTP     │    │ (OLTP     │    │ Node       │
                   │  failover)│    │  failover)│    │ (dedicated │
                   └──────────┘    └──────────┘    │  analytics) │
                                                    └──────┬─────┘
                                                           │
                                                    ┌──────┴─────┐
                                                    │ MongoDB     │
                                                    │ Charts      │
                                                    │ (dashboard) │
                                                    └────────────┘

    ANALYTICS NODE:
    → Replica member tagged { nodeType: "ANALYTICS" }
    → Never becomes primary (priority: 0)
    → Never serves OLTP reads (hidden from default read preference)
    → Dedicated to analytical queries and Charts connections
    → ZERO impact on operational performance
    → THIS is MongoDB's answer to "analytics on live data"

🧩 HOW TO APPROACH IT IN MONGODB

REAL-TIME ANALYTICS STRATEGY — THREE LAYERS:

LAYER 1: PRE-COMPUTED METRICS (fastest, Project 20 pattern)
    → Compute at WRITE time, not READ time
    → $inc counters on every transaction insert
    → Dashboard reads pre-computed values: <5ms
    → Best for: KPIs, counters, running totals
    → Trade-off: more write complexity

LAYER 2: AGGREGATION ON LIVE DATA (medium speed)
    → Run aggregation pipeline on operational data
    → Use analytics node (readPreference: secondary + tag)
    → Dashboard polls every 10-30 seconds
    → Best for: trends, distributions, top-N reports
    → Trade-off: query latency proportional to data scanned

LAYER 3: CHANGE STREAM PUSH (true real-time)
    → Change stream watches for specific events
    → Push to dashboard via WebSocket / SSE
    → Best for: alerts, live feeds, real-time counters
    → Trade-off: complexity of stream processing

WHICH LAYER FOR EACH REQUIREMENT:
    ┌─────────────────────────────────────────────────────────┐
    │ Operations: txn volume/minute     → Layer 1 (counters)  │
    │ Finance: 24h revenue trend        → Layer 2 (aggregation)│
    │ Risk: fraud alerts <5 sec         → Layer 3 (change stream)│
    │ Product: signup rate              → Layer 1 (counters)  │
    │ Executive: global KPIs           → Layer 1 + 2 (hybrid)│

Layer 1: Pre-Computed Metrics Collection

// ═══════════════════════════════════════════════════════
// LAYER 1: PRE-COMPUTED METRICS
// Write-time aggregation — counters updated on every transaction
// Reference: Project 20 (Computed Pattern)
// ═══════════════════════════════════════════════════════

// ─── METRICS COLLECTION DESIGN ───────────────────────
// One document per metric per time bucket
// Updated atomically with $inc on every transaction insert

// Document structure for per-minute transaction metrics:
{
    _id: {
        metric: "txn_volume",
        country: "DE",
        minute: ISODate("2024-06-01T14:30:00Z")    // truncated to minute
    },
    count: 847,                         // number of transactions
    total_amount: NumberDecimal("125430.50"),
    max_amount: NumberDecimal("9500.00"),
    min_amount: NumberDecimal("0.50"),
    currency_breakdown: {
        EUR: { count: 720, total: NumberDecimal("108200.00") },
        USD: { count: 127, total: NumberDecimal("17230.50") }
    },
    type_breakdown: {
        purchase: 512,
        transfer: 245,
        deposit: 90
    },
    last_updated: ISODate("2024-06-01T14:30:58Z")
}

// ─── UPDATE METRICS ON EVERY TRANSACTION INSERT ──────
// This runs IN the application, ALONGSIDE the transaction insert

function recordTransaction(txnDoc) {
    // Step 1: Insert the actual transaction
    db.transactions.insertOne(txnDoc)
    
    // Step 2: Update per-minute metrics (atomic $inc)
    let minuteBucket = new Date(txnDoc.timestamp)
    minuteBucket.setSeconds(0, 0)  // truncate to minute
    
    let amount = parseFloat(txnDoc.amount.toString())
    let absAmount = Math.abs(amount)
    
    // Atomic update — no race conditions even at 50,000 txn/min
    db.metrics_realtime.updateOne(
        {
            _id: {
                metric: "txn_volume",
                country: txnDoc.country,
                minute: minuteBucket
            }
        },
        {
            $inc: {
                count: 1,
                total_amount: absAmount,
                [`type_breakdown.${txnDoc.type}`]: 1,
                [`currency_breakdown.${txnDoc.currency}.count`]: 1,
                [`currency_breakdown.${txnDoc.currency}.total`]: absAmount
            },
            $max: { max_amount: absAmount },
            $min: { min_amount: absAmount },
            $set: { last_updated: new Date() }
        },
        { upsert: true }   // create if doesn't exist for this minute
    )
    
    // Step 3: Update per-hour rollup (for trend charts)
    let hourBucket = new Date(txnDoc.timestamp)
    hourBucket.setMinutes(0, 0, 0)
    
    db.metrics_hourly.updateOne(
        {
            _id: {
                metric: "txn_volume",
                country: txnDoc.country,
                hour: hourBucket
            }
        },
        {
            $inc: { count: 1, total_amount: absAmount },
            $set: { last_updated: new Date() }
        },
        { upsert: true }
    )
    
    // Step 4: Update global daily KPI
    let dayBucket = new Date(txnDoc.timestamp)
    dayBucket.setHours(0, 0, 0, 0)
    
    db.metrics_daily.updateOne(
        {
            _id: { metric: "global_kpi", day: dayBucket }
        },
        {
            $inc: {
                total_transactions: 1,
                total_volume: absAmount,
                [`by_country.${txnDoc.country}.count`]: 1,
                [`by_country.${txnDoc.country}.volume`]: absAmount
            },
            $set: { last_updated: new Date() }
        },
        { upsert: true }
    )
}

Layer 2: Aggregation on Live Data (Analytics Node)

// ═══════════════════════════════════════════════════════
// LAYER 2: AGGREGATION PIPELINE ON LIVE DATA
// For complex analytics that can't be pre-computed
// Runs on ANALYTICS NODE (zero OLTP impact)
// ═══════════════════════════════════════════════════════

// ─── CONFIGURE ANALYTICS NODE READ PREFERENCE ───────
// In mongosh, target the analytics node specifically:
// readPreference: "secondary", readPreferenceTags: [{ nodeType: "ANALYTICS" }]

// ─── QUERY 1: Rolling 24-hour revenue trend ─────────
// 1-minute granularity with rolling 60-minute window
// Uses $setWindowFields (Project 13 — window functions)

function getRevenueTrend(hoursBack = 24) {
    let cutoff = new Date()
    cutoff.setHours(cutoff.getHours() - hoursBack)
    
    return db.transactions.aggregate([
        // Stage 1: Filter recent transactions
        { $match: {
            timestamp: { $gte: cutoff },
            type: { $in: ["purchase", "deposit", "transfer"] },
            amount: { $gt: 0 }
        }},
        
        // Stage 2: Bucket into 1-minute intervals
        { $group: {
            _id: {
                minute: {
                    $dateTrunc: {
                        date: "$timestamp",
                        unit: "minute"
                    }
                }
            },
            revenue: { $sum: { $toDouble: "$amount" } },
            txn_count: { $sum: 1 },
            avg_amount: { $avg: { $toDouble: "$amount" } },
            unique_accounts: { $addToSet: "$account_id" }
        }},
        
        // Stage 3: Sort by time
        { $sort: { "_id.minute": 1 } },
        
        // Stage 4: Rolling 60-minute window
        // Equivalent to Redshift:
        //   SUM(revenue) OVER (ORDER BY minute ROWS BETWEEN 59 PRECEDING AND CURRENT ROW)
        { $setWindowFields: {
            sortBy: { "_id.minute": 1 },
            output: {
                rolling_60min_revenue: {
                    $sum: "$revenue",
                    window: { documents: [-59, 0] }
                },
                rolling_60min_txn_count: {
                    $sum: "$txn_count",
                    window: { documents: [-59, 0] }
                },
                revenue_rank: {
                    $rank: {}
                },
                pct_change_from_prev: {
                    $shift: {
                        output: "$revenue",
                        by: -1,
                        default: 0
                    }
                }
            }
        }},
        
        // Stage 5: Calculate derived metrics
        { $addFields: {
            unique_account_count: { $size: "$unique_accounts" },
            pct_change: {
                $cond: {
                    if: { $eq: ["$pct_change_from_prev", 0] },
                    then: 0,
                    else: {
                        $multiply: [
                            { $divide: [
                                { $subtract: ["$revenue", "$pct_change_from_prev"] },
                                "$pct_change_from_prev"
                            ]},
                            100
                        ]
                    }
                }
            }
        }},
        
        // Stage 6: Project final shape
        { $project: {
            minute: "$_id.minute",
            revenue: { $round: ["$revenue", 2] },
            txn_count: 1,
            avg_amount: { $round: ["$avg_amount", 2] },
            unique_accounts: "$unique_account_count",
            rolling_60min_revenue: { $round: ["$rolling_60min_revenue", 2] },
            rolling_60min_txn_count: 1,
            pct_change: { $round: ["$pct_change", 1] },
            _id: 0
        }},
        
        // Only return last 2 hours for the chart
        { $match: {
            minute: { $gte: new Date(Date.now() - 2 * 60 * 60 * 1000) }
        }}
        
    ], {
        // Route to analytics node
        readPreference: "secondary",
        readPreferenceTags: [{ nodeType: "ANALYTICS" }],
        // Allow disk use for large sorts
        allowDiskUse: true,
        // Set max time to prevent runaway queries
        maxTimeMS: 30000
    }).toArray()
}

function getTopMerchants(country, hoursBack = 24, limit = 10) {
    let cutoff = new Date()
    cutoff.setHours(cutoff.getHours() - hoursBack)
    
    return db.transactions.aggregate([
        { $match: {
            country: country,
            type: "purchase",
            timestamp: { $gte: cutoff },
            merchant: { $exists: true }
        }},
        { $group: {
            _id: "$merchant",
            txn_count: { $sum: 1 },
            total_volume: { $sum: { $abs: { $toDouble: "$amount" } } },
            avg_amount: { $avg: { $abs: { $toDouble: "$amount" } } },
            unique_users: { $addToSet: "$account_id" }
        }},
        { $addFields: {
            unique_user_count: { $size: "$unique_users" }
        }},
        { $sort: { total_volume: -1 } },
        { $limit: limit },
        { $project: {
            merchant: "$_id",
            txn_count: 1,
            total_volume: { $round: ["$total_volume", 2] },
            avg_amount: { $round: ["$avg_amount", 2] },
            unique_users: "$unique_user_count",
            _id: 0
        }}
    ], {
        readPreference: "secondary",
        readPreferenceTags: [{ nodeType: "ANALYTICS" }],
        allowDiskUse: true
    }).toArray()

function getCohortConversion(daysBack = 30) {
    let cutoff = new Date()
    cutoff.setDate(cutoff.getDate() - daysBack)
    
    return db.users.aggregate([
        { $match: {
            created_at: { $gte: cutoff }
        }},
        // Lookup first transaction per user
        { $lookup: {
            from: "transactions",
            let: { uid: "$user_id", ucountry: "$country" },
            pipeline: [
                { $match: {
                    $expr: {
                        $and: [
                            { $eq: ["$account_id", "$$uid"] },
                            { $eq: ["$country", "$$ucountry"] }
                        ]
                    }
                }},
                { $sort: { timestamp: 1 } },
                { $limit: 1 }
            ],
            as: "first_txn"
        }},
        // Calculate days to first transaction
        { $addFields: {
            signup_week: { $dateTrunc: { date: "$created_at", unit: "week" } },
            has_transacted: { $gt: [{ $size: "$first_txn" }, 0] },
            days_to_first_txn: {
                $cond: {
                    if: { $gt: [{ $size: "$first_txn" }, 0] },
                    then: {
                        $dateDiff: {
                            startDate: "$created_at",
                            endDate: { $arrayElemAt: ["$first_txn.timestamp", 0] },
                            unit: "day"
                        }
                    },
                    else: null
                }
            }
        }},
        // Group by signup week
        { $group: {
            _id: "$signup_week",
            total_signups: { $sum: 1 },
            converted: { $sum: { $cond: ["$has_transacted", 1, 0] } },
            avg_days_to_convert: { $avg: "$days_to_first_txn" }
        }},
        { $addFields: {
            conversion_rate: {
                $round: [
                    { $multiply: [{ $divide: ["$converted", "$total_signups"] }, 100] },
                    1
                ]
            }
        }},
        { $sort: { _id: 1 } }
    ], {
        readPreference: "secondary",
        readPreferenceTags: [{ nodeType: "ANALYTICS" }],
        allowDiskUse: true,
        maxTimeMS: 60000    // 60 second limit for complex query
    }).toArray()
}

def record_transaction_with_metrics(txn_doc):
    """
    Insert transaction AND update all metric layers atomically
    
    Redshift equivalent:
        INSERT INTO transactions VALUES (...);
        -- Then wait for ETL to update materialized views
        -- No real-time metric update capability
    """
    
    wc = WriteConcern(w="majority")
    
    # ─── Insert transaction ───────────────────────────
    db.get_collection("transactions", write_concern=wc).insert_one(txn_doc)
    
    # ─── Extract values ───────────────────────────────
    amount = abs(float(str(txn_doc["amount"])))
    country = txn_doc["country"]
    currency = txn_doc["currency"]
    txn_type = txn_doc["type"]
    ts = txn_doc["timestamp"]
    
    # ─── Update minute-level metrics ──────────────────
    minute_bucket = ts.replace(second=0, microsecond=0)
    
    db.metrics_realtime.update_one(
        {
            "_id": {
                "metric": "txn_volume",
                "country": country,
                "minute": minute_bucket
            }
        },
        {
            "$inc": {
                "count": 1,
                "total_amount": amount,
                f"type_breakdown.{txn_type}": 1,
                f"currency_breakdown.{currency}.count": 1,
                f"currency_breakdown.{currency}.total": amount
            },
            "$max": {"max_amount": amount},
            "$min": {"min_amount": amount},
            "$set": {"last_updated": datetime.now()}
        },
        upsert=True
    )
    
    # ─── Update hourly metrics ────────────────────────
    hour_bucket = ts.replace(minute=0, second=0, microsecond=0)
    
    db.metrics_hourly.update_one(
        {
            "_id": {
                "metric": "txn_volume",
                "country": country,
                "hour": hour_bucket
            }
        },
        {
            "$inc": {"count": 1, "total_amount": amount},
            "$set": {"last_updated": datetime.now()}
        },
        upsert=True
    )
    
    # ─── Update daily KPI ─────────────────────────────
    day_bucket = ts.replace(hour=0, minute=0, second=0, microsecond=0)
    
    db.metrics_daily.update_one(
        {"_id": {"metric": "global_kpi", "day": day_bucket}},
        {
            "$inc": {
                "total_transactions": 1,
                "total_volume": amount,
                f"by_country.{country}.count": 1,
                f"by_country.{country}.volume": amount
            },
            "$set": {"last_updated": datetime.now()}
        },
        upsert=True
    )
    
    return txn_doc["txn_id"]

def start_fraud_monitor(duration_seconds=60):

    print(f"{'=' * 70}")
    
    # Pipeline filter: only high-value completed transactions
    pipeline = [
        {"$match": {
            "operationType": "insert",

    total_processed = 0

            pipeline,

                    event = stream.try_next()
                    # Stream already advanced, we need to handle differently

                    pass

                for event in stream:
                    total_processed += 1

def start_live_counter(duration_seconds=30):
    """
    Real-time transaction counter — feeds live dashboard
    Pushes per-second counts by country
    
    Redshift equivalent:
        -- SELECT COUNT(*) FROM transactions
        --   WHERE timestamp >= DATEADD(second, -1, GETDATE());
        -- Must poll repeatedly (1 query/second = 1 WLM slot consumed)
        -- Each query: 1-3 seconds execution
        -- Result: always 1-3 seconds behind
        
    MongoDB change stream:
        -- Push-based: zero polling
        -- Latency: <100ms per event
        -- No query execution overhead
    """
    
    print(f"\n{'=' * 70}")
    print(f"📊 LIVE TRANSACTION COUNTER ({duration_seconds}s)")
    print(f"{'=' * 70}")
    
    counters = {}
    last_print = time.time()
    total_events = 0
    start_time = time.time()
    
    pipeline = [{"$match": {"operationType": "insert"}}]
    
    try:
        with db.transactions.watch(
            pipeline,
            max_await_time_ms=500
        ) as stream:
            
            while time.time() - start_time < duration_seconds:
                event = stream.try_next()
                
                if event is not None:
                    total_events += 1
                    doc = event.get("fullDocument", {})
                    country = doc.get("country", "??")
                    
                    if country not in counters:
                        counters[country] = 0
                    counters[country] += 1
                
                # Print summary every second
                now = time.time()
                if now - last_print >= 1.0:
                    total = sum(counters.values())
                    
                    if total > 0:
                        sorted_countries = sorted(
                            counters.items(),
                            key=lambda x: x[1],
                            reverse=True
                        )
                        summary = " | ".join(
                            f"{c}:{n}" for c, n in sorted_countries[:6]
                        )
                        
                        timestamp = datetime.now().strftime("%H:%M:%S")
                        print(f"  [{timestamp}] {total:>5}/sec | {summary}")
                    
                    counters = {}
                    last_print = now
    
    except KeyboardInterrupt:
        print("\n  Counter stopped by user")
    
    elapsed = round(time.time() - start_time, 1)
    print(f"\n  Total events: {total_events:,} in {elapsed}s")
    print(f"  Avg rate: {total_events / max(elapsed, 1):,.0f}/sec")


def start_anomaly_detector(duration_seconds=60):
    """
    Detect anomalous patterns in real-time:
    - Rapid successive transactions from same account
    - Transactions from unusual countries for an account
    - Sudden volume spikes per country
    
    Redshift equivalent:
        -- Would require:
        -- 1. Kinesis Data Analytics (Apache Flink)
        -- 2. Custom anomaly detection logic
        -- 3. Lambda for alerting
        -- 4. DynamoDB for state management
        -- Total: 4 services, complex orchestration
        
    MongoDB:
        -- Change stream + in-memory state
        -- Single process, single database connection
    """
    
    print(f"\n{'=' * 70}")
    print(f"🔍 ANOMALY DETECTOR — LIVE ({duration_seconds}s)")
    print(f"{'=' * 70}")
    
    # In-memory state for pattern detection
    account_history = {}    # account_id → list of recent timestamps
    country_rates = {}      # country → list of timestamps (last 60 sec)
    anomalies_found = 0
    start_time = time.time()
    
    RAPID_TXN_THRESHOLD = 5      # 5+ transactions from same account
    RAPID_TXN_WINDOW_SEC = 60    # within 60 seconds
    VOLUME_SPIKE_MULTIPLIER = 3  # 3x normal rate = spike
    
    pipeline = [{"$match": {"operationType": "insert"}}]
    
    try:
        with db.transactions.watch(pipeline, max_await_time_ms=500) as stream:
            
            while time.time() - start_time < duration_seconds:
                event = stream.try_next()
                
                if event is None:
                    continue
                
                doc = event.get("fullDocument", {})
                if not doc:
                    continue
                
                account_id = doc.get("account_id", "")
                country = doc.get("country", "")
                now = datetime.now()
                now_ts = now.timestamp()
                
                # ─── Check 1: Rapid successive transactions ──
                if account_id not in account_history:
                    account_history[account_id] = []
                
                # Clean old entries
                account_history[account_id] = [
                    ts for ts in account_history[account_id]
                    if now_ts - ts < RAPID_TXN_WINDOW_SEC
                ]
                account_history[account_id].append(now_ts)
                
                if len(account_history[account_id]) >= RAPID_TXN_THRESHOLD:
                    anomalies_found += 1
                    print(f"\n  🚨 RAPID TXN ANOMALY #{anomalies_found}")
                    print(f"    Account: {account_id}")
                    print(f"    {len(account_history[account_id])} transactions "
                          f"in {RAPID_TXN_WINDOW_SEC}s")
                    print(f"    Country: {country}")
                    
                    db.anomaly_alerts.insert_one({
                        "type": "rapid_transactions",
                        "account_id": account_id,
                        "country": country,
                        "txn_count": len(account_history[account_id]),
                        "window_seconds": RAPID_TXN_WINDOW_SEC,
                        "detected_at": now,
                        "status": "pending_review"
                    })
                
                # ─── Check 2: Country volume spike ───────────
                if country not in country_rates:
                    country_rates[country] = []
                
                country_rates[country] = [
                    ts for ts in country_rates[country]
                    if now_ts - ts < 60
                ]
                country_rates[country].append(now_ts)
                
                # Cleanup memory periodically
                if len(account_history) > 100000:
                    # Evict oldest entries
                    oldest = sorted(
                        account_history.items(),
                        key=lambda x: max(x[1]) if x[1] else 0
                    )[:50000]
                    for k, _ in oldest:
                        del account_history[k]
    
    except KeyboardInterrupt:
        print("\n  Detector stopped by user")
    
    elapsed = round(time.time() - start_time, 1)
    print(f"\n  Detector ran for: {elapsed}s")
    print(f"  Anomalies found: {anomalies_found}")
    print(f"  Accounts tracked: {len(account_history):,}")

       → Without index: full collection scan on every dashboard load
       → With index: efficient range scan on time field
    
    ✅ CONSIDER MongoDB Charts FOR QUICK WINS
       → Built into Atlas — no separate service to manage
       → Connects directly to collections
       → Auto-refresh as fast as every 10 seconds
       → Embed in your app via iframe or Charts SDK
       → Free with Atlas (no per-user cost like QuickSight)

❌ WHAT SHOULD NOT DO

DO NOT:

    ❌ RUN HEAVY AGGREGATIONS ON PRIMARY
       → Full collection scan on primary = OLTP latency spike
       → ALWAYS route analytics to secondary or analytics node
       → Redshift trap: you're used to running any query anywhere
         (Redshift is a dedicated warehouse — no OLTP to protect)
       → MongoDB: operational and analytical share the same cluster
    
    ❌ SKIP PRE-COMPUTATION AND AGGREGATE EVERYTHING LIVE
       → "I'll just aggregate transactions every 10 seconds"
       → At 50,000 txns/minute: scanning 500,000 docs per dashboard refresh
       → Multiply by 10 dashboard users = 5M doc scans per refresh cycle
       → Pre-computed metrics: read 60 docs (one per minute) instead
    
    ❌ USE CHANGE STREAMS FOR EVERYTHING
       → Change streams are perfect for: alerts, events, real-time counters
       → Change streams are WRONG for: complex aggregations, historical trends
       → Processing 50,000 events/minute in a change stream to compute
         hourly trends = reinventing the aggregation pipeline poorly
       → Use the right layer for the right metric type
    
    ❌ STORE UNLIMITED GRANULARITY METRICS
       → Per-second metrics × 50 countries × 365 days = 1.6 billion docs
       → Use appropriate granularity:
         Real-time dashboard: per-minute (fine for 60-minute window)
         Trend charts: per-hour (fine for 90-day window)
         Historical: per-day (fine for multi-year)
       → TTL indexes to auto-expire fine-grained metrics
    
    ❌ ASSUME MONGODB REPLACES REDSHIFT FOR ALL ANALYTICS
       → MongoDB real-time analytics: excellent for operational dashboards
       → Redshift: still wins for complex historical analytics
         (5-year trend analysis, multi-table JOINs, cohort studies)
       → The answer is BOTH, not either/or
       → MongoDB for: "what's happening NOW" (seconds-fresh)
       → Redshift for: "what happened over the last 5 years" (batch-fresh)
    
    ❌ FORGET TO HANDLE CHANGE STREAM FAILURES
       → Change stream disconnects: network blip, server restart
       → WITHOUT resume token: miss events during disconnect
       → ALWAYS store and use resume tokens
       → Reference: Project 25 for resume token pattern
    
    ❌ CREATE TOO MANY METRICS IN ONE DOCUMENT
       → { count: 1, total: 100, by_country: {...}, by_type: {...},
       →   by_currency: {...}, by_merchant: {...}, by_hour: {...} }
       → Document grows too large → update contention
       → Split into separate metric documents per dimension
       → One doc for country metrics, one for type metrics, etc.

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 42:

PITFALL 1: "I'll just run my Redshift SQL as MongoDB aggregation"
    ❌ Converting complex SQL with 5 JOINs and 3 CTEs directly
    ✅ Redesign the query for document model and pre-computation
    WHY: MongoDB aggregation pipeline works differently than SQL
         Pre-computed metrics eliminate most complex queries
         What takes 15 seconds in Redshift SQL may take 60 seconds
         in MongoDB aggregation — but <5ms with pre-computation

PITFALL 2: "Analytics on primary is fine — it's fast"
    ❌ Running dashboard aggregations on primary node
    ✅ ALWAYS use analytics node or secondary read preference
    WHY: Your dashboard query scanning 1M documents competes
         with 50,000 OLTP operations per minute
         One slow analytical query = 1000 slow OLTP operations
         Redshift doesn't have this problem (dedicated warehouse)

PITFALL 3: "I don't need pre-computation — aggregation is fast enough"
    ❌ Real-time aggregation for every dashboard refresh
    ✅ Pre-compute at write time, read pre-computed values
    WHY: 10 dashboard users × 5 charts × refresh every 10 seconds
         = 300 aggregation queries per minute
         Each scanning recent transactions = massive I/O
         Pre-computed: 300 simple document reads per minute = trivial

PITFALL 4: "Change streams can replace my entire analytics pipeline"
    ❌ Processing every event through change stream for all metrics
    ✅ Change streams for alerts/events, pre-computation for metrics
    WHY: Change stream processing at 50K events/sec requires
         significant application resources (CPU, memory)
         Pre-computation ($inc) distributes work across write operations
         Change streams: best for reactive patterns, not aggregation

PITFALL 5: "MongoDB Charts replaces QuickSight completely"
    ❌ Expecting Charts to do everything QuickSight does
    ✅ Charts for operational dashboards, QuickSight for deep analytics
    WHY: Charts: great for real-time, simpler visualizations
         QuickSight: better for ML insights, complex BI, SPICE caching
         Charts: included in Atlas pricing (cost advantage)
         QuickSight: per-user pricing but more features
         Use Charts for ops team, QuickSight + Redshift for analysts
