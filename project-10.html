PROJECT 10: Compound Indexes — Multi-Column Sort Keys

Problem: Query Orders by Customer ID + Date Range Efficiently

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform now has 5 million orders
    The operations team needs these queries to be FAST:
    
    Query A: "Show all orders for customer C-1234, newest first"
        → Customer support looking up a specific customer's order history
        → Filters: customer_id + sort by order_date descending
    
    Query B: "Show all orders in 'shipped' status for customer C-1234"
        → Customer checking their shipped orders
        → Filters: customer_id + status
    
    Query C: "Show all 'pending' orders placed this week, highest value first"
        → Operations dashboard for order fulfillment
        → Filters: status + date range + sort by amount
    
    Query D: "Find orders by customer C-1234 between Jan and March 2024"
        → Finance team auditing specific customer's quarterly orders
        → Filters: customer_id + date range
    
    From Project 9: you know single-field indexes help ONE filter field
    But these queries filter on TWO or THREE fields simultaneously
    
    Do you create separate indexes on each field?
    Or ONE compound index with multiple fields?
    If compound — what ORDER do you put the fields in?
    
    THIS is the exact same decision as choosing
    COMPOUND SORTKEY(col1, col2, col3) order in Redshift

🔴 REDSHIFT MINDSET — How You Would Solve This

Compound Sort Key Thinking You Already Know

-- REDSHIFT: You'd design the sort key at table creation

CREATE TABLE store.orders (
    order_id        BIGINT IDENTITY(1,1),
    customer_id     VARCHAR(20) NOT NULL,
    order_date      TIMESTAMP NOT NULL,
    status          VARCHAR(20),
    total_amount    DECIMAL(12,2),
    items_count     INTEGER,
    shipping_city   VARCHAR(100)
)
DISTSTYLE KEY
DISTKEY(customer_id)
COMPOUND SORTKEY(customer_id, order_date);

-- WHY this sort key order?
-- COMPOUND SORTKEY works LEFT to RIGHT (prefix rule)
-- 
-- Physical sort order on disk:
--   C-0001, 2024-01-01
--   C-0001, 2024-01-15
--   C-0001, 2024-02-03
--   C-0002, 2023-12-01
--   C-0002, 2024-01-20
--   ...
--
-- QUERY A: WHERE customer_id = 'C-1234' ORDER BY order_date
--   → Zone maps skip to C-1234 blocks → FAST
--   → Data already sorted by date within customer → VERY FAST
--
-- QUERY D: WHERE customer_id = 'C-1234' AND order_date BETWEEN ...
--   → Zone maps skip to C-1234 blocks → FAST
--   → Date range within customer → zone maps skip further → VERY FAST
--
-- QUERY B: WHERE customer_id = 'C-1234' AND status = 'shipped'
--   → Zone maps skip to C-1234 blocks → FAST
--   → But must scan ALL C-1234 rows to check status → MEDIUM
--   → status is NOT in sort key
--
-- QUERY C: WHERE status = 'pending' AND order_date > ...
--   → customer_id is NOT in filter → sort key USELESS
--   → Full table scan → SLOW
--   → Because sort key prefix (customer_id) not in query

-- THE COMPOUND SORT KEY PREFIX RULE:
-- SORTKEY(A, B, C) helps:
--   WHERE A = ...                    ← uses A prefix
--   WHERE A = ... AND B = ...        ← uses A,B prefix
--   WHERE A = ... AND B = ... AND C  ← uses full key
--   WHERE B = ...                    ← DOES NOT HELP (A missing)
--   WHERE C = ...                    ← DOES NOT HELP (A,B missing)
--   WHERE A = ... AND C = ...        ← uses A prefix only (B gap)

Redshift's Limited Options

REDSHIFT OPTIONS FOR MULTIPLE QUERY PATTERNS:

OPTION 1: COMPOUND SORTKEY(customer_id, order_date)
    → Great for: queries starting with customer_id
    → Terrible for: queries starting with status or order_date alone
    → You chose customer_id first → locked in

OPTION 2: COMPOUND SORTKEY(status, order_date)
    → Great for: queries starting with status
    → Terrible for: queries starting with customer_id
    → Different trade-off

OPTION 3: INTERLEAVED SORTKEY(customer_id, status, order_date)
    → Helps queries on ANY combination
    → But less effective than compound for any SINGLE pattern
    → VACUUM is much slower for interleaved
    → Compromise in all directions

OPTION 4: Create a second table with different sort key
    → Maintain two copies of data
    → ETL to keep them in sync
    → More storage, more complexity

FUNDAMENTAL LIMITATION:
    → Redshift: ONE sort key definition per table
    → Must choose ONE ordering strategy
    → All other query patterns get suboptimal performance
    → Changing requires table recreation

🟢 MONGODB MINDSET — Compound Indexes

The Liberating Difference

MONGODB COMPOUND INDEX PHILOSOPHY:

    REDSHIFT: ONE sort key per table → ONE optimization strategy
    MONGODB:  MANY compound indexes per collection → MANY optimization strategies

    → Create { customer_id: 1, order_date: -1 } for Query A and D
    → Create { customer_id: 1, status: 1 } for Query B
    → Create { status: 1, order_date: -1, total_amount: -1 } for Query C
    → ALL THREE coexist on the same collection
    → Query planner automatically picks the best index per query

    BUT: each index costs storage + RAM + write overhead
    The ART is creating the MINIMUM indexes that serve ALL queries
    
    THIS IS WHERE THE ESR RULE BECOMES YOUR BEST FRIEND

The ESR Rule — The Golden Rule of Compound Indexes

ESR = Equality → Sort → Range

THE MOST IMPORTANT INDEXING CONCEPT IN MONGODB
(Redshift developers never learn this because Redshift has no indexes)

WHEN BUILDING A COMPOUND INDEX, order fields as:
    1. EQUALITY fields first    (field = exactValue)
    2. SORT fields second       (field used in sort())
    3. RANGE fields last        (field > X, field < Y, field IN [...])

WHY THIS ORDER?

    EQUALITY (first):
        → Narrows to exact match → most selective
        → B-tree jumps directly to matching subtree
        → Like: go to floor 5 in a building (exact)
    
    SORT (second):
        → Within the equality match, data is already sorted
        → No in-memory sort needed
        → Like: rooms on floor 5 are numbered in order
    
    RANGE (last):
        → Scans a portion of the remaining sorted data
        → B-tree reads a contiguous range of leaf nodes
        → Like: rooms 501-520 are next to each other

IF YOU PUT RANGE BEFORE SORT:
    → B-tree scans the range
    → But results within range are NOT in sort order
    → MongoDB must do in-memory sort → SLOW for large results
    → Like: picking random rooms on floor 5, then sorting room numbers

EXAMPLE:
    Query: db.orders.find({ customer_id: "C-1234", order_date: { $gte: start } })
                     .sort({ order_date: -1 })

    E = customer_id (equality: exact match)
    S = order_date  (sort: ORDER BY order_date DESC)
    R = order_date  (range: >= start)

    WAIT — order_date is both Sort AND Range!
    
    When a field is both Sort and Range:
        → Treat it as SORT (put it after equality)
        → The range becomes a "scan direction" on the sorted index
        → Index: { customer_id: 1, order_date: -1 }
        → Equality on customer_id → jump to C-1234
        → Descending order_date → already sorted newest first
        → Range >= start → stop scanning when date < start
        → PERFECT: no in-memory sort, minimal scan

🧩 HOW TO APPROACH IT IN MONGODB

COMPOUND INDEX DESIGN PROCESS:

STEP 1: List all query patterns
    Query A: find({ customer_id: X }).sort({ order_date: -1 })
    Query B: find({ customer_id: X, status: Y })
    Query C: find({ status: Y, order_date: { $gte: D } }).sort({ total_amount: -1 })
    Query D: find({ customer_id: X, order_date: { $gte: D1, $lt: D2 } })

STEP 2: For each query, identify E-S-R components
    Query A: E=customer_id     S=order_date     R=none
    Query B: E=customer_id,status  S=none        R=none
    Query C: E=status          S=total_amount    R=order_date
    Query D: E=customer_id     S=none            R=order_date

STEP 3: Design minimum indexes to cover all patterns
    Index 1: { customer_id: 1, order_date: -1 }
        → Serves Query A perfectly (E + S)
        → Serves Query D well (E + R, range on 2nd field)
        → Serves Query B partially (prefix customer_id)
    
    Index 2: { customer_id: 1, status: 1 }
        → Serves Query B perfectly (E + E)
        → Also helps any customer_id-only query (prefix)
    
    Index 3: { status: 1, total_amount: -1, order_date: -1 }
        → Serves Query C (E + S + R via ESR)

STEP 4: Check for redundancy
    → Index 1 prefix { customer_id: 1 } covers customer_id-only queries
    → Index 2 prefix { customer_id: 1 } duplicates this
    → Both needed? Index 2 adds status → yes, keep both
    → Could combine? { customer_id: 1, status: 1, order_date: -1 }
      → Serves B perfectly, A with prefix only (loses sort optimization)
      → Trade-off: fewer indexes vs optimal per-query performance

STEP 5: Verify with explain() (always!)

THIS PROCESS DOESN'T EXIST IN REDSHIFT
    → Redshift: choose ONE sort key → done
    → MongoDB: analyze ALL queries → design minimal index set → iterate

// ═══════════════════════════════════════════════════════
// CORRECT APPROACH: Compound indexes designed with ESR rule
// ═══════════════════════════════════════════════════════

// ─── INDEX 1: For Queries A and D ────────────────────
// Query A: find({ customer_id: X }).sort({ order_date: -1 })
//   E = customer_id, S = order_date, R = none
//   ESR → { customer_id: 1, order_date: -1 }
//
// Query D: find({ customer_id: X, order_date: { $gte, $lt } })
//   E = customer_id, S = none, R = order_date
//   ESR → { customer_id: 1, order_date: -1 }
//   (same index works for both!)

db.orders.createIndex(
    { customer_id: 1, order_date: -1 },
    { name: "idx_cust_date" }
)

// ─── INDEX 2: For Query B ────────────────────────────
// Query B: find({ customer_id: X, status: Y })
//   E = customer_id, E = status, S = none, R = none
//   ESR → { customer_id: 1, status: 1 }

db.orders.createIndex(
    { customer_id: 1, status: 1 },
    { name: "idx_cust_status" }
)

// ─── INDEX 3: For Query C ────────────────────────────
// Query C: find({ status: Y, order_date: { $gte, $lt } })
//          .sort({ total_amount: -1 })
//   E = status, S = total_amount, R = order_date
//   ESR → { status: 1, total_amount: -1, order_date: -1 }

db.orders.createIndex(
    { status: 1, total_amount: -1, order_date: -1 },

Step 6: The Prefix Rule — What Works and What Doesn't

// ═══════════════════════════════════════════════════════
// THE PREFIX RULE — Identical to Redshift COMPOUND SORTKEY
// ═══════════════════════════════════════════════════════

// Index: { customer_id: 1, order_date: -1 }
// This creates a B-tree sorted by: customer_id THEN order_date
//
// WHICH QUERIES CAN USE THIS INDEX?

// ✅ PREFIX MATCH: customer_id only
db.orders.find({ customer_id: "C-00100" }).explain("executionStats")
// Uses idx_cust_date → IXSCAN (uses first field of compound index)
// Like: SORTKEY(customer_id, order_date) helps WHERE customer_id = ...

// ✅ FULL MATCH: customer_id + order_date
db.orders.find({ customer_id: "C-00100", order_date: { $gte: ISODate("2024-01-01") } })
         .explain("executionStats")
// Uses idx_cust_date → IXSCAN (uses both fields)

// ✅ PREFIX + SORT: customer_id + sort by order_date
db.orders.find({ customer_id: "C-00100" }).sort({ order_date: -1 })
         .explain("executionStats")
// Uses idx_cust_date → IXSCAN (equality on first, sort on second)

// ❌ SECOND FIELD ONLY: order_date without customer_id
db.orders.find({ order_date: { $gte: ISODate("2024-01-01") } })
         .explain("executionStats")
// COLLSCAN! Index CANNOT be used because prefix (customer_id) not in query
// EXACTLY like Redshift: SORTKEY(A, B) doesn't help WHERE B = ...

// ❌ SORT ON SECOND FIELD ONLY: sort by order_date without customer_id filter
db.orders.find({ status: "pending" }).sort({ order_date: -1 })
         .explain("executionStats")
// Cannot use idx_cust_date for sort (prefix not matched)
// Might use another index for status filter, then in-memory sort

// ─── THE RULE (same in both systems) ─────────────────
//
// Compound index { A: 1, B: 1, C: 1 } supports:
//   ✅ Filter on A
//   ✅ Filter on A + B
//   ✅ Filter on A + B + C
//   ✅ Filter on A + sort on B
//   ✅ Filter on A + B + sort on C
//   ❌ Filter on B alone
//   ❌ Filter on C alone
//   ❌ Filter on B + C (without A)
//   ❌ Sort on B alone (without A filter)
//
// Redshift COMPOUND SORTKEY(A, B, C) has IDENTICAL behavior:
//   ✅ WHERE A = ...
//   ✅ WHERE A = ... AND B = ...
//   ❌ WHERE B = ... (sort key prefix A not in filter)

Step 7: Direction Matters — The Compound Index Direction Rule

// ═══════════════════════════════════════════════════════
// DIRECTION IN COMPOUND INDEXES — Redshift can't do this
// ═══════════════════════════════════════════════════════

// SINGLE FIELD: direction doesn't matter (B-tree traverses both ways)
// COMPOUND: direction MATTERS when sort uses multiple fields

// SCENARIO: Sort by customer_id ASC, order_date DESC
db.orders.find().sort({ customer_id: 1, order_date: -1 }).limit(10)

// Index { customer_id: 1, order_date: -1 } → MATCHES sort exactly
//   B-tree stores: A→B→C customers, within each: newest→oldest dates
//   Sort { customer_id: 1, order_date: -1 } reads forward → perfect

// Index { customer_id: 1, order_date: 1 } → does NOT match!
//   B-tree stores: A→B→C customers, within each: oldest→newest dates
//   Sort { customer_id: 1, order_date: -1 } can't use this index
//   Must do in-memory sort

// THE RULE:
// An index can satisfy a sort if:
//   1. Sort fields match index fields in SAME ORDER
//   2. Sort directions match index directions EXACTLY
//      OR all directions are REVERSED (traverse index backwards)

// MATCHING:
// Index { a: 1, b: -1 } satisfies:
//   ✅ sort({ a: 1, b: -1 })    → forward traversal (exact match)
//   ✅ sort({ a: -1, b: 1 })    → backward traversal (all reversed)
//   ❌ sort({ a: 1, b: 1 })     → mixed: can't do (a forward, b forward but index has b backward)
//   ❌ sort({ a: -1, b: -1 })   → mixed: can't do

// PRACTICAL EXAMPLE:
print("=== Direction test: matching sort ===")
let eMatch = db.orders.find({ customer_id: "C-00100" })
                       .sort({ order_date: -1 })    // matches index direction
                       .explain("executionStats")
// Check for SORT stage
let hasSort1 = eMatch.queryPlanner.winningPlan.stage === "SORT"
print(`In-memory sort needed: ${hasSort1}`)    // false → index provides order

print("\n=== Direction test: reversed sort ===")
let eReverse = db.orders.find({ customer_id: "C-00100" })
                         .sort({ order_date: 1 })     // REVERSED from index
                         .explain("executionStats")
// MongoDB traverses index BACKWARDS → still works!
let hasSort2 = eReverse.queryPlanner.winningPlan.stage === "SORT"
print(`In-memory sort needed: ${hasSort2}`)    // false → backward traversal

// REDSHIFT COMPARISON:
// Redshift sort keys are ALWAYS ascending
// No descending sort key exists
// ORDER BY col DESC → scan backwards (similar to MongoDB backward traversal)
// But: Redshift has no compound direction control
//      SORTKEY(customer_id, order_date) = both ascending, period

Step 8: Covered Queries — The Ultimate Optimization

// ═══════════════════════════════════════════════════════
// COVERED QUERIES — When MongoDB doesn't read documents at all
// ═══════════════════════════════════════════════════════

// NORMAL query execution:
//   1. IXSCAN → find matching index entries
//   2. FETCH → follow pointer to read full document from disk
//   3. Return document to client
//
// COVERED query execution:
//   1. IXSCAN → find matching index entries
//   2. Return data directly from index (NO FETCH!)
//   3. FASTEST possible query
//
// WHEN IS A QUERY COVERED?
//   → ALL fields in the query filter are IN the index
//   → ALL fields in the projection (returned fields) are IN the index
//   → _id is excluded from projection (or _id is in the index)
//   → MongoDB never needs to touch the actual document

// EXAMPLE: Count orders per status for a customer
// Index: { customer_id: 1, status: 1 }

// ─── NOT COVERED (returns all fields) ────────────────
print("=== Not covered: all fields returned ===")
let eNotCovered = db.orders.find(
    { customer_id: "C-00100" },
    {}                              // return all fields
).explain("executionStats")
print(`Stages: IXSCAN → FETCH`)   // FETCH needed for other fields
print(`Docs examined: ${eNotCovered.executionStats.totalDocsExamined}`)

// ─── COVERED (only return indexed fields) ────────────
print("\n=== Covered: only indexed fields returned ===")
let eCovered = db.orders.find(
    { customer_id: "C-00100" },
    { customer_id: 1, status: 1, _id: 0 }   // ONLY fields in the index
                                              // _id: 0 → exclude _id
).explain("executionStats")

// Check for FETCH stage
let plan = eCovered.queryPlanner.winningPlan
print(`Stage: ${plan.stage}`)
// If covered: stage is IXSCAN or PROJECTION_COVERED (no FETCH)
// totalDocsExamined should be 0 (never reads actual documents!)
print(`Docs examined: ${eCovered.executionStats.totalDocsExamined}`)
print(`Keys examined: ${eCovered.executionStats.totalKeysExamined}`)
// Keys examined > 0, Docs examined = 0 → COVERED!

// WHY THIS IS POWERFUL:
// → No disk I/O to read documents
// → All data served from index (which lives in RAM)
// → Fastest possible MongoDB query
// → Especially useful for count, exists, list-of-ids queries

// REDSHIFT COMPARISON:
// Redshift is NATURALLY "covered" for column subset queries
// SELECT customer_id, status FROM orders → reads only 2 columns
// Because columnar storage reads ONLY needed columns
// MongoDB must explicitly design indexes to achieve similar behavior

// ─── PRACTICAL COVERED QUERY PATTERNS ────────────────

// Pattern 1: Check if customer exists (covered by idx_cust_date)
db.orders.find(
    { customer_id: "C-00100" },
    { customer_id: 1, _id: 0 }
).limit(1).explain("executionStats")
// IXSCAN only → 0 docs examined → instant

// Pattern 2: Get customer's order dates (covered by idx_cust_date)
db.orders.find(
    { customer_id: "C-00100" },
    { customer_id: 1, order_date: 1, _id: 0 }

// IXSCAN only → 0 docs examined → reads from index

// Pattern 3: Count orders by status for customer (covered by idx_cust_status)
db.orders.find(
    { customer_id: "C-00100", status: "shipped" },
    { _id: 0, customer_id: 1, status: 1 }
).explain("executionStats")
// IXSCAN only → 0 docs examined
# pymongo — Covered query demonstration

# ─── COVERED QUERY (only indexed fields) ──────────────
explain_covered = db.orders.find(

stats = explain_covered["executionStats"]

Step 9: Index Intersection — When Two Indexes Combine

// ═══════════════════════════════════════════════════════
// INDEX INTERSECTION — MongoDB can combine two indexes
// (But compound index is almost always better)
// ═══════════════════════════════════════════════════════

// WHAT IS INDEX INTERSECTION?
// If you have:
//   Index A: { customer_id: 1 }
//   Index B: { status: 1 }
// Query: find({ customer_id: "C-00100", status: "shipped" })
// MongoDB CAN use both indexes and intersect the results
//
// Stage: AND_SORTED or AND_HASH
//   → IXSCAN on idx_A → set of document IDs matching customer_id
//   → IXSCAN on idx_B → set of document IDs matching status
//   → INTERSECT the two sets → documents matching both
//   → FETCH those documents

// WHY COMPOUND INDEX IS USUALLY BETTER:
// Intersection: two index scans + merge step
// Compound:     one index scan → direct result
//
// Intersection: reads MORE keys total (from both indexes)
// Compound:     reads FEWER keys (single precise scan)
//
// MongoDB RARELY chooses intersection over compound index
// It's a fallback mechanism, not a primary strategy

// WHEN INTERSECTION HELPS:
// → You have many single-field indexes already
// → You can't add a compound index (operational constraint)
// → Ad-hoc queries with unpredictable field combinations

// WHEN TO USE COMPOUND INSTEAD:
// → Known, repeated query patterns (almost always)
// → Performance-critical queries
// → Production systems

// REDSHIFT COMPARISON:
// Redshift has NO index intersection
// Only ONE sort key → ONE optimization path
// MongoDB intersection = "emergency fallback" when compound not available

⚡ WHAT COULD BE DONE (Compound Index Optimizations)

1. INDEX WITH INCLUDED FIELDS (covering wider queries)
   → Cannot directly add "included" fields in MongoDB
   → But: add fields to compound index tail to enable covered queries
   → { customer_id: 1, order_date: -1, status: 1, total_amount: 1 }
   → Covers more projection patterns
   → Trade-off: larger index, more write overhead

2. PARTIAL COMPOUND INDEX
   → Only index active orders:
   → db.orders.createIndex(
         { customer_id: 1, order_date: -1 },
         { partialFilterExpression: { status: { $ne: "cancelled" } } }
     )
   → Smaller index (excludes cancelled orders)
   → Query MUST include the partial filter to use it

3. COLLATION-AWARE COMPOUND INDEX
   → Case-insensitive compound index:
   → db.orders.createIndex(
         { customer_id: 1, status: 1 },
         { collation: { locale: "en", strength: 2 } }
     )
   → Queries must specify matching collation

4. WILDCARD INDEX (when query patterns are unpredictable)
   → db.orders.createIndex({ "$**": 1 })
   → Indexes ALL fields in ALL documents
   → Useful for ad-hoc queries but HUGE storage
   → NOT a substitute for well-designed compound indexes
   → Last resort, not first choice

✅ WHAT SHOULD DO

SHOULD DO:
    ✅ Apply ESR rule for EVERY compound index design
       → Equality fields first
       → Sort fields second
       → Range fields last
       → This order minimizes keys examined and avoids in-memory sort
    
    ✅ Verify with explain() that your index eliminates SORT stage
       → If explain shows SORT → your index isn't providing sort order
       → Redesign index field order or direction
    
    ✅ Consider covered queries for high-frequency reads
       → Add frequently projected fields to index tail
       → Check: totalDocsExamined = 0 → covered!
    
    ✅ Look for compound indexes that serve MULTIPLE queries
       → { customer_id: 1, order_date: -1 } serves:
         Query A (customer + sort by date)
         Query D (customer + date range)
         Any customer_id-only query (prefix)
       → One index, three query patterns → efficient
    
    ✅ Prefer compound indexes over multiple single-field indexes
       → Compound: one precise scan → optimal
       → Multiple singles: intersection is unreliable and slower
    
    ✅ Match index direction to sort direction
       → Sort { a: 1, b: -1 } needs index { a: 1, b: -1 }
       → Or reversed: { a: -1, b: 1 } (backward traversal)
       → Mismatched: { a: 1, b: 1 } → in-memory sort required

❌ WHAT SHOULD NOT DO

DO NOT:
    ❌ Create compound index in wrong field order
       → { order_date: -1, customer_id: 1 } for query filtering customer_id
       → Prefix rule: order_date first → customer_id-only queries can't use it
       → EXACTLY like putting wrong column first in Redshift COMPOUND SORTKEY
    
    ❌ Put RANGE field before SORT field (violates ESR)
       → { status: 1, order_date: -1, total_amount: -1 } for:
         find({ status: "pending", order_date: { $gte: ... } }).sort({ total_amount: -1 })
       → Range on order_date (2nd field) breaks sort on total_amount (3rd field)
       → MongoDB must in-memory sort
       → Correct: { status: 1, total_amount: -1, order_date: -1 }
         (E=status, S=total_amount, R=order_date)
    
    ❌ Create too many compound indexes
       → Each compound index is LARGER than single-field
       → 5 compound indexes × 3 fields each = significant RAM
       → Every write updates ALL indexes
       → Aim for MINIMUM indexes that cover all query patterns
    
    ❌ Assume index direction doesn't matter for compound
       → Single field: direction rarely matters (Project 9)
       → Compound: direction is CRITICAL for multi-field sorts
       → { a: 1, b: -1 } ≠ { a: 1, b: 1 } for sort purposes
    
    ❌ Create compound index that duplicates a single-field index prefix
       → If you have { customer_id: 1, order_date: -1 }
       → DON'T also create { customer_id: 1 }
       → The compound index's prefix already serves customer_id-only queries
       → Exception: if single-field index is unique and compound is not
    
    ❌ Think "INTERLEAVED SORTKEY = multiple compound indexes"
       → Redshift interleaved: ONE structure, compromise for all queries
       → MongoDB: SEPARATE indexes, each OPTIMAL for its query pattern
       → MongoDB approach is strictly better (at cost of storage + writes)
    
    ❌ Forget to exclude _id in projections for covered queries
       → find({ a: X }, { a: 1, b: 1 }) → NOT covered (_id included by default)
       → find({ a: X }, { a: 1, b: 1, _id: 0 }) → COVERED if index is { a: 1, b: 1 }

⚙️ HOW IT ONLY WORKS INTERNALLY — Compound B-Tree

MongoDB Compound Index Storage

INDEX: { customer_id: 1, order_date: -1 }

B-TREE LEAF NODES (sorted by customer_id ASC, then order_date DESC):

  ┌─────────────────────────────────────────────────────┐
  │ Leaf Page 1                                          │
  │ C-00001, 2024-05-30 → docPtr_A                       │
  │ C-00001, 2024-04-15 → docPtr_B                       │
  │ C-00001, 2024-02-01 → docPtr_C                       │
  │ C-00001, 2023-11-20 → docPtr_D                       │
  └─────────────────────────────────────────────────────┘
  ┌─────────────────────────────────────────────────────┐
  │ Leaf Page 2                                          │
  │ C-00002, 2024-05-28 → docPtr_E                       │
  │ C-00002, 2024-03-10 → docPtr_F                       │
  │ C-00002, 2024-01-05 → docPtr_G                       │
  │ C-00003, 2024-05-31 → docPtr_H                       │
  └─────────────────────────────────────────────────────┘
  ...

QUERY A: find({ customer_id: "C-00001" }).sort({ order_date: -1 })
    Step 1: B-tree navigates to first entry with customer_id = "C-00001"
    Step 2: Read entries forward (they're already sorted by order_date DESC)
    Step 3: Stop when customer_id changes to "C-00002"
    Step 4: Results come out in order → NO in-memory sort needed
    
    I/O: log(n) to find start + sequential read of matching entries
    
QUERY D: find({ customer_id: "C-00001", order_date: { $gte: 2024-01-01, $lt: 2024-04-01 } })
    Step 1: B-tree navigates to customer_id = "C-00001"
    Step 2: Within C-00001 entries, navigate to order_date < 2024-04-01
            (descending index: 2024-04-01 comes BEFORE 2024-01-01)
    Step 3: Read entries until order_date < 2024-01-01
    Step 4: Stop
    
    I/O: log(n) to find start + read ONLY the matching date range
    VERY efficient — double filtering in one index traversal

QUERY (non-prefix): find({ order_date: { $gte: 2024-01-01 } })
    → B-tree is sorted by customer_id FIRST
    → order_date values are scattered across ALL customer_id groups
    → Cannot use index efficiently
    → Falls back to COLLSCAN
    → THIS is the prefix rule in action

Redshift Compound Sort Key Internals (For Comparison)

SORTKEY(customer_id, order_date) ON DISK:

  Block 001 (1MB):
  ┌──────────────────────────────────────────────────────┐
  │ Zone Map: customer_id min="C-00001" max="C-00050"     │
  │           order_date  min=2023-01-05  max=2024-05-30  │
  │                                                        │
  │ Column: customer_id                                    │
  │ [C-00001, C-00001, C-00001, C-00002, C-00002, ...]    │
  │                                                        │
  │ Column: order_date                                     │
  │ [2023-01-05, 2023-06-15, 2024-05-30, 2023-02-01, ...] │
  │ (sorted within customer_id groups)                     │
  └──────────────────────────────────────────────────────┘

  WHERE customer_id = 'C-00001':
  → Check zone maps: block 001 min="C-00001" → might have it → READ
  → Check zone maps: block 050 min="C-00100" → skip
  → Read matching blocks → scan for C-00001 within block
  
  WHERE order_date > '2024-01-01' (no customer_id filter):
  → Zone map on order_date spans ENTIRE range in every block
    (because data sorted by customer_id first, date second)
  → min=2023-01-05, max=2024-05-30 in EVERY block → NO blocks skipped
  → Full table scan → prefix rule violation (same as MongoDB)

KEY DIFFERENCE IN INTERNALS:
  MongoDB compound index:
    → Separate B-tree structure
    → O(log n) navigation to exact position
    → Sequential read of matching entries
    → Can provide sort order (eliminate in-memory sort)
    
  Redshift compound sort key:
    → Physical data order (not separate structure)
    → Zone map elimination (skip blocks)
    → Minimum read unit: 1MB block (coarser than B-tree)
    → Physical order provides sort (always)
    → But: only ONE sort order per table

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 10 SPECIFIC:

PITFALL 1: "I'll just put the most filtered field first"
    ❌ Always putting highest-cardinality field first
    ✅ Apply ESR rule: Equality → Sort → Range
    WHY: Redshift sort key thinking says "most selective column first"
         MongoDB compound index thinking says "equality first, then sort, then range"
         These are DIFFERENT strategies
    EXAMPLE:
         Query: find({ status: "pending", order_date: { $gte: D } })
                .sort({ total_amount: -1 })
         
         Redshift thinking: order_date has highest cardinality → put first
         ❌ { order_date: -1, status: 1, total_amount: -1 }
            → status is equality but not first → can't jump to "pending"
            → in-memory sort needed for total_amount
         
         ESR thinking: E=status, S=total_amount, R=order_date
         ✅ { status: 1, total_amount: -1, order_date: -1 }
            → jumps to "pending" (equality)
            → reads in total_amount order (sort)
            → filters by order_date range (range)

PITFALL 2: "One big compound index covers everything"
    ❌ { customer_id: 1, status: 1, order_date: -1, total_amount: -1, shipping_city: 1 }
    ✅ Smaller, targeted compound indexes per query pattern
    WHY: 5-field compound index is HUGE in storage
         Only helps queries that start with customer_id (prefix rule)
         Query starting with status → CANNOT use this index
         Better: 3 focused indexes each serving 1-2 query patterns
    EXCEPTION: If ALL queries start with customer_id, one wide index can work

PITFALL 3: "Direction doesn't matter — it didn't for single fields" (Project 9)
    ❌ { customer_id: 1, order_date: 1 } for sort({ customer_id: 1, order_date: -1 })
    ✅ { customer_id: 1, order_date: -1 } matches the sort exactly
    WHY: Single-field indexes traverse both directions (Project 9)
         Compound indexes: mixed directions CAN'T reverse
         { a: 1, b: 1 } can serve sort { a: 1, b: 1 } or { a: -1, b: -1 }
         { a: 1, b: 1 } CANNOT serve sort { a: 1, b: -1 }
         Must create { a: 1, b: -1 } for that sort pattern

PITFALL 4: "Covered queries happen automatically" (Redshift thinking)
    ❌ Expecting MongoDB to only read needed fields naturally
    ✅ Must explicitly project only indexed fields + exclude _id
    WHY: Redshift columnar storage ALWAYS reads only needed columns
         That's its natural behavior — no special design needed
         MongoDB reads ENTIRE documents unless:
           a) Projection specified AND
           b) ALL projected fields are in the index AND
           c) _id is excluded (or _id is in the index)
         You must DESIGN for covered queries in MongoDB

PITFALL 5: "I'll create compound index for every field combination"
    ❌ 10 fields → creating indexes for every 2-field and 3-field combination
       → C(10,2) + C(10,3) = 45 + 120 = 165 indexes!
    ✅ Analyze actual query patterns → create MINIMUM indexes
    WHY: Each index:
         → Consumes storage (30-50% of indexed data per index)
         → Consumes RAM (must fit in WiredTiger cache)
         → Slows every insert/update/delete
         → 165 indexes would make writes nearly impossible
    RULE OF THUMB: 5-10 indexes per collection is healthy
                   >20 indexes = investigate and consolidate

PITFALL 6: "Compound index {A, B} also helps queries on B alone"
    ❌ Assuming { customer_id: 1, status: 1 } helps find({ status: "pending" })
    ✅ Understanding prefix rule: only leftmost fields usable
    WHY: IDENTICAL to Redshift COMPOUND SORTKEY prefix rule
         You already know this from Redshift — apply it here!
         SORTKEY(A, B) doesn't help WHERE B = ... 
         Index {A, B} doesn't help find({ B: ... })
         If you need B-only queries: create separate { B: 1 } index

PITFALL 7: "Index intersection will handle multi-field queries"
    ❌ Creating many single-field indexes and relying on intersection
    ✅ Create compound indexes for known multi-field query patterns
    WHY: MongoDB RARELY uses index intersection
         Query planner usually picks ONE best index, ignores others
         Even when intersection happens: 2 scans + merge > 1 compound scan
         Intersection is a fallback, not a strategy
         Redshift has no equivalent concept — don't look for it in MongoDB

📈 SCALABILITY CONSIDERATIONS

COMPOUND INDEX SCALABILITY:

INDEX SIZE GROWS WITH FIELDS:
    Single field index { email: 1 }:
        → ~40-60 bytes per entry (key + pointer)
        → 10M docs → ~500MB index
    
    Compound index { customer_id: 1, order_date: -1 }:
        → ~60-80 bytes per entry (two keys + pointer)
        → 10M docs → ~700MB index
    
    Compound index { customer_id: 1, status: 1, order_date: -1, total_amount: -1 }:
        → ~90-120 bytes per entry (four keys + pointer)
        → 10M docs → ~1GB index
    
    EACH ADDITIONAL FIELD ADDS ~15-30 bytes per entry
    At 10M docs, that's 150-300MB more index data per field

RAM PRESSURE:
    All indexes should fit in WiredTiger cache (50% of RAM)
    Server with 16GB RAM → ~7.5GB WiredTiger cache
    
    3 compound indexes × ~700MB each = ~2.1GB
    + _id index ~500MB
    + data working set ~2GB
    = ~4.6GB total → fits in 7.5GB cache ✅
    
    10 compound indexes × ~700MB each = ~7GB
    + _id index ~500MB
    + data working set ~2GB
    = ~9.5GB total → EXCEEDS 7.5GB cache ❌ → disk reads → slow
    
    REDSHIFT COMPARISON:
    → Zone maps: ~KB of metadata regardless of table size
    → No RAM pressure from "indexes" (no indexes exist)
    → Redshift RAM used for query execution, not index caching
    → This is a fundamental advantage of Redshift's approach
    → MongoDB must carefully manage index count and size

WRITE OVERHEAD:
    Each compound index updated on every write:
    
    INSERT one document:
        → 0 indexes (impossible): 1 write
        → 1 index (_id only): 2 writes (data + _id index)
        → 4 indexes: 5 writes (data + 4 index updates)
        → 10 indexes: 11 writes (data + 10 index updates)
    
    Write throughput DECREASES linearly with index count
    
    REDSHIFT COMPARISON:
    → INSERT appends to unsorted zone → ONE write
    → VACUUM later re-sorts (deferred cost)
    → No per-write index maintenance overhead
    → But: VACUUM can take hours on large tables
    → Trade-off: MongoDB = real-time index cost, Redshift = deferred sort cost

SHARDING INTERACTION (Preview for Project 21):
    → Shard key MUST be prefix of any compound index
    → Or shard key must be a separate index
    → Compound index design affects shard key choice
    → Planning compound indexes NOW affects sharding decisions LATER

📐 DATA MODELING COMPARISON — Compound Index Impact

HOW COMPOUND INDEXES AFFECT DOCUMENT DESIGN:

REDSHIFT APPROACH:
    → Normalized tables: orders, customers, order_items
    → SORTKEY chosen per table independently
    → JOIN tables at query time
    → Sort key serves the JOIN pattern

MONGODB APPROACH:
    → Document design INTERACTS with index design
    → Compound index fields must exist in the document
    → Embedded data can be indexed (dot notation in compound index)
    → Document shape influences index efficiency

EXAMPLE — Two design options for the same query:

OPTION A: Flat document (Redshift-like)
    {
        customer_id: "C-00100",
        order_date: ISODate("2024-05-31"),
        status: "shipped",
        customer_name: "John Doe",          ← denormalized from customer
        customer_city: "New York"            ← denormalized from customer
    }
    Index: { customer_id: 1, order_date: -1 }
    → Simple, flat index on top-level fields

OPTION B: Nested document (MongoDB-native)
    {
        customer: {                          ← embedded subdocument
            id: "C-00100",
            name: "John Doe",
            city: "New York"
        },
        order_date: ISODate("2024-05-31"),
        status: "shipped"
    }
    Index: { "customer.id": 1, order_date: -1 }    ← DOT NOTATION in index
    → Works! MongoDB indexes can use dot notation for nested fields
    → Same B-tree performance as flat fields

OPTION C: Reference (normalized, Redshift-style)
    // orders collection
    { customer_id: "C-00100", order_date: ..., status: ... }
    
    // customers collection
    { _id: "C-00100", name: "John Doe", city: "New York" }
    
    Index on orders: { customer_id: 1, order_date: -1 }
    → Requires $lookup (JOIN) to get customer name
    → Compound index only helps orders collection
    → Customer name not available in covered query from orders index

KEY INSIGHT:
    Document design determines WHAT you can index
    Index design determines HOW FAST you can query
    They must be designed TOGETHER
    Redshift: table design and sort key are loosely coupled
    MongoDB: document design and indexes are tightly coupled

def deep_explain(collection, filter_doc=None, sort_doc=None,
                 projection=None, limit_val=None, label="Query"):
    """
    Deep analysis of query execution plan.
    Detects: index used, sort elimination, covered query.
    """
    cursor = collection.find(filter_doc or {}, projection or {})
    if sort_doc:
        cursor = cursor.sort(list(sort_doc.items()))
    if limit_val:
        cursor = cursor.limit(limit_val)
    
    explain = cursor.explain()
    stats = explain["executionStats"]
    plan = explain["queryPlanner"]["winningPlan"]
    
    # Walk plan tree
    stages = []
    node = plan
    while node:
        stages.append(node.get("stage", "?"))
        node = node.get("inputStage")
    
    # Detect characteristics
    has_sort_stage = "SORT" in stages
    has_fetch_stage = "FETCH" in stages
    has_ixscan = "IXSCAN" in stages
    has_collscan = "COLLSCAN" in stages
    is_covered = (stats["totalDocsExamined"] == 0 and 
                  stats["totalKeysExamined"] > 0)
    
    # Find index name
    index_name = None
    node = plan
    while node:
        if node.get("indexName"):
            index_name = node["indexName"]
            break
        node = node.get("inputStage")
    
    # Efficiency
    returned = stats["nReturned"]
    examined = stats["totalDocsExamined"]
    keys = stats["totalKeysExamined"]
    ratio = examined / max(returned, 1)
    
    # Rating
    if has_collscan:
        rating = "🔴 COLLSCAN"
    elif has_sort_stage:
        rating = "🟡 INDEX + IN-MEMORY SORT"
    elif is_covered:
        rating = "🟢 COVERED (no doc fetch!)"
    elif ratio <= 1.1:
        rating = "🟢 OPTIMAL"
    elif ratio <= 5:
        rating = "🟡 GOOD"
    else:
        rating = "🟠 SUBOPTIMAL"
    
    print(f"\n  {'─'*60}")
    print(f"  {label}")
    print(f"  {'─'*60}")
    print(f"  Filter:     {filter_doc}")
    print(f"  Sort:       {sort_doc}")
    print(f"  Projection: {projection}")
    print(f"  Stages:     {' → '.join(stages)}")
    print(f"  Index:      {index_name or 'NONE'}")
    print(f"  Keys exam:  {keys:,}")
    print(f"  Docs exam:  {examined:,}")
    print(f"  Returned:   {returned:,}")
    print(f"  Time:       {stats['executionTimeMillis']}ms")
    print(f"  Sort elim:  {'✅ Yes' if not has_sort_stage else '❌ No (in-memory sort)'}")
    print(f"  Covered:    {'✅ Yes' if is_covered else '❌ No'}")
    print(f"  Rating:     {rating}")
    
    return {
        "stages": stages,
        "index": index_name,
        "keys_examined": keys,
        "docs_examined": examined,
        "returned": returned,
        "time_ms": stats["executionTimeMillis"],
        "sort_eliminated": not has_sort_stage,
        "covered": is_covered,
        "rating": rating
    }

# EQUIVALENT REDSHIFT SQL (for reference)
# ═══════════════════════════════════════════════════════
#
# -- Redshift: Only ONE sort key per table — choose wisely
#
# -- OPTION 1: Optimize for customer queries (A, B, D)
# CREATE TABLE store.orders (
#     order_id      BIGINT IDENTITY(1,1),
#     customer_id   VARCHAR(20),
#     order_date    TIMESTAMP,
#     status        VARCHAR(20),
#     total_amount  DECIMAL(12,2),
#     items_count   INTEGER,
#     shipping_city VARCHAR(100)
# )
# DISTSTYLE KEY DISTKEY(customer_id)
# COMPOUND SORTKEY(customer_id, order_date);
#
# -- Query A: ✅ Uses sort key prefix (customer_id) + sort key (order_date)
# SELECT * FROM store.orders
# WHERE customer_id = 'C-00100'
# ORDER BY order_date DESC;
#
# -- Query B: 🟡 Uses sort key prefix (customer_id) but must scan for status
# SELECT * FROM store.orders
# WHERE customer_id = 'C-00100' AND status = 'shipped';
# -- customer_id filter uses zone maps → good
# -- status filter requires scanning within customer blocks → medium
#
# -- Query C: ❌ Sort key prefix (customer_id) NOT in filter
# SELECT * FROM store.orders
# WHERE status = 'pending'
#   AND order_date >= '2024-04-01' AND order_date < '2024-05-01'
# ORDER BY total_amount DESC
# LIMIT 20;
# -- Full table scan → slow
# -- In-memory sort by total_amount → expensive
#
# -- Query D: ✅ Uses sort key prefix + range on sort key
# SELECT * FROM store.orders
# WHERE customer_id = 'C-00100'
#   AND order_date >= '2024-01-01'
#   AND order_date < '2024-04-01';
#
# -- To help Query C, would need different sort key:
# -- OPTION 2: COMPOUND SORTKEY(status, order_date)
# -- But then Queries A, B, D lose sort key benefit!
#
# -- OPTION 3: INTERLEAVED SORTKEY(customer_id, status, order_date)
# -- Helps ALL queries somewhat, but none optimally
# -- VACUUM REINDEX takes much longer
#
# -- MongoDB solution: 3 compound indexes, each OPTIMAL for its query

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY COMPOUND INDEX EFFECTIVENESS:

TECHNIQUE 1: explain() stage analysis
    → IXSCAN without SORT = compound index providing sort order ✅
    → IXSCAN with SORT = index used for filter, sort done in memory 🟡
    → COLLSCAN = index not used at all 🔴
    → PROJECTION_COVERED / IXSCAN without FETCH = covered query 🟢

TECHNIQUE 2: Keys examined vs docs examined vs returned
    → Keys examined = docs examined = returned → perfect compound index
    → Keys examined > docs examined → index scanned extra entries (range)
    → Docs examined > returned → fetched docs that didn't match all filters
    → Keys examined = 0 → COLLSCAN (no index used)

TECHNIQUE 3: Index usage statistics
    → db.orders.aggregate([{ $indexStats: {} }])
    → Shows how many times each index was used
    → Compound index with 0 usage → candidate for dropping
    → Run periodically to find unused indexes

TECHNIQUE 4: Atlas Performance Advisor
    → Automatically suggests compound indexes based on slow queries
    → Shows: suggested index, affected queries, expected improvement
    → Often suggests better compound indexes than manual design
    → Like Redshift Advisor but specifically for index optimization

TECHNIQUE 5: Slow query profiler
    → db.setProfilingLevel(1, { slowms: 50 })
    → Captures queries taking >50ms
    → Shows: query filter, sort, execution time, plan summary
    → Use to discover queries that need compound indexes
