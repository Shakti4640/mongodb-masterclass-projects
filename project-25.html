PROJECT 25: Change Streams — Real-Time Event Streaming

Problem: Trigger Notifications When Order Status Changes, Sync Inventory Across Microservices, and Build a Real-Time Activity Feed

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    You are building an e-commerce platform with three real-time requirements:

    REQUIREMENT 1 — Order Status Notifications
        → When order status changes to "shipped", send push notification
        → When order status changes to "delivered", trigger review request
        → Must NOT miss any status change — exactly-once processing
        → Must work even if notification service restarts

    REQUIREMENT 2 — Inventory Synchronization
        → When a product is purchased, decrement inventory in real-time
        → Multiple microservices need to react: warehouse, analytics, recommendations
        → Each service processes changes independently at its own pace
        → If a service goes down, it must resume where it left off

    REQUIREMENT 3 — Real-Time Activity Feed
        → Show live feed: "Alice just purchased X", "Bob reviewed Y"
        → Stream new events to connected WebSocket clients
        → Filter events by type: only purchases, only reviews, only returns

    IN REDSHIFT:
        → No native way to "watch" for changes
        → Must poll: SELECT * FROM orders WHERE updated_at > last_check
        → Or build external pipeline: Kinesis → Lambda → SNS
        → Redshift is a WAREHOUSE — it receives data, it doesn't emit events

    IN MONGODB:
        → Change Streams: native, real-time, resumable event source
        → The database TELLS YOU when data changes
        → No polling, no external event bus required
        → Built on the oplog (Project 23) with consistency guarantees (Project 24)

🔴 REDSHIFT MINDSET — How You Would Solve This

Redshift Has No Native Change Detection

YOUR REDSHIFT APPROACH TO "REACT WHEN DATA CHANGES":

APPROACH 1: Polling (Simple, Terrible)
    -- Run this query every N seconds
    SELECT order_id, status, updated_at
    FROM orders
    WHERE updated_at > '2024-06-01T10:30:00'  -- last poll time
      AND status IN ('shipped', 'delivered');
    
    PROBLEMS:
    → Polling interval = latency (poll every 5s = up to 5s delay)
    → Empty polls waste compute (most polls return zero rows)
    → Redshift charges per query time — polling is EXPENSIVE
    → Race condition: what if row updates BETWEEN polls?
    → What if Redshift cluster is paused/resizing during a poll?
    → Must track "last poll time" externally
    → Must handle duplicates (same row polled twice)
    → DOES NOT SCALE to multiple consumers

APPROACH 2: External Event Pipeline (Complex, Standard)

    ┌────────────────────────────────────────────────────────┐
    │                                                         │
    │  Application                                            │
    │     │                                                   │
    │     ├──→ Write to Redshift (INSERT/UPDATE)              │
    │     │                                                   │
    │     └──→ ALSO publish event to Kinesis/SNS/SQS          │
    │              │                                          │
    │              ├──→ Lambda → Send push notification        │
    │              ├──→ Lambda → Update inventory service      │
    │              └──→ Lambda → Feed to activity stream       │
    │                                                         │
    └────────────────────────────────────────────────────────┘
    
    PROBLEMS:
    → Application must publish events SEPARATELY from database writes
    → Two writes: database + event bus → can get out of sync
    → "Dual write problem": what if DB write succeeds but event fails?
      → Data in Redshift, but no notification sent
      → Or: event sent, but DB write fails
      → NO ATOMIC guarantee between the two
    → Application complexity: every write path must emit events
    → Must build and maintain the entire event infrastructure
    → Must handle Kinesis shard management, Lambda concurrency, etc.

APPROACH 3: Redshift Streaming Ingestion (Receive Only)
    -- Redshift CAN ingest FROM Kinesis streams
    CREATE EXTERNAL SCHEMA kinesis_schema
    FROM KINESIS
    IAM_ROLE 'arn:aws:iam::role/redshift-kinesis';
    
    -- But this is INBOUND only
    -- Redshift receives events FROM Kinesis
    -- Redshift CANNOT emit events TO Kinesis natively
    -- Direction: Kinesis → Redshift (not Redshift → Kinesis)

FUNDAMENTAL LIMITATION:
    → Redshift is a DATA SINK — data flows IN
    → Redshift is NOT a DATA SOURCE for events
    → To emit events, you need EXTERNAL infrastructure
    → The database and the event system are SEPARATE concerns
    → You maintain TWO systems: warehouse + event pipeline

IN MONGODB:
    → The database IS the event source
    → ONE write to MongoDB = data stored AND event emitted
    → No dual-write problem
    → No external event infrastructure needed (for basic use cases)
    → Change Streams: native, real-time, resumable, filtered

Change Stream Event Structure

WHAT DOES A CHANGE EVENT LOOK LIKE?

When you update an order's status:
    db.orders.updateOne(
        { _id: "ORD-001" },
        { $set: { status: "shipped", shipped_at: new Date() } }
    )

The change stream emits:
    {
        _id: {                              ← RESUME TOKEN
            _data: "8264A5B2C..."           (opaque, used for resuming)
        },
        operationType: "update",            ← WHAT happened
        clusterTime: Timestamp(1718000000, 1),  ← WHEN it happened
        ns: {                               ← WHERE it happened
            db: "ecommerce",
            coll: "orders"
        },
        documentKey: {                      ← WHICH document
            _id: "ORD-001"
        },
        updateDescription: {                ← WHAT changed (for updates)
            updatedFields: {
                status: "shipped",
                shipped_at: ISODate("2024-06-10T...")
            },
            removedFields: [],
            truncatedArrays: []
        },
        fullDocument: {                     ← FULL document (if requested)
            _id: "ORD-001",
            customer_id: "CUST-042",
            status: "shipped",
            items: [...],
            total: 149.99,
            shipped_at: ISODate("2024-06-10T...")
        }
    }

KEY PARTS:
    → _id (resume token): bookmark for exactly-once processing
    → operationType: insert | update | replace | delete | drop | ...
    → documentKey: which document changed
    → updateDescription: WHAT fields changed (delta, not full doc)
    → fullDocument: optionally include the complete document
    → clusterTime: when it happened (ties to Project 24 consistency)

Change Stream Scope Levels

THREE LEVELS OF WATCHING:

LEVEL 1: Collection-level (most common)
    → Watch ONE specific collection
    → db.orders.watch()
    → Emits changes to orders collection only

LEVEL 2: Database-level
    → Watch ALL collections in a database
    → db.watch()
    → Emits changes to any collection in "ecommerce" database
    → Includes collection drops and renames

LEVEL 3: Deployment-level (cluster-wide)
    → Watch ALL databases, ALL collections
    → db.getMongo().watch()
    → Emits every change in the entire MongoDB deployment
    → Includes database drops and cluster-level events

🔧 HOW TO APPROACH IT IN MONGODB

THINKING FRAMEWORK FOR CHANGE STREAMS:

    STEP 1: Identify WHAT changes you care about
        → Which collection(s)?
        → Which operation types? (insert? update? delete? all?)
        → Which documents? (all orders? only status changes? only high-value?)
        → Which fields? (only status field? only price changes?)

    STEP 2: Design the PIPELINE FILTER
        → Use aggregation pipeline stages (Project 12-13 knowledge)
        → $match: filter by operationType, namespace, field changes
        → $project: include only needed fields in the event
        → $addFields: compute derived fields for the consumer

    STEP 3: Choose FULL DOCUMENT behavior
        → "default": no full document (just delta for updates)
        → "updateLookup": include full document at read time
        → "whenAvailable": include full document if available (MongoDB 6.0+)
        → "required": include full document, error if unavailable (MongoDB 6.0+)

    STEP 4: Plan RESUME strategy
        → Store resume token after processing each event
        → On restart: pass resume token to resume where you left off
        → Storage: file, database, Redis — any durable store

    STEP 5: Choose CONSISTENCY level
        → Default: readConcern "majority" (only emits durable changes)
        → This means: change stream will NEVER emit a change
          that gets rolled back (Project 24 knowledge)
        → SAFE by default — unlike readConcern "local" for regular reads

// THE PROBLEM:
// → Your notification service crashes at 3:42 PM
// → 5 orders shipped between 3:42 and 3:45 PM (when service restarts)
// → WITHOUT resume: those 5 notifications are LOST forever
// → WITH resume: service picks up exactly where it left off

// HOW IT WORKS:
// 1. After processing each event, store its resume token
// 2. On restart, pass the stored token to watch()
// 3. MongoDB replays events FROM that point forward
// 4. No events lost, no events duplicated

// Step 1: Check if we have a stored resume token

var watchOptions = {
    fullDocument: "updateLookup"
}

    watchOptions.resumeAfter = stored.token
    print(`Resuming from stored token: ${JSON.stringify(stored.token).substring(0, 50)}...`)
} else {

class ChangeStreamConsumer:
    """
    Production-grade change stream consumer with:
    - Resume token persistence (exactly-once processing)
    - Error handling with automatic retry
    - Graceful shutdown
    - Configurable pipeline filtering
    
    Redshift equivalent: NONE
    → Would require: Kinesis + Lambda + DynamoDB checkpoint table
    → All managed separately, all can fail independently
    → MongoDB: one class, one connection, built-in reliability
    """
    
    def __init__(self, collection, consumer_name: str,
                 pipeline: list = None, full_document: str = "updateLookup"):
        self.collection = collection
        self.consumer_name = consumer_name
        self.pipeline = pipeline or []
        self.full_document = full_document
        self.running = False
        self.events_processed = 0
        self.resume_collection = db["resume_tokens"]
        
    def _get_resume_token(self) -> dict:
        """Load last processed resume token from database"""
        stored = self.resume_collection.find_one(
            {"_id": self.consumer_name}
        )
        if stored and "token" in stored:
            print(f"  [{self.consumer_name}] Resuming from stored token")
            return stored["token"]
        print(f"  [{self.consumer_name}] No stored token — starting fresh")
        return None
    
    def _save_resume_token(self, token: dict):
        """Persist resume token after processing event"""
        self.resume_collection.update_one(
            {"_id": self.consumer_name},
            {
                "$set": {
                    "token": token,
                    "updated_at": datetime.now(),
                    "events_processed": self.events_processed
                }
            },
            upsert=True
        )
    
    def _build_watch_options(self, resume_token) -> dict:
        """Build watch() options including resume token"""
        options = {
            "full_document": self.full_document
        }
        
        # Try fullDocumentBeforeChange if available (MongoDB 6.0+)
        try:
            options["full_document_before_change"] = "whenAvailable"
        except Exception:
            pass
        
        if resume_token:
            options["resume_after"] = resume_token
            
        return options
    
    def process_event(self, event: dict):
        """Override this in subclasses to handle specific event types"""
        raise NotImplementedError("Subclasses must implement process_event()")
    
    def start(self, max_events: int = None):
        """
        Start consuming change stream events.
        
        max_events: stop after N events (for testing)
                    None = run forever (production)
        """
        self.running = True
        resume_token = self._get_resume_token()
        
        while self.running:
            try:
                options = self._build_watch_options(resume_token)
                
                with self.collection.watch(
                    self.pipeline, **options
                ) as stream:
                    
                    print(f"  [{self.consumer_name}] Stream opened — listening...")
                    
                    for event in stream:
                        if not self.running:
                            break
                        
                        # Process the event
                        try:
                            self.process_event(event)
                            self.events_processed += 1
                        except Exception as e:
                            print(f"  [{self.consumer_name}] "
                                  f"Error processing event: {e}")
                            # Continue processing — don't lose the token
                        
                        # ALWAYS save resume token AFTER processing
                        # This is the KEY to exactly-once semantics
                        self._save_resume_token(event["_id"])
                        resume_token = event["_id"]
                        
                        if max_events and self.events_processed >= max_events:
                            print(f"  [{self.consumer_name}] "
                                  f"Reached {max_events} events — stopping")
                            self.running = False
                            break
                            
            except PyMongoError as e:
                print(f"  [{self.consumer_name}] Stream error: {e}")
                print(f"  [{self.consumer_name}] Retrying in 5 seconds...")
                time.sleep(5)
                # Loop continues — will reconnect with last resume token
                # THIS IS WHY RESUME TOKENS MATTER:
                # → Stream broke? Reconnect from last processed event
                # → No events lost, no events duplicated
                
        print(f"  [{self.consumer_name}] Stopped. "
              f"Processed {self.events_processed} events total.")
    
    def stop(self):
        """Graceful shutdown"""
        self.running = False

class OrderNotificationConsumer(ChangeStreamConsumer):
    """
    Watches for order status changes → sends notifications.
    
    Filters: only "shipped" and "delivered" status changes
    Action: log notification (in production: push notification, email)
    """
    
    def __init__(self):
        pipeline = [
            {
                "$match": {
                    "operationType": "update",
                    "$or": [
                        {"updateDescription.updatedFields.status": "shipped"},
                        {"updateDescription.updatedFields.status": "delivered"}
                    ]
                }
            }
        ]
        
        super().__init__(
            collection=db["orders"],
            consumer_name="notification_service",
            pipeline=pipeline,
            full_document="updateLookup"
        )
    
    def process_event(self, event: dict):
        """Process order status change → send notification"""
        new_status = event["updateDescription"]["updatedFields"]["status"]
        order_id = event["documentKey"]["_id"]
        full_doc = event.get("fullDocument", {})
        customer = full_doc.get("customer_name", "Unknown")
        total = full_doc.get("total", 0)
        
        # Determine notification type
        if new_status == "shipped":
            notification = {
                "type": "push_notification",
                "title": "Your order has shipped! 📦",
                "body": f"Order {order_id} (${total:.2f}) is on its way!",
                "customer_id": full_doc.get("customer_id"),
                "customer_name": customer,
                "order_id": order_id,
                "timestamp": datetime.now()
            }
        elif new_status == "delivered":
            notification = {
                "type": "email",
                "subject": "Your order has been delivered! ✅",
                "body": f"Order {order_id} delivered. Please leave a review!",
                "customer_id": full_doc.get("customer_id"),
                "customer_name": customer,
                "order_id": order_id,
                "timestamp": datetime.now()
            }
        
        # In production: send via push service / email service
        # Here: log to notifications collection
        db.notifications_log.insert_one(notification)
        
        print(f"  📬 NOTIFICATION: {notification['title']}")
        print(f"     To: {customer} | Order: {order_id}")
        
        # Check pre-image if available (MongoDB 6.0+)
        pre_image = event.get("fullDocumentBeforeChange")
        if pre_image:
            old_status = pre_image.get("status", "unknown")
            print(f"     Status change: {old_status} → {new_status}")

    """
    Watches for new orders → decrements product inventory.

    """
    
    def __init__(self):

            {
                "$match": {
                    "operationType": "insert"
                }
            }
        ]
        
        super().__init__(
            collection=db["orders"],
            consumer_name="inventory_service",
            pipeline=pipeline,
            full_document="updateLookup"
        )
    
    def process_event(self, event: dict):
        """Process new order → decrement inventory"""
        order = event["fullDocument"]
        order_id = order["_id"]
        
        print(f"  📦 INVENTORY: Processing order {order_id}")
        
        for item in order.get("items", []):
            product_id = item["product_id"]
            qty = item["qty"]
            
            # Atomic decrement
            result = db.products.update_one(
                {"_id": product_id},
                {"$inc": {"stock_qty": -qty}}

            # Check remaining stock
            product = db.products.find_one({"_id": product_id})
            remaining = product["stock_qty"] if product else "N/A"
            
            print(f"     {item['name']}: -{qty} → {remaining} remaining")
            
            # Low stock alert
            if product and product["stock_qty"] < 10:
                print(f"     ⚠️ LOW STOCK: {item['name']} — "

class RobustChangeStreamConsumer:
    """
    Production-hardened change stream consumer.
    
    Handles:
    - Transient network errors (auto-retry)
    - Oplog overflow (resume token expired)
    - Collection drop/rename (invalidate events)
    - Primary stepdown (election during watch)
    - Application-level processing errors
    
    Redshift equivalent: Would require building ALL of this
    into your Kinesis consumer + Lambda error handling +
    DynamoDB checkpoint recovery. MongoDB gives you the
    building blocks natively.
    """
    
    def __init__(self, collection, consumer_name: str,
                 pipeline: list = None):
        self.collection = collection
        self.consumer_name = consumer_name
        self.pipeline = pipeline or []
        self.running = False
        self.resume_collection = db["resume_tokens"]
        self.error_log = db["consumer_errors"]
        self.max_retries = 10
        self.base_retry_delay = 1  # seconds
        self.max_retry_delay = 60  # seconds
    
    def _get_resume_token(self):
        stored = self.resume_collection.find_one({"_id": self.consumer_name})
        return stored.get("token") if stored else None
    
    def _save_resume_token(self, token):
        self.resume_collection.update_one(
            {"_id": self.consumer_name},
            {"$set": {"token": token, "updated_at": datetime.now()}},
            upsert=True
        )
    
    def _clear_resume_token(self):
        """Clear stored token when it's no longer valid"""
        self.resume_collection.delete_one({"_id": self.consumer_name})
    
    def _log_error(self, error_type: str, details: str):
        self.error_log.insert_one({
            "consumer": self.consumer_name,
            "error_type": error_type,
            "details": details,
            "timestamp": datetime.now()
        })
    
    def process_event(self, event: dict):
        """Override in subclass"""
        raise NotImplementedError
    
    def start(self, max_events: int = None):
        self.running = True
        events_processed = 0
        consecutive_errors = 0
        
        while self.running:
            resume_token = self._get_resume_token()
            
            try:
                # Build watch options
                watch_kwargs = {"full_document": "updateLookup"}
                if resume_token:
                    watch_kwargs["resume_after"] = resume_token
                
                with self.collection.watch(
                    self.pipeline, **watch_kwargs
                ) as stream:
                    
                    consecutive_errors = 0  # reset on successful connection
                    print(f"  [{self.consumer_name}] Connected to stream")
                    
                    for event in stream:
                        if not self.running:
                            break
                        
                        try:
                            # ─── PROCESS EVENT ────────────────
                            self.process_event(event)
                            events_processed += 1
                            
                            # ─── SAVE CHECKPOINT ─────────────
                            self._save_resume_token(event["_id"])
                            
                            # ─── CHECK LIMITS ────────────────
                            if max_events and events_processed >= max_events:
                                self.running = False
                                break
                                
                        except Exception as proc_error:
                            # APPLICATION-LEVEL error
                            # Event was received but processing failed
                            # Decision: skip and continue vs retry vs dead-letter
                            
                            print(f"  [{self.consumer_name}] "
                                  f"Processing error: {proc_error}")
                            
                            self._log_error(
                                "processing_error",
                                f"Event: {event['documentKey']}, "
                                f"Error: {str(proc_error)}"
                            )
                            
                            # STILL save resume token
                            # → We don't want to reprocess a broken event forever
                            # → Log it, skip it, move on
                            # → In production: send to dead-letter queue
                            self._save_resume_token(event["_id"])
                            
            except PyMongoError as e:
                error_str = str(e)
                consecutive_errors += 1
                
                # ─── CASE 1: Resume token expired (oplog overflow) ──
                if "resume token" in error_str.lower() or \
                   "oplog" in error_str.lower():
                    print(f"  [{self.consumer_name}] "
                          f"Resume token expired! Oplog overflowed.")
                    print(f"  [{self.consumer_name}] "
                          f"EVENTS WERE LOST between last checkpoint "
                          f"and current oplog start")
                    
                    self._log_error("oplog_overflow",
                                    "Resume token expired — events lost")
                    self._clear_resume_token()
                    # Next iteration: starts from current time
                    # LOST EVENTS: must be reconciled from other sources
                    
                # ─── CASE 2: Invalidate (collection dropped/renamed) ──
                elif "invalidate" in error_str.lower():
                    print(f"  [{self.consumer_name}] "
                          f"Collection was dropped or renamed!")
                    self._log_error("invalidated",
                                    "Collection dropped or renamed")
                    self._clear_resume_token()
                    # Consumer must decide: stop or watch new collection
                    
                # ─── CASE 3: Transient error (network, election) ──
                else:
                    # Exponential backoff with jitter
                    delay = min(
                        self.base_retry_delay * (2 ** (consecutive_errors - 1)),
                        self.max_retry_delay
                    )
                    
                    print(f"  [{self.consumer_name}] "
                          f"Stream error (attempt {consecutive_errors}): {e}")
                    print(f"  [{self.consumer_name}] "
                          f"Retrying in {delay}s...")
                    
                    self._log_error("transient_error", error_str)
                    
                    if consecutive_errors >= self.max_retries:
                        print(f"  [{self.consumer_name}] "
                              f"Max retries reached — stopping")
                        self._log_error("max_retries",
                                        "Consumer stopped after max retries")
                        self.running = False
                        break
                    
                    time.sleep(delay)
        
        print(f"  [{self.consumer_name}] Stopped. "
              f"Total events: {events_processed}")
    
    def stop(self):
        self.running = False

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

ADVANCED CHANGE STREAM OPTIMIZATIONS:

1. PIPELINE PUSHDOWN
   → Filter in the pipeline, not in application code
   → $match in pipeline: MongoDB filters AT THE OPLOG LEVEL
   → Application-side filter: MongoDB sends ALL events, you discard
   → 100x difference in network traffic for selective consumers

   GOOD (filtered at source):
   db.orders.watch([{ $match: { operationType: "update" } }])
   
   BAD (filter in application):
   for event in db.orders.watch():
       if event["operationType"] == "update":  # wastes bandwidth
           process(event)

2. BATCH PROCESSING WITH CHANGE STREAMS
   → Don't process one event at a time
   → Collect events into batches, process batch, save ONE token
   
   batch = []
   for event in stream:
       batch.append(event)
       if len(batch) >= 100:
           process_batch(batch)
           save_token(batch[-1]["_id"])  # save LAST token in batch
           batch = []

3. fullDocument "whenAvailable" vs "updateLookup" (MongoDB 6.0+)
   → "updateLookup": does a SEPARATE findOne after each change event
     → Extra read per event
     → Might return a NEWER version than what triggered the event
   → "whenAvailable": uses post-image stored WITH the oplog entry
     → No extra read
     → Returns EXACT document as it was at change time
     → Requires: changeStreamPreAndPostImages enabled on collection
   → "required": same as "whenAvailable" but ERRORS if not available
   → ALWAYS prefer "whenAvailable" over "updateLookup" on MongoDB 6.0+

4. SHARDED CLUSTER CHANGE STREAMS
   → On sharded collections: mongos merges events from all shards
   → Events arrive in GLOBAL order (by clusterTime)
   → More shards = more internal coordination
   → Pipeline filtering becomes even MORE important
     (reduces cross-shard event merging work)

✅ WHAT SHOULD DO

SHOULD DO:
    ✅ ALWAYS store resume tokens after processing each event
       → Without this: events lost on consumer restart
       → Store in MongoDB, Redis, or any durable store
       → Save AFTER successful processing, not before
    
    ✅ Use pipeline filtering to reduce event volume
       → Filter at the stream level, not in application code
       → $match as first pipeline stage for maximum efficiency
    
    ✅ Handle the "invalidate" event type
       → Emitted when collection is dropped or renamed
       → Your consumer must decide: stop, restart, or watch new collection
    
    ✅ Implement exponential backoff for stream reconnection
       → Transient errors (network, election) are NORMAL
       → Don't retry immediately — backoff to avoid overwhelming server
    
    ✅ Use "whenAvailable" over "updateLookup" on MongoDB 6.0+
       → Faster (no extra read)
       → More accurate (exact document at change time)
       → Enable changeStreamPreAndPostImages on collection first
    
    ✅ Monitor consumer lag
       → Compare consumer's last processed clusterTime to current clusterTime
       → Alert if lag approaches oplog window size
       → Lag > 50% of oplog window = critical alert
    
    ✅ Use separate resume tokens per consumer
       → Notification service: own token
       → Inventory service: own token
       → Each processes at its own pace independently

REDSHIFT HABITS TO UNLEARN:
    🔄 Stop thinking "I need an external event bus for change detection"
       → MongoDB IS the event source — no Kinesis/SNS needed for basic use
    🔄 Stop thinking "polling is the only way to detect changes"
       → Change streams are push-based — no polling overhead
    🔄 Stop thinking "the database just stores data"
       → MongoDB stores data AND emits events — unified
    🔄 Stop building dual-write patterns
       → One write to MongoDB = data stored + event available
       → No separate "publish to event bus" step needed

❌ WHAT SHOULD NOT DO

DO NOT:
    ❌ Forget to store resume tokens
       → Consumer restart = all events since last start are LOST
       → THIS IS THE #1 MISTAKE with change streams
       → Always persist token after EACH event (or each batch)
    
    ❌ Use change streams without a replica set
       → Change streams REQUIRE replica set or sharded cluster
       → Standalone mongod: change streams are NOT available
       → Atlas: always a replica set (works out of the box)
    
    ❌ Open too many change streams on one collection
       → Each stream tails the oplog independently
       → 100 streams on one collection = 100 oplog cursors
       → Better: one stream → fan-out in application code
       → Or: use Kafka as the fan-out layer (Project 44)
    
    ❌ Rely on change streams for guaranteed ordering across collections
       → Within one collection: events ordered by clusterTime
       → Across collections: events from different collections
         may interleave unpredictably
       → For cross-collection ordering: use database-level watch
         with careful clusterTime comparison
    
    ❌ Use change streams as your ONLY event store
       → Oplog has a LIMITED window (hours to days)
       → After the window: events are gone forever
       → For long-term event storage: pipe to Kafka, S3, or a
         dedicated events collection
    
    ❌ Process events with heavy computation inside the stream loop
       → Slow processing = consumer falls behind
       → Consumer falls behind = approaches oplog window limit
       → Oplog window exceeded = events LOST
       → Better: receive event → enqueue for async processing → save token
    
    ❌ Assume fullDocument is always available
       → For "delete" events: document is gone — no fullDocument
       → For "updateLookup": document might have been deleted
         between the change and the lookup
       → Always handle None/missing fullDocument gracefully
    
    ❌ Ignore the "invalidate" event
       → Collection dropped → stream emits "invalidate" → stream closes
       → If you don't handle this: consumer hangs or crashes
       → Must reconnect (possibly to a different/recreated collection)

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Internals — Change Streams and the Oplog

HOW CHANGE STREAMS WORK INSIDE:

    LAYER 1: THE OPLOG (Project 23 foundation)
    ┌─────────────────────────────────────────────────────────┐
    │  The oplog is a CAPPED COLLECTION in the "local" database│
    │  Every write to the replica set → oplog entry            │
    │                                                          │
    │  local.oplog.rs:                                         │
    │  { ts: Timestamp(100,1), op: "i", ns: "ecommerce.orders",│
    │    o: { _id: "ORD-001", status: "processing", ... } }   │
    │  { ts: Timestamp(100,2), op: "u", ns: "ecommerce.orders",│
    │    o2: { _id: "ORD-001" },                               │
    │    o: { $set: { status: "shipped" } } }                  │
    │                                                          │
    │  Capped: oldest entries automatically deleted when full   │
    │  THIS is why the oplog window is limited                 │
    └─────────────────────────────────────────────────────────┘

    LAYER 2: TAILABLE CURSOR ON OPLOG
    ┌─────────────────────────────────────────────────────────┐
    │  Change stream = special tailable cursor on the oplog    │
    │                                                          │
    │  Normal cursor: reads existing data, then closes         │
    │  Tailable cursor: reads existing data, then WAITS for    │
    │                   new data to arrive (like "tail -f")    │
    │                                                          │
    │  The cursor stays open indefinitely                      │
    │  New oplog entries → cursor returns them immediately     │
    │  No polling — cursor is PUSHED new entries by the server │
    └─────────────────────────────────────────────────────────┘

    LAYER 3: SERVER-SIDE PIPELINE PROCESSING
    ┌─────────────────────────────────────────────────────────┐
    │  Your pipeline filter runs ON THE SERVER                 │
    │                                                          │
    │  db.orders.watch([                                       │
    │      { $match: { "updateDescription.updatedFields.status"│
    │                   : "shipped" } }                        │
    │  ])                                                      │
    │                                                          │
    │  MongoDB applies $match to EACH oplog entry              │
    │  Only matching entries are sent to the client            │
    │  Non-matching entries are silently skipped               │
    │  → Saves network bandwidth                              │
    │  → Saves client processing time                         │
    │  → Filter runs at oplog speed (very fast)               │
    └─────────────────────────────────────────────────────────┘

    LAYER 4: EVENT TRANSFORMATION
    ┌─────────────────────────────────────────────────────────┐
    │  Raw oplog entry → Change event document                 │
    │                                                          │
    │  Oplog entry (internal format):                          │
    │  { ts: Timestamp(100,2), op: "u",                        │
    │    ns: "ecommerce.orders",                               │
    │    o2: { _id: "ORD-001" },                               │
    │    o: { $v: 2, diff: { u: { status: "shipped" } } } }   │
    │                                                          │
    │  Change event (user-facing format):                      │
    │  { _id: { _data: "826..." },    ← resume token          │
    │    operationType: "update",      ← human-readable        │
    │    ns: { db: "ecommerce",        ← structured namespace  │
    │          coll: "orders" },                                │
    │    documentKey: { _id: "ORD-001" },                      │
    │    updateDescription: {          ← clear delta format    │
    │        updatedFields: { status: "shipped" },             │
    │        removedFields: []                                 │
    │    } }                                                   │
    │                                                          │
    │  MongoDB transforms the internal oplog format            │
    │  into a clean, documented, stable API                    │
    │  Oplog format can change between versions                │
    │  Change event format is a stable public API              │
    └─────────────────────────────────────────────────────────┘

    LAYER 5: RESUME TOKEN MECHANICS
    ┌─────────────────────────────────────────────────────────┐
    │  Resume token = encoded oplog position                   │
    │                                                          │
    │  Contains:                                               │
    │  → Cluster time (when the change happened)               │
    │  → Token type (identifies the format version)            │
    │  → Document key (which document changed)                 │
    │  → UUID (which collection, surviving rename/drop)        │
    │                                                          │
    │  When you pass resumeAfter = token:                      │
    │  → MongoDB seeks to that position in the oplog           │
    │  → Starts returning events AFTER that position           │
    │  → If oplog has been trimmed past that position:         │
    │    → ERROR: "resume of change stream not possible"       │
    │    → Must start from current time or startAtOperationTime│
    │                                                          │
    │  resumeAfter vs startAfter:                              │
    │  → resumeAfter: fails if collection was dropped          │
    │  → startAfter: works even after drop + recreate          │
    │  → startAfter: recommended for most production use       │
    └─────────────────────────────────────────────────────────┘

    LAYER 6: SHARDED CLUSTER CHANGE STREAMS
    ┌─────────────────────────────────────────────────────────┐
    │  On sharded clusters: each shard has its own oplog       │
    │                                                          │
    │  Client ──→ mongos ──→ opens cursor on EACH shard        │
    │                    ──→ merges events by clusterTime       │
    │                    ──→ returns globally ordered stream    │
    │                                                          │
    │  mongos handles:                                         │
    │  → Opening tailable cursors on all shard primaries       │
    │  → Merging events in clusterTime order                   │
    │  → Handling shard additions/removals                     │
    │  → Resume across all shards from one token               │
    │                                                          │
    │  Performance impact:                                     │
    │  → More shards = more cursors = more coordination        │
    │  → Pipeline filtering reduces per-shard event volume     │
    │  → Global ordering adds slight latency vs single shard   │
    └─────────────────────────────────────────────────────────┘

Redshift Internals — For Comparison

WHY REDSHIFT CANNOT HAVE CHANGE STREAMS:

    1. NO OPLOG EQUIVALENT
       → Redshift's commit log is internal — not exposed
       → No tailable cursor mechanism exists
       → Write path: leader → compute nodes → columnar blocks
       → No replication log that external consumers can read

    2. COLUMNAR STORAGE IS NOT CHANGE-FRIENDLY
       → Changes to one row touch MULTIPLE column files
       → No single "change record" exists for a row update
       → UPDATE in Redshift = soft delete old row + insert new row
       → This happens across column blocks — no unified change event

    3. BATCH-ORIENTED ARCHITECTURE
       → Redshift is designed for BULK operations
       → Single-row changes are expensive (not optimized for)
       → Change detection at row level contradicts the design
       → Redshift thinks in "load millions of rows" not "one row changed"

    4. NO REPLICA SERVING INDEPENDENT READS
       → No secondaries to tail
       → Leader coordinates everything
       → No mechanism for external consumers to "subscribe"

    FUNDAMENTAL DIFFERENCE:
    → MongoDB: document-oriented → one write = one event = one document
    → Redshift: columnar → one write = changes across many column files
    → MongoDB's document model ENABLES change streams naturally
    → Redshift's columnar model makes per-row change detection expensive

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 25:

PITFALL 1: "I'll poll MongoDB like I polled Redshift"
    ❌ setInterval(() => db.orders.find({ updated_at: { $gt: lastPoll } }), 5000)
    ✅ db.orders.watch(pipeline)
    WHY: Polling is your Redshift instinct
         Change streams are PUSH-BASED — zero wasted queries
         100-1000x lower latency, zero cost for idle periods

PITFALL 2: "Change streams will work on my standalone mongod"
    ❌ Running mongod without --replSet and trying to watch()
    ✅ Always use replica set (even single-node replica set for dev)
    WHY: Change streams require the oplog
         Oplog only exists on replica sets
         For local dev: rs.initiate() on single node

PITFALL 3: "I don't need resume tokens — my service never crashes"
    ❌ Starting watch() without resume capability
    ✅ ALWAYS persist resume tokens
    WHY: Services ALWAYS crash eventually
         Deployments require restarts
         Network blips close connections
         Without resume: every restart = lost events

PITFALL 4: "I'll filter events in my application code"
    ❌ for event in collection.watch():
           if event["operationType"] == "update":  # client-side filter
    ✅ collection.watch([{"$match": {"operationType": "update"}}])
    WHY: Client-side filter: ALL events sent over network, then discarded
         Pipeline filter: only matching events sent
         10x-100x bandwidth difference on busy collections

PITFALL 5: "fullDocument always has the data I need"
    ❌ customer_name = event["fullDocument"]["customer_name"]  # may crash
    ✅ customer_name = event.get("fullDocument", {}).get("customer_name", "N/A")
    WHY: fullDocument can be None for delete events
         With "updateLookup": document might be deleted between change and lookup
         With "whenAvailable": might not be available if images disabled
         ALWAYS handle missing fullDocument

PITFALL 6: "I'll open a change stream for each WebSocket client"
    ❌ 10,000 connected users → 10,000 change streams on same collection
    ✅ ONE change stream → fan-out to 10,000 WebSocket connections in app
    WHY: Each stream = one oplog cursor = server resources
         Fan-out is an APPLICATION concern, not a DATABASE concern
         One stream → in-memory pub/sub → many clients
