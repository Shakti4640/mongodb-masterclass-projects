PROJECT 40: Resharding & Scaling Operations

Problem: Original Shard Key Chosen Poorly — Hot Shard Getting 80% of Traffic

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your ride-sharing platform launched 18 months ago
    
    ORIGINAL DECISION (Project 22 thinking):
        → Collection: rides (2.8 billion documents, 4.2TB)
        → Shard key chosen: { city: 1 }
        → Reasoning at launch: "Most queries filter by city"
        → 5-shard cluster, zone-tagged by region (Project 39)
    
    WHAT HAPPENED OVER 18 MONTHS:
        → New York City alone = 38% of all rides
        → Top 5 cities = 74% of all rides
        → Shard holding NYC = 1.6TB (38% of total data)
        → Other 4 shards = 650GB each
        → NYC shard: CPU at 92%, disk I/O saturated
        → Other shards: CPU at 25%, mostly idle
        → Queries for NYC: 800ms average (was 50ms at launch)
        → Queries for small cities: still 30ms
    
    THE DIAGNOSIS:
        → LOW CARDINALITY shard key → uneven distribution
        → "city" has ~500 unique values for 2.8B documents
        → Single chunk for NYC cannot be split further
        → This is a JUMBO CHUNK — the worst sharding failure mode
    
    THE REQUIREMENT:
        → Fix the shard key to { city: 1, ride_date: 1, _id: 1 }
        → Redistribute data evenly WITHOUT taking the platform offline
        → Zero downtime — rides are happening RIGHT NOW
        → Maintain zone sharding (EU data stays in EU — Project 39)
    
    IN REDSHIFT: ALTER TABLE rides ALTER DISTKEY ride_id;
        → Takes hours but conceptually simple
        → Full table rewrite behind the scenes
        → Queries continue during redistribution (degraded performance)
    
    IN MONGODB: This was IMPOSSIBLE before 5.0
        → Pre-5.0: drop collection, re-shard from scratch (massive downtime)
        → MongoDB 5.0+: live resharding (online, zero downtime)
        → MongoDB 7.0+: improved resharding with less overhead
        → But it's COMPLEX — many moving parts

-- CHECK DISTRIBUTION SKEW
-- Redshift provides system tables to see this
SELECT
    slice,
    COUNT(*) as row_count,
    SUM(COUNT(*)) OVER() as total,
    ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER(), 1) as pct
FROM stv_blocklist b
JOIN stv_tbl_perm t ON b.tbl = t.id
WHERE t.name = 'rides'
GROUP BY slice
ORDER BY row_count DESC;

-- RESULT (skewed):
-- slice | row_count   | pct
-- 0     | 1,064,000,000 | 38.0%    ← NYC slice (overloaded)
-- 1     |   448,000,000 | 16.0%
-- 2     |   420,000,000 | 15.0%
-- ...

-- FIX IN REDSHIFT: Change distribution key
ALTER TABLE rides ALTER DISTKEY ride_id;

-- WHAT HAPPENS:
-- → Redshift creates new table behind the scenes
-- → Redistributes ALL rows by ride_id (high cardinality)
-- → Swaps old and new table atomically
-- → Queries continue during redistribution (may be slower)
-- → Takes hours for 4.2TB table
-- → ride_id has billions of unique values → even distribution
-- → ONE COMMAND, conceptually simple

-- ALTERNATIVE: Change to EVEN distribution
ALTER TABLE rides ALTER DISTSTYLE EVEN;
-- → Round-robin: perfectly even, but no co-location benefit

-- VERIFY NEW DISTRIBUTION
SELECT slice, COUNT(*), ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER(), 1)
FROM stv_blocklist b
JOIN stv_tbl_perm t ON b.tbl = t.id
WHERE t.name = 'rides'
GROUP BY slice
ORDER BY 1;
-- Now: ~20% per slice (5 slices) → balanced!

What Redshift Does Internally During Redistribution

REDSHIFT ALTER DISTKEY PROCESS:

Step 1: Create shadow copy of table with new DISTKEY
    → New physical table created on all slices
    → Old table still serving queries

Step 2: Redistribute rows
    → Each row re-hashed on new DISTKEY value
    → Rows shipped to correct new slice
    → Network traffic between slices (MPP interconnect)
    → Progress: viewable in SYS_QUERY_HISTORY

Step 3: Replay changes
    → Any INSERT/UPDATE/DELETE during redistribution captured
    → Replayed on new table after bulk copy

Step 4: Atomic swap
    → Old table pointer → new table
    → Old physical data deleted
    → DONE

CHARACTERISTICS:
    → Single command
    → Automatic process
    → No application changes needed
    → Performance degraded during redistribution
    → Cannot be cancelled mid-way (commit or rollback)
    → Sort key preserved
    → Encoding preserved
    → Takes hours for multi-TB tables

🟢 MONGODB MINDSET — How This Same Problem Is Approached

Why Resharding Was Historically MongoDB's Biggest Weakness

MONGODB SHARD KEY HISTORY:

Pre-MongoDB 4.4:
    → Shard key IMMUTABLE after creation
    → Cannot change shard key AT ALL
    → Only fix: create new collection, reshard, migrate data
    → Massive downtime required

MongoDB 4.4:
    → Refinable shard keys: add suffix fields to existing key
    → { city: 1 } → { city: 1, ride_date: 1 }
    → But: only ADDS fields, cannot CHANGE the base key
    → Helps with jumbo chunks but doesn't fix fundamental skew

MongoDB 5.0+:
    → LIVE RESHARDING: change shard key completely
    → Zero downtime, online operation
    → Application continues reading/writing during resharding
    → THIS changed everything for MongoDB at scale

MongoDB 7.0+:
    → Improved resharding performance
    → Less temporary storage needed
    → Better coordination with balancer
    → Resharding can be combined with zone changes

The Three-Phase Resharding Architecture

│                                                                      │

┌─────────────────────────────────────────────────────────────────────┐
│ PHASE 3: COMMIT (Critical Section — Brief Blocking)                  │
│                                                                      │
│  1. mongos blocks writes momentarily (milliseconds to seconds)       │
│  2. Final oplog entries replayed                                     │
│  3. Catalog updated: collection now points to new sharded layout     │
│  4. mongos resumes routing to new shard key ranges                   │
│  5. Temporary collection BECOMES the real collection                 │
│  6. Old chunks cleaned up asynchronously                             │
│                                                                      │
│  APPLICATION SEES: brief latency spike, then improved performance    │

🧩 HOW TO APPROACH IT IN MONGODB

DIAGNOSIS → DECISION → EXECUTION → VERIFICATION

STEP 1 — DIAGNOSE THE PROBLEM
    → Identify skew: chunk distribution per shard
    → Identify jumbo chunks: chunks that can't be split
    → Identify hot shard: CPU, I/O, latency metrics
    → Determine root cause: shard key cardinality, frequency, monotonicity
    → Reference Project 22 for shard key evaluation criteria

STEP 2 — DECIDE THE NEW SHARD KEY
    → Analyze actual query patterns (not assumed patterns)
    → Choose key with: high cardinality + low frequency + non-monotonic
    → Consider compound key for both distribution AND query targeting
    → Validate with math: unique values >> number of chunks

STEP 3 — CHOOSE THE RESHARDING METHOD
    → Refine (4.4+): add suffix fields — least disruptive
    → Reshard (5.0+): change key completely — most flexible
    → Manual migration: create new collection, migrate — most control

STEP 4 — EXECUTE WITH MONITORING
    → Pre-flight checks: disk space, index readiness, backup
    → Initiate resharding
    → Monitor all three phases
    → Be ready to abort if issues arise

STEP 5 — POST-RESHARDING VERIFICATION
    → Verify even distribution
    → Verify query performance improvement
    → Verify zone assignments still correct (Project 39)
    → Clean up orphaned chunks
    → Update monitoring thresholds

Phase C: Execute Resharding

// ═══════════════════════════════════════════════════════
// STEP 4: EXECUTE LIVE RESHARDING
// ═══════════════════════════════════════════════════════

// ─── OPTION A: REFINE SHARD KEY (MongoDB 4.4+) ──────
// Add suffix fields to EXISTING key
// Less disruptive, but doesn't change base key

// This turns { city: 1 } into { city: 1, ride_date: 1, _id: 1 }
// Existing chunks get the NEW key range boundaries
// Jumbo chunks become splittable (because now city + date + _id)

db.adminCommand({
    refineCollectionShardKey: "rideshare.rides",
    key: { city: 1, ride_date: 1, _id: 1 }
})

// PROS of refine:
//   → Instant (metadata change only — no data movement)
//   → No temporary storage needed
//   → Jumbo chunks become splittable immediately
//   → Balancer will GRADUALLY move data to achieve balance
//
// CONS of refine:
//   → Only ADDS fields, cannot change existing fields
//   → Rebalancing happens slowly via balancer
//   → If base key is fundamentally wrong, this only helps partially
//   → NYC data still starts on shard-a (moves slowly over hours/days)

// ─── OPTION B: FULL RESHARDING (MongoDB 5.0+) ───────
// COMPLETELY change the shard key
// More disruptive but fixes fundamental distribution issues

// IMPORTANT: This is the nuclear option — use when refine isn't enough

db.adminCommand({
    reshardCollection: "rideshare.rides",
    key: { city: 1, ride_date: 1, _id: 1 },
    
    // Optional: set zones for new shard key ranges
    // (replaces Project 39 zone configuration)
    zones: [
        {
            zone: "US-EAST",
            min: { city: "New York", ride_date: MinKey(), _id: MinKey() },
            max: { city: "New York\uffff", ride_date: MaxKey(), _id: MaxKey() }
        },
        {
            zone: "EU",
            min: { city: "London", ride_date: MinKey(), _id: MinKey() },
            max: { city: "London\uffff", ride_date: MaxKey(), _id: MaxKey() }
        }
        // ... more zones as needed
    ],
    
    // Optional: limit write throughput during resharding
    // Prevents resharding from starving application writes
    // numInitialChunks: not specified = auto-calculated
})

// { ok: 1 }
// Resharding has STARTED — it's now running in the background

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

1. PRE-SPLITTING AFTER RESHARDING
   → Don't wait for balancer to split chunks naturally
   → Pre-split into desired ranges for immediate balance:
   
   sh.splitAt("rideshare.rides", {
       city: "New York",
       ride_date: ISODate("2024-06-01"),
       _id: ObjectId("000000000000000000000000")
   })
   
   → Creates split point → balancer distributes faster

2. RESHARDING WITH numInitialChunks
   → During reshardCollection, specify desired chunk count:
   
   db.adminCommand({
       reshardCollection: "rideshare.rides",
       key: { city: 1, ride_date: 1, _id: 1 },
       numInitialChunks: 500
       // 500 chunks / 5 shards = 100 chunks per shard
       // Each chunk ≈ 8.4GB (4.2TB / 500)
   })

3. CONFIGURE CHUNK SIZE
   → Default chunk size: 128MB (MongoDB 6.0+, was 64MB before)
   → Larger chunks = fewer migrations but more skew potential
   → Smaller chunks = more even but more balancer work
   
   use config
   db.settings.updateOne(
       { _id: "chunksize" },
       { $set: { value: 256 } },    // 256MB chunks
       { upsert: true }
   )

4. HASHED SHARD KEY COMPONENT
   → Instead of range-based compound key:
   → { city: 1, ride_date: 1, _id: "hashed" }
   → Hashed _id ensures perfectly even distribution WITHIN city+date
   → Trade-off: range queries on _id no longer efficient

✅ WHAT SHOULD DO

SHOULD DO:

    ✅ MONITOR BEFORE RESHARDING
       → Establish baseline metrics: query latency, CPU, I/O per shard
       → Know what "normal" looks like so you can compare after
    
    ✅ TEST RESHARDING IN STAGING FIRST
       → Clone production data to staging cluster
       → Run resharding there to estimate duration and impact
       → Identify any unexpected issues
    
    ✅ SCHEDULE DURING LOW TRAFFIC
       → Even though it's "zero downtime," there IS performance impact
       → Clone phase: significant disk I/O and network traffic
       → Commit phase: brief write pause
       → Best during maintenance windows or low-traffic periods
    
    ✅ SET BALANCER WINDOWS
       → After resharding, balancer continues fine-tuning
       → Restrict to off-peak hours for production stability
    
    ✅ PREFER refineCollectionShardKey WHEN POSSIBLE
       → Adding suffix fields is instant (metadata-only)
       → Full resharding is a heavy operation
       → If adding fields to the key fixes the skew, prefer refine
    
    ✅ ALWAYS CREATE THE NEW INDEX BEFORE RESHARDING
       → Index build on 2.8B documents takes hours
       → Do this days before the resharding window
       → Resharding fails if supporting index doesn't exist
    
    ✅ VERIFY DISK SPACE ON ALL SHARDS
       → Resharding temporarily doubles collection storage
       → Running out of disk mid-resharding = potential data loss
       → Rule: 2.5x current collection size free per shard
    
    ✅ HAVE ABORT PLAN READY
       → sh.abortReshardCollection() is your emergency exit
       → Know the conditions that trigger abort
       → Document them in your runbook
    
    ✅ UPDATE ZONES DURING RESHARDING (not after)
       → Use the zones parameter in reshardCollection
       → Zones applied atomically with new shard key
       → Avoids temporary zone violations

❌ WHAT SHOULD NOT DO

DO NOT:

    ❌ RESHARD DURING PEAK TRAFFIC
       → Clone phase reads ALL documents from ALL shards
       → Network bandwidth consumed for data transfer
       → Disk I/O increases 2-3x during cloning
       → Application latency will increase 20-50%
       → REDSHIFT TRAP: Redshift ALTER DISTKEY also degrades
         performance, but you're used to it being "just one command"
         — MongoDB resharding requires MORE operational awareness
    
    ❌ IGNORE THE OPLOG GAP DURING MONITORING
       → If oplog entries fetched >> oplog entries applied
       → Gap is GROWING → writes faster than resharding can catch up
       → Resharding may never complete
       → Solution: throttle application writes or increase shard resources
    
    ❌ CHANGE ZONES SEPARATELY AFTER RESHARDING
       → If you reshard WITHOUT specifying zones:
         old zone ranges become invalid (key shape changed)
         data may temporarily violate zone rules
       → Always include zones in the reshardCollection command
    
    ❌ RESHARD TO A MONOTONICALLY INCREASING KEY
       → { _id: 1 } as shard key → all new writes go to ONE shard
       → Same problem you just fixed, different manifestation
       → Project 22 rules still apply to the new key
    
    ❌ ASSUME RESHARDING IS INSTANT LIKE REFINE
       → refineCollectionShardKey = metadata change (seconds)
       → reshardCollection = full data movement (hours to days)
       → Redshift developers expect ALTER DISTKEY speed
         (hours, but unattended)
       → MongoDB resharding needs ACTIVE monitoring throughout
    
    ❌ SKIP THE BACKUP
       → Resharding is the highest-risk operation in MongoDB
       → If something goes catastrophically wrong mid-operation:
         cluster can be left in inconsistent state (rare but possible)
       → Always have a point-in-time recovery option
    
    ❌ RUN MULTIPLE RESHARDING OPERATIONS SIMULTANEOUSLY
       → Only ONE resharding operation per cluster at a time
       → Second reshardCollection command will fail
       → Plan: reshard one collection, verify, then next
    
    ❌ RESIZE SHARDS DURING RESHARDING
       → Don't change shard hardware (Atlas tier) while resharding
       → Compounding two heavy operations = unpredictable behavior
       → Scale shards BEFORE resharding, not during

📍 WHEN TO USE MONGODB OVER REDSHIFT

FOR THIS SPECIFIC PROBLEM (live redistribution):

USE MONGODB WHEN:
    → Zero-downtime redistribution is MANDATORY
       (ride-sharing, payment systems, real-time apps)
    → You need to abort mid-operation safely
    → You need zone-aware redistribution (data locality)
    → You need to add/remove shards without downtime
    → Your data distribution changes over time
       (new cities, new regions, shifting traffic patterns)
    → You need granular control over balancing behavior
       (windows, thresholds, chunk sizes)

📍 WHEN TO STICK WITH REDSHIFT INSTEAD

FOR THIS SPECIFIC PROBLEM (changing data distribution):

USE REDSHIFT WHEN:
    → Simplicity matters more than zero-downtime
       (ALTER TABLE ALTER DISTKEY = one command)
    → Brief performance degradation is acceptable
    → You don't need zone-based data locality
    → Your distribution strategy changes rarely
       (quarterly optimization, not continuous)
    → You prefer Redshift Serverless auto-distribution
       (no DISTKEY decisions at all)
    → Your workload is batch analytics
       (brief downtime during nightly ETL window is fine)

MongoDB Resharding Internals — Deep Dive

RESHARDING COORDINATOR (runs on config server primary):

┌─────────────────────────────────────────────────────────────────┐
│ CONFIG SERVER (Primary)                                          │
│                                                                  │
│ ReshardingCoordinator                                            │
│ ├── State Machine: init → cloning → applying → blocking → done  │
│ ├── Manages temporary resharding collection                      │
│ ├── Coordinates donors (source shards) and recipients (target)   │
│ └── Controls commit protocol                                     │
└─────────────────────────────────────────────────────────────────┘

CLONE PHASE INTERNALS:

    Source Shard (donor):
    ┌──────────────────────────────────────────┐
    │  Original Collection                      │
    │  ┌─────────────────────────────────────┐ │
    │  │ Chunk: { city: "NYC" ... }           │ │
    │  │ 1.06 billion documents               │ │
    │  │                                      │ │
    │  │ ReshardingDonor:                     │ │
    │  │ → Opens cursor on collection         │ │
    │  │ → Reads documents in batches         │ │
    │  │ → Applies NEW shard key hash         │ │
    │  │ → Routes each doc to correct         │ │
    │  │   RECIPIENT shard based on new key   │ │
    │  └─────────────────────────────────────┘ │
    └──────────────────┬───────────────────────┘
                       │ documents flow over network
                       ▼
    Recipient Shard:
    ┌──────────────────────────────────────────┐
    │  Temporary Resharded Collection           │
    │  ┌─────────────────────────────────────┐ │
    │  │ ReshardingRecipient:                 │ │
    │  │ → Receives batches from donors       │ │
    │  │ → Inserts into temporary collection  │ │
    │  │ → Builds indexes concurrently        │ │
    │  │ → Tracks oplog position of donor     │ │
    │  └─────────────────────────────────────┘ │
    └──────────────────────────────────────────┘

CATCH-UP PHASE INTERNALS:

    ┌──────────────────────────────────────────┐
    │  Donor Shard                              │
    │  ┌─────────────────────────────────────┐ │
    │  │ Oplog (replication log):             │ │
    │  │ → Records every write since clone    │ │
    │  │   started                            │ │
    │  │ → Insert ride #2800000001 at ts=T1   │ │
    │  │ → Update ride #1500000000 at ts=T2   │ │
    │  │ → Delete ride #800000000 at ts=T3    │ │
    │  └───────────────┬─────────────────────┘ │
    └──────────────────┼───────────────────────┘
                       │ oplog entries streamed
                       ▼
    ┌──────────────────────────────────────────┐
    │  Recipient Shard                          │
    │  ┌─────────────────────────────────────┐ │
    │  │ Apply oplog entries:                 │ │
    │  │ → Insert: add to temp collection     │ │
    │  │ → Update: apply to temp collection   │ │
    │  │ → Delete: remove from temp collection│ │
    │  │ → Hash each operation's new shard key│ │
    │  │   to determine if THIS recipient     │ │
    │  │   owns the document                  │ │
    │  │                                      │ │
    │  │ Gap shrinks as writes are replayed   │ │
    │  │ until gap ≈ 0                        │ │
    │  └─────────────────────────────────────┘ │
    └──────────────────────────────────────────┘

COMMIT PHASE INTERNALS (the critical section):

    Step 1: Coordinator enters "blocking-writes" state
        → All mongos routers receive notification
        → Writes to collection BLOCKED (reads continue)
        → Duration: typically milliseconds to low seconds
    
    Step 2: Final oplog entries replayed
        → Any writes that snuck in before block → replayed
        → Temporary collection now EXACTLY matches original
    
    Step 3: Catalog swap
        → Config server metadata updated:
          collection "rideshare.rides" now points to new UUID
          chunk ranges updated with new shard key boundaries
          zone tags applied to new key ranges
        → This is an ATOMIC metadata operation
    
    Step 4: mongos routers refresh routing table
        → Next query uses NEW chunk map
        → Queries route to correct shards based on new key
        → Write block released
    
    Step 5: Cleanup (async, background)
        → Old collection data (orphaned chunks) deleted
        → Temporary namespace removed
        → Index for old shard key optionally dropped
        → Disk space reclaimed gradually

TEMPORARY STORAGE DURING RESHARDING:

    Before:
    ┌──────────────────────────────────────────┐
    │ Shard A: 1.6TB (original)                 │
    │ Shard B: 0.7TB (original)                 │
    │ Shard C: 0.6TB (original)                 │
    │ Shard D: 0.5TB (original)                 │
    │ Shard E: 0.8TB (original)                 │
    │ Total: 4.2TB                              │
    └──────────────────────────────────────────┘

    During (peak — clone phase ending):
    ┌──────────────────────────────────────────┐
    │ Shard A: 1.6TB (original) + 0.84TB (new) │ ← needs 2.44TB free
    │ Shard B: 0.7TB (original) + 0.84TB (new) │ ← needs 1.54TB free
    │ Shard C: 0.6TB (original) + 0.84TB (new) │ ← needs 1.44TB free

WiredTiger Behavior During Resharding

ON DONOR SHARDS (reading original data):
    
    ┌─────────────────────────────────────────────┐
    │ WiredTiger Cache                             │
    │                                              │
    │ Normal workload: 60% cache utilization       │
    │ During resharding: 85-95% cache utilization  │
    │                                              │
    │ WHY: Resharding cursor scans ALL documents   │
    │      → Pages loaded into cache sequentially  │
    │      → Evicts application's working set      │
    │      → Application cache miss rate increases  │
    │      → Application queries slow down          │
    │                                              │
    │ MITIGATION:                                  │
    │ → Resharding uses low-priority I/O           │
    │ → But still competes for cache space         │
    │ → Set resharding to use internal cursor      │
    │   with yield points (automatic)              │
    └─────────────────────────────────────────────┘

ON RECIPIENT SHARDS (writing new data):

    ┌─────────────────────────────────────────────┐
    │ WiredTiger Write Path                        │
    │                                              │
    │ 1. Incoming documents → WiredTiger cache     │
    │ 2. Cache fills → checkpoint to disk          │
    │ 3. Indexes built concurrently:               │
    │    → _id index (always)                      │
    │    → New shard key index                     │
    │    → All other existing indexes              │
    │ 4. Journal writes increase 3-5x             │

    │                                              │
    │ IMPACT:                                      │
    │ → Write latency on recipient shards +20-40%  │

    │ → Journal disk can become bottleneck          │

def diagnose_shard_distribution():
    """
    Identify skew in current shard key distribution
    Redshift equivalent:
        SELECT slice, COUNT(*) FROM stv_blocklist
        GROUP BY slice ORDER BY 2 DESC;
    """
    
    print("=" * 70)
    print("SHARD DISTRIBUTION DIAGNOSIS")
    print("=" * 70)
    
    # ─── 1. Get collection shard key info ─────────────
    coll_info = config_db.collections.find_one(
        {"_id": "rideshare.rides"}
    )
    print(f"\nCurrent shard key: {coll_info['key']}")
    # Output: Current shard key: {'city': 1}
    
    # ─── 2. Count chunks per shard ────────────────────
    pipeline = [
        {"$match": {"uuid": coll_info["uuid"]}},
        {"$group": {
            "_id": "$shard",
            "chunkCount": {"$sum": 1},
            "jumboCount": {
                "$sum": {"$cond": [
                    {"$eq": [{"$ifNull": ["$jumbo", False]}, True]},
                    1, 0
                ]}
            }
        }},
        {"$sort": {"chunkCount": -1}}
    ]
    
    print("\nChunk distribution per shard:")
    print(f"{'Shard':<15} {'Chunks':<10} {'Jumbo':<10} {'Estimated %'}")
    print("-" * 50)
    
    total_chunks = 0
    shard_data = []
    for doc in config_db.chunks.aggregate(pipeline):
        total_chunks += doc["chunkCount"]
        shard_data.append(doc)
    
    for doc in shard_data:
        pct = round(100 * doc["chunkCount"] / total_chunks, 1)
        print(f"{doc['_id']:<15} {doc['chunkCount']:<10} "
              f"{doc['jumboCount']:<10} {pct}%")
    
    # ─── 3. Identify jumbo chunks ─────────────────────
    jumbo_chunks = list(config_db.chunks.find(
        {"uuid": coll_info["uuid"], "jumbo": True},
        {"min": 1, "max": 1, "shard": 1}
    ))
    
    if jumbo_chunks:
        print(f"\n⚠️  JUMBO CHUNKS FOUND: {len(jumbo_chunks)}")
        for chunk in jumbo_chunks:
            print(f"  Shard: {chunk['shard']}")
            print(f"  Range: {chunk['min']} → {chunk['max']}")
    else:
        print("\n✅ No jumbo chunks")
    
    # ─── 4. Check shard key cardinality ───────────────
    cardinality = db.rides.aggregate([
        {"$group": {"_id": "$city"}},
        {"$count": "uniqueValues"}
    ])
    result = list(cardinality)
    if result:
        print(f"\nShard key cardinality: {result[0]['uniqueValues']} unique values")
    
    # ─── 5. Top cities by document count ──────────────
    print("\nTop 5 cities by ride count:")
    top_cities = db.rides.aggregate([
        {"$group": {"_id": "$city", "count": {"$sum": 1}}},
        {"$sort": {"count": -1}},
        {"$limit": 5}
    ])
    total_docs = db.rides.estimated_document_count()
    for city in top_cities:
        pct = round(100 * city["count"] / total_docs, 1)
        print(f"  {city['_id']:<20} {city['count']:>15,} ({pct}%)")
    
    return coll_info, total_chunks, jumbo_chunks

def monitor_resharding(interval_seconds=30, max_checks=None):
    """
    Real-time monitoring of resharding progress
    
    Redshift equivalent:
        SELECT * FROM SYS_QUERY_HISTORY
        WHERE query_text LIKE '%ALTER TABLE%ALTER DISTKEY%';
        -- Plus: CloudWatch metrics for cluster performance

    print("=" * 70)
    
    check_count = 0
    
    while True:
        check_count += 1
        if max_checks and check_count > max_checks:
            print("Max checks reached — exiting monitor")
            break
        
        try:
            op = admin_db.command({"currentReshardingOperation": 1})
        except Exception as e:
            print(f"[{datetime.now().strftime('%H:%M:%S')}] "
                  f"No resharding operation (complete or not started)")
            break
        
        state = op.get("state", "unknown")
        
        if state in ("done", "unknown") or not op.get("state"):
            print(f"\n[{datetime.now().strftime('%H:%M:%S')}] "
                  f"Resharding COMPLETE ✅")

        total_docs = op.get("approxDocumentsToCopy", 0)

        copied_bytes = op.get("bytesCopied", 0)

        eta_seconds = op.get("remainingOperationTimeEstimatedSecs", -1)

            eta_str = f"{round(eta_seconds / 3600, 1)}h"
        else:
            eta_str = "calculating..."

        elapsed = op.get("totalCopyTimeElapsedSecs", 0)

        # Format output
        timestamp = datetime.now().strftime('%H:%M:%S')
        
        if state == "cloning":
            print(
                f"[{timestamp}] "
                f"CLONING | "
                f"Docs: {doc_pct}% ({copied_docs:,}/{total_docs:,}) | "
                f"Bytes: {byte_pct}% ({round(copied_bytes/1e9, 1)}GB/"
                f"{round(total_bytes/1e9, 1)}GB) | "
                f"Oplog gap: {oplog_gap:,} | "
                f"Elapsed: {elapsed_str} | "
                f"ETA: {eta_str}"
            )
        
        elif state == "applying":
            print(
                f"[{timestamp}] "
                f"CATCHING UP | "
                f"Oplog: fetched={oplog_fetched:,} "
                f"applied={oplog_applied:,} "
                f"gap={oplog_gap:,} | "
                f"Inserts: {op.get('insertsApplied', 0):,} | "
                f"Updates: {op.get('updatesApplied', 0):,} | "
                f"Deletes: {op.get('deletesApplied', 0):,} | "
                f"ETA: {eta_str}"
            )
        
        elif state == "blocking-writes":
            print(
                f"[{timestamp}] "
                f"⚡ BLOCKING WRITES — COMMIT IN PROGRESS ⚡ | "
                f"This should last only seconds..."
            )
        
        else:
            print(
                f"[{timestamp}] "
                f"State: {state} | "
                f"Details: {op}"
            )
        
        # Check for warning conditions
        if oplog_gap > 10_000_000:
            print(f"    ⚠️  OPLOG GAP LARGE ({oplog_gap:,}) — "
                  f"writes may be outpacing catch-up")
        
        if state == "cloning" and doc_pct > 0:
            rate = copied_docs / max(elapsed, 1)
            remaining_docs = total_docs - copied_docs
            est_clone_hours = remaining_docs / rate / 3600
            if est_clone_hours > 48:
                print(f"    ⚠️  ESTIMATED CLONE TIME: {est_clone_hours:.0f}h — "
                      f"consider aborting and using smaller batches")
        
        time.sleep(interval_seconds)

def verify_resharding():
    """
    Complete post-resharding health check
    
    Redshift equivalent:
        SELECT * FROM SVV_TABLE_INFO WHERE table = 'rides';
        -- Check new distribution
        SELECT slice, COUNT(*) FROM stv_blocklist ...;
    """
    
    print("\n" + "=" * 70)
    print("POST-RESHARDING VERIFICATION")
    print("=" * 70)
    
    # ─── 1. Verify new shard key ──────────────────────
    coll_info = config_db.collections.find_one(
        {"_id": "rideshare.rides"}
    )
    print(f"\n1. Shard key: {coll_info['key']}")
    expected = {"city": 1, "ride_date": 1, "_id": 1}
    if coll_info["key"] == expected:
        print(f"   ✅ Matches expected key")
    else:
        print(f"   ❌ MISMATCH — expected {expected}")
    
    # ─── 2. Verify chunk distribution ─────────────────
    print(f"\n2. Chunk distribution:")
    pipeline = [
        {"$match": {"uuid": coll_info["uuid"]}},
        {"$group": {
            "_id": "$shard",
            "chunks": {"$sum": 1},
            "jumbo": {"$sum": {
                "$cond": [
                    {"$eq": [{"$ifNull": ["$jumbo", False]}, True]},
                    1, 0
                ]
            }}
        }},
        {"$sort": {"chunks": -1}}
    ]
    
    results = list(config_db.chunks.aggregate(pipeline))
    total = sum(r["chunks"] for r in results)
    ideal = total / len(results) if results else 0
    
    print(f"   {'Shard':<15} {'Chunks':<10} {'%':<10} {'Jumbo':<10} {'Status'}")
    print(f"   {'-'*55}")
    
    max_deviation = 0
    for r in results:
        pct = round(100 * r["chunks"] / total, 1)
        deviation = abs(r["chunks"] - ideal) / ideal * 100
        max_deviation = max(max_deviation, deviation)
        status = "✅" if deviation < 20 else "⚠️" if deviation < 40 else "❌"
        print(f"   {r['_id']:<15} {r['chunks']:<10} {pct}%"
              f"{'':>5} {r['jumbo']:<10} {status}")
    
    if max_deviation < 20:
        print(f"\n   ✅ Distribution is BALANCED (max deviation: {max_deviation:.0f}%)")
    else:
        print(f"\n   ⚠️  Distribution has some skew (max deviation: {max_deviation:.0f}%)")
        print(f"       Balancer will continue to improve this")
    
    # ─── 3. Verify no jumbo chunks ────────────────────
    jumbo_count = config_db.chunks.count_documents({
        "uuid": coll_info["uuid"],
        "jumbo": True
    })
    print(f"\n3. Jumbo chunks: {jumbo_count} ", end="")
    print("✅" if jumbo_count == 0 else "❌")
    
    # ─── 4. Verify document count ─────────────────────
    count = db.rides.estimated_document_count()
    print(f"\n4. Document count: {count:,}")
    print(f"   (Compare with pre-resharding count for data integrity)")
    
    # ─── 5. Verify zone assignments ───────────────────
    print(f"\n5. Zone assignments:")
    zones = list(config_db.tags.find({"ns": "rideshare.rides"}))
    for zone in zones:
        print(f"   Zone: {zone['tag']}")
        print(f"     Min: {zone['min']}")
        print(f"     Max: {zone['max']}")
    
    if not zones:
        print(f"   ⚠️  No zones configured")
    
    # ─── 6. Test query performance ────────────────────
    print(f"\n6. Query performance test:")
    
    test_queries = [
        {
            "name": "NYC January 2024",
            "filter": {
                "city": "New York",
                "ride_date": {
                    "$gte": datetime(2024, 1, 1),
                    "$lt": datetime(2024, 2, 1)
                }
            }
        },
        {
            "name": "London last week",
            "filter": {
                "city": "London",
                "ride_date": {
                    "$gte": datetime.now() - timedelta(days=7),
                    "$lt": datetime.now()
                }
            }
        },
        {
            "name": "Small city (Boise) all time",
            "filter": {
                "city": "Boise"
            }
        }
    ]
    
    for test in test_queries:
        # Run explain to check index usage and shard targeting
        explain = db.command(
            "explain",
            {
                "find": "rides",
                "filter": test["filter"],
                "limit": 10
            },
            verbosity="executionStats"
        )
        
        # Extract key metrics
        shards_targeted = len(
            explain.get("queryPlanner", {})
                   .get("winningPlan", {})
                   .get("shards", [])
        )
        
        # Time a real query
        start = time.time()
        result = list(db.rides.find(test["filter"]).limit(10))
        elapsed_ms = round((time.time() - start) * 1000, 1)
        
        print(f"   {test['name']}:")
        print(f"     Latency: {elapsed_ms}ms")
        print(f"     Shards targeted: {shards_targeted}")
        print(f"     Results: {len(result)} docs")
    
    # ─── 7. Check for orphaned documents ──────────────
    print(f"\n7. Orphaned document check:")
    print(f"   Run on each shard: db.rides.countDocuments()")
    print(f"   Sum should match total: {count:,}")
    print(f"   Orphans are cleaned up asynchronously by range deleter")
    
    print("\n" + "=" * 70)
    print("VERIFICATION COMPLETE")
    print("=" * 70)

                  f"{status.get('dbsToMove', [])}")

    return status


def watch_balancer(duration_minutes=10, interval_seconds=30):
    """
    Watch balancer chunk migration activity
    
    No Redshift equivalent — Redshift doesn't have background rebalancing
    """
    
    print(f"\n{'=' * 70}")
    print(f"BALANCER ACTIVITY MONITOR ({duration_minutes} minutes)")
    print(f"{'=' * 70}")
    
    # Get balancer state
    balancer_state = admin_db.command({"balancerStatus": 1})
    print(f"Balancer mode: {balancer_state['mode']}")
    print(f"Currently running: {balancer_state.get('inBalancerRound', False)}")
    
    start_time = datetime.now()
    end_time = start_time + timedelta(minutes=duration_minutes)
    last_seen_time = start_time - timedelta(hours=1)
    
    while datetime.now() < end_time:
        # Query recent migrations
        recent = list(config_db.changelog.find(
            {
                "what": "moveChunk.commit",
                "time": {"$gt": last_seen_time}
            }
        ).sort("time", -1).limit(5))
        
        if recent:
            for entry in recent:
                details = entry.get("details", {})
                print(
                    f"[{entry['time'].strftime('%H:%M:%S')}] "
                    f"CHUNK MOVE: {details.get('from', '?')} → "
                    f"{details.get('to', '?')} | "
                    f"Docs: {details.get('cloned', '?'):,} | "
                    f"Time: {details.get('executionTimeMillis', '?')}ms"
                )
            last_seen_time = recent[0]["time"]
        else:
            print(f"[{datetime.now().strftime('%H:%M:%S')}] "
                  f"No recent migrations")
        
        time.sleep(interval_seconds)

    -- Step 3: Wait (hours)

    --

    check_shard_health()
    
    # PHASE B: Prepare

        return

    print("  B) reshard — Change key completely (hours, zero downtime)")

    # In practice: try refine FIRST, only reshard if refine insufficient

        refine_shard_key()

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 40:

PITFALL 1: "I'll just change the shard key like ALTER DISTKEY"
    ❌ Expecting resharding to be a single simple command
    ✅ Resharding requires: preparation, monitoring, verification
    WHY: Redshift ALTER DISTKEY is a single managed operation
         MongoDB resharding is a distributed multi-phase process
         Both achieve the same goal — different operational complexity

PITFALL 2: "Resharding will fix my performance immediately"
    ❌ Running resharding and expecting instant improvement
    ✅ Resharding takes hours + balancer takes more hours after
    WHY: Clone phase copies ALL data (hours)
         Balancer fine-tunes after resharding (more hours)
         Full benefit realized 24-48 hours after resharding completes
         Redshift ALTER DISTKEY: improvement visible immediately after completion

PITFALL 3: "I don't need to monitor — it's automatic"
    ❌ Starting resharding and walking away
    ✅ Actively monitor throughout — especially oplog gap
    WHY: Oplog gap growing → resharding may never complete
         Disk space exhaustion → data corruption risk
         Application latency spikes → may need abort
         Redshift: you CAN walk away (it's fully managed)
         MongoDB: you SHOULD watch (distributed operation)

PITFALL 4: "I'll reshard during peak traffic — it's zero downtime"
    ❌ Resharding during busiest hours
    ✅ Schedule during lowest traffic period
    WHY: "Zero downtime" ≠ "zero performance impact"
         Clone phase: significant I/O and network load
         Application latency: +20-50% during resharding
         Redshift ALTER DISTKEY: same advice applies

PITFALL 5: "I forgot to check disk space"
    ❌ Starting resharding without 2x free space per shard
    ✅ Verify: free space > collection size on EVERY shard
    WHY: Temporary collection created on all shards
         If any shard runs out of disk: operation fails
         Partial failure during resharding = messy recovery
         Redshift: managed storage (RA3) handles this automatically

PITFALL 6: "I'll change the shard key AND resize shards simultaneously"
    ❌ Running resharding + Atlas tier change at same time
    ✅ One major operation at a time
    WHY: Compounding operations = unpredictable resource contention
         Resharding + scaling = double the I/O, network, CPU load
         Sequence: scale shards → wait → reshard → wait → verify

PITFALL 7: "Refine is always enough"
    ❌ Using refine when fundamental key change is needed
    ✅ Understand when refine helps vs when full reshard is needed
    REFINE HELPS WHEN:
        → Jumbo chunks caused by low cardinality
        → Adding suffix fields creates enough split points
        → Existing base key is still query-relevant
    REFINE DOESN'T HELP WHEN:
        → Base key field itself causes skew (e.g., { status: 1 })
        → Need completely different distribution strategy
        → Need to change from range to hashed or vice versa

PITFALL 8: "I'll skip the backup"
    ❌ No backup before resharding 4.2TB of production data
    ✅ Always snapshot before the highest-risk operation in MongoDB
    WHY: Resharding failure modes are rare but catastrophic
         Config server inconsistency = cluster-wide impact
         Atlas: automatic snapshots make this easy
         Self-managed: mongodump or LVM snapshots required

PITFALL 9: "Oplog size doesn't matter"
    ❌ Default oplog size with 12+ hour resharding operation
    ✅ Ensure oplog window > 2x estimated resharding duration

Capacity Planning Framework

MONGODB CAPACITY PLANNING FOR SCALING DECISIONS:

1. STORAGE CAPACITY PER SHARD
   ┌──────────────────────────────────────────────────────────┐
   │ Formula:                                                  │
   │   Required storage per shard =                            │
   │     (Total data size / Number of shards)                  │
   │     × 1.5 (indexes overhead)                              │
   │     × 1.3 (WiredTiger overhead: journal + oplog)          │
   │     × 2.0 (headroom for resharding/compaction)            │
   │                                                           │
   │ Example (our ride-sharing):                               │
   │   4.2TB / 5 shards = 840GB per shard                     │
   │   × 1.5 = 1,260GB (with indexes)                         │
   │   × 1.3 = 1,638GB (with WiredTiger overhead)             │
   │   × 2.0 = 3,276GB (with resharding headroom)             │
   │   → Each shard needs ~3.3TB storage                       │
   └──────────────────────────────────────────────────────────┘

2. RAM CAPACITY PER SHARD (Working Set)
   ┌──────────────────────────────────────────────────────────┐
   │ Rule of thumb:                                            │
   │   WiredTiger cache = 50% of (RAM - 1GB)                  │
   │   Working set MUST fit in cache for low latency           │
   │                                                           │
   │ Working set = frequently accessed data + active indexes   │
   │                                                           │
   │ If working set > cache:                                   │
   │   → Cache evictions → disk reads → high latency           │
   │   → Project 33 covered this in detail                     │
   │                                                           │
   │ Example:                                                  │
   │   Last 30 days of rides per shard = 50GB                  │
   │   Active indexes per shard = 30GB                         │
   │   Working set = 80GB                                      │
   │   Need: 80GB / 0.5 + 1GB = 161GB RAM per shard           │
   │   → Choose Atlas M60 (128GB) or M80 (256GB)              │
   └──────────────────────────────────────────────────────────┘

3. THROUGHPUT CAPACITY PER SHARD
   ┌──────────────────────────────────────────────────────────┐
   │ Single shard can handle:                                  │
   │   Reads:  ~50,000-100,000 ops/sec (indexed queries)      │
   │   Writes: ~10,000-50,000 ops/sec (depends on doc size)   │
   │                                                           │
   │ When to add shards:                                       │
   │   → Any shard consistently > 70% CPU                     │
   │   → Write latency p99 > 50ms                             │
   │   → Read latency p99 > 100ms                             │
   │   → Cache eviction rate > 100 pages/sec                  │
   │                                                           │
   │ Redshift comparison:                                      │
   │   → Redshift: ~5-50 concurrent queries (WLM limited)     │
   │   → MongoDB: thousands of concurrent operations           │
   │   → Different workloads, different scaling triggers        │
   └──────────────────────────────────────────────────────────┘

4. WHEN TO ADD SHARDS vs UPGRADE SHARD TIER
   ┌──────────────────────────────────────────────────────────┐
   │ ADD SHARDS (horizontal) when:                             │
   │   → Write throughput is the bottleneck                    │
   │   → Data volume growing beyond single shard storage       │
   │   → Need geographic distribution (new zone)               │
   │   → Want to distribute CPU load for reads                 │
   │                                                           │
   │ UPGRADE TIER (vertical) when:                             │
   │   → Working set doesn't fit in RAM (need more memory)     │
   │   → Single-document operations are slow (CPU-bound)       │
   │   → Disk I/O saturated (need faster IOPS)                 │
   │   → Cheaper than adding another full shard                │
   │                                                           │
   │ Redshift equivalent:                                      │
   │   → Add nodes = add shards                                │
   │   → Change node type = upgrade shard tier                 │
   │   → Same trade-off: horizontal vs vertical                │
   └──────────────────────────────────────────────────────────┘

📐 DATA MODELING COMPARISON

HOW SHARD KEY RELATES TO DATA MODEL:

REDSHIFT THINKING:
    → Data model and distribution key are INDEPENDENT decisions
    → You design normalized tables FIRST
    → Then choose DISTKEY for JOIN optimization
    → DISTKEY change doesn't affect table structure
    → Sort key change doesn't affect table structure
    → Data model is STABLE, distribution is TUNABLE

MONGODB THINKING:
    → Shard key is DEEPLY coupled to document design
    → Shard key fields MUST exist in every document
    → Shard key determines which shard owns each document
    → Changing shard key may require rethinking document structure
    → Document design + shard key + index strategy = ONE holistic decision

EXAMPLE — How resharding affects data model:

    BEFORE RESHARDING:
    Shard key: { city: 1 }
    Document design:
    {
        _id: ObjectId("..."),
        city: "New York",              ← shard key field
        rider_id: 12345,
        driver_id: 67890,
        ride_date: ISODate("..."),
        pickup: { lat: 40.7, lng: -74.0 },
        dropoff: { lat: 40.8, lng: -73.9 },
        fare: 25.50,
        status: "completed"
    }

    AFTER RESHARDING:
    Shard key: { city: 1, ride_date: 1, _id: 1 }
    Document design: SAME! (no change needed)
    {
        _id: ObjectId("..."),
        city: "New York",              ← shard key field 1
        rider_id: 12345,
        driver_id: 67890,
        ride_date: ISODate("..."),     ← shard key field 2
        pickup: { lat: 40.7, lng: -74.0 },
        dropoff: { lat: 40.8, lng: -73.9 },
        fare: 25.50,
        status: "completed"
        // _id already exists              ← shard key field 3
    }

    → Document structure UNCHANGED
    → But routing behavior COMPLETELY different
    → Queries without city in filter: scatter-gather (ALL shards)
    → Queries with city: targeted to subset of shards
    → Queries with city + ride_date: targeted to FEWER shards
    → Queries with city + ride_date + _id: targeted to SINGLE shard

    IN REDSHIFT:
    → Table structure also unchanged during ALTER DISTKEY
    → But: no query routing concept (leader node handles all)
    → DISTKEY affects JOIN co-location, not query routing
    → Fundamentally different purpose of distribution
