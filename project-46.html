Where Redshift-Only Architecture BREAKS for IoT

FAILURE POINTS:

1. INGESTION THROUGHPUT
   → 100,000 writes/second sustained
   → Redshift COPY is batch-oriented (minimum 10-second micro-batch)
   → Cannot INSERT 100K individual rows/second (single-digit TPS for INSERT)
   → Must buffer in Kinesis/S3 first → minimum 60-second latency
   → If Kinesis buffer fails → DATA LOSS during the gap
   
   COMPARISON:
   → Redshift: 60-second minimum ingestion latency (via COPY)
   → MongoDB: sub-10ms per write, 100K+/sec sustained throughput

2. REAL-TIME ALERTING (5-second requirement)
   → Data arrives in Redshift after 60+ seconds (Firehose buffer)
   → Query to detect threshold breach: additional 2-10 seconds
   → Total: 70+ seconds from sensor reading to alert
   → Requirement: 5 seconds
   → IMPOSSIBLE with Redshift alone

3. LIVE DASHBOARD (current readings)
   → "Show me current temperature for all sensors in Facility 42"
   → Redshift: data is 60+ seconds stale (batch ingestion)
   → Even with Streaming Ingestion (preview): 10+ second lag
   → Dashboard refreshes every 5 seconds → stale data displayed
   → Users see "5 minutes ago" data → unacceptable for safety monitoring

4. COST AT SCALE
   → 8.64 billion rows/day = 259 billion rows/month
   → Even compressed: ~50TB/month of new data
   → Redshift storage: $0.024/GB = $1,200/month for storage alone
   → After 12 months: 600TB → $14,400/month just for storage
   → Most of this data is rarely queried after 30 days
   → Redshift charges same price for hot and cold data

5. SMALL ROW OVERHEAD
   → Each reading is ~100 bytes of actual data
   → Redshift row overhead: ~20-30 bytes metadata per row
   → 20-30% storage wasted on metadata for tiny rows
   → MongoDB Time Series: automatic bucketing groups readings
     → metadata overhead amortized across hundreds of readings per bucket

6. DATA LIFECYCLE MANAGEMENT
   → Hot data (7 days): query frequently for dashboards/alerts
   → Warm data (90 days): query occasionally for trend analysis
   → Cold data (5 years): query rarely for compliance/audit
   → Redshift: ALL data at same cost, same performance tier
   → No automatic tiering → must manually UNLOAD to S3 + Spectrum
   → Complex lifecycle management with external orchestration

CONCLUSION:
    Redshift CANNOT be the ingestion layer for IoT
    Redshift IS excellent for historical trend analysis
    You need MongoDB for ingestion + real-time + alerting
    You need Redshift for long-term analytics
    You need S3 for cost-effective cold storage
    The architecture must handle DIFFERENT time horizons differently

📦 COMPONENT 2: TIME SERIES INGESTION (The Core Challenge)

Why MongoDB Time Series (References: Projects 16, 30)

THE CRITICAL DIFFERENCE FROM E-COMMERCE (Project 45):

E-COMMERCE:
    → Write: 50K orders/hour = ~14 writes/second
    → Read: 100K product page views/second
    → Ratio: 1:7000 (write:read)
    → Optimize for: READ LATENCY

IOT:
    → Write: 100,000 sensor readings/second
    → Read: ~1,000 dashboard queries/second
    → Ratio: 100:1 (write:read)
    → Optimize for: WRITE THROUGHPUT

THIS RATIO INVERSION changes EVERYTHING about the architecture:
    → Document design optimizes for write batching
    → Index strategy minimizes write amplification
    → Shard key optimizes for write distribution
    → Read queries accept slightly higher latency for much higher write throughput

# ─── BATCH WRITER (accumulate then bulk insert) ──────
class BatchWriter:
    """
    Accumulates sensor readings and bulk-inserts in batches
    
    WHY BATCHING:
    → Individual insertOne() at 100K/sec = 100K network round trips/sec
    → Batch insertMany() of 1000 = 100 network round trips/sec
    → 1000x reduction in network overhead
    → MongoDB Time Series internally batches further into buckets
    
    COMPARISON TO REDSHIFT:
    → Redshift COPY: accumulate in S3 → COPY every 60 seconds
    → MongoDB insertMany: accumulate in memory → insert every 1 second
    → MongoDB latency: 1 second vs Redshift: 60+ seconds
    """
    
    def __init__(self, collection, batch_size=5000, flush_interval_sec=1.0):
        self.collection = collection
        self.batch_size = batch_size
        self.flush_interval = flush_interval_sec
        self.buffer = []
        self.lock = threading.Lock()
        self.stats = {
            "total_inserted": 0,
            "total_batches": 0,
            "total_errors": 0,
            "start_time": time.time()
        }
        
        # Background flush thread
        self.flush_thread = threading.Thread(target=self._periodic_flush, daemon=True)
        self.flush_thread.start()
    
    def add_reading(self, reading):
        with self.lock:
            self.buffer.append(reading)
            if len(self.buffer) >= self.batch_size:
                self._flush()
    
    def _periodic_flush(self):
        while True:
            time.sleep(self.flush_interval)
            with self.lock:
                if self.buffer:
                    self._flush()
    
    def _flush(self):
        if not self.buffer:
            return
        
        batch = self.buffer.copy()
        self.buffer = []
        
        try:
            # ── BULK INSERT ──
            # ordered=False: continue on error (like Redshift MAXERROR)
            # Don't let one bad reading stop the batch
            result = self.collection.insert_many(batch, ordered=False)
            
            self.stats["total_inserted"] += len(result.inserted_ids)
            self.stats["total_batches"] += 1
            
        except Exception as e:
            # BulkWriteError: some succeeded, some failed
            self.stats["total_errors"] += 1
            # Log error, don't crash — IoT ingestion must be resilient
            print(f"Batch error: {e}")
    
    def get_throughput(self):
        elapsed = time.time() - self.stats["start_time"]
        if elapsed > 0:
            return self.stats["total_inserted"] / elapsed
        return 0

class AlertEngine:
    """
    Watches MongoDB Time Series collection via Change Streams
    Fires alerts when sensor readings exceed thresholds
    
    WHY CHANGE STREAMS (not polling):
    → Push-based: MongoDB notifies us immediately on new data
    → No polling interval → no wasted queries
    → Resume token → exactly-once processing guarantee
    → Built on oplog → zero additional load on database
    
    WHY NOT REDSHIFT:
    → Data arrives in Redshift 60+ seconds after sensor reading
    → Alert requirement: <5 seconds
    → Redshift cannot participate in real-time alerting
    → Redshift receives alerts AFTER they're generated (for analytics)
    
    COMPARISON TO E-COMMERCE (Project 45):
    → E-commerce change streams: watch order status changes (~50K/hr)
    → IoT change streams: watch ALL sensor readings (~100K/sec)
    → MUST filter in pipeline to avoid overwhelming the alert engine
    """
    
    def __init__(self):
        self.threshold_cache = {}
        self.alert_counts = {"warning": 0, "critical": 0}
        self._load_thresholds()
    
    def _load_thresholds(self):
        """Cache all sensor thresholds in memory for fast lookup"""
        for sensor in db.sensors.find(
            {"status": "active"},
            {"_id": 1, "thresholds": 1, "type": 1, "location.facility_id": 1}
        ):
            self.threshold_cache[sensor["_id"]] = {
                "thresholds": sensor.get("thresholds", {}),
                "type": sensor["type"],
                "facility_id": sensor["location"]["facility_id"]
            }
        print(f"Loaded thresholds for {len(self.threshold_cache)} sensors")
    
    def _check_thresholds(self, reading):
        """
        Check if reading exceeds any threshold
        Returns alert document or None
        """
        sensor_id = reading["meta"]["sensor_id"]
        sensor_config = self.threshold_cache.get(sensor_id)
        
        if not sensor_config:
            return None
        
        sensor_type = sensor_config["type"]
        thresholds = sensor_config["thresholds"].get(sensor_type, {})
        
        # Get the measurement value
        value = reading.get(sensor_type)
        if value is None:
            return None
        
        # Check critical thresholds first
        if thresholds.get("critical_high") and value >= thresholds["critical_high"]:
            return self._create_alert(reading, sensor_config, value,
                                       thresholds["critical_high"], "critical", "high")
        
        if thresholds.get("critical_low") and value <= thresholds["critical_low"]:
            return self._create_alert(reading, sensor_config, value,
                                       thresholds["critical_low"], "critical", "low")
        
        # Check warning thresholds
        if thresholds.get("warning_high") and value >= thresholds["warning_high"]:
            return self._create_alert(reading, sensor_config, value,
                                       thresholds["warning_high"], "warning", "high")
        
        if thresholds.get("warning_low") and value <= thresholds["warning_low"]:
            return self._create_alert(reading, sensor_config, value,
                                       thresholds["warning_low"], "warning", "low")
        
        return None
    
    def _create_alert(self, reading, sensor_config, value, threshold, severity, direction):
        """Create alert document and store in alerts collection"""
        
        alert = {
            "sensor_id": reading["meta"]["sensor_id"],
            "facility_id": sensor_config["facility_id"],
            "sensor_type": sensor_config["type"],
            "severity": severity,
            "direction": direction,
            "value": value,
            "threshold": threshold,
            "delta": round(abs(value - threshold), 2),
            "reading_ts": reading["ts"],
            "alert_ts": datetime.utcnow(),
            "latency_ms": int(
                (datetime.utcnow() - reading["ts"]).total_seconds() * 1000
            ),
            "status": "active",                        # active | acknowledged | resolved
            "acknowledged_by": None,
            "resolved_at": None,
            "meta": reading["meta"]
        }
        
        # Store alert
        db.alerts.insert_one(alert)
        
        # Update sensor computed stats
        db.sensors.update_one(
            {"_id": reading["meta"]["sensor_id"]},
            {"$inc": {"computed.alerts_7d": 1}}
        )
        
        self.alert_counts[severity] += 1
        
        return alert
    
    def watch_readings(self):
        """
        Main change stream watcher
        Monitors time series collection for new readings
        """
        
        # Pipeline: only watch inserts (not updates/deletes)
        pipeline = [
            {"$match": {"operationType": "insert"}}
        ]
        
        print("Alert engine started — watching readings collection...")
        print(f"Thresholds loaded for {len(self.threshold_cache)} sensors")
        
        # Resume token for exactly-once processing
        resume_token = None
        
        while True:
            try:
                with db.readings.watch(
                    pipeline,
                    resume_after=resume_token,
                    full_document="updateLookup"
                ) as stream:
                    for change in stream:
                        resume_token = stream.resume_token
                        
                        doc = change.get("fullDocument")
                        if not doc:
                            continue
                        
                        # Check thresholds
                        alert = self._check_thresholds(doc)
                        
                        if alert:
                            latency = alert["latency_ms"]
                            print(
                                f"🚨 [{alert['severity'].upper()}] "
                                f"Sensor {alert['sensor_id']} | "
                                f"{alert['sensor_type']}: {alert['value']} "
                                f"({'>' if alert['direction'] == 'high' else '<'} "
                                f"threshold {alert['threshold']}) | "
                                f"Latency: {latency}ms"
                            )
                            
                            # In production: send to notification service
                            # → Kafka topic: alerts
                            # → PagerDuty / SMS / Email
                            # → Facility control room display
                            
            except Exception as e:
                print(f"Change stream error: {e}")
                print("Reconnecting in 5 seconds...")
                time.sleep(5)
                # Resume from last token → no missed events

import boto3

def export_readings_to_s3(date_to_export):
    """
    Export one day of readings from MongoDB to S3 as Parquet
    Runs daily via scheduler (Airflow / EventBridge)
    
    WHY PARQUET:
    → Columnar format → Redshift Spectrum reads efficiently
    → Snappy compression → 10x smaller than JSON
    → Schema preserved → no parsing overhead
    → Compatible with: Spectrum, Athena, Spark, Databricks
    """
    
    start = datetime.combine(date_to_export, datetime.min.time())
    end = start + timedelta(days=1)
    
    print(f"Exporting readings for {date_to_export}")
    
    # Read from MongoDB Time Series in chunks
    batch_size = 100000
    total_exported = 0
    file_number = 0
    
    cursor = db.readings.find(
        {"ts": {"$gte": start, "$lt": end}},
        {"_id": 0}                                    # exclude _id for Parquet
    ).batch_size(batch_size)
    
    batch = []
    for doc in cursor:
        # Flatten the document for columnar storage
        flat = {
            "ts": doc["ts"],
            "sensor_id": doc["meta"]["sensor_id"],
            "facility_id": doc["meta"]["facility_id"],
            "sensor_type": doc["meta"]["type"],
            "region": doc["meta"]["region"],
            "temperature": doc.get("temperature"),
            "pressure": doc.get("pressure"),
            "vibration": doc.get("vibration"),
            "humidity": doc.get("humidity"),
            "quality": doc.get("quality", "unknown")
        }
        batch.append(flat)
        
        if len(batch) >= batch_size:
            _write_parquet_to_s3(batch, date_to_export, file_number)
            total_exported += len(batch)
            file_number += 1
            batch = []
            print(f"  Exported {total_exported:,} readings...")
    
    # Final batch
    if batch:
        _write_parquet_to_s3(batch, date_to_export, file_number)
        total_exported += len(batch)
    
    print(f"Export complete: {total_exported:,} readings → "
          f"{file_number + 1} Parquet files")
    return total_exported


def _write_parquet_to_s3(records, date, file_num):
    """Write batch of records as Parquet to S3"""
    
    # Convert to PyArrow table
    table = pa.Table.from_pylist(records)
    
    # Write to buffer
    buf = io.BytesIO()
    pq.write_table(table, buf, compression='snappy')
    buf.seek(0)
    
    # Upload to S3 with date partitioning
    s3_key = (
        f"readings/year={date.year}/month={date.month:02d}/"
        f"day={date.day:02d}/readings_{file_num:04d}.parquet"
    )
    
    s3.put_object(
        Bucket="sensorflow-data-lake",
        Key=s3_key,
        Body=buf.getvalue()
    )


def generate_hourly_aggregates(date_to_aggregate):
    """
    Generate hourly pre-aggregated summaries
    These go DIRECTLY into Redshift local tables (not Spectrum)
    
    WHY PRE-AGGREGATE:
    → Raw per-second data: 8.64 billion rows/day → too large for Redshift local
    → Hourly aggregates: 2.4 million rows/day (100K sensors × 24 hours) → perfect
    → Most analytical queries need hourly/daily granularity anyway
    → Raw per-second available via Spectrum when needed
    
    COMPARISON TO E-COMMERCE (Project 45):
    → E-commerce: every order row goes to Redshift (important individually)
    → IoT: individual readings are noise → aggregates are signal
    → Different data semantics → different ETL strategy
    """
    
    start = datetime.combine(date_to_aggregate, datetime.min.time())
    end = start + timedelta(days=1)
    
    pipeline = [

def generate_daily_aggregates(date_to_aggregate):
    """

def daily_lifecycle_job():
    """
    Runs daily at 02:00 UTC via Airflow / EventBridge
    
    Steps:
    1. Export yesterday's raw readings to S3 (Parquet)
    2. Generate hourly aggregates → S3
    3. Generate daily aggregates → S3
    4. COPY aggregates into Redshift
    5. MongoDB TTL auto-cleans readings older than 7 days
    6. S3 Lifecycle auto-moves data older than 90 days to Glacier
    """
    
    yesterday = datetime.utcnow().date() - timedelta(days=1)
    
    print(f"═══ DAILY LIFECYCLE JOB: {yesterday} ═══")
    
    # Step 1: Raw export to S3
    print("\n[Step 1] Exporting raw readings to S3...")
    raw_count = export_readings_to_s3(yesterday)
    
    # Step 2: Hourly aggregates
    print("\n[Step 2] Generating hourly aggregates...")
    hourly = generate_hourly_aggregates(yesterday)
    
    # Step 3: Daily aggregates
    print("\n[Step 3] Generating daily aggregates...")
    daily = generate_daily_aggregates(yesterday)
    
    # Step 4: Load into Redshift
    print("\n[Step 4] Loading aggregates into Redshift...")
    load_aggregates_to_redshift(yesterday)

  AND s.sensor_type = 'temperature'

GROUP BY 1
ORDER BY 1;

-- → Columnar: reads ONLY avg_value, max_value columns
-- → LAG window function across months → trivial for Redshift

-- ── QUERY 2: Facility Health Scorecard ───────────────
-- "Rank all facilities by operational health"
WITH facility_metrics AS (
    SELECT
        f.facility_id,
        fac.facility_name,
        fac.region,
        COUNT(DISTINCT f.sensor_id)          AS active_sensors,
        AVG(f.uptime_pct)                    AS avg_uptime,
        AVG(f.data_quality_pct)              AS avg_data_quality,
        SUM(CASE WHEN f.uptime_pct < 95 THEN 1 ELSE 0 END) AS sensors_below_95_uptime,
        SUM(f.suspect_count)                  AS total_suspect_readings
    FROM analytics.fact_readings_daily f
    JOIN analytics.dim_facilities fac ON f.facility_id = fac.facility_id
    WHERE f.date >= CURRENT_DATE - INTERVAL '30 days'
    GROUP BY 1, 2, 3
),
alert_metrics AS (
    SELECT
        facility_id,
        COUNT(*)                                               AS total_alerts,
        SUM(CASE WHEN severity = 'critical' THEN 1 ELSE 0 END) AS critical_alerts,
        AVG(latency_ms)                                         AS avg_alert_latency_ms
    FROM analytics.fact_alerts
    WHERE alert_ts >= CURRENT_DATE - INTERVAL '30 days'
    GROUP BY 1
)
SELECT
    fm.facility_id,
    fm.facility_name,
    fm.region,
    fm.active_sensors,
    ROUND(fm.avg_uptime, 2)               AS avg_uptime_pct,
    ROUND(fm.avg_data_quality, 2)         AS data_quality_pct,
    fm.sensors_below_95_uptime,
    COALESCE(am.total_alerts, 0)          AS alerts_30d,
    COALESCE(am.critical_alerts, 0)       AS critical_alerts_30d,
    COALESCE(am.avg_alert_latency_ms, 0)  AS avg_alert_latency,
    -- Health score: weighted composite
    ROUND(
        fm.avg_uptime * 0.3 +
        fm.avg_data_quality * 0.2 +
        (100 - LEAST(COALESCE(am.critical_alerts, 0), 100)) * 0.3 +
        (100 - fm.sensors_below_95_uptime::FLOAT / NULLIF(fm.active_sensors, 0) * 100) * 0.2
    , 2)                                   AS health_score,
    RANK() OVER (ORDER BY
        fm.avg_uptime * 0.3 +
        fm.avg_data_quality * 0.2 +
        (100 - LEAST(COALESCE(am.critical_alerts, 0), 100)) * 0.3 +
        (100 - fm.sensors_below_95_uptime::FLOAT / NULLIF(fm.active_sensors, 0) * 100) * 0.2
    DESC)                                  AS health_rank
FROM facility_metrics fm
LEFT JOIN alert_metrics am ON fm.facility_id = am.facility_id
ORDER BY health_score DESC;

-- WHY REDSHIFT WINS:
-- → Cross-facility comparison across 500 facilities
-- → CTE with JOIN to alerts table → complex multi-source analysis
-- → RANK() window function for ranking
-- → MongoDB: $lookup across collections + complex $group → impractical at scale

-- ── QUERY 3: Anomaly Pattern Detection Input ─────────
-- "Feed ML pipeline: hourly stats with rolling context"
SELECT
    f.sensor_id,
    f.sensor_type,
    f.facility_id,
    f.hour,
    f.avg_value,
    f.max_value,
    f.min_value,
    f.stddev_value,
    f.reading_count,
    f.suspect_pct,
    -- Rolling 24-hour context (for ML features)
    AVG(f.avg_value) OVER (
        PARTITION BY f.sensor_id
        ORDER BY f.hour
        ROWS BETWEEN 23 PRECEDING AND CURRENT ROW
    )                                       AS rolling_24h_avg,
    STDDEV(f.avg_value) OVER (
        PARTITION BY f.sensor_id
        ORDER BY f.hour
        ROWS BETWEEN 23 PRECEDING AND CURRENT ROW
    )                                       AS rolling_24h_stddev,
    -- Z-score: how far is current hour from 24h rolling average
    (f.avg_value - AVG(f.avg_value) OVER (
        PARTITION BY f.sensor_id
        ORDER BY f.hour
        ROWS BETWEEN 23 PRECEDING AND CURRENT ROW
    )) / NULLIF(STDDEV(f.avg_value) OVER (
        PARTITION BY f.sensor_id
        ORDER BY f.hour
        ROWS BETWEEN 23 PRECEDING AND CURRENT ROW
    ), 0)                                   AS z_score,
    -- Day of week pattern (is this typical for Tuesday 3pm?)
    EXTRACT(DOW FROM f.hour)                AS day_of_week,
    EXTRACT(HOUR FROM f.hour)               AS hour_of_day
FROM analytics.fact_readings_hourly f
WHERE f.hour >= CURRENT_DATE - INTERVAL '90 days'
ORDER BY f.sensor_id, f.hour;

-- WHY REDSHIFT WINS:
-- → Rolling window across 90 days × 100K sensors = 216M rows
-- → Z-score calculation with window STDDEV → native SQL
-- → This feeds directly into SageMaker anomaly detection model
-- → MongoDB: $setWindowFields across 216M documents → hours

✅ WHAT SHOULD DO (IoT-Specific Best Practices)

SHOULD DO:

✅ USE TIME SERIES COLLECTIONS (not regular collections)
    → 7-20x storage savings from automatic bucketing
    → Columnar compression on measurement fields
    → Optimized for time-range queries
    → TTL built-in for automatic data lifecycle
    → If you used manual Bucket Pattern (Project 16): migrate to native

✅ PRE-AGGREGATE BEFORE LOADING TO REDSHIFT
    → Raw per-second data: S3 only (queried via Spectrum when needed)
    → Hourly aggregates: Redshift local tables (most queries)
    → Daily aggregates: Redshift local tables (trend analysis)
    → This reduces Redshift storage 3600x (hourly) or 86400x (daily)

✅ USE write concern w:1 FOR SENSOR READINGS
    → Sensor data is replaceable (sensor will send again)
    → w:majority adds ~2-5ms latency per write × 100K writes/sec
    → Total wasted: 200-500 seconds of compute per second of real time
    → If primary crashes: lose last few seconds → acceptable for IoT
    → CONTRAST with e-commerce orders: use w:majority (money involved)

✅ BATCH INSERTS AGGRESSIVELY
    → Individual insertOne: 100K network round trips/second
    → Batch insertMany(5000): 20 network round trips/second
    → 5000x reduction in network overhead
    → MongoDB Time Series further batches into internal buckets

✅ SEPARATE METADATA FROM READINGS
    → Sensor metadata: regular collection (100K documents, rarely changes)
    → Sensor readings: time series collection (billions, append-only)
    → Different access patterns → different collection types
    → Metadata: read-heavy, cached, polymorphic
    → Readings: write-heavy, time-bucketed, uniform

✅ CACHE THRESHOLDS IN APPLICATION MEMORY
    → Alert engine checks thresholds for every reading (100K/sec)
    → Reading threshold from MongoDB: ~1ms per lookup
    → 100K lookups/sec × 1ms = 100 seconds of latency/sec → impossible
    → Cache all thresholds in memory, refresh every 60 seconds
    → In-memory lookup: <1 microsecond → feasible

❌ WHAT SHOULD NOT DO (IoT Anti-Patterns)

DO NOT:

❌ SHARD BY TIMESTAMP
    → #1 IoT mistake by Redshift developers
    → "Timestamp is my sort key in Redshift, so it should be my shard key"
    → All current writes go to ONE shard (latest time range)
    → That shard becomes a bottleneck at 100K writes/sec
    → Use: facility_id + sensor_id (distributes writes evenly)
    → Reference: Project 22 (monotonic shard key anti-pattern)

❌ STORE RAW PER-SECOND DATA IN REDSHIFT LOCAL TABLES
    → 8.64 billion rows/day → 3.15 trillion rows/year
    → Even compressed: ~15TB/year in Redshift
    → Redshift queries slow when scanning trillions of rows
    → Pre-aggregate to hourly/daily → store raw in S3 → Spectrum for raw access

❌ USE REGULAR COLLECTIONS FOR TIME SERIES DATA
    → Regular collection: each reading = one document = one index entry
    → 100K documents/second × ~200 bytes each = ~20MB/sec index writes
    → Index grows rapidly → doesn't fit in RAM → performance degrades
    → Time Series collection: automatic bucketing → 100x fewer index entries
    → Reference: Project 30

❌ USE w:majority FOR EVERY SENSOR READING
    → Explained above: massive latency overhead
    → Sensor data is inherently replaceable
    → Use w:1 for readings, w:majority for alerts and metadata changes

❌ POLL MONGODB FOR ALERTS (instead of Change Streams)
    → "Check every sensor every second for threshold breaches"
    → 100K queries/second just for alert checking → database overloaded
    → Change Streams: MongoDB PUSHES new readings to alert engine
    → Zero polling overhead → scales to any number of sensors

❌ KEEP ALL DATA IN MONGODB FOREVER
    → "MongoDB can scale horizontally, just add shards"
    → True, but expensive: NVMe SSD storage at $0.10-0.20/GB
    → vs S3 at $0.023/GB or Glacier at $0.004/GB
    → 7-day TTL on hot data → S3 for warm → Glacier for cold
    → 50x cost reduction on data older than 7 days

❌ IGNORE SENSOR CLOCK DRIFT
    → Sensor timestamp may differ from server time
    → MongoDB Time Series uses the ts field AS-IS
    → If sensor clock is wrong → data bucketed incorrectly
    → Always normalize timestamps at ingestion gateway
    → Add server_received_at field alongside sensor ts

⚠️ COMMON PITFALLS AND FAILURE MODES — IoT Specific

PITFALL 1: "I'll use the same document model as e-commerce"
    ❌ Rich nested documents for sensor readings
       { sensor: { id: "...", location: { facility: {...} } }, reading: {...} }
       → 500-byte documents × 100K/sec = 50MB/sec just for metadata overhead
    ✅ Flat time series documents with minimal metaField
       { ts: Date, meta: { sensor_id, facility_id }, temperature: 73.5 }
       → 100-byte documents → 10MB/sec → 5x less I/O
    LESSON: IoT documents should be as SMALL as possible
            E-commerce documents should be as COMPLETE as possible

PITFALL 2: "Let me index every field for fast queries"
    ❌ createIndex({ temperature: 1 })
       createIndex({ "meta.facility_id": 1, temperature: 1 })
       createIndex({ ts: 1, "meta.sensor_id": 1, temperature: 1 })
       → Each index updated on every write → 100K index updates/sec PER index
       → 5 indexes = 500K index operations/sec → write throughput halved
    ✅ Time Series collections auto-create internal indexes on timeField and metaField
       → Add minimal additional indexes ONLY for critical query patterns
       → Accept slower ad-hoc queries to maintain write throughput
    LESSON: IoT = write-optimized → minimize indexes
            E-commerce = read-optimized → generous indexes acceptable

PITFALL 3: "We need per-second data in Redshift for all analysis"
    ❌ Loading 8.64B rows/day into Redshift local tables
       → After 1 month: 259B rows → Redshift grinds to halt
       → VACUUM takes hours → query performance degrades
    ✅ Pre-aggregate to hourly (2.4M rows/day) and daily (100K rows/day)
       → After 1 year: hourly = 876M rows, daily = 36.5M rows → manageable
       → Raw per-second: S3 → Spectrum when needed (rare)
    LESSON: aggregate BEFORE loading to Redshift
            Redshift processes aggregated data 3600x faster than raw

PITFALL 4: "Change Streams can handle 100K events/second for alerting"
    ❌ Alert engine processes EVERY reading via change stream
       → Change stream delivers every insert to the watcher
       → 100K events/sec → alert engine overwhelmed
    ✅ Filter in the change stream pipeline
       → Only pass readings that might trigger alerts
       → OR: sample (every 10th reading per sensor)
       → OR: use MongoDB Atlas Triggers with filtering
       → OR: alert engine processes readings in micro-batches
    LESSON: Change streams are powerful but must be filtered at scale

PITFALL 5: "One MongoDB cluster for everything"
    ❌ Same cluster handles ingestion (100K writes/sec) AND dashboards AND metadata
       → Write-heavy workload starves read queries
       → Dashboard latency spikes during peak ingestion
    ✅ Separate read preference configuration
       → Ingestion: writes to PRIMARY only
       → Dashboards: reads from SECONDARY (readPreference: secondaryPreferred)
       → Metadata lookups: reads from NEAREST
       → OR: dedicated analytics secondary (hidden replica set member)
    LESSON: separate read and write paths at the replica set level

# ─── INGESTION SIMULATION ────────────────────────────
def simulate_sensor_readings(sensors, duration_seconds=10, readings_per_second=1):
    """
    Simulate sensor readings and insert into Time Series collection
    
    At demo scale: 100 sensors × 1 reading/sec = 100 writes/sec
    At production: 100K sensors × 1 reading/sec = 100K writes/sec
    Same code — just different scale
    
    COMPARISON TO REDSHIFT:
    → Redshift: cannot INSERT 100 rows/sec efficiently
       Must buffer → S3 → COPY (60+ second latency)
    → MongoDB: insertMany() directly (1-2 second latency)
    """
    
    # Base values for realistic simulation
    base_values = {
        "temperature": {"base": 72.0, "noise": 3.0, "drift": 0.01},
        "pressure":    {"base": 150.0, "noise": 5.0, "drift": 0.05},
        "vibration":   {"base": 0.5, "noise": 0.15, "drift": 0.001},
        "humidity":    {"base": 45.0, "noise": 2.0, "drift": 0.02}
    }
    
    # Track per-sensor state for realistic continuity
    sensor_state = {}
    for s in sensors:
        stype = s["type"]
        config = base_values[stype]
        sensor_state[s["_id"]] = config["base"] + random.gauss(0, config["noise"])
    
    total_inserted = 0
    total_alerts = 0
    start_time = time.time()
    
    # Threshold cache (in production: loaded from sensors collection)
    threshold_cache = {}
    for s in sensors:
        stype = s["type"]
        threshold_cache[s["_id"]] = {
            "type": stype,
            "thresholds": s["thresholds"].get(stype, {}),
            "facility_id": s["location"]["facility_id"]
        }
    
    print(f"\n⏱ Starting ingestion simulation: {len(sensors)} sensors, "
          f"{duration_seconds} seconds")
    print(f"  Expected: ~{len(sensors) * duration_seconds:,} readings\n")
    
    for second in range(duration_seconds):
        batch = []
        now = datetime.utcnow()
        
        for s in sensors:
            stype = s["type"]
            config = base_values[stype]
            
            # Simulate realistic sensor value with drift and noise
            current = sensor_state[s["_id"]]
            current += random.gauss(0, config["noise"] * 0.3)      # small noise
            current += config["drift"] * random.choice([-1, 1])    # random drift
            sensor_state[s["_id"]] = current
            
            # Occasionally inject anomaly (1 in 500 chance per reading)
            if random.random() < 0.002:
                current += config["noise"] * 5     # spike
                sensor_state[s["_id"]] = current
            
            value = round(current, 2)
            
            # Determine quality
            expected_range = config["noise"] * 3
            quality = "good" if abs(value - config["base"]) < expected_range else "suspect"
            
            # Build reading document
            reading = {
                "ts": now + timedelta(milliseconds=random.randint(0, 999)),
                "meta": {
                    "sensor_id": s["_id"],
                    "facility_id": s["location"]["facility_id"],
                    "type": stype,
                    "region": s["location"]["region"]
                },
                stype: value,
                "quality": quality
            }
            batch.append(reading)
            
            # Check thresholds (inline alert check)
            tc = threshold_cache.get(s["_id"], {})
            thresholds = tc.get("thresholds", {})
            
            if thresholds.get("critical_high") and value >= thresholds["critical_high"]:
                alert = {
                    "sensor_id": s["_id"],
                    "facility_id": tc["facility_id"],
                    "sensor_type": stype,
                    "severity": "critical",
                    "direction": "high",
                    "value": value,
                    "threshold": thresholds["critical_high"],
                    "delta": round(value - thresholds["critical_high"], 2),
                    "reading_ts": reading["ts"],
                    "alert_ts": datetime.utcnow(),
                    "latency_ms": 0,
                    "status": "active"
                }
                db.alerts.insert_one(alert)
                total_alerts += 1
                print(f"  🚨 CRITICAL: {s['_id']} | {stype}: {value} "
                      f"> threshold {thresholds['critical_high']}")
            
            elif thresholds.get("warning_high") and value >= thresholds["warning_high"]:
                alert = {
                    "sensor_id": s["_id"],
                    "facility_id": tc["facility_id"],
                    "sensor_type": stype,
                    "severity": "warning",
                    "direction": "high",
                    "value": value,
                    "threshold": thresholds["warning_high"],
                    "delta": round(value - thresholds["warning_high"], 2),
                    "reading_ts": reading["ts"],
                    "alert_ts": datetime.utcnow(),
                    "latency_ms": 0,
                    "status": "active"
                }
                db.alerts.insert_one(alert)
                total_alerts += 1
                print(f"  ⚠️  WARNING: {s['_id']} | {stype}: {value} "
                      f"> threshold {thresholds['warning_high']}")
        
        # Bulk insert all readings for this second
        insert_start = time.time()
        result = db.readings.insert_many(batch, ordered=False)
        insert_ms = (time.time() - insert_start) * 1000
        
        total_inserted += len(result.inserted_ids)
        
        # Progress report every 2 seconds
        if (second + 1) % 2 == 0:
            elapsed = time.time() - start_time
            throughput = total_inserted / elapsed
            print(f"  [{second+1}s] Inserted: {total_inserted:,} | "
                  f"Throughput: {throughput:,.0f}/sec | "
                  f"Batch insert: {insert_ms:.1f}ms | "
                  f"Alerts: {total_alerts}")
        
        # Pace to 1 reading/second/sensor
        elapsed_this_second = time.time() - (start_time + second)
        if elapsed_this_second < 1.0:
            time.sleep(1.0 - elapsed_this_second)
    
    total_elapsed = time.time() - start_time
    final_throughput = total_inserted / total_elapsed
    
    print(f"\n✅ Ingestion complete:")
    print(f"   Total readings: {total_inserted:,}")
    print(f"   Total alerts: {total_alerts}")
    print(f"   Duration: {total_elapsed:.1f}s")
    print(f"   Throughput: {final_throughput:,.0f} readings/sec")
    print(f"   Avg batch insert time: ~{(total_elapsed/duration_seconds)*1000/len(sensors)*len(sensors):.1f}ms")
    
    return total_inserted, total_alerts

def demo_facility_health_summary():
    """
    QUERY: Aggregate health summary across all facilities
    Use case: Executive dashboard / operations center
    
    MongoDB: <500ms for real-time operational view
    Redshift: 5-15s for comprehensive historical analysis
    
    EACH system serves a DIFFERENT version of this dashboard:
    → MongoDB: "What's happening RIGHT NOW?" (last hour)
    → Redshift: "How have we trended over 6 months?" (historical)
    """
    
    print(f"\n🏭 FACILITY HEALTH SUMMARY")
    print(f"   (Last hour of data)\n")
    
    start = time.time()
    
    one_hour_ago = datetime.utcnow() - timedelta(hours=1)
    
    # Facility-level aggregation from readings
    reading_stats = list(db.readings.aggregate([
        {"$match": {"ts": {"$gte": one_hour_ago}}},
        {
            "$group": {
                "_id": {
                    "facility": "$meta.facility_id",
                    "type": "$meta.type"
                },
                "sensor_count": {"$addToSet": "$meta.sensor_id"},
                "reading_count": {"$sum": 1},
                "avg_value": {
                    "$avg": {
                        "$ifNull": [
                            "$temperature",
                            {"$ifNull": ["$pressure",
                                {"$ifNull": ["$vibration", "$humidity"]}]}
                        ]
                    }
                },
                "suspect_count": {
                    "$sum": {"$cond": [{"$eq": ["$quality", "suspect"]}, 1, 0]}
                }
            }
        },
        {
            "$group": {
                "_id": "$_id.facility",
                "sensor_types": {
                    "$push": {
                        "type": "$_id.type",
                        "sensors": {"$size": "$sensor_count"},
                        "readings": "$reading_count",
                        "avg_value": {"$round": ["$avg_value", 2]},
                        "suspect": "$suspect_count"
                    }
                },
                "total_sensors": {"$sum": {"$size": "$sensor_count"}},
                "total_readings": {"$sum": "$reading_count"},
                "total_suspect": {"$sum": "$suspect_count"}
            }
        },
        {"$sort": {"_id": 1}}
    ]))
    
    # Alert counts per facility
    alert_stats = {doc["_id"]: doc["count"] for doc in db.alerts.aggregate([
        {
            "$match": {
                "alert_ts": {"$gte": one_hour_ago}
            }
        },
        {
            "$group": {
                "_id": "$facility_id",
                "count": {"$sum": 1},
                "critical": {
                    "$sum": {"$cond": [{"$eq": ["$severity", "critical"]}, 1, 0]}
                }
            }
        }
    ])}
    
    elapsed_ms = (time.time() - start) * 1000
    
    for fac in reading_stats:
        fac_id = fac["_id"]
        alerts = alert_stats.get(fac_id, {"count": 0, "critical": 0})
        if isinstance(alerts, int):
            alerts = {"count": alerts, "critical": 0}
        
        # Fetch facility name
        facility = db.facilities.find_one({"_id": fac_id})
        fac_name = facility["name"] if facility else fac_id
        
        data_quality = round(
            (1 - fac["total_suspect"] / max(fac["total_readings"], 1)) * 100, 2
        )
        
        print(f"   🏭 {fac_name} ({fac_id})")
        print(f"      Active sensors: {fac['total_sensors']}")
        print(f"      Readings (1hr): {fac['total_readings']:,}")
        print(f"      Data quality:   {data_quality}%")
        print(f"      Alerts (1hr):   {alerts if isinstance(alerts, int) else alerts.get('count', 0)}")
        print(f"      By type:")
        for st in fac["sensor_types"]:
            print(f"        {st['type']:<14}: {st['sensors']} sensors, "
                  f"avg={st['avg_value']}, {st['readings']:,} readings")
        print()
    
    print(f"   ⏱ Query time: {elapsed_ms:.2f}ms")
    print(f"   (This is OPERATIONAL health — for real-time monitoring)")
    print(f"   (For 6-month trend → Redshift query on pre-aggregated data)")
