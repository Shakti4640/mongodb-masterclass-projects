🔴 REDSHIFT MINDSET — "Exporting Data from Redshift"

What You Already Know About Getting Data OUT of Redshift

YOUR REDSHIFT EXPORT TOOLKIT:

1. UNLOAD command — bulk export to S3
    UNLOAD ('SELECT * FROM analytics.fact_orders')
    TO 's3://bucket/exports/orders_'
    IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftUnloadRole'
    FORMAT AS PARQUET               -- or CSV, JSON
    PARTITION BY (order_year)       -- partitioned output
    ALLOWOVERWRITE;

    → Parallel export across all slices
    → Creates multiple files (one per slice)
    → Compressed automatically
    → THIS is Redshift's strength: bulk operations

2. Direct query via psycopg2/JDBC
    → SELECT with cursor → fetch rows → process in Python
    → Fine for small datasets (<1M rows)
    → Too slow for 500GB (network bottleneck)

3. Redshift Data Sharing
    → Share data with other Redshift clusters
    → NOT useful for MongoDB migration

4. Federated Query (Redshift → external)
    → Query external databases FROM Redshift
    → NOT useful for writing TO MongoDB

THE CHALLENGE:
    → Redshift is great at EXPORTING flat rows
    → MongoDB needs NESTED documents
    → The transformation from flat → nested is THE hard part
    → Redshift has NO concept of "build a JSON document from JOINed tables"
    → You must do this transformation OUTSIDE Redshift

🧩 HOW TO APPROACH IT IN MONGODB

MIGRATION STRATEGY OVERVIEW:

CHALLENGE 1 (Redshift → MongoDB): BULK HISTORICAL MIGRATION
    
    PHASE A: EXPORT from Redshift
        → UNLOAD JOINed data to S3 as JSON/Parquet
        → Pre-JOIN in Redshift SQL (leverage MPP for the heavy JOIN)
        → Export in chunks (by date range or customer segment)
    
    PHASE B: TRANSFORM
        → Read exported flat rows
        → Group by order_id
        → Build nested document structure
        → Handle data type conversions (Redshift types → BSON types)
    
    PHASE C: LOAD into MongoDB
        → insertMany() with ordered:false (continue on errors)
        → Batch size: 1000 documents per batch
        → Monitor progress and errors
    
    PHASE D: VALIDATE
        → Count reconciliation (Redshift rows → MongoDB documents)
        → Value reconciliation (SUM of amounts match)
        → Spot-check random documents against source rows
    
    PHASE E: CUTOVER
        → Point application to MongoDB
        → Start Project 34 pipeline (MongoDB → Redshift for new data)
        → Decommission RetailMax Redshift cluster

CHALLENGE 2 (MongoDB Backfill → Redshift): HISTORICAL BACKFILL
    
    → Export existing MongoDB documents using mongoexport or aggregation
    → Transform using same logic as Project 34 Component 2
    → Load into Redshift using COPY
    → Then enable real-time pipeline for new changes
    
CHALLENGE 3 (Redshift → MongoDB enrichment): TARGETED REVERSE FLOW
    
    → UNLOAD ML scores from Redshift to S3
    → Read scores, match to MongoDB documents by customer_id
    → updateMany() to enrich MongoDB documents with scores
    → NOT a full sync — specific, controlled data flow

# ─── DATA TYPE CONVERTERS ─────────────────────────────
# Redshift types → BSON types
# This is where type mismatches cause silent bugs

def convert_redshift_to_bson(value, target_type="auto"):
    """
    Convert Redshift column values to appropriate BSON types
    
    CRITICAL CONVERSIONS:
    → Redshift DECIMAL  → BSON Decimal128 (for money, preserve precision)
    → Redshift TIMESTAMP → BSON ISODate (datetime object in Python)
    → Redshift BIGINT    → BSON Int64 (or keep as Python int)
    → Redshift VARCHAR    → BSON String (direct)
    → Redshift BOOLEAN    → BSON Boolean (direct)
    → Redshift NULL       → omit field entirely (MongoDB convention)
    
    WHY THIS MATTERS:
    → Storing price as float (29.99) → floating point errors in MongoDB
    → Storing price as Decimal128("29.99") → exact precision preserved
    → Redshift enforces types per column; MongoDB per value
    → One wrong conversion = corrupted data across millions of documents
    """
    if value is None:
        return None
    
    if target_type == "decimal":
        return Decimal128(str(value))
    elif target_type == "datetime":
        if isinstance(value, str):
            return datetime.fromisoformat(value.replace("Z", "+00:00"))
        return value
    elif target_type == "int":
        return int(value)
    elif target_type == "bool":
        return bool(value)
    else:
        return value

# ─── S3 PARQUET READER ────────────────────────────────

def read_parquet_from_s3(prefix):
    """
    Read all Parquet files from an S3 prefix
    Yields rows as dictionaries
    
    WHY PARQUET:
    → Redshift UNLOAD to Parquet preserves types (DECIMAL stays DECIMAL)
    → CSV would lose type information (everything becomes string)
    → Parquet is columnar on disk but we read as rows (pyarrow handles this)
    """
    paginator = s3.get_paginator("list_objects_v2")
    
    for page in paginator.paginate(Bucket=S3_BUCKET, Prefix=prefix):
        for obj in page.get("Contents", []):
            key = obj["Key"]
            if not key.endswith(".parquet"):
                continue
            
            logger.info(f"Reading s3://{S3_BUCKET}/{key}")
            
            # Download parquet file
            response = s3.get_object(Bucket=S3_BUCKET, Key=key)
            parquet_bytes = response["Body"].read()
            
            # Read with pyarrow
            table = pq.read_table(io.BytesIO(parquet_bytes))
            
            # Convert to Python dicts (row by row)
            for batch in table.to_batches():
                for row in batch.to_pydict().values():
                    pass  # handled below
            
            # More efficient: iterate as pandas
            df = table.to_pandas()
            for _, row in df.iterrows():
                yield row.to_dict()


# ─── ORDER ITEMS INDEX BUILDER ────────────────────────

def build_items_index(items_prefix):
    """
    Pre-build an index of order_items grouped by order_id
    
    THE KEY CHALLENGE:
    → Order items are in separate Parquet files from orders
    → Each order has 1-50 items (variable)
    → Must GROUP items by order_id to build items[] array
    
    STRATEGY:
    → Read ALL items into memory grouped by order_id
    → For 800M items this WON'T fit in memory
    → So we process in CHUNKS (by order_id range)
    
    ALTERNATIVE STRATEGIES:
    → Strategy A: Pre-sort items by order_id in Redshift (we did this with ORDER BY)
    →             Then stream-process: when order_id changes, flush items
    → Strategy B: Use temporary MongoDB collection as staging
    →             Insert all items, then aggregate with $group
    → Strategy C: Process in date chunks (match order export chunks)
    
    We'll use Strategy A (streaming) for memory efficiency
    """
    logger.info("Building order items index (streaming by order_id)...")
    
    current_order_id = None
    current_items = []
    items_by_order = {}
    flush_count = 0
    
    for row in read_parquet_from_s3(items_prefix):
        order_id = int(row["order_id"])
        
        item = {
            "sku": row.get("sku", ""),
            "product_name": row.get("product_name", ""),
            "category": row.get("category", ""),
            "subcategory": row.get("subcategory", ""),
            "brand": row.get("brand", ""),
            "unit_price": convert_redshift_to_bson(row.get("unit_price"), "decimal"),
            "quantity": convert_redshift_to_bson(row.get("quantity"), "int"),
            "subtotal": convert_redshift_to_bson(row.get("subtotal"), "decimal"),
            "discount_amount": convert_redshift_to_bson(
                row.get("discount_amount", 0), "decimal"
            )
        }
        
        # Remove None values (MongoDB convention: omit rather than store null)
        item = {k: v for k, v in item.items() if v is not None}
        
        if order_id not in items_by_order:
            items_by_order[order_id] = []
        items_by_order[order_id].append(item)
        
        flush_count += 1
        if flush_count % 1000000 == 0:
            logger.info(f"  Indexed {flush_count:,} items for {len(items_by_order):,} orders")
    
    logger.info(f"  Total: {flush_count:,} items for {len(items_by_order):,} orders")
    return items_by_order


# ─── DOCUMENT BUILDER ─────────────────────────────────

def build_order_document(order_row, items_for_order):
    """
    THE CORE TRANSFORMATION: flat Redshift rows → nested MongoDB document
    
    This is the REVERSE of Project 34's transform_order_to_redshift_rows()
    
    Project 34 (MongoDB → Redshift):
        document.customer.name → dim_customers.name column
        document.items[0].sku  → fact_order_items row with sku column
        1 document → N rows
    
    Project 35 (Redshift → MongoDB):
        dim_customers.name column → document.customer.name
        fact_order_items rows     → document.items[] array
        N rows → 1 document
    """
    
    order_id = int(order_row["order_id"])

# ─── BATCH INSERTER ───────────────────────────────────

class MongoDBBatchInserter:
    """
    Efficiently insert documents into MongoDB in batches
    
    Key considerations:
    → ordered=False: continue on error (don't stop for one bad document)
    → writeConcern w:1: fast writes during migration (not w:majority)
    → Batch size 1000: balance between memory and round-trip overhead
    → Error tracking: log failures for retry
    
    COMPARISON TO REDSHIFT COPY:
    → Redshift COPY: reads from S3, parallel across slices, very fast
    → MongoDB insertMany: sends documents over network, sequential batches
    → MongoDB is SLOWER for bulk load (this is a known trade-off)
    → But MongoDB can handle it — just takes longer
    """
    
    def __init__(self, collection, batch_size=1000):
        self.collection = collection
        self.batch_size = batch_size
        self.buffer = []
        self.total_inserted = 0
        self.total_errors = 0
        self.error_log = []
        self.start_time = time.time()
    
    def add(self, document):
        self.buffer.append(document)
        if len(self.buffer) >= self.batch_size:
            self.flush()
    
    def flush(self):
        if not self.buffer:
            return
        
        try:
            result = self.collection.insert_many(
                self.buffer,
                ordered=False,           # continue past errors
                # For migration: w:1 is faster than w:majority
                # We'll verify data integrity in Phase D
            )
            self.total_inserted += len(result.inserted_ids)
            
        except BulkWriteError as bwe:
            # Some documents may have succeeded even if some failed
            self.total_inserted += bwe.details.get("nInserted", 0)
            errors = bwe.details.get("writeErrors", [])
            self.total_errors += len(errors)
            
            # Log first few errors for debugging
            for err in errors[:5]:
                self.error_log.append({
                    "index": err.get("index"),
                    "code": err.get("code"),
                    "message": err.get("errmsg", "")[:200]
                })
            
            logger.warning(
                f"Batch had {len(errors)} errors. "
                f"Total errors so far: {self.total_errors}"
            )
        
        # Progress logging
        elapsed = time.time() - self.start_time
        rate = self.total_inserted / elapsed if elapsed > 0 else 0
        
        if self.total_inserted % 10000 == 0:
            logger.info(
                f"Progress: {self.total_inserted:>12,} inserted | "
                f"{self.total_errors:>8,} errors | "
                f"{rate:>8,.0f} docs/sec | "
                f"Elapsed: {elapsed/60:>6.1f} min"
            )
        
        self.buffer = []
    
    def finish(self):
        self.flush()
        elapsed = time.time() - self.start_time
        
        logger.info(f"\n{'='*60}")
        logger.info(f"MIGRATION LOAD COMPLETE")
        logger.info(f"  Total inserted: {self.total_inserted:,}")
        logger.info(f"  Total errors:   {self.total_errors:,}")
        logger.info(f"  Error rate:     {self.total_errors/max(self.total_inserted,1)*100:.4f}%")
        logger.info(f"  Duration:       {elapsed/60:.1f} minutes")
        logger.info(f"  Avg rate:       {self.total_inserted/elapsed:,.0f} docs/sec")
        logger.info(f"{'='*60}")
        
        if self.error_log:
            logger.info(f"\nSample errors:")
            for err in self.error_log[:10]:
                logger.info(f"  Code {err['code']}: {err['message']}")
        
        return {
            "inserted": self.total_inserted,
            "errors": self.total_errors,
            "duration_minutes": elapsed / 60
        }

def run_migration(year_chunk=None):
    """
    Main migration function: Redshift (S3 export) → MongoDB
    
    Flow:
    1. Build items index (group items by order_id)
    2. Stream order rows from S3
    3. For each order: look up its items, build document
    4. Batch insert into MongoDB
    5. Return statistics for validation
    """
    
    logger.info(f"{'='*60}")
    logger.info(f"RETAILMAX → SHOPSTREAM MIGRATION")
    logger.info(f"Year chunk: {year_chunk or 'ALL'}")
    logger.info(f"Started: {datetime.now()}")
    logger.info(f"{'='*60}")
    
    # ─── STEP 1: PRE-PROCESS ITEMS ───────────────
    logger.info(f"\nSTEP 1: Building order items index...")
    items_prefix = "retailmax/order_items/"
    items_by_order = build_items_index(items_prefix)
    logger.info(f"  Items index built: {len(items_by_order):,} orders with items")
    
    # ─── STEP 2: PREPARE MONGODB ─────────────────
    logger.info(f"\nSTEP 2: Preparing MongoDB collection...")
    
    # Create indexes AFTER migration (faster insert without indexes)
    # But create unique index on _legacy_order_id to prevent duplicates on re-run
    orders_collection.create_index(
        "_legacy_order_id",
        unique=True,
        name="idx_legacy_order_id"
    )
    logger.info(f"  Unique index on _legacy_order_id created (idempotency)")
    
    # ─── STEP 3: STREAM ORDERS + BUILD + INSERT ──
    logger.info(f"\nSTEP 3: Streaming orders, building documents, inserting...")
    
    orders_prefix = f"retailmax/orders/"
    if year_chunk:
        orders_prefix = f"retailmax/orders/year={year_chunk}/"
    
    inserter = MongoDBBatchInserter(orders_collection, batch_size=BATCH_SIZE)
    orders_without_items = 0
    
    for order_row in read_parquet_from_s3(orders_prefix):
        order_id = int(order_row["order_id"])
        
        # Look up items for this order
        items_for_order = items_by_order.get(order_id, [])
        
        if not items_for_order:
            orders_without_items += 1
        
        # Build the MongoDB document
        document = build_order_document(order_row, items_for_order)
        
        # Add to batch
        inserter.add(document)
    
    # Flush remaining
    stats = inserter.finish()
    
    logger.info(f"\nOrders without items: {orders_without_items:,}")
    logger.info(f"(These may be cancelled orders or data quality issues)")
    
    # ─── STEP 4: CREATE OPERATIONAL INDEXES ──────
    logger.info(f"\nSTEP 4: Creating operational indexes...")
    
    # These indexes support the application queries (Project 9-11)
    indexes_to_create = [
        {"keys": [("customer._legacy_customer_id", 1)], "name": "idx_customer_legacy_id"},
        {"keys": [("status", 1), ("created_at", -1)], "name": "idx_status_date"},
        {"keys": [("created_at", -1)], "name": "idx_created_at"},
        {"keys": [("order_number", 1)], "name": "idx_order_number", "unique": True},
        {"keys": [("items.sku", 1)], "name": "idx_items_sku"},
        {"keys": [("customer.email", 1)], "name": "idx_customer_email"},
    ]
    
    for idx in indexes_to_create:
        unique = idx.pop("unique", False)
        orders_collection.create_index(
            idx["keys"],
            name=idx["name"],
            unique=unique,
            background=True    # don't block reads during index build
        )
        logger.info(f"  Created index: {idx['name']}")
    
    return stats

def validate_counts():

    items_match = rs_items_count == mongo_items_count
    logger.info(f"\n  Order Items:")
    logger.info(f"    Redshift:  {rs_items_count:>15,}")
    logger.info(f"    MongoDB:   {mongo_items_count:>15,}")
    logger.info(f"    Match:     {'✅ PASS' if items_match else '❌ FAIL'}")

    mongo_customer_result = list(mongo_db.orders.aggregate([
        {"$match": {"_migration_source": "retailmax_redshift"}},
        {"$group": {"_id": "$customer._legacy_customer_id"}},

        mongo_customer_result[0]["unique_customers"] if mongo_customer_result else 0

    logger.info(f"    MongoDB unique in orders: {mongo_customer_count:>10,}")

def validate_values():

    CRITICAL: financial data must match EXACTLY
    → Redshift SUM(total) must equal MongoDB sum of payment.total
    → Any discrepancy = data corruption in migration
    """
    
    rs = get_redshift()
    rs_cur = rs.cursor()
    mongo_db = get_mongo()

    logger.info("LEVEL 2: VALUE VALIDATION")
    logger.info("=" * 60)
    
    # ─── Total revenue comparison ─────────────────
    rs_cur.execute("""
        SELECT
            SUM(total)          AS total_revenue,
            SUM(subtotal)       AS total_subtotal,
            SUM(shipping_cost)  AS total_shipping,
            SUM(tax)            AS total_tax,
            AVG(total)          AS avg_order_value
        FROM retailmax.fact_orders;
    """)
    rs_row = rs_cur.fetchone()
    rs_revenue = float(rs_row[0])
    rs_subtotal = float(rs_row[1])
    
    mongo_result = list(mongo_db.orders.aggregate([
        {"$match": {"_migration_source": "retailmax_redshift"}},
        {"$group": {
            "_id": None,
            "total_revenue": {"$sum": {"$toDouble": "$payment.total"}},
            "total_subtotal": {"$sum": {"$toDouble": "$payment.subtotal"}},
            "avg_order_value": {"$avg": {"$toDouble": "$payment.total"}}
        }}
    ]))
    mongo_revenue = mongo_result[0]["total_revenue"] if mongo_result else 0
    mongo_subtotal = mongo_result[0]["total_subtotal"] if mongo_result else 0
    
    revenue_diff = abs(rs_revenue - mongo_revenue)
    revenue_match = revenue_diff < 1.00  # $1 tolerance for floating point
    
    logger.info(f"\n  Total Revenue:")
    logger.info(f"    Redshift:   ${rs_revenue:>15,.2f}")
    logger.info(f"    MongoDB:    ${mongo_revenue:>15,.2f}")
    logger.info(f"    Difference: ${revenue_diff:>15,.2f}")
    logger.info(f"    Match:      {'✅ PASS' if revenue_match else '❌ FAIL'}")
    
    # ─── Revenue by year comparison ───────────────
    rs_cur.execute("""

        FROM retailmax.fact_orders

    rs_yearly = {int(row[0]): {"revenue": float(row[1]), "orders": row[2]}

            "revenue": {"$sum": {"$toDouble": "$payment.total"}},

        {"$sort": {"_id": 1}}

    logger.info(f"    {'Year':<8} {'RS Revenue':>15} {'Mongo Revenue':>15} {'RS Orders':>12} {'Mongo Orders':>12} {'Status'}")

            f"{rs_data['orders']:>12,} {m_data['orders']:>12,} {status}"
        )
    
    rs_cur.close()
    rs.close()
    
    return revenue_match

# ─── LEVEL 3: SAMPLE VALIDATION ──────────────────────

def validate_samples(num_samples=100):
    """
    Randomly sample orders and verify field-by-field
    
    For each sampled order:
    → Fetch from Redshift (by order_id)
    → Fetch from MongoDB (by _legacy_order_id)
    → Compare every field
    → Report any mismatches
    """
    
    rs = get_redshift()
    rs_cur = rs.cursor()
    mongo_db = get_mongo()
    
    logger.info("\n" + "=" * 60)
    logger.info(f"LEVEL 3: SAMPLE VALIDATION ({num_samples} random orders)")
    logger.info("=" * 60)
    
    # Get random order IDs from Redshift
    rs_cur.execute(f"""
        SELECT order_id FROM retailmax.fact_orders
        ORDER BY RANDOM()
        LIMIT {num_samples};
    """)
    sample_ids = [row[0] for row in rs_cur.fetchall()]
    
    mismatches = 0
    
    for order_id in sample_ids:
        # Fetch from Redshift
        rs_cur.execute("""
            SELECT order_id, order_number, status, total, customer_id
            FROM retailmax.fact_orders
            WHERE order_id = %s;
        """, (order_id,))
        rs_order = rs_cur.fetchone()
        
        if not rs_order:
            logger.warning(f"  Order {order_id}: NOT FOUND in Redshift")
            mismatches += 1
            continue
        
        # Fetch from MongoDB
        mongo_order = mongo_db.orders.find_one(
            {"_legacy_order_id": order_id}
        )
        
        if not mongo_order:
            logger.warning(f"  Order {order_id}: NOT FOUND in MongoDB")
            mismatches += 1
            continue
        
        # Compare key fields
        checks = [
            ("status", rs_order[2], mongo_order.get("status")),
            ("total", float(rs_order[3]), float(str(mongo_order.get("payment", {}).get("total", 0)))),
            ("customer_id", rs_order[4], mongo_order.get("customer", {}).get("_legacy_customer_id")),
        ]
        
        for field, rs_val, mongo_val in checks:
            if field == "total":
                if abs(rs_val - mongo_val) > 0.01:
                    logger.warning(
                        f"  Order {order_id}: {field} mismatch: "
                        f"RS={rs_val} vs Mongo={mongo_val}"
                    )
                    mismatches += 1
            else:
                if str(rs_val) != str(mongo_val):
                    logger.warning(
                        f"  Order {order_id}: {field} mismatch: "
                        f"RS={rs_val} vs Mongo={mongo_val}"
                    )
                    mismatches += 1
    
    match_rate = (num_samples - mismatches) / num_samples * 100
    logger.info(f"\n  Samples checked: {num_samples}")
    logger.info(f"  Mismatches:      {mismatches}")
    logger.info(f"  Match rate:      {match_rate:.1f}%")
    logger.info(f"  Status:          {'✅ PASS' if match_rate >= 99.9 else '❌ FAIL'}")
    
    rs_cur.close()
    rs.close()
    
    return match_rate >= 99.9

# │  (NO pipeline)   │  (CDC active)  │
# │                  │                │
# └──────────────────┴────────────────┘
#   BACKFILL NEEDED    PIPELINE HANDLES
#   (this code)        (Project 34)
#
# CRITICAL REQUIREMENT:
# → Backfill must NOT interfere with live CDC pipeline
# → Backfill must NOT duplicate data already in Redshift
# → Backfill must be RESTARTABLE (resume from failure point)
# ──────────────────────────────────────────────────────

def backfill_to_redshift(
    cutoff_date,           # documents BEFORE this date need backfill
    batch_size=5000,
    s3_bucket="shopstream-redshift-staging"
):
    """
    Backfill historical MongoDB data into Redshift
    
    Strategy:
    1. Query MongoDB for documents older than cutoff_date
    2. Process in date-range chunks (one week at a time)
    3. Transform using same logic as Project 34 Component 2
    4. Write to S3 with special prefix (backfill/)
    5. Redshift COPY picks up from backfill prefix
    
    WHY DATE CHUNKS:
    → 50M documents don't fit in memory
    → Date chunks are restartable (if week 12 fails, restart from week 12)
    → Each chunk can be validated independently
    → Matches Redshift sort key (order_date) for efficient loading
    """
    
    client = MongoClient("mongodb+srv://admin:pass@cluster0.xxxxx.mongodb.net/")
    db = client["shopstream"]
    orders = db["orders"]
    s3_client = boto3.client("s3")
    
    logger.info(f"Backfilling documents created before {cutoff_date}")
    
    # Find the earliest document
    earliest = orders.find_one(
        sort=[("created_at", 1)],
        projection={"created_at": 1}
    )
    
    if not earliest:
        logger.info("No documents found")
        return
    
    start_date = earliest["created_at"]
    logger.info(f"Earliest document: {start_date}")
    logger.info(f"Cutoff date: {cutoff_date}")
    
    # Process in weekly chunks
    chunk_start = start_date
    chunk_number = 0
    total_exported = 0
    
    while chunk_start < cutoff_date:
        chunk_end = min(chunk_start + timedelta(days=7), cutoff_date)
        chunk_number += 1
        
        logger.info(f"\nChunk {chunk_number}: {chunk_start.date()} → {chunk_end.date()}")
        
        # ─── READ FROM MONGODB ────────────────────
        # Use hint to force index usage on created_at
        # Read preference: secondaryPreferred (don't impact primary)
        
        cursor = orders.find(
            {
                "created_at": {
                    "$gte": chunk_start,
                    "$lt": chunk_end
                }
            }
        ).hint("idx_created_at").batch_size(batch_size)
        
        # ─── TRANSFORM + WRITE TO S3 ─────────────
        # Reuse Project 34 transformation logic
        # but write to a SEPARATE S3 prefix (backfill/)
        
        fact_orders_rows = []
        fact_items_rows = []
        dim_customers_rows = []
        chunk_doc_count = 0
        
        for doc in cursor:
            chunk_doc_count += 1
            
            # ── Transform order header ────────────
            order_id = str(doc["_id"])
            customer = doc.get("customer", {})
            payment = doc.get("payment", {})
            shipping = doc.get("shipping", {})
            address = shipping.get("address", {})
            
            fact_orders_rows.append({
                "order_id":         order_id,
                "order_number":     doc.get("order_number", ""),
                "order_date":       doc.get("created_at", ""),
                "status":           doc.get("status", ""),
                "customer_id":      str(customer.get("customer_id",
                                        customer.get("_legacy_customer_id", ""))),
                "item_count":       doc.get("item_count", 0),
                "subtotal":         str(payment.get("subtotal", 0)),
                "shipping_cost":    str(payment.get("shipping", 0)),
                "tax":              str(payment.get("tax", 0)),
                "total":            str(payment.get("total", 0)),
                "payment_method":   payment.get("method", ""),
                "source_channel":   doc.get("source", ""),
                "campaign_id":      doc.get("marketing_campaign", ""),
                "shipping_state":   address.get("state", ""),
                "shipping_country": address.get("country", ""),
                "created_at":       str(doc.get("created_at", "")),
                "updated_at":       str(doc.get("updated_at", ""))
            })
            
            # ── Transform line items (array → rows) ──
            for item in doc.get("items", []):
                fact_items_rows.append({
                    "order_id":         order_id,
                    "order_number":     doc.get("order_number", ""),
                    "order_date":       str(doc.get("created_at", "")),
                    "order_status":     doc.get("status", ""),
                    "sku":              item.get("sku", ""),
                    "product_name":     item.get("product_name", ""),
                    "unit_price":       str(item.get("unit_price", 0)),
                    "quantity":         item.get("quantity", 0),
                    "subtotal":         str(item.get("subtotal", 0)),
                    "customer_id":      str(customer.get("customer_id",
                                            customer.get("_legacy_customer_id", ""))),
                    "product_category": item.get("category", ""),
                    "shipping_state":   address.get("state", ""),
                    "shipping_country": address.get("country", ""),
                    "payment_method":   payment.get("method", ""),
                    "source_channel":   doc.get("source", ""),
                    "campaign_id":      doc.get("marketing_campaign", ""),
                    "created_at":       str(doc.get("created_at", "")),
                    "updated_at":       str(doc.get("updated_at", "")),
                    "_mongo_doc_id":    order_id
                })
            
            # ── Extract customer dimension ────────
            cust_id = str(customer.get("customer_id",
                          customer.get("_legacy_customer_id", "")))
            if cust_id:
                dim_customers_rows.append({
                    "customer_id":  cust_id,
                    "name":         f"{customer.get('first_name', '')} {customer.get('last_name', customer.get('name', ''))}".strip(),
                    "email":        customer.get("email", ""),
                    "segment":      customer.get("segment", ""),
                    "region":       customer.get("region", "")
                })
        
        # ─── WRITE CHUNK TO S3 ────────────────────
        timestamp = datetime.utcnow().strftime("%Y%m%d_%H%M%S")
        chunk_label = f"{chunk_start.strftime('%Y%m%d')}_{chunk_end.strftime('%Y%m%d')}"
        
        tables_to_write = {
            "fact_orders":      fact_orders_rows,
            "fact_order_items": fact_items_rows,
            "dim_customers":    dim_customers_rows
        }
        
        for table_name, rows in tables_to_write.items():
            if not rows:
                continue
            
            # Write as gzipped CSV
            output = io.StringIO()
            writer = csv.DictWriter(output, fieldnames=rows[0].keys())
            writer.writeheader()
            writer.writerows(rows)
            
            compressed = gzip.compress(output.getvalue().encode("utf-8"))
            
            # SEPARATE prefix: backfill/ vs cdc-events/
            # This prevents collision with live pipeline
            s3_key = (
                f"backfill/"
                f"table={table_name}/"
                f"chunk={chunk_label}/"
                f"{table_name}_{timestamp}.csv.gz"
            )
            
            s3_client.put_object(
                Bucket=s3_bucket,
                Key=s3_key,
                Body=compressed
            )
            
            logger.info(f"  {table_name}: {len(rows):,} rows → s3://{s3_bucket}/{s3_key}")
        
        total_exported += chunk_doc_count
        logger.info(f"  Chunk {chunk_number}: {chunk_doc_count:,} docs ({total_exported:,} total)")
        
        # Move to next chunk
        chunk_start = chunk_end
    
    logger.info(f"\n{'='*60}")
    logger.info(f"BACKFILL EXPORT COMPLETE")
    logger.info(f"  Total documents exported: {total_exported:,}")
    logger.info(f"  Total chunks: {chunk_number}")
    logger.info(f"  S3 prefix: s3://{s3_bucket}/backfill/")
    logger.info(f"{'='*60}")
    
    return total_exported


# ─── BACKFILL REDSHIFT LOADER ─────────────────────────
# Loads backfill data into Redshift
# Uses SAME tables as live pipeline but DIFFERENT S3 prefix

def load_backfill_to_redshift():
    """
    Load backfill S3 files into Redshift
    
    CRITICAL CONSIDERATIONS:
    → Backfill data goes into SAME tables as live pipeline data
    → Must handle potential overlaps at the cutoff boundary
    → Use staging + upsert pattern to prevent duplicates
    → Run AFTER backfill export completes, BEFORE enabling live pipeline
    
    ORDERING:
    1. Export backfill to S3 (above function)
    2. Load backfill into Redshift (this function)
    3. THEN enable live CDC pipeline (Project 34)
    4. This ensures no gap and no overlap
    """
    
    import psycopg2
    
    conn = psycopg2.connect(
        host="my-cluster.xxxxx.region.redshift.amazonaws.com",
        port=5439, dbname="shopstream_analytics",
        user="etl_user", password="secure_password"
    )
    
    iam_role = "arn:aws:iam::123456789012:role/RedshiftCopyRole"
    s3_bucket = "shopstream-redshift-staging"
    
    backfill_tables = {
        "fact_orders": {
            "target": "analytics.fact_orders",
            "s3_prefix": f"s3://{s3_bucket}/backfill/table=fact_orders/",
            "merge_key": "order_id"
        },
        "fact_order_items": {
            "target": "analytics.fact_order_items",
            "s3_prefix": f"s3://{s3_bucket}/backfill/table=fact_order_items/",
            "merge_key": None      # append only
        },
        "dim_customers": {
            "target": "analytics.dim_customers",
            "s3_prefix": f"s3://{s3_bucket}/backfill/table=dim_customers/",
            "merge_key": "customer_id"
        }
    }
    
    with conn.cursor() as cur:
        for table_name, config in backfill_tables.items():
            logger.info(f"\nLoading backfill: {table_name}")
            
            target = config["target"]
            s3_prefix = config["s3_prefix"]
            merge_key = config["merge_key"]
            
            # Stage into temp table first
            staging = f"staging.backfill_{table_name}"
            cur.execute(f"DROP TABLE IF EXISTS {staging};")
            cur.execute(f"CREATE TABLE {staging} (LIKE {target});")
            
            # COPY from backfill prefix
            cur.execute(f"""
                COPY {staging}
                FROM '{s3_prefix}'
                IAM_ROLE '{iam_role}'
                CSV
                IGNOREHEADER 1
                GZIP
                DATEFORMAT 'auto'
                TIMEFORMAT 'auto'
                BLANKSASNULL
                EMPTYASNULL
                MAXERROR 1000;
            """)
            
            cur.execute(f"SELECT COUNT(*) FROM {staging};")
            staged_count = cur.fetchone()[0]
            logger.info(f"  Staged: {staged_count:,} rows")
            
            if merge_key:
                # Upsert: remove existing + insert new (atomic)
                cur.execute("BEGIN TRANSACTION;")
                cur.execute(f"""
                    DELETE FROM {target}
                    WHERE {merge_key} IN (SELECT {merge_key} FROM {staging});
                """)
                cur.execute(f"INSERT INTO {target} SELECT * FROM {staging};")
                cur.execute("END TRANSACTION;")
                logger.info(f"  Upserted {staged_count:,} rows into {target}")
            else:
                # Append: just insert
                cur.execute(f"INSERT INTO {target} SELECT * FROM {staging};")
                logger.info(f"  Appended {staged_count:,} rows into {target}")
            
            # Cleanup
            cur.execute(f"DROP TABLE IF EXISTS {staging};")
            
            # Analyze for fresh statistics
            cur.execute(f"ANALYZE {target};")
            logger.info(f"  ANALYZE complete on {target}")
    
    conn.commit()
    conn.close()
    
    logger.info(f"\nBackfill loading complete!")

def export_ml_scores_from_redshift():
    """
    Redshift has computed ML scores via:
    → Redshift ML (CREATE MODEL)
    → Or external SageMaker model invoked via UDF
    → Or pre-computed aggregation stored in analytics table
    
    Export these scores to S3 for MongoDB consumption
    """
    
    conn = psycopg2.connect(
        host="my-cluster.xxxxx.region.redshift.amazonaws.com",
        port=5439, dbname="shopstream_analytics",
        user="ml_user", password="secure_password"
    )
    
    # The ML scores table in Redshift
    # (computed by a separate ML pipeline — out of scope here)
    export_sql = """
        UNLOAD ($$
            SELECT
                customer_id,
                clv_score,
                churn_risk_score,
                recommended_segment,
                top_category_affinity,
                score_computed_at
            FROM analytics.ml_customer_scores
            WHERE score_computed_at >= DATEADD(day, -1, GETDATE())
        $$)
        TO 's3://shopstream-enrichment/ml-scores/latest/'
        IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftUnloadRole'
        FORMAT AS CSV
        HEADER
        GZIP
        ALLOWOVERWRITE;
    """
    
    with conn.cursor() as cur:
        cur.execute(export_sql)
        logger.info("ML scores exported to S3")
    
    conn.close()


# ─── STEP 2: READ SCORES AND UPDATE MONGODB ──────────

def enrich_mongodb_with_scores():
    """
    Read ML scores from S3 and update MongoDB customer documents
    
    KEY DESIGN DECISIONS:
    
    1. WHERE to store scores in MongoDB:
       ❌ Embedded in every order document (denormalization nightmare)
       ✅ In a separate customers collection OR
       ✅ As a dedicated sub-document in orders collection
       
       We'll update a customers collection with ml_scores sub-document
    
    2. HOW to update:
       → bulkWrite with UpdateOne operations
       → upsert: false (only update existing customers)
       → $set specific fields (don't overwrite entire document)
    
    3. WHY this is safe:
       → Only ADDS fields to existing documents
       → Never modifies operational fields (name, email, orders)
       → Application reads ml_scores for personalization
       → If scores are stale, personalization degrades gracefully
       → NOT a source-of-truth flow — enrichment only
    """
    
    s3_client = boto3.client("s3")
    mongo_client = MongoClient("mongodb+srv://admin:pass@cluster0.xxxxx.mongodb.net/")
    db = mongo_client["shopstream"]
    customers = db["customers"]
    
    # Ensure index on customer_id for fast lookups
    customers.create_index("customer_id", unique=True)
    
    # Read scores from S3
    bucket = "shopstream-enrichment"
    prefix = "ml-scores/latest/"
    
    paginator = s3_client.get_paginator("list_objects_v2")
    
    update_operations = []
    total_scores = 0
    
    for page in paginator.paginate(Bucket=bucket, Prefix=prefix):
        for obj in page.get("Contents", []):
            key = obj["Key"]
            if not key.endswith(".csv.gz"):
                continue
            
            logger.info(f"Reading {key}")
            
            response = s3_client.get_object(Bucket=bucket, Key=key)
            content = gzip.decompress(response["Body"].read()).decode("utf-8")
            
            reader = csv.DictReader(io.StringIO(content))
            
            for row in reader:
                customer_id = row.get("customer_id", "").strip()
                if not customer_id:
                    continue
                
                # Build the update operation
                # $set only the ml_scores sub-document
                # This ADDS or REPLACES ml_scores without touching other fields
                
                update_operations.append(
                    UpdateOne(
                        # FILTER: find customer by ID
                        {"customer_id": customer_id},
                        
                        # UPDATE: set ml_scores sub-document
                        {
                            "$set": {
                                "ml_scores": {
                                    "clv_score": float(row.get("clv_score", 0)),
                                    "churn_risk": float(row.get("churn_risk_score", 0)),
                                    "recommended_segment": row.get("recommended_segment", ""),
                                    "top_category": row.get("top_category_affinity", ""),
                                    "computed_at": datetime.fromisoformat(
                                        row.get("score_computed_at", datetime.utcnow().isoformat())
                                    ),
                                    "_source": "redshift_ml",
                                    "_enriched_at": datetime.utcnow()
                                }
                            }
                        },
                        upsert=False    # CRITICAL: don't create new customers
                                        # Only enrich EXISTING customers
                    )
                )
                
                total_scores += 1
                
                # Execute in batches of 1000
                if len(update_operations) >= 1000:
                    result = customers.bulk_write(
                        update_operations,
                        ordered=False    # continue past individual failures
                    )
                    logger.info(
                        f"  Batch: {result.modified_count} modified, "
                        f"{result.matched_count} matched"
                    )
                    update_operations = []
    
    # Flush remaining
    if update_operations:
        result = customers.bulk_write(update_operations, ordered=False)
        logger.info(
            f"  Final batch: {result.modified_count} modified, "
            f"{result.matched_count} matched"
        )
    
    logger.info(f"\nEnrichment complete: {total_scores:,} scores processed")
    
    # ─── VERIFY ENRICHMENT ────────────────────────
    # Spot check a customer document
    sample = customers.find_one(
        {"ml_scores": {"$exists": True}},
        {"customer_id": 1, "name": 1, "ml_scores": 1}
    )
    
    if sample:
        logger.info(f"\nSample enriched customer:")
        logger.info(f"  Customer: {sample.get('name', 'N/A')}")
        logger.info(f"  CLV Score: {sample['ml_scores']['clv_score']}")
        logger.info(f"  Churn Risk: {sample['ml_scores']['churn_risk']}")
        logger.info(f"  Segment: {sample['ml_scores']['recommended_segment']}")
        logger.info(f"  Source: {sample['ml_scores']['_source']}")

⚡ WHAT COULD BE DONE (Migration-Specific Optimizations)

OPTIMIZATION 1: PARALLEL MIGRATION WITH MULTIPROCESSING
    → Instead of single-threaded year-by-year migration
    → Use Python multiprocessing: one process per year chunk
    → Each process reads its S3 files + inserts into MongoDB independently
    → 3 years in parallel ≈ 3x faster
    → MongoDB handles concurrent inserts well (document-level locking)

OPTIMIZATION 2: DISABLE INDEXES DURING BULK LOAD
    → Drop all indexes BEFORE migration (except _id)
    → Insert all documents (much faster without index maintenance)
    → Rebuild indexes AFTER migration completes
    → Same concept as Redshift: load THEN sort (VACUUM SORT)
    → Can be 2-5x faster for large migrations

OPTIMIZATION 3: USE MONGOIMPORT FOR SIMPLE STRUCTURES
    → If documents are simple (no complex nesting needed)
    → Export from Redshift as JSON directly
    → mongoimport --uri "..." --collection orders --file orders.json --jsonArray
    → Faster than pymongo for pure bulk load
    → But LESS control over transformation logic

OPTIMIZATION 4: USE AWS DMS FOR AUTOMATED MIGRATION
    → AWS Database Migration Service supports MongoDB ↔ Redshift
    → DMS handles: full load + ongoing replication (CDC)
    → Less custom code needed
    → Trade-off: limited transformation capability
    → Complex nesting (arrays, deeply nested objects) may not transform correctly
    → Best for: simple schema migrations

OPTIMIZATION 5: PRE-JOIN IN REDSHIFT AS MATERIALIZED VIEW
    → Instead of exporting 5 tables separately and joining in Python
    → Create a materialized view that pre-joins everything
    → UNLOAD the materialized view as one dataset
    → Reduces Python transformation complexity
    → Redshift does the JOIN work (it's fast at JOINs)
    → Trade-off: can't produce nested arrays in SQL
    → Still need Python for items[] array assembly

    → These enable: traceability, debugging, rollback, validation
    → In Redshift: _mongo_doc_id serves the same purpose (Project 34)
    → BOTH directions need cross-reference IDs

✅ MIGRATE IN CHUNKS — NOT ALL AT ONCE
    → Date-based chunks: by year, by month, by week
    → Size-based chunks: 100K documents per batch
    → WHY: restartable, monitorable, debuggable
    → If chunk 47 of 200 fails, restart from chunk 47
    → Full re-migration of 500GB = days of wasted time

✅ VALIDATE AT EVERY LEVEL
    → Level 1: Counts (rows in = documents out)
    → Level 2: Values (SUM of money fields match)
    → Level 3: Samples (random spot-checks)
    → Level 4: Application smoke test (can the app work with migrated data?)
    → NEVER skip validation — data corruption is silent

✅ CREATE INDEXES AFTER BULK LOAD
    → Inserting with indexes = slower (each insert updates B-tree)
    → Insert THEN create indexes = faster (one B-tree build)
    → Exception: unique index on legacy_id (prevents duplicate on re-run)

✅ USE WRITE CONCERN w:1 DURING MIGRATION
    → Migration is bulk, not real-time
    → w:"majority" adds latency per batch (waits for replication)
    → w:1 is sufficient (data on primary, will replicate)
    → Validate AFTER migration completes
    → Switch to w:"majority" for live operational writes

✅ PLAN FOR CUTOVER CAREFULLY
    → The moment between "old system" and "new system" is dangerous
    → Strategy: run BOTH systems in parallel during transition
    → Dual-write: application writes to BOTH for N days
    → Validate both systems agree
    → THEN cut over reads to new system
    → THEN stop writes to old system

✅ DOCUMENT THE SCHEMA MAPPING
    → Create a formal mapping document:
      Redshift column → MongoDB field path
    → Review with stakeholders
    → Version control the mapping
    → This becomes the contract for the transformer code

❌ WHAT SHOULD NOT DO (Anti-Patterns for Migration)

❌ DO NOT MIGRATE REDSHIFT SCHEMA STRUCTURE INTO MONGODB
    WRONG:
    → One collection per Redshift table
    → db.fact_orders, db.fact_order_items, db.dim_customers
    → Using $lookup to JOIN them (recreating Redshift in MongoDB)
    
    WHY: defeats the purpose of MongoDB
    → You moved data but didn't transform the MODEL
    → $lookup is slow compared to Redshift JOINs
    → You get the worst of both worlds
    
    RIGHT:
    → Denormalize: JOIN in the migration, embed in the document
    → fact_orders + fact_order_items + dim_customers → ONE document
    → Design documents for APPLICATION access patterns
    → This is the ENTIRE point of moving to MongoDB

❌ DO NOT ASSUME TYPE COMPATIBILITY
    WRONG:
    → Redshift DECIMAL(10,2) → Python float → MongoDB double
    → 29.99 in Redshift → 29.[REDACTED:CREDIT_DEBIT_NUMBER] in MongoDB
    
    WHY: floating point precision loss
    → Financial data MUST use Decimal128 in MongoDB
    → convert_redshift_to_bson() function handles this (shown in code)
    
    RIGHT:
    → Redshift DECIMAL → Python Decimal → BSON Decimal128
    → Explicit type mapping for every column

❌ DO NOT RUN MIGRATION AGAINST PRODUCTION MONGODB
    WRONG:
    → Inserting 150M documents directly into production cluster
    → Competes with live application traffic
    → WiredTiger cache pressure (Project 33)
    → Application latency degrades during migration
    
    RIGHT:
    → Migrate into a SEPARATE cluster or replica set
    → Validate on separate cluster
    → Use mongodump/mongorestore to move to production
    → OR: use Atlas Live Migration to move data with zero downtime

❌ DO NOT SKIP THE BACKFILL GAP PROBLEM
    WRONG:
    → Start CDC pipeline on Monday
    → "We'll backfill historical data later"
    → Later never comes → Redshift has a gap
    → Reports show zero data before Monday
    
    RIGHT:
    → Backfill FIRST (Challenge 2), validate
    → THEN start CDC pipeline from known cutoff point
    → Verify no gap at the boundary

❌ DO NOT BUILD A GENERIC MIGRATION TOOL
    WRONG:
    → "Let's build a tool that migrates ANY Redshift schema to MongoDB"
    → Weeks of engineering on generic transformations
    → Complex configuration, edge cases everywhere
    
    RIGHT:
    → Build migration code SPECIFIC to your schema
    → One transformer per entity type (orders, customers, products)
    → Simpler, testable, maintainable
    → Generic tools (DMS) exist if you need them

❌ DO NOT IGNORE TIMEZONE HANDLING
    WRONG:
    → Redshift TIMESTAMP (no timezone) → MongoDB ISODate (UTC assumed)
    → All timestamps shifted by your server's local timezone offset
    → Orders at 11pm Pacific suddenly appear as next day UTC
    
    RIGHT:
    → Confirm Redshift timezone setting: SHOW timezone;
    → Convert explicitly: AT TIME ZONE 'UTC' in Redshift
    → Or: handle in Python with pytz/zoneinfo
    → Store in MongoDB as UTC (always)
    → Application converts to local timezone for display

⚙️ HOW IT ONLY WORKS INTERNALLY — The Transformation Engine

THE FUNDAMENTAL CHALLENGE: STRUCTURAL IMPEDANCE MISMATCH

Redshift stores data RELATIONALLY:
    ┌─────────────────────────────────────────────────┐
    │ fact_orders                                       │
    │ [order_id][customer_id][total][status][date]      │
    │     1         100       117.60 shipped  May31     │
    └─────────────────────────────────────────────────┘
    ┌─────────────────────────────────────────────────┐
    │ fact_order_items                                   │
    │ [item_id][order_id][sku][qty][subtotal]           │
    │     1        1      A    2    59.98               │
    │     2        1      B    1    35.99               │
    └─────────────────────────────────────────────────┘
    ┌─────────────────────────────────────────────────┐
    │ dim_customers                                     │
    │ [customer_id][name][segment]                      │
    │     100       Sarah premium                       │
    └─────────────────────────────────────────────────┘

    → Data SPREAD across tables
    → Connected by foreign keys
    → JOIN at query time to reconstruct full picture

MongoDB stores data as DOCUMENTS:
    ┌─────────────────────────────────────────────────┐
    │ { _id: 1,                                        │
    │   customer: { name: "Sarah", segment: "premium" }│
    │   items: [                                       │
    │     { sku: "A", qty: 2, subtotal: 59.98 },      │
    │     { sku: "B", qty: 1, subtotal: 35.99 }       │
    │   ],                                             │
    │   payment: { total: 117.60 },                    │
    │   status: "shipped"                              │
    │ }                                                │
    └─────────────────────────────────────────────────┘

    → Data CO-LOCATED in one document
    → No foreign keys needed
    → No JOIN at query time

THE TRANSFORMATION OPERATIONS:

REDSHIFT → MONGODB (normalized → denormalized):
    Operation 1: JOIN
        → Combine rows from multiple tables using foreign keys
        → fact_orders JOIN dim_customers ON customer_id
        → Redshift does this FAST (MPP hash join)
        → Do this IN Redshift before export (leverage its strength)
    
    Operation 2: NEST
        → Flat columns → nested object
        → customer_id, first_name, last_name, email
        → → customer: { id, first_name, last_name, email }
        → This is pure structural transformation (Python dict building)
    
    Operation 3: COLLECT (the hardest)
        → Multiple rows → array inside one document
        → fact_order_items WHERE order_id = 1
        → → items: [ {row1}, {row2}, {row3} ]
        → Requires GROUPING rows by parent key
        → Then COLLECTING child rows into array
        → This is what makes Redshift→MongoDB harder than the reverse

MONGODB → REDSHIFT (denormalized → normalized):
    Operation 1: FLATTEN
        → Nested object → flat columns
        → customer: { name, email } → customer_name, customer_email
        → Simple dot-notation traversal
    
    Operation 2: EXPLODE
        → Array → multiple rows
        → items: [{A}, {B}] → row for A, row for B
        → Each array element becomes its own row
        → Parent fields duplicated across child rows
    
    Operation 3: EXTRACT
        → Embedded sub-document → separate dimension row
        → customer: {...} → one row in dim_customers
        → Deduplicate across multiple source documents

WHY REDSHIFT → MONGODB IS HARDER:
    → COLLECT operation requires grouping + memory management
    → 800M order_items rows must be grouped by 150M order_ids
    → Each group (1-50 items) becomes one array
    → Can't fit all items in memory → streaming/chunking needed
    → No equivalent of Redshift's parallel MPP processing in Python
    
    → MongoDB → Redshift (FLATTEN + EXPLODE) is simpler because:
      → Each document is self-contained
      → Process one document at a time
      → No grouping needed — just iterate and flatten

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "THE 16MB DOCUMENT LIMIT SURPRISE"
    ❌ Order with 10,000 line items → items[] array → document > 16MB
    WHY: Redshift has no row size limit (columns stored separately)
         MongoDB has 16MB max document size
    WHEN: bulk/wholesale orders, B2B transactions, subscription bundles
    ✅ FIX: 
    → Check max items per order in Redshift BEFORE migrating:
      SELECT MAX(cnt) FROM (
          SELECT order_id, COUNT(*) cnt FROM fact_order_items GROUP BY 1
      );
    → If max > ~500 items: use Outlier Pattern (Project 17)
    → Embed first N items, reference overflow in separate collection

PITFALL 2: "DUPLICATE DOCUMENTS ON RE-RUN"
    ❌ Migration fails at chunk 47, restart inserts chunks 1-46 again
    WHY: insertMany without idempotency = duplicate documents
    ✅ FIX:
    → Unique index on _legacy_order_id (shown in code)
    → ordered:false continues past duplicate key errors
    → Re-run is safe: existing documents rejected, new ones inserted

PITFALL 3: "TIMEZONE SHIFT ACROSS ALL DATES"
    ❌ Redshift stored timestamps in US/Pacific
       MongoDB receives them as UTC
       All orders shifted by 7-8 hours
    WHY: Redshift TIMESTAMP has no timezone info
         MongoDB ISODate is always UTC
    ✅ FIX:
    → Check: SHOW timezone; in Redshift (often UTC, but verify)
    → Convert explicitly in Python: 
      .replace(tzinfo=ZoneInfo("US/Pacific")).astimezone(ZoneInfo("UTC"))

PITFALL 4: "ITEMS ARRAY EMPTY FOR ALL DOCUMENTS"
    ❌ Migrated 150M orders but all have items: []
    WHY: Order items exported to separate S3 files
         Items index build failed silently
         Or: order_id types don't match (BIGINT vs string)
    ✅ FIX:
    → Validate items index BEFORE starting document assembly:
      assert len(items_by_order) > 0
    → Validate type matching: int(order_row["order_id"]) matches items key type
    → Spot-check: first 10 documents should have items

PITFALL 5: "MEMORY EXHAUSTION ON ITEMS INDEX"
    ❌ 800M order items loaded into Python dict → OOM killed
    WHY: each item is ~500 bytes × 800M = ~400GB memory needed
    ✅ FIX:
    → Process in date chunks (our approach: year by year)
    → Or: use streaming approach (items sorted by order_id)
    → Or: use MongoDB as staging (insert items into temp collection,
      then aggregate with $group to build arrays)
    → Or: use Spark/Dask for distributed transformation

PITFALL 6: "ENRICHMENT OVERWRITES OPERATIONAL DATA"
    ❌ Challenge 3: $set on customer document accidentally replaces all fields
    WHY: Using $set at document root instead of $set on sub-document
    ✅ FIX:
    → Always $set a SPECIFIC sub-path: "ml_scores.clv_score"
    → Never $set at root level for enrichment
    → Test on staging collection first
    → Verify document structure before and after enrichment

BOTTLENECKS AND SOLUTIONS:

Bottleneck 1: Redshift UNLOAD speed
    → Typically 1-5 GB/minute depending on cluster size
    → 500GB ≈ 100-500 minutes (2-8 hours)
    → Parallel UNLOAD across slices is automatic
    → Not much you can do to speed this up
    → PARQUET format is faster than CSV

Bottleneck 2: Python transformation speed
    → Single-threaded: ~5,000-10,000 documents/second
    → With multiprocessing: ~20,000-50,000 documents/second
    → With Spark: ~100,000+ documents/second
    → The items[] array assembly is the slowest part
    → Pre-sorting items by order_id helps (sequential access)

Bottleneck 3: MongoDB insert speed
    → Single connection: ~5,000-10,000 inserts/second
    → Multiple connections (connection pool): ~20,000-50,000/second
    → Sharded cluster: ~100,000+ inserts/second (parallel across shards)
    → Indexes slow inserts: create AFTER migration
    → Write concern w:1 is 2-3x faster than w:"majority"

Bottleneck 4: Network bandwidth
    → S3 → EC2 (same region): ~10 Gbps
    → EC2 → MongoDB Atlas (VPC peering): ~5-10 Gbps
    → Run transformer on EC2 in SAME region as both S3 and MongoDB
    → Network is rarely the bottleneck if co-located

    → They move STRUCTURE, not transform it
    → Custom transformation code is almost always needed
    → Unless your data is already flat (no JOINs needed)
