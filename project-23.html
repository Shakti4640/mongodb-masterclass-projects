PROJECT 23: Replica Sets — Multi-AZ Reimagined

Problem: Production Database Needs Zero-Downtime Failover and Tunable Read/Write Guarantees

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform is now mission-critical
    
    CURRENT STATE:
        → Single MongoDB server handling all traffic
        → 15 million customers, 800 million orders
        → 3,000 reads/second, 500 writes/second at peak
        → Revenue: $2.4M/day flowing through the system
    
    LAST WEEK:
        → Server had a disk failure at 2:47 AM
        → Database was DOWN for 4 hours and 12 minutes
        → Estimated revenue loss: $420,000
        → Customer trust damage: immeasurable
        → CTO: "This can NEVER happen again"
    
    REQUIREMENTS:
        1. ZERO data loss on server failure
        2. Automatic failover in < 10 seconds
        3. No single point of failure
        4. Read scaling for analytics without impacting production
        5. Geographic redundancy (survive datacenter outage)
    
    ADDITIONAL COMPLEXITY:
        → Some operations MUST be consistent (payment processing)
        → Some operations CAN be slightly stale (product browsing)
        → Some operations need to read from nearest server (global users)
    
    Your Redshift brain says: "Enable Multi-AZ — one checkbox, done"
    MongoDB says: "Replica sets give you all this PLUS tunable consistency per operation"
    
    THIS is both simpler AND more powerful than what you're used to

🔴 REDSHIFT MINDSET — How You Would Solve This

Redshift's High Availability (What You Already Know)

REDSHIFT HA — THE MODEL YOU HAVE:

    OPTION 1: Multi-AZ Deployment (Redshift RA3)
    ─────────────────────────────────────────────
    → Enable Multi-AZ: ONE CHECKBOX in console
    → Redshift creates a standby in another Availability Zone
    → Automatic failover if primary AZ fails
    → Recovery Point Objective (RPO): 0 (zero data loss)
    → Recovery Time Objective (RTO): ~30-60 seconds
    → No application changes needed
    → No configuration tuning
    → You don't choose WHICH AZ, WHEN to failover, or HOW
    → Redshift handles EVERYTHING

    ┌─────────────────────┐        ┌─────────────────────┐
    │  PRIMARY (AZ-1)      │ sync   │  STANDBY (AZ-2)      │
    │  ┌────────────────┐  │ ────→  │  ┌────────────────┐  │
    │  │ Leader Node     │  │        │  │ Leader (standby)│  │
    │  │ Compute Nodes   │  │        │  │ Compute (warm)  │  │
    │  │ RA3 Storage     │  │        │  │ RA3 Storage     │  │
    │  └────────────────┘  │        │  └────────────────┘  │
    └─────────────────────┘        └─────────────────────┘
         ↑ reads + writes            (no reads, no writes)
         ↑ all traffic               (pure standby)

    WHAT YOU CAN'T DO IN REDSHIFT:
    ✗ Read from the standby (it's purely passive)
    ✗ Choose consistency level per query
    ✗ Route reads to nearest server
    ✗ Have more than 2 copies (primary + standby)
    ✗ Configure write durability per operation
    ✗ Use standby for analytics while primary serves OLTP

    OPTION 2: Cross-Region Snapshots
    ─────────────────────────────────
    → Automated snapshots copied to another region
    → Manual restore in case of region failure
    → RPO: up to 8 hours (snapshot interval)
    → RTO: 30-60 minutes (restore time)
    → NOT automatic failover

    OPTION 3: Data Sharing (Read Replicas)
    ──────────────────────────────────────
    → Share data with consumer clusters (read-only)
    → Different from HA — for read scaling, not failover
    → Consumer clusters can be in different regions
    → Near-real-time (seconds of lag)

Why Redshift's Approach Is Simpler But Less Flexible

REDSHIFT HA PHILOSOPHY:
    → "We handle everything — you don't need to think about it"
    → ONE consistency model: serializable (always fully consistent)
    → ONE failover strategy: automatic, managed by AWS
    → ONE read path: always from primary
    → ZERO configuration needed
    
    ADVANTAGE: Simple. Reliable. No expertise needed.
    DISADVANTAGE: No flexibility. No read scaling from standby.
                  No tunable consistency. No geographic routing.

MONGODB HA PHILOSOPHY:
    → "We give you KNOBS — you tune for your specific needs"
    → TUNABLE consistency: per operation (read concern, write concern)
    → CONFIGURABLE failover: election priority, voting members
    → MULTIPLE read paths: primary, secondary, nearest
    → GEOGRAPHIC routing: read from closest server
    
    ADVANTAGE: Flexible. Read scaling. Geographic distribution. 
               Per-operation consistency tuning.
    DISADVANTAGE: More complex. Requires understanding. 
                  Wrong settings = data loss or slow writes.

The Oplog — How Replication Works

OPLOG (Operations Log) — MongoDB's Replication Mechanism:

    WHAT IT IS:
        → A capped collection on the PRIMARY: local.oplog.rs
        → Contains EVERY write operation (insert, update, delete)
        → Secondaries TAIL the oplog and replay operations
        → Like a database transaction log / WAL
        → Similar concept: Redshift's commit log, MySQL's binlog

    HOW IT WORKS:
    
    1. Application writes to PRIMARY:
       db.orders.insertOne({ customer: "alice", amount: 299.99 })
    
    2. PRIMARY writes to its data files AND appends to oplog:
       {
           ts: Timestamp(1717200000, 1),     // operation timestamp
           op: "i",                           // i=insert, u=update, d=delete
           ns: "ecommerce.orders",            // namespace
           o: {                               // the document
               _id: ObjectId("..."),
               customer: "alice",
               amount: 299.99
           }
       }
    
    3. SECONDARY 1 reads this oplog entry:
       → Connects to PRIMARY's oplog
       → Reads new entries since last checkpoint
       → Applies the insert to its own data files
       → Updates its own oplog (for cascading replication)
    
    4. SECONDARY 2 does the same (independently)
    
    TIMING:
        → Oplog replication: typically < 1 second lag
        → Under load: can be seconds to minutes
        → Network issues: lag can grow significantly
        → REPLICATION LAG = time between PRIMARY write and SECONDARY apply

    │                                                               │

    REDSHIFT COMPARISON:
        → Redshift Multi-AZ: synchronous block-level replication
        → Every write simultaneously written to primary AND standby
        → No "lag" — standby is always identical
        → But: standby is NOT readable
        
        → MongoDB: asynchronous oplog replication (by default)
        → Primary writes first, secondaries catch up
        → Small lag (usually < 1 second)
        → But: secondaries ARE readable (with caveats)
        → CAN be made synchronous with w:"majority" (Project 24)

Elections — Automatic Failover

ELECTION PROCESS — What Happens When Primary Fails:

    NORMAL STATE:
        Primary (AZ-1): ✅ serving reads + writes
        Secondary 1 (AZ-2): ✅ replicating, available for reads
        Secondary 2 (AZ-3): ✅ replicating, available for reads

    FAILURE:
        Primary (AZ-1): ❌ CRASHES (disk failure, network, OOM)
        Secondary 1 (AZ-2): ✅ detects primary is gone
        Secondary 2 (AZ-3): ✅ detects primary is gone

    DETECTION (1-5 seconds):
        → Each member sends heartbeats every 2 seconds
        → If no heartbeat from primary for 10 seconds → primary declared down
        → electionTimeoutMillis: 10,000ms (default, configurable)
        → In practice: detection in 5-10 seconds

    ELECTION (1-3 seconds):
        → Eligible secondaries "volunteer" to become primary
        → Election requires MAJORITY of voting members
        → 3-member set: majority = 2 votes needed
        → Secondary 1 votes for itself + Secondary 2 votes for Secondary 1
        → 2 votes ≥ majority → Secondary 1 becomes new PRIMARY
        → Total failover time: ~5-12 seconds

    POST-ELECTION:
        Primary (AZ-1): ❌ still down
        NEW Primary (AZ-2): ✅ now serving reads + writes
        Secondary (AZ-3): ✅ now replicating from NEW primary

    ┌────────────────────────────────────────────────────────┐
    │                                                         │
    │  TIMELINE:                                              │
    │                                                         │
    │  T+0s:   Primary crashes                                │
    │  T+2s:   Secondaries miss first heartbeat               │
    │  T+4s:   Secondaries miss second heartbeat              │
    │  T+10s:  Election timeout reached → election starts     │
    │  T+11s:  Votes exchanged                                │
    │  T+12s:  New primary elected                            │
    │  T+12s:  Driver detects new topology → routes to new    │
    │          primary                                        │
    │                                                         │
    │  TOTAL DOWNTIME: ~12 seconds                            │
    │  (vs Redshift Multi-AZ: ~30-60 seconds)                 │
    │                                                         │
    │  DURING ELECTION:                                       │
    │  → Writes: REJECTED (no primary to accept them)         │
    │  → Reads from secondary: STILL WORK                     │
    │  → Application must handle "no primary" errors           │
    │    (retry logic in driver — usually automatic)           │
    │                                                         │
    └────────────────────────────────────────────────────────┘

    WHY ODD NUMBER OF MEMBERS:
        → 3 members: majority = 2 → survives 1 failure
        → 5 members: majority = 3 → survives 2 failures
        → EVEN numbers cause split-brain risk:
          → 4 members: majority = 3
          → If network splits 2+2 → neither side has majority
          → NO election possible → entire set read-only
        → ODD numbers: one side always has majority

    ELECTION PRIORITY:
        → Each member has a priority (0-1000, default 1)
        → Higher priority = more likely to become primary
        → Priority 0 = can NEVER become primary (passive member)
        → Use case: analytics secondary in remote datacenter
          → Should replicate data but never become primary
          → Set priority: 0

    REDSHIFT COMPARISON:
        → Redshift: AWS manages failover internally
        → No election, no voting, no priority configuration
        → Simpler but no control
        → MongoDB: YOU control who can become primary
        → More power but more responsibility

🧩 HOW TO APPROACH IT IN MONGODB

Read Preferences — Where to Read From

READ PREFERENCES — THE POWER REDSHIFT DOESN'T GIVE YOU:

    MongoDB lets you choose WHERE each read goes:

    1. primary (DEFAULT)
        → All reads go to PRIMARY
        → Always see latest data (no stale reads)
        → Same behavior as Redshift (all reads from primary cluster)
        → Use for: payment processing, inventory checks, anything needing latest data

    2. primaryPreferred
        → Read from PRIMARY if available
        → Fall back to SECONDARY if primary is down
        → Use for: most application reads (best of both worlds)

    3. secondary
        → Always read from a SECONDARY
        → PRIMARY only handles writes → reduced load
        → Data may be slightly stale (replication lag)
        → Use for: analytics queries, reports, dashboards

    4. secondaryPreferred
        → Read from SECONDARY if available
        → Fall back to PRIMARY if no secondaries available
        → Use for: read-heavy workloads where slight staleness is OK

    5. nearest
        → Read from the member with LOWEST network latency
        → Could be primary or any secondary
        → Use for: global deployments (read from closest server)

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                  │
    │  MAPPING TO APPLICATION NEEDS:                                   │
    │                                                                  │
    │  Payment processing → primary (must see latest inventory)        │
    │  Product page load  → primaryPreferred (fresh data, failover OK) │
    │  Analytics dashboard → secondary (offload from primary)          │
    │  Global user profile → nearest (minimize latency)                │
    │  Report generation   → secondaryPreferred (never impact primary) │

    │  → Every query goes to the primary cluster                       │

    │  → Read scaling requires separate consumer clusters              │
    │  → No per-query routing control                                 │
    │                                                                  │

Write Concern — How Durable Is Each Write?

WRITE CONCERN — CHOOSING DURABILITY PER OPERATION:

    When you write to MongoDB, you choose HOW SURE you need to be
    that the write survived:

    w: 1 (DEFAULT — acknowledged by primary only)
        → Primary writes to memory + journal → returns success
        → If primary crashes BEFORE replicating → DATA LOST
        → Fastest option
        → Use for: non-critical data (page views, logs, temp data)

    w: "majority" (acknowledged by majority of members)
        → Primary writes → replicates to at least 1 secondary → returns success
        → If primary crashes → data exists on at least 1 secondary
        → ZERO data loss on failover (new primary has the data)
        → Slower (~2-5ms extra latency)
        → Use for: orders, payments, user data, anything important

    w: 0 (fire and forget — no acknowledgment)
        → Primary accepts write, returns immediately
        → No confirmation of write success
        → Fastest possible, but no guarantees
        → Use for: metrics, telemetry, non-critical logging

    j: true (journaled — written to disk journal before ack)
        → Combines with w: ensures journal write before response
        → Protects against: server crash (not just process crash)
        → Use for: { w: "majority", j: true } = maximum durability

class ECommerceService:
    """
    Demonstrates correct write concern and read preference
    for each operation in an e-commerce order flow.
    
    Redshift equivalent:
        → All operations use same consistency (serializable)
        → No per-operation tuning possible
        → Simpler but less optimized
    
    MongoDB:
        → Critical operations: w:"majority", read from primary
        → Non-critical operations: w:1, read from secondary
        → Each operation tuned for its specific needs
    """
    
    def __init__(self, db):
        # Critical path: orders, payments, inventory
        self.orders = db.get_collection(

        self.events = db.get_collection(
            "events_flow",
            write_concern=WriteConcern(w=1),
            read_preference=ReadPreference.SECONDARY_PREFERRED
        )
        
        self.products_browse = db.get_collection(
            "inventory_flow",
            read_preference=ReadPreference.SECONDARY_PREFERRED
        )
        
        self.analytics = db.get_collection(
            "daily_stats_flow",

        Browse products — READ from secondary (stale OK).

            → Product catalog doesn't change every second
            → Stale by 1-2 seconds is invisible to user
            → Offloads primary for write-heavy order processing

            SELECT * FROM products WHERE category = :cat;
            → Always from primary (no choice)

    def check_inventory(self, product_id):
        """
        Check inventory — READ from primary (MUST be latest).
        
        WHY primary:
            → About to commit a purchase
            → MUST see latest stock count
            → Stale read → overselling → customer anger
        
        Redshift equivalent:
            SELECT stock FROM inventory WHERE product_id = :pid;
            → Always from primary (no choice — but correct here)

        return product
    
    def place_order(self, customer_id, product_id, quantity=1):
        """
        Place order — WRITE with majority concern (MUST be durable).
        
        WHY w:"majority", j:true:
            → Customer is paying money
            → Order MUST survive server failure
            → If primary crashes after ack → order exists on secondaries
            → Zero data loss guaranteed
        
        Redshift equivalent:
            BEGIN;
            INSERT INTO orders (...) VALUES (...);
            UPDATE inventory SET stock = stock - :qty WHERE product_id = :pid;
            COMMIT;
            → Always fully durable (no choice — but correct here)
        """
        # Step 1: Check inventory (primary read — latest data)
        product = self.check_inventory(product_id)
        if not product or product["stock"] < quantity:
            print(f"  ❌ Insufficient stock for {product_id}")
            return None
        
        # Step 2: Decrement inventory (majority write)

        inv_result = self.inventory.update_one(
            {"_id": product_id, "stock": {"$gte": quantity}},
            {"$inc": {"stock": -quantity}}

        if inv_result.modified_count == 0:
            print(f"  ❌ Race condition: stock depleted between check and update")
            return None
        
        print(f"\n  📉 Decrement Inventory (writeConcern: majority)")
        print(f"     Product: {product_id}, Qty: -{quantity}")
        print(f"     Time: {inv_ms:.2f}ms")
        
        # Step 3: Create order (majority write + journal)
        order = {
            "customer_id": customer_id,
            "product_id": product_id,
            "product_name": product["name"],
            "quantity": quantity,
            "unit_price": product["price"],
            "total_amount": round(product["price"] * quantity, 2),
            "status": "confirmed",
            "created_at": datetime.now()
        }
        
        start = time.time()
        order_result = self.orders.insert_one(order)
        order_ms = (time.time() - start) * 1000
        
        print(f"\n  📝 Create Order (writeConcern: majority, j:true)")
        print(f"     Order ID: {order_result.inserted_id}")
        print(f"     Amount: ${order['total_amount']:.2f}")
        print(f"     Time: {order_ms:.2f}ms")
        print(f"     Durability: survives ANY single server failure")
        
        # Step 4: Log analytics event (w:1 — non-critical)
        start = time.time()
        self.events.insert_one({
            "type": "purchase",
            "customer_id": customer_id,
            "product_id": product_id,
            "amount": order["total_amount"],
            "created_at": datetime.now()
        })
        event_ms = (time.time() - start) * 1000
        
        print(f"\n  📊 Log Event (writeConcern: w:1)")
        print(f"     Time: {event_ms:.2f}ms")
        print(f"     Durability: primary only (if lost, analytics slightly off — acceptable)")
        
        # Step 5: Update computed stats (w:1 — from Project 20)

        print(f"     Time: {stats_ms:.2f}ms")
        
        total_ms = inv_ms + order_ms + event_ms + stats_ms

        print(f"     Total time: {total_ms:.2f}ms")
        print(f"     Critical writes (majority): {inv_ms + order_ms:.2f}ms")
        print(f"     Non-critical writes (w:1): {event_ms + stats_ms:.2f}ms")
        
        return order_result.inserted_id

def resilient_write(collection, document, max_retries=3):
    """
    Write with automatic retry on failover.
    
    WHAT HAPPENS DURING FAILOVER:
        1. Primary crashes
        2. Application's next write → ConnectionError or AutoReconnect
        3. pymongo driver detects topology change
        4. Election completes (~5-12 seconds)
        5. Driver discovers new primary
        6. Retry succeeds on new primary
    
    WHY retryWrites=True HANDLES MOST CASES:
        → pymongo automatically retries once on network errors
        → Handles: elections, brief network blips, connection resets
        → Does NOT handle: sustained outages, application-level retries
    
    Redshift equivalent:
        → Connection drops during Redshift failover (~30-60s)
        → Application must reconnect
        → Most JDBC drivers auto-reconnect
        → Retry logic similar but less nuanced
    """
    
    for attempt in range(max_retries):
        try:
            result = collection.insert_one(document)
            if attempt > 0:
                print(f"    ✅ Write succeeded on retry {attempt + 1}")
            return result
        
        except Exception as e:
            error_name = type(e).__name__
            
            if attempt < max_retries - 1:
                # Determine wait time based on error type
                if "AutoReconnect" in error_name or "ConnectionFailure" in error_name:
                    wait_time = 2 ** attempt  # exponential backoff: 1s, 2s, 4s
                    print(f"    ⚠️ Attempt {attempt + 1} failed ({error_name})")
                    print(f"       Waiting {wait_time}s before retry (election may be in progress)")
                    time.sleep(wait_time)
                elif "NotPrimaryError" in error_name or "NotWritablePrimary" in str(e):
                    wait_time = 5  # election in progress, wait longer
                    print(f"    ⚠️ No primary available — election in progress")
                    print(f"       Waiting {wait_time}s for new primary")
                    time.sleep(wait_time)
                else:
                    raise  # unknown error — don't retry
            else:
                print(f"    ❌ All {max_retries} attempts failed")
                raise


def resilient_read(collection, query, max_retries=3):
    """
    Read with automatic retry and fallback read preference.
    
    Strategy:
        1. Try primary read
        2. If primary down → fall back to secondary read
        3. Accept stale data during failover (better than error)
    """
    
    for attempt in range(max_retries):
        try:
            if attempt == 0:
                # First attempt: try configured read preference
                result = collection.find_one(query)
            else:
                # Retry: fall back to secondaryPreferred
                fallback_coll = collection.with_options(
                    read_preference=ReadPreference.SECONDARY_PREFERRED
                )
                result = fallback_coll.find_one(query)
                print(f"    ⚠️ Read from secondary (primary unavailable)")
            
            return result
        
        except Exception as e:
            if attempt < max_retries - 1:
                wait_time = 2 ** attempt
                print(f"    ⚠️ Read attempt {attempt + 1} failed: {type(e).__name__}")
                print(f"       Retrying in {wait_time}s...")
                time.sleep(wait_time)
            else:
                raise

# -- → Automatic standby in another AZ
# -- → Automatic failover (~30-60 seconds)

# -- → Standby is NOT readable
# -- → No per-query consistency tuning
# --
# -- Option 2: Cross-region snapshot copy
# -- ALTER CLUSTER my-cluster MODIFY 
# --   SNAPSHOT_COPY_DESTINATION 'us-west-2';
# -- → Snapshots copied to another region
# -- → Manual restore for DR
# -- → RPO: up to 8 hours
# --
# -- Option 3: Data sharing (read scaling)
# -- CREATE DATASHARE my_share SET PUBLICACCESSIBLE TRUE;
# -- ALTER DATASHARE my_share ADD TABLE store.orders;
# -- → Consumer clusters can read shared data
# -- → Near-real-time (seconds of lag)
# -- → Like MongoDB secondary reads
# -- → But: separate cluster, separate cost
# --
# -- KEY DIFFERENCES:
# -- Redshift: ONE consistency model (serializable)
# --           ONE failover strategy (AWS-managed)
# --           ONE read path (primary only, unless data sharing)
# --           ZERO configuration needed
# --
# -- MongoDB: TUNABLE consistency (per operation)
# --          CONFIGURABLE failover (priority, votes, delays)
# --          MULTIPLE read paths (primary, secondary, nearest)
# --          GEOGRAPHIC routing (read from closest)
# --          MORE POWER but MORE COMPLEXITY
# ═══════════════════════════════════════════════════════

✅ WHAT SHOULD DO (Best Practices)

✅ USE w:"majority" FOR ALL BUSINESS-CRITICAL WRITES
    → Orders, payments, user accounts, inventory changes
    → The ~2-5ms extra latency is negligible vs data loss risk
    → Set as DEFAULT on your database connection
    → Override to w:1 only for explicitly non-critical operations

✅ USE readPreference: "secondaryPreferred" FOR READ-HEAVY COLLECTIONS
    → Product catalog, user profiles, search results
    → Offloads primary for write-heavy operations
    → Automatically falls back to primary if no secondaries

✅ USE retryWrites=True AND retryReads=True
    → pymongo default since 3.9 but verify in your connection string
    → Handles transient errors during elections automatically
    → No application code needed for basic retry

✅ ALWAYS DEPLOY ODD NUMBER OF VOTING MEMBERS
    → 3 members: survives 1 failure (majority = 2)
    → 5 members: survives 2 failures (majority = 3)
    → Even numbers: split-brain risk during network partition
    → Atlas: always 3 members by default

✅ SPREAD MEMBERS ACROSS AVAILABILITY ZONES
    → Primary: AZ-1, Secondary: AZ-2, Secondary: AZ-3
    → Survives entire AZ failure
    → Atlas: automatic multi-AZ deployment
    → Same principle as Redshift Multi-AZ

✅ MONITOR REPLICATION LAG
    → Alert if lag > 10 seconds (sustained)
    → Alert if lag > 60 seconds (critical)
    → Atlas: built-in replication lag charts
    → Self-managed: rs.status() in monitoring scripts

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT USE w:1 FOR FINANCIAL OPERATIONS
    REDSHIFT THINKING: "Every write is automatically durable"
    MONGODB TRAP: w:1 (default) only confirms primary write
    → Primary crashes → last writes LOST
    → For payments: ALWAYS { w: "majority", j: true }

❌ DO NOT READ FROM SECONDARY FOR CONSISTENCY-CRITICAL OPERATIONS
    → Inventory check before purchase → READ FROM PRIMARY
    → Secondary may be 1-2 seconds behind
    → Stale inventory read → overselling
    → Use readPreference: "primary" for anything involving decisions

❌ DO NOT DEPLOY EVEN NUMBER OF VOTING MEMBERS
    → 4 members: network split 2+2 → neither has majority → no primary → all writes fail
    → 2 members: 1 fails → no majority → no election → stuck
    → ALWAYS 3, 5, or 7 voting members (odd numbers)

❌ DO NOT CONNECT DIRECTLY TO A SPECIFIC MEMBER
    WRONG: MongoClient("mongodb://primary-host:27017/")
    RIGHT: MongoClient("mongodb://host1:27017,host2:27017,host3:27017/?replicaSet=rs0")
    → Direct connection: if that member goes down → application fails
    → Replica set connection: driver discovers all members, handles failover

❌ DO NOT IGNORE REPLICATION LAG FOR SECONDARY READS
    → If secondary is 30 seconds behind → dashboard shows 30-second-old data
    → If secondary is 5 minutes behind → something is wrong
    → Monitor lag and set maxStalenessSeconds in read preference
    → db.collection.find().readPref("secondary", [], { maxStalenessSeconds: 30 })
