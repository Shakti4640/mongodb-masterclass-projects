PROJECT 14: Embedding vs Referencing — The JOIN Decision

Problem: Design a Blog Platform with Posts, Authors, and Comments

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    You're building a blogging platform
    
    The data relationships:
    ├── AUTHORS write POSTS                    (one-to-many)
    ├── POSTS have COMMENTS                    (one-to-many, potentially unbounded)
    ├── Each AUTHOR has a PROFILE              (one-to-one)
    ├── POSTS have TAGS                        (many-to-many)
    └── COMMENTS are written by USERS          (many-to-one)
    
    Access patterns the application needs:
    ├── Display a blog post page (post + author name + recent comments)
    ├── Display author profile page (author info + list of their posts)
    ├── Display all posts with a specific tag
    ├── Add a new comment to a post
    ├── Update author bio (reflected everywhere)
    └── Analytics: most commented posts, most active authors
    
    In Redshift: you would normalize into 5+ tables and JOIN
    In MongoDB: you must CHOOSE — embed or reference — for EACH relationship
    
    THIS DECISION SHAPES EVERYTHING:
    → Query performance
    → Write performance
    → Data consistency
    → Document size
    → Application complexity
    
    Get it wrong → rewrite your entire data layer
    Get it right → MongoDB feels magical

🔴 REDSHIFT MINDSET — How You Would Solve This

Step 1: Normalize Everything (Redshift Default Thinking)

YOUR INSTINCT:
    → Identify entities: authors, posts, comments, tags
    → Create one table per entity
    → Use foreign keys to link them
    → JOIN at query time
    → Never duplicate data
    → Third Normal Form (3NF) or star schema
    
    THIS IS CORRECT FOR REDSHIFT
    → Columnar storage makes JOINs on large datasets efficient
    → Deduplication saves storage (columnar compression)
    → Updates hit ONE row in ONE table
    → BI tools expect normalized relational schemas

Step 2: The Redshift Schema Design

-- REDSHIFT: Fully normalized blog platform

-- ─── AUTHORS ─────────────────────────────────────────
CREATE TABLE blog.authors (
    author_id       BIGINT IDENTITY(1,1) PRIMARY KEY,
    username        VARCHAR(50) NOT NULL,
    email           VARCHAR(255) NOT NULL,
    bio             VARCHAR(2000),
    avatar_url      VARCHAR(500),
    joined_at       TIMESTAMP DEFAULT GETDATE()
)
DISTSTYLE ALL          -- small table → replicate to all nodes
SORTKEY(username);     -- for fast author lookups

-- ─── POSTS ───────────────────────────────────────────
CREATE TABLE blog.posts (
    post_id         BIGINT IDENTITY(1,1) PRIMARY KEY,
    author_id       BIGINT NOT NULL,       -- FK → authors
    title           VARCHAR(500) NOT NULL,
    body            VARCHAR(65535),         -- max VARCHAR in Redshift
    status          VARCHAR(20) DEFAULT 'draft',
    view_count      INTEGER DEFAULT 0,
    published_at    TIMESTAMP,
    created_at      TIMESTAMP DEFAULT GETDATE()
)
DISTSTYLE KEY
DISTKEY(author_id)             -- co-locate posts with same author
SORTKEY(published_at);         -- time-range queries

-- ─── COMMENTS ────────────────────────────────────────
CREATE TABLE blog.comments (
    comment_id      BIGINT IDENTITY(1,1) PRIMARY KEY,
    post_id         BIGINT NOT NULL,       -- FK → posts
    user_id         BIGINT NOT NULL,       -- FK → authors (commenter)
    body            VARCHAR(5000) NOT NULL,
    created_at      TIMESTAMP DEFAULT GETDATE()
)
DISTSTYLE KEY
DISTKEY(post_id)               -- co-locate comments with their post
SORTKEY(created_at);

-- ─── TAGS ────────────────────────────────────────────
CREATE TABLE blog.tags (
    tag_id          BIGINT IDENTITY(1,1) PRIMARY KEY,
    name            VARCHAR(100) NOT NULL UNIQUE
)
DISTSTYLE ALL;                 -- small reference table

-- ─── POST_TAGS (junction table for many-to-many) ────
CREATE TABLE blog.post_tags (
    post_id         BIGINT NOT NULL,
    tag_id          BIGINT NOT NULL,
    PRIMARY KEY (post_id, tag_id)
)
DISTSTYLE KEY
DISTKEY(post_id);

-- ═══ QUERY: Display a blog post page ════════════════
SELECT
    p.post_id,
    p.title,
    p.body,
    p.published_at,
    p.view_count,
    a.username      AS author_name,
    a.avatar_url    AS author_avatar,
    c.comment_id,
    c.body          AS comment_body,
    c.created_at    AS comment_date,
    ca.username     AS commenter_name
FROM blog.posts p
JOIN blog.authors a     ON p.author_id = a.author_id
LEFT JOIN blog.comments c   ON p.post_id = c.post_id
LEFT JOIN blog.authors ca   ON c.user_id = ca.author_id
WHERE p.post_id = 12345
ORDER BY c.created_at DESC;

-- RESULT: 5 tables involved, 3 JOINs, distributed across nodes
-- Redshift handles this efficiently because:
--   → authors is DISTSTYLE ALL (no data movement for JOIN)
--   → comments DISTKEY(post_id) co-locates with posts
--   → Columnar storage: only reads needed columns

Step 3: What Redshift Does Well Here

REDSHIFT STRENGTHS IN THIS DESIGN:
    ✅ Author updates their bio → ONE row updated → reflected everywhere
    ✅ Analytical queries across ALL posts → columnar scan is fast
    ✅ No data duplication → storage efficient
    ✅ Standard SQL → any BI tool connects immediately
    ✅ Adding new relationships → just add new junction table
    
REDSHIFT WEAKNESSES:
    ❌ Every blog post page view = 3+ JOINs across tables
    ❌ High-concurrency reads (1000 users viewing posts) → WLM queue saturation
    ❌ Single comment insert → write to comments table + possible sort key penalty
    ❌ Schema change → ALTER TABLE across multiple tables
    ❌ Latency: even simple post lookup = 50-500ms (MPP overhead)

🟢 MONGODB MINDSET — How This Same Problem Is Approached

Step 1: The Fundamental Question Redshift Never Asks

IN REDSHIFT:
    → You NEVER ask "should I embed or reference?"
    → Answer is ALWAYS: normalize into separate tables, JOIN at query time
    → This is the ONLY option
    
IN MONGODB:
    → For EVERY relationship you must ask:
      "Should this data be EMBEDDED inside the parent document
       or REFERENCED via ID and looked up separately?"
    
    → This is NOT a syntax question — it's an ARCHITECTURE decision
    → The answer depends on:
      1. How is this data ACCESSED? (together or independently?)
      2. How does this data GROW? (bounded or unbounded?)
      3. How often does this data CHANGE? (rarely or frequently?)
      4. What is the CARDINALITY? (one-to-few? one-to-millions?)
      5. Who OWNS this data? (parent or independent entity?)

Step 2: The Three Fundamental Approaches

APPROACH 1 — FULL EMBEDDING (Denormalization)
    → Child data stored INSIDE parent document
    → ONE query returns EVERYTHING
    → Like: Redshift materialized view with all JOINs pre-computed
    → Trade-off: data duplication, document size growth

APPROACH 2 — FULL REFERENCING (Normalization)  
    → Separate collections linked by IDs
    → MULTIPLE queries or $lookup to assemble
    → Like: Redshift normalized tables with JOINs
    → Trade-off: multiple round trips, $lookup overhead

APPROACH 3 — HYBRID (Partial Embedding + References)
    → Embed FREQUENTLY accessed subset
    → Reference RARELY accessed or independently managed data
    → Like: Redshift star schema (some denormalization, some JOINs)
    → THIS IS USUALLY THE RIGHT ANSWER

🧠 THE DECISION FRAMEWORK — When to Embed vs Reference

ASK THESE QUESTIONS FOR EACH RELATIONSHIP:

┌─────────────────────────────────────┬──────────────┬──────────────┐
│ QUESTION                             │ EMBED ✅      │ REFERENCE ✅  │
├─────────────────────────────────────┼──────────────┼──────────────┤
│ Is child data always read WITH       │              │              │
│ the parent?                          │ YES          │ NO           │
├─────────────────────────────────────┼──────────────┼──────────────┤
│ Does child data grow unboundedly?    │ NO           │ YES          │
│ (no cap on array size)               │ (bounded)    │ (unbounded)  │
├─────────────────────────────────────┼──────────────┼──────────────┤
│ Does child need to be queried        │              │              │
│ independently?                       │ NO           │ YES          │
├─────────────────────────────────────┼──────────────┼──────────────┤
│ How often does child data change?    │ RARELY       │ FREQUENTLY   │
├─────────────────────────────────────┼──────────────┼──────────────┤
│ Is the relationship "owned"?         │ YES          │ NO           │
│ (child meaningless without parent)   │ (owned)      │ (independent)│
├─────────────────────────────────────┼──────────────┼──────────────┤
│ Cardinality?                         │ one-to-few   │ one-to-many  │
│                                      │ one-to-dozen │ one-to-millions│
├─────────────────────────────────────┼──────────────┼──────────────┤
│ Total embedded size stays            │              │              │
│ under ~1MB comfortably?              │ YES          │ NO           │
│ (hard limit: 16MB per document)      │              │              │
├─────────────────────────────────────┼──────────────┼──────────────┤
│ Need atomic updates across           │              │              │
│ parent + child?                      │ YES (free)   │ NO (or use   │
│                                      │              │ transactions)│
└─────────────────────────────────────┴──────────────┴──────────────┘

MEMORY AID — The "GOUF" Rule:
    G — Growth:       bounded → embed, unbounded → reference
    O — Ownership:    owned → embed, independent → reference  
    U — Usage:        read together → embed, read separately → reference
    F — Frequency:    rarely updated → embed, frequently updated → reference

🏗️ APPLYING THE FRAMEWORK TO OUR BLOG PLATFORM

Relationship 1: Author ↔ Profile (One-to-One)

ANALYSIS:
    Growth:      Profile is fixed size (bio, avatar, social links)     → EMBED ✅
    Ownership:   Profile is meaningless without author                  → EMBED ✅
    Usage:       Profile always displayed with author                   → EMBED ✅
    Frequency:   Profile rarely changes                                 → EMBED ✅
    
    VERDICT: EMBED profile inside author document
    
    ┌──────────────────────────────────────────────────────┐
    │ REDSHIFT: 2 tables (authors + author_profiles)       │
    │           JOIN on every query                         │
    │                                                       │
    │ MONGODB:  1 document with profile nested inside       │
    │           Zero JOINs ever                             │
    └──────────────────────────────────────────────────────┘

Relationship 2: Post ↔ Author (Many-to-One)

ANALYSIS:
    Growth:      One author per post (fixed)                            → EMBED ✅
    Ownership:   Author exists independently of any post                → REFERENCE ✅
    Usage:       Author name shown on post, but full profile accessed   → HYBRID
                 separately on author page
    Frequency:   Author changes bio → must update everywhere if         → REFERENCE ✅
                 embedded
    
    VERDICT: HYBRID — embed SUBSET of author (name, avatar)
             + store author_id reference for full profile lookup
    
    ┌──────────────────────────────────────────────────────┐
    │ REDSHIFT: posts.author_id → JOIN authors             │
    │           Every post page = JOIN                      │
    │                                                       │
    │ MONGODB:  Embed { author_id, name, avatar } in post  │
    │           Post page = ZERO lookups for author info    │
    │           Author profile page = separate query        │
    │           Author changes name = update posts too      │
    │           (accepted trade-off for read performance)   │
    └──────────────────────────────────────────────────────┘

Relationship 3: Post ↔ Comments (One-to-Many, UNBOUNDED)

ANALYSIS:
    Growth:      A viral post could get 50,000+ comments               → REFERENCE ✅
    Ownership:   Comments somewhat tied to post but have own identity   → DEPENDS
    Usage:       Post page shows latest 10 comments, not all            → HYBRID
    Frequency:   New comments added constantly                          → REFERENCE ✅
    Size:        50,000 comments × 500 bytes = 25MB → exceeds 16MB     → REFERENCE ✅
    
    VERDICT: REFERENCE — separate comments collection
             + optionally embed latest 3-5 comments in post (subset pattern)
    
    ┌──────────────────────────────────────────────────────┐
    │ REDSHIFT: posts + comments tables, JOIN on post_id   │
    │           Same approach — normalized                  │
    │                                                       │
    │ MONGODB:  comments in separate collection             │
    │           + latest 3 comments embedded in post        │
    │           Post page = ONE read (embedded comments)    │
    │           "Load more" = query comments collection     │
    │           New comment = insert to comments +          │
    │                         update post's embedded array  │
    └──────────────────────────────────────────────────────┘

Relationship 4: Post ↔ Tags (Many-to-Many)

ANALYSIS:
    Growth:      A post has 1-10 tags (bounded, small)                  → EMBED ✅
    Ownership:   Tags on a post are "owned" by that post                → EMBED ✅
    Usage:       Tags always displayed with post                        → EMBED ✅
    Frequency:   Tags rarely change after publishing                    → EMBED ✅
    Size:        10 tags × 30 bytes = 300 bytes (trivial)              → EMBED ✅
    Query need:  "Find all posts with tag X" → needs index on array    → EMBED ✅
    
    VERDICT: EMBED tags as array of strings inside post document
    
    ┌──────────────────────────────────────────────────────┐
    │ REDSHIFT: tags table + post_tags junction table      │
    │           2 JOINs to get tags for a post             │
    │           3 JOINs to find posts by tag               │
    │                                                       │
    │ MONGODB:  tags: ["javascript", "mongodb", "tutorial"]│
    │           Array embedded directly in post             │
    │           Index on tags field → fast tag search       │
    │           NO junction table needed                    │
    │           NO JOINs needed                             │
    └──────────────────────────────────────────────────────┘
    
    THIS IS A MASSIVE SIMPLIFICATION:
    → Redshift: 3 tables + 2 JOINs for many-to-many
    → MongoDB:  1 array field + 1 multikey index

🔧 HOW TO APPROACH IT IN MONGODB

THE DESIGN PROCESS (rewire from Redshift thinking):

REDSHIFT DESIGN PROCESS:
    1. Identify entities → tables
    2. Identify relationships → foreign keys
    3. Normalize to 3NF
    4. Denormalize for performance (star schema)
    5. Choose DISTKEY/SORTKEY
    6. Done — schema is locked

MONGODB DESIGN PROCESS:
    1. Identify entities (same)
    2. List ALL access patterns (what queries will the app run?)
    3. For EACH relationship, apply the GOUF decision framework
    4. Design documents around the MOST FREQUENT access pattern
    5. Accept trade-offs for less frequent patterns
    6. Iterate as access patterns evolve (schema is flexible)

FOR OUR BLOG PLATFORM:
    Most frequent access pattern: "Display a single blog post page"
    → This drives the ENTIRE document design
    → Everything needed for post page should be ONE read
    → Less frequent patterns (author page, analytics) accept more work

Design B: Full Referencing (Maximum Flexibility, Multiple Queries)

// ═══════════════════════════════════════════════════════
// DESIGN B — FULL REFERENCING
// Separate collections for everything
// Exactly like Redshift normalized tables
// ═══════════════════════════════════════════════════════

// ─── AUTHORS COLLECTION ─────────────────────────────
// Equivalent to: blog.authors table in Redshift
{
    _id: ObjectId("auth001"),
    username: "sarah_dev",
    email: "sarah@example.com",
    bio: "Full-stack developer with 10 years experience...",
    avatar_url: "https://cdn.example.com/avatars/sarah.jpg",
    joined_at: ISODate("2022-01-15T00:00:00Z")
}

// ─── POSTS COLLECTION ───────────────────────────────
// Equivalent to: blog.posts table
// author_id is a REFERENCE (like a foreign key)
{
    _id: ObjectId("post001"),
    author_id: ObjectId("auth001"),     // ← REFERENCE (like FK)
    title: "Getting Started with MongoDB",
    body: "MongoDB is a document database that...",
    status: "published",
    published_at: ISODate("2024-06-15T10:00:00Z"),
    view_count: 15234,
    tag_ids: [                           // ← REFERENCES to tags
        ObjectId("tag001"),
        ObjectId("tag002")
    ]
}

// ─── COMMENTS COLLECTION ────────────────────────────
// Equivalent to: blog.comments table
{
    _id: ObjectId("comm001"),
    post_id: ObjectId("post001"),       // ← REFERENCE to post
    user_id: ObjectId("auth002"),       // ← REFERENCE to commenter
    body: "Great introduction! Very helpful.",
    created_at: ISODate("2024-06-15T12:30:00Z")
}

// ─── TAGS COLLECTION ────────────────────────────────
// Equivalent to: blog.tags table
{
    _id: ObjectId("tag001"),
    name: "mongodb",
    post_count: 42
}

// ─── QUERY: Display blog post page ──────────────────
// Requires MULTIPLE queries or $lookup pipeline
// (covered in Project 13)

db.posts.aggregate([
    // Step 1: Find the post
    { $match: { _id: ObjectId("post001") } },
    
    // Step 2: JOIN author (like Redshift JOIN blog.authors)
    { $lookup: {
        from: "authors",
        localField: "author_id",
        foreignField: "_id",
        as: "author"
    }},
    { $unwind: "$author" },
    
    // Step 3: JOIN comments (like Redshift LEFT JOIN blog.comments)
    { $lookup: {
        from: "comments",
        let: { pid: "$_id" },
        pipeline: [
            { $match: { $expr: { $eq: ["$post_id", "$$pid"] } } },
            { $sort: { created_at: -1 } },
            { $limit: 10 },
            // Sub-JOIN: get commenter info
            { $lookup: {
                from: "authors",
                localField: "user_id",
                foreignField: "_id",
                as: "commenter"
            }},
            { $unwind: "$commenter" },
            { $project: {
                body: 1,
                created_at: 1,
                "commenter.username": 1,
                "commenter.avatar_url": 1
            }}
        ],
        as: "comments"
    }},
    
    // Step 4: JOIN tags
    { $lookup: {
        from: "tags",
        localField: "tag_ids",
        foreignField: "_id",
        as: "tags"
    }},
    
    // Step 5: Shape output
    { $project: {
        title: 1,
        body: 1,
        published_at: 1,
        view_count: 1,
        "author.username": 1,
        "author.avatar_url": 1,
        comments: 1,
        "tags.name": 1
    }}
])

// ─── ANALYSIS ───────────────────────────────────────
// ✅ Author updates bio → ONE document in authors collection
// ✅ 50,000 comments → no document size problem
// ✅ "All comments by john_reader" → simple query on comments collection
// ✅ Each entity independently queryable
//
// ❌ Blog post page requires 4 $lookups (JOINs)
// ❌ $lookup is MUCH slower than Redshift JOIN
//    (not optimized with distribution keys / zone maps)
// ❌ Multiple network round trips if done as separate queries
// ❌ Complex pipeline code vs simple SQL JOIN
// ❌ You're basically using MongoDB like Redshift → losing the advantage

// VERDICT: ⚠️ WORKS BUT SUBOPTIMAL
// You're paying MongoDB prices for Redshift behavior

🔧 WHAT COULD BE DONE (MongoDB-Specific Optimizations)

OPTIMIZATION 1: Multikey Index on Tags Array
    db.posts.createIndex({ tags: 1, status: 1, published_at: -1 })
    → MongoDB automatically creates multikey index for arrays
    → Each array element gets its own index entry
    → { tags: "mongodb" } query uses index efficiently
    → Redshift has NO equivalent — needs junction table + JOIN

OPTIMIZATION 2: Compound Index for Comments Collection
    db.comments.createIndex({ post_id: 1, created_at: -1 })
    → Supports "load more comments" query perfectly
    → ESR principle from Project 10:
      Equality (post_id) → Sort (created_at desc)

OPTIMIZATION 3: Partial Index for Published Posts Only
    db.posts.createIndex(
        { slug: 1 },
        { partialFilterExpression: { status: "published" } }
    )
    → Only indexes published posts → smaller index
    → Draft posts not in index → saves memory
    → Redshift has NO equivalent (indexes entire sort key range)

OPTIMIZATION 4: Projection to Reduce Data Transfer
    // Blog listing page — don't transfer full body
    db.posts.find(
        { status: "published" },
        { title: 1, slug: 1, excerpt: 1, author: 1, tags: 1, 
          published_at: 1, comment_count: 1, view_count: 1 }
    )
    // body field excluded → much smaller document transferred
    // Like: SELECT title, excerpt, ... (not SELECT *)

✅ WHAT SHOULD DO (Best Practices vs Redshift Habits to Unlearn)

SHOULD DO:
    ✅ Design documents around the PRIMARY access pattern
       → "What does the blog post page need?" drives everything
       → Secondary patterns (author page, tag page) accept trade-offs
    
    ✅ Embed data that is READ TOGETHER
       → Author name + avatar on post = always read together → embed
       → Author full bio = only read on author page → reference
    
    ✅ Use SUBSET embedding for unbounded relationships
       → All comments = comments collection (unbounded)
       → Latest 3 comments = embedded in post (bounded)
    
    ✅ Accept data duplication as a FEATURE, not a bug
       → Author username duplicated in posts and comments
       → This is INTENTIONAL — eliminates JOINs for reads
       → Manage consistency via application logic or change streams
    
    ✅ Maintain computed fields for counts
       → comment_count: 50 → updated via $inc
       → Avoids: db.comments.countDocuments({ post_id: x }) on every read
       → Redshift equivalent: materialized view with COUNT
    
    ✅ Keep embedded arrays BOUNDED
       → recent_comments: $slice to 3
       → tags: rarely more than 10
       → Use $push with $slice to auto-limit

REDSHIFT HABITS TO UNLEARN:
    🔄 "Never duplicate data"
       → MongoDB: duplication is a TOOL for read performance
       → You're trading write complexity for read speed
    
    🔄 "Always normalize to 3NF"
       → MongoDB: normalize only when you MUST (unbounded, independent)
       → Default posture is embed, not normalize
    
    🔄 "JOINs are cheap"
       → MongoDB: $lookup is MUCH more expensive than Redshift JOIN
       → $lookup = nested loop join (no hash join, no merge join)
       → Redshift JOIN = optimized with distribution keys, zone maps
       → Avoid $lookup on hot paths — embed instead
    
    🔄 "One table per entity"
       → MongoDB: sometimes one collection serves multiple purposes
       → A post document contains author info, tags, and comments
       → This feels wrong from Redshift perspective — it's correct here
    
    🔄 "Schema = source of truth"
       → MongoDB: application code = source of truth for structure
       → The document shape is defined by what you insert
       → Use schema validation for guardrails, not DDL

❌ WHAT SHOULD NOT DO (Anti-Patterns)

ANTI-PATTERN 1: "Redshift in MongoDB" — Full Normalization
    ❌ Creating: authors, posts, comments, tags, post_tags collections
       Then $lookup for every single query
    WHY IT'S WRONG:
       → $lookup is expensive (no distribution optimization)
       → You're paying MongoDB prices for Redshift-style queries
       → Use Redshift if you want full normalization
    FIX: Embed what's read together

ANTI-PATTERN 2: "Massive Document" — Full Embedding
    ❌ Putting ALL comments inside the post document
    WHY IT'S WRONG:
       → 16MB document limit will be hit
       → Every comment read transfers entire post + all comments
       → Write amplification: updating one comment rewrites entire document
    FIX: Separate collection for unbounded data + subset embed

ANTI-PATTERN 3: "Foreign Key Thinking" — Storing Only IDs
    ❌ Post document: { author_id: "auth001", tag_ids: ["t1","t2"] }
       Then $lookup for author name on every post page view
    WHY IT'S WRONG:
       → Forces $lookup on the HOTTEST path (post page view)
       → In Redshift: DISTSTYLE ALL on authors makes this cheap
       → In MongoDB: $lookup is ALWAYS a separate collection scan
    FIX: Embed the SUBSET you need: { author: { username, avatar } }

ANTI-PATTERN 4: "No References Ever" — Embedding Everything
    ❌ Author's full profile (including email, social, settings)
       embedded in every post
    WHY IT'S WRONG:
       → Author changes bio → update hundreds of posts
       → Massive data duplication for rarely-needed fields
       → Email in every post = security concern
    FIX: Embed ONLY what the post page needs (username + avatar)

ANTI-PATTERN 5: "Array as Join Table" — Unbounded ID Arrays
    ❌ Author document: { post_ids: [id1, id2, ... id5000] }
       Storing ALL post IDs in author document
    WHY IT'S WRONG:
       → Array grows unboundedly as author writes more posts
       → Querying "all posts by author" should use posts collection index
       → Large arrays degrade update and read performance
    FIX: Query posts collection: db.posts.find({ author_id: x })
         NOT: db.authors.findOne({ _id: x }).post_ids → then find posts

ANTI-PATTERN 6: "Premature Optimization" — Over-Embedding
    ❌ Embedding 10 levels deep before you have real traffic data
    WHY IT'S WRONG:
       → You don't know actual access patterns yet
       → Over-embedding makes future changes harder
       → Under-embedding is easier to fix (add fields) than
         over-embedding (must split collections)
    FIX: Start slightly more normalized than you think you need
         → Add embedding as access patterns become clear

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Internals — Embedded vs Referenced Reads

EMBEDDED DOCUMENT READ:
    db.posts.findOne({ slug: "getting-started-with-mongodb" })
    
    WHAT HAPPENS INSIDE WIREDTIGER:
    1. Index lookup on slug index → find document location
    2. Read ONE page from B-tree containing the document
    3. Decompress page (snappy/zstd)
    4. Extract complete document (post + author + tags + comments)
    5. Return to client
    
    I/O OPERATIONS: 1 index seek + 1 data page read
    DISK PAGES: 1-2 (if document fits in one page)
    NETWORK: 1 round trip
    
    THE DOCUMENT IS STORED AS A CONTIGUOUS BSON BLOB:
    ┌─────────────────────────────────────────────┐
    │ WiredTiger Page                               │
    │ ┌─────────────────────────────────────────┐ │
    │ │ Document (complete BSON)                  │ │
    │ │ ├── title: "Getting Started..."          │ │
    │ │ ├── body: "MongoDB is a..."              │ │
    │ │ ├── author: { username, avatar }         │ │ ← EMBEDDED
    │ │ ├── tags: ["mongodb", "tutorial"]        │ │ ← EMBEDDED
    │ │ ├── recent_comments: [{...},{...},{...}] │ │ ← EMBEDDED
    │ │ └── view_count: 15234                    │ │
    │ └─────────────────────────────────────────┘ │
    │ ┌─────────────────────────────────────────┐ │
    │ │ Next Document...                         │ │
    │ └─────────────────────────────────────────┘ │
    └─────────────────────────────────────────────┘
    
    ALL related data is PHYSICALLY ADJACENT on disk
    → One seek reads everything
    → THIS is why embedding is fast


REFERENCED DOCUMENT READ ($lookup):
    db.posts.aggregate([
        { $match: { slug: "..." } },
        { $lookup: { from: "authors", ... } },
        { $lookup: { from: "comments", ... } }
    ])
    
    WHAT HAPPENS INSIDE:
    1. Index lookup on slug → find post document
    2. Read post page → get author_id
    3. Switch to authors collection → index lookup on _id
    4. Read author page → get author document
    5. Switch to comments collection → index lookup on post_id
    6. Read multiple comment pages → get comment documents
    7. Merge results in memory
    8. Return to client
    
    I/O OPERATIONS: 3+ index seeks + 3+ data page reads
    DISK PAGES: 5-10+ (documents in different locations)
    NETWORK: 1 round trip (aggregation pipeline runs server-side)
    
    DATA IS PHYSICALLY SCATTERED:
    ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
    │ Posts Collection   │  │ Authors Coll.     │  │ Comments Coll.    │
    │ ┌──────────────┐ │  │ ┌──────────────┐ │  │ ┌──────────────┐ │
    │ │ Post doc     │─│──│→│ Author doc   │ │  │ │ Comment 1    │ │
    │ │ (author_id)──│ │  │ │              │ │  │ │              │ │
    │ │ (no comments)│ │  │ └──────────────┘ │  │ ├──────────────┤ │
    │ └──────────────┘ │  │                   │  │ │ Comment 2    │ │
    │                   │  │                   │  │ │              │ │
    └──────────────────┘  └──────────────────┘  │ ├──────────────┤ │
                                                  │ │ Comment 3    │ │
    3 SEPARATE SEEKS                              │ └──────────────┘ │
    across 3 collections                          └──────────────────┘
    
    Related data is PHYSICALLY SCATTERED across different B-trees
    → Multiple seeks needed
    → THIS is why $lookup is slower than embedding

Why $lookup Is Slower Than Redshift JOIN

REDSHIFT JOIN INTERNALS:
    SELECT p.*, a.username FROM posts p JOIN authors a ON ...
    
    1. Authors table is DISTSTYLE ALL → already on every node
       → ZERO data movement needed for JOIN
    2. Columnar storage → reads ONLY username column from authors
       → Reads ONLY needed columns from posts
    3. Zone maps → skips blocks that don't match WHERE clause
    4. Hash join or merge join algorithm → O(n) or O(n log n)
    5. Parallel execution across all compute slices
    
    RESULT: JOINs are OPTIMIZED at the storage and execution level

MONGODB $lookup INTERNALS:
    db.posts.aggregate([{ $lookup: { from: "authors", ... } }])
    
    1. For EACH post document matched:
       → Execute a SEPARATE query against authors collection
       → This is a NESTED LOOP join (the least efficient join type)
    2. No distribution optimization
       → If sharded: $lookup may need to go to DIFFERENT shards
       → Network hop for each lookup
    3. Document storage → must read ENTIRE author document
       → Can't read just username column (not columnar)
    4. No parallel execution across shards for $lookup
    5. Results assembled in memory on the mongos/primary
    
    RESULT: $lookup is a LAST RESORT in MongoDB, not a first choice

    ┌─────────────────────────────────────────────────┐
    │ THIS IS THE #1 REASON TO EMBED IN MONGODB       │
    │                                                   │
    │ Redshift JOINs are CHEAP → normalize everything   │
    │ MongoDB $lookups are EXPENSIVE → embed when you can│
    └─────────────────────────────────────────────────┘

Redshift Internals — Why Normalization Works There

REDSHIFT JOIN: posts ⟕ authors (LEFT JOIN)

STEP 1: Distribution
    → authors is DISTSTYLE ALL → replicated on every slice
    → posts DISTKEY(author_id) → each slice has subset of posts
    → JOIN is LOCAL on each slice → ZERO network data movement

    │ Slice 0  │  │ Slice 1  │  │ Slice 2  │  │ Slice 3  │
    │          │  │          │  │          │  │          │
    │ Posts    │  │ Posts    │  │ Posts    │  │ Posts    │
    │ (auth    │  │ (auth    │  │ (auth    │  │ (auth    │
    │  1-250)  │  │  251-500)│  │  501-750)│  │  751-1000│
    │          │  │          │  │          │  │          │
    │ Authors  │  │ Authors  │  │ Authors  │  │ Authors  │
    │ (ALL)    │  │ (ALL)    │  │ (ALL)    │  │ (ALL)    │
    │          │  │          │  │          │  │          │
    │ JOIN     │  │ JOIN     │  │ JOIN     │  │ JOIN     │
    │ locally  │  │ locally  │  │ locally  │  │ locally  │
    └─────────┘  └─────────┘  └─────────┘  └─────────┘
    
    Each slice JOINs its posts with its local copy of authors
    → PARALLEL execution
    → ZERO data movement
    → THIS is why Redshift JOINs are cheap

STEP 2: Columnar Reading
    → SELECT a.username → reads ONLY the username column block
    → Not the entire author row (email, bio, social, etc.)
    → Compression on single column → very compact reads
    
    EXAMPLE: 1000 authors, each with 500-byte username
    → Redshift reads: 1000 × 500 bytes = 500KB (just usernames)
    → MongoDB reads: 1000 × 2KB avg document = 2MB (entire documents)
    → 4x more data read by MongoDB for same information

STEP 3: Zone Maps
    → WHERE post_id = 12345
    → Zone map on post_id column says:
      "Block 47 has min=12300, max=12400 → MIGHT contain 12345"
      "Block 48 has min=12401, max=12500 → SKIP"
    → Only reads relevant blocks
    → MongoDB equivalent: B-tree index (similar but different mechanism)

STEP 4: Result Assembly
    → Leader node collects results from all slices
    → Merges and returns to client
    → Parallel assembly → fast even for complex JOINs

Write Path Comparison — Embedded vs Referenced vs Redshift

SCENARIO: New comment added to a blog post

─── PATH A: MONGODB EMBEDDED (all comments in post document) ───

    1. Find post document in B-tree index
    2. Read ENTIRE post document into memory (could be large)
    3. Append new comment to comments array (in-memory)
    4. If document grew beyond its allocated page space:
       → Document MUST BE MOVED to a new location
       → Old location marked as free space
       → All indexes pointing to this document UPDATED
       → THIS IS CALLED "DOCUMENT MIGRATION"
       → Extremely expensive for frequently growing documents
    5. Write modified document back to disk
    6. Update journal
    
    COST: HIGH for growing documents
    → Every comment makes the document bigger
    → Eventually triggers document migration
    → THIS is why unbounded arrays are dangerous
    
    WiredTiger uses "padding factor" to leave extra space
    → Anticipates growth
    → But eventually exceeded by large arrays


─── PATH B: MONGODB REFERENCED (comments in separate collection) ───

    1. Insert new document into comments collection
       → Find empty page or allocate new page
       → Write compact comment document (~500 bytes)
       → Update _id index for comments collection
       → Update post_id index for comments collection
    2. Update post document: $inc comment_count + $push recent_comments
       → $push with $slice: bounded array, predictable size
       → Document size stays STABLE (recent_comments always 3)
       → No document migration
    3. Write journal entries for both operations
    
    COST: LOW — two small operations
    → Comment document is small and fixed-size
    → Post document doesn't grow (bounded embedded array)
    → No document migration ever
    
    THIS IS WHY HYBRID DESIGN WINS FOR WRITES TOO


─── PATH C: REDSHIFT NORMALIZED ───

    1. INSERT INTO comments → row sent to compute slice (DISTKEY post_id)
    2. Row split into columns → each column value appended to its block
    3. Zone maps updated for new block ranges
    4. Sort key (created_at) → row appended to unsorted region
       → VACUUM needed later to re-sort
    5. Commit log entry written
    
    COST: MEDIUM
    → Single table write is straightforward
    → BUT: unsorted region grows → query performance degrades
    → Need periodic VACUUM to re-sort (Redshift-specific overhead)
    → No equivalent problem in MongoDB (B-tree always sorted)


WRITE PERFORMANCE COMPARISON:
    ┌──────────────────────────┬──────────┬────────────┬──────────┐
    │ OPERATION                 │ EMBED    │ REFERENCE  │ REDSHIFT │

    REFERENCE approach: write time is CONSTANT regardless of comment count
    EMBEDDED approach: write time DEGRADES as array grows
    REDSHIFT approach: write time constant but needs maintenance (VACUUM)

Data Consistency — The Duplication Trade-Off Internals

SCENARIO: Author "sarah_dev" changes username to "sarah_engineer"

─── REDSHIFT (Normalized) ───
    UPDATE blog.authors SET username = 'sarah_engineer' WHERE author_id = 1;
    → ONE row updated
    → Every JOIN with authors table automatically reflects new name
    → ZERO additional work
    → Consistency: GUARANTEED by normalization

─── MONGODB (Hybrid — username duplicated) ───
    Places where "sarah_dev" exists:
    1. authors collection → { username: "sarah_dev" }
    2. Every post by sarah → { author: { username: "sarah_dev" } }
    3. Every comment by sarah → { username: "sarah_dev" }
    
    Must update ALL locations:
    
    // Step 1: Update source of truth
    db.authors.updateOne(
        { _id: ObjectId("auth001") },
        { $set: { username: "sarah_engineer" } }
    )
    
    // Step 2: Update all her posts
    db.posts.updateMany(
        { author_id: ObjectId("auth001") },
        { $set: { "author.username": "sarah_engineer" } }
    )
    
    // Step 3: Update all her comments
    db.comments.updateMany(
        { user_id: ObjectId("auth001") },
        { $set: { username: "sarah_engineer" } }
    )
    
    // Step 4: Update embedded comments inside other people's posts
    db.posts.updateMany(
        { "recent_comments.user_id": ObjectId("auth001") },    // ← WRONG FIELD
        // Hmm... we didn't store user_id in recent_comments
        // We stored username directly
        // This is HARD to update without user_id reference
    )
    
    → MULTIPLE operations needed
    → Brief inconsistency window between updates
    → Some locations might be missed
    → Consistency: APPLICATION MUST MANAGE
    
    MITIGATION STRATEGIES:
    1. Use Change Streams (Project 25) to auto-propagate changes
    2. Accept eventual consistency (username changes are rare)
    3. Store user_id in embedded subsets for update targeting
    4. Use multi-document transactions (Project 28) for atomicity

    │ THE FUNDAMENTAL TRADE-OFF MADE EXPLICIT:                 │
    │                                                          │
    │ EMBEDDING gives you:                                     │
    │   + Fast reads (no JOIN)                                 │
    │   + Atomic single-document operations                    │
    │   - Data duplication                                     │
    │   - Complex updates when duplicated data changes         │
    │   - Risk of inconsistency                                │
    │                                                          │
    │ REFERENCING gives you:                                    │
    │   + Single source of truth (like Redshift)               │

    │   - Slow reads ($lookup needed)                          │
    │   - Multiple round trips or complex pipelines            │
    │   - No free atomicity across collections                 │
    │                                                          │
    │ HYBRID gives you:                                        │
    │   + Fast reads for primary pattern                       │
    │   + Manageable duplication (only subset embedded)         │
    │   + Reference available for secondary patterns           │
    │   - Some update complexity (bounded)                     │
    │   - Must track what's duplicated where                   │
    │                                                          │
    │ REDSHIFT gives you:                                      │
    │   + Zero duplication                                     │
    │   + Zero consistency concerns                            │
    │   + Simple schema management                             │

PITFALL 1: "I'll Just $lookup Everything" (Redshift Developer Default)

    FIX:

        → Embed data needed by those patterns

PITFALL 2: "Embed Everything for Maximum Speed"
    SYMPTOM:

        → Write operations getting slower over time
        → Memory pressure increasing (large documents in cache)

        → Returns ENTIRE post documents just to get comments

    FIX:

        → Plus subset embedded in posts (for post page display)

    SYMPTOM:
        → Author shows as "sarah_dev" on some posts
        → Author shows as "sarah_engineer" on newer posts

    ROOT CAUSE:
        → Updated authors collection but forgot to update
          embedded author info in posts and comments
    FIX:

          Option A: Application-level cascade on author update
          Option B: Change stream listener auto-propagates
          Option C: Background job periodically reconciles
        → ALWAYS store the reference ID alongside embedded data
          → { author_id: x, username: "..." }
          → author_id enables targeted updates

PITFALL 5: "Document Size Creep" (Invisible Until It's Too Late)
    SYMPTOM:
        → Application works for 6 months
        → Suddenly: insert failures, slow reads
        → Investigation: average document size grew from 2KB to 200KB
    ROOT CAUSE:
        → Embedded arrays grew slowly (10 items/day for 6 months = 1800 items)
        → Nobody monitored document sizes
    FIX:
        → Monitor: db.collection.stats().avgObjSize
        → Set up alerts when avg size exceeds threshold
        → Use $slice on all $push operations
        → Periodically audit: 
          db.posts.aggregate([
              { $project: { size: { $bsonSize: "$$ROOT" } } },
              { $sort: { size: -1 } },
              { $limit: 10 }
          ])

PITFALL 6: "I Need Transactions for Every Write Now"
    SYMPTOM:
        → Hybrid design requires updating 2 collections per comment
        → Developer wraps every comment in a multi-document transaction
        → Throughput drops 60%
    ROOT CAUSE:
        → Treating MongoDB transactions like Redshift transactions
        → MongoDB transactions have significant overhead (lock escalation)
    FIX:
        → Ask: "What happens if step 2 fails?"
        → If comment inserted but post.recent_comments not updated:
          → Post shows slightly stale comments → acceptable
          → Background job can reconcile
        → Use transactions ONLY for financial/critical operations
        → For most hybrid writes: eventual consistency is fine

📈 SCALABILITY CONSIDERATIONS

HOW EMBEDDING VS REFERENCING AFFECTS SCALING:

─── EMBEDDED DESIGN AT SCALE ───

    SHARDING:
    → Post with embedded data = ONE document on ONE shard
    → Reading a post = ONE shard contacted (targeted query)
    → No cross-shard operations for single post read
    → EXCELLENT for read scalability
    
    CACHE:
    → Large documents consume more WiredTiger cache
    → Fewer documents fit in RAM
    → Working set may exceed cache → disk reads → latency spikes
    → WATCH: db.serverStatus().wiredTiger.cache.
              "pages read into cache" should be low
    
    WRITE SCALING:
    → Document updates on one shard only
    → But large document updates = more I/O per write
    → Document migration (growth) creates write amplification
    
    REPLICATION:
    → Larger documents = larger oplog entries
    → Replication lag may increase with large document updates
    → Secondary reads may be more stale

─── REFERENCED DESIGN AT SCALE ───

    SHARDING:
    → Posts on one shard, comments on another
    → $lookup across shards = VERY expensive (scatter-gather)
    → Must shard both collections on compatible keys
      → posts: shard key includes _id
      → comments: shard key includes post_id
      → Comments for one post co-located on same shard
    
    CACHE:
    → Small documents = more documents fit in RAM
    → Better cache hit ratio
    → Working set more manageable
    
    WRITE SCALING:
    → Small document writes = fast, predictable
    → No document migration concerns
    → Each collection scales independently
    
    REPLICATION:
    → Smaller oplog entries
    → Replication stays current more easily

def add_comment(post_id, user_id, username, avatar_url, body):
    """
    Add a comment using hybrid pattern:
    1. Insert full comment into comments collection
    2. Update post: increment count + maintain recent_comments (max 3)
    
    In Redshift this would be:
        INSERT INTO comments (...) VALUES (...);
    Just one operation. But every READ would need JOINs.
    """
    now = datetime.now()
    comment_id = ObjectId()
    
    comment_doc = {
        "_id": comment_id,
        "post_id": post_id,
        "user_id": user_id,
        "username": username,
        "avatar_url": avatar_url,
        "body": body,
        "created_at": now,
        "likes": 0,
        "replies": []
    }
    
    comment_subset = {
        "comment_id": comment_id,
        "username": username,
        "avatar_url": avatar_url,
        "body": body,
        "created_at": now
    }
    
    # Operation 1: Insert into comments collection
    db.comments.insert_one(comment_doc)
    
    # Operation 2: Update post — $inc count + $push with $slice
    db.posts.update_one(
        {"_id": post_id},
        {
            "$inc": {"comment_count": 1},
            "$push": {
                "recent_comments": {
                    "$each": [comment_subset],
                    "$sort": {"created_at": -1},
                    "$slice": 3
                }
            },
            "$set": {"updated_at": now}
        }
    )
    
    return comment_id
