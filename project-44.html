PROJECT 44: MongoDB + Kafka Integration

Problem: MongoDB Operational Data Must Flow to Redshift, Elasticsearch, and ML Pipelines Simultaneously

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your fintech platform now has a mature event-sourced MongoDB system
    (Project 43) and multiple downstream consumers need the data
    
    CURRENT ARCHITECTURE:
        → MongoDB Atlas Global Cluster (Project 41)
        → Event sourcing with change streams (Project 43)
        → Real-time analytics on MongoDB (Project 42)
        → Redshift for historical analytics (Project 34)
    
    THE PROBLEM — MULTIPLE CONSUMERS, ONE SOURCE:
        → Redshift: needs ALL account events for historical analytics
        → Elasticsearch: needs user profiles + transactions for search
        → ML Pipeline: needs transaction features for fraud model training
        → Compliance Engine: needs specific event types for regulatory reporting
        → Notification Service: needs account state changes for user alerts
        → Data Lake (S3): needs raw events for archival and ad-hoc analysis
    
    CURRENT APPROACH (Project 43 — direct change streams):
        → One change stream per consumer → 6 separate change streams
        → Each stream independently reads from MongoDB oplog
        → Each consumer must manage its own resume token
        → If one consumer fails: only that consumer falls behind
        → If MongoDB primary steps down: ALL consumers reconnect
    
    THE SCALING PROBLEM:
        → 6 change stream consumers × 50,000 events/minute = heavy oplog load
        → Each consumer processes at different speeds
        → Slow consumer blocks itself (no buffering)
        → Adding a 7th consumer means another oplog reader
        → No replay capability for new consumers (missed historical events)
        → No dead letter queue for failed processing
    
    THE SOLUTION — KAFKA AS EVENT BUS:
        → ONE change stream → Kafka → MANY consumers
        → Kafka buffers events (configurable retention: hours/days/forever)
        → Each consumer reads at its own pace (consumer groups)
        → New consumer can replay from beginning (Kafka retention)
        → Dead letter topic for failed events
        → Schema registry for event format governance
        → Exactly-once semantics with proper configuration
    
    IN REDSHIFT:
        → Kinesis Data Streams → Kinesis Firehose → S3 → COPY
        → Or: Kinesis → Redshift Streaming Ingestion (newer)
        → Or: MSK (Managed Kafka) → Lambda → Redshift
        → Multiple AWS services to orchestrate
    
    IN MONGODB:
        → MongoDB Kafka Connector (official, production-grade)
        → Change stream → Kafka topic (one connector)
        → Kafka topic → any consumer (standard Kafka ecosystem)
        → Single integration point, unlimited downstream consumers

-- REDSHIFT: Receiving data via Kinesis Streaming Ingestion

-- Step 1: Create external schema for Kinesis
CREATE EXTERNAL SCHEMA kinesis_schema
FROM KINESIS
IAM_ROLE 'arn:aws:iam::123456789012:role/redshift-kinesis-role'
REGION 'us-east-1';

-- Step 2: Create materialized view that reads from stream
CREATE MATERIALIZED VIEW mv_account_events
AUTO REFRESH YES AS
SELECT
    approximate_arrival_timestamp AS ingestion_time,
    partition_key AS account_id,
    JSON_EXTRACT_PATH_TEXT(
        from_varbyte(kinesis_data, 'utf-8'), 'event_type'
    ) AS event_type,
    JSON_EXTRACT_PATH_TEXT(
        from_varbyte(kinesis_data, 'utf-8'), 'event_data'
    ) AS event_data_json,
    CAST(JSON_EXTRACT_PATH_TEXT(
        from_varbyte(kinesis_data, 'utf-8'), 'resulting_state', 'balance'
    ) AS DECIMAL(18,2)) AS balance,
    JSON_EXTRACT_PATH_TEXT(
        from_varbyte(kinesis_data, 'utf-8'), 'country'
    ) AS country,
    JSON_EXTRACT_PATH_TEXT(
        from_varbyte(kinesis_data, 'utf-8'), 'metadata', 'correlation_id'
    ) AS correlation_id,
    refresh_time AS last_refresh
FROM kinesis_schema.account_events_stream
WHERE is_valid = true
  AND LEN(kinesis_data) > 0;

-- Step 3: Query the materialized view
SELECT event_type, COUNT(*), SUM(balance)
FROM mv_account_events
WHERE ingestion_time >= DATEADD(hour, -1, GETDATE())
GROUP BY event_type;

-- LIMITATIONS:
-- 1. AUTO REFRESH has minimum interval (depends on data volume)
-- 2. Materialized view refresh is not instant
-- 3. Complex JSON parsing in SQL (verbose, error-prone)
-- 4. No schema validation on incoming data
-- 5. Kinesis as intermediary adds latency + cost
-- 6. Must manage Kinesis separately from Redshift

-- ALTERNATIVE: Traditional COPY from S3
-- Kinesis Firehose → S3 → COPY command
-- Higher latency but more reliable for large volumes

-- Step 1: Firehose delivers to S3
-- s3://fintech-events/year=2024/month=06/day=01/hour=14/

-- Step 2: COPY into Redshift
COPY account_events
FROM 's3://fintech-events/year=2024/month=06/day=01/'
IAM_ROLE 'arn:aws:iam::123456789012:role/redshift-s3-role'
FORMAT AS JSON 'auto'
TIMEFORMAT 'auto'
REGION 'us-east-1';

🧩 HOW TO APPROACH IT IN MONGODB

KAFKA INTEGRATION DESIGN FRAMEWORK:

STEP 1 — CHOOSE WHAT TO CAPTURE
    → Which collections need CDC?
    → Which operation types? (insert only? insert + update + delete?)
    → Which fields to include/exclude?
    → Full document or just changed fields?

STEP 2 — DESIGN KAFKA TOPICS
    → Topic naming: mongo.<database>.<collection>
    → Partitioning key: stream_id / account_id / country
    → Partition count: based on consumer parallelism
    → Retention: based on replay requirements

STEP 3 — CONFIGURE SOURCE CONNECTOR
    → MongoDB Kafka Source Connector (Confluent-certified)
    → Change stream pipeline for filtering
    → Output format: JSON / Avro / Protobuf

STEP 4 — CONFIGURE SINK CONNECTORS (per destination)
    → Redshift: Kafka → S3 → COPY (or direct Redshift connector)
    → Elasticsearch: Kafka Elasticsearch Sink Connector
    → S3: Kafka S3 Sink Connector
    → MongoDB (another cluster): MongoDB Kafka Sink Connector

STEP 5 — MONITORING AND ERROR HANDLING
    → Connector status monitoring
    → Consumer lag monitoring
    → Dead letter topic for failed messages
    → Alert on processing delays

// Watches MongoDB change streams → publishes to Kafka topics

// ─── CONNECTOR CONFIGURATION: Events Collection ─────
// File: mongo-source-events.json
// Deploy via Kafka Connect REST API

{
    "name": "mongo-source-events",
    "config": {
        // ─── CONNECTOR CLASS ─────────────────────────
        "connector.class": "com.mongodb.kafka.connect.MongoSourceConnector",
        "tasks.max": "1",
        
        // ─── MONGODB CONNECTION ──────────────────────
        "connection.uri": "mongodb+srv://kafka-user:password@fintech-global.xxxxx.mongodb.net/",
        "database": "fintech",
        "collection": "events",
        
        // ─── KAFKA TOPIC ─────────────────────────────
        "topic.prefix": "mongo",
        // Produces to topic: mongo.fintech.events
        "topic.separator": ".",
        
        // ─── CHANGE STREAM PIPELINE ──────────────────
        // Filter: only capture insert operations
        // (events are append-only per Project 43)
        "pipeline": "[{\"$match\": {\"operationType\": \"insert\"}}]",
        
        // ─── OUTPUT FORMAT ───────────────────────────
        "output.format.key": "json",
        "output.format.value": "json",
        
        // Full document in the message value
        "publish.full.document.only": "true",
        
        // Use stream_id as Kafka message key
        // → All events for same account go to same partition
        // → Guarantees ordering per account within partition
        "output.json.formatter": "com.mongodb.kafka.connect.source.json.formatter.SimplifiedJson",
        "key.converter": "org.apache.kafka.connect.storage.StringConverter",
        "key.converter.schemas.enable": "false",
        "value.converter": "org.apache.kafka.connect.json.JsonConverter",
        "value.converter.schemas.enable": "false",
        
        // ─── PARTITIONING ────────────────────────────
        // Route messages by stream_id for ordering guarantee
        "topic.creation.default.partitions": "12",
        "topic.creation.default.replication.factor": "3",
        
        // ─── OFFSET MANAGEMENT ───────────────────────
        // Store offsets in Kafka (default)
        // Resume from last position on restart
        "offset.partition.name": "mongo-source-events",
        
        // ─── ERROR HANDLING ──────────────────────────
        "errors.tolerance": "all",
        "errors.log.enable": "true",
        "errors.log.include.messages": "true",
        "errors.deadletterqueue.topic.name": "mongo.fintech.events.dlq",
        "errors.deadletterqueue.topic.replication.factor": "3",
        
        // ─── HEARTBEAT ──────────────────────────────
        // Send heartbeat to detect connector is alive
        "heartbeat.interval.ms": "10000",
        "heartbeat.topic.name": "__mongodb_heartbeats"
    }
}

def mongodb_to_kafka_producer(collection_name, topic_name,
                               duration_seconds=60,
                               pipeline=None):
    """
    Python-based CDC: MongoDB Change Stream → Kafka Producer
    Use when Kafka Connect is not available
    
    For production: use the official MongoDB Kafka Source Connector
    This is the manual/custom alternative
    
    Redshift equivalent:
        -- Custom Lambda reading DynamoDB Streams → Kinesis
        -- Or: AWS DMS task (MongoDB source → Kinesis target)
        -- Both require significant configuration
    """
    
    print(f"\n{'=' * 70}")
    print(f"CDC PRODUCER: {collection_name} → {topic_name}")
    print(f"{'=' * 70}")
    
    # ─── Kafka Producer ──────────────────────────────
    producer = Producer({
        "bootstrap.servers": KAFKA_BOOTSTRAP,
        "client.id": f"mongo-cdc-{collection_name}",
        "acks": "all",                     # Wait for all replicas
        "enable.idempotence": True,         # Exactly-once semantics
        "compression.type": "lz4",
        "linger.ms": 5,                    # Small batching for low latency
        "batch.size": 16384,
        "retries": 3,
        "retry.backoff.ms": 100
    })
    
    # ─── Resume token management ─────────────────────
    resume_doc = mongo_db.cdc_offsets.find_one(
        {"_id": f"cdc-{collection_name}"}
    )
    
    watch_options = {"full_document": "updateLookup"}
    if resume_doc and resume_doc.get("resume_token"):
        watch_options["resume_after"] = resume_doc["resume_token"]
        print(f"  Resuming from saved position")
    else:
        print(f"  Starting from current position")
    
    if pipeline is None:
        pipeline = [{"$match": {"operationType": {"$in": [
            "insert", "update", "replace", "delete"
        ]}}}]
    
    # ─── Delivery callback ───────────────────────────
    delivered = {"count": 0, "errors": 0}
    
    def delivery_callback(err, msg):
        if err:
            delivered["errors"] += 1
            print(f"  ❌ Delivery failed: {err}")
        else:
            delivered["count"] += 1
    
    # ─── Watch and produce ───────────────────────────
    collection = mongo_db[collection_name]
    produced = 0
    start_time = time.time()
    
    try:
        with collection.watch(pipeline, **watch_options) as stream:
            while time.time() - start_time < duration_seconds:
                change = stream.try_next()
                
                if change is None:
                    producer.poll(0.1)
                    continue
                
                # ─── Build Kafka message ─────────────
                doc = change.get("fullDocument")
                operation = change.get("operationType")
                document_key = change.get("documentKey", {})
                
                # Message key: use document _id for partitioning
                # All changes to same document → same partition → ordered
                msg_key = str(document_key.get("_id", "unknown"))
                
                # Message value: full change event
                msg_value = {
                    "operation": operation,
                    "database": change.get("ns", {}).get("db"),
                    "collection": change.get("ns", {}).get("coll"),
                    "document_key": str(document_key.get("_id")),
                    "timestamp": datetime.now().isoformat(),
                    "cluster_time": str(change.get("clusterTime", "")),
                    "full_document": _serialize_document(doc) if doc else None,
                    "update_description": change.get("updateDescription")
                }
                
                # ─── Produce to Kafka ─────────────────
                producer.produce(
                    topic=topic_name,
                    key=msg_key.encode("utf-8"),
                    value=json.dumps(msg_value, default=str).encode("utf-8"),
                    callback=delivery_callback,
                    headers={
                        "source": "mongodb-cdc",
                        "collection": collection_name,
                        "operation": operation
                    }
                )
                
                produced += 1
                
                # Flush periodically
                if produced % 100 == 0:
                    producer.flush(timeout=5)
                    
                    # Save resume token
                    mongo_db.cdc_offsets.update_one(
                        {"_id": f"cdc-{collection_name}"},
                        {"$set": {
                            "resume_token": change["_id"],
                            "last_produced": datetime.now(),
                            "messages_produced": produced
                        }},
                        upsert=True
                    )
                
                if produced % 500 == 0:
                    print(f"  Produced: {produced} | "
                          f"Delivered: {delivered['count']} | "
                          f"Errors: {delivered['errors']}")
    
    except KeyboardInterrupt:
        print("\n  Producer stopped by user")
    
    finally:
        producer.flush(timeout=30)
        
        elapsed = round(time.time() - start_time, 1)
        print(f"\n  {'=' * 50}")
        print(f"  Producer ran: {elapsed}s")
        print(f"  Messages produced: {produced}")
        print(f"  Messages delivered: {delivered['count']}")
        print(f"  Delivery errors: {delivered['errors']}")
        print(f"  {'=' * 50}")
    
    return produced


def _serialize_document(doc):
    """Convert MongoDB document to JSON-serializable dict"""
    if doc is None:
        return None
    
    result = {}
    for key, value in doc.items():
        if isinstance(value, Decimal128):
            result[key] = str(value)
        elif isinstance(value, datetime):
            result[key] = value.isoformat()
        elif isinstance(value, dict):
            result[key] = _serialize_document(value)
        elif isinstance(value, list):
            result[key] = [
                _serialize_document(v) if isinstance(v, dict)
                else str(v) if isinstance(v, (Decimal128, datetime))
                else v
                for v in value
            ]
        elif hasattr(value, '__str__') and not isinstance(value, (str, int, float, bool)):
            result[key] = str(value)
        else:
            result[key] = value
    
    return result

    finally:
        consumer.close()
        elapsed = round(time.time() - start_time, 1)

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

1. CHANGE STREAM PRE-FILTERING IN SOURCE CONNECTOR
   → Filter at MongoDB level BEFORE sending to Kafka
   → Reduces Kafka throughput and storage
   
   "pipeline": "[
       {\"$match\": {
           \"operationType\": \"insert\",
           \"fullDocument.country\": {\"$in\": [\"US\",\"DE\",\"JP\"]}
       }}
   ]"
   
   → Only US, DE, JP events flow to Kafka
   → Other countries: processed locally in MongoDB only
   → Redshift equivalent: Kinesis filter Lambda (separate service)

2. AVRO SERIALIZATION WITH SCHEMA REGISTRY
   → JSON messages: human-readable but large
   → Avro: binary, compact, schema-enforced
   → 3-5x smaller messages → lower Kafka storage + network cost
   
   "value.converter": "io.confluent.connect.avro.AvroConverter",
   "value.converter.schema.registry.url": "http://schema-registry:8081",
   "value.converter.schemas.enable": "true"
   
   → Schema evolution: add fields without breaking consumers
   → Backward compatible: old consumers read new messages
   → Forward compatible: new consumers read old messages

3. TOPIC-PER-EVENT-TYPE ROUTING
   → Instead of one topic for all events:
   → Route by event_type to specialized topics
   
   "topic.mapper": "com.mongodb.kafka.connect.source.topic.mapping.DefaultTopicMapper",
   "topic.namespace.map": "{
       \"fintech.events\": \"mongo.fintech.events.${event_type}\"
   }"
   
   → Topics: mongo.fintech.events.DepositMade
   →         mongo.fintech.events.TransferSent
   →         mongo.fintech.events.AccountFrozen
   → Consumers subscribe only to relevant event types
   → Compliance: only subscribe to TransferSent + AccountFrozen

4. EXACTLY-ONCE WITH KAFKA TRANSACTIONS
   → Source connector: enable.idempotence + transactional.id
   → Sink connector: read_committed isolation
   → End-to-end exactly-once from MongoDB to destination
   
   Source:
   "producer.override.enable.idempotence": "true",
   "producer.override.transactional.id": "mongo-source-events-txn"
   
   Sink:
   "consumer.override.isolation.level": "read_committed"

5. COMPACTED TOPICS FOR CURRENT STATE
   → Events topic: "delete" policy (time-based retention)
   → Users/accounts topic: "compact" policy (keep latest per key)
   → Consumers always see latest user profile
   → No need to replay history for current state
   
   → Redshift equivalent: materialized view AUTO REFRESH
   → But: Kafka compaction is continuous, not scheduled

SHOULD DO:

       → Every connector should have a DLQ topic
       → Failed messages → DLQ for investigation
       → Don't lose data on processing errors
       → Monitor DLQ topic for alerts

    ✅ MONITOR CONSUMER LAG
       → Lag = produced offset - consumed offset per partition
       → Alert when lag exceeds threshold (e.g., 10,000 messages)
       → Lag growing = consumer too slow or stuck
       → Tools: Kafka Manager, Burrow, Confluent Control Center

    ✅ USE SCHEMA REGISTRY FOR PRODUCTION
       → Validates every message against registered schema
       → Prevents bad data from entering pipeline
       → Schema evolution rules prevent breaking changes
       → Consumers can trust message format

    ✅ COMMIT OFFSETS AFTER SUCCESSFUL PROCESSING
       → enable.auto.commit: false for critical consumers
       → Process message → write to destination → commit offset
       → If crash between process and commit: reprocess (at-least-once)
       → With idempotent writes: effectively exactly-once

❌ WHAT SHOULD NOT DO

DO NOT:

    ❌ CREATE ONE CHANGE STREAM PER CONSUMER (without Kafka)
       → 6 consumers = 6 change streams = 6x oplog read load
       → MongoDB oplog is a shared resource
       → Kafka: ONE change stream → Kafka → 6 consumers
       → Kafka absorbs the fan-out, not MongoDB

    ❌ USE KAFKA AS THE SOURCE OF TRUTH
       → Kafka is a TRANSPORT layer, not a database
       → MongoDB events collection = source of truth (Project 43)
       → Kafka = distribution mechanism
       → If Kafka loses data: replay from MongoDB change stream
       → If MongoDB loses data: Kafka retention helps temporarily

    ❌ PUT TRANSFORMATION LOGIC IN THE SOURCE CONNECTOR
       → Source connector should capture raw change events
       → Transformation happens in consumers or Kafka Streams
       → Keeps source simple and reliable
       → Different consumers need different transformations

    ❌ USE ONE GIANT TOPIC FOR EVERYTHING
       → mongo.fintech.everything → hard to manage
       → Consumers must filter irrelevant messages (waste)
       → Better: one topic per collection or per event category
       → Consumer subscribes only to relevant topics

    ❌ IGNORE CONSUMER LAG MONITORING
       → Silent lag buildup = data freshness degradation
       → Redshift getting events 2 hours late → stale dashboards
       → ML pipeline lagging → fraud model training on old data
       → Set up alerts: lag > 5,000 → warning, lag > 50,000 → critical

    ❌ SKIP DEAD LETTER QUEUES
       → Poison message (malformed JSON, unexpected schema)
       → Without DLQ: consumer crashes repeatedly on same message
       → With DLQ: bad message moves aside, consumer continues
       → Investigate DLQ messages manually

    ❌ USE JSON WITHOUT SCHEMA REGISTRY IN PRODUCTION
       → Schema drift: MongoDB document gains new field
       → Downstream consumer breaks on unexpected field
       → Schema registry: validates format, manages evolution
       → JSON in dev, Avro + Schema Registry in production

    ❌ ASSUME KAFKA ORDERING = MONGODB ORDERING
       → MongoDB change stream: ordered by oplog timestamp (global)
       → Kafka: ordered WITHIN partition only
       → If events for same account go to different partitions:
         ordering lost!
       → ALWAYS partition by entity key (stream_id / account_id)

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 44:

PITFALL 1: "I'll use AWS DMS for MongoDB CDC"
    ❌ DMS has LIMITED MongoDB CDC support
       → Document mode only (not real change stream)

       → No pipeline filtering
       → No resume token management

    WHY: Built ON change streams (native MongoDB CDC)

         Automatic resume token management
         Confluent-certified, production-tested

PITFALL 2: "Kafka is just like Kinesis — same mental model"
    ❌ Treating Kafka topics like Kinesis streams
    ✅ Understand fundamental differences
    KEY DIFFERENCES:
        → Retention: Kafka unlimited vs Kinesis 7 days max
        → Consumer groups: Kafka native vs Kinesis manual
        → Replay: Kafka from any offset vs Kinesis from trim horizon
        → Throughput: Kafka millions/sec vs Kinesis 1MB/sec/shard
        → Schema: Kafka has Schema Registry vs Kinesis none
        → Exactly-once: Kafka native vs Kinesis at-least-once

PITFALL 3: "I'll produce messages synchronously one at a time"
    ❌ producer.produce(); producer.flush() per message
    ✅ Batch produces, flush periodically
    WHY: Synchronous produce: ~1ms per message = 1000 msg/sec max
         Async batch produce: ~0.01ms per message = 100,000 msg/sec
         Use linger.ms + batch.size for automatic batching
         Flush every N messages or every N seconds

PITFALL 4: "Consumer lag doesn't matter — data gets there eventually"
    ❌ Ignoring consumer lag metrics
    ✅ Alert on lag thresholds per consumer group
    WHY: Lag = data freshness degradation
         Redshift consumer lagging 2 hours → dashboards show stale data
         Compliance consumer lagging → regulatory reporting delayed
         ML consumer lagging → fraud model training on old data
         Each consumer has different acceptable lag threshold

PITFALL 5: "I'll commit offsets before processing is complete"
    ❌ consumer.commit() → then process message
    ✅ Process message → verify success → THEN consumer.commit()
    WHY: Crash after commit but before processing → message LOST
         Commit-after-process: crash means reprocess (safe)
         With idempotent destination writes: no duplicates
         This is the fundamental at-least-once → effectively-once pattern

PITFALL 6: "One connector handles everything"
    ❌ Single Kafka Connect instance for all connectors
    ✅ Distributed Kafka Connect cluster with resource isolation
    WHY: Source connector failure shouldn't affect sink connectors
         Heavy sink connector shouldn't starve source connector
         Use separate Connect worker groups for source vs sink
         Monitor each connector independently

PITFALL 7: "I don't need the DLQ — errors won't happen"
    ❌ errors.tolerance: "none" (fail on first error)
    ✅ errors.tolerance: "all" + DLQ topic configured
    WHY: Production data is messy — unexpected formats, encoding issues

THROUGHPUT BENCHMARKS:

    MongoDB → Kafka (Source Connector):
        → Sustained: 30,000-80,000 events/sec (single connector)
        → Burst: 100,000+ events/sec
        → Bottleneck: usually MongoDB oplog read speed
    
    Kafka → S3 (S3 Sink Connector):
        → Sustained: 50,000-200,000 events/sec
        → Depends on: file format, compression, flush settings
        → Parquet + snappy: best compression + query performance
    
    S3 → Redshift (COPY):
        → Sustained: 100,000-500,000 rows/sec (depends on row size)
        → Optimized with: manifest files, multiple files, parallel load
    
    End-to-end latency:
        MongoDB write → Kafka topic: 100-500ms
        Kafka topic → S3 file: 60-300 seconds (flush interval)
        S3 file → Redshift table: 5-60 seconds (COPY execution)
        Total: ~2-6 minutes (MongoDB → Redshift)
        
    Compare with direct change stream (Project 34):
        MongoDB write → Redshift: custom ETL → 15-30 minutes
        
    Compare with MongoDB real-time (Project 42):
