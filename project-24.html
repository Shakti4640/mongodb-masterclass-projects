PROJECT 24: Read/Write Concern — Consistency Guarantees

Problem: Financial Transaction Ledger Where Some Reads Must Be Absolutely Current and Others Can Tolerate Staleness

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    You are building a financial services platform with three distinct access patterns:

    PATTERN A — Account Balance Check (customer-facing)
        → Customer checks their balance on mobile app
        → Must see their OWN most recent transaction
        → Cannot show stale balance AFTER customer just made a deposit
        → But microsecond-perfect consistency not required across ALL accounts

    PATTERN B — Fraud Detection Engine (internal, real-time)
        → Scans recent transactions for anomalies
        → Slight staleness is acceptable (1-2 seconds behind)
        → But needs HIGH THROUGHPUT (thousands of reads/second)
        → Spreading reads across secondaries is essential

    PATTERN C — End-of-Day Reconciliation (regulatory report)
        → Must see EXACTLY the committed state of ALL accounts
        → Zero tolerance for dirty reads, phantom reads, stale data
        → Runs once per day — latency is acceptable
        → Equivalent to "serializable isolation" in Redshift

    IN REDSHIFT: You have NO CHOICE — every query gets serializable isolation
    IN MONGODB: You CHOOSE the consistency level PER OPERATION

    The question: How do you configure MongoDB to give each pattern
    EXACTLY the consistency guarantee it needs — no more, no less?

🔴 REDSHIFT MINDSET — How You Would Solve This

Redshift's Consistency Model: One Size Fits All

IN REDSHIFT — THERE IS NO DECISION TO MAKE:

    Every single query — whether it's:
        → A simple SELECT balance FROM accounts WHERE id = 123
        → A complex 15-table JOIN for reconciliation
        → A quick dashboard refresh
    
    Gets the SAME consistency guarantee:
        → SERIALIZABLE ISOLATION (strongest possible)
        → Snapshot-based: query sees data as of query start time
        → No dirty reads — ever
        → No non-repeatable reads — ever
        → No phantom reads — ever
        → All committed data visible, no uncommitted data visible

REDSHIFT'S TRANSACTION MODEL:
    → BEGIN; ... COMMIT; (standard SQL transactions)
    → Default: each statement is auto-committed
    → Explicit transactions: BEGIN → statements → COMMIT/ROLLBACK
    → Isolation level: SERIALIZABLE (only option)
    → No READ UNCOMMITTED, no READ COMMITTED, no REPEATABLE READ
    → You CANNOT choose a weaker isolation level for better performance

    -- This is the ONLY way to query in Redshift
    SELECT balance FROM accounts WHERE account_id = 'ACC-001';
    -- Always sees committed, consistent data
    -- No configuration needed or possible

    -- Explicit transaction
    BEGIN;
    UPDATE accounts SET balance = balance - 100 WHERE account_id = 'ACC-001';
    UPDATE accounts SET balance = balance + 100 WHERE account_id = 'ACC-002';
    COMMIT;
    -- Either BOTH succeed or NEITHER succeeds
    -- Standard ACID — no tuning knobs

WHY REDSHIFT CAN DO THIS SIMPLY:
    → Single leader node coordinates ALL queries
    → No replicas serving reads (unlike MongoDB secondaries)
    → No distributed consensus needed (MPP but centrally coordinated)
    → Write path: leader → compute nodes (synchronous within cluster)
    → Read path: leader distributes query → nodes return results → leader merges
    → EVERYTHING goes through the leader — single source of truth

THE TRADE-OFF REDSHIFT MAKES:
    → ALWAYS correct (strong consistency)
    → But LIMITED concurrency (WLM queue slots)
    → But HIGHER latency per query (must ensure consistency)
    → But NO ability to trade consistency for performance
    → For analytical workloads: this is FINE (queries take seconds anyway)
    → For operational workloads: this is a BOTTLENECK

What Redshift Developers Never Think About

CONCEPTS THAT DON'T EXIST IN YOUR REDSHIFT BRAIN:

    1. "How stale can this read be?"
       → In Redshift: never stale. Not a question.
       → In MongoDB: fundamental question for every operation.

    2. "How many nodes must confirm a write before I consider it done?"
       → In Redshift: the cluster handles this. Not your problem.
       → In MongoDB: you choose. Per write. Per operation.

    3. "Should this read go to the primary or a replica?"
       → In Redshift: no replicas serve queries (leader coordinates all).
       → In MongoDB: you already learned read preference in Project 23.
         Now we go deeper — WHAT DATA does each node serve?

    4. "Can I sacrifice consistency for lower latency?"
       → In Redshift: no. Not possible.
       → In MongoDB: yes. Explicitly. With fine-grained control.

    5. "What happens if I read from a node that hasn't received the latest write?"
       → In Redshift: impossible scenario.
       → In MongoDB: common scenario. Read concern controls what you see.

🟢 MONGODB MINDSET — How This Same Problem Is Approached

The Fundamental Insight: Tunable Consistency

MONGODB'S PHILOSOPHY:
    → Different operations have DIFFERENT consistency needs
    → Forcing serializable isolation on EVERY operation wastes resources
    → A mobile app balance check and a regulatory audit
      DO NOT need the same guarantees
    → MongoDB gives you TWO INDEPENDENT KNOBS:

    ┌─────────────────────────────────────────────────────────────┐
    │                                                              │
    │  KNOB 1: WRITE CONCERN                                       │
    │  "How many nodes must acknowledge a write                    │
    │   before the driver tells your application 'success'?"       │
    │                                                              │
    │  KNOB 2: READ CONCERN                                        │
    │  "What version of the data am I willing to see?              │
    │   Only majority-committed? Or whatever is locally available?" │
    │                                                              │
    │  KNOB 3: READ PREFERENCE (covered in Project 23)             │
    │  "WHICH node should I read from?"                            │
    │                                                              │
    │  THREE KNOBS — independent — combined = your consistency     │
    │                                                              │
    └─────────────────────────────────────────────────────────────┘

    Redshift: ONE fixed dial welded to "MAXIMUM"
    MongoDB:  THREE independent dials you tune per operation

📊 WRITE CONCERN — Deep Dive

What Write Concern Controls

WRITE CONCERN ANSWERS:
    "When my application calls insertOne() or updateOne(),
     at what point does the driver return 'acknowledged: true'?"

    REMEMBER FROM PROJECT 23:
    ┌─────────────────────────────────────────────┐
    │  PRIMARY ──oplog──→ SECONDARY 1             │
    │     │                                        │
    │     └──oplog──→ SECONDARY 2                  │
    │                                              │
    │  Application writes to PRIMARY               │
    │  Secondaries replicate asynchronously         │
    └─────────────────────────────────────────────┘

    THE QUESTION: How many of these nodes must confirm
                  before your app gets "success"?

Write Concern Levels — From Least to Most Durable

┌────────────────┬───────────────────────────────────────────────────────────┐
│ WRITE CONCERN   │ BEHAVIOR                                                  │
├────────────────┼───────────────────────────────────────────────────────────┤
│                 │                                                           │
│ w: 0            │ FIRE AND FORGET                                           │
│ "Unacknowledged"│ → Driver sends write, does NOT wait for ANY response      │
│                 │ → Application gets "success" immediately                  │
│                 │ → You don't even know if the write REACHED the server     │
│                 │ → Network errors are the ONLY thing reported              │
│                 │ → FASTEST possible write                                  │
│                 │ → ZERO durability guarantee                               │
│                 │                                                           │
│                 │ USE WHEN: Logging, metrics, telemetry                     │
│                 │           Losing occasional writes is acceptable          │
│                 │ NEVER FOR: Financial data, user data, anything important  │
│                 │                                                           │
│                 │ Redshift equivalent: NOTHING — Redshift never allows this │
│                 │                                                           │
├────────────────┼───────────────────────────────────────────────────────────┤
│                 │                                                           │
│ w: 1            │ PRIMARY ACKNOWLEDGED (DEFAULT)                            │
│ "Acknowledged"  │ → Driver waits until PRIMARY writes to memory             │
│                 │ → Application gets "success" when primary confirms        │
│                 │ → Secondaries may NOT have the write yet                  │
│                 │ → If primary crashes BEFORE replicating: WRITE IS LOST    │
│                 │                                                           │
│                 │ TIMELINE:                                                 │
│                 │   App → write → Primary (memory) → "acknowledged: true"  │
│                 │                      │                                    │
│                 │                      └──→ Secondary 1 (async, later)      │
│                 │                      └──→ Secondary 2 (async, later)      │
│                 │                                                           │
│                 │ RISK WINDOW: Time between primary memory write            │
│                 │              and replication to secondaries               │
│                 │              (~few milliseconds usually)                  │
│                 │                                                           │
│                 │ USE WHEN: General application writes                      │
│                 │           Acceptable risk for most operations             │
│                 │ CAUTION:  NOT safe for financial transactions             │
│                 │                                                           │
├────────────────┼───────────────────────────────────────────────────────────┤
│                 │                                                           │
│ w: "majority"   │ MAJORITY ACKNOWLEDGED (RECOMMENDED FOR IMPORTANT DATA)    │
│                 │ → Driver waits until MAJORITY of replica set members      │
│                 │   confirm the write                                       │
│                 │ → For 3-member set: majority = 2 (primary + 1 secondary) │
│                 │ → For 5-member set: majority = 3                         │
│                 │ → Write SURVIVES even if primary crashes                  │
│                 │   (at least one secondary has it)                         │
│                 │                                                           │
│                 │ TIMELINE:                                                 │
│                 │   App → write → Primary (memory)                         │
│                 │                      │                                    │
│                 │                      ├──→ Secondary 1 confirms ──┐       │
│                 │                      │                            │       │
│                 │                      │    Majority reached ◄──────┘       │
│                 │                      │              │                     │
│                 │                      │    "acknowledged: true" → App      │
│                 │                      │                                    │
│                 │                      └──→ Secondary 2 (may be later)      │
│                 │                                                           │
│                 │ USE WHEN: Financial transactions, user data               │
│                 │           Anything that MUST NOT be lost                  │
│                 │                                                           │
│                 │ THIS IS CLOSEST TO REDSHIFT'S DEFAULT BEHAVIOR            │
│                 │                                                           │
├────────────────┼───────────────────────────────────────────────────────────┤
│                 │                                                           │
│ w: <number>     │ SPECIFIC COUNT ACKNOWLEDGED                               │
│ (e.g., w: 3)   │ → Wait for exactly N members to confirm                  │
│                 │ → w: 3 on a 3-member set = ALL members                   │
│                 │ → STRONGER than majority but SLOWER                      │
│                 │ → Risk: if one node is down, write HANGS (use wtimeout)  │
│                 │                                                           │
│                 │ USE WHEN: Maximum durability required                     │
│                 │ CAUTION:  Availability decreases (any node down = stuck) │
│                 │                                                           │
├────────────────┼───────────────────────────────────────────────────────────┤
│                 │                                                           │
│ j: true         │ JOURNAL ACKNOWLEDGED (orthogonal to w)                    │
│ (add to any w)  │ → Wait until write is flushed to ON-DISK journal         │
│                 │ → Without j:true, w:1 means "in memory" only             │
│                 │ → With j:true, survives process crash (not just failover)│
│                 │                                                           │
│                 │ w:1, j:false  → primary memory only (fast, risky)        │
│                 │ w:1, j:true   → primary memory + disk journal (safer)    │
│                 │ w:"majority", j:true → majority nodes + disk journal     │
│                 │                         (SAFEST possible write)           │
│                 │                                                           │
│                 │ Redshift equivalent: every write is journaled — always    │
│                 │                                                           │
└────────────────┴───────────────────────────────────────────────────────────┘

📊 READ CONCERN — Deep Dive

What Read Concern Controls

READ CONCERN ANSWERS:
    "When I read data, what VERSION of the data am I willing to see?"

    THIS IS NOT THE SAME AS READ PREFERENCE:
    → Read Preference (Project 23): WHICH node do I read FROM?
    → Read Concern (this project):  WHAT data do I see on that node?

    CRITICAL DISTINCTION:
    ┌─────────────────────────────────────────────────────────┐
    │  Read Preference = WHERE to read (primary/secondary)    │
    │  Read Concern    = WHAT to read (committed? local?)     │
    │                                                          │
    │  You set BOTH independently                              │
    │  Combined effect = your consistency guarantee            │
    └─────────────────────────────────────────────────────────┘

Read Concern Levels — From Weakest to Strongest

┌──────────────────┬─────────────────────────────────────────────────────────┐
│ READ CONCERN      │ BEHAVIOR                                                │
├──────────────────┼─────────────────────────────────────────────────────────┤
│                   │                                                         │
│ "local"           │ RETURN WHATEVER THIS NODE HAS (DEFAULT)                 │
│                   │                                                         │
│                   │ → Returns the most recent data on the node you're       │
│                   │   reading from                                          │
│                   │ → On PRIMARY: most recent data (usually up-to-date)     │
│                   │ → On SECONDARY: might be slightly behind primary        │
│                   │ → Does NOT guarantee the data has been replicated       │
│                   │   to majority                                           │
│                   │                                                         │
│                   │ RISK: Data might ROLL BACK                              │
│                   │   → Primary accepts write with w:1                      │
│                   │   → You read it with "local" — you see it              │
│                   │   → Primary crashes BEFORE replicating                  │
│                   │   → New primary elected — that write never existed      │
│                   │   → You saw data that was ROLLED BACK                   │
│                   │                                                         │
│                   │ USE WHEN: Most application reads                        │
│                   │           Slight inconsistency acceptable               │
│                   │           Dashboard displays, product browsing          │
│                   │                                                         │
│                   │ Redshift equivalent: N/A — Redshift never shows         │
│                   │                      uncommitted or rollback-able data  │
│                   │                                                         │
├──────────────────┼─────────────────────────────────────────────────────────┤
│                   │                                                         │
│ "available"       │ RETURN WHATEVER IS AVAILABLE (sharded clusters)          │
│                   │                                                         │
│                   │ → Same as "local" for replica sets                      │
│                   │ → On SHARDED clusters: returns data even during         │
│                   │   chunk migration (might return orphaned documents)     │
│                   │ → FASTEST possible read on sharded clusters             │
│                   │                                                         │
│                   │ RISK: Orphaned documents during chunk migrations        │
│                   │       (temporary duplicates or missing data)            │
│                   │                                                         │
│                   │ USE WHEN: Absolute lowest latency needed on sharded     │
│                   │           cluster, temporary inconsistency acceptable   │
│                   │ RARE: Most applications use "local" instead             │
│                   │                                                         │
├──────────────────┼─────────────────────────────────────────────────────────┤
│                   │                                                         │
│ "majority"        │ RETURN ONLY MAJORITY-COMMITTED DATA                     │
│                   │                                                         │
│                   │ → Returns data that has been confirmed written to       │
│                   │   a majority of replica set members                     │
│                   │ → This data WILL NOT be rolled back                     │
│                   │ → Even if primary fails, this data survives             │
│                   │                                                         │
│                   │ HOW IT WORKS INTERNALLY:                                │
│                   │   → Each node tracks "majority commit point"            │
│                   │   → This is the oplog timestamp confirmed by majority   │
│                   │   → Read concern "majority" returns data ≤ this point   │
│                   │   → Data AFTER this point exists but isn't returned     │
│                   │                                                         │
│                   │ TRADE-OFF: Might return SLIGHTLY OLDER data             │
│                   │   → Primary has write at T=100ms                        │
│                   │   → Majority confirmed up to T=85ms                     │
│                   │   → "majority" read returns data as of T=85ms          │
│                   │   → 15ms of data is "invisible" until majority catches  │
│                   │                                                         │
│                   │ USE WHEN: Financial reads, balance checks               │
│                   │           Any read that must not see rolled-back data   │
│                   │                                                         │
│                   │ THIS IS CLOSEST TO REDSHIFT'S BEHAVIOR                  │
│                   │ (only committed data visible)                           │
│                   │                                                         │
├──────────────────┼─────────────────────────────────────────────────────────┤
│                   │                                                         │
│ "linearizable"    │ ABSOLUTE LATEST COMMITTED DATA (STRONGEST)              │
│                   │                                                         │
│                   │ → Returns data that is BOTH:                            │
│                   │   a) Majority committed (won't roll back)               │
│                   │   b) Reflects ALL successful writes before this read    │
│                   │ → Effectively: "read the absolute truth right now"      │
│                   │ → Waits for any in-progress majority commits            │
│                   │                                                         │
│                   │ RESTRICTIONS:                                           │
│                   │   → ONLY works on PRIMARY                               │
│                   │   → ONLY works on single-document reads                 │
│                   │   → CANNOT use with read preference "secondary"         │
│                   │   → SLOWEST read concern (must confirm majority)        │
│                   │                                                         │
│                   │ HOW IT DIFFERS FROM "majority":                         │
│                   │   → "majority": returns majority-committed snapshot     │
│                   │     (might be a few ms behind absolute latest)          │
│                   │   → "linearizable": confirms that NO newer majority     │
│                   │     write exists — truly the latest                     │
│                   │                                                         │
│                   │ USE WHEN: Reading a lock/semaphore document             │
│                   │           Distributed locking patterns                  │
│                   │           Absolute truth required for single document   │
│                   │                                                         │
│                   │ EXACTLY EQUIVALENT TO REDSHIFT (for single reads)       │
│                   │ But at MUCH higher latency cost                         │
│                   │                                                         │
├──────────────────┼─────────────────────────────────────────────────────────┤
│                   │                                                         │
│ "snapshot"        │ TRANSACTIONAL SNAPSHOT (MULTI-DOCUMENT)                  │
│                   │                                                         │
│                   │ → Returns data from a consistent snapshot in time       │
│                   │ → ALL documents in the query see the SAME point-in-time│
│                   │ → ONLY available inside multi-document transactions     │
│                   │   (Project 28)                                          │
│                   │ → Guarantees: no dirty reads, no non-repeatable reads,  │
│                   │   no phantom reads                                      │
│                   │                                                         │
│                   │ THIS IS EXACTLY REDSHIFT'S DEFAULT BEHAVIOR             │
│                   │ → Every Redshift query is a snapshot read               │
│                   │ → MongoDB makes you explicitly opt into this            │
│                   │   and it requires a transaction context                 │

📊 CAUSAL CONSISTENCY & SESSIONS — The "Read Your Own Writes" Problem

The Problem That Doesn't Exist in Redshift

THE SCENARIO THAT BREAKS NAIVE MONGODB USAGE:

    1. Customer deposits $500 into account
       → Write goes to PRIMARY (w: "majority" — confirmed)
    
    2. Customer immediately checks balance
       → Read goes to SECONDARY (readPreference: "secondaryPreferred")
       → Secondary hasn't received the replication yet
       → Customer sees OLD balance WITHOUT the $500 deposit
       → Customer panics: "Where's my money?!"

    IN REDSHIFT: This CANNOT happen
        → All reads go through the same leader node
        → No replicas serve independent reads
        → Your write is immediately visible to your next read

    IN MONGODB: This is a REAL problem when:
        → Write concern < majority, OR
        → Read goes to a different node than the write, OR
        → Replication lag exists (always exists, even if tiny)

THE SOLUTION: CAUSAL CONSISTENCY via CLIENT SESSIONS

How Sessions Enable Causal Consistency

CAUSAL CONSISTENCY GUARANTEES (within a session):

    1. READ YOUR OWN WRITES
       → If you write X, then read, you WILL see X
       → Even if read goes to a different node

    2. MONOTONIC READS
       → If you read X at time T, next read won't see older than T
       → You never go "backwards" in time

    3. MONOTONIC WRITES
       → Your writes are applied in the order you issued them
       → Write A before Write B → A is applied before B

    4. WRITES FOLLOW READS
       → If you read data, then write based on it,
         your write is guaranteed to be AFTER the read's snapshot

HOW IT WORKS INTERNALLY:

    ┌──────────────────────────────────────────────────────┐
    │  CLIENT SESSION                                       │
    │                                                       │
    │  Write to Primary                                     │
    │  → Gets back: operationTime = T1                      │
    │  → Gets back: clusterTime = T1                        │
    │  → Session stores T1                                  │
    │                                                       │
    │  Read from Secondary (next operation in same session) │
    │  → Session sends: afterClusterTime = T1               │
    │  → Secondary checks: "Am I caught up to T1?"          │
    │     → YES: return data                                │
    │     → NO:  WAIT until replication catches up to T1    │
    │            THEN return data                           │
    │                                                       │
    │  RESULT: Read ALWAYS sees the write, even on          │
    │          a different node                              │
    └──────────────────────────────────────────────────────┘

    THE SECONDARY WAITS for replication to catch up
    → Adds latency (maybe 1-50ms depending on replication lag)
    → But GUARANTEES you see your own writes
    → Without sessions: secondary returns immediately (stale data)

📊 CAP THEOREM — MongoDB's Position vs Redshift's

CAP THEOREM REFRESHER:
    In a distributed system, you can guarantee at most TWO of:
    → Consistency: every read returns the most recent write
    → Availability: every request receives a response
    → Partition tolerance: system works despite network partitions

    REALITY: Network partitions ALWAYS happen in distributed systems
             So you're really choosing between C and A during partitions

MONGODB'S POSITION: CP (Consistency + Partition Tolerance)
    → During a network partition:
      → Minority side CANNOT accept writes (no primary)
      → Majority side elects new primary, continues
      → Minority side is UNAVAILABLE for writes
      → CONSISTENCY is preserved at the cost of AVAILABILITY
    
    → BUT with read concern "local" + secondary reads:
      → MongoDB can ACT like AP for reads
      → Stale reads from isolated secondaries ARE available
      → You're trading consistency for availability PER OPERATION

    → MongoDB's TRUE position: tunable on the CP ↔ AP spectrum
      → w:"majority" + readConcern:"majority" = strong CP
      → w:1 + readConcern:"local" + secondaryPreferred = more AP

REDSHIFT'S POSITION: CA (Consistency + Availability)
    → Redshift is NOT a distributed system in the CAP sense
    → Single cluster, single leader node
    → No network partitions WITHIN the cluster (shared nothing but co-located)
    → If a node fails: Redshift handles internally (automatic recovery)
    → No partition tolerance CHOICE because no real partitions
    → Multi-AZ: still one logical cluster, AWS handles failover

    → Redshift avoids CAP trade-offs by being a single logical system
    → MongoDB faces CAP trade-offs because it's truly distributed

class FinancialDatabase:
    """
    Production-grade MongoDB configuration for financial services.
    Different consistency guarantees for different access patterns.
    
    Redshift equivalent: N/A — Redshift gives ONE consistency level
    for ALL operations. This granular control is MongoDB-specific.
    """
    
    def __init__(self, connection_string: str):
        # Base client — safe defaults
        self.client = MongoClient(
            connection_string,
            retryWrites=True,
            retryReads=True,
            w="majority",
            wTimeoutMS=5000,
            readConcernLevel="majority"
        )
        
        db = self.client["finservices"]
        
        # ─── CRITICAL PATH: Customer-facing balance operations ───
        # Strongest guarantees — must see own writes
        self.accounts = db.get_collection(
            "accounts",
            write_concern=WriteConcern(w="majority", j=True, wtimeout=5000),
            read_concern=ReadConcern("majority"),
            read_preference=ReadPreference.PRIMARY
        )
        
        # ─── CRITICAL PATH: Financial transactions ───────────────
        # Strongest guarantees — must never lose a transaction
        self.transactions = db.get_collection(
            "transactions",
            write_concern=WriteConcern(w="majority", j=True, wtimeout=5000),
            read_concern=ReadConcern("majority"),
            read_preference=ReadPreference.PRIMARY
        )
        
        # ─── HIGH THROUGHPUT: Fraud detection reads ──────────────
        # Weaker guarantees — slight staleness acceptable
        self.fraud_transactions = db.get_collection(
            "transactions",  # same collection, different handle
            write_concern=WriteConcern(w=1),  # won't write via this handle
            read_concern=ReadConcern("local"),
            read_preference=SecondaryPreferred(max_staleness=120)
        )
        
        # ─── FAST WRITES: Audit log / telemetry ─────────────────
        # Weakest guarantees — losing occasional log entries is OK
        self.audit_log = db.get_collection(
            "audit_log",
            write_concern=WriteConcern(w=1, j=False),
            read_concern=ReadConcern("local"),
            read_preference=ReadPreference.SECONDARY_PREFERRED
        )
    
    def customer_deposit(self, account_id: str, amount: float,
                         description: str) -> dict:
        """
        Customer deposit — uses causal session for read-your-writes.
        After deposit, customer can immediately see updated balance.
        """
        with self.client.start_session(causal_consistency=True) as session:
            # Write with majority + journal
            self.accounts.update_one(
                {"_id": account_id},
                {
                    "$inc": {"balance": amount},
                    "$set": {"updated_at": datetime.now()}
                },
                session=session
            )
            
            self.transactions.insert_one(
                {
                    "account_id": account_id,
                    "type": "deposit",
                    "amount": amount,
                    "timestamp": datetime.now(),
                    "description": description
                },
                session=session
            )
            
            # Read within same session — guaranteed to see the deposit
            updated = self.accounts.find_one(
                {"_id": account_id},
                session=session
            )
            
            # Audit log — fire-and-forget (different concern)
            self.audit_log.insert_one({
                "action": "deposit",
                "account_id": account_id,
                "amount": amount,
                "timestamp": datetime.now(),
                "session_id": str(session.session_id)
            })
            
            return {
                "status": "success",
                "account_id": account_id,
                "new_balance": updated["balance"],
                "deposit_amount": amount
            }
    
    def customer_balance_check(self, account_id: str) -> dict:
        """
        Customer balance check — reads from primary with majority concern.
        No session needed if not preceded by a write.
        """
        account = self.accounts.find_one({"_id": account_id})
        return {
            "account_id": account_id,
            "balance": account["balance"],
            "as_of": account["updated_at"]
        }
    
    def fraud_scan_large_transactions(self, threshold: float,
                                       hours: int = 1) -> list:
        """
        Fraud detection — reads from secondaries, local concern.
        High throughput, slight staleness acceptable.
        """
        cutoff = datetime.now() - timedelta(hours=hours)
        return list(self.fraud_transactions.find(
            {
                "amount": {"$gt": threshold},
                "timestamp": {"$gte": cutoff}
            }
        ).sort("amount", -1).limit(100))
    
    def reconciliation_report(self) -> dict:
        """
        End-of-day reconciliation — snapshot transaction.
        All reads see consistent point-in-time.
        Equivalent to Redshift's default query behavior.
        """
        with self.client.start_session() as session:
            session.start_transaction(
                read_concern=ReadConcern("snapshot"),
                write_concern=WriteConcern(w="majority")
            )
            
            try:
                accounts = list(self.accounts.find(
                    {}, session=session
                ).sort("_id", 1))
                
                total_balance = sum(a["balance"] for a in accounts)
                
                txn_summary = list(self.transactions.aggregate([
                    {"$group": {
                        "_id": "$type",
                        "count": {"$sum": 1},
                        "total": {"$sum": "$amount"}
                    }}
                ], session=session))
                
                session.commit_transaction()
                
                return {
                    "status": "success",
                    "total_accounts": len(accounts),
                    "total_balance": total_balance,
                    "transaction_summary": txn_summary,
                    "timestamp": datetime.now()
                }
                
            except Exception as e:
                session.abort_transaction()
                return {"status": "error", "message": str(e)}

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Internals — Majority Commit Point & Oplog

HOW readConcern "majority" WORKS INSIDE MONGODB:

    STEP 1: Primary writes to its oplog
        → oplog entry: { ts: Timestamp(100), op: "u", ... }
        → Primary's oplog timestamp: T=100

    STEP 2: Secondaries replicate from primary's oplog
        → Secondary 1 applies up to T=98
        → Secondary 2 applies up to T=95

    STEP 3: Primary tracks each secondary's progress
        → Primary knows: "Secondary 1 is at T=98, Secondary 2 at T=95"

    STEP 4: Primary calculates MAJORITY COMMIT POINT
        → 3-member set: majority = 2
        → Primary is at T=100, Secondary 1 at T=98
        → 2 members (majority) have confirmed T=98
        → MAJORITY COMMIT POINT = T=98

    STEP 5: readConcern "majority" query arrives
        → Query can only see data with oplog timestamp ≤ T=98
        → Data at T=99 and T=100 EXISTS on primary
          but is INVISIBLE to this read concern
        → Those writes haven't reached majority yet

    STEP 6: As secondaries catch up
        → Secondary 1 reaches T=100, Secondary 2 reaches T=98
        → New majority commit point: T=100
        → Now readConcern "majority" can see T=99 and T=100

    ┌─────────────────────────────────────────────────────┐
    │  PRIMARY oplog:    T=95 T=96 T=97 T=98 T=99 T=100  │
    │  Secondary 1:      T=95 T=96 T=97 T=98             │
    │  Secondary 2:      T=95                              │
    │                                                      │
    │  Majority commit:  T=98 (primary + sec1 = 2 = maj)  │
    │                                                      │
    │  readConcern "local":    sees up to T=100 (on primary)│
    │  readConcern "majority": sees up to T=98              │
    │  readConcern "linearizable": confirms T=100 is        │
    │                              majority-committed,      │
    │                              then returns it          │
    └─────────────────────────────────────────────────────┘


HOW CAUSAL SESSIONS WORK INSIDE:

    STEP 1: Client writes with session
        → Server returns: operationTime = T=100, clusterTime = T=100
        → Session STORES this: "my latest operation was T=100"

    STEP 2: Client reads with SAME session (possibly different node)
        → Session sends: afterClusterTime = T=100
        → Target node receives this parameter

    STEP 3: Target node checks
        → "Am I caught up to T=100?"
        → If YES (e.g., secondary at T=102): return data immediately
        → If NO (e.g., secondary at T=95): WAIT for replication
          → Apply oplog entries until reaching T=100
          → THEN return data
        → Wait has a timeout (serverSelectionTimeoutMS)

    RESULT: Client ALWAYS sees its own writes
            Even when reading from a different node
            Cost: possible wait for replication catch-up


HOW readConcern "linearizable" WORKS INSIDE:

    STEP 1: Read request arrives at PRIMARY (required)
    STEP 2: Primary initiates a no-op write to oplog
    STEP 3: Primary waits for MAJORITY to confirm the no-op
    STEP 4: If majority confirms: the current state IS the latest
            committed state — no newer majority-committed write exists
    STEP 5: Return the data

    WHY THE NO-OP WRITE?
    → Confirms primary is still the REAL primary
    → If a network partition happened and a new primary was elected,
      this old primary's no-op would NOT get majority confirmation
    → Prevents stale reads from a deposed primary

    COST: Extra round-trip for majority confirmation
          Slowest read concern (~10-50ms added)
          Only works on single documents (cannot scan with this)

Redshift Internals — For Comparison

HOW REDSHIFT ACHIEVES CONSISTENCY:

    SIMPLE: Single-leader architecture eliminates the problem

    STEP 1: Query arrives at LEADER NODE
    STEP 2: Leader assigns a SNAPSHOT timestamp
    STEP 3: Leader distributes query to compute nodes
    STEP 4: Each compute node reads data as of snapshot timestamp
    STEP 5: Compute nodes return results to leader
    STEP 6: Leader merges and returns to client

    NO REPLICATION LAG: compute nodes share storage (RA3)
                        or have data pre-distributed (DC2)
    NO MAJORITY COMMIT: leader is the single source of truth
    NO READ CONCERNS:   every read is snapshot-isolated
    NO WRITE CONCERNS:  every write is synchronously committed

    WHY THIS IS SIMPLER:
    → One logical system, not a distributed replica set
    → No independent nodes that can fall behind
    → No CAP trade-offs (no real network partitions within cluster)
    → Cost: limited concurrency, no read scaling, single point of coordination

WHY THE INTERNALS CAUSE DIFFERENT BEHAVIOR:

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 24:

PITFALL 1: "I'll just use the defaults — worked fine in Redshift"
    ❌ Leave everything at w:1, readConcern "local" (MongoDB defaults)
    ✅ Explicitly set w:"majority" + readConcern "majority" for critical data
    WHY: Redshift defaults = strongest consistency
         MongoDB defaults = fastest, NOT strongest
         Your Redshift instinct ("defaults are safe") is WRONG here

PITFALL 2: "I'll set everything to maximum — safest possible"
    ❌ w:"majority",j:true + readConcern "linearizable" on EVERY operation
    ✅ Tune per operation: critical = strong, non-critical = weak
    WHY: Redshift trained you that "one consistency level is fine"
         In MongoDB, maximum consistency everywhere = 10-50x slower writes
         Telemetry writes at w:"majority",j:true = massive waste

PITFALL 3: "w:majority without wtimeout"
    ❌ writeConcern: { w: "majority" }  // no timeout
    ✅ writeConcern: { w: "majority", wtimeout: 5000 }
    WHY: If a secondary is down, majority may be unreachable
         Without wtimeout: application thread hangs FOREVER
         With wtimeout: get error after 5s, can retry or alert

PITFALL 4: "Reading from secondary = same as reading from primary"
    ❌ readPreference: "secondary" without thinking about consistency
    ✅ Always pair secondary reads with appropriate readConcern
    WHY: Redshift has no replicas — you never thought about this
         In MongoDB: secondary reads are STALE by definition
         How stale depends on replication lag (ms to minutes)

PITFALL 5: "Causal sessions are overkill — I'll just read from primary"
    ❌ Force all reads to primary to avoid staleness issues
    ✅ Use causal sessions + secondary reads for read-your-writes
    WHY: Forcing primary reads defeats the purpose of replica sets
         Primary becomes bottleneck (same as Redshift leader node)
         Causal sessions give you BOTH: secondary scaling + consistency

PITFALL 6: "readConcern majority = I see ALL committed writes"
    ❌ Assuming readConcern "majority" shows the absolute latest data
    ✅ Understand it shows majority-COMMITTED data (may lag by ms)
    WHY: There's a window between primary write and majority commit
         In that window: data exists on primary but isn't in majority snapshot
         For absolute latest: use "linearizable" (single doc only)
         For practical purposes: "majority" is close enough for 99.9% of cases

PITFALL 7: "I wrote with w:1 and read with readConcern majority — why can't I see my write?"
    ❌ writeConcern: w:1 → readConcern: "majority"
    ✅ writeConcern: w:"majority" → readConcern: "majority"
    WHY: w:1 means only primary confirmed
         readConcern "majority" only shows data confirmed by majority
         Your w:1 write might not be in majority yet
         MISMATCH: write guarantee weaker than read requirement
         RULE: write concern must be ≥ read concern for consistency

      2. Concurrency scaling → burst read capacity
      3. CANNOT tune consistency for throughput
    
    → MongoDB scales BOTH capacity AND consistency independently
    → Redshift scales ONLY capacity

📐 DATA MODELING COMPARISON

HOW CONSISTENCY REQUIREMENTS AFFECT DATA MODELING:

REDSHIFT (no impact on modeling):
    → Consistency is fixed → doesn't affect table design
    → Design for query performance only
    → Normalize or denormalize based on JOIN costs

MONGODB (MAJOR impact on modeling):
    → If you need transaction-level consistency across documents:
      → PREFER embedding (single-document atomicity — free!)
      → AVOID multi-document transactions when possible
      → Model data so common operations touch ONE document
    
    → If you can tolerate eventual consistency:
      → References are fine ($lookup is OK)
      → Separate collections for different access patterns
      → Update propagation can be async
    
    EXAMPLE: Account + Recent Transactions
    
    APPROACH A: Embedded (single-document atomicity)
    {
        _id: "ACC-001",
        balance: 10500,
        recent_transactions: [         // ← embedded
            { type: "deposit", amount: 500, ts: ISODate("...") },
            { type: "purchase", amount: 42, ts: ISODate("...") }
        ]
    }
    → Balance update + transaction recording = ONE updateOne()
    → Atomic without transactions
    → readConcern "majority" on this ONE document = consistent view
    
    APPROACH B: Referenced (needs multi-document transaction)
    // accounts collection
    { _id: "ACC-001", balance: 10500 }
    // transactions collection
    { account_id: "ACC-001", type: "deposit", amount: 500 }
    → Balance update + transaction insert = TWO operations
    → NEEDS multi-document transaction for atomicity
    → More expensive, more complex

    The consistency model DRIVES your document design in MongoDB
    In Redshift: consistency doesn't affect table design at all
