Problem: Find All Customers From "New York" Who Spent > $500

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce platform now has 1000 customers (loaded in Project 2)
    The marketing team needs answers:
    
    1. "Show me all customers from New York who spent over $500"
    2. "Show me just names and emails of premium customers"
    3. "Show me the top 10 spenders"
    4. "Show me customers aged 25-35 in Texas or California"
    5. "Show me customers who prefer Electronics"
    6. "Show me customers who DON'T have a phone number"
    7. "How many customers signed up each month?"

🔴 REDSHIFT MINDSET — How You Would Solve This

All 7 Queries in Redshift SQL

-- REDSHIFT: Your comfort zone — pure SQL

-- ─── QUERY 1: Basic filter with AND ─────────────────
SELECT *
FROM store.customers
WHERE state = 'NY'
  AND total_spent > 500;

-- ─── QUERY 2: Select specific columns (projection) ──
SELECT name, email
FROM store.customers
WHERE is_premium = TRUE;

-- ─── QUERY 3: Sort and limit ────────────────────────
SELECT name, email, total_spent
FROM store.customers
ORDER BY total_spent DESC
LIMIT 10;

-- ─── QUERY 4: Multiple conditions with OR + BETWEEN ─
SELECT *
FROM store.customers
WHERE age BETWEEN 25 AND 35
  AND state IN ('TX', 'CA');

-- ─── QUERY 5: Query JSON/SUPER column ───────────────
-- Redshift with SUPER type:
SELECT *
FROM store.customers
WHERE JSON_EXTRACT_PATH_TEXT(preferences, 'preferred_categories')
    LIKE '%Electronics%';
-- OR with SUPER type:
-- WHERE c.preferences.preferred_categories[0] = 'Electronics';
-- CLUNKY — Redshift wasn't designed for this

-- ─── QUERY 6: Find NULL values ──────────────────────
SELECT *
FROM store.customers
WHERE phone IS NULL;

-- ─── QUERY 7: Count by month ────────────────────────
SELECT DATE_TRUNC('month', signup_date) AS signup_month,
       COUNT(*) AS customer_count
FROM store.customers
GROUP BY 1
ORDER BY 1;

What Redshift Does Behind the Scenes

FOR QUERY 1: WHERE state = 'NY' AND total_spent > 500

    1. Leader node receives query, creates execution plan
    2. Plan pushed to compute slices
    3. Each slice checks ZONE MAPS:
       → state column: skip blocks where max(state) < 'NY' or min(state) > 'NY'
       → total_spent column: skip blocks where max(total_spent) <= 500
    4. Qualifying blocks scanned COLUMN BY COLUMN:
       → Read ONLY state column → filter for 'NY'
       → Read ONLY total_spent column → filter for > 500
       → Combine row IDs that pass BOTH filters
       → Read remaining columns for matching rows
    5. Results sent back to leader node → returned to client
    
    KEY INSIGHT: Redshift reads ONLY the columns it needs
    → SELECT * reads all columns (expensive)
    → SELECT name, email reads only 2 columns (cheap)
    → This is the columnar advantage for analytical queries

🟢 MONGODB MINDSET — How This Same Problem Is Approached

The Fundamental Difference in Query Philosophy

REDSHIFT QUERY MODEL:
    SELECT [columns] FROM [table] WHERE [conditions] ORDER BY [columns] LIMIT [n]
    → Declarative SQL: you describe WHAT you want
    → Optimizer decides HOW to get it
    → Everything is a flat row with fixed columns

MONGODB QUERY MODEL:
    db.collection.find( {filter}, {projection} ).sort({field: 1}).limit(n)
    → Method chaining: each method modifies the cursor
    → Filter is a JSON document describing conditions
    → Projection is a JSON document describing which fields to return
    → Documents can be nested — queries can reach INTO nested structures

THE MENTAL MODEL:
    Redshift: "Filter ROWS by COLUMN values"
    MongoDB:  "Filter DOCUMENTS by FIELD values — including nested fields and arrays"
    
    find() = SELECT + FROM + WHERE combined
    projection = SELECT column list
    sort() = ORDER BY
    limit() = LIMIT
    skip() = OFFSET

🧩 HOW TO APPROACH IT IN MONGODB

QUERY ANATOMY — DECONSTRUCTING find():

    db.collection.find( filter, projection ).sort().limit().skip()
                        │         │            │       │       │
                        │         │            │       │       └─ OFFSET
                        │         │            │       └─ LIMIT
                        │         │            └─ ORDER BY
                        │         └─ SELECT columns (inclusion/exclusion)
                        └─ WHERE clause (as a JSON document)

EVERY find() CALL HAS TWO ARGUMENTS:
    Argument 1: FILTER (which documents to match) — the WHERE clause
    Argument 2: PROJECTION (which fields to return) — the SELECT list
    
    Both are JSON documents
    Both are optional

BUILDING QUERIES — THE THINKING PROCESS:

    REDSHIFT BRAIN:
        "SELECT name, email FROM customers WHERE state='NY' AND age > 30"
        → Table → Columns → Conditions → Done
    
    MONGODB BRAIN:
        Step 1: What documents do I want? → { "address.state": "NY", age: { $gt: 30 } }
        Step 2: What fields do I want back? → { name: 1, email: 1, _id: 0 }
        Step 3: Any sorting? → .sort({ name: 1 })
        Step 4: Any limiting? → .limit(100)
        
        db.customers.find(
            { "address.state": "NY", age: { $gt: 30 } },
            { name: 1, email: 1, _id: 0 }
        ).sort({ name: 1 }).limit(100)

The 7 Marketing Team Queries — MongoDB Solutions

// ═══════════════════════════════════════════════════════
// ALL 7 QUERIES — MONGOSH
// Using the 1000 customers loaded in Project 2
// ═══════════════════════════════════════════════════════

use ecommerce

// ═══════════════════════════════════════════════════════
// QUERY 1: "Customers from New York who spent over $500"
// Redshift: SELECT * FROM customers WHERE state = 'NY' AND total_spent > 500
// ═══════════════════════════════════════════════════════

db.customers.find({
    "address.state": "NY",           // dot notation → reaches into nested object
    total_spent: { $gt: 500 }        // $gt = greater than
})

// BREAKDOWN:
// ┌──────────────────────────────────────────────────────────────┐
// │ { "address.state": "NY", total_spent: { $gt: 500 } }        │
// │   ↑                        ↑                                 │
// │   DOT NOTATION              COMPARISON OPERATOR               │
// │   reaches into:             $gt = >                           │
// │   { address: { state: "NY" } }                               │
// │                                                               │
// │   IMPLICIT AND:                                               │
// │   Multiple fields in one filter = AND condition               │
// │   No $and keyword needed for simple cases                    │
// └──────────────────────────────────────────────────────────────┘

// HOW MANY matched?
db.customers.countDocuments({
    "address.state": "NY",
    total_spent: { $gt: 500 }
})


// ═══════════════════════════════════════════════════════
// QUERY 2: "Just names and emails of premium customers"
// Redshift: SELECT name, email FROM customers WHERE is_premium = TRUE
// ═══════════════════════════════════════════════════════

db.customers.find(
    { is_premium: true },            // FILTER: which documents
    { name: 1, email: 1, _id: 0 }   // PROJECTION: which fields
)

// PROJECTION RULES:
// ┌──────────────────────────────────────────────────────────────┐
// │ { name: 1, email: 1 }        → Include name, email, _id     │
// │                                  (_id is ALWAYS included      │
// │                                   unless explicitly excluded) │
// │                                                               │
// │ { name: 1, email: 1, _id: 0 } → Include name, email         │
// │                                  Exclude _id                  │
// │                                                               │
// │ { preferences: 0, phone: 0 }  → Include ALL fields           │
// │                                  EXCEPT preferences and phone │
// │                                                               │
// │ RULE: Cannot MIX inclusion and exclusion                      │
// │       (except _id: 0 which is always allowed)                │
// │                                                               │
// │ { name: 1, phone: 0 }         → ERROR! Can't mix 1 and 0    │
// │ { name: 1, _id: 0 }           → OK! _id is the exception    │
// └──────────────────────────────────────────────────────────────┘

// REDSHIFT COMPARISON:
// In Redshift: SELECT name, email → reads only 2 column files (fast)
// In MongoDB:  projection { name:1, email:1 } → reads full document
//              then strips fields before returning to client
//              STILL reads entire BSON document from storage
//              Projection saves NETWORK bandwidth, not disk I/O
//              (Covered indexes are the exception — Project 10)


// ═══════════════════════════════════════════════════════
// QUERY 3: "Top 10 spenders"
// Redshift: SELECT name, email, total_spent FROM customers
//           ORDER BY total_spent DESC LIMIT 10
// ═══════════════════════════════════════════════════════

db.customers.find(
    {},                                         // no filter — all documents
    { name: 1, email: 1, total_spent: 1, _id: 0 }  // projection
).sort(
    { total_spent: -1 }                         // -1 = descending
).limit(10)

// SORT DIRECTIONS:
//   1  = ascending  (ASC)  → smallest first
//   -1 = descending (DESC) → largest first

// MULTI-FIELD SORT (like ORDER BY col1 ASC, col2 DESC):
// .sort({ total_spent: -1, name: 1 })
// → Sort by total_spent descending, then name ascending for ties

// IMPORTANT PERFORMANCE NOTE:
// Without an index on total_spent:
//   → MongoDB must load ALL documents → sort in memory → return top 10
//   → If result set > 100MB: sort FAILS with error
//   → Redshift: sort key on total_spent would make this instant
//   → We'll fix this with indexes in Project 9


// ═══════════════════════════════════════════════════════
// QUERY 4: "Customers aged 25-35 in Texas or California"
// Redshift: SELECT * FROM customers
//           WHERE age BETWEEN 25 AND 35
//           AND state IN ('TX', 'CA')
// ═══════════════════════════════════════════════════════

db.customers.find({
    age: { $gte: 25, $lte: 35 },                    // BETWEEN equivalent
    "address.state": { $in: ["TX", "CA"] }           // IN equivalent
})

// ALTERNATIVE with explicit $and (same result, more verbose):
db.customers.find({
    $and: [
        { age: { $gte: 25 } },
        { age: { $lte: 35 } },
        { "address.state": { $in: ["TX", "CA"] } }
    ]
})

// WHEN IS EXPLICIT $and NEEDED?
// When you have MULTIPLE conditions on the SAME field with different operators:
db.customers.find({
    $and: [
        { age: { $gt: 20 } },
        { age: { $lt: 30 } },
        { age: { $ne: 25 } }      // exclude exactly 25
    ]
})
// The shorthand { age: { $gt: 20, $lt: 30, $ne: 25 } } also works here
// But explicit $and is needed when same field appears with SAME operator:
// $and: [ { tags: "A" }, { tags: "B" } ]  ← need BOTH tags (can't shorthand)


// ═══════════════════════════════════════════════════════
// QUERY 5: "Customers who prefer Electronics"
// Redshift: Extremely painful with SUPER type or LIKE on JSON string
// MongoDB: NATURAL — arrays are first-class queryable
// ═══════════════════════════════════════════════════════

db.customers.find({
    "preferences.preferred_categories": "Electronics"
})

// WHAT JUST HAPPENED:
// ┌──────────────────────────────────────────────────────────────┐

// │ {                                                            │
// │   "preferences": {                                           │
// │     "preferred_categories": ["Electronics", "Books"]          │
// │   }                                                          │
// │ }                                                            │
// │                                                               │
// │ Query: { "preferences.preferred_categories": "Electronics" }  │
// │                                                               │
// │ MongoDB AUTOMATICALLY checks if "Electronics" is             │
// │ CONTAINED IN the array                                       │
// │                                                               │
// │ No special array function needed                             │
// │ No unnesting, no LIKE '%Electronics%'                        │
// │ Just query the field — MongoDB knows it's an array           │
// │ and searches WITHIN it                                       │
// │                                                               │
// │ THIS IS MONGODB'S SUPERPOWER — arrays are queryable natively │
// └──────────────────────────────────────────────────────────────┘

// MORE ARRAY QUERY EXAMPLES:

// Customers who prefer BOTH Electronics AND Books:
db.customers.find({
    "preferences.preferred_categories": { $all: ["Electronics", "Books"] }
})

// Customers who prefer EITHER Electronics OR Books:
db.customers.find({

})

// Customers who prefer EXACTLY ["Electronics", "Books"] (exact match, order matters):
db.customers.find({
    "preferences.preferred_categories": ["Electronics", "Books"]
})

// Customers with EXACTLY 3 preferred categories:
db.customers.find({
    "preferences.preferred_categories": { $size: 3 }
})

// REDSHIFT COMPARISON:
// To query an array in Redshift SUPER type:
//   SELECT * FROM customers c, c.preferences.preferred_categories AS cat
//   WHERE cat = 'Electronics';
// OR:
//   SELECT * FROM customers
//   WHERE JSON_EXTRACT_PATH_TEXT(preferences, 'preferred_categories')
//       LIKE '%Electronics%';
//
// Both are CLUNKY compared to MongoDB's natural array querying
// Redshift was not designed for this — MongoDB was


// ═══════════════════════════════════════════════════════
// QUERY 6: "Customers who DON'T have a phone number"
// Redshift: SELECT * FROM customers WHERE phone IS NULL
// MongoDB: THREE different interpretations!
// ═══════════════════════════════════════════════════════

// Interpretation A: phone field doesn't exist at all
db.customers.find({
    phone: { $exists: false }
})

// Interpretation B: phone is explicitly null
db.customers.find({
    phone: { $type: "null" }
})

// Interpretation C: phone is null OR missing (Redshift IS NULL equivalent)
db.customers.find({
    phone: null
})

// Interpretation D: phone exists and has a real value
db.customers.find({
    phone: { $exists: true, $ne: null }
})

// COUNT each category:
let total = db.customers.countDocuments({})
let phoneMissing = db.customers.countDocuments({ phone: { $exists: false } })
let phoneNull = db.customers.countDocuments({ phone: { $type: "null" } })
let phoneExists = db.customers.countDocuments({ phone: { $exists: true, $ne: null } })

print(`Total: ${total}`)
print(`Phone missing: ${phoneMissing}`)
print(`Phone null: ${phoneNull}`)
print(`Phone has value: ${phoneExists}`)

⚡ WHAT COULD BE DONE (MongoDB-Specific Optimizations)

OPTIMIZATION 1: CURSOR METHODS — CONTROL HOW RESULTS ARE RETURNED

    // Batch size — control network round trips
    db.customers.find({}).batchSize(50)
    // Default: 101 documents in first batch, then 16MB batches
    // Small batchSize = more round trips but less memory per batch
    // Large batchSize = fewer round trips but more memory per batch
    // Redshift: no equivalent — results stream as one response
    
    // Hint — force specific index usage
    db.customers.find({ "address.state": "NY" }).hint({ "address.state": 1 })
    // Like Redshift: /*+ USE_SORT_KEY */ (hypothetical — Redshift auto-optimizes)
    // Useful when query planner makes wrong index choice
    // We'll use this in Project 9 after creating indexes

OPTIMIZATION 2: $elemMatch — PRECISE ARRAY QUERYING

    // When array contains OBJECTS, $elemMatch matches within SINGLE element
    // Example: orders array with { product: "X", qty: 5 }
    // Find documents where ONE element matches BOTH conditions:
    
    db.orders.find({
        items: {
            $elemMatch: {
                product: "Widget",
                qty: { $gt: 10 }
            }
        }
    })
    // Without $elemMatch: could match product:"Widget" in one element
    // and qty > 10 in a DIFFERENT element — incorrect!
    // Redshift: doesn't have this problem (no arrays of objects natively)

OPTIMIZATION 3: $regex WITH INDEX SUPPORT

    // Regex that STARTS with anchor (^) can use index
    db.customers.find({ name: /^Customer_00/ })    // ← CAN use index on name
    db.customers.find({ name: /Customer_00/ })     // ← CANNOT use index (full scan)
    
    // Redshift equivalent:
    // WHERE name LIKE 'Customer_00%'  → can use sort key (if name is sort key)
    // WHERE name LIKE '%Customer_00%' → full table scan (same as MongoDB)

✅ WHAT SHOULD DO

BEST PRACTICES FOR QUERYING IN MONGODB:

    ✅ USE PROJECTION TO LIMIT RETURNED FIELDS
       → db.customers.find({}, { name: 1, email: 1, _id: 0 })
       → Reduces network transfer (documents can be large)
       → Redshift equivalent: SELECT col1, col2 (but Redshift saves disk I/O too)
       → MongoDB saves network I/O but still reads full document from storage
       → Exception: covered index queries (Project 10)
    
    ✅ USE DOT NOTATION FOR NESTED FIELDS
       → "address.state" reaches into { address: { state: "NY" } }
       → "preferences.preferred_categories" reaches into nested arrays
       → Can go multiple levels: "a.b.c.d"
       → Redshift has NO equivalent (flat columns only, or SUPER type with ugly syntax)
    
    ✅ USE findOne() WHEN YOU EXPECT EXACTLY ONE RESULT
       → db.customers.findOne({ email: "alice@example.com" })
       → Returns the DOCUMENT directly (not a cursor)
       → More efficient than find().limit(1) in some cases
       → Redshift equivalent: SELECT * FROM customers WHERE email = '...' LIMIT 1
    
    ✅ USE countDocuments() INSTEAD OF find().count()
       → find().count() is deprecated
       → countDocuments({}) is the correct way
       → estimatedDocumentCount() is faster but approximate (reads metadata)
       → Redshift: COUNT(*) always exact
    
    ✅ COMBINE SORT + LIMIT FOR "TOP N" QUERIES
       → .sort({ total_spent: -1 }).limit(10) — top 10 spenders
       → MongoDB optimizes this internally (doesn't sort ALL documents)
       → IF an index exists on the sort field
       → Without index: must sort all in memory (100MB limit)
    
    ✅ USE $in INSTEAD OF MULTIPLE $or FOR SAME FIELD
       → { "address.state": { $in: ["TX", "CA", "NY"] } }
       → NOT: { $or: [{"address.state":"TX"}, {"address.state":"CA"}, ...] }
       → $in is optimized to use index range scans
       → Redshift equivalent: WHERE state IN ('TX', 'CA', 'NY')

REDSHIFT HABITS TO UNLEARN:
    🔄 "SELECT * is fine for exploration"
       → In MongoDB, find({}) returns ENTIRE documents including nested objects
       → Can be very verbose — use projection or findOne() for exploration
    
    🔄 "WHERE clause is the only filtering mechanism"
       → MongoDB has $exists, $type, $size, $elemMatch, $all, $regex
       → These have NO direct SQL equivalents — learn them
    
    🔄 "NULL means the value is missing"
       → In MongoDB: null ≠ field-not-existing (Project 2 lesson applied here)
       → Query carefully: { field: null } matches BOTH

❌ WHAT SHOULD NOT DO

ANTI-PATTERNS — ESPECIALLY TRAPS FOR REDSHIFT USERS:

    ❌ USING find() FOR ANALYTICS (GROUP BY / SUM / AVG)
       # WRONG — trying to do GROUP BY with find()
       db.customers.find({}).forEach(doc => {
           // manually aggregating in application code
           if (!counts[doc.address.state]) counts[doc.address.state] = 0;
           counts[doc.address.state]++;
       })
       
       # RIGHT — use aggregation pipeline
       db.customers.aggregate([
           { $group: { _id: "$address.state", count: { $sum: 1 } } }
       ])
       
       WHY: find() retrieves documents — aggregate() computes results
            Doing aggregation in app code transfers ALL data over network
            Aggregation pipeline runs ON the server
            Same as Redshift: never SELECT * then aggregate in Python

    ❌ MIXING INCLUSION AND EXCLUSION IN PROJECTION
       # WRONG
       db.customers.find({}, { name: 1, phone: 0 })
       # → Error: Cannot do inclusion on field name in exclusion projection
       
       # RIGHT — include what you want
       db.customers.find({}, { name: 1, email: 1, _id: 0 })
       
       # RIGHT — exclude what you don't want
       db.customers.find({}, { preferences: 0, phone: 0 })
       
       RULE: Pick ONE mode — include or exclude (except _id)

    ❌ USING skip() FOR PAGINATION AT LARGE OFFSETS
       # WRONG — slow for large offsets
       db.customers.find({}).sort({ _id: 1 }).skip(100000).limit(20)
       # → MongoDB must SCAN and SKIP 100,000 documents → SLOW
       
       # RIGHT — range-based pagination
       db.customers.find({ _id: { $gt: lastSeenId } }).sort({ _id: 1 }).limit(20)
       # → Jumps directly to the right position using _id index
       
       # Redshift equivalent problem:
       # SELECT * FROM customers ORDER BY id OFFSET 100000 LIMIT 20
       # → Also slow in Redshift (must scan and skip rows)
       # → Redshift: use WHERE id > last_seen_id (same solution)

    ❌ QUERYING WITHOUT UNDERSTANDING DOCUMENT STRUCTURE
       # WRONG — assuming flat structure like Redshift
       db.customers.find({ state: "NY" })
       # → Returns NOTHING because state is NESTED inside address
       
       # RIGHT — use dot notation
       db.customers.find({ "address.state": "NY" })
       
       WHY: In Redshift, all columns are top-level
            In MongoDB, fields can be nested multiple levels deep
            You MUST know your document structure to query correctly
            Use db.customers.findOne() to inspect structure first

    ❌ USING $regex WITHOUT ANCHORING
       # WRONG — full collection scan
       db.customers.find({ name: { $regex: "Johnson" } })
       # → Scans every document (like LIKE '%Johnson%' in Redshift)
       
       # BETTER — anchor to start
       db.customers.find({ name: { $regex: "^Johnson" } })
       # → Can use index on name field (like LIKE 'Johnson%')
       
       # BEST — use text index for full-text search (Project 11)

    ❌ RELYING ON DOCUMENT ORDER IN find() RESULTS
       # WRONG — assuming documents come back in insertion order
       db.customers.find({})
       # → Order is NOT guaranteed without explicit .sort()
       # → MongoDB returns in "natural order" (internal storage order)
       # → Can change after updates, compaction, etc.
       
       # RIGHT — always sort if order matters
       db.customers.find({}).sort({ signup_date: -1 })
       
       # Redshift equivalent:
       # SELECT * FROM customers  → order not guaranteed without ORDER BY
       # Same rule applies in both systems

⚙️ HOW IT ONLY WORKS INTERNALLY

MongoDB Query Execution — What Happens During find()

WHEN YOU RUN: db.customers.find({ "address.state": "NY", total_spent: { $gt: 500 } })

STEP 1: QUERY PARSING
    → mongod receives the query
    → Parses the filter document into internal query representation
    → Identifies: field paths, operators, values
    → "address.state" → path: ["address", "state"] → value: "NY"
    → "total_spent" → operator: $gt → value: 500

STEP 2: QUERY PLANNING (OPTIMIZER)
    → Query planner evaluates available indexes
    → Currently: only _id index exists (auto-created in Project 1)
    → NO index on "address.state" or "total_spent"
    → Plan: COLLSCAN (collection scan — read EVERY document)
    
    ┌────────────────────────────────────────────────────────┐
    │ QUERY PLAN (no indexes):                                │
    │                                                         │
    │   COLLSCAN ──→ FILTER ──→ PROJECTION ──→ RETURN        │
    │   (read all)   (match     (strip         (send to       │
    │                 NY+>500)   fields)        client)       │
    │                                                         │
    │ REDSHIFT EQUIVALENT:                                    │
    │   SEQ SCAN ──→ FILTER ──→ PROJECT ──→ RETURN            │
    │   (without sort key benefit)                            │
    └────────────────────────────────────────────────────────┘

STEP 3: DOCUMENT SCANNING (COLLSCAN)
    → WiredTiger reads documents from B-tree leaf pages
    → Each document: BSON deserialized → fields extracted
    → CRITICAL DIFFERENCE FROM REDSHIFT:
    
    MongoDB COLLSCAN:
    ┌─────────────────────────────────────────────────────┐
    │ Doc 1: { _id, name, email, age, address, ... }      │ ← Read ENTIRE doc
    │ Doc 2: { _id, name, email, age, address, ... }      │ ← Read ENTIRE doc
    │ Doc 3: { _id, name, email, age, address, ... }      │ ← Read ENTIRE doc
    │ ... (read all 1000 documents)                        │
    │                                                      │
    │ For each doc:                                        │
    │   Navigate to address.state → check == "NY"          │
    │   Navigate to total_spent → check > 500              │
    │   If both match → include in results                 │
    └─────────────────────────────────────────────────────┘
    
    Redshift SEQ SCAN (for comparison):
    ┌─────────────────────────────────────────────────────┐
    │ Column "state":      [NY, WA, TX, NY, CA, ...]      │ ← Read ONLY this column
    │ Column "total_spent": [1250, 450, 3200, 89, ...]    │ ← Read ONLY this column
    │                                                      │
    │ Combine: row IDs where state=NY AND total_spent>500  │
    │ THEN read remaining columns for matching rows only   │
    │                                                      │
    │ MUCH LESS DATA READ FROM DISK                        │
    └─────────────────────────────────────────────────────┘
    
    THIS IS WHY:
    → Full table scans are FASTER in Redshift (columnar = read less data)
    → Point lookups are FASTER in MongoDB (document = one read)

STEP 4: CURSOR CREATION
    → Results NOT all loaded into memory at once
    → MongoDB creates a CURSOR (pointer to result set)
    → Client fetches results in BATCHES:
      → First batch: up to 101 documents (or 16MB, whichever comes first)
      → Subsequent batches: up to 16MB each
    → Cursor expires after 10 minutes of inactivity (configurable)
    
    REDSHIFT COMPARISON:
    → Redshift also uses cursors (DECLARE CURSOR)
    → But typically streams entire result set
    → MongoDB cursors are more like database cursors in traditional RDBMS

STEP 5: PROJECTION APPLICATION
    → AFTER documents are fetched from storage
    → BEFORE documents are sent over the network
    → Fields not in projection are stripped
    → Saves network bandwidth, NOT disk I/O
    → Exception: covered index queries (all fields in index — Project 10)
    
    REDSHIFT COMPARISON:
    → Redshift projection happens AT DISK LEVEL (columnar)
    → SELECT name, email → only name and email column files read
    → MongoDB: full document read → then name and email extracted
    → THIS IS A FUNDAMENTAL ARCHITECTURAL DIFFERENCE

Why Dot Notation Works — BSON Traversal

DOCUMENT ON DISK (BSON):
    {
        _id: ObjectId("..."),
        name: "Alice",
        address: {              ← nested BSON object
            city: "Austin",
            state: "TX"         ← the target field
        }
    }

BSON BINARY LAYOUT (simplified):
    [total_size][type:0x07][_id\0][ObjectId_bytes]
                [type:0x02][name\0][string_length][Alice\0]
                [type:0x03][address\0][sub_doc_size]
                    [type:0x02][city\0][string_length][Austin\0]
                    [type:0x02][state\0][string_length][TX\0]
                [0x00]  ← end of sub-document
    [0x00]  ← end of document

QUERY: { "address.state": "TX" }

TRAVERSAL:
    1. Parse dot notation: ["address", "state"]
    2. In BSON document: find field "address"
    3. "address" is type 0x03 (embedded document)
    4. Enter sub-document
    5. Find field "state" within sub-document
    6. Compare value "TX" with query value "TX" → MATCH

    THIS IS O(n) where n = number of fields at each level
    NOT O(1) like a hash lookup
    BSON is length-prefixed so fields can be skipped efficiently
    But deep nesting = more traversal = slightly slower

REDSHIFT COMPARISON:
    → Redshift: column "state" is a direct pointer to columnar block
    → O(1) to find the column, then sequential scan of values
    → No traversal needed — flat structure
    → MongoDB: must navigate document structure for each query

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 3:

PITFALL 1: "Forgetting dot notation for nested fields"
    # WRONG — field "state" doesn't exist at top level
    db.customers.find({ state: "NY" })           // returns NOTHING
    
    # RIGHT — navigate into nested object
    db.customers.find({ "address.state": "NY" }) // returns matches
    
    REDSHIFT: all columns are top-level → no navigation needed
    MONGODB: fields can be nested → MUST use dot notation

PITFALL 2: "Expecting find() to return all results at once"
    # In Python:
    results = collection.find({"address.state": "NY"})
    # results is a CURSOR, not a list!
    # It's lazy — fetches documents in batches
    
    # To get all results as a list:
    results_list = list(collection.find({"address.state": "NY"}))
    # WARNING: for millions of documents, this loads ALL into memory!
    
    # BETTER — iterate:
    for doc in collection.find({"address.state": "NY"}):
        process(doc)
    
    REDSHIFT: psql/JDBC typically fetches all rows into memory
    MONGODB: cursor is lazy — iterate to consume

PITFALL 3: "Projection confusion — include vs exclude"
    # WRONG — mixing include and exclude
    db.customers.find({}, { name: 1, phone: 0 })    // ERROR
    
    # RIGHT — include mode (list what you WANT)
    db.customers.find({}, { name: 1, email: 1, _id: 0 })
    
    # RIGHT — exclude mode (list what you DON'T want)
    db.customers.find({}, { preferences: 0, phone: 0 })
    
    # REMEMBER: _id: 0 is ALWAYS allowed regardless of mode

PITFALL 4: "Assuming projection saves disk I/O"
    REDSHIFT TRAINED YOU: SELECT name, email → reads only 2 column files
                          SELECT * → reads all column files
                          Huge difference in I/O and performance
    
    MONGODB REALITY: Projection ONLY saves network bandwidth
                     MongoDB ALWAYS reads the full document from storage
                     Then strips unwanted fields before sending to client
                     Exception: covered index queries (Project 10)
    
    IMPLICATION: 
    → In Redshift: projection is a MAJOR performance optimization
    → In MongoDB: projection is a MINOR optimization (network only)
    → Document DESIGN matters more than projection in MongoDB

PITFALL 5: "Using .sort() without index on large collections"
    db.customers.find({}).sort({ total_spent: -1 })
    
    Without index:
    → MongoDB loads ALL matching documents into memory
    → Sorts them in memory
    → IF result set > 100MB → SORT FAILS WITH ERROR:
      "Sort exceeded memory limit of 104857600 bytes"
    
    FIX OPTIONS:
    → Create index on sort field (Project 9)
    → Use allowDiskUse: true (slower but handles large sorts)
      db.customers.find({}).sort({total_spent:-1}).allowDiskUse()
    → Limit results before sorting (if appropriate)
    
    REDSHIFT: sorts are handled by disk-based merge sort across slices
              No memory limit for sorts — designed for massive data
              Sort key pre-sorts data → many sorts are nearly free

PITFALL 6: "Type mismatch in queries"
    # If age was stored as STRING (Project 2 CSV trap):
    db.customers.find({ age: { $gt: 30 } })    // returns NOTHING
    # Because "34" (string) is not > 30 (number)
    
    # Must match the STORED type:
    db.customers.find({ age: { $gt: "30" } })  // works but WRONG semantics
    # "9" > "30" in string comparison!
    
    REDSHIFT: types enforced by DDL → this can't happen
    MONGODB: types determined at insert time → must be consistent
    LESSON: Always verify types after loading (Project 2 verification)

📐 DATA MODELING COMPARISON

HOW DATA MODEL AFFECTS QUERIES:

REDSHIFT — FLAT TABLES + JOINS:
    
    Query: "Find premium customers from NY with their preferences"
    
    -- Must JOIN because preferences are in separate table
    SELECT c.name, c.email, p.newsletter, p.preferred_categories
    FROM customers c
    JOIN customer_preferences p ON c.customer_id = p.customer_id
    WHERE c.state = 'NY' AND c.is_premium = TRUE;
    
    → Requires JOIN across distribution keys
    → If DISTKEY mismatch → broadcast join (slow)
    → If DISTKEY match → co-located join (fast)

MONGODB — NESTED DOCUMENTS + NO JOIN:
    
    // Same query — no JOIN needed, all data in ONE document
    db.customers.find(
        {
            "address.state": "NY",
            is_premium: true,
            preferences: { $exists: true }
        },
        {
            name: 1,
            email: 1,
            "preferences.newsletter": 1,
            "preferences.preferred_categories": 1,
            _id: 0
        }
    )
    
    → ZERO joins — data is already together
    → Single document read per customer
    → Faster for this access pattern
    
    → BUT: if you need "all customers who share category X"
      MongoDB must scan all documents
      Redshift can use a reverse lookup table

LESSON:
    MongoDB's document model makes RELATED DATA QUERIES fast
    Redshift's normalized model makes CROSS-ENTITY ANALYTICS fast
    → Design your documents for YOUR most common queries

PAGE_SIZE = 5

# --- Approach A: skip/limit (simple but slow for large offsets) ---

    page = list(
        customers.find({}, {"name": 1, "_id": 0})
        .sort("name", 1)
        .skip(page_num * PAGE_SIZE)
        .limit(PAGE_SIZE)
    )
    names = [d["name"] for d in page]
    print(f"  Page {page_num + 1}: {names}")

# --- Approach B: range-based (fast, scalable) ---
print("\n--- 8b. Range-based pagination (RECOMMENDED) ---")
last_id = None
for page_num in range(3):
    if last_id is None:
        query = {}
    else:
        query = {"_id": {"$gt": last_id}}

    page = list(
        customers.find(query, {"name": 1, "_id": 1})
        .sort("_id", 1)
        .limit(PAGE_SIZE)
    )

    if not page:
        break

    last_id = page[-1]["_id"]
    names = [d["name"] for d in page]
    print(f"  Page {page_num + 1}: {names}")

KEY INSIGHT:
    → Without indexes: MongoDB queries degrade LINEARLY with data size
    → With indexes: MongoDB queries degrade LOGARITHMICALLY with data size
    → Indexes are NOT optional — they are ESSENTIAL for production
    → This is like Redshift sort keys: without them, scans are full table
    → Project 9 will fix this — but first we need to complete CRUD (Project 4-5)
    
REDSHIFT COMPARISON:
    → Redshift without sort keys: full table scan but COLUMNAR (still fast for analytics)
    → Redshift with sort keys: zone map skipping (fast for filtered queries)
    → MongoDB without indexes: COLLSCAN (slow for everything at scale)
    → MongoDB with indexes: IXSCAN (fast for targeted queries)
    → The PENALTY for missing indexes is WORSE in MongoDB than in Redshift
      because MongoDB reads full documents (not just columns)

📐 DATA MODELING COMPARISON

HOW DOCUMENT DESIGN AFFECTS QUERIES:

SCENARIO: "Find all customers who prefer Electronics"

REDSHIFT NORMALIZED MODEL:
    ┌───────────────────┐      ┌────────────────────────────┐
    │ customers          │      │ customer_preferences        │
    │ id | name | state  │      │ customer_id | category      │
    │ 1  | Alice | TX    │      │ 1           | Electronics   │
    │ 2  | Bob   | NY    │      │ 1           | Books         │
    │ 3  | Carol | CA    │      │ 2           | Clothing      │
    └───────────────────┘      └────────────────────────────┘
    
    SELECT c.* FROM customers c
    JOIN customer_preferences p ON c.id = p.customer_id
    WHERE p.category = 'Electronics';
    
    → Requires JOIN
    → Distribution key mismatch → broadcast join possible (slow)
    → Clean, normalized, but query requires coordination

MONGODB EMBEDDED MODEL:
    ┌─────────────────────────────────────────────────────────┐
    │ { name: "Alice", state: "TX",                           │
    │   preferences: { preferred_categories: ["Electronics",   │
    │                                         "Books"] } }     │
    │ { name: "Bob", state: "NY",                              │
    │   preferences: { preferred_categories: ["Clothing"] } }  │
    │ { name: "Carol", state: "CA" }  ← no preferences at all │
    └─────────────────────────────────────────────────────────┘
    
    db.customers.find({ "preferences.preferred_categories": "Electronics" })
    
    → ZERO joins
    → Single collection scan (or index scan)
    → Array query is native and fast
    → BUT: "what are ALL distinct categories?" requires scanning all documents
           vs Redshift: SELECT DISTINCT category FROM preferences (one column scan)
