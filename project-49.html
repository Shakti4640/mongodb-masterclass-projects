Your Typical Redshift Monitoring Queries

-- REDSHIFT: Find slow queries (last 24 hours)
SELECT 
    query,
    userid,
    TRIM(querytxt) AS sql_text,
    starttime,
    endtime,
    DATEDIFF(seconds, starttime, endtime) AS duration_sec,
    aborted
FROM stl_query
WHERE starttime > DATEADD(hour, -24, GETDATE())
  AND duration_sec > 30
ORDER BY duration_sec DESC
LIMIT 20;

-- REDSHIFT: Check table skew (data distribution problems)
SELECT 
    "table" AS table_id,
    "schema" AS schema_name,
    tbl_rows,
    skew_rows,
    skew_sortkey1,
    unsorted
FROM svv_table_info
WHERE "schema" = 'store'
ORDER BY skew_rows DESC;

-- REDSHIFT: WLM queue saturation
SELECT 
    service_class,
    num_queued_queries,
    num_executing_queries,
    queue_elapsed_time
FROM stv_wlm_service_class_state
WHERE service_class > 4;

-- REDSHIFT: Disk space alert
SELECT 
    SUM(used) AS used_mb,
    SUM(capacity) AS capacity_mb,
    ROUND(100.0 * SUM(used) / SUM(capacity), 2) AS pct_used
FROM stv_partitions
WHERE part_begin = 0;

-- REDSHIFT: Lock contention
SELECT 
    l.pid,
    l.table_id,
    t.name AS table_name,
    l.lock_mode,
    l.granted
FROM pg_locks l
JOIN stv_tbl_perm t ON l.table_id = t.id
WHERE l.granted = false;

🧩 HOW TO APPROACH IT IN MONGODB

MONITORING PHILOSOPHY COMPARISON:

REDSHIFT APPROACH (what you're used to):
    → Reactive: query STL tables when something seems slow
    → Periodic: CloudWatch dashboards checked occasionally
    → Batch-oriented: problems discovered after queries finish
    → DBA-centric: requires SQL knowledge to interrogate system tables
    → Limited knobs: WLM configuration is the main tuning lever

MONGODB APPROACH (what you need to learn):
    → Proactive: Performance Advisor catches issues before impact
    → Real-time: RTPP shows LIVE operation patterns
    → Continuous: alerts fire on threshold breaches automatically
    → Ops-friendly: Atlas UI requires no MQL knowledge for basics
    → Many knobs: read/write concern, profiler level, index strategy,
                   connection pool, WiredTiger cache, read preference

THE MONITORING HIERARCHY (apply to both systems):

    Level 5: BUSINESS METRICS     ← "Are users happy?"
        ↑     (API latency, error rates, conversion rates)
    Level 4: APPLICATION METRICS  ← "Is the app healthy?"
        ↑     (response times, queue depths, throughput)
    Level 3: DATABASE METRICS     ← "Is the database healthy?" ← THIS PROJECT
        ↑     (query times, connections, replication, cache)
    Level 2: INFRASTRUCTURE       ← "Are the machines healthy?"
        ↑     (CPU, memory, disk, network)
    Level 1: AVAILABILITY         ← "Is it up?"
              (ping, port check, health endpoint)

    YOU MUST MONITOR ALL LEVELS
    Most teams only monitor Level 1-2 and wonder why things break

Part 2: The Database Profiler — MongoDB's Slow Query Log

// ═══════════════════════════════════════════════════════
// DATABASE PROFILER — Equivalent to Redshift STL_ALERT_EVENT_LOG
// ═══════════════════════════════════════════════════════

// ─── PROFILER LEVELS ─────────────────────────────────
// Level 0: OFF (default) — no profiling
// Level 1: SLOW OPERATIONS ONLY — log operations exceeding threshold
// Level 2: ALL OPERATIONS — log everything (HUGE overhead — dev only)
//
// REDSHIFT EQUIVALENT:
//   STL_QUERY logs ALL completed queries automatically
//   No equivalent of "only log slow ones"
//   Redshift Query Monitoring Rules can LOG queries matching criteria

// ─── ENABLE PROFILER (Level 1 — slow operations) ────
db.setProfilingLevel(1, { slowms: 100 })
// Log any operation taking > 100ms
// Output: { was: 0, slowms: 100, ... }

// ADVANCED: Profile only specific operations
db.setProfilingLevel(1, {
    slowms: 50,
    filter: {
        "op": { "$in": ["query", "update"] },   // only queries and updates
        "ns": { "$nin": ["ecommerce.logs"] }     // exclude logs collection
    }
})

// ─── CHECK PROFILER STATUS ───────────────────────────
db.getProfilingStatus()
// { was: 1, slowms: 100, ... }

// ─── QUERY PROFILER OUTPUT ───────────────────────────
// Profiled operations are stored in: system.profile (capped collection)

db.system.profile.find({
    ts: { $gt: new Date(Date.now() - 3600000) }  // last hour
}).sort({ millis: -1 }).limit(10)
// Output per slow operation:
// {
//   op: "query",
//   ns: "ecommerce.orders",
//   command: {
//     find: "orders",
//     filter: { status: "pending" },
//     sort: { created_at: -1 },
//     limit: 100
//   },
//   keysExamined: 0,           ← no index used!
//   docsExamined: 5000000,     ← scanned 5M documents
//   nreturned: 100,            ← returned only 100
//   millis: 3847,              ← took 3.8 seconds
//   planSummary: "COLLSCAN",   ← collection scan (WORST case)
//   ts: ISODate("2024-06-15T14:30:22Z"),
//   responseLength: 48293
// }

// ─── THE GOLDEN RATIO: Query Targeting ──────────────
// docsExamined / nreturned = query targeting ratio
// IDEAL: ratio close to 1.0 (examine only what you return)
// BAD:   ratio > 100 (examining 100x more than returning)
// TERRIBLE: ratio > 1000 (collection scan territory)
//
// In this example: 5000000 / 100 = 50000 ← TERRIBLE
// FIX: createIndex({ status: 1, created_at: -1 })
//
// REDSHIFT EQUIVALENT CONCEPT:
//   Rows scanned vs rows returned in EXPLAIN output
//   Zone map effectiveness (blocks skipped vs blocks scanned)

// ─── FIND WORST QUERY SHAPES ─────────────────────────
// Group slow queries by shape (query pattern)
db.system.profile.aggregate([
    { $match: { millis: { $gt: 100 } } },
    { $group: {
        _id: {
            op: "$op",
            ns: "$ns",
            planSummary: "$planSummary"
        },
        count: { $sum: 1 },
        avgMillis: { $avg: "$millis" },
        maxMillis: { $max: "$millis" },
        totalMillis: { $sum: "$millis" },
        avgDocsExamined: { $avg: "$docsExamined" },
        avgKeysExamined: { $avg: "$keysExamined" }
    }},
    { $sort: { totalMillis: -1 } },
    { $limit: 10 }
])
// This tells you: "Which query PATTERNS cause the most total pain?"
//
// REDSHIFT EQUIVALENT:
//   SELECT TRIM(querytxt), COUNT(*), AVG(elapsed)/1000000 AS avg_sec
//   FROM stl_query
//   WHERE elapsed > 30000000  -- > 30 seconds
//   GROUP BY TRIM(querytxt)
//   ORDER BY COUNT(*) * AVG(elapsed) DESC
//   LIMIT 10;


// ─── DISABLE PROFILER WHEN DONE ──────────────────────
db.setProfilingLevel(0)
// IMPORTANT: Level 1 has ~5% overhead, Level 2 has ~30% overhead

from pymongo import MongoClient
from datetime import datetime, timedelta
import time
import json

class MongoDBHealthMonitor:
    """
    Production MongoDB health monitor
    Checks all critical metrics and generates alerts
    
    Redshift equivalent: Custom SQL scripts querying STL/STV tables
    + CloudWatch alarm definitions
    """
    
    def __init__(self, connection_string, alert_callback=None):
        self.client = MongoClient(connection_string)
        self.admin_db = self.client.admin
        self.alert_callback = alert_callback or self._default_alert
        
        # ── THRESHOLDS (tune per your workload) ─────────
        self.thresholds = {
            # Connection thresholds
            "connection_utilization_warn": 70,      # percent
            "connection_utilization_critical": 85,
            
            # Cache thresholds
            "cache_utilization_warn": 80,           # percent
            "cache_dirty_warn": 15,                 # percent
            "cache_dirty_critical": 25,
            
            # Ticket thresholds (WiredTiger concurrency)
            "read_tickets_available_warn": 20,
            "read_tickets_available_critical": 5,
            "write_tickets_available_warn": 20,
            "write_tickets_available_critical": 5,
            
            # Replication thresholds
            "replication_lag_warn_seconds": 5,
            "replication_lag_critical_seconds": 15,
            "oplog_window_warn_hours": 24,
            "oplog_window_critical_hours": 8,
            
            # Query targeting
            "query_targeting_warn": 100,            # docs examined / returned
            "query_targeting_critical": 1000,
            
            # Operation latency
            "read_latency_warn_ms": 50,
            "read_latency_critical_ms": 200,
            "write_latency_warn_ms": 20,
            "write_latency_critical_ms": 100,
            
            # Disk
            "disk_utilization_warn": 75,            # percent
            "disk_utilization_critical": 90,
        }
        
        # ── Previous sample for rate calculations ────────
        self._prev_status = None
        self._prev_time = None
    
    def _default_alert(self, severity, component, message, value, threshold):
        """Default alert handler — print to console"""
        timestamp = datetime.now().isoformat()
        alert = {
            "timestamp": timestamp,
            "severity": severity,        # "WARN" or "CRITICAL"
            "component": component,
            "message": message,
            "current_value": value,
            "threshold": threshold
        }
        prefix = "🟡" if severity == "WARN" else "🔴"
        print(f"{prefix} [{severity}] {component}: {message} "
              f"(value={value}, threshold={threshold})")
        return alert
    
    # ─── CONNECTION HEALTH ────────────────────────────
    def check_connections(self, server_status):
        """
        Check connection pool health
        
        Redshift equivalent:
            SELECT COUNT(*) FROM stv_sessions;
            CloudWatch: DatabaseConnections
        """
        conn = server_status["connections"]
        current = conn["current"]
        available = conn["available"]
        total_capacity = current + available
        utilization = (current / total_capacity) * 100
        
        result = {
            "current": current,
            "available": available,
            "total_capacity": total_capacity,
            "utilization_pct": round(utilization, 2),
            "active": conn.get("active", 0),
            "status": "OK"
        }
        
        if utilization > self.thresholds["connection_utilization_critical"]:
            result["status"] = "CRITICAL"
            self.alert_callback(
                "CRITICAL", "connections",
                f"Connection utilization at {utilization:.1f}%",
                round(utilization, 1),
                self.thresholds["connection_utilization_critical"]
            )
        elif utilization > self.thresholds["connection_utilization_warn"]:
            result["status"] = "WARN"
            self.alert_callback(
                "WARN", "connections",
                f"Connection utilization at {utilization:.1f}%",
                round(utilization, 1),
                self.thresholds["connection_utilization_warn"]
            )
        
        return result
    
    # ─── WIREDTIGER CACHE HEALTH ─────────────────────
    def check_cache(self, server_status):
        """
        Check WiredTiger cache utilization and eviction pressure
        
        Redshift equivalent:
            No direct equivalent — Redshift manages block cache internally
            Closest: checking if queries spill to disk
        """
        cache = server_status["wiredTiger"]["cache"]
        
        bytes_in_cache = cache["bytes currently in the cache"]
        max_bytes = cache["maximum bytes configured"]
        dirty_bytes = cache["tracked dirty bytes in the cache"]
        
        cache_pct = (bytes_in_cache / max_bytes) * 100
        dirty_pct = (dirty_bytes / max_bytes) * 100
        
        # Eviction metrics
        evicted_unmodified = cache.get(
            "unmodified pages evicted", 0)
        evicted_modified = cache.get(
            "modified pages evicted", 0)
        
        result = {
            "cache_used_gb": round(bytes_in_cache / (1024**3), 2),
            "cache_max_gb": round(max_bytes / (1024**3), 2),
            "cache_utilization_pct": round(cache_pct, 2),
            "dirty_pct": round(dirty_pct, 2),
            "evicted_clean_pages": evicted_unmodified,
            "evicted_dirty_pages": evicted_modified,
            "status": "OK"
        }
        
        # Cache utilization alerts
        if cache_pct > self.thresholds["cache_utilization_warn"]:
            result["status"] = "WARN"
            self.alert_callback(
                "WARN", "wiredtiger_cache",
                f"Cache utilization at {cache_pct:.1f}%",
                round(cache_pct, 1),
                self.thresholds["cache_utilization_warn"]
            )
        
        # Dirty cache alerts
        if dirty_pct > self.thresholds["cache_dirty_critical"]:
            result["status"] = "CRITICAL"
            self.alert_callback(
                "CRITICAL", "wiredtiger_cache",
                f"Dirty cache at {dirty_pct:.1f}% — "
                f"checkpointing may be falling behind",
                round(dirty_pct, 1),
                self.thresholds["cache_dirty_critical"]
            )
        elif dirty_pct > self.thresholds["cache_dirty_warn"]:
            result["status"] = "WARN" if result["status"] == "OK" else result["status"]
            self.alert_callback(
                "WARN", "wiredtiger_cache",
                f"Dirty cache at {dirty_pct:.1f}%",
                round(dirty_pct, 1),
                self.thresholds["cache_dirty_warn"]
            )
        
        return result
    
    # ─── WIREDTIGER TICKETS (CONCURRENCY) ─────────────
    def check_tickets(self, server_status):
        """
        Check WiredTiger read/write ticket availability
        
        Redshift equivalent:
            WLM queue slots
            SELECT num_executing_queries, num_queued_queries
            FROM stv_wlm_service_class_state;
        """
        tickets = server_status["wiredTiger"]["concurrentTransactions"]
        
        read_avail = tickets["read"]["available"]
        read_total = tickets["read"]["totalTickets"]
        write_avail = tickets["write"]["available"]
        write_total = tickets["write"]["totalTickets"]
        
        result = {
            "read_available": read_avail,
            "read_total": read_total,
            "read_in_use": read_total - read_avail,
            "write_available": write_avail,
            "write_total": write_total,
            "write_in_use": write_total - write_avail,
            "status": "OK"
        }
        
        if read_avail < self.thresholds["read_tickets_available_critical"]:
            result["status"] = "CRITICAL"
            self.alert_callback(
                "CRITICAL", "wiredtiger_tickets",
                f"Only {read_avail} read tickets available "
                f"(of {read_total})",
                read_avail,
                self.thresholds["read_tickets_available_critical"]
            )
        elif read_avail < self.thresholds["read_tickets_available_warn"]:
            result["status"] = "WARN"
            self.alert_callback(
                "WARN", "wiredtiger_tickets",
                f"Only {read_avail} read tickets available",
                read_avail,
                self.thresholds["read_tickets_available_warn"]
            )
        
        if write_avail < self.thresholds["write_tickets_available_critical"]:
            result["status"] = "CRITICAL"
            self.alert_callback(
                "CRITICAL", "wiredtiger_tickets",
                f"Only {write_avail} write tickets available "
                f"(of {write_total})",
                write_avail,
                self.thresholds["write_tickets_available_critical"]
            )
        
        return result
    
    # ─── REPLICATION HEALTH ──────────────────────────
    def check_replication(self):
        """
        Check replica set health and replication lag
        
        Redshift equivalent:
            Multi-AZ is AWS-managed — NO user visibility
            THIS is a MongoDB advantage: full replication transparency
        """
        try:
            rs_status = self.admin_db.command("replSetGetStatus")
        except Exception as e:
            return {
                "status": "UNKNOWN",
                "error": f"Not a replica set or cannot check: {e}"
            }
        
        members = rs_status.get("members", [])
        primary_optime = None
        result = {
            "set_name": rs_status.get("set", "unknown"),
            "members": [],
            "max_lag_seconds": 0,
            "status": "OK"
        }
        
        # Find primary's optime
        for member in members:
            if member.get("stateStr") == "PRIMARY":
                primary_optime = member.get("optimeDate")
                break
        
        for member in members:
            member_info = {
                "name": member["name"],
                "state": member["stateStr"],
                "health": member["health"],
                "uptime_hours": round(
                    member.get("uptime", 0) / 3600, 1),
                "ping_ms": member.get("pingMs", 0)
            }
            
            # Check member health
            if member["health"] != 1:
                result["status"] = "CRITICAL"
                self.alert_callback(
                    "CRITICAL", "replication",
                    f"Member {member['name']} is DOWN "
                    f"(state: {member['stateStr']})",
                    0, 1
                )
            
            # Calculate replication lag for secondaries
            if (member.get("stateStr") == "SECONDARY"
                    and primary_optime):
                member_optime = member.get("optimeDate")
                if member_optime and primary_optime:
                    lag = (primary_optime -
                           member_optime).total_seconds()
                    member_info["lag_seconds"] = round(lag, 2)
                    result["max_lag_seconds"] = max(
                        result["max_lag_seconds"], lag)
                    
                    if lag > self.thresholds[
                            "replication_lag_critical_seconds"]:
                        result["status"] = "CRITICAL"
                        self.alert_callback(
                            "CRITICAL", "replication",
                            f"Secondary {member['name']} lag: "
                            f"{lag:.1f}s",
                            round(lag, 1),
                            self.thresholds[
                                "replication_lag_critical_seconds"]
                        )
                    elif lag > self.thresholds[
                            "replication_lag_warn_seconds"]:
                        result["status"] = max(
                            result["status"], "WARN",
                            key=lambda x: {"OK": 0, "WARN": 1,
                                         "CRITICAL": 2}[x])
                        self.alert_callback(
                            "WARN", "replication",
                            f"Secondary {member['name']} lag: "
                            f"{lag:.1f}s",
                            round(lag, 1),
                            self.thresholds[
                                "replication_lag_warn_seconds"]
                        )
            
            result["members"].append(member_info)
        
        return result
    
    # ─── OPERATION RATES (ops/sec) ────────────────────
    def check_operation_rates(self, server_status):
        """
        Calculate current operations per second by type
        
        Redshift equivalent:
            CloudWatch: QueriesCompletedPerSecond
            Custom: COUNT(*) from stl_query per minute
        """
        current_counters = server_status["opcounters"]
        current_time = time.time()
        
        result = {
            "insert_per_sec": 0,
            "query_per_sec": 0,
            "update_per_sec": 0,
            "delete_per_sec": 0,
            "total_per_sec": 0,
            "status": "OK"
        }
        
        if self._prev_status:
            elapsed = current_time - self._prev_time
            if elapsed > 0:
                prev = self._prev_status["opcounters"]
                for op in ["insert", "query", "update", "delete"]:
                    rate = (current_counters[op] -
                            prev[op]) / elapsed
                    result[f"{op}_per_sec"] = round(rate, 1)
                result["total_per_sec"] = round(
                    sum(result[f"{op}_per_sec"]
                        for op in ["insert", "query",
                                   "update", "delete"]), 1)
        
        self._prev_status = server_status
        self._prev_time = current_time
        
        return result
    
    # ─── LONG RUNNING OPERATIONS ─────────────────────
    def check_long_running_ops(self, threshold_seconds=30):
        """
        Find operations running longer than threshold
        
        Redshift equivalent:
            SELECT pid, user_name, query, starttime,
                   DATEDIFF(sec, starttime, GETDATE()) as runtime
            FROM stv_recents
            WHERE status = 'Running'
              AND DATEDIFF(sec, starttime, GETDATE()) > 30;
        """
        long_ops = list(self.admin_db.aggregate([
            {"$currentOp": {"allUsers": True, "idleConnections": False}},
            {"$match": {
                "active": True,
                "secs_running": {"$gt": threshold_seconds},
                "op": {"$ne": "none"}
            }},
            {"$project": {
                "opid": 1,
                "secs_running": 1,
                "op": 1,
                "ns": 1,
                "command": 1,
                "planSummary": 1,
                "client": 1,
                "appName": 1
            }},
            {"$sort": {"secs_running": -1}}
        ]))
        
        result = {
            "count": len(long_ops),
            "operations": [],
            "status": "OK"
        }
        
        for op in long_ops:
            op_info = {
                "opid": op.get("opid"),
                "seconds_running": op.get("secs_running", 0),
                "operation": op.get("op"),
                "namespace": op.get("ns"),
                "plan": op.get("planSummary", "unknown"),
                "client": op.get("client", "unknown"),
                "app": op.get("appName", "unknown")
            }
            result["operations"].append(op_info)
            
            # Alert on COLLSCAN operations running long
            if (op.get("planSummary", "").startswith("COLLSCAN")
                    and op.get("secs_running", 0) > 60):
                result["status"] = "CRITICAL"
                self.alert_callback(
                    "CRITICAL", "long_running_ops",
                    f"COLLSCAN running {op['secs_running']}s on "
                    f"{op.get('ns', 'unknown')}",
                    op["secs_running"], threshold_seconds
                )
            elif len(long_ops) > 5:
                result["status"] = "WARN"
        
        return result
    
    # ─── COLLECTION HEALTH CHECK ─────────────────────
    def check_collection_health(self, db_name, collection_name):
        """
        Check specific collection metrics
        
        Redshift equivalent:
            SELECT * FROM svv_table_info
            WHERE "table" = 'orders';
        """
        db = self.client[db_name]
        stats = db.command("collStats", collection_name)
        
        # Index stats
        indexes = list(
            db[collection_name].aggregate([{"$indexStats": {}}]))
        unused_indexes = [
            idx for idx in indexes
            if idx["accesses"]["ops"] == 0
            and idx["name"] != "_id_"
        ]
        
        result = {
            "namespace": f"{db_name}.{collection_name}",
            "document_count": stats.get("count", 0),
            "avg_doc_size_bytes": stats.get("avgObjSize", 0),
            "data_size_mb": round(
                stats.get("size", 0) / (1024**2), 2),
            "storage_size_mb": round(
                stats.get("storageSize", 0) / (1024**2), 2),
            "index_count": stats.get("nindexes", 0),
            "total_index_size_mb": round(
                stats.get("totalIndexSize", 0) / (1024**2), 2),
            "unused_indexes": [

            "status": "OK"
        }

                f"{stats['nindexes']} indexes — "
                f"high write overhead",

            )
        
        # Alert on unused indexes
        if len(unused_indexes) > 0:
            self.alert_callback(
                "WARN", "collection_health",
                f"{db_name}.{collection_name} has "
                f"{len(unused_indexes)} unused indexes: "
                f"{[i['name'] for i in unused_indexes]}",
                len(unused_indexes), 0
            )
        
        return result

import requests

class AtlasAlertManager:
    """

    via boto3 or AWS CLI
    """
    
    def __init__(self, public_key, private_key, group_id):
        self.base_url = (
            f"https://cloud.mongodb.com/api/atlas/v2"
            f"/groups/{group_id}"
        )
        self.auth = HTTPDigestAuth(public_key, private_key)
        self.headers = {
            "Content-Type": "application/json",
            "Accept": "application/vnd.atlas.2023-11-15+json"

    def create_alert_config(self, alert_config):
        """Create a new alert configuration"""
        response = requests.post(
            f"{self.base_url}/alertConfigs",
            auth=self.auth,
            headers=self.headers,
            json=alert_config

    # ─── MONGODB HEALTH ──────────────────────────────
    def check_mongodb(self) -> ComponentHealth:
        """Comprehensive MongoDB health check"""
        try:
            status = self.mongo_admin.command("serverStatus")
        except Exception as e:
            return ComponentHealth(
                name="MongoDB",
                status="CRITICAL",
                alerts=[f"Cannot connect: {e}"]
            )
        
        health = ComponentHealth(name="MongoDB", status="OK")
        
        # Connections
        conn = status["connections"]
        conn_pct = conn["current"] / (
            conn["current"] + conn["available"]) * 100
        health.metrics["connections_current"] = conn["current"]
        health.metrics["connections_pct"] = round(conn_pct, 1)
        
        if conn_pct > 85:
            health.status = "CRITICAL"
            health.alerts.append(
                f"Connection utilization: {conn_pct:.1f}%")
        elif conn_pct > 70:
            health.status = "WARN"
            health.alerts.append(
                f"Connection utilization: {conn_pct:.1f}%")
        
        # WiredTiger Cache
        cache = status["wiredTiger"]["cache"]
        cache_used = cache["bytes currently in the cache"]
        cache_max = cache["maximum bytes configured"]
        cache_pct = (cache_used / cache_max) * 100
        dirty_bytes = cache["tracked dirty bytes in the cache"]
        dirty_pct = (dirty_bytes / cache_max) * 100
        
        health.metrics["cache_used_gb"] = round(
            cache_used / (1024**3), 2)
        health.metrics["cache_pct"] = round(cache_pct, 1)
        health.metrics["cache_dirty_pct"] = round(dirty_pct, 1)
        
        if cache_pct > 95:
            health.status = "CRITICAL"
            health.alerts.append(
                f"WT Cache: {cache_pct:.1f}%")
        if dirty_pct > 25:
            health.status = "CRITICAL"
            health.alerts.append(
                f"WT Dirty Cache: {dirty_pct:.1f}%")
        
        # Tickets
        tickets = status["wiredTiger"]["concurrentTransactions"]
        read_avail = tickets["read"]["available"]
        write_avail = tickets["write"]["available"]
        health.metrics["read_tickets_available"] = read_avail
        health.metrics["write_tickets_available"] = write_avail
        
        if read_avail < 5 or write_avail < 5:
            health.status = "CRITICAL"
            health.alerts.append(
                f"Ticket exhaustion: read={read_avail} "
                f"write={write_avail}")
        elif read_avail < 20 or write_avail < 20:
            health.status = max(
                health.status, "WARN",
                key=lambda x: {"OK": 0, "WARN": 1,
                             "CRITICAL": 2}[x])
            health.alerts.append(
                f"Tickets low: read={read_avail} "
                f"write={write_avail}")
        
        # Operations per second
        opcounters = status["opcounters"]
        health.metrics["total_ops"] = sum(
            opcounters[op]
            for op in ["insert", "query", "update", "delete"]
        )
        
        # Replication
        try:
            rs = self.mongo_admin.command("replSetGetStatus")
            primary_optime = None
            max_lag = 0
            
            for member in rs.get("members", []):
                if member["stateStr"] == "PRIMARY":
                    primary_optime = member.get("optimeDate")
                if member["health"] != 1:
                    health.status = "CRITICAL"
                    health.alerts.append(
                        f"Member DOWN: {member['name']}")
            
            for member in rs.get("members", []):
                if (member["stateStr"] == "SECONDARY"
                        and primary_optime):
                    optime = member.get("optimeDate")
                    if optime:
                        lag = (primary_optime -
                               optime).total_seconds()
                        max_lag = max(max_lag, lag)
            
            health.metrics["max_replication_lag_sec"] = round(
                max_lag, 2)
            
            if max_lag > 15:
                health.status = "CRITICAL"
                health.alerts.append(
                    f"Replication lag: {max_lag:.1f}s")
            elif max_lag > 5:
                health.status = max(
                    health.status, "WARN",
                    key=lambda x: {"OK": 0, "WARN": 1,
                                 "CRITICAL": 2}[x])
                health.alerts.append(
                    f"Replication lag: {max_lag:.1f}s")
        except Exception:
            health.metrics["replication"] = "standalone"
        
        return health

    def check_cdc_pipeline(self) -> ComponentHealth:
        """
        End-to-end CDC pipeline health
        MongoDB → Kafka Connector → Kafka → Sink Connector → Redshift
        
        This is the GLUE between MongoDB and Redshift
        from Projects 34, 35, 44
        """
        health = ComponentHealth(
            name="CDC Pipeline", status="OK")
        
        try:
            # Check MongoDB oplog window
            # If oplog window is too small, connector can lose position
            try:
                rs = self.mongo_admin.command("replSetGetStatus")
                primary = next(
                    (m for m in rs["members"]
                     if m["stateStr"] == "PRIMARY"), None)
                
                if primary:
                    local_db = self.mongo_client["local"]
                    oplog = local_db["oplog.rs"]
                    
                    first_entry = oplog.find().sort(
                        "$natural", 1).limit(1)
                    last_entry = oplog.find().sort(
                        "$natural", -1).limit(1)
                    
                    first = next(first_entry, None)
                    last = next(last_entry, None)
                    
                    if first and last:
                        first_ts = first["ts"].time
                        last_ts = last["ts"].time
                        oplog_window_hours = (
                            last_ts - first_ts) / 3600
                        
                        health.metrics[
                            "oplog_window_hours"] = round(
                            oplog_window_hours, 1)
                        
                        if oplog_window_hours < 8:
                            health.status = "CRITICAL"
                            health.alerts.append(
                                f"Oplog window: "
                                f"{oplog_window_hours:.1f}h "
                                f"— connector may lose position")
                        elif oplog_window_hours < 24:
                            health.status = "WARN"
                            health.alerts.append(
                                f"Oplog window: "
                                f"{oplog_window_hours:.1f}h")
            except Exception as e:
                health.metrics["oplog_check"] = f"error: {e}"
            
            # Check for stale data in Redshift
            # If latest MongoDB data is much newer than Redshift data,
            # pipeline is lagging
            try:
                mongo_db = self.mongo_client["ecommerce"]
                latest_mongo = mongo_db.orders.find_one(
                    sort=[("updated_at", -1)],
                    projection={"updated_at": 1}
                )
                
                if latest_mongo:
                    mongo_latest = latest_mongo.get(
                        "updated_at", datetime.now())
                    health.metrics[
                        "mongo_latest_update"] = (
                        mongo_latest.isoformat())
                    
                    # In production: also query Redshift for latest
                    # record timestamp and compare
                    # lag = mongo_latest - redshift_latest
            except Exception as e:
                health.metrics["freshness_check"] = f"error: {e}"
        
        except Exception as e:
            health.status = "CRITICAL"
            health.alerts.append(
                f"CDC pipeline check failed: {e}")
        
        return health

✅ WHAT SHOULD DO (Best Practices)

MONITORING BEST PRACTICES — MONGODB:

    ✅ IMPLEMENT THE "FOUR GOLDEN SIGNALS" (Google SRE):
       1. Latency:     read/write operation time
       2. Traffic:      operations per second
       3. Errors:       failed operations, timeout rate
       4. Saturation:   cache utilization, ticket availability, disk

    ✅ ALERT ON SYMPTOMS, NOT CAUSES
       → GOOD: "Read latency > 100ms for 5 minutes"
       → BAD:  "CPU > 80%" (CPU can be 80% and system is fine)
       → GOOD: "Replication lag > 10 seconds"
       → BAD:  "Disk IOPS > 3000" (might be normal for your workload)

    ✅ USE TIERED ALERTING
       → INFO:     Log only (query targeting ratio > 50)
       → WARN:     Slack notification (replication lag > 5s)
       → CRITICAL: PagerDuty page (no primary, host down)

    ✅ SET ALERT THRESHOLDS BASED ON BASELINES
       → Run for 2 weeks → establish normal ranges
       → Set WARN at 1.5x normal, CRITICAL at 2x normal
       → Not arbitrary numbers

    ✅ MONITOR THE CDC PIPELINE END-TO-END
       → Not just MongoDB, not just Kafka, not just Redshift
       → Monitor DATA FRESHNESS: "How old is the newest record
         in Redshift compared to MongoDB?"
       → This is the metric that matters to the business

    ✅ ENABLE DATABASE PROFILER STRATEGICALLY
       → Production: Level 1, slowms=200
       → Staging: Level 1, slowms=50
       → Debug session: Level 2 (TEMPORARILY — heavy overhead)
       → Atlas: Use built-in Query Profiler (no performance impact)

    ✅ REVIEW PERFORMANCE ADVISOR WEEKLY
       → Treat it like a code review — mandatory, scheduled
       → Create recommended indexes in staging → test → promote

❌ WHAT SHOULD NOT DO (Anti-Patterns)

MONITORING ANTI-PATTERNS — ESPECIALLY FOR REDSHIFT DEVELOPERS:

    ❌ ALERTING ON EVERYTHING
       → "CPU > 50%" → you'll get 100 alerts/day
       → Alert fatigue = missed real problems
       → REDSHIFT TRAP: you're used to CloudWatch being noisy
         and ignoring most alarms

    ❌ RUNNING db.serverStatus() EVERY SECOND
       → serverStatus itself has overhead
       → 10-30 second intervals are sufficient
       → Atlas provides 10-second granularity natively

    ❌ ENABLING PROFILER LEVEL 2 IN PRODUCTION
       → Logs EVERY operation → massive I/O overhead (30%+)
       → System.profile collection grows rapidly
       → Use Level 1 with appropriate slowms threshold

    ❌ IGNORING QUERY TARGETING RATIO
       → Redshift developers don't have this concept
       → In Redshift: full table scan is NORMAL (columnar = efficient)
       → In MongoDB: full collection scan is CATASTROPHIC
       → docsExamined:docsReturned of 10000:1 means you're reading
         10,000 documents to return 1
       → THIS IS THE #1 METRIC REDSHIFT DEVELOPERS IGNORE IN MONGODB
       → Treat query targeting ratio like Redshift treats skew_rows

    ❌ MONITORING ONLY THE PRIMARY NODE
       → Redshift: leader node is the single point of monitoring
       → MongoDB: secondaries have INDEPENDENT health
       → A secondary with bad disk I/O will cause:
         ├── Replication lag → stale reads if readPreference=secondary
         ├── Failed elections → if primary dies, slow failover
         └── Oplog falling behind → potential full resync needed
       → MONITOR EVERY MEMBER of every replica set

    ❌ TREATING MongoDB MONITORING LIKE Redshift MONITORING
       → Redshift: check once per hour, mostly batch analytics
       → MongoDB: check every 30 seconds, real-time operational
       → Redshift: "query took 45 seconds" = normal
       → MongoDB: "query took 45 seconds" = EMERGENCY
       → Different latency expectations = different monitoring cadence

    ❌ IGNORING INDEX SIZE IN MONITORING
       → Redshift: no indexes to worry about
       → MongoDB: indexes consume RAM (WiredTiger cache)
       → 50 indexes on a collection = massive write penalty + cache pressure
       → MUST monitor: total index size vs available cache
       → If indexes alone exceed 60% of cache: you're in trouble

    ❌ NOT MONITORING OPLOG WINDOW
       → Redshift: no concept of oplog
       → MongoDB: oplog is the LIFELINE of replication AND change streams
       → If oplog window shrinks below your longest possible outage:
         ├── Secondary falls behind → requires FULL RESYNC (hours)
         ├── Change stream loses position → missed events → data loss
         └── Kafka connector loses resume token → full re-snapshot
       → This is a SILENT KILLER — everything works until it doesn't

    ❌ USING SHELL COMMANDS AS YOUR ONLY MONITORING
       → Running db.serverStatus() manually = NOT monitoring
       → That's like checking Redshift by running SQL once a day
       → You need AUTOMATED, CONTINUOUS, ALERTING monitoring
       → Atlas provides this free — use it
       → Self-managed: Prometheus + MongoDB exporter + Grafana

    ❌ SKIPPING COLLECTION-LEVEL MONITORING
       → Monitoring only server-level metrics misses:
         ├── One collection growing unboundedly (no TTL, no archival)
         ├── One collection with 30 unused indexes
         ├── One collection's average document size doubling
         └── One collection becoming the hot spot
       → Schedule weekly collection health audits

    ❌ NOT CORRELATING ACROSS MongoDB + Kafka + Redshift
       → Monitoring each in isolation = blind spots
       → EXAMPLE: MongoDB ops/sec drops → looks like MongoDB problem
         → Actually: Kafka consumer lag spiked → back-pressure
         → Actually: Redshift COPY command failed → Kafka backed up
       → ROOT CAUSE was in Redshift, SYMPTOM appeared in MongoDB
       → Unified monitoring (Part 5 above) prevents this

MongoDB Monitoring Internals

WHY MONGODB MONITORING WORKS THE WAY IT DOES:

1. serverStatus() — HOW IT COLLECTS METRICS
   ┌─────────────────────────────────────────────────────┐
   │  mongod process                                       │
   │  ├── In-memory counters (opcounters, connections)     │
   │  │   → Atomic increments on every operation           │
   │  │   → Zero overhead — counters always maintained     │
   │  │   → serverStatus() just READS these counters       │
   │  │                                                    │
   │  ├── WiredTiger statistics                            │
   │  │   → WT maintains its own internal counters         │
   │  │   → Cache bytes, evictions, checkpoints            │
   │  │   → serverStatus() calls WT's statistics API       │
   │  │                                                    │
   │  ├── Replication metrics                              │
   │  │   → Oplog position tracked per member              │
   │  │   → Heartbeats every 2 seconds between members     │
   │  │   → rs.status() reads heartbeat state              │
   │  │                                                    │
   │  └── Lock metrics                                     │
   │      → WiredTiger uses document-level locking          │
   │      → Intent locks tracked at collection/database     │
   │      → currentOp() reads lock manager state            │
   └─────────────────────────────────────────────────────┘

   COST OF serverStatus():
   → Acquires NO locks
   → Reads in-memory counters
   → Typical latency: 5-20ms
   → Safe to call every 10-30 seconds in production

2. DATABASE PROFILER — HOW IT CAPTURES SLOW QUERIES
   ┌─────────────────────────────────────────────────────┐
   │  When profiler is enabled (Level 1):                  │
   │                                                       │
   │  EVERY operation goes through this path:              │
   │  ├── Operation starts → timer begins                  │
   │  ├── Operation completes → timer stops                │
   │  ├── IF elapsed > slowms threshold:                   │
   │  │   ├── Capture operation details:                   │
   │  │   │   command, namespace, planSummary,              │
   │  │   │   keysExamined, docsExamined, millis            │
   │  │   ├── Write to system.profile collection           │
   │  │   │   (capped collection — auto-deletes oldest)    │
   │  │   └── This write IS an additional I/O operation    │
   │  │                                                    │
   │  └── IF elapsed <= slowms: nothing happens (no cost)  │
   │                                                       │
   │  OVERHEAD:                                            │
   │  ├── Level 1: ~2-5% (only slow ops trigger writes)   │
   │  ├── Level 2: ~15-30% (EVERY op triggers a write)    │
   │  └── Level 0: 0% (profiler disabled)                  │
   │                                                       │
   │  system.profile collection:                           │
   │  ├── Default size: 1MB (capped)                       │
   │  ├── Holds ~1000-5000 profiled operations             │
   │  ├── Oldest entries auto-deleted when full             │
   │  └── Can resize: db.setProfilingLevel(0);             │
   │      db.system.profile.drop();                        │
   │      db.createCollection("system.profile",            │
   │        {capped:true, size:10485760}) // 10MB          │
   └─────────────────────────────────────────────────────┘

3. ATLAS MONITORING — HOW IT WORKS WITHOUT PROFILER OVERHEAD
   ┌─────────────────────────────────────────────────────┐
   │  Atlas uses a DIFFERENT approach than profiler:       │
   │                                                       │
   │  ├── Atlas Agent runs alongside mongod                │
   │  │   → Calls serverStatus() every 10 seconds          │
   │  │   → Calls replSetGetStatus() periodically          │

   │  │   → Samples slow query log (not full profiler)     │
   │  │   → Analyzes query shapes (parameterized patterns) │
   │  │   → Runs HYPOTHETICAL explain() on suggested       │
   │  │     indexes to estimate improvement                │
   │  │   → NO performance impact on production workload   │
   │  │                                                    │
   │  ├── Real-Time Performance Panel:                     │
   │  │   → Samples currentOp() every 1 second             │
   │  │   → Groups by operation type and namespace          │
   │  │   → Shows live throughput without profiler overhead │
   │  │                                                    │
   │  └── Data sent to Atlas control plane:                │
   │      → Metrics stored for 8 hours (free) to 2 years   │
   │      → Queried via Atlas UI or Admin API              │
   │      → Alert evaluation happens in control plane      │
   │      → NO storage consumed on your cluster             │

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE IN MONGODB MONITORING:

PITFALL 1: "I'll just check the dashboard once a day"
    → Redshift habit: queries take minutes, check STL tables occasionally
    → MongoDB reality: operations take milliseconds, problems escalate in seconds
    → A slow query in Redshift wastes minutes
    → A slow query in MongoDB at 15,000 ops/sec blocks THOUSANDS of operations
    FIX: Automated alerting with <5 minute detection time

PITFALL 2: "CPU and disk are fine, so the database is fine"
    → Redshift habit: CPU + disk + WLM = sufficient monitoring
    → MongoDB reality: WiredTiger cache can be thrashing at 50% CPU
    → Cache eviction rate is INVISIBLE in CPU metrics
    → Replication lag is INVISIBLE in disk metrics
    FIX: Monitor MongoDB-specific metrics (cache, tickets, oplog, targeting)

PITFALL 3: "COLLSCAN is like a Redshift sequential scan — acceptable"
    → Redshift: sequential scan on sorted data = efficient (zone maps skip)
    → MongoDB: COLLSCAN = read EVERY document = ALWAYS bad
    → There are NO zone maps in MongoDB — COLLSCAN reads everything
    → Even on 100M documents: COLLSCAN examines all 100M
    FIX: ZERO tolerance for COLLSCAN on collections > 10,000 documents

PITFALL 4: "I don't need to monitor replication — HA is automatic"
    → Redshift: Multi-AZ is invisible, AWS handles everything
    → MongoDB: replica set health is YOUR responsibility
    → Elections, lag, oplog window — all need active monitoring
    → A secondary 30 seconds behind = potential data loss on failover
    FIX: Alert on replication lag > 5 seconds, oplog window < 24 hours

PITFALL 5: "Monitoring the database is enough"
    → You have a PIPELINE: MongoDB → Kafka → Redshift
    → Database metrics can look perfect while pipeline is broken
    → Kafka consumer lag growing = Redshift data going stale
    → Nobody notices until analyst says "yesterday's data is missing"
    FIX: Monitor DATA FRESHNESS end-to-end, not just component health

PITFALL 6: "I'll tune thresholds later"
    → Deploying alerts with default thresholds = alert fatigue within a week
    → Baseline your workload for 2 weeks FIRST
    → Then set thresholds at 1.5x baseline (WARN) and 2x baseline (CRITICAL)
    FIX: Mandatory 2-week baseline period before production alerting

KEY SCALING DIFFERENCE:
    MongoDB: monitoring complexity grows with SHARD COUNT
             → each shard = independent replica set to monitor
             → N shards × 3 members = 3N nodes to track

              → more analysts = more queries = more STL data
              → STL retention shrinks with more activity
              → Need external storage for long-term query history
