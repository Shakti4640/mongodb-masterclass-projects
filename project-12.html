Problem: Revenue Per Product Category Per Month From Order Data

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    Your e-commerce analytics team needs these reports:
    
    Report 1: "Total revenue and order count per product category"
        → Classic GROUP BY with SUM and COUNT
    
    Report 2: "Monthly revenue trend for the last 12 months"
        → GROUP BY with DATE_TRUNC and SUM
    
    Report 3: "Top 5 customers by total spending"
        → GROUP BY customer, SUM, ORDER BY DESC, LIMIT 5
    
    Report 4: "Average order value per status, only statuses with > 100 orders"
        → GROUP BY with AVG and HAVING clause
    
    Report 5: "Revenue breakdown: category → month → with running totals"
        → Multi-level GROUP BY with computed fields
    
    In Redshift: you write SQL with SELECT...GROUP BY...HAVING...ORDER BY
    In MongoDB: you build a PIPELINE — a sequence of stages that data flows through
    
    THE PIPELINE IS THE SINGLE MOST IMPORTANT MONGODB CONCEPT AFTER DOCUMENTS
    It replaces: SELECT, WHERE, GROUP BY, HAVING, ORDER BY, LIMIT, JOINs,
                 subqueries, CTEs, and window functions — ALL in one framework

The SQL You Already Know

-- ═══════════════════════════════════════════════════════
-- REPORT 1: Revenue and order count per category
-- ═══════════════════════════════════════════════════════
SELECT
    category,
    COUNT(*) AS order_count,
    SUM(total_amount) AS total_revenue,
    AVG(total_amount) AS avg_order_value,
    MIN(total_amount) AS min_order,
    MAX(total_amount) AS max_order
FROM store.orders
GROUP BY category
ORDER BY total_revenue DESC;

-- ═══════════════════════════════════════════════════════
-- REPORT 2: Monthly revenue trend
-- ═══════════════════════════════════════════════════════
SELECT
    DATE_TRUNC('month', order_date) AS month,
    COUNT(*) AS order_count,
    SUM(total_amount) AS revenue
FROM store.orders
WHERE order_date >= DATEADD(month, -12, GETDATE())
GROUP BY DATE_TRUNC('month', order_date)
ORDER BY month;

-- ═══════════════════════════════════════════════════════
-- REPORT 3: Top 5 customers by spending
-- ═══════════════════════════════════════════════════════
SELECT
    customer_id,
    COUNT(*) AS order_count,
    SUM(total_amount) AS total_spent,
    AVG(total_amount) AS avg_order
FROM store.orders
GROUP BY customer_id
ORDER BY total_spent DESC
LIMIT 5;

-- ═══════════════════════════════════════════════════════
-- REPORT 4: Average order value per status (HAVING)
-- ═══════════════════════════════════════════════════════
SELECT
    status,
    COUNT(*) AS order_count,
    AVG(total_amount) AS avg_value
FROM store.orders
GROUP BY status
HAVING COUNT(*) > 100
ORDER BY avg_value DESC;

-- ═══════════════════════════════════════════════════════
-- REPORT 5: Category + Month breakdown
-- ═══════════════════════════════════════════════════════
SELECT
    category,
    DATE_TRUNC('month', order_date) AS month,
    COUNT(*) AS order_count,
    SUM(total_amount) AS revenue,
    ROUND(SUM(total_amount) / COUNT(*), 2) AS avg_order_value
FROM store.orders
GROUP BY category, DATE_TRUNC('month', order_date)
ORDER BY category, month;

How SQL Executes (What You Already Know)

SQL EXECUTION ORDER (not the writing order):

    1. FROM      → identify the table(s)
    2. WHERE     → filter rows before grouping
    3. GROUP BY  → group remaining rows
    4. HAVING    → filter groups after aggregation
    5. SELECT    → compute output columns
    6. ORDER BY  → sort results
    7. LIMIT     → take top N

    You WRITE:  SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT
    It EXECUTES: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

    This is a DECLARATIVE model: you say WHAT you want, the engine decides HOW

🟢 MONGODB MINDSET — The Aggregation Pipeline

The Fundamental Concept: Pipeline = UNIX Pipes for Data

MONGODB AGGREGATION PIPELINE:

    Instead of one SQL statement with clauses:
        SELECT ... FROM ... WHERE ... GROUP BY ... HAVING ... ORDER BY ...
    
    You build a SEQUENCE OF STAGES that data flows through:
        Collection → $match → $group → $sort → $project → $limit → Results
    
    ANALOGY:
        SQL = one complex sentence with many clauses
        Pipeline = a series of simple commands connected by pipes
    
    UNIX PIPE ANALOGY (you probably know this):
        cat file.txt | grep "error" | sort | uniq -c | sort -rn | head -10
        
        MongoDB equivalent:
        db.logs.aggregate([
            { $match: { level: "error" } },          // grep "error"
            { $group: { _id: "$source", count: {$sum:1} } },  // uniq -c
            { $sort: { count: -1 } },                // sort -rn
            { $limit: 10 }                           // head -10
        ])
    
    EACH STAGE:
        → Receives documents from the previous stage
        → Transforms them
        → Passes results to the next stage
        → Independent: each stage only sees what the previous stage output
    
    THIS IS A PROCEDURAL model: you say HOW to process, step by step
    SQL is DECLARATIVE: you say WHAT you want
    
    BOTH produce the same results
    Pipeline gives you MORE control over execution order
    SQL lets the optimizer choose execution order

🧩 HOW TO APPROACH IT IN MONGODB

MENTAL MODEL FOR PIPELINE CONSTRUCTION:

REDSHIFT SQL THINKING:
    "I need category revenue"
    → SELECT category, SUM(amount) FROM orders GROUP BY category ORDER BY 2 DESC
    → One statement, engine figures out execution

MONGODB PIPELINE THINKING:
    "I need category revenue"
    → Step 1: Start with all documents in orders collection
    → Step 2: Group by category, accumulate sum of amount
    → Step 3: Sort by total descending
    → Build each step as a stage object
    → Chain them in an array

THE TRANSLATION RECIPE:
    SQL:    SELECT columns FROM table WHERE filter GROUP BY keys HAVING condition ORDER BY col LIMIT n
    
    Pipeline:
    [
        { $match:   { filter } },           ← WHERE (ALWAYS put first for index use!)
        { $group:   { _id: keys, accumulators } },  ← GROUP BY + SELECT aggregates
        { $match:   { condition } },         ← HAVING (second $match = post-group filter)
        { $sort:    { col: -1 } },           ← ORDER BY
        { $limit:   n },                     ← LIMIT
        { $project: { reshape output } }     ← SELECT (final column selection)
    ]

CRITICAL RULE:
    $match FIRST → uses indexes (like WHERE before GROUP BY)
    $match LATER → filters in memory (like HAVING after GROUP BY)
    ALWAYS filter as early as possible in the pipeline

db.orders.aggregate([
    // STAGE 1: $group = GROUP BY category
    {
        $group: {
            _id: "$category",                    // GROUP BY category
                                                  // $ prefix = field reference
            order_count: { $sum: 1 },             // COUNT(*)
            total_revenue: { $sum: "$total_amount" }, // SUM(total_amount)
            avg_order_value: { $avg: "$total_amount" }, // AVG(total_amount)
            min_order: { $min: "$total_amount" },  // MIN(total_amount)
            max_order: { $max: "$total_amount" }   // MAX(total_amount)
        }
    },
    // STAGE 2: $sort = ORDER BY total_revenue DESC
    {
        $sort: { total_revenue: -1 }
    },
    // STAGE 3: $project = reshape output (optional but clean)
    {
        $project: {
            _id: 0,                              // hide the _id field
            category: "$_id",                    // rename _id → category
            order_count: 1,                      // include as-is
            total_revenue: { $round: ["$total_revenue", 2] },

            _id: {

            order_count: { $sum: 1 },
            revenue: { $sum: "$total_amount" }

    {
        $sort: { "_id.year": 1, "_id.month": 1 }
    },
    // STAGE 4: $project = clean output
    {
        $project: {
            _id: 0,
            month: {
                $dateToString: {                         // FORMAT_DATE equivalent
                    format: "%Y-%m",
                    date: {
                        $dateFromParts: {
                            year: "$_id.year",
                            month: "$_id.month",
                            day: 1
                        }
                    }
                }

Pipeline Performance: Index Usage in Aggregation

// ═══════════════════════════════════════════════════════
// CRITICAL: How indexes interact with aggregation
// ═══════════════════════════════════════════════════════

// RULE: $match as FIRST stage → can use indexes
//       $match later → processes in memory (no index)
//       $sort early → can use indexes if no prior stage modifies docs

// ─── EXAMPLE: $match FIRST (uses index) ──────────────
print("=== $match FIRST: uses index ===")
let e1 = db.orders.explain("executionStats").aggregate([
    { $match: { customer_id: "C-00100" } },         // FIRST → uses idx_cust_date
    { $group: { _id: "$status", count: { $sum: 1 } } },
    { $sort: { count: -1 } }
])
// Check if index was used
let stage1 = e1.stages[0].$cursor.executionStats
print(`Keys examined: ${stage1.totalKeysExamined}`)
print(`Docs examined: ${stage1.totalDocsExamined}`)
// Should show IXSCAN — only examines C-00100's documents

// ─── EXAMPLE: $match NOT FIRST (no index) ────────────
print("\n=== $group FIRST, $match SECOND: no index ===")
let e2 = db.orders.explain("executionStats").aggregate([
    { $group: { _id: "$customer_id", total: { $sum: "$total_amount" } } },
    { $match: { total: { $gt: 1000 } } },           // AFTER $group → no index
    { $sort: { total: -1 } }
])
// Full collection scan — $group must read all documents

// ─── BEST PRACTICE: Filter early, aggregate later ────
// GOOD:
// [
//     { $match: { filter } },       ← filter FIRST (index!)
//     { $group: { aggregate } },     ← group filtered data
//     { $match: { having } },        ← filter groups
//     { $sort: { order } },
//     { $limit: n }
// ]
//
// BAD:
// [
//     { $group: { aggregate } },     ← groups ALL data (no filter first)
//     { $match: { filter } },        ← too late — already scanned everything
// ]

// ─── PIPELINE OPTIMIZATION ───────────────────────────
// MongoDB automatically optimizes some pipeline patterns:
//
// 1. $match + $match → merged into single $match
//    [{ $match: { a: 1 } }, { $match: { b: 2 } }]
//    → [{ $match: { a: 1, b: 2 } }]
//
// 2. $sort + $limit → merged (TopN sort, doesn't sort everything)
//    [{ $sort: { a: -1 } }, { $limit: 10 }]
//    → Only finds top 10, doesn't fully sort
//
// 3. $match moves before $project/$addFields when possible
//    [{ $project: { a: 1 } }, { $match: { a: 5 } }]
//    → [{ $match: { a: 5 } }, { $project: { a: 1 } }]
//    → $match moved earlier → can use index
//
// Redshift optimizer does similar rewriting automatically
// MongoDB does SOME rewriting but you have more control

print("\n=== Pipeline Index Usage Analysis ===")

# With $match first (uses index)
explain_indexed = db.orders.aggregate([

// MEMORY LIMITS — 100MB per stage
// ═══════════════════════════════════════════════════════

// Each pipeline stage has a 100MB memory limit by default
// If exceeded: ERROR (unless allowDiskUse is enabled)

// WHEN YOU HIT THE LIMIT:
// → $group on high-cardinality field (50K unique customers)
// → $sort on large unsorted result set
// → $bucket on fine-grained ranges

// SOLUTION: allowDiskUse

    {
        $group: {
            _id: "$customer_id",                // 50K unique customers

⚡ WHAT COULD BE DONE (Pipeline Optimizations)

1. $match FIRST — ALWAYS (Project 12 core lesson)
   → Reduces documents entering pipeline
   → Uses indexes (Projects 9-11)
   → Dramatic performance improvement

2. $project EARLY — Remove unneeded fields
   → Less data flowing through subsequent stages
   → Less memory used per stage
   → db.orders.aggregate([
         { $match: { customer_id: "C-00100" } },
         { $project: { total_amount: 1, category: 1, order_date: 1 } },  ← early
         { $group: { ... } }
     ])

3. $sort + $limit TOGETHER — TopN optimization
   → MongoDB auto-optimizes: doesn't fully sort, just finds top N
   → [{ $sort: { total: -1 } }, { $limit: 10 }]
   → Internally: heap of size 10, scan once → O(n) not O(n log n)

4. $out / $merge — Materialized views
   → Save aggregation results to a new collection
   → Re-run periodically (like Redshift MATERIALIZED VIEW REFRESH)
   → Queries read pre-computed results → instant
   → db.orders.aggregate([ ...pipeline..., { $out: "monthly_revenue" } ])
   → We'll cover deeply in Project 20 (Computed Pattern)

5. USE $facet FOR MULTIPLE AGGREGATIONS IN ONE PASS
   → Instead of running 3 separate aggregate() calls:
   → Run one aggregate() with $facet stage
   → $facet runs parallel sub-pipelines on same data
   → Like running multiple GROUP BYs in one query

✅ WHAT SHOULD DO

SHOULD DO:
    ✅ Put $match as FIRST stage for index utilization
       → This is the #1 performance rule for aggregation
       → Without $match first: full collection scan every time
    
    ✅ Think in STAGES, not SQL clauses
       → Each stage is a discrete transformation
       → Debug by running pipeline with one stage at a time
       → Add stages incrementally until you get the result
    
    ✅ Use $project/$addFields to reduce data between stages
       → Less data per document = more documents in 100MB memory limit
       → Only carry fields you need to the next stage
    
    ✅ Use $round for financial results
       → $sum of doubles can produce imprecise results (Project 8)
       → $round: ["$field", 2] cleans up display
       → Better: use Decimal128 from the start
    
    ✅ Use $addToSet for DISTINCT aggregation
       → Redshift: COUNT(DISTINCT category) or LISTAGG(DISTINCT ...)
       → MongoDB: $addToSet collects unique values, $size counts them
    
    ✅ Use explain() on aggregation pipelines
       → db.collection.explain("executionStats").aggregate([...])
       → Verify index usage in first stage
       → Check for COLLSCAN → add $match or index
    
    ✅ Use allowDiskUse for analytical queries
       → If $group or $sort exceeds 100MB → enable disk
       → But consider: should this query be in Redshift instead?

❌ WHAT SHOULD NOT DO

DO NOT:
    ❌ Put $group before $match
       → $group without prior $match = scan entire collection
       → ALWAYS filter first, then aggregate
       → Exception: when you truly need to aggregate ALL data
    
    ❌ Use $push to collect ALL documents into one array
       → $push: "$$ROOT" in $group → creates one document with ALL docs as array
       → 16MB document limit → crashes on large groups
       → Use $out to write results, not $push to collect them
    
    ❌ Think "aggregation = slow, find = fast"
       → Aggregation WITH $match first and indexes = fast
       → Aggregation WITHOUT $match = slow (full scan)
       → It's not the pipeline that's slow — it's missing the $match
    
    ❌ Write one giant pipeline stage when multiple simple stages work
       → Hard to debug, hard to maintain
       → Break complex transformations into multiple $project/$addFields stages
       → Each stage should do ONE clear thing
    
    ❌ Forget that $group _id: null aggregates the ENTIRE collection
       → _id: null = no GROUP BY = one output document
       → _id: "$field" = GROUP BY field = one output per unique value
       → _id: { a: "$x", b: "$y" } = GROUP BY x, y = one per combo
       → Getting _id wrong = wrong results (not an error, just wrong data)
    
    ❌ Use JavaScript ($accumulator, $function) when pipeline operators exist
       → JavaScript in pipeline = 10-100x slower than native operators
       → Almost everything can be done with native pipeline stages
       → Only use JS as absolute last resort
    
    ❌ Assume pipeline execution order is optimized like SQL
       → SQL: optimizer reorders clauses for best performance
       → Pipeline: stages execute in YOUR specified order
       → MongoDB does SOME optimization (merging $match stages)
       → But mostly: YOUR order = execution order
       → This is POWER (control) and RESPONSIBILITY (must optimize yourself)

             ▼
    ┌─────────────────┐
    │  $sort            │  In-memory sort (3 docs = trivial)
    │  {count: -1}      │  Output: 3 documents (sorted)
    └────────┬─────────┘
             │ 3 documents
             ▼
    ┌─────────────────┐
    │  $limit           │  Take first 2
    │  2                │  Output: 2 documents
    └────────┬─────────┘
             │ 2 documents
             ▼
    ┌─────────────────┐
    │  CLIENT           │  Results returned to application
    └─────────────────┘

KEY BEHAVIORS:
    1. STREAMING: Most stages process documents one at a time
       → $match: check each doc, pass if matches, skip if not
       → $project: transform each doc, pass to next stage
       → These don't need to hold all docs in memory

    2. BLOCKING: Some stages must see ALL input before producing output
       → $group: must see all docs to compute aggregates
       → $sort: must see all docs to determine order
       → $bucket: must see all docs to assign buckets
       → These stages buffer in memory (100MB limit)

    3. PIPELINE COALESCENCE: MongoDB merges adjacent stages when safe
       → $match + $match → merged
       → $sort + $limit → TopN optimization
       → $project + $project → merged
       → $match moves before $project when possible

REDSHIFT COMPARISON:
    Redshift MPP execution:
    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ Slice 1   │  │ Slice 2   │  │ Slice 3   │  Parallel execution
    │ scan→agg  │  │ scan→agg  │  │ scan→agg  │  across compute nodes
    └─────┬────┘  └─────┬────┘  └─────┬────┘
          │             │             │
          └─────────────┼─────────────┘
                        ▼
               ┌──────────────┐
               │ Leader Node   │  Merges partial aggregates
               │ merge→sort    │  Final sort + limit
               │ →return       │  Return to client
               └──────────────┘

    Redshift PARALLELIZES across slices:
        → Each slice scans its portion of data (columnar, compressed)
        → Each slice computes PARTIAL aggregates locally
        → Leader node MERGES partial aggregates
        → Final sort/limit on leader node
        → THIS IS WHY Redshift is faster for full-table aggregation

    MongoDB SINGLE-THREADED pipeline (within one query):
        → One thread processes documents sequentially
        → No automatic parallelism within a single aggregate()
        → $match uses index → fast filter
        → $group uses hash table → sequential accumulation
        → THIS IS WHY MongoDB is slower for full-collection analytics
        → But FASTER when $match narrows to small subset first

    PARALLELISM AT SCALE:
        → Redshift: automatic MPP across nodes (always parallel)
        → MongoDB unsharded: single-threaded pipeline
        → MongoDB SHARDED: pipeline runs on EACH shard in parallel
          → mongos merges partial results (like Redshift leader node)
          → Sharding gives MongoDB Redshift-like parallelism
          → But still document-oriented, not columnar
          → We'll cover this in Project 21 (Sharding)

⚠️ COMMON PITFALLS AND FAILURE MODES

MISTAKES REDSHIFT DEVELOPERS MAKE — PROJECT 12 SPECIFIC:

PITFALL 1: "$match AFTER $group when it should be BEFORE"
    ❌ aggregate([
           { $group: { _id: "$customer_id", total: { $sum: "$total_amount" } } },
           { $match: { "_id": "C-00100" } }     ← filters AFTER grouping all 500K
       ])
    ✅ aggregate([
           { $match: { customer_id: "C-00100" } },  ← filters FIRST (index!)
           { $group: { _id: "$status", total: { $sum: "$total_amount" } } }
       ])
    WHY: SQL optimizer reorders WHERE before GROUP BY automatically
         MongoDB executes stages in YOUR order — no automatic reordering
         You MUST put $match first yourself
    IMPACT: 500K docs scanned vs 10 docs scanned — 50,000x difference

PITFALL 2: "Using $count instead of $sum:1 in $group"
    ❌ $group: { _id: "$category", count: { $count: {} } }
       → $count is NOT an accumulator inside $group
    ✅ $group: { _id: "$category", count: { $sum: 1 } }
       → $sum: 1 = add 1 per document = COUNT(*)
    WHY: $count is a STAGE (standalone), not an accumulator
         db.orders.aggregate([{ $count: "total" }]) → { total: 500000 }
         Inside $group, use $sum: 1 for counting
    NOTE: MongoDB 5.0+ added $count as accumulator in $group
          But $sum: 1 works in all versions — safer to use

PITFALL 3: "Forgetting $ prefix on field references"
    ❌ $group: { _id: "category" }        → groups ALL docs under literal string "category"
    ✅ $group: { _id: "$category" }       → groups by the VALUE of category field
    WHY: Without $, MongoDB treats it as a literal string
         "category" = the string "category" (same for all docs → one group)
         "$category" = the field reference (different per doc → many groups)
    IMPACT: Returns 1 group instead of 10 — results completely wrong
    THIS IS THE #1 AGGREGATION MISTAKE — always check for $ prefix

PITFALL 4: "Expecting SQL-like NULL handling in $group"
    ❌ Assuming NULL values are excluded from $avg like SQL
    ✅ Understanding MongoDB $avg SKIPS null/missing but $sum treats as 0
    WHY: Redshift AVG() excludes NULLs from count and sum
         MongoDB $avg also excludes null/missing from count and sum ← same!
         BUT: $sum: "$field" where field is missing → adds 0 (not NULL)
         Subtlety: a document with no total_amount field adds 0 to $sum
                   but doesn't affect $avg denominator

PITFALL 5: "Expecting pipeline results to be automatically sorted"
    ❌ Running $group and assuming results come in alphabetical order
    ✅ ALWAYS add explicit $sort stage if order matters
    WHY: SQL GROUP BY doesn't guarantee order either (need ORDER BY)
         But Redshift often returns in sort key order (implicit)
         MongoDB $group returns in HASH TABLE order (unpredictable)
         Never rely on implicit ordering — always $sort explicitly

PITFALL 6: "Using aggregate() when find() is sufficient"
    ❌ aggregate([ { $match: { email: "user@test.com" } } ])
       → overhead of pipeline framework for simple lookup
    ✅ find({ email: "user@test.com" })
       → direct index lookup, faster
    WHY: Aggregation has pipeline setup overhead
         For simple queries without GROUP BY, use find()
         Use aggregate() when you need: grouping, transformations,
         $lookup, $unwind, computed fields, $facet

PITFALL 7: "Building the entire pipeline at once and debugging"
    ❌ Writing 8-stage pipeline, running, getting wrong results, staring at it
    ✅ Build incrementally:
       Stage 1: run, check output
       Stage 1+2: run, check output
       Stage 1+2+3: run, check output
       Add one stage at a time
    WHY: SQL is one statement — you debug the whole thing
         Pipeline is sequential — debug stage by stage
         Each stage's output is the next stage's input
         Print intermediate results to verify each transformation

PITFALL 8: "Forgetting allowDiskUse for large aggregations"
    ❌ $group on 50K unique customer_ids with $push → exceeds 100MB → ERROR
    ✅ aggregate([...], { allowDiskUse: true })
    WHY: Redshift auto-spills to disk (no option needed)
         MongoDB requires explicit opt-in
         Error message: "Exceeded memory limit for $group"
         Fix: add allowDiskUse: true
         Better fix: filter earlier with $match to reduce data volume

📐 DATA MODELING COMPARISON — Aggregation Impact

HOW AGGREGATION NEEDS AFFECT DOCUMENT DESIGN:

REDSHIFT:
    → Normalized tables optimized for columnar scan
    → GROUP BY on any column works efficiently (columnar reads one column)
    → JOINs between tables are expected in aggregation queries
    → Star schema: fact table + dimension tables → JOIN on aggregate

MONGODB:
    → Denormalized documents optimized for filtered aggregation
    → $group works best when all needed fields are IN the document
    → $lookup (JOIN) in aggregation is expensive — avoid when possible
    → Embed frequently-aggregated fields directly in the document

EXAMPLE: Revenue by category with category name

REDSHIFT (normalized):
    orders: order_id, category_id, amount
    categories: category_id, category_name
    → SELECT c.category_name, SUM(o.amount)
      FROM orders o JOIN categories c ON o.category_id = c.category_id
      GROUP BY c.category_name
    → JOIN is natural and efficient in columnar MPP

MONGODB (denormalized):
    orders: { category: "Electronics", amount: 129.99 }
    → No JOIN needed — category name embedded
    → aggregate([ { $group: { _id: "$category", total: {$sum:"$amount"} } } ])
    → Faster: no $lookup, single collection scan

MONGODB (normalized — anti-pattern for aggregation):
    orders: { category_id: ObjectId("..."), amount: 129.99 }
    categories: { _id: ObjectId("..."), name: "Electronics" }
    → Must $lookup to get category name → expensive
    → Requires reading from TWO collections → slow
    → This is the Redshift-style approach in MongoDB → DON'T

KEY INSIGHT:
    In Redshift: normalize freely, JOINs are efficient
    In MongoDB: denormalize for aggregation, avoid $lookup in hot paths
    Your document design determines your aggregation performance
    This is why Project 14 (Embedding vs Referencing) is critical

def run_pipeline(collection, pipeline, label, show_results=True,
                 max_results=10, allow_disk=False):
    """
    Run aggregation pipeline with timing, count, and optional explain.
    """
    print(f"\n  {'='*60}")
    print(f"  {label}")
    print(f"  {'='*60}")

    # Show pipeline stages
    print(f"  Pipeline ({len(pipeline)} stages):")
    for i, stage in enumerate(pipeline):
        stage_name = list(stage.keys())[0]
        print(f"    Stage {i+1}: {stage_name}")

    # Run with timing
    start = time.time()
    results = list(collection.aggregate(pipeline, allowDiskUse=allow_disk))
    elapsed = (time.time() - start) * 1000

    print(f"  {'─'*60}")
    print(f"  Results: {len(results)} documents")
    print(f"  Time: {elapsed:.1f}ms")

    if show_results:
        for doc in results[:max_results]:
            # Clean up for display
            display = {}
            for k, v in doc.items():
                if isinstance(v, float):
                    display[k] = round(v, 2)
                elif isinstance(v, list) and len(v) > 5:
                    display[k] = f"[{len(v)} items]"
                else:
                    display[k] = v
            pprint.pprint(display)

        if len(results) > max_results:
            print(f"  ... and {len(results) - max_results} more")

    return results
