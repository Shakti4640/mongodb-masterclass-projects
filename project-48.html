💰 SECTION 1: MONGODB ATLAS PRICING — Complete Breakdown

Atlas Pricing Model (2024)

MONGODB ATLAS PRICING STRUCTURE:
    You pay for: Instance + Storage + Backup + Data Transfer + Optional Features

    → No separate compute vs storage pricing (until serverless)
    → Instance tier determines BOTH compute AND included storage
    → Backup is included (continuous) but extra storage costs
    → Data transfer costs are SIGNIFICANT (often overlooked)

💰 SECTION 2: AMAZON REDSHIFT PRICING — Complete Breakdown

Redshift Pricing Model (2024)

REDSHIFT PRICING STRUCTURE:
    You pay for: Compute Nodes + Managed Storage (RA3) + Spectrum + Extras
    
    UNLIKE MONGODB:
    → Compute and storage are SEPARATE (RA3 nodes)
    → You pay for compute even when idle (unless paused/serverless)
    → Spectrum charges per TB scanned (not per hour)
    → Concurrency scaling charges for burst capacity

💰 SECTION 6: COST OPTIMIZATION STRATEGIES

MONGODB ATLAS COST OPTIMIZATION:

1. RIGHT-SIZE INSTANCES
   → Start with M30, monitor, upgrade only when needed
   → Atlas Performance Advisor tells you when to upgrade
   → Over-provisioning is the #1 MongoDB cost waste
   → Check: db.serverStatus().wiredTiger.cache
     → If cache hit ratio > 95%: instance is right-sized
     → If < 90%: consider upgrading

2. USE AUTO-SCALING (Atlas feature)
   → Set min/max instance tiers
   → Atlas scales up during peak, down during off-peak
   → Can save 20-40% vs fixed provisioning
   → Example: M50 during day, M40 at night

3. OPTIMIZE INDEXES
   → Unused indexes waste RAM and write throughput
   → db.collection.aggregate([{$indexStats:{}}])
   → Drop indexes with 0 accesses in 30 days
   → Each dropped index saves ~10% write overhead

4. USE TTL INDEXES AGGRESSIVELY
   → Don't pay to store data you don't need
   → Session data: TTL 24 hours
   → Cart data: TTL 7 days
   → Logs: TTL 30 days
   → Each TB of TTL'd data saves $250/month in storage

5. COMPRESS DATA (application level)
   → MongoDB compresses with snappy/zstd by default
   → But large text fields can be pre-compressed
   → Store binary blobs in S3, reference in MongoDB

6. RESERVED INSTANCES (for stable workloads)
   → Atlas offers reserved capacity discounts
   → Contact MongoDB sales for enterprise pricing
   → Typical savings: 30-50% vs on-demand


REDSHIFT COST OPTIMIZATION:

1. USE RA3 INSTEAD OF DC2
   → RA3 separates compute from storage
   → Pay for storage you use, not storage you might use
   → Can resize compute without moving data

2. USE RESERVED INSTANCES
   → 1-year reserved: ~35-40% savings
   → 3-year reserved: ~55-65% savings
   → Example: RA3.xlplus on-demand $793/month → 1yr RI ~$515/month
   → ONLY commit to reserved for stable baseline

3. PAUSE WHEN NOT IN USE
   → Dev/test clusters: pause overnight and weekends
   → 730 hours/month → ~250 hours active = 66% savings
   → Production: cannot pause, but consider serverless for intermittent

4. USE REDSHIFT SERVERLESS FOR INTERMITTENT WORKLOADS
   → If analysts query <8 hours/day
   → Serverless auto-pauses after 5 minutes idle
   → Break-even: ~6 hours of active querying per day
   → Below 6 hours: serverless cheaper
   → Above 6 hours: provisioned cheaper

5. OPTIMIZE SORT KEYS AND DISTRIBUTION
   → Bad sort key = full table scans = more compute needed
   → Good sort key = block skipping = same query on smaller cluster
   → Review: SELECT * FROM svv_table_info ORDER BY unsorted DESC

6. MOVE COLD DATA TO SPECTRUM
   → Data older than 90 days → S3 Parquet → Spectrum
   → S3 storage: $0.023/GB vs Redshift: $0.024/GB (similar)
   → BUT: smaller Redshift cluster needed (less data to manage)
   → Spectrum: pay per query ($5/TB scanned)
   → If cold data queried rarely: massive savings

7. USE MATERIALIZED VIEWS
   → Pre-compute expensive aggregations
   → Dashboard queries hit MV (milliseconds) instead of base tables (seconds)
   → Smaller cluster can serve same workload
   → Refresh MVs during off-peak (scheduled)

8. COMPRESS DATA PROPERLY
   → Run ANALYZE COMPRESSION on tables
   → Choose optimal encoding per column
   → Can reduce storage 5-10x → smaller cluster needed

✅ WHAT SHOULD DO (Cost Management Best Practices)

SHOULD DO:

✅ START SMALL, MEASURE, THEN SCALE
    → Don't provision for peak on day 1
    → Atlas M10 → M30 → M50 as traffic grows
    → Redshift Serverless → 2-node → 4-node as queries justify
    → Each upgrade based on ACTUAL metrics, not projections

✅ USE RESERVED INSTANCES FOR STABLE BASELINE
    → Reserve 70% of steady-state capacity
    → Use on-demand/serverless for the 30% variable portion
    → Review reservations annually

✅ MONITOR COST PER TRANSACTION
    → Track: infrastructure cost / number of operations
    → E-commerce: cost per order ($0.0004 is healthy)
    → IoT: cost per sensor per month ($0.15 is healthy)
    → SaaS: cost per tenant per month (should be <20% of plan price)

✅ SET BILLING ALERTS
    → Atlas: budget alerts at 50%, 80%, 100% of expected
    → AWS: CloudWatch billing alerts
    → Kafka: monitor throughput → cost scales with throughput

✅ REVIEW ARCHITECTURE QUARTERLY
    → Are all MongoDB indexes being used? (drop unused → save RAM → smaller instance)
    → Is Redshift cluster right-sized? (check WLM queue depth)
    → Is Spectrum cheaper than local storage for cold data?
    → Can any workload move to serverless?

✅ ACCOUNT FOR ENGINEERING TIME IN TCO
    → A $5K/month infrastructure saving that requires
      1 FTE ($15K/month) to manage is a NET LOSS
    → Managed services (Atlas, MSK) cost more than self-managed
      but save 0.5-1 FTE → usually net positive

❌ WHAT SHOULD NOT DO (Cost Anti-Patterns)

DO NOT:

❌ CHOOSE ARCHITECTURE BASED ON COST ALONE
    → Cheapest option that fails performance SLAs = most expensive option
    → A $500/month savings that causes 100ms latency increase
      → At 1% conversion impact on $10M revenue = $100K/year lost
    → Always evaluate: cost + performance + reliability

❌ OVER-PROVISION "JUST IN CASE"
    → M80 cluster "because we might need it someday"
    → RA3.4xlarge × 8 "for future growth"
    → Paying for capacity you don't use = waste
    → Both Atlas and Redshift can scale up in hours when needed

❌ IGNORE DATA TRANSFER COSTS
    → Cross-region replication: $0.02/GB adds up fast
    → 1TB/month cross-region = $20/month (seems small)
    → 100TB/month at scale = $2,000/month (significant)
    → Design architecture to minimize cross-region traffic

❌ KEEP ALL DATA IN HOT STORAGE
    → MongoDB at $0.25/GB vs S3 at $0.023/GB vs Glacier at $0.004/GB
    → 10TB of cold data in MongoDB: $2,500/month
    → Same 10TB in Glacier: $40/month
    → TTL + S3 lifecycle = 60x cost reduction on cold data

❌ FORGET TO CLEAN UP DEV/TEST RESOURCES
    → That M40 dev cluster nobody uses: $2,277/month
    → That Redshift cluster from last quarter's POC: $1,586/month
    → Tag all resources with environment + owner + expiry date
    → Monthly cleanup review saves 10-20% of total bill

❌ COMPARE ONLY INSTANCE COSTS
    → MongoDB M50 ($4,380) vs Redshift ra3.xlplus×4 ($3,172)
    → "Redshift is cheaper!" — WRONG
    → Add: OpenSearch ($2,100) + ElastiCache ($1,200) + DMS ($730)
      to replace MongoDB features
    → True comparison: $4,380 vs $7,202
    → Always compare TOTAL system cost, not component cost
