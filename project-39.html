PROJECT 39: Zone Sharding — Data Locality & Compliance

WHERE Data Lives Is as Important as WHO Can Access It

🎯 THE HYPOTHETICAL PROBLEM

SCENARIO:
    ShopStream has gone global. The platform now serves customers in:
    → North America (US, Canada) — 60% of orders
    → European Union (Germany, France, Netherlands) — 25% of orders
    → Asia-Pacific (Japan, Australia, Singapore) — 15% of orders
    
    THREE COMPLIANCE MANDATES HIT SIMULTANEOUSLY:
    
    MANDATE 1 — GDPR (European Union):
        → EU customer personal data MUST be stored in EU data centers
        → "Personal data" = name, email, address, phone, order history
        → Data CANNOT leave EU boundaries (with limited exceptions)
        → Violation penalty: up to 4% of global annual revenue
        → ShopStream's EU revenue: $50M → potential fine: $2M
        → CURRENT STATE: all data in us-east-1 (Virginia) → VIOLATION
    
    MANDATE 2 — APPI (Japan) / Privacy Act (Australia):
        → Japanese customer data preferably stored in Japan/APAC region
        → Australian Privacy Act: data should remain in Australia where practical
        → Not as strict as GDPR but increasingly enforced
        → CURRENT STATE: APAC data in us-east-1 → non-compliant
    
    MANDATE 3 — PERFORMANCE (Latency Requirements):
        → EU customers experience 150ms latency (cross-Atlantic round-trip)
        → APAC customers experience 250ms latency (cross-Pacific)
        → Target: <50ms for all regions
        → Customers abandoning carts due to slow checkout
        → Revenue impact: estimated $3M/year in lost conversions
    
    ADDITIONAL BUSINESS REQUIREMENTS:
    
    REQUIREMENT 4 — TIERED STORAGE:
        → Hot data (last 30 days): fast NVMe SSDs
        → Warm data (30-180 days): standard SSDs
        → Cold data (>180 days): archive to S3 (Project 36)
        → Want to use zone sharding for storage tiers too
    
    REQUIREMENT 5 — DISASTER RECOVERY:
        → Each region must survive independent data center failure
        → EU data must have replicas WITHIN EU (not failover to US)
        → RPO (Recovery Point Objective): <1 second
        → RTO (Recovery Time Objective): <30 seconds
    
    THE QUESTION:
        How do you ensure data lives in the RIGHT geographic region
        while maintaining a SINGLE logical database?
    
    THE ANSWER:
        → MongoDB Zone Sharding: tag shards with regions,
          route data to correct shards based on shard key
        → ONE database, ONE connection string, THREE regions
        → Application doesn't know where data lives — routing is automatic
        → Compliance built into the database topology
    
    REDSHIFT COMPARISON:
        → Redshift has NO equivalent of zone sharding
        → Redshift approach: separate cluster per region
        → Separate clusters = separate databases = separate queries
        → No unified view without Redshift Data Sharing (read-only, limited)
        → MongoDB zone sharding is fundamentally more elegant

🔴 REDSHIFT MINDSET — "How You Handle Multi-Region in Redshift"

Redshift Multi-Region Architecture (What You Know)

REDSHIFT APPROACH TO DATA LOCALITY:

OPTION A: SEPARATE CLUSTERS PER REGION (most common)

    ┌────────────────────┐   ┌────────────────────┐   ┌────────────────────┐
    │  REDSHIFT CLUSTER   │   │  REDSHIFT CLUSTER   │   │  REDSHIFT CLUSTER   │
    │  us-east-1          │   │  eu-west-1          │   │  ap-northeast-1    │
    │                     │   │                     │   │                     │
    │  US customer data   │   │  EU customer data   │   │  APAC customer data│
    │  US orders          │   │  EU orders          │   │  APAC orders       │
    │  US analytics       │   │  EU analytics       │   │  APAC analytics    │
    └────────────────────┘   └────────────────────┘   └────────────────────┘
    
    PROBLEMS:
    → Three separate databases to manage
    → Three separate ETL pipelines
    → Three separate BI dashboard configurations
    → "Global revenue report" requires querying all three + merging
    → Schema changes must be applied to all three clusters
    → No single view of all customer data
    → 3x operational overhead

OPTION B: REDSHIFT DATA SHARING (cross-region, limited)

    ┌────────────────────┐
    │  PRODUCER CLUSTER   │ ───── Data Sharing ────► Consumer Cluster
    │  us-east-1          │                           eu-west-1 (read-only)
    │  (all data)         │
    └────────────────────┘
    
    PROBLEMS:
    → Data sharing is READ-ONLY (consumer can't write)
    → Data still LIVES in producer region (us-east-1)
    → GDPR violation: EU data stored in US, shared to EU
    → Cross-region latency for reads
    → Not a data locality solution

OPTION C: APPLICATION-LEVEL ROUTING

    ┌──────────────────────────────────────────────────────┐
    │  APPLICATION                                          │
    │                                                        │
    │  if customer.region == "EU":                           │
    │      connect_to("redshift-eu-west-1")                 │
    │  elif customer.region == "APAC":                       │
    │      connect_to("redshift-ap-northeast-1")            │
    │  else:                                                 │
    │      connect_to("redshift-us-east-1")                 │
    └──────────────────────────────────────────────────────┘
    
    PROBLEMS:
    → Application must know about all clusters
    → Routing logic in application code (brittle)
    → Cross-region queries impossible without application merging
    → Connection management complexity
    → Failover logic in application (not database)

NONE OF THESE ARE ELEGANT:
    → Redshift was designed for SINGLE-REGION analytical workloads
    → Multi-region is an AFTERTHOUGHT in Redshift
    → MongoDB zone sharding was DESIGNED for this from the ground up

🧩 HOW TO APPROACH IT IN MONGODB

ZONE SHARDING SETUP — STEP BY STEP THINKING:

STEP 1: CHOOSE THE SHARD KEY
    → Must include the field that determines which zone/region
    → Our case: customer_region ("US", "EU", "APAC")
    → Compound shard key: { customer_region: 1, _id: 1 }
    → customer_region: routes to correct zone
    → _id: ensures uniqueness and distribution within zone
    → (Project 22: shard key selection principles apply here)

STEP 2: DEPLOY SHARDS IN CORRECT REGIONS
    → Shard 1: replica set in us-east-1
    → Shard 2: replica set in eu-west-1
    → Shard 3: replica set in ap-northeast-1
    → Each shard = 3-node replica set (within its region)
    → Atlas: handles deployment automatically

STEP 3: TAG SHARDS WITH ZONE NAMES
    → Tag shard 1 as "US"
    → Tag shard 2 as "EU"
    → Tag shard 3 as "APAC"

STEP 4: DEFINE ZONE RANGES
    → customer_region: "US" → zone "US" (shard 1)
    → customer_region: "EU" → zone "EU" (shard 2)
    → customer_region: "APAC" → zone "APAC" (shard 3)

STEP 5: ENABLE SHARDING AND VERIFY
    → Shard the collection with chosen key
    → Verify zone assignments
    → Insert test documents and confirm routing
    → Monitor balancer to ensure correct placement

🔧 HOW CAN WE SOLVE IT IN MONGODB

Step 1: Shard Key Design for Zone Sharding

// ═══════════════════════════════════════════════════════
// STEP 1: SHARD KEY DESIGN
//
// The shard key MUST include the zone-determining field
// In our case: customer_region
//
// Shard key: { customer_region: 1, _id: 1 }
//
// WHY this compound key:
// → customer_region: determines WHICH zone (geographic routing)
// → _id: ensures high cardinality WITHIN each zone
//   (avoids jumbo chunks — Project 22)
// → Without _id: all "EU" documents could land in one chunk
//   (if customer_region alone has only 3 values = 3 chunks max)
//
// COMPARISON TO REDSHIFT:
// → Redshift DISTKEY: determines which slice gets the row
// → MongoDB shard key: determines which SHARD gets the document
// → Zone tag: determines which REGION the shard lives in
// → Redshift has no zone concept — distribution is within one cluster
// ═══════════════════════════════════════════════════════

// Document design with zone-routing field
// EVERY document must have customer_region at the top level

{
    _id: ObjectId(),
    order_number: "ORD-2024-EU-001",
    
    // THIS FIELD DETERMINES GEOGRAPHIC ROUTING
    customer_region: "EU",           // ← shard key prefix → routes to EU shard
    
    customer: {
        customer_id: ObjectId("..."),
        name: "Hans Mueller",
        email: "hans@example.de",
        country: "DE"                // Germany
    },
    items: [ /* ... */ ],
    payment: { total: 89.99 },
    shipping: {
        address: {
            street: "Hauptstraße 42",
            city: "Berlin",
            country: "DE"
        }
    },
    status: "confirmed",
    created_at: ISODate("2024-05-31T14:30:22Z")
}

// CRITICAL DESIGN DECISIONS:
//
// 1. customer_region is DENORMALIZED into every document
//    → Not derived from address.country at query time
//    → Must be set at insert time
//    → Cannot change after insert (shard key is immutable — Project 22)
//    → Application must determine region BEFORE inserting
//
// 2. Region mapping logic (in application):
//    DE, FR, NL, IT, ES, ... → "EU"
//    US, CA                  → "US"
//    JP, AU, SG, KR, ...    → "APAC"
//
// 3. What about customers who MOVE between regions?
//    → customer_region is set at signup time
//    → If customer moves EU → US: complex (must migrate document)
//    → Best practice: create new document, soft-delete old
//    → This is a LIMITATION of zone sharding (shard key immutability)

Step 2: Configure Zone Sharding (mongosh)

// STEP 2: CONFIGURE ZONE SHARDING
//
// Prerequisites:
// → Sharded cluster deployed with shards in 3 regions
// → Config servers running
// → mongos routers running
// → Connect to mongos (NOT directly to shard)
//
// On Atlas: this is done via Atlas UI (Global Clusters)
// On self-managed: manual configuration via mongosh
// ═══════════════════════════════════════════════════════

// Connect to mongos
// mongosh "mongodb://mongos-host:27017/admin"

// ─── STEP 2A: VERIFY SHARDS ARE AVAILABLE ───────────

sh.status()
// Output shows:
// shards:
//   { _id: "shard-us",   host: "shard-us/us-node1:27017,us-node2:27017,us-node3:27017" }
//   { _id: "shard-eu",   host: "shard-eu/eu-node1:27017,eu-node2:27017,eu-node3:27017" }
//   { _id: "shard-apac", host: "shard-apac/apac-node1:27017,apac-node2:27017,apac-node3:27017" }


// ─── STEP 2B: TAG SHARDS WITH ZONE NAMES ────────────
// This tells MongoDB: "shard-eu IS the EU zone"

sh.addShardTag("shard-us", "US")
sh.addShardTag("shard-eu", "EU")
sh.addShardTag("shard-apac", "APAC")

// Verify tags
db.adminCommand({ listShards: 1 })
// Each shard now has tags: ["US"], ["EU"], ["APAC"]

// REDSHIFT EQUIVALENT: None
// → Redshift clusters don't have "tags" that control data placement
// → You physically create separate clusters in separate regions
// → MongoDB does it within ONE logical cluster


// ─── STEP 2C: ENABLE SHARDING ON DATABASE ───────────

sh.enableSharding("shopstream")


// ─── STEP 2D: SHARD THE COLLECTION ──────────────────

sh.shardCollection(
    "shopstream.orders",
    { customer_region: 1, _id: 1 }    // compound shard key
)

// COMPARISON TO REDSHIFT:
// CREATE TABLE orders (...) DISTSTYLE KEY DISTKEY(customer_region);
// → Same concept: distribute data by a key field
// → BUT: Redshift distributes across SLICES within ONE cluster
// → MongoDB distributes across SHARDS potentially in DIFFERENT regions


// ─── STEP 2E: DEFINE ZONE RANGES ────────────────────
// This is THE critical step: map shard key values to zones
//
// "Any document where customer_region = 'EU' → EU zone"
// "Any document where customer_region = 'US' → US zone"
// "Any document where customer_region = 'APAC' → APAC zone"

// Zone range for US
sh.addTagRange(
    "shopstream.orders",
    { customer_region: "US",   _id: MinKey },    // range start
    { customer_region: "US",   _id: MaxKey },    // range end
    "US"                                          // zone tag
)
// All chunks where customer_region = "US" → US zone → shard-us

// Zone range for EU
sh.addTagRange(
    "shopstream.orders",
    { customer_region: "EU",   _id: MinKey },
    { customer_region: "EU",   _id: MaxKey },
    "EU"
)
// All chunks where customer_region = "EU" → EU zone → shard-eu

// Zone range for APAC
sh.addTagRange(
    "shopstream.orders",
    { customer_region: "APAC", _id: MinKey },
    { customer_region: "APAC", _id: MaxKey },
    "APAC"
)
// All chunks where customer_region = "APAC" → APAC zone → shard-apac


// ─── STEP 2F: VERIFY ZONE CONFIGURATION ─────────────

sh.status()
// Output now shows:
//
// databases:
//   shopstream:

//       shard key: { customer_region: 1, _id: 1 }

Step 3: Application Code — Transparent Routing

# ═══════════════════════════════════════════════════════
# STEP 3: APPLICATION CODE
#
# THE BEAUTIFUL PART: application code doesn't change!
# mongos handles ALL routing transparently
# Application connects to ONE endpoint
# ═══════════════════════════════════════════════════════

from pymongo import MongoClient
from datetime import datetime
import logging

logger = logging.getLogger("zone_sharding_demo")

# ─── SINGLE CONNECTION (application doesn't know about zones) ──

client = MongoClient(
    "mongodb+srv://app_service:pass@shopstream-global.mongodb.net/"
    # ONE connection string for the ENTIRE global cluster
    # mongos router handles geographic routing
)
db = client["shopstream"]
orders = db["orders"]

# ─── REGION DETERMINATION (application's responsibility) ──

COUNTRY_TO_REGION = {
    # EU countries
    "DE": "EU", "FR": "EU", "NL": "EU", "IT": "EU",
    "ES": "EU", "PT": "EU", "BE": "EU", "AT": "EU",
    "IE": "EU", "FI": "EU", "SE": "EU", "DK": "EU",
    "PL": "EU", "CZ": "EU", "RO": "EU", "GR": "EU",
    
    # North America
    "US": "US", "CA": "US",
    
    # APAC
    "JP": "APAC", "AU": "APAC", "SG": "APAC",
    "KR": "APAC", "NZ": "APAC", "IN": "APAC",
}

def get_region(country_code):
    """Determine region from country code"""
    return COUNTRY_TO_REGION.get(country_code, "US")  # default to US


# ─── INSERT: Application sets customer_region ─────────

def create_order(customer, items, shipping_country):
    """
    Create an order with correct region routing
    
    Application's ONLY responsibility:
    → Set customer_region field based on shipping country
    → mongos routes to correct shard automatically
    
    Application does NOT:
    → Choose which shard to write to
    → Manage connection to specific region
    → Know about zone configuration
    """
    
    region = get_region(shipping_country)
    
    order = {
        # SHARD KEY (determines geographic routing)
        "customer_region": region,    # ← THIS is all the app needs to set
        
        "order_number": f"ORD-2024-{region}-{datetime.utcnow().strftime('%H%M%S')}",
        "customer": customer,
        "items": items,
        "shipping": {
            "address": {
                "country": shipping_country
            }
        },
        "payment": {
            "total": sum(item["price"] * item["qty"] for item in items)
        },
        "status": "confirmed",
        "created_at": datetime.utcnow()
    }
    
    result = orders.insert_one(order)
    logger.info(f"Order {order['order_number']} → region: {region} → routed automatically")
    
    return result.inserted_id

# German customer → customer_region: "EU" → shard-eu (eu-west-1)
eu_order_id = create_order(

sh.addShardTag("shard-eu-warm", "EU-WARM")

// → customer_region: geographic routing

    → { customer_region: 1, _id: "hashed" }
    → Hashed _id ensures even distribution within each zone
    → Prevents hot spots within a region
    → Trade-off: hashed component can't do range queries on _id

OPTIMIZATION 3: SECONDARY ZONE TAGS FOR READ PREFERENCE
    → Tag shards with multiple tags: ["EU", "EU-ANALYTICS"]

OPTIMIZATION 4: BALANCER WINDOW SCHEDULING

    → Schedule balancer during off-peak hours
    → sh.setBalancerState(true) / sh.setBalancerState(false)
    → Or: set balancer window: 02:00 - 06:00 UTC
    → Prevents migration-related latency during peak hours

❌ WHAT SHOULD NOT DO (Anti-Patterns)

❌ DO NOT USE LOW-CARDINALITY SHARD KEY ALONE
    WRONG: shard key: { customer_region: 1 }
    WHY: only 3 possible values → only 3 chunks maximum
    → Cannot split chunks further → jumbo chunks → unbalanceable
    → (Project 22: cardinality is critical for shard key design)
    ✅ FIX: compound key { customer_region: 1, _id: 1 }
    → _id adds cardinality → chunks can split within each region

❌ DO NOT ALLOW DOCUMENTS WITHOUT customer_region
    WRONG: insert document without customer_region field
    → MongoDB routes to "default" zone or random shard
    → EU customer data could end up on US shard → GDPR violation
    ✅ FIX:
    → Schema validation (Project 7): require customer_region
    → Application validation: reject orders without region
    → Monitoring: alert on documents with null/missing region

❌ DO NOT DISABLE THE BALANCER PERMANENTLY
    WRONG: sh.setBalancerState(false) and forgetting
    WHY: new chunks won't migrate to correct zones
    → Zone violations accumulate silently
    → Data locality guarantees degrade over time
    ✅ FIX: use balancer windows (schedule, don't disable)

❌ DO NOT PUT mongos IN A SINGLE REGION
    WRONG: all mongos routers in us-east-1
    WHY: EU customers connect to US mongos → cross-Atlantic latency
    → Even though data is on EU shard, the ROUTER is in US
    ✅ FIX: deploy mongos in EACH region
    → EU users → EU mongos → EU shard (all within EU)
    → Atlas handles this automatically with Global Clusters

❌ DO NOT QUERY ACROSS REGIONS FOR LATENCY-SENSITIVE OPERATIONS
    WRONG: application page load requires scatter-gather across all zones
    WHY: scatter-gather = cross-region round-trips → 150-250ms added
    ✅ FIX:
    → Design queries to include customer_region (targeted)
    → Cache global aggregations (compute periodically, serve from cache)
    → Separate operational queries (targeted) from analytical (scatter-gather OK)

❌ DO NOT ASSUME INSTANT CHUNK MIGRATION
    WRONG: insert document → expect it on correct shard immediately
    REALITY: balancer moves chunks periodically (not instantly)
    → New document may briefly exist on "wrong" shard until balancer runs
    → For COMPLIANCE: this temporary state may be concerning
    ✅ FIX:
    → Pre-split chunks for each zone range before data arrives
    → This ensures new inserts go directly to correct shard
    → No balancer migration needed for correctly pre-split ranges

⚙️ HOW IT ONLY WORKS INTERNALLY

ZONE SHARDING INTERNALS — WHAT HAPPENS WHEN YOU INSERT:

APPLICATION:
    orders.insert_one({ customer_region: "EU", ... })
        │
        ▼
MONGOS ROUTER:
    1. Extracts shard key: { customer_region: "EU", _id: ObjectId("...") }
    2. Looks up chunk map in config server cache:
       → Chunk ranges → which shard owns this key range?
       → { customer_region: "EU", _id: MinKey } to { customer_region: "EU", _id: MaxKey }
       → This range is tagged "EU"
       → Zone "EU" is assigned to shard "shard-eu"
       → Therefore: this document → shard-eu
    3. Routes insert to shard-eu PRIMARY
        │
        ▼
SHARD-EU PRIMARY (in eu-west-1):
    1. Receives insert
    2. Writes to WiredTiger (encrypted at rest — Project 38)
    3. Replicates to EU secondaries (also in eu-west-1)
    4. Returns acknowledgment to mongos
        │
        ▼
MONGOS:
    Returns success to application
    
TOTAL LATENCY (EU user → EU mongos → EU shard):
    ~5-10ms (all within eu-west-1 region)

WITHOUT ZONE SHARDING (data in us-east-1):
    EU user → US mongos → US shard → back to EU user
    ~150-200ms (cross-Atlantic round-trip)
    
IMPROVEMENT: 15-30x latency reduction for regional users


WHAT HAPPENS WHEN BALANCER RUNS:

CONFIG SERVER tracks:
    → Chunk ownership: which chunks on which shards
    → Zone tags: which zones assigned to which shards
    → Zone ranges: which shard key ranges belong to which zones

BALANCER (runs periodically):
    1. Scans all chunks
    2. For each chunk: is it on the CORRECT shard for its zone?
    3. If chunk on wrong shard → schedule migration
    4. Migration process:
       a. Source shard sends chunk data to destination shard
       b. Destination shard receives and indexes data
       c. Config server updates chunk ownership
       d. Source shard deletes migrated data
    5. Migration is ONLINE (reads/writes continue during migration)
    6. Brief lock at final cutover point (~1-5ms)

WHEN CHUNKS ARE ON WRONG SHARD:
    → After initial sharding of existing data
    → After adding new zone ranges
    → After adding new shards to existing zones
    → Balancer fixes these automatically (may take minutes to hours)


COMPARISON WITH REDSHIFT DISTRIBUTION:

    REDSHIFT DISTRIBUTION:
    → DISTKEY determines which SLICE gets the row
    → All slices are in SAME REGION (same cluster)
    → No geographic distribution concept
    → Redistribution (resize): takes hours, requires maintenance window
    → No zone concept, no tagging, no geographic routing

    MONGODB ZONE SHARDING:
    → Shard key determines which SHARD gets the document
    → Shards can be in DIFFERENT REGIONS (geographic distribution)
    → Zone tags map shards to geographic regions
    → Zone ranges map shard key values to zones
    → Balancer handles chunk migration automatically
    → Online (no maintenance window for rebalancing)
    → mongos handles routing transparently

    KEY DIFFERENCE:
    → Redshift: data distribution is a PERFORMANCE optimization (co-locate for JOINs)
    → MongoDB: data distribution is BOTH performance AND compliance (geographic routing)
    → Redshift DISTKEY: "put related data on same slice for fast JOINs"
    → MongoDB zone sharding: "put EU data on EU servers for GDPR compliance"

NEW TERMS IN THIS PROJECT:

    Zone:
        → A named grouping of shards (e.g., "EU", "US", "APAC")
        → Documents with matching shard key ranges route to tagged shards
        → Enables geographic data locality and storage tiering

    Zone Range:
        → Mapping from shard key value range to zone name
        → { customer_region: "EU" } → zone "EU"
        → Tells balancer: keep these chunks on "EU"-tagged shards

    Shard Tag:
        → Label attached to a shard (e.g., shard-eu tagged as "EU")
        → Links physical shard to logical zone
        → One shard can have multiple tags (geographic + storage tier)

    Balancer:
        → Background process that moves chunks between shards
        → Ensures zone rules are satisfied
        → Runs on config server
        → Can be scheduled (balancer window) for off-peak hours

    Targeted Query:
        → Query includes shard key → mongos routes to ONE shard
        → Fast: no cross-region network hops
        → Example: find({ customer_region: "EU", ... })

    Scatter-Gather Query:
        → Query doesn't include shard key → mongos queries ALL shards
        → Slower: cross-region network round-trips
        → Example: find({ status: "shipped" }) → hits US + EU + APAC

    Data Sovereignty:
        → Legal requirement for data to reside in specific geographic region
        → GDPR (EU), APPI (Japan), Privacy Act (Australia)
        → Zone sharding provides database-level enforcement

    Crypto-Shredding + Zone Sharding (combined):
        → Project 38: encrypt PII fields with per-region keys
        → Project 39: keep data in correct region
        → Together: EU data encrypted with EU key, stored on EU shard
        → Delete EU key → EU data unreadable (even on EU shard)
        → Complete compliance: WHERE + HOW data is protected

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "ALL DATA ON ONE SHARD AFTER ENABLING ZONES"
    SITUATION: sharded collection, added zone tags, but data doesn't move
    WHY: balancer may be disabled or slow for large collections
    ✅ FIX:
    → Verify balancer is running: sh.isBalancerRunning()
    → Check migration status: sh.status()
    → For large collections: migration may take hours/days
    → Pre-split chunks before adding zone ranges (faster convergence)

PITFALL 2: "SCATTER-GATHER QUERY TIMEOUT"
    SITUATION: global aggregation times out
    WHY: query spans 3 regions → cross-region latency × 3
    → Large result sets transferring across regions
    ✅ FIX:
    → Add $match with customer_region when possible (targeted)
    → Use allowDiskUse: true for large aggregations
    → Increase socket timeout for global queries
    → Pre-compute global aggregations periodically (Project 20 computed pattern)
    → Store global summaries in a non-sharded collection

PITFALL 3: "CUSTOMER MOVES REGIONS — SHARD KEY CAN'T CHANGE"
    SITUATION: German customer moves to US
    → customer_region: "EU" can't change (shard key immutable)
    → New orders should route to US but old orders stuck in EU
    WHY: shard key immutability (fundamental MongoDB limitation)
    ✅ FIX:
    → Soft-delete EU document: { _archived: true, _moved_to: "US" }
    → Create new document with customer_region: "US"
    → Application handles transition gracefully
    → This is rare (<1% of customers) — design for the 99%

PITFALL 4: "mongos IN WRONG REGION"
    SITUATION: EU customer connects to US mongos
    → mongos routes to EU shard correctly
    → BUT: client → US mongos → EU shard → US mongos → client
    → Two cross-Atlantic hops instead of zero
    WHY: mongos deployment doesn't match user geography
    ✅ FIX:
    → Deploy mongos in EVERY region where users exist
    → DNS-based routing: EU users → EU mongos endpoint
    → Atlas Global Clusters: handles this automatically
    → Application: use region-specific connection string or global DNS

PITFALL 5: "ZONE VIOLATION DURING COMPLIANCE AUDIT"
    SITUATION: auditor runs sh.status() and sees EU chunks on US shard
    WHY: 
    → Recently added zone ranges, balancer hasn't migrated yet
    → Balancer was disabled during maintenance and not re-enabled
    → New data inserted faster than balancer can migrate
    ✅ FIX:
    → Pre-split chunks before zone configuration
    → Never disable balancer for more than a few hours
    → Automated monitoring: alert on zone violations
    → Schedule compliance checks AFTER balancer convergence

ZONE SHARDING IMPACTS DOCUMENT DESIGN:

WITHOUT ZONE SHARDING (Projects 1-38):
    {
        _id: ObjectId(),
        order_number: "ORD-2024-001",
        customer: {
            name: "Hans Mueller",

        },
        items: [...],
        payment: { total: 89.99 },

    → Country is inside nested customer object
    → Routing is not a concern

WITH ZONE SHARDING (Project 39):
    {
        _id: ObjectId(),
        customer_region: "EU",              ← NEW: top-level, part of shard key
        order_number: "ORD-2024-EU-001",
        customer: {
            name: "Hans Mueller",
            email: "hans@example.de",
            country: "DE"
        },
        items: [...],
        payment: { total: 89.99 },
        status: "confirmed"
    }
    
    → customer_region MUST be top-level (shard key fields must be top-level)
    → customer_region MUST be set at insert time (immutable after)
    → customer_region is DENORMALIZED (derived from country, stored separately)
    → Every query SHOULD include customer_region for targeted routing

DESIGN RULES FOR ZONE-SHARDED COLLECTIONS:

    1. Zone-determining field MUST be top-level
       ❌ Shard key: { "customer.country": 1 }  → nested field complications
       ✅ Shard key: { customer_region: 1, _id: 1 }  → clean, explicit

    2. Zone-determining field MUST be set before insert
       ❌ Insert without region → MongoDB can't route correctly
       ✅ Application determines region → sets field → then inserts

    3. Zone-determining field CANNOT change
       ❌ db.orders.updateOne({ _id: x }, { $set: { customer_region: "US" } })
         → Error: cannot modify shard key
       ✅ Create new document in new region, archive old document

    4. Every operational query SHOULD include zone field
       ❌ db.orders.find({ status: "shipped" })  → scatter-gather (slow)
       ✅ db.orders.find({ customer_region: "EU", status: "shipped" })  → targeted (fast)

    5. Analytical queries MAY omit zone field (scatter-gather acceptable)
       ✅ db.orders.aggregate([{ $group: { _id: "$customer_region", ... } }])
         → Scatter-gather is fine for analytics (runs less frequently)

COMPARISON WITH REDSHIFT:
    → Redshift DISTKEY doesn't require data model changes
    → You just specify DISTKEY(column) in CREATE TABLE
    → MongoDB zone sharding requires: new field + application logic + shard key design
    → More effort upfront, but the result is much more powerful
    → Geographic routing is IMPOSSIBLE in Redshift without separate clusters

# ═══════════════════════════════════════════════════════
# ZONE SHARDING MONITORING & COMPLIANCE VERIFICATION
#
# Production monitoring script for zone-sharded clusters
# Run periodically (hourly) for compliance assurance
# ═══════════════════════════════════════════════════════

from pymongo import MongoClient
from datetime import datetime
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("zone_monitor")

MONGO_URI = "mongodb+srv://admin:pass@shopstream-global.mongodb.net/"

EXPECTED_ZONES = {
    "US": "shard-us",
    "EU": "shard-eu",
    "APAC": "shard-apac"
}

def monitor_zone_compliance():
    """
    Complete zone sharding monitoring:
    1. Verify zone configuration is intact
    2. Check document distribution per zone
    3. Verify no zone violations exist
    4. Check balancer health
    5. Measure cross-region query latency
    """
    
    client = MongoClient(MONGO_URI)
    admin_db = client["admin"]
    config_db = client["config"]
    db = client["shopstream"]
    
    logger.info(f"{'='*70}")
    logger.info(f"ZONE SHARDING COMPLIANCE MONITOR")
    logger.info(f"Timestamp: {datetime.utcnow().isoformat()}")
    logger.info(f"{'='*70}")
    
    all_passed = True
    
    # ─── CHECK 1: ZONE CONFIGURATION ─────────────────
    logger.info(f"\n--- CHECK 1: Zone Configuration ---")
    
    # Verify shard tags
    shards = list(config_db.shards.find())
    for shard in shards:
        shard_name = shard["_id"]
        tags = shard.get("tags", [])
        logger.info(f"  Shard {shard_name}: tags = {tags}")
        
        # Verify expected tags
        for zone, expected_shard in EXPECTED_ZONES.items():
            if expected_shard == shard_name and zone not in tags:
                logger.error(f"  ❌ Shard {shard_name} MISSING tag '{zone}'")
                all_passed = False
    
    # Verify zone ranges
    tags = list(config_db.tags.find({"ns": "shopstream.orders"}))
    logger.info(f"\n  Zone ranges for shopstream.orders:")
    for tag in tags:
        logger.info(f"    {tag['tag']}: {tag['min']} → {tag['max']}")
    
    if len(tags) < len(EXPECTED_ZONES):
        logger.error(f"  ❌ Expected {len(EXPECTED_ZONES)} zone ranges, found {len(tags)}")
        all_passed = False
    else:
        logger.info(f"  ✅ All {len(tags)} zone ranges configured")
    
    # ─── CHECK 2: DOCUMENT DISTRIBUTION ──────────────
    logger.info(f"\n--- CHECK 2: Document Distribution ---")
    
    distribution = list(db.orders.aggregate([
        {"$group": {
            "_id": "$customer_region",
            "count": {"$sum": 1}
        }},
        {"$sort": {"_id": 1}}
    ]))
    
    total_docs = 0
    for region in distribution:
        count = region["count"]
        total_docs += count
        logger.info(f"  Region {region['_id']}: {count:>12,} documents")
    logger.info(f"  Total:       {total_docs:>12,} documents")
    
    # ─── CHECK 3: CHUNK DISTRIBUTION PER SHARD ───────
    logger.info(f"\n--- CHECK 3: Chunk Distribution ---")
    
    chunks = list(config_db.chunks.find(
        {"ns": "shopstream.orders"},
        {"shard": 1, "min": 1, "max": 1}
    ))
    
    shard_chunk_counts = {}
    zone_violations = 0
    
    for chunk in chunks:
        shard = chunk["shard"]
        shard_chunk_counts[shard] = shard_chunk_counts.get(shard, 0) + 1
        
        # Check zone violation: is this chunk on the correct shard?
        min_region = chunk["min"].get("customer_region", "")
        
        if min_region in EXPECTED_ZONES:
            expected_shard = EXPECTED_ZONES[min_region]
            if shard != expected_shard:
                zone_violations += 1
                if zone_violations <= 5:  # log first 5
                    logger.warning(
                        f"  ⚠️ Zone violation: region={min_region} "
                        f"on shard={shard} (expected {expected_shard})"
                    )
    
    for shard, count in sorted(shard_chunk_counts.items()):
        logger.info(f"  Shard {shard}: {count} chunks")
    
    if zone_violations > 0:
        logger.error(f"  ❌ {zone_violations} zone violations detected!")
        logger.error(f"     Balancer should resolve these automatically")
        all_passed = False
    else:
        logger.info(f"  ✅ No zone violations — all chunks on correct shards")
    
    # ─── CHECK 4: BALANCER HEALTH ────────────────────
    logger.info(f"\n--- CHECK 4: Balancer Health ---")
    
    balancer_status = admin_db.command("balancerStatus")
    is_running = balancer_status.get("mode", "off") != "off"
    in_balancer_round = balancer_status.get("inBalancerRound", False)
    
    logger.info(f"  Balancer enabled: {is_running}")
    logger.info(f"  Currently balancing: {in_balancer_round}")
    
    if not is_running:
        logger.error(f"  ❌ Balancer is DISABLED — zone violations won't auto-resolve")
        all_passed = False
    else:
        logger.info(f"  ✅ Balancer is active")
    
    # Check recent migrations
    recent_migrations = list(config_db.changelog.find(
        {"what": "moveChunk.commit"},
        {"time": 1, "details.from": 1, "details.to": 1}
    ).sort("time", -1).limit(5))
    
    if recent_migrations:
        logger.info(f"\n  Recent chunk migrations:")
        for m in recent_migrations:
            details = m.get("details", {})
            logger.info(
                f"    {m['time']}: {details.get('from', '?')} → {details.get('to', '?')}"
            )
    
    # ─── CHECK 5: TARGETED QUERY LATENCY PER REGION ──
    logger.info(f"\n--- CHECK 5: Targeted Query Latency ---")
    
    import time
    
    for region in ["US", "EU", "APAC"]:
        start = time.time()
        result = db.orders.find_one(
            {"customer_region": region},
            {"order_number": 1}
        )
        elapsed = (time.time() - start) * 1000  # ms
        
        status = "✅" if elapsed < 50 else "⚠️" if elapsed < 200 else "❌"
        logger.info(f"  {region} targeted query: {elapsed:.1f}ms {status}")
    
    # Scatter-gather (no region filter)
    start = time.time()
    result = db.orders.find_one({"status": "confirmed"})
    elapsed = (time.time() - start) * 1000
    logger.info(f"  Global scatter-gather: {elapsed:.1f}ms")
    
    # ─── SUMMARY ─────────────────────────────────────
    logger.info(f"\n{'='*70}")
    logger.info(f"COMPLIANCE STATUS: {'✅ ALL CHECKS PASSED' if all_passed else '❌ ISSUES FOUND'}")
    logger.info(f"{'='*70}")
    
    if not all_passed:
        logger.error("ACTION REQUIRED: Review zone violations and balancer status")
        logger.error("GDPR compliance may be at risk")
    
    client.close()
    return all_passed

def generate_compliance_report():
    """
    Generate formal compliance report for auditors
    
    Output: structured report showing:
    → Zone configuration (what SHOULD happen)
    → Current state (what IS happening)
    → Verification results (are they aligned?)
    → Timestamp and operator information
    """
    
    client = MongoClient(MONGO_URI)
    config_db = client["config"]
    db = client["shopstream"]
    
    report = {
        "report_type": "GDPR Data Locality Compliance",
        "generated_at": datetime.utcnow().isoformat(),
        "cluster": "shopstream-global",
        "collection": "shopstream.orders",
        "zones": [],
        "document_counts": {},
        "zone_violations": 0,
        "balancer_status": "",
        "overall_status": "COMPLIANT"
    }
    
    # Zone configuration
    shards = list(config_db.shards.find())
    tags = list(config_db.tags.find({"ns": "shopstream.orders"}))
    
    for shard in shards:
        zone_info = {
            "shard_name": shard["_id"],
            "host": shard["host"],
            "tags": shard.get("tags", []),
            "zone_ranges": []
        }
        
        for tag in tags:
            if tag["tag"] in shard.get("tags", []):
                zone_info["zone_ranges"].append({
                    "zone": tag["tag"],
                    "min": str(tag["min"]),
                    "max": str(tag["max"])
                })
        
        report["zones"].append(zone_info)
    
    # Document distribution
    distribution = list(db.orders.aggregate([
        {"$group": {"_id": "$customer_region", "count": {"$sum": 1}}}
    ]))
    report["document_counts"] = {d["_id"]: d["count"] for d in distribution}
    
    # Zone violations
    chunks = list(config_db.chunks.find({"ns": "shopstream.orders"}))
    violations = 0
    for chunk in chunks:
        min_region = chunk["min"].get("customer_region", "")
        if min_region in EXPECTED_ZONES:
            if chunk["shard"] != EXPECTED_ZONES[min_region]:
                violations += 1
    
    report["zone_violations"] = violations
    if violations > 0:
        report["overall_status"] = "NON-COMPLIANT"
    
    # Balancer
    balancer = client["admin"].command("balancerStatus")
    report["balancer_status"] = balancer.get("mode", "unknown")
    
    client.close()
    
    # Print report
    logger.info(f"\n{'='*70}")
    logger.info("FORMAL COMPLIANCE REPORT")
    logger.info(f"{'='*70}")
    
    import json
    logger.info(json.dumps(report, indent=2, default=str))
    
    return report


if __name__ == "__main__":
    passed = monitor_zone_compliance()
    
    if not passed:
        logger.info("\nGenerating formal compliance report...")
        report = generate_compliance_report()
